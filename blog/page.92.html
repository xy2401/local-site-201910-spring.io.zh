Spring Cloud Data Flow 1.2 RC1发布

发布
马克·波拉克
2017年4月20日

我很高兴代表团队宣布Spring Cloud Data Flow 1.2的第一个候选版本。

注意：开始使用此新版本的一种好方法是遵循参考文档中的《 入门指南》 。

1.2 RC1版本的要点：

组成的任务

此版本引入了组合任务 ！此功能提供了将任务流编排为内聚的工作单元的能力。复杂的ETL管道可以包括顺序执行，并行执行，条件转换或所有上述各项的组合。组合任务功能具有DSL原语和交互式图形界面，可更轻松地快速构建这些类型的拓扑。您可以从参考指南中了解更多信息。

例如，一个ETL作业可能包括多个步骤。拓扑中的每个步骤都可以构建为有限的短期Spring Cloud Task应用程序。借助数据流任务DSL，可以轻松地将多个任务编排为步骤。

task create simple-etl --definition "extractDbToHDFS &&
      <analysisInSpark || enrichAndLoadHawq> &&
      <populateMgmtDashboard || runRegulatoryReport || loadAnalyticsStore>"

这将首先运行extractDbToHDFS然后运行analysisInSpark和enrichAndLoadHawq并行执行，在并行运行其余三个任务之前，等待它们都完成，然后在结束作业之前等待它们全部完成。该拓扑的图形表示如下所示。

实时指标和监控

实时指标现已成为已部署流的操作视图的一部分。流中的应用程序发布其Spring Boot中包含的指标/metrics执行器端点。这包括发送和接收消息的费率。新的服务器Spring Cloud Data Flow Metrics Collector收集这些指标并计算总消息速率。数据流服务器查询Metrics Collector以支持在UI和外壳程序中显示消息速率。有关该体系结构的更多详细信息，请参考参考指南中的“ 监视已部署的应用程序”部分。

下面的屏幕截图显示了time | log具有三个实例的流time和log应用程序。主应用程序框下方的每个点显示每个单独应用程序的消息速率以及一个guid可用于在运行平台的平台上标识应用程序的值。

如下所示，“ 运行时”选项卡也进行了改进，以显示消息速率和平台公开的任何其他指标。对于精通脚本的用户，shell体验还通过runtime apps命令。

伴侣神器

在1.2 M3中引入的随行工件支持得到了一些改进。现在，批量注册工作流急切地为所有现成的应用程序解析和下载元数据工件。查看每个应用程序的受支持属性时，这在Shell或UI中很方便。

OAUTH改进

此更改将为REST-API用户提供其他选项。而不是提供username:password通过BasicAuth组合，用户现在可以在从安全的Spring Cloud Data Flow设置调用RESTful调用时，直接从其OAuth2提供程序检索OAuth2访问令牌，然后在HTTP标头中提供访问令牌。

Spring Cloud Stream应用程序入门-培根。发布

培根。现在可以普遍使用RELEASE 。所有开箱即用的流应用程序都基于Spring Cloud Stream Chelsea构建。发布和Spring Cloud Dalston。发布基础。现有应用程序进行了一些增强和错误修复，此发行系列还带来了新应用程序，例如MongoDB接收器，聚合器处理器，Header-Enricher处理器和PGCopy接收器。

Spring Cloud Task 1.2.0。RC1现在可用

发布
格伦·伦弗罗（Glenn Renfro）
2017年4月14日

我们很高兴地宣布Spring Cloud Task 1.2.0。RC1现在可以通过Github和Pivotal下载存储库获得。非常感谢所有对此版本做出贡献的人。

Spring Cloud Task 1.2.0。RC1提供以下功能：

  • 升级到Spring Cloud Stream切尔西GA。
  • 更名closecontext.enable至closecontext.enabled与启用的属性的Spring Boot样式匹配。
  • 在创建任务时可以设置任务名称TaskExecution在外部，允许启动器在永久性庄园中设置名称。
  • 添加Ordered与任务事件和批处理作业事件的接口，以便用户可以确定何时从其Spring Cloud Task应用程序发出任务或批处理事件。
  • 清理了Spring Cloud Task依赖项。
  • 更新默认任务名称创建，以防止与JMX冲突。
