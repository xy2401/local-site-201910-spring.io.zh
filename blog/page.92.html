Spring Cloud Data Flow 1.2 RC1发布

发布
马克·波拉克
2017年4月20日

我很高兴代表团队宣布Spring Cloud Data Flow 1.2的第一个候选版本。

注意：开始使用此新版本的一种好方法是遵循参考文档中的《 入门指南》 。

1.2 RC1版本的要点：

组成的任务

此版本引入了组合任务 ！此功能提供了将任务流编排为内聚的工作单元的能力。复杂的ETL管道可以包括顺序执行，并行执行，条件转换或所有上述各项的组合。组合任务功能带有DSL原语和交互式图形界面，可更轻松地快速构建这些类型的拓扑。您可以从参考指南中了解更多信息。

例如，一个ETL作业可能包括多个步骤。拓扑中的每个步骤都可以构建为有限的短期Spring Cloud Task应用程序。借助数据流任务DSL，可以轻松地将多个任务编排为步骤。

task create simple-etl --definition "extractDbToHDFS &&
       &&
      "

这将首先运行extractDbToHDFS ，然后并行运行analysisInSpark和enrichAndLoadHawq在并行运行其余三个任务之前等待它们都完成，并在结束作业之前等待它们全部完成。该拓扑的图形表示如下所示。

实时指标和监控

实时指标现已成为已部署流的操作视图的一部分。流中的应用程序发布包含在其Spring Boot /metrics执行器端点中的/metrics 。这包括发送和接收消息的费率。新的服务器Spring Cloud Data Flow Metrics Collector收集这些指标并计算总消息速率。数据流服务器查询Metrics Collector以支持在UI和外壳程序中显示消息速率。有关该体系结构的更多详细信息，请参考参考指南中的“ 监视已部署的应用程序”部分。

下面的屏幕截图显示了一段time | log的总消息速率time | log带有三个time和log应用程序实例的time | log流。主应用程序框下方的每个点显示每个单独应用程序的消息速率以及一个guid值，该guid值可用于标识运行它们的平台上的应用程序。

如下所示，“ 运行时”选项卡也进行了改进，以显示消息速率和平台公开的任何其他指标。对于精通脚本的用户，shell体验还可以通过runtime apps命令包含这些详细信息。

伴侣神器

在1.2 M3中引入的随行工件支持得到了一些改进。现在，批量注册工作流急切地为所有现成的应用程序解析和下载元数据工件。查看每个应用程序的受支持属性时，这在Shell或UI中很方便。

OAUTH改进

此更改将为REST-API用户提供其他选项。现在，当用户针对安全的Spring Cloud Data Flow设置调用RESTful调用时，用户现在可以直接从其OAuth2提供程序检索OAuth2访问令牌，然后在HTTP标头中提供访问令牌，而无需通过BasicAuth提供username:password组合。 。

Spring Cloud Stream应用程序入门-培根。发布

培根。现在可以普遍使用RELEASE 。所有开箱即用的流应用程序都基于Spring Cloud Stream Chelsea构建。发布和Spring Cloud Dalston。发布基础。现有应用程序进行了一些增强和错误修复，此发行系列还带来了新应用程序，例如MongoDB接收器，聚合器处理器，Header-Enricher处理器和PGCopy接收器。

为了方便起见，我们生成了bit.ly链接 ，其中包含docker和maven工件的最新坐标。

Spring Cloud Task应用入门-Belmont。RC1

App Starters Belmont。RC1版本现已完成。为了在Spring Cloud Data Flow中支持组合任务功能，我们添加了一个名为Composed Task Runner的新应用程序。这是一个在有向图中执行其他任务的任务，该有向图中由通过--graph命令行参数传入的DSL指定。

贝尔蒙特。RC1基于Spring Cloud Task 1.2 RC1和Spring Cloud Dalston。发布基础。

为了方便起见，我们生成了bit.ly链接 ，其中包含docker和maven工件的最新坐标。

下一步是什么？

1.2.0。发布即将到来。我们的目标是在未来2-3周内完成包装。核心版本发布后，Spring Cloud Data Flow的运行时实现将立即赶上并适应这一基础。

反馈很重要。请在StackOverflow和GitHub上与我们联系，以获取问题和功能要求。我们也欢迎您的贡献！感谢您对改善Spring Cloud Data Flow生态系统的任何帮助。
