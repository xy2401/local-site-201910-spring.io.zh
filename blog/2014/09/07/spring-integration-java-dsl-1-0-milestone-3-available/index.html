Spring Integration Java DSL 1.0里程碑3可用

发布
阿尔特姆·比兰（Artem Bilan）
2014年9月7日

我们很高兴地宣布，已经发布了用于Spring Integration 1.0 Milestone 3的Java DSL。
请结合使用Milestone仓库和Maven或Gradle，
下载一个
发行档案 ，
或查看项目主页
想要查询更多的信息。

自上一个里程碑2以来，
此里程碑包括一些新功能和进一步的改进，以及许多错误修复，
GA版本将于10月底左右发布。

首先，感谢所有对此Spring Integration Extension感兴趣的人，提供反馈，
分享他们的想法，甚至做出贡献。

以下是主要更改的摘要：

Spring Integration 4.1

当前里程碑提供了与
Spring Integration 4.1
以及Spring Framework 4.1 。

更改包括：

  • Spring Integration 4.1提供Channel names late resolution ，
    允许DSL IntegrationFlow的定义仅基于渠道名称，并避免了所需的解决方法
    @DependsOn从一种流向另一种流

  • 当Spring集成时，Spring Framework 4.1中包含了许多消息传递性能改进。
    （及其DSL）基于Spring Messaging模块；

  • Spring Framework 4.1引入了[SpEL编译器]（ http：//docs.spring
    .io / spring / docs / current / spring-framework-reference / html / expressions.html＃expressions-spel-compilation）。
    对于Spring Integration（在运行时广泛使用SpEL）非常有用。您可以使用以下命令启用SpEL编译器：
    spring.expression.compiler.mode系统属性，值为IMMEDIATE要么MIXED 。

当然，与Spring Integration 4.0的兼容性仍然保持。

IntegrationFlow Lambda

的IntegrationFlow bean定义现在可以表示为Lambda同样，为了简化DSL使用：

@Bean
public IntegrationFlow lambdaFlow() {
    return f -> f.filter("World"::equals)
                 .transform("Hello "::concat)
                 .handle(System.out::println);
}

其实我们只是做了IntegrationFlow单一方法的功能界面
define(IntegrationFlowDefinition flow)和IntegrationFlowBeanPostProcessor只是转换那个“假” bean
到真实StandardIntegrationFlow基于隐式IntegrationFlowBuilder 。唯一的限制是这样的
定义，它基于隐式DirectChannel与lambdaFlow.inputBean name称。

支持增加

如您在最后所注意到的IMAP采样.javaMailProperties()和.enrichHeaders()语法流利
指定Properties和Map作为内联对象。不幸的是，Java没有提供Builder API
对于那些简单的类，因此我们为您做到这一点并提供功能接口来构建Properties和Map
来自Lambdas。例如， Mail.headers()提供一个MailHeadersBuilder -的扩展MapBuilder ，
可以在.enrichHeaders()以及：
java @Bean public IntegrationFlow sendMailFlow() { return f -> f.enrichHeaders(Mail.headers().subject("foo").from("[email protected]").to("[email protected]")) .handle(Mail.outboundAdapter("smtp.gmail.com") .credentials("user", "pw") .protocol("smtp"))); }

当然，这相当于从XML配置。
