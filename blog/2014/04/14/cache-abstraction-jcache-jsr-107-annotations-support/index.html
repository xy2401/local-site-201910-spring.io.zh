缓存抽象：JCache（JSR-107）注释支持

工程
斯特凡·尼科尔（StéphaneNicoll）
2014年4月14日

Spring的缓存抽象从Spring 3.1开始可用，是时候向它展示更多的爱了。在本文中，我想向您介绍该领域的主要改进，即JCache（JSR-107）批注支持。

您可能已经听说过， JSR-107毕竟是最终提案，比最初的提案晚了13年。对于那些熟悉Spring缓存注释的人，下表描述了Spring注释和JSR-107对应项之间的映射：

Spring JSR-107
@Cacheable @CacheResult
@CachePut @CachePut
@CacheEvict @CacheRemove
@CacheEvict(allEntries=true) @CacheRemoveAll

JCache批注

首先，让我们看一下每个注释并描述如何使用它们。这将是一个机会，可以更好地了解它们对您使用Spring注释所使用的内容以及更重要的是这些注释所带来的新功能的支持。

@CacheResult

@CacheResult相当类似于@Cacheable ，以下改写原始示例使用@CacheResult注解：

@CacheResult(cacheName = "books")
public Book findBook(ISBN isbn) {...}

密钥生成可以使用CacheKeyGenerator接口。如果未指定具体实现，则每个规范的默认实现都采用所有参数，除非一个或多个参数用@CacheKey注释，在这种情况下，仅使用那些注释。假设上面的方法现在需要一个额外的属性，该属性不应该是键的一部分，这就是我们使用JCache编写它的方式：

@CacheResult(cacheName = "book")
public Book findBook(@CacheKey ISBN isbn, boolean checkWarehouse) { ... }

@CacheResult带来了异常缓存的概念：每当方法执行失败时，就有可能缓存为防止再次调用该方法而引发的异常。假设InvalidIsbnNotFoundException如果ISBN的结构无效，则抛出该异常。这是一个永久性的失败，无法使用该参数检索任何书籍。以下内容缓存了该异常，以便使用相同的无效ISBN进行的进一步调用直接引发该缓存的异常，而不是再次调用该方法。

@CacheResult(cacheName = "books", exceptionCacheName = "failures"
             cachedExceptions = InvalidIsbnNotFoundException.class)
public Book findBook(@CacheKey ISBN isbn) { ... }

当然，盲目地抛出缓存的异常可能会非常混乱，因为调用堆栈可能与当前调用上下文不匹配。我们通过复制具有一致调用堆栈的异常来尽力确保stacktrace匹配。

JCache有一个很酷的概念CacheResolver允许解析缓存以在运行时使用。由于JCache支持常规缓存和异常缓存，因此CacheResolver要使用的实例由CacheResolverFactory 。显而易见的默认设置是根据cacheName和exceptionCacheName属性分别。但是，也可以自定义工厂以用于每个操作。

@CacheResult(cacheName = "books", cacheResolverFactory = MyFactory.class)
public Book findBook(@CacheKey ISBN isbn) { ... }

最后， @CacheResult有个skipGet可以启用以始终调用该方法的属性，而不管缓存的状态如何。这实际上与我们自己使用@CachePut 。

@CachePut

尽管批注具有相同的名称，但JCache中的语义完全不同。我们这本书的一个简单更新就是这样写的

@CachePut(value = "books", key = "#p0")
public Book update(ISBN isbn, Book updatedBook) { ... }

虽然JCache会要求您这样写

@CachePut(cacheName = "books")
public void update(ISBN isbn, @CacheValue Book updatedBook) { ... }

请注意，即使updatedBook不应该是键的一部分，我们不必添加@CacheKey第一个论点。这是因为用注释的参数@CacheValue自动从密钥生成中排除。

至于@CacheResult ， @CachePut允许管理在执行该方法时抛出的任何异常，从而防止在抛出的异常与注释中指定的过滤器匹配时发生put操作。

最后，可以控制是否在调用带注释的方法之前或之后更新缓存。当然，如果以前进行了更新，则不会发生异常处理。

@CacheRemove和@CacheRemoveAll

这些真的类似于@CacheEvict和@CacheEvict(allEntries = true)分别。 @CacheRemove具有特殊的异常处理，以防止被注释的方法引发与注释上指定的过滤器匹配的异常而驱逐。

其它功能

缓存默认值

@CacheDefaults是类级别的注释，它使您可以共享在类上定义的任何缓存操作上的通用设置。这些是：

  • 缓存名称
  • 风俗CacheResolverFactory
  • 风俗CacheKeyGenerator

在下面的示例中，任何与缓存相关的操作都将使用books快取：

@CacheDefaults(cacheName = "books")
public class BookRepositoryImpl implements BookRepository {

    @CacheResult
    public Book findBook(@CacheKey ISBN isbn) { ... }
}

启用JSR-107支持

JCache支持的实现使用我们自己的Cache和CacheManager抽象意味着您可以使用现有的CacheManager基础架构，但仍使用标准注释！

要启用对Spring缓存注释的支持，您习惯了@EnableCaching或者 xml元素，例如：

@Configuration
@EnableCaching
public class AppConfig {
    @Bean
    public CacheManager cacheManager() { ...}

    ...
}

那么，如何将标准注释的支持带入混合呢？好吧，不多。只需添加JCache API和spring-context-support类路径中的模块（如果尚未安装），将被设置。

现有的基础结构实际上是在寻找JCache API的存在，当与Spring的JCache支持一起发现时，它还将配置必要的基础结构以支持标准注释。

包起来

简而言之，如果您已经在使用Spring的缓存抽象，并且想尝试标准的批注，那么开始就需要在项目中再添加两个依赖项。

想要试一试？抓住Spring 4.1 的每晚SNAPSHOT版本并添加javax.cache:cache-api:1.0.0和org.springframework:spring-context-support:4.1.0.BUILD-SNAPSHOT项目的依赖项。如果您需要更多详细信息， 文档也已更新。
