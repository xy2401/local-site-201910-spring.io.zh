JavaScript模块化（没有流行语）

工程
布赖恩·克洛泽尔
2014年4月11日

大约十年前， Adrian Colyer撰写了一篇令人难忘的博客文章 ，其中给出了有关面向方面编程（AOP）的最佳解释：简洁明了的样式，准确的内容，没有流行语。如果您已经看过本系列的前两篇 文章 ，您可能已经注意到Sagan应用程序的客户端模块中的一些体系结构选择，包括JavaScript模块的使用 。

在这篇文章中，我想向您介绍Adrian帖子风格的JavaScript模块基础知识：清晰，简单，准确，没有流行语！

为什么JavaScript也需要模块化

如果像我一样具有Java背景，那么某些JavaScript语言功能对您来说可能会有些奇怪：

  • 其强大的原型性质
  • 没有软件组件单元：没有使用该语言构建的软件包或模块
  • 浏览器应用程序和服务器应用程序之间的编程风格差异

简单的Web应用程序通常如下所示：

<html>
<head>
	<!-- scripts are loaded **sequentially** -->
	<script src="jquery.js"></script>
	<script src="widget.js"></script>
	<!-- all objects are in the global namespace, 
			 no way to isolate the code completely! -->
	<script>
		window.$; // this is jquery
	</script>
</head>
</html>

开发人员经常尝试重用功能块并管理它们之间的依赖关系。在前一种情况下，这确实很难实现：您必须以正确的顺序声明脚本，并避免名称冲突， 而无需任何依赖项的元数据 。随着应用程序的增长，这项任务成为噩梦：想象一下在没有软件包或依赖项管理基础结构的情况下管理Java应用程序！

了解JavaScript创作格式：commonJS和AMD

幸运的是，JS社区非常了解模块化的必要性，并且在最近几年中，出现了两种广泛采用的解决此问题的方法：

AMD（异步模块定义）是其中之一：

// defining a module that depends on "graphmodule",
// injected in the function definition as an argument
define(["graphmodule"],
  function (graph) {
    // this module exports a "draw" function
    return function draw(data) {
      return graph.piechart(data);
    }
  }
);

由于以下原因，AMD格式经常用于浏览器应用程序：

  • 它允许变量在工厂函数内部进行作用域化（相对于将所有内容置于窗口全局作用域中）
  • 依赖项可能无法同步使用，例如，它们需要通过异步网络请求来获取
  • 模块在回调中定义，然后在所有依赖项加载后运行

CommonJS是另一种创作格式 ，通常用在服务器端的Node.js应用程序中，也用在浏览器应用程序中。


/**
 * Our module's dependencies and API 
 **/
// we fetch the graph module as a dependency
var graph = require('graphmodule');

// we decorate the `exports` variable to export our function
exports.draw = drawFunc;

/**
 * API implementation
 **/
function drawFunc(data) {
    return graph.piechart(data);
}

您可能发现了与AMD的一些区别：

  • 此创作格式可以在浏览器和服务器上使用！
  • 因此，您可以在无头浏览器和 Node.js单元测试中测试模块！
  • 模块定义没有包装在define

下一步是什么？

下一版本的JavaScript（技术上称为ECMAScript 6）将标准化JavaScript模块化 ，尽管所使用的语法与我们上面看到的AMD或commonJS格式略有不同。
此外，EcmaScript 6将解决该领域的许多问题，但还将提供语法糖来定义类似OO的类。

无论您是在客户端还是服务器上使用JavaScript的任何地方，都应考虑采用模块化方法-出于我们在Java中所做的所有相同原因。要精通JS模块技术，学习曲线很小，但是结果值得。

如果您不熟悉 JavaScript模块化，建议您阅读“编写CommonJS模块”教程以及我们在Sagan中使用的模块加载器： curl 。
并且一旦您感到熟悉，就可以在Sagan应用程序中查看我们的模块定义 。
