介绍Spring Sync

工程
克雷格·沃尔斯
2014年10月22日

今天早些时候，我宣布了Spring Sync的第一个里程碑版本，这是一个新项目，它通过使用基于补丁的交换来解决客户端应用程序与Spring后端之间的有效通信。由于这是一个新项目，所以我认为这是个向您展示Spring Sync可以做什么的好时机。

此处给出的示例引用了Spring REST Todos示例和/或Todo该示例项目中的类。

创建和应用补丁

在最低层次上，Spring Sync提供了一个用于生成补丁并将其应用于Java对象的库。的Patch class是此库的核心，捕获可应用于对象的更改以使其与另一个对象同步。

的Patch类的目标是通用的，不与补丁的任何特定表示直接关联。就是说，它受到JSON补丁的启发，Spring Sync提供了对创建和序列化的支持Patch实例作为JSON补丁。Spring Sync的未来版本可能包括对其他补丁表示的支持。

创建补丁的最简单方法是在两个Java对象之间进行区别：

Todo original = ...;
Todo modified = ...;
Patch patch = Diff.diff(original, modified);

在这里Diff.diff()方法将比较两者Todo对象并产生一个Patch描述了它们之间的区别。

一旦你有一个Patch ，可以通过将对象传递到apply()方法：

Todo patched = patch.apply(original, Todo.class);

请注意diff()和apply()方法是彼此相反的。因此，补丁Todo在这些示例中，应与修改后的相同Todo将补丁应用到原始文件后。

如我所说， Patch与任何特定的补丁表示分离。但是Spring Sync提供了JsonPatchMaker作为要转换的实用程序类Patch往/返杰克逊的东西JsonNode实例中JsonNode是一个ArrayNode每个JSON补丁规范中包含零个或多个操作。例如，将Patch到JsonNode包含JSON补丁：

JsonNode jsonPatchNode = JsonPatchMaker.toJsonNode(patch);

同样， Patch可以从JsonNode像这样：

Patch patch = JsonPatchMaker.fromJsonNode(jsonPatchNode);

注意JsonPatchMaker是（反）序列化的临时解决方案Patch与JSON补丁之间的对象。在以后的版本中，它将被更永久的解决方案取代。

应用差分同步

创建补丁要求您同时具有对象实例的前后，才能从中计算差异。尽管未将它们称为“之前”和“之后”，但尼尔·弗雷泽（Neil Fraser）在论文中描述的差分同步算法本质上定义了一种控制器方式，通过该方式，可以在两个或多个网络节点之间创建，共享和应用补丁。 （也许是客户端和服务器，但不一定仅适用于客户端-服务器方案）。

应用差异同步时，每个节点维护一个资源的两个副本：

  • 本地节点可能会更改的自己的工作副本。
  • 卷影副本，它是本地节点对远程节点的工作副本的外观的了解。

节点可以对其资源的本地副本进行所需的任何更改。定期地，该节点将通过将本地节点与其为远程节点维护的卷影副本进行比较来产生补丁。然后，它将补丁发送到远程节点。发送补丁程序后，假定远程节点将应用补丁程序，那么节点将在阴影上复制其本地副本，因此，其对远程节点资源的理解与本地资源同步。

收到补丁程序后，节点必须将补丁程序应用于发送给该补丁程序的节点所保留的阴影以及其自己的本地副本（自身可能已更改）。

Spring Sync通过其同步支持差分同步DiffSync类。创建一个DiffSync ，您必须为其提供一个ShadowStore以及可以为其应用修补程序的对象类型：

ShadowStore shadowStore = new MapBasedShadowStore();
shadowStore.setRemoteNodeId("remoteNode");
DiffSync diffSync = new DiffSync(shadowStore, Todo.class);

一旦你有一个DiffSync您可以使用它来应用Patch到一个对象：

Todo patched = diffSync.apply(patch, todo);

的apply()方法会将修补程序应用于给定对象以及同一对象的卷影副本。如果尚未创建卷影副本，它将通过深度克隆给定对象来创建一个卷影副本。

的ShadowStore在哪DiffSync维护其远程节点的卷影副本的副本。对于任何给定的节点，可能会有多个影子存储，每个与其处理的远程节点一个。如您在示例中所见， remoteNodeId属性设置为唯一标识远程节点。在客户端-服务器拓扑中，服务器可以使用会话ID来标识远程节点。同时，客户端（可能仅与一个中央服务器共享资源）可以使用他们想要标识服务器节点的任何标识符。

DiffSync也可以用来创建一个Patch从存储的卷影副本中：

Patch patch = diffSync.diff(todo);

创建补丁时，将从阴影中检索存储的阴影ShadowStore并与给定的对象进行比较。与差分同步流保持一致，一旦生成了补丁，给定的对象将被复制到阴影上。

值得注意的是DiffSync与...合作Patch与任何特定修补程序表示分离的对象。因此， DiffSync本身也与补丁表示分离。

将DiffSync上网

在单个节点上创建和应用补丁有点没有意义。当两个或多个节点共享并处理同一资源并且您需要每个节点保持同步（合理时）时，差异同步才真正发挥作用。因此，Spring Sync还提供DiffSyncController一个Spring MVC控制器，该控制器处理HTTP PATCH请求，对资源应用差分同步。

最简单的配置方式DiffSyncController是创建一个带有注解的Spring配置类@EnableDifferentialSynchronization并扩展DiffSyncConfigurerAdapter类：

@Configuration
@EnableDifferentialSynchronization
public class DiffSyncConfig extends DiffSyncConfigurerAdapter {

	@Autowired
	private PagingAndSortingRepository<Todo, Long> repo;
	
	@Override
	public void addPersistenceCallbacks(PersistenceCallbackRegistry registry) {
		registry.addPersistenceCallback(new JpaPersistenceCallback<Todo>(repo, Todo.class));
	}
	
}

除其他事项外， @EnableDifferentialSynchronization声明一个DiffSyncController bean ，为其提供一个PersistenceCallbackRegistry和一个ShadowStore 。

的PersistenceCallbackRegistry是的注册表PersistenceCallback通过哪个对象DiffSyncController将检索并保留修补的资源。的PersistenceCallback界面启用DiffSyncController与资源的特定于应用程序的持久性选择脱钩。例如，这是一个实现PersistenceCallback与Spring Data一起使用 CrudRepository坚持Todo对象：

package org.springframework.sync.diffsync.web;

import java.util.List;

import org.springframework.data.repository.CrudRepository;
import org.springframework.sync.diffsync.PersistenceCallback;

class JpaPersistenceCallback<T> implements PersistenceCallback<T> {
	
	private final CrudRepository<T, Long> repo;
	private Class<T> entityType;

	public JpaPersistenceCallback(CrudRepository<T, Long> repo, Class<T> entityType) {
		this.repo = repo;
		this.entityType = entityType;
	}
	
	@Override
	public List<T> findAll() {
		return (List<T>) repo.findAll();
	}
	
	@Override
	public T findOne(String id) {
		return repo.findOne(Long.valueOf(id));
	}
	
	@Override
	public void persistChange(T itemToSave) {
		repo.save(itemToSave);
	}
	
	@Override
	public void persistChanges(List<T> itemsToSave, List<T> itemsToDelete) {
		repo.save(itemsToSave);
		repo.delete(itemsToDelete);
	}

	@Override
	public Class<T> getEntityType() {
		return entityType;
	}
	
}

至于ShadowStore给DiffSyncController ，这将是MapBasedShadowStore默认。但是您可以覆盖getShadowStore()来自的方法DiffSyncConfigurerAdapter指定不同的影子存储实现。例如，您可以这样配置基于Redis的影子存储：

@Autowired
private RedisOperations<String, Object> redisTemplate;

@Override
public ShadowStore getShadowStore() {
	return new RedisShadowStore(redisTemplate);
}

不管执行哪个ShadowStore您选择时，将声明一个会话范围的bean，以确保每个客户端都将收到他们自己的影子存储实例。

在处理PATCH请求时， DiffSyncController将应用差分同步流的一个周期：

  1. 它将修补程序应用于资源的服务器副本和发送PATCH的客户端的卷影副本。
  2. 通过将其本地资源与卷影副本进行比较，它将创建一个新补丁。
  3. 它将用资源的本地副本替换卷影副本。
  4. 它将把响应上的新补丁发送给客户端。

就像Patch和DiffSync ， DiffSyncController与任何特定的修补程序格式分离。Spring Sync确实提供JsonPatchHttpMessageConverter但是，这样DiffSyncController可以使用JSON Patch格式的补丁来接收和响应，并指定内容类型为“ application / json-patch + json”。

结论

如您在此处看到的，Spring Sync旨在提供一种在客户端和服务器（或共享资源的任何节点集）之间进行有效通信和同步的方法。它为生成和应用补丁程序提供了低级支持，并为使用差分同步提供了高级支持。尽管它支持JSON修补程序，但在很大程度上与任何特定的修补程序格式无关。

这仅仅是开始。除其他外，我们希望…

  • 补充DiffSyncController与WebSocket / STOMP的基于HTTP的差分同步，用于全双工补丁程序通信。
  • 不断完善差分同步实现，以支持资源版本控制和其他技术，以避免补丁冲突。
  • 支持在客户端Android应用程序中使用Spring Sync。
