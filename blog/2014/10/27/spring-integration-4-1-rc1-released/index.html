Spring Integration 4.1 RC1发布

发布
阿尔特姆·比兰（Artem Bilan）
2014年10月27日

尊敬的Spring社区，

我们很高兴宣布Spring Integration 4.1 Release Candidate可用。
请使用带有Maven或Gradle的Milestone资料库 ，或下载发行资料档案库 ，以进行试用。

该版本包括许多新功能和改进，以及许多错误修复。GA版本
计划在11月初。

首先，感谢所有提供反馈的人
4.1里程碑1和
提交的报告（错误或新功能）。特别感谢那些通过请求请求提供贡献的人。
这是自里程碑以来的主要变化的摘要：

Web套接字支持

此功能是在4.1里程碑1中引入的，但是已经解决了几个问题，我们现在
提供了一些示例，以更好地了解如何在Spring Integration应用程序中使用Web套接字：
基本和
踩聊天 。

JDK8可选的一致处理

如果您使用的是Java 8，则可以使用Optional服务方法参数的容器。例如：

public void optionals(@Payload("@myConvert.conv(payload)") Optional<Bar> payload,
        @Header(value="foo") Optional<String> header)

在这种情况下@myConvert.conv(payload)退货null ， payload变量将包含一个Optional.empty() 。
同样的事情header变量-如果没有foo请求中的标头Message 。可以用
作为替代required上的属性@Header注解。

布线滑模

现在支持“ 路由清单”模式。而是
比简单static list of channel names ，我们介绍了RoutingSlipRouteStrategy ，提供
基于请求的动态运行时路由Message和reply object 。SpEL也受支持：

<header-enricher input-channel="input" output-channel="process">
	<routing-slip value="channel1; request.headers[myRoutingSlipChannel];
	                        routingSlipRoutingStrategy;"/>
</header-enricher>

这种模式在复杂，动态的情况下非常有用，在这种情况下，可能难以配置多个路由器来确定
消息流。通过此增强功能，当出现一条消息时
到达没有端点的端点output-channel ，请参考路由清单以确定下一个要访问的频道
消息将被发送。布线单用完后，正常replyChannel处理简历。

幂等接收器模式

在此版本中，我们实现了
幂等接收器是一流的功能。
以前，用户必须通过使用自定义来实现此模式MessageSelector在一个 ， 例如。
该框架现在支持此功能，因为Advice可以应用于任何使用端点的组件：

<idempotent-receiver endpoint="endpoint1, foo*"
				     metadata-store="store"
					 discard-channel="duplicates"
					 key-expression="payload.invoiceNumber"/>

这将创建一个AOP IdempotentReceiverInterceptor这适用于MessageHandler#handleMessage在端点内
在哪里id匹配提供的其中之一endpoint模式。

如果discard-channel省略，重复的消息仍将发送到消息处理程序，但它将包含一个
duplicateMessage标头，允许用户代码采取进一步的措施。

对于JavaConfig， @IdempotentReceiver提供了注释，但是IdempotentReceiverInterceptor@Bean必须
也可以配置：

散布聚集模式

散布集会
现在提供企业集成模式：

<!--Auction scenario-->
<scatter-gather input-channel="inputAuction" output-channel="output"
                scatter-channel="auctionChannel">
	<gatherer release-strategy-expression="^[payload gt 5] != null or size() == 3"/>
</scatter-gather>

<!--Distribution scenario-->
<scatter-gather input-channel="inputDistribution" output-channel="output"
                gather-channel="gatherChannel">
	<scatterer apply-sequence="true">
		<recipient channel="distribution1Channel"/>
		<recipient channel="distribution2Channel"/>
		<recipient channel="distribution3Channel"/>
	</scatterer>
	<gatherer release-strategy-expression="^[payload gt 5] != null or size() == 3"/>
</scatter-gather>

这是一个复合端点，它结合了publish-subscribe逻辑与aggregation功能。
当然，它可以以前使用现有的集成流程实现publish-subscribe-channel ，
要么recipient-list-router ，以及aggregator
组件，但是此新功能可以更清晰地实现诸如best quote 。

Redis队列网关

一对request-reply （入站和出站）网关组件基于Redis List已添加到Redis模块：

<int-redis:queue-outbound-gateway request-channel="sendChannel" queue="foo"/>

<int-redis:queue-inbound-gateway request-channel="requestChannel" queue="foo"/>

反应堆的PersistentQueue

的QueueChannel已更改为允许注入任何Queue实施。这样做是为了允许
在Reactor项目中使用Chronicle-Queue实现：

@Bean QueueChannel queueChannel() {
   return new QueueChannel(new PersistentQueueSpec<Message<?>>()
                           		.codec(new JavaSerializationCodec<>())
                           		.basePath("/usr/queuePath")
                           		.get());
}

跳过投票

使用轮询端点时，有时有必要“跳过”轮询，这可能是因为某些下游情况可能
导致失败，或者说任务执行器池没有可用线程。此版本增加了PollSkipAdvice那
可以通过基于用户提供的代码的跳过逻辑插入到轮询者的建议链中。

笔记

  1. Spring Integration 4.1需要Spring Framework 4.1
  2. 虽然现在需要JDK8来构建 Spring Integration，但该框架在运行时仍与Java 6兼容。
  3. 我们希望在本周晚些时候宣布Spring Integration Java DSL候选版本的可用性。
