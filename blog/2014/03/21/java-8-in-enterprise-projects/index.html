企业项目中的Java 8

新闻与活动
于尔根·霍勒（Juergen Hoeller）
2014年3月21日

随着Java 8 GA版本的发布终于在本周到来，让我们所有人欢喜和庆祝-然后醒来，想知道如何将其引入我们的日常工作环境中……

这对于原型制作来说很容易做到：下载JDK 8，IntelliJ IDEA或Eclipse最近完成的Java 8支持，然后就可以使用了。最具挑战性的部分是后面的步骤：您需要弄清楚如何将应用程序部署到支持Java 8的环境。

当然，实际上，这种考虑可能早就出现了：在您愿意或被允许尝试Java 8之前，首先需要制定启用Java 8的部署计划。而且，在很多情况下，您甚至都无需打扰，因为您已经预先知道在此时无法将Java 8引入生产系统。

不可能？真？让我们重新考虑特定的部署环境…

Java 8和已建立的Java EE 6应用程序服务器

对于Java EE服务器，通常会有捆绑的JDK版本形式的升级showtopper。使用WebSphere，即使在WAS 8.5上，默认的JDK仍然是Java 6，并且可以选择使用Java 7。使用WebLogic，最新版本已经更加强大地基于Java 7。但是，这些服务器都不会让您自由选择要使用的JDK代。

特定应用JDK版本通常与应用服务器堆栈进行深度集成，因此不建议仅在其他JDK安装上运行它。即使针对JDK 8进行了启动烟雾测试，当前的应用服务器也无法幸免。

借助GlassFish和JBoss AS等灵活的产品，您可能可以将它们指向JDK 8安装并启动服务器并运行。但是，通常仅在使用“ -target 1.7”时才适用；一旦开始使用Java 8语言功能，服务器的某些部分和某些集成的服务提供商就会选择“ -target 1.8” ...

Java 8和最近发布的Java EE 7应用程序服务器

WildFly 8是JBoss AS 7的继任者，实际上是第一款具有基本JDK 8兼容性的EE服务器。它的开源版本已经普遍可用。但是，尚没有基于WildFly 8的JBoss EAP版本形式的Red Hat官方支持。

至于GlassFish 4，Oracle去年停止了商业支持，此后没有发布维护版本，也没有正式的支持合同。不过，今年晚些时候的GlassFish 4.0.1版本中可能会提供基本的JDK 8支持。

更新（9月30日）：半年后，GlassFish 4.1现在可以使用JDK 8支持。此外，Oracle还通过JDK 8对最近发布的WebLogic 12.1.3进行了认证-支持一些EE 7规范，但尚不支持完整的EE 7平台服务器。

Java 8和Tomcat / Jetty

诸如Tomcat和Jetty之类的精益Web应用程序服务器在这里具有明显的优势，因为它们不依赖于任何JDK版本的细节。Tomcat 8和Jetty 9已经针对OpenJDK 8进行了半年的测试，并且通常声明与Java 8兼容。甚至Tomcat 7也加入了这个俱乐部，并且还接受了升级的Eclipse编译器等。

因此，Tomcat和Jetty都将自己展示为可与JDK 8一起使用的即时生产质量选项。从Pivotal的角度来看，我们正在尽最大努力通过参与Apache Tomcat本身以及通过tc Server产品来在JDK 8上支持Tomcat。

当然，您通常不会使用普通的Tomcat或Jetty，而是向部署单元中添加特定的框架和选择的库。显然，每个库都必须与Java 8兼容。他们中的许多人自然是，但是请仔细检查您的持久性提供程序和绑定框架，因为它们可能尚不支持Java 8字节码。

Java 8和Spring

从一开始就已经以Java 8兼容的方式设计了Spring Framework 4世代。实际上，我们甚至在第一个OpenJDK 8候选发布版本发布之前就发布了具有全面Java 8开发支持的Spring Framework 4.0 GA。

从我们的角度来看，Java 8上的Spring Framework 4在许多环境中都是很好的组合，不仅可以与Tomcat，Jetty或WildFly结合使用，而且还可以与最近的嵌入式引擎（例如Reactor或Undertow）以及其他类型的自定义应用程序架构结合使用。

随着JDK 8进入GA，现在是在Java 8上尝试这种基于Spring的安排的最佳时机！您会惊讶于您已经走了多远。结合IntelliJ IDEA或Eclipse中与Java 8兼容的最新工具，您可能会立即找到所需的所有内容。

因此，请不要等到2016年尝试Java 8。
