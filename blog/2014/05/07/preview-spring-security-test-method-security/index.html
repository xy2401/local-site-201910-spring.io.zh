预览Spring安全性测试：方法安全性

工程
罗伯·温奇
2014年5月7日

2015年3月31日更新
该博客已过时，不再维护。请参考参考文档的“ 测试”部分以获取更新的文档。

周一， 我宣布了Spring Security 4.0.0的发布。M1。这是介绍Spring Security Testing支持的三部分博客系列文章中的第一篇。该系列的轮廓如下所示：

  • 在这篇文章中，我们将讨论spring-security-test如何使基于方法的安全性测试更容易
  • 我们的第二篇文章将演示如何在Spring MVC Test中使用spring-security-test
  • 我们的最后一篇文章将演示如何在Spring Test MVC HtmlUnit中使用spring-security-test

测试方法安全

基于测试方法的安全性一直非常简单。但是，这并不意味着它会更好。让我们探索一个非常简单的示例，看看如何使用Spring Security Test支持使基于方法的安全测试更加容易。

我们首先介绍一个MessageService要求对用户进行身份验证才能访问它。

源代码
您可以在github上找到此博客系列的完整源代码。

public class HelloMessageService implements MessageService {

	@PreAuthorize("authenticated")
	public String getMessage() {
		Authentication authentication = SecurityContextHolder.getContext()
                                                             .getAuthentication();
		return "Hello " + authentication;
	}
}

的结果getMessage是一个字符串，向当前的Spring Security说“ Hello” Authentication 。输出示例如下所示。

Hello org.springframew[email protected]ca25360: Principal: [email protected]: Username: user; Password: [PROTECTED]; Enabled: true; AccountNonExpired: true; credentialsNonExpired: true; AccountNonLocked: true; Granted Authorities: ROLE_USER; Credentials: [PROTECTED]; Authenticated: true; Details: null; Granted Authorities: ROLE_USER

安全测试设置

在使用Spring Security Test支持之前，我们必须执行一些设置。可以在下面看到一个示例：

@RunWith(SpringJUnit4ClassRunner.class)
@ContextConfiguration
@TestExecutionListeners(listeners={ServletTestExecutionListener.class,
		DependencyInjectionTestExecutionListener.class,
		DirtiesContextTestExecutionListener.class,
		TransactionalTestExecutionListener.class,
		WithSecurityContextTestExcecutionListener.class})
public class WithMockUserTests {

这是如何设置Spring Security Test的基本示例。亮点可以很好地看到：

  • @RunWith指示spring-test模块应该创建ApplicationContext。这与使用现有的Spring Test支持没什么不同。有关更多信息，请参考Spring Reference。
  • @ContextConfiguration指示弹簧测试配置用于创建ApplicationContext 。由于未指定任何配置，因此将尝试使用默认配置位置。这与使用现有的Spring Test支持没有什么不同。有关更多信息，请参考Spring Reference。
  • @TestExecutionListeners指示spring-test模块除了默认的侦听器之外，还使用WithSecurityContextTestExcecutionListener这将确保我们的测试是针对正确的用户进行的。它通过填充SecurityContextHolder在运行我们的测试之前。测试完成后，它将清除SecurityContextHolder 。

注意

据了解， @TestExecutionListeners相当冗长，有许多现有的JIRA有望在将来对此进行改进。有关最新信息，请参见SEC-2585 。


记住我们添加了@PreAuthorize我们的注释HelloMessageService因此，它需要经过身份验证的用户才能调用它。如果我们运行以下测试，我们期望以下测试将通过：

@Test(expected = AuthenticationCredentialsNotFoundException.class)
public void getMessageUnauthenticated() {
    messageService.getMessage();
}

@WithMockUser

问题是我们如何最轻松地以特定用户身份运行测试。由于我们正在使用WithSecurityContextTestExcecutionListener ，将以用户名“ user”，密码“ password”和角色“ ROLE_USER”的用户身份运行以下测试。

@Test
@WithMockUser
public void getMessageWithMockUser() {
  String message = messageService.getMessage();
  ...
}

具体来说，以下是正确的：

  • 用户名“ user”的用户不必存在，因为我们在模拟用户
  • 的Authentication填充在SecurityContext是类型UsernamePasswordAuthenticationToken
  • 校长Authentication是一个User
  • 的User将具有用户名“ user”，密码“ password”和GrantedAuthority是一个名为“ ROLE_USER”的单个。

我们的示例很好，因为它提供了很多默认值。如果我们想使用其他用户名运行测试该怎么办？以下测试将使用用户名“ customUser”运行。同样，用户不需要实际存在。

@Test
@WithMockUser("customUsername")
public void getMessageWithMockUserCustomUsername() {
	String message = messageService.getMessage();
  ...
}

我们还可以轻松地自定义角色。例如，将使用用户名“ admin”以及角色“ ROLE_USER”和“ ROLE_ADMIN”调用此测试。

@Test
@WithMockUser(username="admin",roles={"USER","ADMIN"})
public void getMessageWithMockUserCustomUser() {
	String message = messageService.getMessage();
	...
}

当然，在每种测试方法上放置注释可能会有些乏味。相反，我们可以将注释放置在类级别，并且每个测试都将使用指定的用户。例如，以下将对用户名为“ admin”，密码为“ password”，角色为“ ROLE_USER”和“ ROLE_ADMIN”的用户运行每个测试。

@RunWith(SpringJUnit4ClassRunner.class)
@ContextConfiguration
@TestExecutionListeners(listeners={ServletTestExecutionListener.class,
    DependencyInjectionTestExecutionListener.class,
    DirtiesContextTestExecutionListener.class,
    TransactionalTestExecutionListener.class,
    WithSecurityContextTestExcecutionListener.class})
@WithMockUser(username="admin",roles={"USER","ADMIN"})
public class WithMockUserTests {

@WithUserDetails

而@WithMockUser这是一种非常方便的入门方法，可能无法在所有情况下都起作用。例如，应用程序通常期望Authentication主体是特定类型。这样做是为了使应用程序可以将委托人称为自定义类型，并减少Spring Security上的耦合。

自定义主体通常是自定义返回的时间UserDetailsService返回一个实现两个目标的对象UserDetails和自定义类型。在这种情况下，使用自定义创建测试用户非常有用UserDetailsService 。那正是@WithUserDetails做。

假设我们有一个UserDetailsService作为bean公开，下面的测试将使用Authentication类型的UsernamePasswordAuthenticationToken以及从UserDetailsService用户名“ user”。

@Test
@WithUserDetails
public void getMessageWithUserDetails() {
	String message = messageService.getMessage();
	...
}

我们还可以自定义用于从我们的用户中查找用户的用户名UserDetailsService 。例如，此测试将使用从UserDetailsService用户名“ customUsername”。

@Test
@WithUserDetails("customUsername")
public void getMessageWithUserDetailsCustomUsername() {
	String message = messageService.getMessage();
	...
}

喜欢@WithMockUser我们还可以将注释放在类级别，以便每个测试使用同一用户。但是不像@WithMockUser ， @WithUserDetails要求用户存在。

@WithSecurityContext

我们已经看到@WithMockUser如果我们不使用自定义，则是一个很好的选择Authentication主要。接下来我们发现@WithUserDetails将允许我们使用自定义UserDetailsService创建我们的Authentication主体，但要求用户存在。现在，我们将看到一个具有最大灵活性的选项。

我们可以使用@WithSecurityContext创建任何SecurityContext我们想要。例如，我们可以创建一个名为@WithMockCustomUser如下所示：

@WithSecurityContext(factory = WithMockCustomUserSecurityContextFactory.class)
public @interface WithMockCustomUser {

	String username() default "rob";

	String name() default "Rob Winch";
}

你可以看到@WithMockCustomUser带有注释@WithSecurityContext注解。这就是向Spring Security Test支持发出信号的信号，我们打算创建一个SecurityContext进行测试。的@WithSecurityContext批注要求我们指定一个SecurityContextFactory这将创建一个新的SecurityContext鉴于我们@WithMockCustomUser注解。你可以找到我们的WithMockCustomUserSecurityContextFactory实施如下：

public class WithMockCustomUserSecurityContextFactory implements WithSecurityContextFactory<WithMockCustomUser> {
	@Override
	public SecurityContext createSecurityContext(WithMockCustomUser customUser) {
		SecurityContext context = SecurityContextHolder.createEmptyContext();

		CustomUserDetails principal =
			new CustomUserDetails(customUser.name(), customUser.username());
		Authentication auth =
			new UsernamePasswordAuthenticationToken(principal, "password", principal.getAuthorities());
		context.setAuthentication(auth);
		return context;
	}
}

现在，我们可以使用我们的新注释和Spring Security的注释来注释测试类或测试方法。 WithSecurityContextTestExcecutionListener将确保我们的SecurityContext被适当地填充。

当创建自己的WithSecurityContextFactory实现，很高兴知道可以使用标准的Spring注释对其进行注释。例如， WithUserDetailsSecurityContextFactory使用@Autowired注释以获取UserDetailsService ：

final class WithUserDetailsSecurityContextFactory implements WithSecurityContextFactory<WithUserDetails> {

    private UserDetailsService userDetailsService;

    @Autowired
    public WithUserDetailsSecurityContextFactory(UserDetailsService userDetailsService) {
        this.userDetailsService = userDetailsService;
    }

    public SecurityContext createSecurityContext(WithUserDetails withUser) {
        String username = withUser.value();
        Assert.hasLength(username, "value() must be non empty String");
        UserDetails principal = userDetailsService.loadUserByUsername(username);
        Authentication authentication = new UsernamePasswordAuthenticationToken(principal, principal.getPassword(), principal.getAuthorities());
        SecurityContext context = SecurityContextHolder.createEmptyContext();
        context.setAuthentication(authentication);
        return context;
    }
}
