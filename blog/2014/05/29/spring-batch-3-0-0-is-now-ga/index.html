Spring Batch 3.0.0现在已发布

发布
迈克尔·米内拉
2014年5月29日

今天，我们很高兴地宣布发布Spring Batch 3.0。此版本的Spring Batch提供了对JSR-352的支持-我们致力于为我们现有的Spring Batch社区提供该支持以及许多新功能。


Spring Batch 3.0中的功能


此版本的主要功能包括：



  • JSR-352支持

  • 促进将Spring Batch集成到Spring Batch中

  • 彻底检查依赖项

  • 支持SQLite

  • 工作范围

  • 删除“原型”以支持Spring Boot

  • 从Maven切换到Gradle以构建Spring Batch


JSR-352支持


JSR-352提供了JEE7中最重要的新功能之一，将批处理的标准化引入了Java堆栈。Spring团队不仅与专家组的两名团队成员一起为规范提供了大量投入，而且Spring Batch成为了大多数编程模型的灵感来源。Spring Batch 3.0通过提供最经生产测试的代码库来实现此规范，继承了Spring对JSR-352的承诺。


重要的是要注意JSR-352和Spring Batch之间的区别。JSR-352提供了三件事：



  • 用于构建批处理组件的API

  • 基于XML的DSL，用于配置批处理组件

  • 批处理组件如何交互的合同


JSR-352不提供批处理组件的任何实现 。Spring Batch提供了17个阅读器，16个编写器以及无数其他在企业生产环境中经过多年测试的实用程序和扩展。Spring Batch提供了超越单个JVM的可伸缩性选项。最终，Spring Batch通过Spring for Apache Hadoop项目提供了大数据支持，并作为Spring XD的基石。


要了解有关Spring Batch JSR-352实施的更多信息，请在此处访问我们的参考文档： http://docs.spring.io/spring-batch/trunk/reference/html/jsr-352.html


将Spring Batch集成到Spring Batch


从外部看，确定何时使用Spring Batch和何时使用Spring Integration可能会造成混淆。实话实说，并非总是有一个确定的答案。在许多情况下，两种技术的结合是提供可靠，可扩展的解决方案的最佳选择。这就是开发Spring Batch Integration的原因。这个模块是Spring Batch扩展的集合，允许开发人员使用Spring Integration扩展Spring Batch的功能。该模块不提供自定义代码和最少配置，即可提供多JVM可伸缩性和消息驱动的作业启动之类的功能。


Spring Batch Integration中提供的功能将批处理扩展到JSR-352提供的功能之外，以提供强大的企业级解决方案。您可以在参考文档中阅读有关Spring Batch Integration及其组件的更多信息： http://docs.spring.io/spring-batch/trunk/reference/html/springBatchIntegration.html 。


彻底检查依赖项


作为整个Spring产品组合不断努力的一部分，Spring Batch第三方依赖项已更新。这使Spring Batch的用户可以使用他们信任的Spring Batch中最新和最好的库。


支持SQLite


大多数集成测试方案都使用HSQLDB作为数据源，因为它能够轻松创建内存数据库。但是，有时候简单的单个文件选项会更好。因此，我们添加了SQLite作为Spring Batch存储库的受支持数据库。


工作范围


Spring Batch提供了两个自定义bean作用域。第一个步骤作用域已经存在多年了，它是一种延迟bean初始化直到执行该bean使用的步骤的方法。使用Spring Batch 3.0，我们添加了一个工作范围。这将使您能够延迟初始化作业级别组件（例如，JobExecutionListeners），并为这些组件提供诸如作业参数注入之类的功能。您可以在以下参考手册中阅读有关工作范围的更多信息： http://docs.spring.io/spring-batch/trunk/reference/html/configureStep.html#job-scope 。

删除“原型”以支持Spring Boot


传统上，Spring Batch提供了一个示例Maven项目，作为引导Spring Batch项目的一种方式。使用Spring Boot，我们已经删除了那些项目，转而使用Spring Boot提供的动态引导功能。我们认为，Boot提供了比旧“原型”所提供的更好的选择。


从Maven移至Gradle


最终，此版本是Spring Batch从Maven迁移到Gradle进行内部构建系统的第一个版本。此更改对使用jar的开发人员的影响为零。它们仍将通过Maven Central提供给Maven用户。


期待


在此版本中，我们期待着计划和开发我们的3.1版本。3.1发行版将有两个主要主题：简化和支持Spring XD。


为简化起见，我们希望将Spring Batch的可用性提高到可以减少“繁重”过程的程度。通过集中精力，它可以使开发人员“扩展”解决方案。很多时候，事情开始时都是简单的脚本，后来演变成企业需要依靠的关键任务功能流程。我们希望在脚本阶段考虑Spring Batch，以便当该功能需要企业级解决方案时就已经存在。


除了核心的Spring Batch功能之外，Spring Batch还作为Spring XD的核心技术。我们将继续通过Spring XD和Spring for Apache Hadoop扩展Spring Batch的功能，以支持大数据解决方案。


结论


3.0.0代表Spring Batch的下一个主要里程碑的完成。它为我们的社区带来了JSR-352提供的标准，并为高级用户提供了详尽的其他功能集合。我们期待您在论坛，社交媒体以及SpringOne2GX上的亲自反馈！
