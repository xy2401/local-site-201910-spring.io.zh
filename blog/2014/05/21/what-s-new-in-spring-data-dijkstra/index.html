Spring Data Dijkstra有什么新功能？

工程
奥利弗·德罗博姆（Oliver Drotbohm）
2014年5月21日

我们刚刚宣布了名为Dijkstra的Spring Data发布系列的GA发布版本。我想借此机会向您介绍我们在此版本中添加的一些功能。

5个新模块加入火车

该发行版的第一个主要功能是在发行版中增加了5个模块。它们中的大多数已经存在了一段时间，但今后我们将与其他模块同步发布它们。新添加的模块是Spring Data Elasticsearch ， Cassandra ， Couchbase ， Gemfire和Redis 。

春季数据共享区

发布火车的许多改进通常最终出现在Commons模块中，因此各个商店模块实际上都可以从新添加的功能中受益。这是Dijkstra最重要的：

支持包装类型作为返回值

对于声明为返回域类型的单个实例的Spring Data存储库方法，如果无法获得结果，则存储库将管理返回null 。但是，如果我们从头开始创建API而不是直接返回实例，我们可能宁愿使用Optional来确保客户端不会意外忘记null检查。

目前，许多Java库都提供了Optional类型。Google Guava拥有它，更重要的是，JDK 8也提供了它。因此，通过Dijkstra发布火车，我们提供了将这些类型用作返回类型的包装的可能性，并让Spring Data存储库基础结构自动为您包装null 。

interface CustomerRepository extends Repository {

  Optional findOne(Long id);

  Optional findByEmailAddress(EmailAddress emailAddress);
}

您在这里看到的第一个方法是CrudRepository.findOne(…)的变体。注意，存储库接口不会扩展CrudRepository因为这将导致编译错误，因为我们无法重新声明findOne(…)方法并更改返回类型。因此，我们建议您简单地构建自己的基础存储库接口（如参考文档中所述，以防您想要更改findOne(…)的行为。

第二种方法是一种简单的查询方法，该方法使用查询派生使Spring Data基础结构从方法名派生查询。在此，我们还将检测Optional作为包装类型，执行查询并将结果自动包装到Optional实例中。

异步存储库方法调用

我们以一种相关的方式开始向存储库方法添加异步执行功能。查询方法现在可以返回Future如果使用@Asyncannotation，将导致它异步执行。

interface CustomerRepository extends Repository {

  @Async
  @Query(" … long running query declaration … ")
  Future> findByLongRunningQuery(String lastname);
}

然后，执行基于Spring对异步方法调用的支持。我们还正在研究更高级的异步执行模型，例如将在以后的Spring Data版本中添加的promise（可能基于Project Reactor ）。

地理空间类型

MongoDB模块（以及其他模块）始终支持Mongo的地理空间操作-在MongoTemplate和存储库抽象中均如此。在当前版本中，我们将核心值类型（例如Shape ， Point ， Box ， Distance等）移到了Spring Data Commons模块中。这将使您无所不在地引用地理类型并与支持地理空间功能的所有Spring Data模块进行交互，而无需将这些类型相互映射。查看JavaDoc了解详细信息。

切片

Spring Data一直在存储库编程模型中支持分页。它使您可以逐页访问数据以遍历大型数据集。除了普通页面内容之外， Page接口还公开API，以查找可用元素和页面的总数。

计算该数字可能会占用大量资源（通常通常必须执行附加查询），并且通常唯一感兴趣的页面元数据是页面是否具有下一个页面，以便客户端可以继续获取更多数据。例如，您可以在Facebook时间线中看到这种模式。

Dijkstra版本现在引入了一个精简版的Page称为“ Slice ，它使您可以了解当前切片是否具有以下切片：

interface BlogPostRepository extends Repository {

  Slice findByAuthorOrderByDateDesc(Author author, Pageable pageable);
}

现在，您可以使用Pageable来表达您想要获取的页码和页面大小，Spring Data基础结构将读取比请求更多的项目，并使用该项目的存在或不存在作为下一个可用切片的指示符。

派生删除查询

查询方法派生功能现在支持deleteBy…和removeBy…方法前缀，以派生根据给定条件删除托管域类型的查询。

interface BlogPostRepository extends Repository {

  int deleteByDateBefore(Date date);
}

这已在Dijkstra的JPA，MongoDB和Solr模块中实现，并将在以后添加到其他商店模块中。

JPA 2.1支持

为Dijkstra开发JPA模块期间的一个核心主题是对JavaEE 7 / JPA 2.1功能的支持。我们使用此发行版解决的核心领域是对查询执行以及存储过程执行中的实体图的支持。

实体图

假设我们具有以下域类型定义：

@Entity
@NamedEntityGraph(name = "summary", attributeNodes = { 
  @NamedAttributeNode("firstname"),
  @NamedAttributeNode("lastname")})
class Customer {
  // properties ommitted
}

现在，我们可以通过@EntityGraphannotation引用命名的实体图，以指示我们希望将此图应用于正在执行的查询。

interface CustomerRepository extends Repository {

  @EntityGraph("summary")
  Optional findByEmailAddress(EmailAddress emailAddress);
}

这将仅导致急切地加载firstname和lastname属性，而所有其他准备在访问时延迟加载的属性。

储存程序

JPA 2.1添加了通过EntityManager API执行存储过程的功能。与上面的示例类似，可以以域类型声明存储过程的元数据。假设您要触发一个存储过程，该存储过程为客户随机创建密码：

@Entity
@NamedStoredProcedureQuery(name = "Customer.generateNewPassword", 
  procedureName = "generateNewPassword", parameters = {
    @StoredProcedureParameter(
      mode = ParameterMode.IN, name = "username", type = String.class),
    @StoredProcedureParameter(
      mode = ParameterMode.OUT, name = "password", type = String.class)})
class Customer {
  // properties ommitted
}

可以使用存储库查询方法执行存储过程，如下所示：

interface CustomerRepository extends Repository {

  @Procedure
  String generateNewPassword(@Param("username") String username);
}

默认情况下，我们将使用通过众所周知的DomainType.methodName模式找到的存储过程元数据，并将其与方法声明进行匹配。对于非常简单的过程映射（如此处所示），您甚至可以省略元数据声明，因为所有元数据都可以从方法名称派生。在参考文档中找到有关存储过程支持的更多信息。

Redis的交易支持

最新版本的Redis模块添加了功能，以累积可以批量执行的一组操作。为此， RedisTemplate现在可以通过将enableTransactionSupport属性配置为true （它默认为false ）来与Spring的事务同步集成。

启用此功能将导致RedisConnection绑定到当前Thread并发出MULTI命令，从而允许基础Redis驱动程序潜在地执行命令队列。如果事务没有错误完成，则发出EXEC命令；如果事务失败，则使用DISCARD命令丢弃累积的命令。

一旦启用，连接将绑定到当前线程，确保将每个写入操作通过管道传递到同一连接，并排队等待周围的事务完成。读取操作（例如KEYS命令）仍将通过使用全新的无线程绑定连接立即执行。

@Bean
public StringRedisTemplate redisTemplate() {

	StringRedisTemplate template = 
	  new StringRedisTemplate(redisConnectionFactory());
	// Enable transaction synchronization support
	template.setEnableTransactionSupport(true);

	return template;
}

这样配置的RedisTemplate可以用于以下语义：

// Executed on thread bound connection
template.opsForValue().set("foo", "bar");

// Read operation executed on separate connection
template.keys("*");

// Returns null as values set within transaction are not visible
// prior to transaction flush
template.opsForValue().get("foo");

Spring Data Solr中的复杂查询支持

使用Spring Data Solr的条件API创建更复杂的查询已经有一段时间了。因此，我们决定重写部分实现，以使API与之前的版本兼容。

基本上，我们从相当平坦的表示形式转向了树状模型，保留了我们习惯的所有流畅的API样式。

Solr查询q=name:solr OR (type:spring AND category:data)现在可以表示为：

new SimpleQuery(
  where("name").is("solr").or(
    where("type").is("spring").and("category").is("data")));

Spring Data REST中的投影

Spring Data REST公开的REST资源的一个非常普遍的要求是能够制作自定义表示。这意味着，用户希望减少响应中呈现的属性的数量或内联关联的实体以节省服务器往返次数。借助Spring Data REST 2.1，我们现在提供了在服务器端定义自定义投影的可能性。为此，您声明一个接口以完全包含要公开的属性：

@Projection(name = "summary", types = Order.class)
interface OrderSummary {
	
  LocalDate getOrderedDate();

  CustomerSummary getCustomer();

  @Value("#{@shop.calculateTotal(target)}")
  Money getTotal();
}

该接口可以放置在Order所在的同一包（或子包）中，并由Spring Data REST自动检测。它将使暴露单个订单或订单集合的所有资源在URI模板中携带附加参数，以指示投射功能：

{ _links : {
    orders : { href : "…/orders{?projection}", templated : true }
  }
}

如果客户现在扩大与模板summary了projection ，我们将创建在服务器端的代理，将被移交给杰克逊编组。每个获取器都将转发到实际目标类上的属性查找（在本例中为Order ）。

在上面的示例中， getCustomer()引用了一个相关实体，该实体仅在非投影场景中才作为链接公开。通过使用投影，我们检测到该方法的返回类型不是Customer 。这将依次导致创建投影代理，以便您完全控制公开的属性。投影界面当然可以携带Jackson注释，以进一步自定义呈现的表示。

对于高级用例，您甚至可以为投影方法配备@Value以将SpEL表达式的结果返回给编组器。在这里的示例中，我们在Spring bean上调用一个名为shop的方法，并将代理目标实例交给它来计算订单总额，其中可以考虑回扣，税金等。

摘要
