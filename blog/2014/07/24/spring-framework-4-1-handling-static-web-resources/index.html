Spring Framework 4.1-处理静态Web资源

工程
布赖恩·克洛泽尔
2014年7月24日

本周， Juergen宣布了Spring Framework 4.1发行候选版 。现在是时候测试这些新功能，看看它们如何使您的应用程序更好！

这些新功能之一是灵活的静态Web资源解析和转换。Spring Framework 已经允许您使用以下方法提供静态资源ResourceHttpRequestHandlers 。此功能为您提供了更多功能和新的可能性。

ResourceResolvers和ResourceTransformers

ResourceResolvers和ResourceTransformers是此新功能的核心。

ResourceResolvers可以根据资源的URL路径解析资源。给定其内部资源路径，他们还可以解析供客户端使用的面向外部的公共 URL路径。 ResourceTransformers可以修改已解析资源的内容。

这是一个图表，说明使用ResourceHttpRequestHandlers ：

  Request for Resource
      |
      | HTTP request
      v
  Resolvers chain: FirstResolver, SecondResolver, ThirdResolver
  (each resolver can return the resource or delegate to the next one)
      |
      | Resolved Resource
      v
  Transformers chain: FirstTransformer, SecondTransformer
  (each transformer can transform the resource or just pass it along without modification)
      |
      | Transformed Resource
      v
  HTTP Response with Resource content

这是另一个展示如何ResourceResolvers可以更新资源链接以供HTTP客户端使用：


 Resource link in a template source file
 |
 | Resource path (like “/css/main.css”)
 v
 Resolvers chain: FirstResolver, SecondResolver, ThirdResolver
 (each resolver can modify the resource path or delegate to the next one)
 |
 | Updated resource path (like “/css/main-0e37f12.css”)
 v
 Resource link in a rendered template

现在，让我们看一下ResourceResolvers实现必须提供：

解析器名称 目标
PathResourceResolver 在与请求路径匹配的已配置位置（类路径，文件系统等）下查找资源
CachingResourceResolver 从缓存实例解析资源或委托链中的下一个解析器
GzipResourceResolver HTTP客户端支持gzip压缩时，查找扩展名为“ .gz”的资源的变体
VersionResourceResolver 解析包含版本字符串（即有关所请求资源的版本信息）的请求路径。通过在资源更新时更改其URL，此解析器对于设置HTTP缓存策略很有用。

现在， ResourceTransformers ：

变压器名称 目标
CssLinkResourceTransformer 修改CSS文件中的链接以匹配应向客户端公开的公共URL路径
CachingResourceTransformer 将转换结果缓存在Cache中或委托给链中的下一个Transformer
AppCacheManifestTransfomer 帮助处理HTML5脱机应用程序的HTML5 AppCache清单中的资源

这些新添加的主要目标ResourceHttpRequestHandlers是为了轻松优化和使用优化的静态资源以实现前端性能 。

还有另一个资产管道？

许多图书馆和框架都通过完整的集成资产管道解决了这些问题，这些管道通常会为要使用的编程语言，技术和项目结构提供强有力的，自以为是的解决方案。这些资产管道在创建可部署应用程序时和/或在应用程序运行时会进行资源优化。

在Spring Framework 4.1中，我们选择了一条路径，该路径依赖于在构建时使用最佳工具为应用程序优化资源，并在运行时利用解析器和Transformer。对于JavaScript应用程序，我们希望利用JavaScript开发人员使用的相同工具链（例如grunt和gulp）在构建时优化资源。关于Dart和TypeScript，同样的事情-本机工具始终提供最佳体验。

这些生态系统很丰富（实际上比Java中可用的选项要丰富得多）并且还在不断发展。我们认为，依靠这些生态系统和框架中的灵活解决方案是这里的最佳方法。

因此，您的应用程序应该找到适当的平衡和杠杆作用：
*在构建时使用本机工具为客户端应用程序转译，最小化，级联……任务
*框架随附的解析器和转换器（也可以创建自己的解析器和转换器！）

查看即将到来的标准（例如HTTP / 2和ECMAScript 6） ，这更有意义-定义更改将在未来几年发生。

资源版本控制

在将Web应用程序推向生产环境时，静态Web资源版本控制是一个主要问题，而在服务器端则非常重要。Spring Framework 4.1旨在通过各种策略提供一流的支持，这些策略包括基于内容的哈希（例如在Git中，也称为指纹）以及适用于整个文件集的版本（例如，与JavaScript模块加载程序配合使用时所需要的版本）。

所有这一切的基础都是“缓存破坏”的概念，其中通过积极的HTTP缓存指令（例如，未来1年）为资源提供服务，并在必要时依靠URL中与版本相关的更改来“破坏”缓存。这可能是基于内容的哈希版本，它会在文件内容更改时更改，也可能是通过其他方式（例如，简单属性，git commit sha等）确定的版本。

源代码布局

另一个非常重要的问题是您的源代码位于何处以及应用程序的组织方式-作为Java开发人员，我们习惯于在src/main/webapp 。但这真的是最好的位置吗？

如今，大多数Web应用程序都由运行在浏览器中的客户端应用程序和服务器应用程序组成，二者均通过HTTP或Websocket进行通信。每个对象都可以具有自己的依赖项，测试，构建工具等。那么，为什么我们不能将它们解耦并在我们的代码库中反映出关注点的分离呢？

将Web应用程序分解为模块（客户端模块和服务器模块）可以极大地改善您的开发体验，并提供应用程序所需的自由度。

我们在Sagan项目中使用了相同的布局，而我在先前的截屏视频Project Sagan中详细讨论了其背后的原理：客户端架构 。

这是项目布局的示例：

spring-app/
|- build.gradle
|- client/
|  |- src/
|  |  |- css/
|  |  |- js/
|  |     |- main.js
|  |- test/
|  |- build.gradle
|  |- gulpfile.js
|- server/
|  |- src/main/java/
|  |– build.gradle

解析器/转换器和构建工具都可以在资源处理方面提供类似的功能。那么我们应该使用哪一个呢？

Spring Resource Handling展示应用程序

在Spring Resource Handling展示应用程序中 ，我们展示了几个关键功能：

  • 在HTML响应，CSS文件和HTML5 appcache清单中缓存清除静态资源
  • 如前所述的新项目布局
  • 模板引擎集成，例如Groovy标记模板和Handlebars
  • 使用LESS作为CSS替代方案，在开发过程中使用客户端预处理器，并在生产中使用构建处理器
  • 使用Gradle和gulp的完整构建工具链 ；将来的示例可以使用grunt，maven等演示相同的功能

请注意，这种新的项目布局具有两个关键优势：
1。更好的开发人员体验，因为未优化的资源在开发时直接从磁盘提供
2。生产中的最佳性能，因为静态资源是通过构建进行优化并打包在webJAR中的-因此最终将在生产中的类路径中为它们提供服务
