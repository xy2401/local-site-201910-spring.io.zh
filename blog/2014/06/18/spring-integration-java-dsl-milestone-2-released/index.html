Spring Integration Java DSL Milestone 2发布

发布
阿尔特姆·比兰（Artem Bilan）
2014年6月18日

我们很高兴地宣布Java DSL扩展第二里程碑的发布
对于Spring Integration ！

的org.springframework.integration:spring-integration-java-dsl:1.0.0.M2工件可以从
Spring IO里程碑存储库 。

里程碑2包括一些错误修复，一些新功能和进一步的改进。

感谢所有尝试里程碑1的人，他们提供了反馈，提出了问题并分享了他们的想法。

这是自里程碑1以来的主要变化的摘要：

Lambda处理程序

您可能已经注意到，Java 8 Lambdas的使用是使此DSL方便易读的强大工具。我们收到了一个社区要求
是为了允许宣布Lambda .handle() EIP方法，而不必声明POJO并将其用作方法调用。但是，一个担心是不要丢失“运行时类型转换”。
但是你
无法获得Lambda的通用类型 。
经过一番调查，我们找到了解决方案，方法是添加一个type论点。因此，
几种新方法已添加到IntegrationFlowBuilder ：


<P> IntegrationFlowBuilder handle(GenericHandler<P> handler)

<P> IntegrationFlowBuilder handle(GenericHandler<P> handler,
		EndpointConfigurer<GenericEndpointSpec<ServiceActivatingHandler>> endpointConfigurer) 


<P> IntegrationFlowBuilder handle(Class<P> payloadType, GenericHandler<P> handler)

<P> IntegrationFlowBuilder handle(Class<P> payloadType, GenericHandler<P> handler,
		EndpointConfigurer<GenericEndpointSpec<ServiceActivatingHandler>> endpointConfigurer)

如果将方法变量与显式变量一起使用payloadType论点和handler是Lambda，
最后一个参数将包装到LambdaMessageProcessor与一个ConversionService 。和消息payload
将转换为适当的type在运行时。这样，我们可以实现更好的松耦合。
这是一个简单的示例来演示：
java @Bean public IntegrationFlow integerFlow() { return IntegrationFlows.from("input") .transform(p - > new String(p, "UTF-8")) .handle(Integer.class, (p, h) -> p * 2) .get(); }
的ConversionService防止ClassCastException: String cannot be cast to Integer 。

相同的附加类型参数已添加到带有Lambda的其他EIP方法中： .transform() ， .filter() ， .split()等等

变压器厂

方便，流利Transformers已添加factory用作内联目标对象定义.transform() EIP方法：
java @Bean public IntegrationFlow transformFlow() { return IntegrationFlows.from("input") .transform(Transformers.xpath("/root/myJson", XPathEvaluationType.STRING_RESULT)) .transform(Transformers.fromJson(MyPojo.class)) .transform(Transformers.serializer()) .get(); }
它避免了使用setter进行编码的不便，并使流定义更加简单。
注意Transformers可以用来声明目标Transformer作为@Bean ，并再次使用它们IntegrationFlow定义。不过，如果尚未将内联对象定义为Bean，则DSL解析器会注意它们的声明。

.gateway（）EIP方法

因为IntegrationFlow定义看起来类似于从Spring Integration XML中，我们介绍了
的.gateway() EIP方法，其作用与在一个 -发送消息到
requestChannel一些其他消息流，并等待其结果replyChannel ，
要么TemporaryReplyChannel默认：
``java
@豆
@DependsOn（“ gatewayRequestFlow”）
公共IntegrationFlow gatewayFlow（）{
返回IntegrationFlows.from（“ gatewayInput”）
.gateway（“ gatewayRequest”，g-> g.errorChannel（“ gatewayError”）。replyTimeout（10L））
。得到（）;
}

@豆
公共IntegrationFlow gatewayRequestFlow（）{
返回IntegrationFlows.from（“ gatewayRequest”）
.filter（“ foo” :: equals，f-> f.throwExceptionOnRejection（true））
。 transform（String :: toUpperCase）
。得到（）;
}
``

协议专用适配器

当然，Spring Integration的大部分价值是通过与某些外部系统的交互来提供的，其中
协议适配器提供了该功能。借助Spring Integration Java DSL，我们可以继续
使用通用bean定义（ @Bean ）用于任何最终系统适配器（例如MarshallingWebServiceInboundGateway ），
但是DSL的目的是提供一个高级API，以类似于Spring提供的方式声明组件。
集成XML配置。

由于您现在熟悉我们的Builder和Lambda能力，我们将基于这些技术。
引入了带有一组静态方法的类，以委托给一些基础方法IntegrationComponentSpec
实施。这些类可以被视为“名称空间工厂”，因为它们与XML名称空间起着相同的作用
用于特定于协议特定的Spring Integration模块的组件。当前，Spring Integration Java DSL
仅支持Amqp和Jms命名空间工厂：
``java
@豆
公共IntegrationFlow amqpFlow（）{
返回IntegrationFlows.from（Amqp.inboundGateway（this.rabbitConnectionFactory，queue（）））
.transform（“ hello” :: concat）
.transform（String.class，String :: toUpperCase）
。得到（）;
}

@豆
公共IntegrationFlow amqpOutboundFlow（）{
返回IntegrationFlows.from（“ amqpOutboundInput”）
.handle（Amqp.outboundAdapter（this.amqpTemplate）.routingKeyExpression（“ headers.routingKey”））
。得到（）;
}

@豆
公共IntegrationFlow jmsInboundFlow（）{
返回IntegrationFlows
.from（Jms.inboundAdapter（this.jmsConnectionFactory）
.configureJmsTemplate（t->
t.deliveryPersistent（true）
.jmsMessageConverter（myMessageConverter（））
.destination（“ jmsInbound”））
。 transform（String :: toUpperCase）
.channel（this.jmsOutboundInboundReplyChannel（））
。得到（）;
}

@豆
公共IntegrationFlow jmsOutboundGatewayFlow（）{
返回IntegrationFlows.from（“ jmsOutboundGatewayChannel”）
.handle（Jms.outboundGateway（this.jmsConnectionFactory）
.replyContainer（c->
c.concurrentConsumers（3）
.sessionTransacted（true））
.requestDestination（“ jmsPipelineTest”））
。得到（）;
}
``

我们在这里展示名称空间工厂作为内联适配器声明的用法，
但是它们可以从@Bean定义使IntegrationFlow方法链更具可读性。

我们在向其他名称空间工厂投入精力之前，正在征求社区的反馈。我们也会
感谢接下来我们应该支持哪些优先级的适配器/网关。

一定要具体spring-integration-[PROTOCOL].jar及其对类路径的必需依赖关系
因为spring-integration-java-dsl声明为optional以避免不必要的最终应用程序开销。

DSL解析器更改

但是，此M2版本的一般目的是解决一个关键问题，即DSL解析器的位置不正确。
现在它从IntegrationConfigurationBeanFactoryPostProcessor到IntegrationFlowBeanPostProcessor和春季整合
Java DSL不再影响应用程序上下文-只是遵循标准的Spring bean定义
生命周期。您可能需要对现有DSL应用程序进行一些更改才能使用此版本。

在大多数情况下，这仅限于channel auto-declaration ，当我们不定义显式MessageChannel豆
定义，但从集成组件中引用它。如果您注意到.gateway()上面的样本我们使用
@DependsOn注解。这是因为在声明bean的过程中，它们一一注册并初始化
在里面@Configuration类。由于我们不使用bean定义MessageChannel s，应用程序上下文
无法自动声明dependsOn对于使用通道的bean，另一方面，我们不声明
MessageChannel豆，我们只有一种选择要依靠IntegrationFlow豆。

因此，由您决定MessageChannel明确或使用bean @DependsOn在适当的时候IntegrationFlow豆的定义
对于下游IntegrationFlow声明隐式通道。
