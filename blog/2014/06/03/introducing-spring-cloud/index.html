介绍Spring Cloud

工程
拉姆尼瓦斯·拉达德（Ramnivas Laddad）
2014年6月3日

开发，部署和操作云应用程序应该与本地应用程序一样容易（如果不比本地应用程序容易的话）。那是并且应该成为任何云平台，库或工具背后的管理原则。Spring Cloud是一个开放源代码库，可轻松为云开发JVM应用程序。借助它，应用程序可以连接到服务，并在Cloud Foundry和Heroku等多个云中轻松发现有关云环境的信息。此外，您可以将其扩展到其他云平台和新服务。

在这个博客（系列文章的第一篇）中，我将介绍Spring Cloud并从应用程序开发人员的角度展示其用法。我们将开发一个简单的应用程序并将其部署到Cloud Foundry和Heroku 。在随后的博客中，我将探讨可扩展性角度。

简而言之的春天云

在云中运行应用程序的众多优势之一是易于获得各种服务。无需管理硬件，安装，操作，备份等，只需单击按钮或shell命令即可创建和绑定服务。

应用程序如何访问这些服务？例如，如果您有一个关系数据库绑定到您的应用程序，则需要创建一个DataSource基于该服务的对象。这就是Spring Cloud的帮助。它消除了访问和配置服务连接器所需的所有工作，并使您专注于使用这些服务。它还公开了应用程序实例信息（主机地址，端口，名称等）。

Spring Cloud通过Cloud Connector的概念以独立于云的方式来完成所有这些工作。虽然它为Cloud Foundry和Heroku提供了实现，但是您（或云提供商）可以通过实现接口并利用库的其余部分，将其扩展到其他云。然后，只需将包含扩展名的库添加到应用程序的类路径中即可；无需分叉并构建Spring Cloud。

Spring Cloud还认识到它不可能满足每个云上的每个服务。因此，在开箱即用地支持许多常见服务的同时，它允许您（或服务提供商）将其功能扩展到其他服务。就像扩展其他云一样，您将包含服务扩展的jar添加到应用程序的类路径中。

最后，它为Spring应用程序提供了特殊支持（在单独的模块中），包括Java和XML配置支持形式的Spring Boot应用程序，并以易于使用的形式公开了应用程序和服务属性。这是Spring Cloud中唯一依赖Spring的模块。其他框架提供者可以以类似的方式为其框架提供特定的支持。

让我们看看Spring Cloud的实际应用。

行动中的Spring Cloud

我们将从基于Spring Boot的简单应用程序（ 源代码 ）开始（传统的Spring MVC应用程序可以很好地工作，尽管会花费更多的设置代码）。该应用程序包括一个注入了表示绑定服务的bean的控制器和一个打印与绑定到应用程序的服务有关的信息的主页。

@Controller
public class HomeController {
    @Autowired(required = false) DataSource dataSource;
    @Autowired(required = false) RedisConnectionFactory redisConnectionFactory;
    @Autowired(required = false) MongoDbFactory mongoDbFactory;
    @Autowired(required = false) ConnectionFactory rabbitConnectionFactory;
    
    @Autowired ApplicationInstanceInfo instanceInfo;

    @RequestMapping("/")
    public String home(Model model) {
        Map<Class<?>, String> services = new LinkedHashMap<Class<?>, String>();
        services.put(dataSource.getClass(), toString(dataSource));
        services.put(mongoDbFactory.getClass(), toString(mongoDbFactory));
        services.put(redisConnectionFactory.getClass(), toString(redisConnectionFactory));
        services.put(rabbitConnectionFactory.getClass(), toString(rabbitConnectionFactory));
        model.addAttribute("services", services.entrySet());
        
        model.addAttribute("instanceInfo", instanceInfo);
        
        return "home";
    }
 
    // ... various toString() methods to create a string representation for each service
 
}

HomeController有四个注入的依赖项，它们表示可以绑定到应用程序的服务，而另一个依赖于ApplicationInstanceInfo 。的"/”路由将代表每个服务的字符串及其类添加到模型以及实例信息。关联的视图将呈现所有这些信息。

对于配置，我们添加CloudConfig如下：

@Configuration
@ServiceScan
@Profile("cloud")
public class CloudConfig extends AbstractCloudConfig {
    @Bean
    public ApplicationInstanceInfo applicationInfo() {
        return cloud().getApplicationInstanceInfo();
    }
}

全班延伸AbstractCloudConfig ，这是在Spring Cloud中使用Java配置方法的方法。我们设置@Profile(“cloud”)以确保仅在云环境中加载此配置。的@ServiceScan注解扫描所有绑定的服务，并为每个服务创建一个bean（然后将其自动连接到HomeController ）。如果您想知道两者之间的平行性@ComponentScan和@ServiceScan ， 你是对的。前者扫描可以实例化为bean的候选类，而后者扫描绑定的服务。我们还创建了一个与bean对应的应用程序实例信息。

将应用程序部署到Cloud Foundry

我们包括以下内容manifest.yml绑定了我们用于说明目的的所有四个服务（您将需要使用以下内容创建这些服务cf create-service命令）：

---
applications:
- name: hello-spring-cloud
  memory: 512M
  instances: 1
  host: hello-spring-cloud-${random-word}
  domain: cfapps.io
  path: target/hello-spring-cloud-0.0.1-SNAPSHOT.jar
  services:
	- postgres-service
	- amqp-service
	- mongodb-service
	- redis-service
  env:
    SPRING_PROFILES_DEFAULT: cloud

现在，我们需要做的就是构建并推送：

`sh $ mvn package $ cf push

部署到Heroku

我们可以将相同的应用程序部署到Heroku。我们需要添加几个文件（都不是特定于Spring Cloud的文件）： system.properties使Heroku使用Java 7和Procfile使它执行正确的命令以启动应用程序并启用cloud轮廓。我们将应用程序推送到Heroku如下：

$ heroku apps:create
$ heroku addons:add mongolab
$ heroku addons:add rediscloud
$ heroku addons:add cloudamqp
$ heroku config:set SPRING_CLOUD_APP_NAME=hello-spring-cloud
$ git push heroku master

在这里，我们为MongoDb，Redis和AMQP服务提供商创建附加组件（类似于Cloud Foundry服务）。Heroku自动提供Postgres服务，因此我们不需要显式添加它。Heroku应用程序的环境与Cloud Foundry不同，它不会公开应用程序名称，因此我们使用heroku config:set进行明确设置（如果没有，Spring Cloud会将其设置为 ）。Spring Cloud如何适应这两个云之间的差异还有其他一些差异。我们将在以后的博客中介绍这些内容。

这就是我们要做的。当我们访问我们的应用程序时，它显示所有服务信息的方式与在Cloud Foundry上显示的方式非常相似。

控制一些

指某东西的用途@ServiceScan轻松获取所有服务并开始使用它们。但是实际上，您通常需要对创建服务连接器进行更多控制，例如设置其池参数。在这种情况下，您可以使用Spring Cloud的Java Config或XML config支持。让我们改变CloudConfig类如下：

@Configuration
@Profile("cloud")
public class CloudConfig extends AbstractCloudConfig {
    @Bean
    public ConnectionFactory rabbitConnectionFactory() {
    	return connectionFactory().rabbitConnectionFactory();
    }
   
    @Bean
    public DataSource dataSource() {
    	return connectionFactory().dataSource();
    }
 
    @Bean
    public MongoDbFactory mongoDb() {
    	return connectionFactory().mongoDbFactory();
    }
 
    @Bean
    public RedisConnectionFactory redisConnectionFactory() {
    	return connectionFactory().redisConnectionFactory();
    }

    @Bean
    public ApplicationInstanceInfo applicationInfo() {
        return cloud().getApplicationInstanceInfo();
    }
}

与第一个版本相比，我们删除了@ServiceScan注解。相反，我们使用AbstractCloudConfig为每个服务创建bean。目前，以这种方式创建的bean与通过以下方式创建的bean相同@ServiceScan ，但是我们现在可以进一步对其进行配置。例如，如果我们想绑定DataSource Bean到特定服务（可能是在绑定到应用程序的许多服务中）并使用特定池配置对其进行初始化，我们可以进行以下更改：

@Bean
public DataSource dataSource() {
    PoolConfig poolConfig = new PoolConfig(20, 200);
    ConnectionConfig connectionConfig =
        new ConnectionConfig("sessionVariables=sql_mode='ANSI';characterEncoding=UTF-8");
    DataSourceConfig serviceConfig = 
        new DataSourceConfig(poolConfig, connectionConfig);
     return connectionFactory().dataSource("my-service", serviceConfig);
}

的DataSource以这种方式创建的最大池大小为20，最大等待时间为200毫秒，以及特定的连接属性字符串。

摘要

Spring Cloud提取了连接到云服务的抽象，并且可以毫不费力地将同一应用程序部署到多个云中。在此博客中，我们仅介绍了Spring Cloud提供的内容。在下一个博客中，我们将探索有关Java和XML配置的更多信息，以及如何在非Spring应用程序中使用其核心API。在随后的博客中，我们将更深入地探讨Spring Cloud的可扩展性。
