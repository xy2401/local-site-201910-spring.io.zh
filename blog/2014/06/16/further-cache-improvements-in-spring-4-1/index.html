Spring 4.1中进一步的缓存改进

工程
斯特凡·尼科尔（StéphaneNicoll）
2014年6月16日

这篇文章是我先前有关JSR-107的文章的后续。添加JSR-107支持使我们有机会回顾自己的观点，并了解两者如何快乐地生活在一起。Spring 4.1还包含社区报告的一系列改进。

我也很高兴地宣布，已经发布了专门用于缓存抽象的新入门指南，请检查Spring的缓存数据 ！

缓存解析器

我们在JSR-107中发现的最好的功能之一是能够解析缓存以在运行时使用，它基于实际的方法执行。到目前为止，我们自己的支持依赖于在注释（或方面定义）级别指定的缓存名称。提出了几个问题来报告，当一个以上CacheManager使用，必须能够自定义这些名称解析到的管理器。

我们最终要做的是创建一个类似的CacheResolver抽象以提供最灵活的选择。您可能会怀疑，默认实现需要一个CacheManager并根据提供的缓存名称解析要使用的缓存。由于不再需要这样的名称，因此Spring 4.1允许您编写以下内容：

@Cacheable
public Book findBook(ISBN isbn) {...}

当然，您必须指定自己的CacheResolver知道如何处理此特定调用，或者您需要提供缓存以其他方式使用（保持读取）。

就像value注释的属性不再需要， CacheManager可以省略，只要CacheResolver被设置。您实际上可能有一个不依赖于CacheManager如果您正在管理自己的Cache实例以不同的方式。

操作级别的自定义

社区还报告说，他们希望能够以更精细的方式控制缓存行为。Spring 4.1允许您指定CacheResolver （要么CacheManager ）和KeyGenerator每次操作使用。

例如，以下应用自定义KeyGenerator对那个特定的操作

@Cacheable(value="book", keyGenerator="myKeyGenerator")
public Book findBook(ISBN isbn, boolean checkWarehouse, boolean includeUsed)

这看起来KeyGenerator豆与名字myKeyGenerator 。CacheManager和缓存解析器实例可以以类似方式设置。

类级别的自定义

社区的另一个反馈是能够在方法级别共享那些自定义项， 而无需启用任何默认的缓存行为。遇见@CacheConfig它在类级别提供相同的自定义设置，即缓存名称， CacheResolver （要么CacheManager ）和KeyGenerator 。

@CacheConfig(cacheNames="book", keyGenerator="myKeyGenerator") 
public class BookRepository {

    @Cacheable
    public Book findBook(ISBN isbn) {...}

    public Book thisIsNotCached(String someArg) {...}
}

JCache集成

实施JSR-107支持时的主要目标之一是利用我们自己抽象中已有的东西，以便现有应用程序可以非常平滑地过渡。JCache支持实际上是在内部使用Spring的缓存抽象，这使您可以使用标准注释重用现有的缓存基础架构。当您指定JSR-107定制组件时，例如javax.cache.annotation.CacheResolver或javax.cache.annotation.CacheKeyGenerator ，我们使用适配器通过抽象路由它。

具体而言，您可以保留现有的基础架构并选择所需的任何机制。例如，您可以ConcurrentMapCacheManager使用ConcurrentHashMap带有标准JSR-107批注的幕后花絮。

如果要尝试这样做，请检查使用JCache而不是Spring注释的指南的本叉 （commit 32fea97显示实际更改的内容）

包起来

Spring 4.1将于今年夏天发布，它将大大增强其缓存抽象。添加对JSR-107的支持有助于我们进一步改进它。这篇文章中没有涉及其他一些小的增强，例如更好的异常处理和便捷的putIfAbsent上的方法Cache接口。

与往常一样，我们欢迎社区提供反馈，请尝试使用这些功能，如果遇到任何问题，请告知我们。
