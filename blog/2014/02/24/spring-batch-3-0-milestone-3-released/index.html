Spring Batch 3.0 Milestone 3发布

发布
迈克尔·米内拉
2014年2月24日

今天，我们很高兴地宣布针对Spring Batch 3.0的最终里程碑版本（ 下载 ）。此版本演示了我们对JSR-352的支持，并且当前支持其所有必需的功能。我们还为现有Spring Batch用户向Spring Batch添加了许多新功能。Spring Batch是用于开发强大的批处理应用程序的轻量级全面框架。


Milestone 3中的功能


作为Spring Batch 3.0.0版本的最终里程碑，此版本的所有主要功能均已完成。此版本的主要功能包括：



  • JSR-352支持

  • 将Spring Batch集成到Spring Batch

  • 升级以支持Spring 4和Java 8

  • 其他修复和改进


JSR-352支持


Spring为我们对该规范所做的贡献投入了大量时间和资源。我们通过JCP与其他行业专家的合作，JSR-352验证了Spring Batch在过去六年中在无数生产环境中实施并经过测试的批处理模式是构建企业中关键任务批处理应用程序的最佳方法。


3.0.0。M3是Spring Batch的第一个兼容JSR-352的版本，它通过了所有152个SE TCK测试。JSR-352的Spring Batch实现允许用户使用Spring Batch熟悉的组件，并且仍然保持与Java标准的一致性。Spring Batch支持通过基于JSR-352的XML配置作业，使用JSR中定义的注入方法（Spring的依赖注入，batch.xml和内联配置）注入针对javax.batch。*接口开发的批处理工件，以及JSR-352中提供的所有可伸缩性选项（通过线程进行拆分和分区）。


我们实现的目标很简单：遵循JSR的规则而不破坏Spring Batch已经定义的规则。换句话说，在不影响现有Spring Batch用户的情况下实现规范。为此，我们提供了两个实现批处理作业的“路径”。第一种是传统的Spring Batch方法。使用Spring Batch XML或Spring的java配置，针对Spring Batch接口进行开发等。这不会在3.0版本中更改。第二个“路径”是JSR路径。这种开发方法使用JSR-352定义的XML模式来配置您的作业，使用JSR的JobOperator用于启动作业和管理元数据，并且开发人员根据JSR的API编写代码。重要的是要注意，这是两条截然不同的路径。虽然使用Spring Batch组件可以处理基于JSR-352的作业，但它们在这些上下文中的使用将由JSR的规则规定。例如，注入JSR-352中定义的上下文（ StepContext和JobContext ）。如果使用传统的Spring Batch方法配置作业并通过常规Spring Batch组件启动了作业，那么这些上下文将不适用于批处理工件。但是，如果您通过JSR-352 XML配置作业并通过JSR-352启动它JobOperator ，这些上下文将在适当的地方注入。


重要的是要指出，Spring Batch不仅实现了JSR-352。它在很多方面都比规范更进一步：



  • 组件-Spring Batch提供17种不同的ItemReader实施，16 ItemWriter实施以及许多其他在生产环境中经过多年测试的组件。

  • 可扩展性-JSR-352仅为单个JVM提供扩展选项（通过线程进行分区和拆分）。Spring Batch提供了多种JVM可伸缩性选项，包括远程分区和远程分块。

  • Spring依赖注入-尽管JSR-352提供了一种“依赖注入光”形式，但它在批处理工件的构造上存在许多限制（例如，必须使用no-arg构造函数）。Spring Batch是基于Spring构建的，并受益于Spring Framework功能的强大功能。

  • 基于Java的配置-尽管众所周知基于Spring的基于XML的配置选项，但Spring（特别是Spring Batch）提供了使用基于Java的配置的类型安全性来配置作业的选项。

  • Hadoop /大数据集成-Spring Batch是用于与Spring生态系统中的Hadoop和其他大数据存储进行交互的基础工具。Spring for Apache Hadoop提供了许多与批处理相关的扩展，以使用Spring Batch来协调Hadoop集群上的工作。Spring XD通过在提供所有功能的同时提供执行功能和类似于Spring Batch Admin的管理功能的基础上构建Spring Batch。

在进行维护修订时，Spring将继续参与JSR-352的发展，并期待为JCP流程做出进一步的贡献。


将Spring Batch集成到Spring Batch


几年来，Spring Batch集成一直是Spring Batch Admin鲜为人知的子项目。但是，考虑到该库提供的功能多么强大以及它在企业开发中的有用性，我们决定将其从Spring Batch Admin总括下移至Spring Batch总括下。Spring Batch Integration旨在提供桥接批处理和集成处理的有用组件，为启动和扩展批处理应用程序提供了许多选项。功能包括：



  • 异步ItemProcessor / ItemWriter-执行ItemProcessor另一个线程上的逻辑，返回一个Future到ItemWriter 。一旦Future返回，结果被写入。

  • JobLaunchingMessageHandler / JobLaunchingMessageGateway -提供通过通道接收的Spring Messages启动作业的功能。

  • 远程分块-提供执行能力ItemProcessor通过主/从配置进行远程（跨多个JVM）逻辑。

  • 远程分区-提供通过主/从配置远程执行完整块（跨多个JVM进行读取/处理/写入）的功能。


这些组件使Spring Batch应用程序可以在许多企业中无缝集成和扩展。


升级以支持Spring 4和Java 8


为了使依赖关系保持最新状态，我们更新了Spring Batch及其支持Spring 4的各种模块。由于这些更新，Spring 4将成为Spring Batch 3.0.0的最低Spring支持版本。M3。这不仅带来了Spring 4中大量的新功能，而且还提供了立即在Java 8上运行的功能。


其他修复和改进


除了上述大型新功能之外，我们还添加了一些较小的功能并继续修复错误。我们添加的新功能包括添加JobScope并支持SQLite。


候选发布路线图


那还剩下什么呢？当我们接近发布候选者时，仍有一些小工作要做。我们仍在仔细研究Spring Batch的依赖关系，并希望尽可能地升级。


结论


3.0.0。M3标志着Spring Batch的关键里程碑。它使开发人员可以测试基于JSR-352的功能，同时仍具有Spring和Spring Batch所依赖的基础。它还为框架带来了新功能。我们期待您在论坛，社交媒体和当面的评论和反馈！
