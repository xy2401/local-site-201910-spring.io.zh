Spring YARN应用模型

工程
珍妮·瓦尔基亚拉赫蒂
2014年2月6日

现在，Thomas刚刚为Apache Hadoop 发布了Spring的第五个里程碑，我想借此机会谈谈其新功能Spring YARN的最新开发。

Spring IO平台的一项优势是其技术的互操作性。一个很好的例子是Spring Boot和Spring YARN如何共同为Hadoop YARN应用程序开发创建更好的模型。在此博客文章中，我想展示一个很大程度上基于Spring Boot的新Spring Yarn Application模型的示例。

春季纱介绍

从开发人员开始工作到有人在Hadoop集群上实际执行应用程序的时间，开发生命周期要比仅创建几行代码复杂得多。让我们看看需要考虑什么：

  • 什么是应用程序代码的项目结构
  • 项目是如何构建和打包的
  • 打包的应用程序如何配置
  • 如何执行申请

我们认为，Spring YARN和Spring Boot提供了一个简单的编程模型来开发应用程序，而不是可以轻松地作为YARN应用程序或传统应用程序进行测试和部署。

在较高的层次上，Spring YARN提供了三个组件， YarnClient ， YarnAppmaster和YarnContainer反映了YARN体系结构中的关键过程。总之，这三个组件提供了Spring YARN应用程序模型的基础。

在Hadoop集群上打包和执行自己的代码一直是一个繁琐的过程。需要将已编译的程序包放在Hadoop的类路径中，或者让Hadoop的工具在作业提交期间将您的程序包复制到Hadoop中。一旦超过了WordCount，您的代码将依赖于Hadoop默认类路径中不存在的第三方库。您应该如何打包依赖库？此外，如果您的依赖项与已经属于Hadoop默认类路径一部分的库发生冲突该怎么办。

Spring Boot可以为这些构建和打包问题提供一个优雅的解决方案。您可以创建一个可执行的jar（有时称为uber或fat jar），将您的应用程序代码及其所有依赖项捆绑到一个.jar文件中，或者创建一个zip文件，可以在执行代码之前将其分解。两种打包格式之间的主要区别在于，后者允许您重新使用Hadoop的默认类路径中已经可用的jar。

在本指南中，我们将展示这三个组件YarnClient ， YarnAppmaster和YarnContainer使用Spring Boot将它们打包到可执行jar中。在内部，Spring Boot严重依赖于应用程序自动配置，Spring YARN添加了自己的自动配置魔术。然后，用户可以专注于自己的代码和应用程序配置，而不必花费大量时间尝试了解所有组件应如何相互集成。

现在，我们将向您展示创建自定义应用程序并将其部署到Hadoop集群有多么简单。注意，没有必要使用XML。

创建一个纱线容器

在这里创建ContainerApplication和HelloPojo类。

@Configuration
@EnableAutoConfiguration
public class ContainerApplication {

	public static void main(String[] args) {
		SpringApplication.run(ContainerApplication.class, args);
	}

	@Bean
	public HelloPojo helloPojo() {
		return new HelloPojo();
	}

}

在上面ContainerApplication ，请注意我们如何添加@Configuration在课堂上@Bean为一个helloPojo()方法。我们之前提到YarnContainer组件，它是您要在容器中执行的操作的接口。您可以定义自己的习惯YarnContainer实现此接口并将所有逻辑包装在该实现中。

但是，Spring YARN默认为DefaultYarnContainer如果未定义任何容器，并且此默认实现期望找到一个具有实际逻辑的特定bean，那么该容器应该执行什么操作。这有效地创建了一个简单的模型，其中至少只有一个简单的模型POJO是必需的。

@YarnContainer
public class HelloPojo {

	private static final Log log = LogFactory.getLog(HelloPojo.class);

	@Autowired
	private Configuration configuration;

	@OnYarnContainerStart
	public void publicVoidNoArgsMethod() {
		log.info("Hello from HelloPojo");
		log.info("About to list from hdfs root content");
		FsShell shell = new FsShell(configuration);
		for (FileStatus s : shell.ls(false, "/")) {
			log.info(s);
		}
	}

}

HelloPojo上课很简单POJO从某种意义上说，它没有扩展任何Spring YARN基类。我们在这堂课上做了什么：

  • 我们增加了一个班级@YarnContainer注解。
  • 我们添加了一个方法级别@OnYarnContainerStart注解
  • 我们已经@Autowired Hadoop的Configuration类

@YarnContainer是具有Spring的构造型注释本身@Component在其中定义。这会自动将一个班级标记为具有@YarnContainer功能。

在本课程中，我们可以使用@OnYarnContainerStart注释，用于标记没有返回类型或参数的公共方法，这是需要在Hadoop上执行的操作。

为了证明我们实际上在此类中具有某些实际功能，我们仅使用Spring Hadoop的@FsShell从列表的根目录列出条目HDFS文件系统。为此，我们需要访问Hadoop的Configuration为您准备好了，以便您可以自动为其布线。

创建一个纱线客户端

在这里创建一个ClientApplication类。

@EnableAutoConfiguration
public class ClientApplication {

	public static void main(String[] args) {
		SpringApplication.run(ClientApplication.class, args)
			.getBean(YarnClient.class)
			.submitApplication();
	}

}
  • @EnableAutoConfiguration告诉Spring Boot根据类路径设置，其他bean和各种属性设置开始添加bean。
  • 由于Spring YARN在类路径上，因此会针对Spring YARN组件进行特定的自动配置。

的main()方法使用Spring Boot的SpringApplication.run()启动应用程序的方法。从那里我们只需要一个类型为的bean YarnClient并执行它submitApplication()方法。接下来会发生什么取决于应用程序配置，我们将在本指南的后面部分进行介绍。

创建一个纱线Appmaster

在这里您创建一个AppmasterApplication类。

@EnableAutoConfiguration
public class AppmasterApplication {

	public static void main(String[] args) {
		SpringApplication.run(AppmasterApplication.class, args);
	}

}

申请类别YarnAppmaster看起来比我们刚刚做的还要简单ClientApplication 。再次main()方法使用Spring Boot的SpringApplication.run()启动应用程序的方法。

有人可能会争辩说，如果您使用这种虚拟类来基本上启动您的应用程序，我们是否可以为此使用通用类？好简单的答案是，我们甚至有一个通用的SpringYarnBootApplication为此目的而上课。您将其定义为可执行jar的主类，并在gradle构建期间完成此操作。

但是，在现实生活中，您很可能需要开始向应用程序组件中添加更多的自定义功能，而您可以通过开始添加更多的bean来做到这一点。为此，您需要定义一个Spring @Configuration要么@ComponentScan 。 AppmasterApplication然后将作为您定义更多自定义功能的主要起点。

创建应用程序配置

创建一个新的yaml配置文件。

spring:
  yarn:
    appName: yarn-boot-simple
    applicationDir: /app/yarn-boot-simple/
    fsUri: hdfs://localhost:8020
    rmAddress: localhost:8032
    schedulerAddress: localhost:8030
    client:
      appmasterFile: yarn-boot-simple-appmaster-0.1.0.jar
      files:
       - "file:build/libs/yarn-boot-simple-container-0.1.0.jar"
       - "file:build/libs/yarn-boot-simple-appmaster-0.1.0.jar"
    appmaster:
      containerCount: 1
      containerFile: yarn-boot-simple-container-0.1.0.jar

应用程序的最后一部分是其运行时配置，该配置将所有组件粘合在一起，然后可以将其称为Spring YARN应用程序。该配置充当Spring Boot的源@ConfigurationProperties并包含相关的配置属性，这些属性无法自动发现，或者需要由最终用户覆盖的选项。

然后，您可以为自己的环境编写自己的默认值。因为这些@ConfigurationProperties在运行时由Spring Boot解决的情况下，您甚至可以使用命令行选项或提供其他配置属性文件来覆盖这些属性。

编译应用

该博客中使用的示例代码可以从我们在GitHub上的spring-hadoop-samples回购中找到。

签出示例后，请从以下位置发出gradle build命令boot/yarn-boot-simple目录。

$ cd boot/yarn-boot-simple
$ ./gradlew clean build

对于此样本，我们希望保持项目结构简单。我们不在此博客中介绍gradle构建文件，但有个故事就是，我们将从一个项目中创建三个不同的jar文件。实际上，人们可能会使用多项目模型，其中每个子项目都将创建自己的jar文件。
