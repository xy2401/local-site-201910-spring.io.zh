Spring Data Codd RC1有哪些新功能？

工程
奥利弗·德罗博姆
2014年2月7日

随着我们正在接近当前正在开发中的Spring Data发布系列的延伸，我想是时候向您简要概述我们将在此版本中引入的一些新功能。

万一您还不知道，发布系列是多个Spring Data模块的协调发布，以确保它们无缝兼容。当前的一个以关系模型Edgar F. Codd的发明者的名字命名。

Spring Data 共享区

当Spring Data Commons模块为发布系列中包含的所有其他模块奠定基础时，各个商店模块中也提供了其中的新功能。

对于Codd版本，我们将所需的最低Spring版本提高到3.2.7，并改进了与Spring 4.0的兼容性，以在更新到迄今为止最新和最大的Spring版本时提供无缝的体验。现在，使用Spring 4和Java 8运行应用程序时，Spring Data审计功能现在可以使用JSR-310类型填充实体的创建和修改日期：

abstract class BaseEntity {
    
    @CreatedDate ZonedDateTime createdDate;
    @LastModifiedDate ZonedDateTime modifiedDate;
}

此外，现在通常可以使用特定于模块的模块来激活审核@Enable…Auditing注释（有关详细信息，请参阅Thomas Darimont的[blog post]（
有关第一个里程碑中引入的新功能，请参见https://spring.io/blog/2013/12/04/what-s-new-in-spring-data-mongodb-1-4-m1 ）。

从Codd开始，即使没有注入点在容器启动时显式请求存储库实例的情况下，Spring Data存储库也将急切地实例化，以确保对存储库接口的验证被迅速触发。要恢复原来的行为，请使用@Lazy就像其他Spring bean定义一样，在存储库接口上添加了注释。

分页链接的URI模板

在对Spring MVC的网络支持方面，Spring Data的PagedResourcesAssembler现在创建符合RFC-6570中定义的标准的页面导航链接。因此，假设您有一个像这样的Spring MVC控制器：

@Controller
class OrderController {
    
    @RequestMapping("/orders")
    HttpEntity<PagedResources<Resource<Order>>> showOrders(Pageable pageable, 
        PagedResourcesAssembler<Order> assembler) {

        Page<Order> orders = orderRepository.findAll(pageable);
        return new ResponseEntity<>(assembler.toResource(orders), HttpStatus.OK);
    }
}

的PagedResourcesAssembler将检查给定Page的Order ，呈现内容，页面元信息，并添加prev和next适当的链接。

GET /orders?page=0&size=10
Accept: application/hal+json

200 OK
{ _links : { self : { href : "/orders{?page,size,sort}",
                      templated : true },
             next : { href : "/orders?page=1&size=10" }},

  _embedded :  { orders : [ … ] },

  page : {
    size : 1,
    totalElements : 2,
    totalPages : 2,
    number : 0
  }
}

如您所见， self链接清楚地指示资源接受以自定义响应的参数。的next链接提供对与给定当前参数相对应的下一页的访问。

为了解决问题，Commons模块对配置的评估方式进行了重大更改，这将显着缩短引导时间，尤其是在具有大量存储库的应用程序中。

Spring Data JPA

Spring Data JPA模块在与JPA和CDI交互方面的主要改进之处在于内部改进和新功能：我们已升级到EclipseLink 2.5.1，并改进了与Hibernate 4.3的兼容性，以确保我们的用户可以使用基于JPA 2.1的持久性提供程序。

EntityManager的构造方法注入

多次要求提供的功能也加入了此版本：注入功能EntityManager设计应用程序组件时，将实例转换为构造函数。如果编写需要访问JPA的类EntityManager ，您通常使用@PersistenceContext触发它的注入。但是，出于各种原因 ，我们通常认为构造函数注入是最佳实践，因此不必回退到字段注入就很有用EntityManagers 。

不幸的是， @PersistenceContext当前不能在构造函数参数上使用注释。为此，我已经在JPA Bug跟踪器中提交了票证 ，但是我们当然不必等待JPA 2.2发布就可以看到该功能。从Spring Data JPA 1.5版开始，您将能够编写如下的应用程序组件：

class UserRepositoryImpl implements UserRepositoryCustom {
    
    private final EntityManager em;

    @Autowired // or @Inject
    public UserRepositoryImpl(EntityManager em) {
        this.em = em;
    }

    …
}

默认情况下，我们将注入EntityManager由创建的实例EntityManagerFactoryBean在您的声明ApplicationContext 。如果定义了多个，则可以使用@Qualifier指向的Bean名称EntityManagerFactoryBean您想要获得EntityManager从。

CDI环境中急切的存储库实例化

当在Spring容器中使用Spring Data存储库时，存储库会被实例化。甚至在Codd之前，客户端组件通过注入点请求存储库后就立即实例化它们。因此，只要将存储库注入客户端，就可以确保获得完全初始化的应用程序组件 ，就像Spring组件模型可以保证的那样。

但是，在CDI世界中，情况略有不同。当您访问应用程序组件时，无法保证其依赖项已被初始化，因为在CDI组件模型中，仅注入代理并在首次使用时触发实际实例创建就可以了。

实际上，这可能导致在应用程序代码已经在运行JPA事务时实例化Spring Data存储库。存储库实例化的一部分是检查JPA命名查询以返回查询方法。根据JPA规范，此类命名查询的不存在（在Spring Data上下文中是完全预期的状态）必须通过抛出异常来表示，并且根据定义，该异常必须触发JPA中的事务回滚。

这意味着通过将应用程序组件初始化与CDI中正在运行的业务代码混合在一起，您有可能在调用前者时破坏后者。不幸的是，没有办法在CDI中开箱即用地对应用程序组件进行即时实例化。有一些特定于容器的方法，但是没有一种方法可以对所有方法都可靠地工作（如果您碰巧找到了一种，我会非常感兴趣）。

使用Spring Data JPA 1.5，我们将发布一个@Eager注释将导致我们的CDI扩展触发器存储库，从而避免应用程序组件创建和业务代码执行的重叠。

Spring Data REST

Spring Data REST中最核心的变化是我们的重要重构，尤其是在模块的映射配置和自定义部分。这还包括我们为单个资源公开的默认关系类型的更改。

对于每个存储库，Spring Data REST都会遵循集合资源模式公开一个专用的REST资源。这意味着我们拥有专用于实体集合的资源以及集合中每个项目的个体资源。

由于不应该使用适当的REST客户端来创建与其交互的URI，而是使用超媒体来跟踪服务器提供的链接，因此我们需要公开两种不同的关系类型，以指示集合资源和项目资源之间的差异。

从Spring Data REST 2.0开始，项目资源的关系类型是从存储库管理的域类的名称派生的。然后，通过使用Evo Inflector库对项目资源rel进行复数运算 ，从而从中得出集合资源的关系类型。因此，对于存储库管理Order实例，您将看到名为orders指向收集资源并order指向项目资源。

关系类型的这种变化是Spring Data REST 1.x基于服务的超媒体客户端的一个重大突破，这是主要原因之一，该模块的版本已提高到2.0。另一个方面是，使用模块的M1，我们移到了HAL作为要由Spring Data REST公开的默认媒体类型。我们正在寻求对其他超媒体格式（例如Collection + JSON等）的支持，但是HAL似乎吸引了很多关注。亚马逊最近刚刚发布了他们的AppStream REST API，并使用HAL作为表示格式。

Spring Data Neo4j / MongoDB

Spring Data Neo4j模块中最大的更改（因此也转移到了3.0版本号）支持Neo4j 2.0。因此，要从该版本的最新和最强大的功能中受益，请确保您尝试一下Codd版本的候选人。

在MongoDB方面，最重要的新功能是聚合框架中对Spring Expression语言的支持。Spring Data工程师Thomas Darimont撰写了一篇详细的博客文章 ，介绍了Codd M1已经提供的所有功能，以防您错过。

如上所述，当然，Codd中引入的Spring Data Commons中的所有新功能实际上也都在Neo4j和MongoDB模块中冒泡了。
