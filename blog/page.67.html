Spring Cloud Data Flow 1.3 GA发布

发布
冈纳·希尔特（Gunnar Hillert）
2018年2月1日

我很高兴地代表团队宣布Spring Cloud Data Flow 1.3在各种平台上的普遍可用性

遵循本地服务器 ， Cloud Foundry和Kubernetes的入门指南

发布要点

流更新和回滚

精心策划为一系列微服务应用程序的流数据管道一直是Spring Cloud Data Flow设计的核心价值。在Data Flow 1.3中，我们提供了独立更新源，处理器和接收器的功能，而不必取消部署和重新部署整个流。

通过将部署过程委托给一个名为Skipper的新Spring Cloud项目来实现流更新和回滚功能。Skipper是一个轻量级的Spring Boot应用程序，旨在填补Data Flow中的这一功能空白。船长定义了一种包裹格式，很像helm要么brew并且还可以将应用程序部署/取消部署到多个云平台：本地，Cloud Foundry和Kubernetes。它使用了从一开始就已包含在Data Flow中的相同Spring Cloud Deployer库。SpringOne 2017上的最新演讲更深入地介绍了Skipper以及与Data Flow的集成 。

部署Stream时，Data Flow会创建Skipper程序包，以描述Stream和属于Stream定义的应用程序。然后，Skipper将应用程序部署到所需的平台。请求流更新时，仅自动重新部署一个或多个需要更改的应用程序。一个由Spring Statemachine实例管理的简单策略执行更新或回滚步骤。

数据流包括新的流命令以进行升级和回滚操作。

dataflow:>app register --name transform --type processor --uri maven://com.eg:transformer:0.0.1

dataflow:>stream create mystream --definition "jdbc | transform | mongodb"

dataflow:>app register --name transform --type processor --uri maven://com.eg:transformer:0.0.2

dataflow:>stream update mystream --properties “version.transform=0.0.2”

dataflow:>stream rollback mystream

在这一系列命令中，使用转换器的0.0.1版部署流。的jdbc和mongodb源和接收器已经注册。然后，将流更新为使用转换器的0.0.2版。只有transform应用程序已更新，其中部署了0.0.2版，而未部署了0.0.1版。的jdbc和mongodb应用程序保持原样。rollback命令执行相反的操作，将流返回到转换器0.0.1版的状态。

注意：要使用Data Flow和Skipper，必须在Data Flow Server和Shell中都启用Skipper的Data Flow 功能切换 。
