模块化客户端：Angular JS和Spring Security第VII部分

工程
戴夫·瑟
2015年5月13日

在本文中，我们继续讨论如何在“单页应用程序”中将Spring Security与Angular JS一起使用。在这里，我们展示了如何对客户端代码进行模块化，以及如何使用不带Angular默认使用但大多数用户都不喜欢的片段表示法（例如“ /＃/ login”）的“漂亮” URL路径。这是系列文章中的第七篇，您可以了解应用程序的基本构建块，或者通过阅读第一篇文章从头开始构建它，或者直接进入Github中的源代码 。我们将能够从本系列其余部分的JavaScript代码中整理出很多零散的内容，同时展示如何使其非常紧密地适合由Spring Security和Spring Boot构建的后端服务器。

分解应用程序

到目前为止，我们在本系列中使用的示例应用程序都很简单，以至于我们可以只用一个JavaScript源文件就可以了。即使更大的应用程序从一开始就是这样的生活，也不会以这种方式结束，因此，为了在样本中模拟真实的生活，我们将进行分解。一个良好的起点是采用第二部分中的“单个”应用程序，并在源代码中查看其结构。这是静态内容的目录列表（不包括服务器上的“ application.yml”）：

static/
 js/
   hello.js
 home.html
 login.html
 index.html

这有一些问题。一个显而易见：所有JavaScript都在一个文件中（ hello.js ）。另一个更微妙：我们在应用程序内部具有HTML“部分”（视图“ login.html”和“ home.html”），但它们全部为平面结构，并且与使用它们的控制器代码无关。

让我们仔细看一下JavaScript，我们会发现Angular使我们很容易将其分解为更易于管理的部分：

注意：所有客户端代码都在一个单独的目录“ js”下（ index.html除外，因为这是一个“欢迎”页面，并且会从“静态”目录自动加载）。这是有意的，因为它可以轻松地将单个Spring Security访问规则应用于所有静态资源。这些都是不安全的（因为/js/**在Spring Boot应用程序中默认是不安全的），但是对于其他应用程序可能需要其他规则，在这种情况下，您将选择其他路径。

使用“自然”路线

默认情况下，Angular $routeProvider与URL路径中的片段定位器一起使用，例如，登录页面在hello.js被指定为“ / login”的路由，并且在实际URL中将其翻译为“ /＃/ login”（一个您会在浏览器窗口中看到）。这样，通过根路径“ /”加载的index.html中的JavaScript在所有路由上都保持活动状态。片段命名对于用户来说有点陌生，有时使用“自然”路由会更方便，因为URL路径与Angular路由声明相同，例如“ / login”代表“ / login”。如果只有静态资源，则无法执行此操作，因为只能以一种方式加载index.html ，但是如果堆栈中有一些活动组件（代理或某些服务器端逻辑），则可以安排它通过从所有Angular路径加载index.html来工作。

在本系列中，您具有Spring Boot，因此当然具有服务器端逻辑，并且使用简单的Spring MVC控制器，您可以在应用程序中归纳路由。您需要的是一种枚举服务器中Angular路由的方法。在这里，我们选择通过命名约定来做到这一点：所有不包含句点（并且尚未显式映射）的路径都是Angular路由，应该转发到主页：

@RequestMapping(value = "/{[path:[^\\.]*}")
public String redirect() {
  return "forward:/";
}

该方法只需要在Spring应用程序中的某个位置的@Controller （而不是@RestController ）中。我们使用“转发”（而不是“重定向”），以便浏览器记住“真实”路由，这就是用户在URL中看到的内容。这也意味着Spring Security中与身份验证有关的所有保存请求机制都可以立即使用，尽管我们不会在此应用程序中利用它。

注意： github上的示例代码中的应用程序有一条额外的路线，因此您可以看到功能更全的功能，因此希望是现实的应用程序（“ / home”和“ / message”是不同的模块，视图略有不同）。

提取身份验证问题

当您对上面的应用程序进行模块化时，您应该已经发现代码只是通过将其拆分为模块而起作用，但是有一个小问题，我们仍在使用$rootScope在控制器之间共享状态。对于如此小的应用程序，这没有什么可怕的错误，它使我们可以很快地使用一个不错的原型，因此，我们不必为此感到难过，但是现在我们可以借此机会将所有身份验证问题提取到一个单独的模块中。用Angular术语来说，您需要的是“服务”，因此在“ home”和“ navigation”模块旁边创建一个新模块（“ auth”）：

static/
  js/
    auth/
      auth.js
    home/
      home.js
      home.html
    navigation/
      navigation.js
      login.html
    hello.js
  index.html

在编写auth.js代码之前，我们可以预期其他模块中的更改。首先，在navigation.js ，应使“导航”模块依赖于新的“身份验证”模块，并将“身份验证”服务注入控制器中（当然不再需要$rootScope ）：

angular.module('navigation', ['auth']).controller(
		'navigation',

		function($scope, auth) {

			$scope.credentials = {};

			$scope.authenticated = function() {
				return auth.authenticated;
			}

			$scope.login = function() {
				auth.authenticate($scope.credentials, function(authenticated) {
					if (authenticated) {
						console.log("Login succeeded")
						$scope.error = false;
					} else {
						console.log("Login failed")
						$scope.error = true;
					}
				})
			};

			$scope.logout = function() {
              auth.clear();
            }

		});

它与旧的控制器没有太大区别（它仍需要用于用户操作，登录和注销的功能，以及用于保存登录凭据的对象），但是它已将实现抽象为新的“ auth”服务。“ auth”服务将需要一个authenticate()函数来支持login() ，以及一个clear()函数来支持logout() 。它还具有一个已authenticated的标志，该标志替换了旧控制器中的$rootScope.authenticated 。我们使用authenticated标志与连接到同一名称的函数$scope控制器，使角将会继续检查其价值和更新用户界面，当用户登录英寸

然后终于可以编写代码在上面用铅笔的三种功能（ authenticate() clear()和init()这是大多数代码：

angular.module('auth', []).factory(
    'auth',

    function($http, $location) {

      var auth = {

        authenticated : false,

        loginPath : '/login',
        logoutPath : '/logout',
        homePath : '/',

        authenticate : function(credentials, callback) {

          var headers = credentials && credentials.username ? {
            authorization : "Basic "
                + btoa(credentials.username + ":"
                    + credentials.password)
          } : {};

          $http.get('user', {
            headers : headers
          }).success(function(data) {
            if (data.name) {
              auth.authenticated = true;
            } else {
              auth.authenticated = false;
            }
            $location.path(auth.homePath);
            callback && callback(auth.authenticated);
          }).error(function() {
            auth.authenticated = false;
            callback && callback(false);
          });

        },
        
        clear : function() { ... },
        
        init : function(homePath, loginPath, logoutPath) { ... }

      };

      return auth;

    });

“ auth”模块为auth服务创建工厂（例如，您已经将其注入“ navigation”控制器中）。工厂只是一个返回对象（ auth ）的函数，该对象必须具有我们上面期望的三个函数和标志。上面，我们显示了authenticate()函数的实现，该函数与“导航”控制器中的旧函数基本相同，它在“ / user”处调出后端资源，设置一个经过authenticated的标志并调用带有标志值的可选回调。如果成功，它还会使用$location服务将用户发送到homePath （我们将在一分钟内对此进行改进）。

这是init()函数的基本实现，它仅设置了您不想在auth模块中进行硬编码的各种路径：

init : function(homePath, loginPath, logoutPath) {
  auth.homePath = homePath;
  auth.loginPath = loginPath;
  auth.logoutPath = logoutPath;
}

接下来是clear()函数的实现，但这很简单：

clear : function() {
  auth.authenticated = false;
  $location.path(auth.loginPath);
  $http.post(auth.logoutPath, {});
}

它将取消设置已authenticated标志，将用户发送回登录页面，然后将HTTP POST发送到注销路径。POST成功是因为我们仍然具有原始“单一”应用程序中的CSRF保护功能。如果看到403，请查看错误消息和服务器日志，然后检查是否已安装该过滤器以及是否正在发送XSRF cookie。

重定向到登录页面

到目前为止，我们实现主页的方式具有一些内容，可以在对用户进行身份验证时显示该内容（它只是邀请他们登录）。有些应用程序以这种方式工作，而有些则没有。有些提供了不同的用户体验，用户在经过身份验证之前从登录页面看不到任何东西，因此让我们看看如何将应用程序转换为这种模式。

隐藏登录页面中的所有内容是一个典型的横切关注点：您不希望所有显示登录页面的逻辑都卡在所有UI模块中（它会在所有地方重复，从而使代码更难阅读且更难理解保持）。Spring Security与服务器中的所有交叉问题有关，因为它建立在Filters和AOP拦截Filters之上。不幸的是，这在单页应用程序中对我们没有多大帮助，但是幸运的是Angular还具有一些易于实现我们想要的模式的功能。在此为我们提供帮助的功能是，您可以安装“路由更改”的侦听器，因此，每当用户移动到新路由（即单击菜单栏等）时，或首次加载页面时，您都可以检查路线，如果需要，可以更改它。

要安装侦听器，您可以在auth.init()函数中编写一小段额外的代码（因为已经安排在主“ hello”模块加载时运行）：

angular.module('auth', []).factory(
    'auth',

    function($rootScope, $http, $location) {

      var auth = {
      
        ...

        init : function(homePath, loginPath, logoutPath) {
          ...
          $rootScope.$on('$routeChangeStart', function() {
            enter();
          });
        }

      };

      return auth;

    });

我们注册了一个简单的侦听器，该侦听器仅委派给新的enter()函数，因此现在您还需要在“ auth”模块工厂函数中实现该函数（可以访问工厂对象本身）：

enter = function() {
  if ($location.path() != auth.loginPath) {
    auth.path = $location.path();
    if (!auth.authenticated) {
      $location.path(auth.loginPath);
    }
  }          
}

逻辑很简单：如果路径只是更改为登录页面以外的内容，则记录路径值，然后，如果用户未通过身份验证，请转到登录页面。我们保存路径值的原因是，这样我们可以在成功进行身份验证后返回到该值（Spring Security具有此功能的服务器端，对用户来说非常好）。通过在成功处理程序中添加一些代码，您可以在authenticate()函数中进行此操作：

结论

在本文中，我们了解了如何模块化Angular应用程序（以该系列的 第二部分中的应用程序为起点），如何使其重定向到登录页面以及如何使用可以键入的“自然”路由或由用户轻松添加书签。我们从本系列的最后几篇文章中退后了一步，将精力集中在客户端代码上，暂时放弃了我们在第III-VI部分中构建的分布式体系结构。这并不意味着此处所做的更改不能应用于其他应用程序（实际上是微不足道的）-只是为了简化服务器端代码，同时我们正在学习如何在客户端上做事。有一对夫妇服务器端的功能，我们使用或简要讨论虽然（例如Spring MVC中使用“前进”的观点，以使“自然”路线），所以我们不断的角度和春天一起工作主题，并显示它们在这里和那里的细微调整中效果都很好。
