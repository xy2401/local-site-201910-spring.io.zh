如何不讨厌2016年的 Spring

工程
菲尔·韦伯
2015年11月29日

在感恩节周末，萨姆·阿特金森（Sam Atkinson）于2014年发表了一篇名为《为什么我讨厌 Spring 》的文章，开始在Twitter上进行巡回讨论。听到批评意见以了解我们如何才能改进Spring总是很有趣的， Spring Boot的大部分源于听人们谈论框架所面临的问题 。

在这篇博客文章中，我将尝试解决Sam文章中讨论的一些问题，并描述我个人的Spring“最佳实践”。但是，在我们过多讨论细节之前，有必要考虑一下为什么框架中仍然存在不尽如人意的做事方式。

向后兼容的诅咒

除了人们写自己讨厌它的东西以外，在Spring上工作时压力更大的方面之一是，功能往往有很长的尾巴。Spring对向后兼容性有着坚定的承诺，这意味着一旦添加了功能，就很难从根本上对其进行更改。开发框架提出了应用程序开发人员通常没有的独特挑战。诸如YAGNI （您将不需要它）之类的敏捷技术以及主动重构的难度要大得多。我们拥有SMNI （可能需要它），并且谨慎地进行重构。仍支持在Java中存在lambda，泛型甚至注释之前有意义的功能。很容易争辩说，如今XML并不是连接bean的好方法，但是要完全删除XML却很难说。我个人认为我可以轻松升级较旧的Spring应用程序并逐渐迁移到新技术这一事实。

还要记住，就像商业软件一样，开源具有有限的资源和时间压力。有时，我们作为开发人员会犯错，最终需要和他们一起生活一段时间。

太大，太大？

从本质上讲，Spring是一个集成框架。它为许多不同的技术提供了一致的编程模型。可以公平地说，Spring可以发挥很大的作用，但是如果不与许多技术集成，就不可能提供与许多技术的集成。尽管它看起来似乎不堪重负，但是一旦您了解了核心概念，就很容易只选择所需的Spring部分。如果您有一些特定的目标需要实现，那么spring.io上的指南可能是有用的起点。Spring Boot启动器列表或使用start.spring.io也是将Spring限制在特定区域的好方法。

依存关系

一个典型的基于Spring的应用程序将依赖于任意数量的第三方库。从历史上看，很难确切知道各种库的哪个版本可以很好地协同工作。如果您今天开始一个新项目，我强烈建议您使用Spring Boot的父POM或Spring Platform 。两者都提供了一组精心设计的依赖项，它们可以很好地协同工作。

我们甚至开发了Gradle插件，当您没有Maven的父依赖项功能的好处时， 该插件可让您使用 POM 的托管依赖项部分。

如果由于某种原因而不能使用托管依赖项，那么仍然可以参考Spring Boot文档以获取已知的兼容版本列表 。

注射

始终在bean中使用基于构造函数的依赖项注入 。始终对强制性依赖项使用断言 。有关为何基于字段的注入为何有害的更多背景知识，您可以阅读Oliver Gierke的这篇文章 。

与往常一样，该规则有一个例外，当您使用测试时，最好在测试中使用基于字段的注入SpringJUnit4ClassRunner 。

漏泉

我见过几个人认为，应用程序代码不应该有任何的框架依赖。即org.springframework导入到任何地方（配置代码除外）都是不好的。尽管您绝对可以做到这一点（对标准注释（例如@Inject ），我往往没有那么明确。我更喜欢在这里应用80/20规则。我认为对框架类和注解有依赖性是很好的，但是如果我要离开Spring，我不想重写超过20％的类。

我想要的是易于测试的应用程序代码。我应该可以手动创建我的业务服务的实例并对其进行测试，而无需完整的实例ApplicationContext 。

组态

我个人喜欢保持我的Spring配置相当轻巧。我喜欢使用组件扫描来查找应用程序Bean，并且希望保留自己的应用程序@Configuration用于偏离Spring Boot默认自动配置的特定于框架的配置。这些天我都不倾向于使用XML配置。

有人认为最好对所有已定义的bean进行显式配置，但实际上，我倾向于发现这是一个障碍。遵循这种方法的应用程序往往只有一个@Configuration该类最终了解系统中的所有内容，并且包含一堆小的@Bean方法。我还发现注释的自我记录性质非常有用。我可以轻松搜索@Service并找到我所有的服务 bean 。

山姆（Sam）的文章说： “……注释仍然是魔术。在运行您的应用程序之前，您不知道它是否正确连接”，但我真的不明白这与拥有一个手动连接所有内容的类有何不同。您仍然需要在某个时候运行整个应用程序，以确保它可以运行。

分层

任何规模很大的系统都需要考虑架构层。我几乎所有的Spring应用程序最终都在底部有一个“域”层，在顶部有一个“网络”层。我发现最好在各层之间严格分隔，并在单个方向上保持依赖关系。使用以下方法注释域服务异常非常容易@ResponseStatus但我很少发现这在实践中是个好主意。同样，将一些杰克逊注释添加到@Entity因此可以将其作为JSON编组，但是从长远来看，这通常会引起问题。

我建议使用诸如Structure 101和SonarQube之类的工具，这些工具可以检查代码中的包装缠结和设计问题。

Spring MVC

我通常发现最好将业务逻辑保留在Spring MVC之外@Controllerbeans。我喜欢将Spring MVC视为一个小适配器层。它负责获取HTTP输入，对其进行验证，将其传递给服务然后返回响应。我会经常添加调解器@Service Web层的bean负责协调对业务服务的调用。通常，这使测试事情变得更加容易，尤其是在使用MVC Test framework时 。

记录中

登录Java确实应该没有那么困难。有太多的日志记录库可供选择，并且有许多不同的组合，很容易陷入困境。幸运的是，对于典型的Spring Boot应用程序，您实际上不必担心太多。我的建议始终是：

  • 在代码中使用SLF4J记录事物。
  • 如果可能，请坚持使用Spring Boot的默认登录选项。
  • 仅登录到控制台，并使用Splunk之类的工具进行捕获和存储。

测试中

Spring单元测试的黄金法则是将Spring排除在单元测试之外！应该可以对大多数bean进行单元测试，而无需增加应用程序上下文。在测试世界中，关于模拟的争论很多，但是我通常认为明智地使用Mockito （我喜欢BDD Mockito ）非常有帮助。

在进行集成测试时，您通常需要涉及Spring。我倾向于发现我的集成测试经常配置应用程序的“层”。例如，域测试可能会启动Hibernate + Spring Data +内存数据库。对于Spring MVC测试，我倾向于不理数据库，而注入模拟服务。上面讨论的分层体系结构在这里确实开始产生回报，因为很容易模拟或存根“ Web层”所需的内容。

启动时间

一个典型的Spring Boot REST应用程序启动大约需要2.5秒，一旦开始混合使用Hibernate之类的技术，事情就会变慢。Spring旨在“快速失败并尽早失败”，不幸的是，这确实会影响启动时间。从好的方面来说，一旦应用程序启动，很可能就处于运行状态。

如果您确实有启动缓慢的问题，请考虑查看Spring Boot Devtools ，它可以使重新启动更快。您还应该看看使用@Lazy如果您有自己的创建缓慢的bean，则进行初始化。

Spring Boot

Spring Boot于2013年底推出，专门解决了Spring用户面临的一些常见问题。不幸的是，我已经看到“ Spring现在是如此复杂，以至于它拥有自己的框架。”很多时候都引用了这样的话。Spring Boot故意不同于Spring Framework，它具有不同的目标，不同的依赖关系和不同的发布时间表。我更喜欢将Spring Framework视为原材料，而将Spring Boot视为完全烘焙的蛋糕。您可以随意随意混合配料，但谁又不喜欢蛋糕？

选择与开源

开源最大的优点之一就是它为您提供了极大的自由。如果您想要依赖注入，但您不喜欢Spring，请看看Guice 。如果要构建REST服务，但不喜欢Spring MVC，请尝试Ratpack 。如果您想要一个集成良好，经过良好测试的，记录良好的框架，并且拥有良好的业绩记录，请尝试使用Spring with Spring Boot 。

如果您想更改开源的工作方式，请尝试贡献 ！
