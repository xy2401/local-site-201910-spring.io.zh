将Spring Cloud Apps从Spring Boot 1.2迁移到1.3

工程
戴夫·瑟
2015年11月25日

Spring Boot 1.3中有一些有趣的新功能，现在可以在Brixton发行版的Spring Cloud中使用。Spring Cloud的Angel发布系列与Spring Boot 1.3部分不兼容，因此在升级时，需要注意一些重要事项。本文可帮助您浏览更改并更新任何现有应用程序以使用新功能。通常，在尝试将Spring项目的新版本引入现有代码库时，它通常也应有所帮助。

提示：您可以使用mvn dependency:tree要么gradle dependencies列出项目中的依赖项并检查版本。

依赖管理

如果您使用的是较早版本的Spring Boot，则Maven POM中可能包含以下内容：

<parent>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-parent</artifactId>
  <version>1.2.7.RELEASE</version>
  <relativePath /> <!-- lookup parent from repository -->
</parent>

要么

<dependencyManagement>
  <dependencies>
    <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-parent</artifactId>
      <version>1.2.7.RELEASE</version>
      <type>pom</type>
      <scope>import</scope>
    </dependency>
  </dependencies>
</dependencyManagement>

或者，如果您使用的是Gradle，

buildscript {
	ext {
		springBootVersion = '1.2.7.RELEASE'
	}
	dependencies {
		classpath("org.springframework.boot:spring-boot-gradle-plugin:${springBootVersion}")
	}
}

要升级到Spring Boot 1.3.0，您可以将上面的“ 1.2.7”更改为“ 1.3.0”。到目前为止很简单。

提示：要查看带有最新版本的Spring Boot的“典型” Maven POM，可以curl start.spring.io/pom.xml 。要添加Spring Cloud，您可以追加-d style=cloud-config-client 。可以通过添加来更改Spring Boot版本-d bootVersion=1.3.1.BUILD-SNAPSHOT （例如）。为Gradle使用做同样的事情build.gradle代替pom.xml 。

在Spring Boot中使用Spring Cloud

由于Spring Cloud是基于Spring Boot构建的，因此可能会造成混淆，并且很难找到可以一起使用的组合。在下面的内容中，我们描述了一些升级场景，并说明了依赖管理可以实现的目标。

大升级

通常，最大的变化是升级时（将Spring Boot 1.2升级到1.3，或将Spring Cloud Angel升级到Brixton）。如果您从Spring Initializr下载了一个项目，那么它将使用Spring Boot父POM：

<parent>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-parent</artifactId>
  <version>1.2.7.RELEASE</version>
  <relativePath/> <!-- lookup parent from repository -->
</parent>

和一个Spring Cloud BOM 部分：

<dependencyManagement>
  <dependencies>
    <dependency>
      <groupId>org.springframework.cloud</groupId>
      <artifactId>spring-cloud-starter-parent</artifactId>
      <version>Angel.SR4</version>
      <type>pom</type>
      <scope>import</scope>
    </dependency>
  </dependencies>
</dependencyManagement>

在Gradle中，您会看到以下内容：

buildscript {
    ext {
        springBootVersion = '1.2.7.RELEASE'
    }
    repositories {
        mavenCentral()
    }
    dependencies {
        classpath("org.springframework.boot:spring-boot-gradle-plugin:${springBootVersion}") 
    }
}
dependencyManagement {
  imports { 
    mavenBom "org.springframework.cloud:spring-cloud-starter-parent:Angel.SR4" 
  }
}

在任何一种情况下，仅更新Spring Boot版本都将无法正常工作，因为Spring Cloud Angel BOM具有旧版本的Spring Boot和Spring（还有其他功能）。因此，我们确实需要同时升级Spring Boot和Spring Cloud。例如在Maven中：

<parent>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-parent</artifactId>
  <version>1.3.0.RELEASE</version>
  <relativePath/> <!-- lookup parent from repository -->
</parent>
<dependencyManagement>
  <dependencies>
    <dependency>
      <groupId>org.springframework.cloud</groupId>
      <artifactId>spring-cloud-starter-parent</artifactId>
      <version>Brixton.M3</version>
      <type>pom</type>
      <scope>import</scope>
    </dependency>
  </dependencies>
</dependencyManagement>

在Gradle中：

buildscript {
    ext {
        springBootVersion = '1.3.0.RELEASE'
    }
    repositories {
        mavenCentral()
    }
    dependencies {
        classpath("org.springframework.boot:spring-boot-gradle-plugin:${springBootVersion}") 
    }
}
dependencyManagement {
  imports { 
    mavenBom "org.springframework.cloud:spring-cloud-starter-parent:Brixton.M3" 
  }
}

注意：Brixton。M2和所有早期版本的Spring Cloud与Spring Boot 1.3.0 不兼容。发布。您至少需要Brixton。M3。

升级1.3.0之后的Spring Boot

假设您要使用Spring Boot快照，或者在发布时升级到1.3.1，但是Spring Cloud没有明确取决于您想要的Boot版本的版本。

在Maven中，请记住，如果使用现成的父POM之一，则它们包含并将优先。考虑到这一点，如果您使用那些父POM，请确保使用与您的依赖关系设置最接近的父POM（在本方案中为Boot 1）。

<parent>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-parent</artifactId>
  <version>1.3.1.BUILD-SNAPSHOT</version>
  <relativePath/> <!-- lookup parent from repository -->
</parent>
<dependencyManagement>
  <dependencies>
    <dependency>
      <groupId>org.springframework.cloud</groupId>
      <artifactId>spring-cloud-starter-parent</artifactId>
      <version>Brixton.M3</version>
      <type>pom</type>
      <scope>import</scope>
    </dependency>
  </dependencies>
</dependencyManagement>

在Gradle中，原则上更简单，因为没有“父母”的概念。实际上，除非您也手动应用依赖项管理插件，否则不能将Spring Boot插件与依赖项管理使用其他版本。所以你必须在build.gradle ：

buildscript {
    ext {
        springBootVersion = '1.3.0.RELEASE'
    }
    repositories {
        mavenCentral()
    }
    dependencies {
        classpath "io.spring.gradle:dependency-management-plugin:0.5.3.RELEASE"
        classpath("org.springframework.boot:spring-boot-gradle-plugin:${springBootVersion}") 
    }
}

apply plugin: "io.spring.dependency-management"
...

dependencyManagement {
  imports { 
    mavenBom "org.springframework.cloud:spring-cloud-starter-parent:Brixton.M3" 
    mavenBom "org.springframework.bootspring-boot-starter-parent:1.3.1.BUILD-SNAPSHOT" 
  }
}

apply plugin: 'spring-boot'

规则是您必须a）在Spring Boot 之前手动导入依赖项管理插件，b）声明dependencyManagement 在应用Spring Boot插件之前。完成后，您可以在dependencyManagement声明， 最后一个获胜（与Maven相反）。

注意：对声明顺序的这种敏感性是工具当前版本的“功能”。在将来的版本中可能会有所不同。有关更多详细信息，请参见Gradle工具中的此问题 。

将Maven与自定义父级一起使用

如果您不使用现成的父POM，则可以自由使用不包含父POM的POM。 ，这使事情更易于控制。在这种情况下，您需要将Spring Boot和Spring Cloud放入而且顺序很重要：第一个获胜（Gradle的最后一个获胜）。例如，使用Spring Boot 1.3.1。BUILD-SNAPSHOT和Spring Cloud Brixton。Maven中的M3：

<dependencyManagement>
  <dependencies>
    <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-parent</artifactId>
      <version>1.3.1.BUILD-SNAPSHOT</version>
      <type>pom</type>
      <scope>import</scope>
    </dependency>
    <dependency>
      <groupId>org.springframework.cloud</groupId>
      <artifactId>spring-cloud-starter-parent</artifactId>
      <version>Brixton.M3</version>
      <type>pom</type>
      <scope>import</scope>
    </dependency>
  </dependencies>
</dependencyManagement>

BOM的顺序

请注意，在Maven和Gradle中，BOM的顺序都很重要：首先声明的材料通常在Maven中获胜（在Gradle中的最后一个），如果顶层存在冲突（明确声明的依赖关系）。与Maven的最大区别在于，父对象是特殊的：如果它包含它总是赢。

要了解某个特定的依赖版本是否会以您需要的方式解决，就很复杂。这取决于物料清单的顺序， 并在传递依赖关系树，你的依赖声明的深度上。例如，Spring Boot BOM为一个显式（1级）声明依赖项管理spring-core但不能在其他任何Spring Framework jar上使用（这些jar是通过对Spring Framework BOM的引用引入的）。规则是，第一次声明成功，但是包括整个树（包括所有BOM），从顶部开始逐级搜索。

注意：如果没有Spring Boot（或Spring Dependency Management）插件，Gradle没有这个“最后一个获胜”规则。要使用“本机” Gradle构建执行相同的操作，通常需要仔细而乏味的工作，手动修复可传递依赖项版本。

进一步处理依赖项版本

如果您想要使依赖项的版本超出Spring Boot和Spring Cloud BOM中指定的范围，事情可能会变得很复杂。广义上讲，有2个选项：属性和其他BOM。第一个（属性）与现成的父POM一起使用，而另一个则不行。第二（更BOM表）只有在有可供你有兴趣，并且仅当传递依赖不要与你的要求相冲突的相关性的BOM工作。例如，所有Spring Cloud项目和Spring Framework都有自己的BOM，因此这是一个开始。

物产

Spring Boot父POM（如果使用，则为Spring Cloud的POM，因为它是从Boot 1继承的）将其所有依赖项版本提取到 。因此，您通常可以只更改属性值。Maven中的示例：

<parent>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-parent</artifactId>
  <version>1.3.0.RELEASE</version>
  <relativePath/> <!-- lookup parent from repository -->
</parent>
<properties>
  <spring.version>4.2.4.BUILD-SNAPSHOT</spring.version>
</properties>

Gradle中有一个相应的功能是ext财产，例如

ext['spring.version'] = '4.2.4.BUILD-SNAPSHOT'

其他物料清单

Spring Framework 具有自己的BOM，因此我们可以使用它来管理Spring版本。在具有自定义父项的Maven中（不包含 ）：

<dependencyManagement>
  <dependencies>
    <dependency>
      <groupId>org.springframework</groupId>
      <artifactId>spring-framework-bom</artifactId>
      <version>4.2.4.BUILD-SNAPSHOT</version>
      <type>pom</type>
      <scope>import</scope>
    </dependency>
  </dependencies>
</dependencyManagement>

注意：此示例实际上不适用于Spring Boot父POM（除非它碰巧具有相同的功能） ），因为Spring Framework 版本已由父级修复。要将Spring Boot父级与Spring Framwork快照一起使用，最好使用属性方法（上述）。

在Gradle中，它更简单（因为没有父级来设置有冲突的版本）：

dependencyManagement {
	imports { 
      mavenBom "org.springframework:spring-framework-bom:4.2.4.BUILD-SNAPSHOT" 
      mavenBom "org.springframework.boot:spring-boot-starter-parent:1.3.0.RELEASE" 
	}
}

结论

依赖管理很困难，但希望我们通过概述升级Spring Boot和Spring Cloud时的一些常见方案来减轻影响。根据您选择的是Maven还是Gradle，会有一些稍微不同的行为，但是至少如果您选择Gradle并使用Spring Boot插件，则差异会最小化。归根结底，Spring项目具有不同的发布时间表，因此总会有冲突，但是它们通常总是趋向于趋同，因此，如果等待足够长的时间，事情就会变得均衡。诸如Spring Cloud，Spring Boot和Spring IO Platform之类的伞项目也有助于消除麻烦：如果您可以使用其中之一来管理所有依赖项，事情就会变得简单得多。

现在，Spring Guides中的示例应用程序已全部更新到Spring Boot 1.3，即使这意味着它们依赖于Spring Cloud的里程碑（这仅适用于Zuul代理示例）。许多人不再需要Spring Cloud。如果您需要GA版本的Spring Cloud，则需要立即使用Spring Boot 1.2。可以从git历史记录中提取该组合的样本。
