Java不会吸-摇滚JVM

工程
布赖恩·杜索
2015年2月11日

最近，James Ward撰写了一篇很棒的博客文章， “ Java不会烂-您只是在错误地使用它” ，其中强调了企业Java开发人员在构建Java应用程序的日常例程中面临的众多挑战。好消息是，突破开发痕迹比您想象的要容易得多。在过去的几年中，Spring重新定义了现代Java应用程序的构建方式，同时极大地提高了开发速度。在这篇文章中，我将以James Ward的博客文章为背景，来解释Spring如何帮助开发人员（使用Java）在解决James概述的每个问题时动摇JVM（使用Java）。

即时开发环境设置过程艰难

在James的博客中，他断言“设置Dev Environments Suck的10页Wiki”。我们都已经在职业生涯中的某个时刻创建了这些Wiki页面，并试图保持最新状态而烦恼。这些Wiki长期以来一直使新团队成员感到沮丧，因为他们经常到处都是过时的信息。好消息是，Spring Boot的构建支持使此过程的自动化非常容易。

克隆源存储库并运行该应用程序非常简单，从而可以轻松使用现有应用程序。Spring Boot提供了对Java最流行的构建工具Maven和Gradle的支持。使用您喜欢的构建工具来运行功能齐全的应用程序很简单：

Maven的例子：

mvn spring-boot:run

摇篮示例：

gradle bootRun

由于Spring Boot应用程序是一个独立的工作负载，将应用程序代码，服务器和第三方的依赖关系组合到一个构建和部署单元（可执行的.jar文件）中，因此无论IDE如何，都可以确保开发团队获得直接，可重复的开发经验。开发人员从繁琐的环境设置说明中解放了出来，可以专注于构建出色的应用程序。可以通过执行以下操作来完成在构建之外运行Spring Boot应用程序：

java -jar target/my-application-1.0.1-SNAPSHOT.jar

提示：想自己尝试一下吗？遵循spring.io上的许多入门指南之一。对于刚接触Spring Boot的开发人员来说，一个很好的起点是使用Spring Boot构建应用程序指南。

一致的部署环境

James Ward强调的下一个挑战是“在将开发版本从开发阶段升级到生产阶段时，将风险最小化，每个环境之间唯一应该改变的就是配置”。手动修改部署工件是灾难的根源，最终将导致部署延迟或失败。特定于环境的配置应该外部化，以确保在开发中测试的相同代码是将用于生产的“黄金副本”。这是证明您所测试的就是您所部署的东西的唯一经济有效的方法！

Spring Boot使您可以轻松地使用非常特殊的PropertySource顺序来外部化配置，该顺序旨在合理地覆盖值。通常希望在项目中提供用于本地开发的默认配置，但在通过环境提升代码时会覆盖这些值。Spring Boot提供了通过命令行参数，JNDI属性外部配置的全面支持，Java系统属性，操作系统的环境变量，配置文件，配置文件驱动的配置变型，和更多 。Spring Boot对环境变量的支持使遵循“ 十二要素”应用程序配置最佳实践 （从配置中严格分离代码）变得轻而易举。

提示1：当使用Cloud Foundry等云平台时，Spring Boot应用程序可以利用Spring Cloud Connectors自动绑定到Cloud Foundry服务，例如数据库和消息传递系统。这样做的好处是减少了应用程序需要维护的特定于环境的配置属性的数量，从而大大降低了跨环境升级代码时出错的风险。

提示2：好奇为什么十二要素应用很重要？查看此博客文章，“ 为什么要考虑12种因素的应用程序模式，微服务和CloudFoundry问题” 。

快速的服务器启动过程

Spring Boot提供了对轻量级，可嵌入式容器/服务器的支持，这些容器/服务器可以快速启动。一个简单的REST应用程序可以在短短3秒钟内启动。从Spring Boot 1.2开始，已经为Tomcat（默认容器），Jetty和Undertow提供了嵌入式应用服务器支持。Spring Boot不仅为领先的轻量级容器提供支持，而且使开发人员可以轻松地替换默认值，从而使开发人员可以完全控制这些决策。

以下Gradle构建配置可支持全栈Web开发，包括对嵌入式Tomcat和spring-webmvc的支持。

摇篮示例：

...

apply plugin: 'java'

repositories { jcenter() }
dependencies {
    compile("org.springframework.boot:spring-boot-starter-web:1.2.0.RELEASE")
}

...

正如詹姆斯·沃德（James Ward）在其原始博客文章中指出的那样，通过将单片式部署划分为微服务，可以进一步缩短启动时间。在本文的稍后部分，将进一步详细讨论微服务主题（请参阅标题为“微服务样式架构的岩石”部分 ）。

托管依存关系摇滚

现代开发人员需要工具和技术，使他们能够以最小的摩擦迅速上手。他们还需要模块化，轻巧和自以为是的技术来优化生产力。Spring Boot着眼于快速启动和运行同时极大地提高开发速度的问题。

正如詹姆斯·沃德（James Ward）正确指出的那样：“如果您的任何库依赖项都不由构建工具管理，那将很糟糕”。Spring Boot不仅通过支持现代构建技术，而且还提供了方便的依赖项描述符（称为starter POMS），使这种最佳实践成为第一种实践。入门POM为常见的开发工作负载提供了相关性组，您可以将它们简单地包含在应用程序中。

要开始使用Spring Boot，可以将浏览器指向Spring Initializer- http://start.spring.io 。 Spring Initializer提供了一个基于Web的界面，允许开发人员选择应用程序/工作负载和相关的依赖项。然后它将生成具有构建支持的入门应用程序（支持Maven POM，Maven项目，Gradle Config和Gradle项目）。

Spring Initializer屏幕截图：

对于喜欢命令行的开发人员，可以通过发出以下命令来安装Spring Boot CLI：

curl http://start.spring.io/install.sh | sh

只需发出以下命令，就可以使用命令行来初始化新项目：

spring init --dependencies=web,data-jpa my-project

初始化的结果（通过Spring Initializer或Boot CLI）是一个完全可运行的应用程序，其中所需的依赖项已由Spring Boot包含并自动配置。

Spring IO Platform （包括Spring Boot）还提供了由Spring Boot的启动程序POM管理的依赖关系的超集。 Spring IO级别设置了整个Spring产品组合的依赖版本，并且发布周期比Spring Boot更长，从而为企业提供了经过测试且已知可以协同工作的依赖版本的快照。Spring IO Platform发行版不是库的整体下载，开发人员可以自由选择仅需要在应用程序中使用的部分。Spring IO Platform发行版包含Spring模块，测试库，日志记录框架，数据库管理，SQL / No-SQL等版本。对于希望以逐步的方式使用Spring IO Platform依赖性的企业而言，Spring IO发行版是理想的选择。

以下Gradle配置演示了如何配置Spring IO Platform物料清单并利用Spring Boot的Web Starter POM：


buildscript {
    repositories {
        jcenter()
    }
    dependencies {
        classpath 'io.spring.gradle:dependency-management-plugin:0.3.0.RELEASE'
    }
}

apply plugin: 'io.spring.dependency-management'

repositories {
    mavenCentral()
}

dependencyManagement {
    imports {
        mavenBom 'io.spring.platform:platform-bom:1.1.0.RELEASE'
    }
}

dependencies {
    compile 'org.springframework.boot:spring-boot-starter-web'
}

短的开发/验证周期确实很艰难

大多数现代应用程序都使用丰富的用户界面与后端服务进行通信。以这种方式构建应用程序的好处是可以将UI和服务器端逻辑之间的关注点高度分离。大多数现代的IDE提供静态资源的动态重载，使开发人员无需重新启动服务器即可查看更改。

Spring Boot还支持服务器端模板技术，例如Thymeleaf，Freemarker和Groovy。Spring Boot允许动态重新加载这些模板技术，而无需重启服务器。对于Thymeleaf，这就像在application.properties文件中设置以下属性一样简单：

spring.thymeleaf.cache: false

在服务器方面，将您的整体应用程序分解为微服务将减少Spring Boot应用程序的启动时间（由于使用了轻量级嵌入式容器，因此已经相当快了）。另外，Spring Boot提供了对测试实用程序自动配置的支持。启用Boot的测试自动配置就像添加spring-boot-starter-test启动器POM一样容易。这样可以启用Spring Test，JUnit，Hamcrest和Mockito依赖项，从而可以轻松地将测试驱动开发（TDD）纳入日常工作流程中。TDD使开发人员可以立即收到有关其代码更改的反馈。

最后，JRebel为类重载提供了全面的支持，并支持80多个Java框架（包括Spring）。观看有关Josh Long和Adam Koblentz的网络研讨会Spring Boot和JRebel 6 ，了解更多详细信息。

微服务风格架构

James的声明“大型发行很烂”对于大型或复杂的分布式系统来说都是存在的。大多数开发人员希望以更迭代的方式（敏捷）工作并更频繁地发布。微服务风格的体系结构已成为一种流行的方法，可以应对单片应用程序的挑战。对于微服务新手，Martin Fowler在他的博客中对这种架构风格进行了很好的描述。

在构建分布式应用程序（包括微服务样式应用程序）时，会出现许多常见的系统模式。通过提供开箱即用的服务来应对最常见的挑战，Spring Cloud使采用这些模式变得简单。Spring Cloud实现了一些模式，例如配置管理，服务发现，断路器，智能路由，微代理和控制总线。Spring Boot使将这些功能集成到您的应用程序中变得非常简单。

例如，Spring Cloud Netflix项目使在Netflix的Eureka上建立服务发现服务变得如此简单：


@SpringBootApplication 
@EnableEurekaServer

public class Application {
    public static void main(String[] args) {
        new SpringApplicationBuilder(Application.class).web(true).run(args);
    }

}

客户端应用程序只需使用@EnableEurekaClient即可注册为Eureka客户端：


@SpringBootApplication
@EnableEurekaClient
@RestController
public class Application {

    @RequestMapping("/")
    public String home() {
        return "Hello world";
    }

    public static void main(String[] args) {
        new SpringApplicationBuilder(Application.class).web(true).run(args);
    }

}

提示1：Spring Cloud提供了对众多分布式处理模式的支持。为了帮助您入门，请查看Spring Cloud Samples github存储库。Dave Syer和Spencer Gibb的Spring Cloud，Spring Boot和Netflix OSS SpringOne会话是理解和学习如何实现微服务风格架构的另一种好方法。

提示2：如果您要构建Hypermedia风格的REST API，请务必查看《入门指南， 使用REST访问JPA数据》 。 Spring Data REST将帮助您加速服务开发。Spring Data REST支持SQL和No-SQL存储库。

拥有允许您快速编写微服务代码的技术只是成功的一半。开发人员常常将其代码投入生产平台时遇到许多延迟。一些最常见的障碍包括手动部署，测试，基础结构设置和服务设置。持续交付旨在使将代码推广到生产中的过程自动化，同时最大程度地降低风险。这使组织可以对应用程序进行增量更改，从而使部署成为业务决策，而不是资源决策。它本身是一个很大的话题，但是我强烈推荐Matt Stine的演讲， 即使用Spring和Cloud Foundry开发用于PaaS的微服务 ，它的出色表现突出了将敏捷工程实践与Pivotal Cloud Foundry自动化相结合的好处。

无状态应用程序

James指出：“粘性会话和服务器状态通常是杀死性能和弹性的最佳方法之一。会话状态（从传统的Servlet的角度来看）使得执行持续交付和水平扩展真的非常困难。

从应用程序中删除会话状态可以大大简化操作，从而可以重新部署，终止或扩展应用程序，而不必担心丢失会话数据。存在用于维护状态的有效用例，例如共享身份验证状态，但是此状态应保留在应用程序外部（通常存储在高性能存储库中，例如NoSQL，分布式缓存，甚至是内存中的数据存储区）。外部化应用程序状态不必费劲，Spring Session提供了通用的基础结构来使此过程变得简单且可移植。春季会议提供：

  • 支持厂商中立的集群
  • RESTful API支持-支持标头中的会话ID
  • 用于确定会话ID的可插拔策略
  • 支持在WebSocket处于活动状态时使HttpSession保持活动状态
  • 支持Redis和Hazelcast支持的会话存储
  • 能够在单个浏览器中管理多个同时进行的会话（即类似于多个Google帐户支持）

使用Redis配置Spring Session非常简单：

@EnableRedisHttpSession
public class Config {

    @Bean
    public JedisConnectionFactory connectionFactory() {
        return new JedisConnectionFactory();
    }
}

提示：Spring Session 1.0最近加入了GA ，因此请尝试使用Spring Session示例以查看其运行情况。Dave Syer最近发表了一篇名为“ API网关模式：Angular JS和Spring Security第IV部分 ”的博客文章，其中详细介绍了共享身份验证状态用例。强烈推荐阅读！

最后，如果您使用的是Pivotal Cloud Foundry，则Cloud Foundry Java构建包提供了另一个选项，可以通过自动配置的CF会话状态存储轻松地将状态外部化。查看此博客文章以了解详细信息。

非阻塞应用摇滚

许多现代应用程序体系结构可从请求的异步和非阻塞处理中大大受益。这些用例可以包括组成多个后端服务调用和WebSocket样式的应用程序。

Project Reactor（Spring IO Platform的一部分）为构建这些异步，非阻塞应用程序提供了基础。在2.0版本中，反应器将提供一个全面实施的反应流规格与其他无流实现比如开辟了整合阿卡流 ， Ratpack ， RxJava 。

以下代码段显示了如何使用Reactive Streams API创建流，向其中添加业务逻辑，然后将数据发布到其中：

// by default Streams use the Disruptor RingBufferDispatcher
Broadcaster<String> helloStream = Streams.broadcast(env);

helloStream.map(s -> "Hello " + s + "!")
           .consume(log::info);

helloStream.onNext("World");

提示：Spring Boot为项目Reactor提供支持，使入门变得简单。遵循《 使用Reactor创建异步，事件驱动的应用程序入门指南》中的步骤来创建您的第一个反应性应用程序。另外，请务必查看网络研讨会， 使用Reactor for Asynch，无阻塞微服务 。

Spring Framework 4引入了对WebSocket样式，事件驱动的应用程序的支持。这种务实的方法远远超出了JSR-356，它包括SockJS的客户端后备选项，对消息传递子协议（STOMP）的支持，安全性（Spring Security 4），消息代理支持，基于Reactor的MessageChannel用于消息传递，客户端断开连接处理以及熟悉的Spring编程模型。

提示：Spring Boot使配置WebSocket应用程序变得轻而易举。遵循《 使用WebSocket》来构建交互式Web应用程序指南以进行滚动。

Java语言的岩石

Spring Boot为开发人员提供了使用Java 6、7、8以及Groovy来构建下一代应用程序的选项。他在James的文章中指出“ Java Language Kinda Sucks”，但是随着Java 8的发布，Java得到了推动，并提供了许多强大的语言功能，这些功能将提高生产率。Java 8的一些出色功能包括对Lambda表达式，流和并发改进的支持。Spring Boot使Java 8或Groovy入门变得毫不费力。

提示：如果您不熟悉Java 8的功能，我强烈建议您使用Venkat Subramaniam的书《 Java中的函数式编程：利用Java 8 Lambda表达式的力量》及其SpringOne2GX 2014演讲

结论

构建现代Java应用程序不必是痛苦的经历。Spring Boot结束了构建应用程序的仪式，再次使Java变得有趣。Spring从应用程序的每一层中删除了样板-业务逻辑（Spring Foundation项目），配置和运行时（Spring Boot）以及分布式系统模式（Spring Cloud）。最好的入门方法是研究Spring的入门指南，然后将其部署到Pivotal Web Services （Cloud Foundry的公共托管版本，免费使用60天）。
