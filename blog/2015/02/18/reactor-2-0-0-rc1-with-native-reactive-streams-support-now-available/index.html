Reactor 2.0.0。RC1具有本机反应流支持现已可用！

发布
乔恩·布里斯宾
2015年2月18日

Reactor团队很高兴宣布2.0.0版本。RC1，现在可在spring.io Maven存储库以及Maven central中使用 。2.0版是#uberupdate来自Reactor 1.1版，包含几个新组件以及重要类的完整重写，例如Stream ，现在实现了Reactive Streams标准 。

请注意，Reactor 2.0的Maven坐标与Reactor 1.x的Maven坐标已更改。新坐标全部归入组ID io.projectreactor而不是以前的org.projectreactor 。Gradle项目的示例依赖项块可能如下所示：

ext {
  reactorVersion = '2.0.0.RC1'
}

repositories {
  maven { url 'http://repo.spring.io/libs-milestone' }
  mavenCentral()
}

compile "io.projectreactor:reactor-core:$reactorVersion",
    "io.projectreactor:reactor-net:$reactorVersion",
    "io.projectreactor.spring:reactor-spring-context:$reactorVersion"

什么是反应堆？

如果您是Reactor的完全新手，则可能需要先跳转到http://projectreactor.io上这个崭新的网站并熟悉它，然后再进行一些了解。

变更日志TL; DR

以及2.0.0中宣布的更改。M1版本 ，以下是1.1版的重要更改的简要列表：

  • Stream已被重写以实现Reactive Streams标准，速度提高了5-10倍，并且开销比以前的版本要少得多。
  • 从Reactor 1.x代码升级并不难，只需对现有代码进行一些更改即可适应新的签名。 Stream和重命名Reactor至EventBus 。有关此过渡的文档正在进行中。
  • 反应堆Stream API源自Reactive Extensions，并反映了其许多命名约定。通过利用常用的词汇和行为，可以轻松地将Rx.NET和RxJava示例转换为Reactor。
  • Reactor的API已在静态辅助方法和工厂之间进行了协调，以使嵌入尽可能容易。
  • TCP支持已被完全利用：Reactive Streams背压，HTTP服务器和客户端， Stream集成，DSL帮助程序方法。
  • 一个新网站 ，其中包含适当的参考手册的开头，更多示例和更多javadoc。
  • 社区的贡献不仅仅是核心项目提交者。

反应性流完全是老板

如果我们不得不只谈论一个更改，那么对Reactive Streams的本地和基础支持就可以了。很难夸大基本反应堆已成为Reactor的基础。流处理是新的 难题 ，Reactor从项目开始就开始接受它。但是，增加了反应性流及其对背压支持的传播，使云应用程序更易于访问实时或近实时处理大量数据。现在的模式如stop-read在负载下batch flush要么adaptive batch开箱即用。

反应器中的每个步骤Stream是Reactive Streams组件，可根据当前资源约束下的处理速率正确传播需求和背压。通过使用反应流，Reactor 2.0可以创建处理流，以自动调整其资源使用情况。由于Reactive Streams背压在上游进行通信的方式，您可以影响将新物料拉入系统的速率。这意味着，如果当前处理正在使用所有可用资源，则缓慢的下游组件可以将所有路径一直推回源，从而减慢摄取速率。

Pool<Connection> pool;
Stream<Message> input;

input.capacity(1)
     .batchConsume(msg -> {
       pool.getConnection().merge(msg);
     }, requestMore -> Math.max(pool.getSize() - pool.getActive(), 1));

在上面的代码段中，我们根据池中可用连接的数量来调整要处理的项目数量。的Consumer作为第一个参数传递给batchConsume方法将由requestMore从返回的值Function作为第二个参数传递。在这种情况下，我们将预取等于池中空闲连接数的消息，或者如果所有连接均处于活动状态，则仅预取一条消息（在这种情况下，我们将依赖于连接池的背压）。

如果我们想确保流不是资源消耗，我们还可以更改容量算法以返回小于可用连接数的数字，这将使某些连接对我们应用程序中的其他组件可用。

Reactor现在适用于Android

从Reactor 2.0.0开始。RC1，您可以通过简单地排除gs-collections库，否则会因其大小而迫使您跳过一些箍。我们已经实施了SimpleCachingRegistry为了EventBus那没用gs-collections 。未来的改进可能包括专用的UI事件循环Dispatcher确保您的事件处理程序在正确的线程上运行。

我们非常感兴趣地看到Reactor如何促进Android设备上的响应式应用程序，以及如何将其与Reactor在服务器端的超高容量，低延迟功能联系在一起。如果您在Android上使用Reactor，请告诉我们，是否可以做些事情来改善体验。

HTTP和更好的TCP和ZeroMQ支持

RC1基于Reactor对Netty 4的使用引入了对HTTP的新支持。它还不全面，但是它提供了一些辅助方法和一些有用的抽象，用于构建（和访问）基于REST的无阻塞微服务和纳米服务。我不会尝试使用它来构建生产服务，因为在GA之前还需要进行一些改进。您可以使用Reactor嵌入微服务，而无需直接使用Netty API。

下面的代码使用路径参数创建一个基于Netty的嵌入式HTTP服务器，该服务器将任务分派到共享服务器上。 RingBufferDispatcher 。

HttpServer<String, String> server = NetStreams.httpServer(
  spec -> spec.listen(3000)
              .codec(StandardCodecs.STRING_CODEC)
              .dispatcher(Environment.sharedDispatcher())
);

server.get("/echo/{greeting}", ch -> {
  String greeting = ch.param("greeting") + " World!";

  ch.transfer(Transfer.NON_CHUNKED)
    .responseHeader("Content-Length", "" + greeting.length())
    .log("server");

  return Streams.just(greeting);
});

server.start();

我们还更新了TCP和ZeroMQ支持，以更好地利用我们对Stream 。最重要的是，TCP服务器和客户端利用Reactive Streams背压支持来实现诸如“停止读取”之类的模式，以防止服务器通过从客户端读取过多数据来防止下游处理溢出，然后再有足够的资源来处理它。

通用航空之路

在发布Reactor 2.0 GA之前，我们将至少再进行一次RC。在对它的可预测性感到满意之前，我们仍然需要对复杂的fork / join调度进行一些调整。我们希望对HTTP支持进行一些补充，因为第一次切入是一组相当简单的功能。在边缘情况下，我们可能还会遇到更多错误。

我们对该候选版本感到非常满意，我们鼓励您尝试一下。如果您要进行新的开发，那么我们绝对鼓励您在Reactor 2.0的Reactive Streams基础上，而不是1.1中功能较弱的pre-Reactive Streams版本上构建。如果要升级现有的Reactor代码，则路径实际上很简单。在几乎所有情况下，您的代码都将大大简化。
