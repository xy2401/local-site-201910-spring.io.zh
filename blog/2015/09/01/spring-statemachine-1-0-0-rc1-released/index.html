Spring Statemachine 1.0.0。RC1发布

发布
珍妮·瓦尔基亚拉赫蒂
2015年9月1日

我们很高兴地宣布Spring Statemachine 1.0.0的第一个发行版本。RC1。

此版本的重点是使核心框架更稳定，并最终为分布式状态机添加jepsen测试。我们还添加了测试支持的第一个版本。可以从RC1问题中找到已解决的github票证。我们即将发布一个发行版本，这意味着如果没有重大弹出窗口，那么下一个发行版本将是1.0.0.RELEASE 。如果出现紧急情况，我们会做1.0.0.RC2发布之前。

现在我们在这里，让我们对其进行破解，看看在此版本中我们拥有哪些新功能。

除了常见的错误修复以外，这里还列出了一些主要的新功能：

  • 测试支持
  • 分布式状态机的Jepsen测试

测试支持

测试状态机并不是最容易完成的任务，因此我们引入了新的spring-statemachine-test该模块将简化为Spring Statemachine进行单元测试的过程。由于依赖关系，它没有在核心系统中使用，但是对于recipes和一个Zookeeper集成已经用于测试那些模块。

在测试中测试简单状态机看起来很简单：

StateMachine<String, String> machine = buildMachine();
StateMachineTestPlan<String, String> plan =
  StateMachineTestPlanBuilder.<String, String>builder()
    .defaultAwaitTime(2)
    .stateMachine(machine)
    .step()
      .expectStates("SI")
      .and()
    .step()
      .sendEvent("E1")
      .expectStateChanged(1)
      .expectStates("S1")
      .and()
    .build();
plan.test();

如果状态机定义为：

StateMachine<String, String> buildMachine()
    throws Exception {
  StateMachineBuilder.Builder<String, String> builder =
    StateMachineBuilder.builder();

  builder.configureConfiguration()
    .withConfiguration()
      .taskExecutor(new SyncTaskExecutor())
      .autoStartup(true);

  builder.configureStates()
    .withStates()
      .initial("SI")
      .state("S1");

  builder.configureTransitions()
    .withExternal()
      .source("SI").target("S1")
      .event("E1");

   return builder.build();
}

吉普车测试

事实证明，我们对使用Zookeeker使用分布式状态的支持相对难以通过一组正常的单元测试进行测试，因此，我们遇到了覆盖测试范围的难题。我想借此机会介绍一些有关使用jepsen测试分布式系统的信息。

Kyle Kingsbury的Jepsen测试框架（又名@aphyr ）是一个可用于测试分布式系统的系统，并具有将发送到节点的事件同步并引起网络大脑分裂等功能。Jepsen将成为我们系统的核心，以确保我们的分布式支持能够实现其应有的功能。这是测试由计算机支持的Spring Distributed Statemachine的一流系统Zookeeper 。

将会有一个单独的博客文章，但是初步结果可以从我们的参考文档《 分布式状态机技术论文》中找到 。敬请关注该博客文章！

在这里，我们从吉普森测试中窥视一下，以显示当集群遭受分裂而导致动物园管理员整体完全崩溃时发生的情况，以及当网络崩溃并恢复正常时会发生的情况。

在上图中，我们有一个5节点集群，它们共享一个相同的状态机配置，并由一个Zookeeper每个节点都连接到其自己的本地实例的集合。首先一个事件C发送到所有机器（只有一台机器将处理分布式状态更改），这将启动从状态的分布式转换S21进入S211 。然后网络中断，图表显示每台计算机最终将如何陷入错误状态。当网络和Zookeeper合奏随后被修复，所有机器将重新加入合奏并同步其状态。最终事件K将再次发送给所有计算机，以表明在解决网络问题后所有计算机都在正常工作。

如我们的文档中所述，如果将现有的Zookeer领导者保留为少数，则所有实例都将与集合断开连接，从而导致所有状态机进入错误状态。然后，当网络恢复正常并且Zookeeper组件自行修复并且连接到该状态机的状态机可以重置其自身状态时，此情况将自动解决。
