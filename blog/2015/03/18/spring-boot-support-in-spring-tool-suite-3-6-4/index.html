Spring Tool Suite 3.6.4中的Spring Boot支持

工程
克里斯·德·沃尔德
2015年3月18日

Spring Boot STS教程

Spring Tool Suite 3.6.4刚刚于上周发布。这篇博客文章是一个教程，展示了STS提供的一些新功能，这些功能可以创建和使用Spring Boot应用程序。

在本教程中，您将学习如何：

  • 使用STS创建一个简单的Spring Boot应用程序
  • 从STS启动和调试引导应用程序
  • 使用新的STS属性编辑器来编辑配置属性。
  • 在代码中使用@ConfigurationProperties可以为您自己的配置属性获得相同的编辑器支持。

创建启动应用

我们使用“ New Spring Starter”向导创建一个基本的Spring Boot应用程序。

Spring Boot提供了所谓的“启动器”。入门者是一组类路径依赖关系，它与Spring Boot自动配置一起使您无需进行任何配置即可开始使用应用程序。我们选择“ web”启动器，因为我们将构建一个简单的“ Hello”休息服务。

该向导是一个GUI前端，它在后台使用了start.spring.io上的Web服务来生成一些基本的脚手架。您可以直接使用Web服务，下载生成的zip，解压缩，导入，等等。使用STS向导，只需单击一个按钮即可完成所有这些操作，并确保正确配置了项目，以便您可以立即开始编码。

单击完成按钮后，工作空间将如下所示：

的HelloBootApplication由start.spring.io生成的Java主类是当前应用程序中的唯一代码。多亏了spring boot的“魔力”，并且由于我们在依赖项中添加了“ web”启动器，因此这一小段代码已经是功能齐全的Web服务器！它只是还没有任何实际内容。在添加一些内容之前，让我们学习如何运行该应用程序，并验证它是否在该过程中实际运行。

在STS中运行启动应用

向导创建的Spring Boot应用程序有两种形式：“ jar”或“ war”。入门向导让您在“打包”选项中进行选择。spring-boot的一大功能是，您可以轻松创建包含功能齐全的嵌入式Web服务器的独立“ jar”打包项目。运行您的应用程序所需要做的就是运行其Java Main类型，就像您执行其他任何普通Java应用程序一样。这是一个巨大的优势，因为您不必费心设置本地或远程Tomcat服务器，战争包装和部署。如果您真的想“艰难地”做事，您仍然可以选择“战争”包装。但是，实际上没有必要这样做，因为：

  1. 您可以随时将“ jar”应用转换为“ war”应用
  2. Cloud Foundry平台直接支持部署独立的Java应用程序 。

注意：我们不会在这里介绍如何将应用程序部署到Cloud Foundry，但是在本文中，您可以了解更多有关使用Cloud Foundry Eclipse直接从IDE进行操作的信息。

现在，如果您理解了我刚才说的话，那么您可能意识到实际上并不需要STS的任何“特殊”工具即可在本地运行该应用程序。只需单击Java Main类型，然后选择“运行方式>> Java应用程序”，然后瞧。同样，您所有的标准Eclipse Java调试工具都将“正常工作”。但是，STS提供了专用的发射器，该发射器执行的功能基本相同，但是增加了一些有用的功能。因此，让我们改用它。

您的应用程序应该启动，并且您应该在控制台视图中看到一些输出：

您可以在http：// localhost：8080打开本地运行的应用程序。您将获得的只是一个404错误页面，但这完全符合预期，因为我们尚未向我们的应用添加任何真实内容。

现在，我答应的钟声如何？“运行方式>>引导应用程序”几乎是一个普通的Java启动器，但提供了一些其他选项来自定义其创建的启动配置。要查看这些选项，我们需要打开“启动配置编辑器”，可从要么工具栏按钮：

如果您在Eclipse中使用过Java启动配置编辑器，则应该看起来很熟悉。对于引导启动配置，“主要”选项卡略有不同，并包含一些其他内容。我不会讨论所有其他功能，您可以在STS 3.6.4发行说明中找到更多信息。因此，让我们做一些简单的事情，例如，覆盖默认的http端口8080其他的东西8888 。您可能会猜到可以通过设置系统属性来完成。在“纯” Java启动器中，您可以通过命令行参数设置此类属性。但是，您可能想知道，该属性的名称究竟是“ spring.port”，“ http.port”，“ spring.server.port”吗？幸运的是，启动配置编辑器会有所帮助。覆盖属性表提供了一些基本的内容帮助。您只需键入“端口”，它会提出一些建议：

选择server.port增加价值8888在右列中，然后单击“运行”。

如果您完全按照步骤进行操作，则启动可能会立即终止，但会出现以下异常：

Error: Exception thrown by the agent : java.rmi.server.ExportException: Port already in use: 31196; nested exception is: 

    java.net.BindException: Address already in use

这可能有点令人惊讶，因为我们刚刚更改了端口，不是吗？实际上，这里的端口冲突不是来自http端口，而是来自用于启用“ Live Bean Graph支持”的JMX端口（我不会在此Blog文章中讨论此功能，请参阅STS 3.6.4发行说明 ）。

我们可以采取一些措施来避免该错误。我们可以再次打开编辑器并更改JMX端口，也可以禁用“ Live Bean支持”。但在这种情况下，我们可能真的不想运行多个应用程序副本。因此，我们应该在启动新实例之前停止已经运行的实例。由于这是很常见的事情，因此STS提供了工具栏按钮仅用于此目的。单击按钮，正在运行的应用程序将停止并重新启动，并且您刚刚对启动配置所做的更改现在生效。如果成功了，您现在应该有一个404错误页面位于http://localhost:8888代替8080 。（请注意：如果您还没有启动任何功能，则“重新启动”按钮将不起作用，因为它在当前会话的启动历史记录中有效。但是，如果您至少启动过一次应用程序，则可以“重新启动”已经终止的应用程序）

编辑属性文件

从“启动配置”编辑器中覆盖默认属性值对于“快速覆盖”很方便，但是长期依靠它来配置许多属性并管理更复杂的配置可能不是一个好主意。为此，最好在可以提交到SCM的属性文件中管理属性。入门向导已经方便地创建了一个空白application.properties为了我们。

为了帮助您编辑application.properties STS 3.6.4提供了全新的Spring属性编辑器。该编辑器提供了很好的内容辅助和错误检查：

上面的屏幕截图显示了内容辅助和错误检查的一些“混乱”。现在显示的唯一对我们非常简单的“错误页面应用程序”真正有意义的属性是server.port 。尝试更改属性文件中的端口，当您再次运行该应用程序时，该端口将自动被拾取。但是请注意， 启动配置中覆盖的属性优先于application.properties 。因此，您必须取消选中或删除server.port在启动配置中查看属性。

使我们的应用更有趣

让我们的应用程序更加有趣。这是我们要做的：

  1. 创建一个“ Hello”休息服务，该服务返回“问候”消息。
  2. 通过Spring属性使问候消息可配置。
  3. 设置项目，以便用户定义的属性获得良好的编辑器支持。

创建一个简单的Hello Rest服务

要创建其余服务，您可以按照本指南进行操作 。悬停我们正在做的事情更简单，更直接。

继续使用以下代码创建一个控制器类：

package demo;

import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RequestParam;
import org.springframework.web.bind.annotation.RestController;

@RestController
public class HelloController {

	@RequestMapping("/hello")
	public String hello(@RequestParam String name) {
		return "Hello "+name;
	}
}

重新启动（ ）您的应用。网址http://localhost:8888/hello?name=Kris应该返回一条短信“ Hello Kris”。

使问候语可配置

这实际上很容易做到，并且您可能熟悉Spring的@Value注释。但是，使用@Value您将无法获得不错的内容帮助。Spring Properties Editor不会意识到您以这种方式定义的属性。要理解为什么，了解一下Spring Properties Editor如何获取有关已知属性的信息很有用。

从1.2.x版本开始的某些Spring Boot Jars包含特殊的JSON元数据文件，编辑器会在项目的类路径中查找并解析这些文件。这些文件包含有关已知配置属性的信息。如果您稍作研究，可以从STS中找到这些文件。例如，打开“ spring-boot-autoconfigure-1.2.2”。RELEASE.jar”（在“ Maven依赖关系”下）并浏览到“ META-INF / spring-configuration-metadata.json”。您会发现类似的属性server.port被记录在那里。

为了让编辑器可以使用我们自己的用户定义属性，我们必须创建此元数据。幸运的是，只要您使用Spring Boot @ConfigurationProperties定义属性，就可以轻松实现此自动化。因此，定义一个这样的类：

package demo;

import org.springframework.boot.context.properties.ConfigurationProperties;
import org.springframework.stereotype.Component;

@Component
@ConfigurationProperties("hello")
public class HelloProperties {

	/**
	 * Greeting message returned by the Hello Rest service.
	 */
	private String greeting = "Welcome ";

	public String getGreeting() {
		return greeting;
	}

	public void setGreeting(String greeting) {
		this.greeting = greeting;
	}
}

的@ConfigurationProperties("hello")告诉Boot从以下开始获取配置属性hello.并尝试将它们注入到相应的Bean属性中HelloProperties豆。的@Component批注标记了该类，以便Spring Boot会在其上进行扫描，扫描类路径并将其转换为Bean。因此，如果配置文件（或另一个属性源）包含一个属性hello.greeting那么该属性的值将被注入setGreeting我们的HelloProperties豆。

现在，要实际使用此属性，我们需要的是对bean的引用。例如，要自定义其余服务返回的消息，我们可以添加一个@Autowired场到HelloController并称之为getGreeting方法：

@RestController
public class HelloController {

	@Autowired
	HelloProperties props;
	
	@RequestMapping("/hello")
	public String hello(@RequestParam String name) {
		return props.getGreeting()+name;
	}
}

再次重新启动您的应用，然后尝试访问http://localhost:8888/hello?name=yourname 。您应该获得默认的“ Welcome yourname”消息。

现在继续尝试编辑application.properties并将问候语更改为其他内容。尽管我们已经具备在运行时正确定义属性的所有条件，但是您会注意到编辑器仍未意识到我们新创建的属性：

使编辑者知道的仍然缺少的是spring-configuration-metadata.json文件。该文件是在构建时由spring-boot-configuration-processor这是Java注释处理器。我们必须将此处理器添加到我们的项目中，并确保在项目构建期间执行该处理器。

将此添加到pom.xml ：

<dependency>
	<groupId>org.springframework.boot</groupId>
	<artifactId>spring-boot-configuration-processor</artifactId>
</dependency>

然后执行“ Maven >>更新项目”以触发项目配置更新。STS提供的Maven项目配置器将配置JDT APT并激活用于Eclipse构建的处理器。该警告将立即从编辑器中消失。您还将获得适当的悬停信息：

注释处理器已经激活，以后对您的任何更改HelloProperties类将触发json元数据的自动更新。您可以通过添加一些额外的属性或重命名您的属性来进行尝试greeting财产到别的东西。警告将适当出现/消失。如果您好奇元数据文件在哪里，可以在以下位置找到它target/classes/META-INF 。该文件在那里，即使Eclipse会尽力将其隐藏起来。Eclipse使用项目输出文件夹中的所有文件来执行此操作。您可以通过使用Navigator该视图不会过多地过滤文件，并且会向您显示有关工作空间中实际资源的更直接视图。通过“窗口>>显示视图>>其他>>导航器”打开此视图：

注意：我们知道手动添加处理器似乎是不必要的复杂操作。我们计划在将来进一步实现自动化 。
