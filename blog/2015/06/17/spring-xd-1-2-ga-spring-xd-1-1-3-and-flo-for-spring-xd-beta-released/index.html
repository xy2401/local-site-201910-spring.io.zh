发布了Spring XD 1.2 GA，Spring XD 1.1.3和适用于Spring XD Beta的Flo。

发布
马克·波拉克
2015年6月17日

今天，我们很高兴地宣布Spring XD 1.2，Spring XD 1.1.3的全面可用性以及针对Spring XD Beta的Flo的发布。

  • 1.2.0。GA： 拉链
  • 1.1.3。发布： zip
  • Spring XD Beta的Flo

您还可以使用brew和rpm安装XD 1.2

1.2版包含许多新功能和改进。发行过程是一个多事的旅程，主要是由于Spring XD在众多不同的组中非常受欢迎，每个组都有各自的请求优先级。但是，Spring XD团队迎接了挑战，回顾并回顾为满足我们对简化大数据复杂性的承诺而交付的创新数量，这是非常有益的。

这是过去三个月我们一直在忙的事情的摘要，以及为社区和我们的客户创造的价值。

用于Spring XD和UI改进的Flo

Flo for Spring XD是在Spring XD运行时之上运行的HTML5 canvas应用程序，提供了用于创建，管理和监视流数据管道的图形界面。这是一个简短的截屏视频，向您显示如何构建高级流定义。


您可以浏览文档以获取其他信息，并链接到实际使用的Flo的其他屏幕截图。

XD管理员屏幕还包括一个新的“分析”部分 ，可让您轻松查看量规，计数器，字段值计数器和聚合计数器。

性能改进

预期高吞吐量和低延迟的IoT需求会不断增加，我们在底层消息总线实现中进行了一些性能优化，以每秒每秒数百万条消息的速度在使用Kafka作为传输的Spring XD容器之间传输。通过这些优化，我们现在可以与Kafka自己的测试工具的性能相提并论。但是，我们使用功能更丰富的Spring Integration Kafka客户端，而不是Kafka的高级消费者库。

对于有兴趣复制这些数字的任何人，请参考XD基准测试博客 ，该博客详细描述了所执行的测试和所使用的基础结构。

Apache Ambari和Pivotal HD

为了帮助在ApacheHadoop®集群上自动化Spring XD部署，我们为Spring XD添加了ApacheAmbari®插件 。 Pivotal HD 3.0和Hortonworks HDP 2.2发行版均支持该插件。我们还在Spring XD中增加了对Pivotal HD 3.0的支持 ，从而使受支持的Hadoop版本总数达到了五个。

新的源，处理器，接收器和批处理作业

Spring XD最大的价值主张之一就是其完整的现成的数据连接适配器集，可用于创建实时和基于批处理的数据管道，并且这些管道几乎不需要用户代码即可使用-情况。在社区的帮助下，我们现在将MongoDB ， VideCap和FTP作为源模块，将XSLT转换器处理器和FTP接收器模块。XD团队还开发了Cassandra接收器和语言检测处理器 。

认识到在Pivotal大数据产品组合中的重要作用，我们还通过gpfdist接收器添加了与Pivotal Greenplum数据库和Pivotal HAWQ的本机集成，以实现实时流传输，并且还支持基于gpload的批处理作业 。

除了我们开发人员的生产力主题以及将Spring XD用于大量数据提取用例的生产中，我们很高兴认识到一直在运行Spring的Simon Tao和Yu Cao（CTO和Labs中国的EMC办事处） XD数据管道自2014年起投入生产，也为VideCap源模块做出了贡献。它们的用例和实现细节（用自己的话）在下面。

“对于视频监控行业，从大量非结构化视频流中提取见解存在着巨大的需求。在由数据科学家分析之前，首先需要摄取视频监视数据。为了应对这一挑战，我们使用Spring XD构建了高度可扩展和可扩展的视频数据摄取平台。该平台已准备就绪，可以将各种视频源摄取到集中的Big Data Lake中。鉴于Spring XD具有开箱即用的功能，该平台旨在支持丰富的视频内容处理功能，例如视频转码和对象检测等。

该平台还支持各种类型的视频源-数据处理器和数据导出目的地（例如HDFS，Gemfire XD和Spark），它们是在Spring XD中作为自定义模块构建的，具有高度可重用性和可组合性。使用声明性DSL，视频摄取流将由定义为模块的有向非循环图的视频摄取管道处理。该管道设计为部署在群集环境中，上游模块通过消息总线将数据高效地传输到下游模块。Spring-XD分布式运行时允许管道中的每个模块具有多个实例，这些实例在不同节点上并行运行。通过水平扩展，我们的系统能够支持具有大量视频数据和复杂数据处理工作负载的大规模视频监控部署。”

自定义模块注册表和HA支持

尽管我们可以灵活地配置共享网络位置以用于自定义模块的分布式可用性（通过xd.customModule.home），但我们也认识到在故障情况下使模块注册表具有弹性的重要性-因此，我们拥有HDFS支持模块注册表。具有这种用于生产部署的设置，可以根据业务需求提供一致的自定义模块位可用性和选择的灵活性。

Pivotal Cloud Foundry集成

进一步推动了Pivotal Cloud Foundry集成工作，我们对Spring XD运行时进行了一些基础级别的更改，因此我们能够在Lattice和Diego中将Spring XD模块作为云原生应用程序运行。我们有雄心勃勃的路线图，计划在迭戈本身上启动Spring XD。在研究Diego的Receptor API （用Go！编写）时，我们创建了Java Receptor API ，现在将其提议给Cloud Foundry进行孵化。

下一步

我们即将出现一些非常有趣的发展。也许最重要的是，我们将启动专注于消息驱动和面向批处理的“数据微服务”的新项目。这些将分别直接在Spring Boot以及Spring Integration和Spring Batch上构建。我们的主要目标是为创建以云为本机，以数据为中心的微服务应用程序提供最简单的开发人员体验。反过来，Spring XD 2.0将被重构为这些项目之上的一层，以支持将这些数据微服务组合成流和作业以及它今天提供的所有“即服务”方面，但是它将具有主要致力于向Cloud Foundry和Lattice的部署。我们将很快在这些新项目上发布更多信息，敬请期待！
