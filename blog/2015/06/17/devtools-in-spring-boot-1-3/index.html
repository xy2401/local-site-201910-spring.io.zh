Spring Boot 1.3中的DevTools

工程
菲尔·韦伯
2015年6月17日

Spring Boot 1.3将附带一个名为的全新模块spring-boot-devtools 。该模块的目的是尝试和改进在Spring Boot应用程序上工作时的开发时间体验。

要使用该模块，您只需要将其作为依赖项添加到您的Maven POM中：


    
        org.springframework.boot
        spring-boot-devtools

或您的Gradle构建文件：

dependencies {
    compile("org.springframework.boot:spring-boot-devtools")
}

一旦包含， spring-boot-devtools模块提供了许多不错的功能，我们将在下面进行介绍（如果您不介意阅读文本，请跳到帖子的末尾以观看简短的视频）。

属性默认值

如果您在Spring Boot 1.2中使用了模板技术（例如Thymeleaf），则可能对诸如spring.thymeleaf.cache 。这些属性用于禁用缓存，并允许您更新页面而无需重新启动应用程序。拥有对这些属性的支持非常方便，但是记住在开发过程中进行设置一直很麻烦。

现在，当您使用spring-boot-devtools模块，您不再需要记住设置属性。在开发Thymeleaf的缓存期间，Freemarker，Groovy模板，Velocity和Moustache都将自动禁用。

自动重启

您过去可能曾经使用过诸如JRebel或Spring Loaded之类的工具来为Java应用程序提供即时重载。这些工具很棒，但是它们通常需要额外的配置或IDE插件才能工作（其中有些甚至要花钱！）

使用Spring Boot 1.3，我们一直在进行一些比这些“即时重新加载”技术慢一些的工作，而是通过重新启动应用程序来工作。当你有spring-boot-devtools包含模块，任何类路径文件更改将自动触发应用程序重新启动。我们采取了一些技巧来尝试保持快速重启，因此对于许多微服务风格的应用程序来说，这种技术可能已经足够了。

LiveReload

有了合理的“缓存属性”和“自动重启”功能，每次更改开始变得有些乏味时，都需要手动单击浏览器刷新按钮。为了帮助您节省鼠标按钮，Spring Boot 1.3 DevTools包含一个嵌入式LiveReload服务器。LiveReload是一个简单的协议，允许您的应用程序在发生任何变化时自动触发浏览器刷新。可从livereload.com免费获得适用于Chrome，Firefox和Safari的浏览器扩展。

远程调试隧道

如果您曾经尝试使用Docker托管Java应用程序，或者曾经尝试过Lattice之类的微型PaaS，则可能会对调试代码的难度感到沮丧。您需要配置Java才能开始-Xdebug并以某种方式转发适当的端口，以便可以连接远程调试器。

为了解决这个问题，Spring Boot 1.3可以通过HTTP将JDWP（Java调试线协议）直接隧道传输到您的应用程序。这甚至可以与仅公开端口的部署到Internet Cloud提供商的应用程序一起使用80和443 （尽管由于JDWP是一个健谈协议，所以它可能很慢）。

远程更新和重启

DevTools提供的最后一个技巧是支持远程应用程序更新和重新启动。通过监视本地类路径中的文件更改，然后将其推送到远程服务器，然后重新启动，可以起作用。与本地重新启动一样，您也可以将此功能与LiveReload结合使用。
