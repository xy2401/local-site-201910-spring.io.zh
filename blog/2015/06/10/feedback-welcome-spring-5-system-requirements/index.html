欢迎反馈：Spring 5系统要求

工程
于尔根·霍勒（Juergen Hoeller）
2015年6月10日

正如您可能从我的Java EE 7博客文章中所了解的那样 ，我们正在计划在2016年实现可用性的Spring Framework 5.0版本。由于我们的主要主题之一是对JDK 9的全面支持，因此我们将跟踪JDK 9的候选版本。

Spring 5的功能计划仍在制定中。我们今年将在SpringOne上提出更深入的计划，敬请期待！不过，我想借此机会与您联系，以获取有关我们预期的系统要求的反馈：

我们一定会将最低要求提高到JDK 8+。
这是前提条件，因为它使我们能够在整个框架中拥有一个更干净的代码库，在我们的核心接口中引入默认的方法实现，并在我们的核心抽象中依赖于JDK 8 API类型（例如，CompletableFuture，java.util.function接口） 。

我们还打算软升级EE基准。
现在，这有点棘手，因为我们在这里实际上有各自的要求-并且我们需要考虑生产环境中的企业采用级别：

  • 我们肯定会提高到Servlet 3.0+（从我们目前的Servlet 2.5运行时兼容性开始），但是不会更高，因为我们希望Spring 5应用程序仍可以在EE 6基准服务器上运行。鉴于Java EE 7的市场情况以及仍然基于Servlet 3.0 API的众多服务器，请参阅我的上一篇博客文章 ，讨论为什么这是不可避免的原因。
  • 我们将保持与JMS 1.1+的兼容性，因为除了EE 7问题外，我们还希望企业界中的消息代理没有必要升级到JMS 2.0。Spring的JMS支持无论如何都会自动适应JMS 2.0，因此不应缺少任何功能。我们不得不继续支持2002年代的JMS 1.1 API，这只是可耻的……
  • 我们想提高到JPA 2.1+和Bean Validation 1.1+，但我们的双手似乎束手无策：TomEE 1.7和JBoss EAP 6.4具有硬JPA 2.0和Bean Validation 1.0 API，而WebLogic 12.1.3具有JPA 2.1，但没有Bean验证1.1 API（尽管它们是相关的）。
  • 这意味着我们将必须不断检测JPA 2.1 / BV 1.1，并自动对其进行调整-否则，我们将需要本地捆绑JPA 2.1 / BV 1.1 API jar和相应的提供程序。可能的结果是，我们将简化针对JPA 2.1的设置，只是在运行时通过回退检查来容忍JPA 2.0，类似于我们目前处理Servlet 3.0与2.5的方式。

从您的角度来看，上述假设是否可行？您是否有将EE 6服务器分别升级到JPA 2.1 / Bean Validation 1.1的经验，例如通过本地交付应用程序中相应的API和提供程序？欢迎任何反馈！

请注意：
如果您坚持使用JDK 6或7，则无需担心：我们还计划在一般Spring 4系统要求之内，于明年初推出Spring Framework 4.3。但是，这将是该行中的最后一个功能发布，随后Spring Framework 4.x进入简单的维护阶段。
