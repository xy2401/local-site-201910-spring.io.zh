Java EE 7，第二个生日快乐！生产过程如何？

新闻与活动
于尔根·霍勒（Juergen Hoeller）
2015年6月4日

请注意，有一篇有关Spring 5系统需求的后续博客文章。如果您主要对Spring 5计划过程感兴趣，那么您可能想从这里开始。

在寻求Java EE集成的过程中，我们试图积极采用最新一代的规范，例如JPA，Bean验证以及Servlet和JMS API。从Spring 4开始，我们将并排支持Java EE 6和7级别的规范 。我们希望将其提升到EE 7+级别（JPA 2.1，Bean验证1.1，尤其是Servlet 3.1和JMS 2.0），但是面临一个根本性的问题： 缺乏EE 7平台的采用。

Java EE 7平台已于2013年5月发布，因此已经存在两年了。令人惊讶的是，几乎没有在生产中发现它。但这确实并不令人感到意外：同时，尽管有几个项目已通过EE 7认证 ，但主要供应商的匮乏显而易见：目前尚无主要的EE 7服务器具有生产支持，而不是针对Web配置文件和针对完整的平台。截至2015年6月，普通EE供应商仍在销售基于2009年代Java EE 6 API的服务器许可证。这不仅是传统的犯罪嫌疑人：

  • Tomitribe提供对TomEE 1.7的支持， TomEE 1.7是基于Tomcat 7的EE 6 Web Profile堆栈，基于2009年代EE 6 API，但至少与JSR-356 WebSockets和JDK 8兼容。TomEE 7.0（最近从TomEE 2.0重命名）将是基于Tomcat 8的Java EE 7 Web Profile产品，但尚未发布（截至2015年6月）。
  • 红帽的最新支持产品是JBoss EAP 6.4 ，它是基于JBoss 7的EE 6堆栈，具有JSR-356 WebSockets和JDK 8支持。请注意，从Red Hat的角度来看，WildFly是一个R＆D项目：它带有频繁的功能发布（为此而致意），但不幸的是，没有维护版本或任何生产支持。
  • Oracle随附了WebLogic 12.1.3 ，这是EE 6服务器，带有一点EE 7（JPA 2.1，JAX-RS 2.0，JSR-356 WebSockets）和JDK 8支持。尚未发布完整的EE 7服务器的WebLogic 12.2.1（之前宣布为12.1.4一段时间）（截至2015年6月）。从Oracle的角度来看，GlassFish 4只是一个RI，使用WebLogic作为生产升级路径。
  • IBM WebSphere团队在实现WebSphere Liberty Profile （WAS 8.5.5）的EE 7规范方面做得很好，但尚未完成工作。至少作为生产Liberty Profile的一种功能包，已经在生产中支持了某些EE 7模块和JDK 8，并且WebSphere的完整Java EE 7认证很可能会在2015年实现。
    热门新闻（6月9日）：适用于WAS 8.5.5的EE 7修订包将于6月26日正式发布 。恭喜，IBM！

尽管EE 7的一些规范已被个别采用，例如通过Hibernate 4.3的JPA 2.1和通过Tomcat 8和Jetty 9的Servlet 3.1 / JSR-356 WebSockets ，但可以说Java EE 7未能作为标准进入市场。平台整体。毕竟，“平台”的重点是广泛的主流可用性。具有讽刺意味的是，即使在EE领域，后来发布的JDK 8（2014年3月）也很快投入生产！因此，截至2015年中的最新水平是在生产环境中的JDK 8上运行的，由供应商支持的Java EE 6服务器。

我们的结果：考虑到Spring 4和Java 8的采用水平，我们将把Spring Framework 5的最低要求提高到JDK 8+。但是，由于缺乏Java EE 7平台的采用，我们必须保留与当前应用服务器的兼容性：允许将即将发布的Spring 5应用部署到生产中常见的基于JDK 8的EE 6服务器上-就像我们已经使用Spring 4一样，但是至少有了将JDK 8+用于我们的框架代码库及其所有核心接口的额外好处。

PS（6月6日）：

FWIW，我非常感谢GlassFish和WildFly作为开源工程的努力。Spring对此提供了专门的支持， Undertow HTTP服务器（在WildFly的保护下）非常适合使用Spring Boot进行嵌入式部署。这并没有改变项目所有者（分别为Oracle和Red Hat）不支持他们的事实，而是选择出于生产目的而投资WebLogic 12和JBoss EAP 6。Payara之类的外部支持（适用于GlassFish）只能在某种程度上缓解这种情况，2015年，Java EE市场的很大一部分将与供应商生产产品（全部基于EE 6）绑定在一起。

有关开放源代码项目本身提供优质生产支持的示例，请参见Tomcat 。Tomcat项目在修复错误以及特别是安全漏洞方面拥有非常出色的记录，即使在服务器的前三代中也是如此。因此，我并不是在争取商业支持本身，而是在争取适当的维护版本，例如Tomcat一样（而且我敢说：像Spring一样），无论是来自开源项目本身还是来自商业支持订阅。例如，WildFly没有任何一个。 GlassFish没有Oracle的支持，但至少通过Payara具有支持选项。
