STS中的Spring Boot仪表板-第3部分：重新加载了Spring Boot Devtools

工程
马丁·利珀特
2015年10月22日

欢迎回到Spring社区，

有关Spring Tool Suite中新的Spring Boot Dashboard的系列文章的最后一部分。在最后一部分中，我们将更深入地研究将Spring Boot Devtools与启动仪表板结合使用。同样，此功能是Spring Boot 1.3的新增功能，因此您必须使用该版本才能使用以下功能。

快速部署代码更改

对于本地应用程序，使用Spring Boot Devtools非常简单明了。一旦将Spring Boot Devtools作为依赖项添加到项目中（上下文菜单的Spring类别中有一个简单的菜单选项）并启动您的应用程序，它将监听配置文件和类文件的本地更改，并自动为您启动应用程序的重启。您甚至不需要自己重新启动应用程序，Spring Boot Devtools会自动为您完成此操作。由于每当您保存文件时STS / Eclipse都会生成和更新类文件，因此在STS中工作时，所有这些都会自动发生。

如果您在诸如Cloud Foundry之类的远程运行时上运行Spring Boot应用程序，那么这个故事将变得更加有趣。原则上，您也可以在这样的远程设置中使用Spring Boot Devtools，但这需要更多的工作。好消息是，Spring Boot仪表板可以帮助您实现这一目标。

一旦您在Cloud Foundry上部署或重新启动（并因此更新）了Spring Boot应用程序（使用启动仪表板），并且其类路径上具有Spring Boot Devtools，启动仪表板将在CF上配置启动应用程序以远程使用devtools自动。这主要包括远程机密的设置-允许远程devtools访问它。

一旦启用了devtools的启动应用程序在Cloud Foundry上运行，您就可以轻松启动该启动器的启动devtools远程客户端应用程序。该客户端应用程序在您的计算机上本地运行，并连接到Cloud Foundry上的远程应用程序。它监视文件更改，将其上载到Cloud Foundry上的应用程序，并触发Cloud Foundry上的应用程序重启。

由于远程客户端应用程序将监视本地计算机上项目中的文件更改，因此您可以照常继续在IDE中工作。更改的文件将由远程客户端应用程序自动更新为该应用程序的Cloud Foundry版本。一旦将这些更改的文件存储到Cloud Foundry上的应用程序，Cloud Foundry上的对应方将重新启动引导应用程序。这适用于新编译的源代码以及更改的资源文件。

即使您将项目部署到远程云运行时，这也可以使您在处理项目时获得快速的周转周期。

在云中调试

除了自动重启功能之外，Spring Boot Devtools还可以在云运行时上对远程应用程序进行全面调试。STS中的Spring Boot Dashboard也支持此功能。如果您的应用程序的类路径上有Spring Boot Devtools，则可以按该应用程序的Cloud Foundry实例的（重新）调试按钮。这将以调试模式在Cloud Foundry上重新启动应用程序，自动在您的计算机上启动相应的远程客户端应用程序，然后将Eclipse调试器连接到该应用程序。

您可以以与本地应用程序相同的方式调试在云运行时上运行的应用程序，包括设置断点，检查变量，甚至进行热交换代码。但是，自动重启功能不适用于在调试模式下运行的应用程序（由于当前boot devtools的技术限制）。

如果您尝试在Cloud Foundry上对应用程序进行远程调试，则会发现调试速度明显下降。这是由于JVM的远程调试协议通过HTTP连接的隧道传输引起的，而远程调试协议并不是为该协议而设计的。我们正在通过使用其他运输机制来改善这一状况。但是您应该很少在Cloud Foundry上对应用程序进行远程调试。对于更频繁的调试，您可能需要考虑在本地运行和调试应用程序，并通过本系列博客上一部分中介绍的ngrok隧道功能将其与其他应用程序服务连接。

结论

结束于Spring Boot Dashboard上的博客系列，这是自Spring Tool Suite 3.7.1版本以来的一项新功能。让我们知道您使用仪表板的经历。
