What we actually got into this first release:

  • Easy to use flat one level state machine for simple use cases.
  • Hierarchical state machine structure to ease complex state configuration.
  • State machine regions to provide even more complex state configurations.
  • Usage of triggers, transitions, guards and actions.
  • Type safe configuration adapter.
  • Builder pattern for easy instantiation for use outside of Spring Application context
  • Recipes for usual use cases
  • Distributed state machine based on a Zookeeper
  • State machine event listeners.
  • Spring IOC integration to associate beans with a state machine.

Spring Statemachine 1.0.0发布

发布
珍妮·瓦尔基亚拉赫蒂
十月13，2015

我们很高兴地宣布发布Spring Statemachine 1.0.0 。首先，我要感谢所有为实现这一目标而做出任何贡献的人。可以从Maven Central或从Spring Repository获得工件。

我们在第一个版本中实际得到的是：

  • 易于使用的单层状态机，用于简单的用例。
  • 分层状态机结构可简化复杂的状态配置。
  • 状态机区域提供甚至更复杂的状态配置。
  • 触发器，过渡，防护和操作的使用。
  • 键入安全配置适配器。
  • 用于在Spring Application上下文之外使用的易于实例化的Builder模式
  • 常用案例食谱
  • 基于Zookeeper的分布式状态机
  • 状态机事件监听器。
  • Spring IOC集成，将Bean与状态机关联。

让我们快速回顾一下该项目的诞生方式以及它从最初的github导入到发行版的演变过程。这也为新的春季项目如何诞生或可能如何诞生提供了一些背景知识。该项目是在今年年初从Spring Hadoop的基础工作开始的。

对于在达拉斯举行的SpringOne 2014，我们正在将新的容器分组功能推入Spring YARN中，该功能在YARN容器之上添加了更高级别的功能。如今，Spring XD的YARN运行时和新的Spring Cloud Dataflow YARN部署程序都基于此。与Hadoop YARN资源管理器的通信本质上是异步的，因此，在不使用适当的状态机概念的情况下尝试实现代码库的这一特定部分时，我最终遇到了很多麻烦。当我说我尝试过时，请相信我，我非常努力地尝试不使用适当的状态，大约一个星期后，我不得不面对现实，并面对一个事实，即我试图对自己的代码进行过度编码。我最终得到了一些可以正常工作的东西，但是如果我触摸了该代码的任何部分，地狱就会崩溃。我整理了所有内容，对自己说：“该死的詹妮，我需要一台状态机”。

在完成状态机的基本实现之后，我的所有问题都消失了，仅仅是因为状态机现在控制着必须按特定顺序发生的所有逻辑，而与YARN资源管理器的所有通信仍然异步发生。是的，现在我们已经进入发布阶段，它为我们提供了一个用此版本替换Spring YARN内部状态机的选项。

想法诞生了，将特定的状态机代码派生到其自己的项目中，对其进行一些增强，然后启动一个新的Spring Project，看看它是否具有吸引力，这可能是一个非常好的主意。如果说实话，我对使用这项已有50年历史的这项技术产生了多少兴趣感到有些惊讶。好的和扎实的概念不会死，也不需要死！

关于从构想到发布的过程中我所面临的挑战，我只字未提：

  • 没有嵌套状态和实现挂钩（即监听器）的简单状态机非常易于实现。
  • 引入深层嵌套状态会使事情变得有些复杂，尤其是当您加入其他各种功能（例如进入/退出操作，不同的过渡类型和防护）时。对于具有不同父状态的子状态之间的转换，我遇到了很多麻烦。
  • 添加正交区域会使事情变得更加复杂，这仅仅是因为理论上状态机是单线程的，但是区域是独立的并且可以并行执行。为独立执行区域提供了基础支持而进行了大量工作。
  • 基于Zookeeper（通过SPI抽象）的分布式状态机是一件疯狂的事情。有些人甚至不会尝试，测试是一场噩梦（无法通过简单的单元测试来完成）。我用了Aphyr's Jepsen框架为此（如果说得通一点的话）把事情弄成碎片，但是当您退出单个JVM并开始使用分布式JVM时，它最终使我能够发现各种错误。这是一个痛苦而有趣的旅程。

很多人问我们是否有路线图？简短的回答是是和不是。是的，因为我们肯定有很多东西要实施，而不是因为项目是由社区的要求推动的。我开始遵循UML状态机规范的基本功能，但最终约有50％的额外功能是用户要求的。UML规范对其某些规范功能非常含糊，并且在实现过程中留下了很多细节。如果您需要什么，请说出并转到GitHub Issues 。想要贡献，PR的东西（即使是一个简单的拼写错误，也非常感激！）。

到目前为止我们所知道的：

  • 1.0.x版不会添加任何会改变核心概念的新功能，但会继续添加常规的维护/错误修复。我们可能会添加新功能，以增强现有概念。
  • 1.1.x版将侧重于安全性和执行模型等主题。我们希望与Spring Security ， Spring Session确保状态机动作。为了执行，我们希望替换或给Reactor一个镜头，以支持正常的Framework任务调度/执行。
  • 2.x版本（在可预见的将来）将通过Spring Framework 5和JDK8提高基线。1.x系列将保持Spring Framework 4和JDK7的基线。

试试看，感受一下，嗅探一下，让我们知道您的想法！
