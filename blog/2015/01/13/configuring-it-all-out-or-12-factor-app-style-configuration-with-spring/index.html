“全部配置”或“使用Spring的12要素App样式配置”

工程
乔什·朗（Josh Long）
2015年1月13日

在开始之前，让我们建立一些词汇表。当我们谈论Spring中的配置时，我们通常是在谈论Spring Framework 的各种输入。 ApplicationContext帮助容器了解您要完成的操作的实现。这可能是要送入ClassPathXmlApplicationContext ，或Java类以某种方式注释了AnnotationConfigApplicationContext 。

就像在12要素应用程序清单中很好地描述的那样 ，另一种配置类型是可能在部署之间（阶段，生产，开发人员环境等）不同的任何应用程序配置，例如服务凭证和主机名。

第二种类型的配置应该驻留在已部署的应用程序外部，自从PropertyPlaceholderConfigurer课程介绍。从那时起，Spring对这种类型的配置的支持已经走了很长一段路，在本博客中，我们将研究这种进展。

的PropertyPlaceholderConfigurer

春天的提供PropertyPlaceholderConfigurer自2003年以来。Spring 2.5引入了XML名称空间支持，并随之提供了XML名称空间对属性占位符解析的支持。例如可以让我们用XML配置中的bean定义文字值替换分配给（外部）属性文件中的键的值（在这种情况下， simple.properties可以在类路径上，也可以在应用程序外部）。该属性文件可能如下所示：

# Database Credentials
configuration.projectName = Spring Framework

的Environment抽象化

该解决方案在3.0之前将Java配置引入Spring Framework之前。Spring 3使您可以轻松地使用Java将配置值注入Java组件配置中@Value -注释，如下所示：

@Value("${configuration.projectName}") 
private String projectName;

Spring 3.1引入了Environment抽象。它在运行中的应用程序与运行环境之间提供了一些运行时间接性。的Environment充当键和值的映射。您可以通过贡献a来配置从中读取这些值的位置。注入类型的对象Environment任何您想问的地方。默认情况下，Spring加载系统环境键和值，例如line.separator 。您可以告诉Spring从文件中加载配置密钥，特别是使用@PropertySource注解。

package env;

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.context.annotation.*;
import org.springframework.context.support.*;
import org.springframework.core.env.Environment;

@Configuration
@ComponentScan
@PropertySource("file:/path/to/simple.properties")
public class Application {

	@Bean
	static PropertySourcesPlaceholderConfigurer placeholderConfigurer() {
		return new PropertySourcesPlaceholderConfigurer();
	}

	@Value("${configuration.projectName}")
	void setProjectName(String projectName) {
		System.out.println("setting project name: " + projectName);
	}

	@Autowired
	void setEnvironment(Environment env) {
		System.out.println("setting environment: " + 
                      env.getProperty("configuration.projectName"));
	}

	public static void main(String args[]) throws Throwable {
		new AnnotationConfigApplicationContext(Application.class);
	}
}

本示例从文件加载值， simple.properties ，然后具有一个值， configuration.projectName ，使用@Value注释，然后再从Spring的Environment抽象。为了能够使用@Value注释，我们需要注册一个PropertySourcesPlaceholderConfigurer 。在这种情况下，输出为Spring Framework 。

的Environment还带来了配置文件的想法。它使您可以将标签（配置文件）分配给Bean分组。使用概要文件来描述从一种环境变为另一种环境的bean和bean图形。您可以一次激活一个或多个配置文件。未为其分配概要文件的Bean始终被激活。具有配置文件的豆default仅在没有其他配置文件处于活动状态时激活。

通过概要文件，您可以描述在一个环境与另一个环境中需要以不同方式创建的Bean集。例如，您可能使用嵌入式H2 javax.sql.DataSource在您当地dev个人资料，但随后切换到javax.sql.DataSource通过JNDI查找或通过从Cloud Foundry中的环境变量读取属性来解析的PostgreSQL时， prod个人资料已激活。在这两种情况下，您的代码都有效： javax.sql.DataSource ，但有关使用哪个专门实例的决定由一个或多个活动配置文件决定。

您应谨慎使用此功能。理想情况下，一个环境与另一个环境之间的对象图应保持相当固定。

引导配置

Spring Boot大大改善了性能。Spring Boot将读取以下内容src/main/resources/application.properties默认。如果配置文件处于活动状态 ，它还将根据配置文件名称自动读取配置文件，例如src/main/resources/application-foo.properties哪里foo是当前配置文件。如果Snake YML库位于类路径中，则它还将自动加载YML文件。是的，再读一遍。YML非常好，值得一游！这是一个示例YML文件：

configuration:
	projectName : Spring Boot
	someOtherKey : Some Other Value

在一般情况下，Spring Boot还使获得正确结果变得更加容易。它使-D的论点java过程和环境变量可用作属性。它甚至将它们标准化，因此环境变量$CONFIGURATION_PROJECTNAME或-D形式的论点-Dconfiguration.projectname都可以通过钥匙访问configuration.projectName 。

配置值是字符串，如果您有足够的配置值，尝试确保这些键本身不会成为代码中的魔术字符串可能很麻烦。Spring Boot引入了@ConfigurationProperties组件类型。使用以下注释POJO @ConfigurationProperties并指定一个前缀，Spring将尝试将以该前缀开头的所有属性映射到POJO的属性。在下面的示例中， configuration.projectName将被映射到POJO的实例，所有代码随后可以注入该实例并取消引用以读取（类型安全）值。这样，您只能将一个键的映射放在一个位置。

在下面的示例中，属性将从以下位置自动解析src/main/resources/application.yml 。

package boot;

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.boot.context.properties.ConfigurationProperties;
import org.springframework.stereotype.Component;

// reads a value from src/main/resources/application.properties first
// but would also read:
//  java -Dconfiguration.projectName=..
//  export CONFIGURATION_PROJECTNAME=..

@SpringBootApplication
public class Application {

	@Autowired
	void setConfigurationProjectProperties(ConfigurationProjectProperties cp) {
		System.out.println("configurationProjectProperties.projectName = " + cp.getProjectName());
	}

	public static void main(String[] args) {
		SpringApplication.run(Application.class);
	}
}

@Component
@ConfigurationProperties("configuration")
class ConfigurationProjectProperties {

	private String projectName;

	public String getProjectName() {
		return projectName;
	}

	public void setProjectName(String projectName) {
		this.projectName = projectName;
	}
}

Spring Boot使用@ConfigurationProps大量机制让用户覆盖系统位。您可以查看哪些属性键可用于更改内容，例如，通过添加org.springframework.boot:spring-boot-starter-actuator依赖于基于Spring Boot的Web应用程序，然后访问http://127.0.0.1:8080/configprops 。这将基于运行时类路径上显示的类型为您提供受支持的配置属性的列表。当您添加更多Spring Boot类型时，您将看到更多属性。

带有Spring Cloud配置支持的集中式日志配置

到目前为止，还不错，但是到目前为止，该方法还存在一些不足：

  • 更改应用程序的配置需要重新启动
  • 没有可追溯性：我们如何确定在生产中引入了哪些更改，并在必要时进行回滚？
  • 配置是分散式的，现在尚不清楚去哪里更改。
  • 有时为了安全起见，应该对配置值进行加密和解密。对此没有现成的支持。

Spring Cloud建立在Spring Boot的基础上，并集成了用于处理微服务的各种工具和库，包括Netflix OSS堆栈 ，它提供了配置服务器和该配置服务器的客户端。这种支持共同解决了最后三个问题。

让我们看一个简单的例子。首先，我们将设置配置服务器。配置服务器是基于Spring Cloud的一组应用程序或微服务之间共享的东西。您必须在某个地方使它运行一次。然后，所有其他服务仅需要知道在哪里可以找到配置服务。配置服务充当其从在线或磁盘上的Git存储库中读取的配置密钥和值的代理。

package cloud.server;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.cloud.config.server.EnableConfigServer;

@SpringBootApplication
@EnableConfigServer
public class Application {

	public static void main(String[] args) {
		SpringApplication.run(Application.class, args);
	}
}

如果您正确地管理事情，那么与任何服务一起存在的唯一配置应该是告诉配置服务在哪里可以找到Git存储库的配置以及告诉其他客户端服务在哪里可以找到配置服务的配置。

这是配置服务的配置， src/main/resources/application.yml ：

server:
	port: 8888

spring:
	cloud:
		config:
			server:
				git :
					uri: https://github.com/joshlong/microservices-lab-configuration

这告诉Spring Cloud配置服务在我的GitHub帐户的Git存储库中查找单个客户端服务的配置文件。当然，URI可以很容易地成为我本地文件系统上的Git存储库。用于URI的值也可以是属性引用，其形式为${SOME_URI} ，它可能引用了一个称为SOME_URI 。

运行该应用程序，将浏览器指向，即可验证您的配置服务是否正常运行http://localhost:8888/SERVICE/master哪里SERVICE是您的客户服务的ID boostrap.yml 。基于Spring Cloud的服务将查找名为src/main/resources/bootstrap.(properties,yml)它期望找到-您猜对了！ -引导服务。期望在其中找到的一件事bootstrap.yml file是指定为属性的服务的ID， spring.application.name 。这是我们的配置客户端bootstrap.yml ：

spring:
	application:
		name: config-client
		cloud:
			config:
				uri: http://localhost:8888

当Spring Cloud微服务运行时，它将看到它的spring.application.name是config-client 。它将与配置服务联系（我们已经告知Spring Cloud运行在http://localhosst:8080 ，尽管它也可能是环境变量），并要求其进行任何配置。配置服务会返回JSON，其中包含application.(properties,yml)文件以及中的任何特定于服务的配置config-client.(yml,properties) 。它还会加载给定服务和特定配置文件的任何配置，例如， config-client-dev.properties 。

这一切都是自动发生的。在以下示例中，从配置服务中读取配置值。

package cloud.client;

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.cloud.context.config.annotation.RefreshScope;
import org.springframework.core.env.Environment;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RestController;

@SpringBootApplication
public class Application {

	public static void main(String[] args) {
		SpringApplication.run(Application.class, args);
	}

	@Autowired
	void setEnvironment(Environment e) {
		System.out.println(e.getProperty("configuration.projectName"));
	}
}

@RestController
@RefreshScope
class ProjectNameRestController {

	@Value("${configuration.projectName}")
	private String projectName;

	@RequestMapping("/project-name")
	String projectName() {
		return this.projectName;
	}
}

的ProjectNameRestController带有注释@RefreshScope ，这是一个自定义的Spring Cloud 范围 ，它允许任何bean就地重新创建自身（并从配置服务中重新读取配置值）。有多种触发刷新的方法：发送POST请求http://127.0.0.1:8080/refresh （例如： curl -d{} http://127.0.0.1:8080/refresh ），使用自动暴露的JMX刷新端点或使用Spring Cloud Bus。

Spring Cloud Bus通过RabbitMQ供电总线链接所有服务。这是特别强大的。您可以告诉一个（或数千个！）微服务通过将一条消息发送到消息总线来刷新自身。这可以防止停机时间和比不得不重新启动系统的单个服务或节点更加友好。

要查看所有这些操作，请运行配置客户端和配置服务器，并确保将配置服务器指向您可以控制和更改的Git存储库。点击REST端点并确认您看到了Spring Cloud 。然后至少在Git中更改配置文件git commit他们。然后针对配置客户端触发刷新，然后再次访问REST端点。您应该看到更新后的值！

Spring Cloud配置支持还 包括对安全性和加密的一流支持 。我将让您自己探索最后一英里，但这相当琐碎，而且足以配置有效的密钥。

下一步

我们在这里介绍了很多内容 ！有了所有这些，应该很容易打包一个工件，然后将该工件从一个环境移动到另一个环境，而无需更改工件本身。如果您今天要启动一个应用程序，建议您从Spring Boot和Spring Cloud上启动，尤其是现在我们已经查看了默认情况下带给您的所有好处。不要忘记查看所有这些示例背后的代码 。
