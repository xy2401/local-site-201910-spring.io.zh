Spring Batch和Spring Batch管理员版本

发布
迈克尔·米内拉
2015年1月16日

我们很高兴今天宣布许多与Spring Batch相关的版本。Spring Batch的错误修复版本，Spring Batch Admin的错误修复版本以及Spring Batch Admin的新版本的里程碑版本现在都可用。

春季批次3.0.3

Spring Batch 3.0.3代表了Spring Batch的最新维护版本，解决了许多增强功能和较小的错误修复。Spring Batch 3.0.3的新功能包括：

改进了对JSR-352定制基础结构的支持

在此版本之前，为JSR-352配置的作业开箱即用地覆盖Spring Batch提供的基础结构的唯一方法是在作业的上下文中包含配置。这阻止了为真正共享的组件配置全局替代的功能。使用3.0.3，您现在可以通过System属性指定Spring配置的位置，该属性可以覆盖现有基础架构。

用于分区结果的可选轮询

使用远程分区时，主服务器需要一种方法来通知从分区已完成其工作。从历史上看，这是通过每个从属服务器答复主服务器来完成的，其结果被汇总，然后通知主服务器所有从属服务器已完成。但是，这需要不必要的额外配置。由于分区作业中的从属服务器与主服务器在同一个作业存储库中维护其状态，因此主服务器可以轮询作业存储库以查看从属服务器是否完整。此版本增加了配置MessageChannelPartitionHandler轮询作业存储库，而不是等待响应消息。您可以在以下文档中阅读有关此新功能配置的更多信息： MessageChannelPartitionHandler 。

这不是完整的更新列表，而是突出显示主要的新功能。您可以在这里查看Spring Batch的Jira问题跟踪器中的确切更改： https : //jira.spring.io/browse/BATCH/

Spring Batch管理员

1.3.1

我们今天提供的两个Spring Batch Admin版本中的第一个是Spring Batch Admin 1.3系列的第一个维护版本。此版本解决了一些小错误，可在Jira中找到以下小错误： https : //jira.spring.io/browse/BATCHADM/

2.0.0。M1

去年，我在SpringOne2GX上收到的最大问题是“ Spring Batch Admin发生了什么？“ Spring Batch Admin的最新更新是在7月（去年SpringOne2GX不久之前），但这并不是功能上的升级。它被发布以更新依赖关系并解决一些错误。从那时起，我们一直在努力更新许多功能以使Spring Batch Admin保持最新。今天，我们宣布实现该目标的第一个里程碑。

JSR-352支持

Spring Batch Admin，作为2.0.0的一部分。M1版本现在将支持JSR-352配置的作业。通过将基于XML的配置放在/META-INF/batch-jobs在规范要求的目录中，Spring Batch Admin将加载作业，以由REST端点和当前UI启动。 Spring Batch Admin提供的所有监视方面（查看执行，启动/停止/重启等）均可用。

Java配置支持

由于Spring社区内部的工作已经从基于XML的配置转移到基于Java的配置，Spring Batch Admin保持了发展势头。从此版本开始，Spring Batch Admin支持配置软件包以扫描java配置的Spring Batch作业。与其他批处理作业一样，这些作业将被加载并可以执行，就像它们的XML对应作业一样。

重要的是要注意，尽管Spring Batch Admin现在支持基于Java的配置，但您不想使用@EnableBatchProcessing用它。这是有原因的。 @EnableBatchProcessing提供了Spring Batch Admin开箱即用提供的基础架构集合。通过java config配置作业以在Spring Batch Admin中使用与使用之前完全相同@EnableBatchProcessing ...不使用该注释。您仍然可以自动接线JobBuilderFactory和StepBuilderFactory就像平时一样

REST端点更新

作为Spring XD团队在其管理UI中所做的工作的一部分，他们创建了一套全新的与批处理相关的REST端点集。这个版本将这些端点迁移到Spring Batch Admin中供所有人使用。关/batch path包含一组端点，这些端点提供的功能类似于现有的REST API，但是更进一步。新的API遵循HATEOAS原则，允许API发现和遍历。虽然HATEOAS的支持尚在开发中，但此版本提供了有关我们将如何处理此问题的信息。

Spring Batch Admin的未来

与所有Spring项目一样，我们努力使事情尽可能向后兼容。因此，Spring Batch Admin 2.0正在进行一些重大更改，以允许将来向后兼容。这些更改包括删除“官方” UI，以及弃用旧版本的REST API。 此版本不包含任何这些更改。 这些更改将在Spring Batch Admin 2.0普遍可用之前进行。我们希望对项目的方向保持开放和透明。

过去几年中遵循UI空间变化率的任何人都可以很快看出使用现代前端技术并能够在可预见的将来保持向后兼容性的可能性。目前，打破变更的速度实在太大了。考虑到我们希望在相关领域提供最前沿的工具时，我们决定删除UI，将其作为项目的正式组成部分。话虽如此，我们仍然了解到，客户端是Spring Batch Admin带来的重要内容。因此，我们计划提供一些示例项目的集合，以演示几个不同的客户选项。这将使我们能够以不妨碍我们向后兼容的方式独立于核心框架/ API来开发客户端选项。

我们还将弃用旧版本的REST API。新的API在REST API成熟度模型中向前迈出了一步，从基于HTTP的基本CRUD转变为支持真正的资源。尽管新的REST端点仍在开发中，但一旦功能完善，我们将弃用旧的REST端点，以将开发工作集中在改进和发展新的端点上。

我们从这里去

我们将继续进行上述更改，目前计划在今年第二季度初发布另一个版本。我们对Spring Batch Admin的未来感到兴奋，并期待您在Jira ， Github ， StackOverflow和社交媒体上的反馈！
