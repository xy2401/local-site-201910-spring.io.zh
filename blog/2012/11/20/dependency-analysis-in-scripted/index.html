脚本化的依赖性分析

工程
克里斯·德·沃尔德
2012年11月20日

上个月， 在此博客上宣布了脚本脚本，来自VMWare的JavaScript编辑器。在本文中，我们将深入研究Scripted的Dependency Analysis Engine 。但是在深入研究细节之前，先激励我们为什么需要它。

主要动机：跨文件内容辅助


为了提供出色的JavaScript编辑体验，Scripted需要提供有关可在当前编辑器上下文中使用的功能，方法或标识符的准确建议。

[标题ID =“ attachment_12178” align =“ aligncenter” width =“ 533”标题=“跨文件内容辅助”] [/字幕]

两个组件协同工作以实现此目标：


  • 细粒度类型推断分析引擎

  • 粗粒度依赖性分析引擎


推理引擎解析您的代码，并遍历每个声明，语句和表达式。这使得它可以确定在给定上下文中哪些标识符有效，并对可以存储在这些变量中的事物的种类做出很好的猜测。然后，此信息用于提出内容辅助建议。

如果您只想将所有代码放入一个大文件中，那么仅凭一个高质量的推断器就足以提供一些很好的内容帮助。实际上，项目将分为模块。在上面的示例中，“ main”模块导入了“ utils”模块。在编辑主模块时，“脚本化”会根据需要从“ utils”中提出功能。依赖性分析引擎使之成为可能。

依赖性分析的其他用途


脚本化脚本还使用依赖性分析将未解决的依赖性标记为错误。例如，如果我们尝试导入“假”模块，则Scripted将显示错误标记 。

[caption id =” attachment_12180” align =” aligncenter” width =“ 518” caption =”未解决的模块的错误标记”] [/字幕]

脚本化脚本还使用依赖关系分析来支持轻松导航。按住Shift键或按Ctrl键单击已解析的依赖项名称，将带您到相应的文件。

将来，依赖关系分析还可能使我们能够实施重构工具。例如，如果将.js文件拖放到其他目录，则Scripted可以根据需要自动更新相对路径引用。

它有什么作用？


依赖性分析引擎仅向其客户端提供一个getDGraph函数，即类型推断引擎：
getDGraph :: <path-to-js-file> -> <dependency-graph>

此函数计算依赖关系图的JSON表示形式。此图包含目标文件直接或传递依赖的所有文件的节点。如果将“ main”模块传递给它，它将返回类似以下内容的内容：
getDGraph(‘/sample/main.js’) ==>
{
 …
 “/NODE-NATIVES/stream.js”: {
 “kind”: “commonjs”,
 “refs”: { … }
 },
 “/NODE-NATIVES/fs.js”: {
 “kind”: “commonjs”,
 “refs”: {
 “stream”: {
 “kind”: “commonjs”,
 “name”: “stream”,
 “path”: “/NODE-NATIVES/stream.js”
 },
 …
 }
 },
 “/sample/utils.js”: {
 “kind”: “commonjs”,
 “refs”: {}
 },
 “/sample/main.js”: {
 “kind”: “commonjs”,
 “refs”: {
 “fs”: {
 “kind”: “commonjs”,
 “name”: “fs”,
 “path”: “/NODE-NATIVES/fs.js”
 },
 “./utils”: {
 “kind”: “commonjs”,
 “name”: “./utils”,
 “path”: “/sample/utils.js”
 }
 }
 }
}

此JSON对象中的每个属性代表图中的一个节点。'refs'属性包含边缘。每个边都对应一个模块导入。

一个有趣的细节是，依赖关系分析器为本机节点模块返回特殊的路径字符串。当推理引擎请求此类路径的源代码时，用JavaScript编写并在Node.js上运行的脚本服务器从其自己的Node.js进程中提取源代码。推理引擎像普通的JavaScript文件一样对其进行分析。结果是为内置节点模块提供了很好的内容辅助：

[caption id =” attachment_12184” align =” aligncenter” width =” 587” caption =“内置节点模块的推断建议”] [/字幕]

支持哪些模块系统？


目前，我们仅提供对AMD和CommonJS的支持。对于CommonJS，我们使用增强型解析库来解析对路径的引用。对于AMD，我们目前使用定制的解析器。如果找到满足我们需求的库，则可以用现有库替换。

它是如何工作的？


该过程从目标文件 （即，传递给getDGraph函数的参数）开始。它大致按以下步骤进行：

  1. 检测模块类型（CommonJS与AMD）。

  2. 在目标模块中查找引用。

  3. 解析引用（即，确定将为引用加载的实际文件的路径）。

  4. 对每个已解析的引用重复从步骤1开始的过程。


步骤1基于检测一些典型的代码模式。例如，“未嵌套在'define'调用内的'require'调用”表示我们正在处理CommonJS模块。

根据步骤1中的模块类型，将步骤2和3分派到不同的支持模块。添加对其他模块类型的支持应该相对容易（前提是可以使第1步检测器识别新的模块类型）。

自动解析器配置


依赖性分析器尝试发现它需要知道的内容，而不需要手动配置。这是Scripted中的一般原理，依赖性分析器也不例外。实际上，没有办法手动配置依赖关系分析器或其任何组件，尽管我们很可能在将来使它成为可能。

对于Node / CommonJS来说，它工作得很好，主要是因为实际上并不需要太多配置。即，如果我们假设使用标准的Node.js加载程序算法，那实际上就是我们需要的所有信息。

不幸的是，对于AMD而言，情况更加复杂。典型的AMD加载器（例如requirejs ）是高度可配置的。此外，在项目的源代码中表达此配置的方式往往因项目而异。这给在随机项目中确定在哪里找到所需信息的挑战。

我们采用的方法是查看一些示例项目及其使用的“典型”模式。发现通过识别这些模式而起作用。希望是，如果我们支持足够的通用模式，最终发现将适用于大多数项目。对于失败的情况，我们可能还会添加一些手动配置选项作为最后的选择。

结论


我们看了一下Scripted的幕后故事。我们介绍了脚本化依赖关系分析引擎。它目前了解AMD和CommonJS模块系统。依赖性分析为类型推断引擎提供了依赖性图。基于此图的跨文件分析使我们可以针对其他模块中定义的功能提出准确的内容辅助建议。依赖项信息还用于为无法解析的模块引用创建错误标记，并允许快速导航到已解决的依赖项。将来，依存关系分析还可以实现准确的重构工具来移动或重命名模块。
