Spring Framework 3.2 RC1：Spring MVC测试框架

工程
罗森·斯托扬切夫（Rossen Stoyanchev）
2012年11月12日

更新2012年12月19日

最终的Spring Framework参考文档包含有关迁移的指南以及有关Spring MVC Test 的完整部分 。

上周Juergen Hoeller宣布
Spring Framework 3.2 RC1的发布和Sam Brannen讨论了其弹簧测试模块中令人兴奋的新增功能，例如对WebApplicationContext以及用于加载上下文层次结构的计划。今天我将继续这个主题，并描述另一个令人兴奋的spring-test加成。在3.2 RC1中，我们增加了对测试Spring MVC应用程序（客户端和服务器端）的一流支持。

背景

此处讨论的Spring MVC测试框架源自Github上的一个独立项目 ，该功能经过一年多的发展并得到了许多用户的不断反馈。感谢所有的早期采用者，所有贡献 ， 报告问题 ，发表评论的人，以及所有写博客或谈论它的人。

从Spring 3.2 RC1开始，独立项目中的代码已添加到Spring Framework中 ，并且可以在spring-test模块，在稍微修改的包名称下，并支持3.2特定功能，例如异步请求和其他功能。独立项目将继续存在，以针对Spring MVC 3.1进行应用程序测试。

今天，让我们更仔细，更详细地了解一下。

服务器端支持

您今天如何测试Spring MVC控制器？最有可能通过简单的单元测试，可能涉及MockHttpServletRequest和-Response 。这样做很琐碎，但测试不够。控制器具有注释，这些注释表示如何映射它们，需要提取，转换和验证哪些请求数据，是否写入响应主体，如何处理异常以及持续运行。如果仅编写简单的单元测试，则框架由于这些注释而进行的所有操作均未经测试。

如果您可以重写这些控制器单元测试，而不是直接调用控制器，而是通过DispatcherServlet来完成，就像在运行时那样，该怎么办？如果可以使用流利的API来指定要执行的请求和期望的响应，该怎么办？所有这些都不需要servlet容器。这就是Spring MVC Test所做的。这是一个例子：


@RunWith(SpringJUnit4ClassRunner.class)
@WebAppConfiguration
@ContextConfiguration("servlet-context.xml")
public class SampleTests {

  @Autowired
  private WebApplicationContext wac;

  private MockMvc mockMvc;

  @Before
  public void setup() {
    this.mockMvc = webAppContextSetup(this.wac).build();
  }

  @Test
  public void getFoo() throws Exception {
    this.mockMvc.perform(get("/foo").accept("application/json"))
        .andExpect(status().isOk())
        .andExpect(content().mimeType("application/json"))
        .andExpect(jsonPath("$.name").value("Lee"));
  }
}

静态进口
流利的API依赖于以下静态导入：
MockMvcBuilders.*
MockMvcRequestBuilders.*
MockMvcResultMatchers.*
为了获得代码完成帮助，请在Eclipse首选项中将它们添加为“收藏夹类型”，或者简单地记住以MockMvc* 。

如您所见，我们正在使用新的@WebAppConfiguration注释以加载我们的Spring MVC配置。然后我们注入结果WebApplicationContext进入测试类字段并使用它来创建一个MockMvc ，依次用于执行请求和定义期望。

上下文缓存
TestContext框架跨测试套件甚至跨JVM缓存加载的Spring配置。因此，测试速度应非常理想。

就像现有的控制器单元测试一样，Spring MVC Test建立在模拟请求和响应上spring-test并且不需要运行中的servlet容器。主要区别在于实际的Spring MVC配置是通过TestContext框架加载的，而请求是通过实际调用DispatcherServlet以及在运行时使用的所有相同的Spring MVC基础结构。

同样类似于现有的控制器单元测试，您可以考虑向控制器注入模拟服务，以便专注于测试Web层，例如避免访问数据库。因此，您可以加载创建模拟的配置，而不是加载实际的业务和持久性服务。例如：


@Configuration
public class MyConfig {

    @Bean
    public FooService fooService() {
        return Mockito.mock(FooService.class);
    }

}

或在XML配置中：


<bean class="org.mockito.Mockito" factory-method="mock">
    <constructor-arg value="org.example.FooService"/>
</bean>

假设我们没有在实际的servlet容器中运行，那么到底有多少会起作用？在大多数情况下，一切都将像在运行时一样工作。您甚至可以注册Servlet过滤器，从而启用诸如Spring Security之类的功能。大多数渲染技术（如JSON / XML，Freemarker，Velocity，Thymeleaf，Excel，PDF等）都可以使用。唯一排除的渲染技术是JSP，因为这需要Servlet容器。对于JSP，您仍然可以验证转发了请求的JSP，模型中的哪些属性，是否引发了任何异常等等。

客户端REST测试

客户端REST测试背后的想法是什么？如果您使用RestTemplate ，您可能需要对其进行测试，并且可以将其定位为正在运行的服务器或模拟RestTemplate。客户端REST测试支持提供了第三种选择，即使用实际的RestTemplate但使用自定义配置ClientHttpRequestFactory根据实际请求检查期望并返回存根响应。


RestTemplate restTemplate = new RestTemplate();
MockRestServiceServer mockServer = MockRestServiceServer.createServer(restTemplate);

mockServer.expect(requestTo("/greeting"))
  .andRespond(withSuccess("Hello world", "text/plain"));

// use RestTemplate ...

mockServer.verify();

静态进口
流利的API需要静态导入：

MockRestRequestMatchers.*
MockRestResponseCreators.*
为了获得代码完成帮助，请在Eclipse首选项中将它们添加为“收藏夹类型”，或者简单地记住以"MockRest*" 。

如您所见，我们创建了一个实例RestTemplate并传递给MockRestServiceServer进行配置。然后，我们定义预期请求的特征并提供要返回的存根响应。我们可以定义任意数量的预期请求和存根响应。在测试结束时，我们可以使用verify检查所有预期请求是否已执行的方法。

下一步

我们可以讨论更多的内容，但是最好尝试一下使用Spring Framework 3.2 RC1的自己的项目，或者是与Spring Framework 3.1一起使用的Github上的独立项目 。

最近，我亲自为spring-mvc-showcase的所有控制器方法添加了一套全面的测试 ，以完成练习。这是一个有用的练习，因为它帮助我找到了一个错误。因此，我希望它对其他人也有用。

如果您需要更多示例，则在Spring Framework中还有许多演示测试 ，包括带有异步请求的 测试，带有过滤器的测试 ， JSON响应 ， XML响应等。
