春季项目基础架构更新

工程
克里斯·比姆斯
2012年6月27日

介绍

在过去的一年中，我们用于保持Spring系列项目顺利运行的基础架构和流程进行了许多重大更改。您可能会看到其中一些事件的个别公告，而其他消息可能已被您忽略。我将在下面概述这些更改。当放在一起时，他们描绘了一个更大的图景。



GitHub项目托管


各个Spring项目已经迁移到Git和GitHub已有一段时间了。您可能还记得我们去年圣诞节宣布的Spring Framework 本身已采取行动。随着Spring Web Flow的最新迁移，我们很高兴地宣布，所有主要的Spring项目现在都托管在GitHub的SpringSource组织之下。

在迁移到Git和GitHub之后，项目提交者和Spring用户都会受益。GitHub具有出色的UI，可用于代码浏览，更改历史记录和提交评论。而且，由于已经在GitHub托管了数量惊人的开放源代码项目，这意味着您正在使用一个易于理解的UI，并且您已经知道如何浏览源代码控制，检查最近的更改等。但是GitHub的真正力量在于它鼓励和支持社区贡献的方式。这一点将在下面的“贡献过程”部分中进一步讨论。

现在，如果还没有，请查看GitHub上的SpringSource组织。您会看到我们像往常一样忙。



工件管理


SpringSource存储库位于 http://repo.springsource.org是一个Artifactory实例，致力于托管Spring项目工件的快照，里程碑，RC和GA版本以及它们的传递依赖项。

GA版本的Spring工件将继续一如既往地发布到Maven Central ，但是repo.springsource.org用更强大的解决方案替换了我们以前在maven.springframework.org上的基础结构–正确的UI和搜索功能，与CI的深度集成build.springsource.org上的服务器，以及许多其他有助于项目团队简化发布过程的功能。

对于Spring用户而言，SpringSource存储库是一站式购物，可满足您所有的Spring依赖需求。

如果您的构建脚本中包含maven.springframework.org URL，它们将无限期地工作*，但应将这些URL视为“已弃用”，以便以后使用repo.springsource.org。

可以在spring-framework GitHub Wiki的下载的Spring构件页面上找到进一步的说明和使用说明。SpringSource存储库常见问题解答应该能够回答出现的任何问题。如果没有，请告诉我们！

非常感谢JFrog的团队-Artifactory的制造商-与我们共同努力实现了这一目标。在如此多繁忙的项目之间切换存储库基础架构会带来灾难，但是在他们的帮助下，它很顺利。同时还要感谢Sonatype团队一直以来对Maven Central的需求做出迅速而有益的回应。



基于Gradle的构建


将我们的构建从Ant，Ivy和Maven迁移到Gradle是另一个迁移过程，该迁移过程已经在每个项目的基础上进行了一段时间。而且随着Gradle 1.0 GA公告仍然是新闻新鲜事 ，现在是一个很好的时机，提到大多数Spring项目现在都是使用Gradle ** 构建的 。

Gradle具有许多使人感到愉悦的创新功能，例如可显着减少构建时间的增量构建系统，仅向您显示所需知识的简约命令行输出以及使您能够轻松掌握的简洁语法。使用非常小的可读性的构建脚本来构建像Spring Framework这样的大型代码库。

对于Spring用户而言，用于构建Spring项目的系统似乎无关紧要。确实，就像大多数Linux用户不编译自己的内核一样，大多数Spring用户也从未从源代码构建Spring构件。但是像任何好的开源项目一样，有些也可以。无论是仅仅为了更好地理解Spring还是做出更改并为项目做出贡献，从源头构建Spring的人数都在增长，这在一定程度上要归功于Gradle所做的琐碎工作。Gradle的强大功能之一就是所谓的Gradle Wrapper ，它本质上是一个平台本机的Shell脚本，可以下载并安装项目使用的确切Gradle版本。这意味着从源代码构建基于Gradle的Spring项目实际上就像键入一样简单：

$ git clone https://github.com/SpringSource/spring-framework.git
$ cd spring-framework
$ ./gradlew build

如您所见，那里的“ gradlew”脚本已签入项目源代码管理。因此，随着用于构建项目的Gradle版本的更改，包装程序将保持最新状态，并在必要时自动下载新版本的Gradle。这意味着永远不会听到“构建已损坏”而只是发现正在使用两个不同版本的构建工具。这意味着Spring项目可以立即保持对Gradle的最新更改的最新信息，而不必冒险破坏其他提交者和贡献者的版本。

在Spring Framework自述文件的源代码部分中查看构建中的完整（但仍然非常简单）说明。将您看到的100个单词与我几年前写的有关Building Spring 3的1500个单词的博客帖子进行比较。我希望您会发现它是一个很大的进步。试试看–您甚至可能喜欢：

Gradle团队在此过程中提供了巨大的帮助-在过去的几年中，春季项目经常将Gradle推向极限，并向团队提出要求的挑战，而Gradle团队则只不过是反应敏捷且乐于助人。对于Gradle来说，今天的许多春季项目当然情况会更好，并且我们希望Gradle自身至少可以更好地应对我们所面临的挑战。恭喜团队1.0！



贡献过程


正如我上面提到的，关于GitHub的最好的事情之一是pull request的概念。阅读一下（如果您还不熟悉的话），但可以说拉动请求就像补丁和代码审查一样，都集成到一个紧密的过程和简单的UI中。查看一下Spring Integration ， Spring Framework以及许多其他Spring项目的拉取请求历史记录，您将看到很多有趣且有用的示例。这个过程比将补丁文件附加到JIRA上要好得多。当您将它与Git的功能以及使用Gradle进行构建的便捷性相结合时，这意味着为Spring项目做贡献从未如此简单。

自2011年底Spring Framework 移至GitHub以来，至2012年6月撰写本文时，该项目已收到100多个拉取请求。并非每个拉取请求都会进入。结果取决于审查过程，但是，尽管有时经过大量讨论和完善，但其中许多贡献已经发挥作用。我们根据这段时期的经验整理了一份贡献者准则文档；如果您有什么想回馈给框架的内容，请阅读一下。这是将您的问题摆在最前沿的好方法。自然而然地，我们会优先考虑那些花时间整理高质量作品的用户。感谢所有已经这样做的人，并在此先感谢将来的贡献者！



投稿人邮件列表


自从我们开始处理请求请求以来，我们已经注意到了很多讨论。在某些情况下，它们包含的问题可能会在提交请求之前的讨论中受益。此外，这些讨论在某种程度上仍然不被关注，并且仅对那些单独提出请求的人可见。毫无疑问，你们中的许多人希望能够关注此类讨论并在必要时参与。建立框架贡献者渠道的好处很多。

我们很高兴在此博客文章中宣布由Google网上论坛托管的新的spring-framework-contrib邮件列表。该列表专门针对希望为Spring Framework项目贡献代码的开发人员。在这里，您可以在编写代码之前或同时直接与提交者和其他贡献者讨论您的想法。我们鼓励任何想提出拉客请求的人加入该小组，并在那里与我们讨论您的想法。



问题生命周期


由于有许多用户，Spring Framework项目的问题特别多。可以预见，我们将在今年晚些时候看到第10,000期JIRA问题。伴随着这么多问题的出现以及Spring Framework 的多样化功能，您可以想象保持所有即将到来的问题之上的困难。在过去的一年中，我们精心设计了分类和积压流程，可帮助每个参与人员准确了解每个问题的进展。这对我们非常有帮助，我们已在问题的生命周期中对其进行了描述，以便每个人都可以理解此过程。

您还将注意到有关Spring Framework JIRA项目的一些较小更改：一组修订的组件 ， 简化的问题创建屏幕以及许多其他旨在使问题报告尽可能有用且轻松的调整。



社区论坛


多年来， 春季论坛一直是并将继续成为春季相关问答的好地方。在撰写本文时，有超过100,000个线程（总共近40万个帖子！）超过100,000名成员。这就是说，每天没有成千上万的访客访问而没有登录。毫无疑问，在很长的一段时间内，这将继续是宝贵的资源。

我们还想提供另一个问答资源，以确保Spring社区中的每个人都知道。如您所知，在过去的几年中， 堆栈溢出已成为编程问答的一种非常流行的资源。与Spring相关的问答环节也不例外。如果您访问 http://stackoverflow.com/tags并输入“ spring”，您将看到已经有许多与Spring相关的标签，其中最流行的是spring和spring-mvc 。我和Spring团队的其他成员经常对Stack Overflow的交谈质量印象深刻，我们非常高兴看到并感谢大家提供的出色答案。

与Spring论坛提供的严格的面向项目的类别相比，像Stack Overflow那样的基于标签的系统本质上要难定义一些，但是如果您有特定于Spring Framework 的问题，请提出建议。适当使用'spring'或'spring-mvc'标签。如果您有与Spring Integration相关的问题，请使用spring-integration标记等。的确是不言自明的。



摘要


希望您会同意，上述更改使我们所有人比以往任何时候都更容易合作，这是任何开源项目成功的关键因素。我们鼓励您花时间查看这篇文章中提到的资源，并尝试并利用可用的资源。请记住，贡献有多种形式-无论是在JIRA中定义明确的问题，参与讨论还是提交请求请求，都在促进整个对话和整个社区的发展！



脚注


*较旧的maven.springframework.org URL继续起作用，因为maven.springframework.org的DNS条目现在指向repo.springsource.org，并且/ snapshot，/ milestone和/ release路径现在映射到具有以下内容的“虚拟存储库”在repo.springsource.org上具有相同的名称和内容。

**各个Spring项目团队可以自由选择最适合他们的构建工具。这意味着尽管许多项目已转移到Gradle，但可以预料其他项目将坚持使用Maven。
