Spring Integration 2.2中的新增功能（第2部分-事务同步）

工程
加里·罗素
2012年9月26日

介绍


这是一系列博客文章的第二部分，重点介绍了在最近发布的Release Candidate 1之后，Spring Integration 2.2中提供的一些新功能。第一部分讨论MongoDB适配器。

Spring Integration 2.2引入了扩展支持，用于将非事务性资源与事务同步。

背景


多年来，Spring 为同步资源和事务提供了一流的支持 。在许多情况下，此工具用于同步由多个事务管理器管理的事务，以实现Dave Syer出色的JavaWorld文章中描述的“最大努力1PC”模式。

例如，它通常用于将JMS提交与JDBC提交同步。

长期以来，Spring Integration通过定义一个元素上的一个轮询。此版本通过允许非事务性资源与事务同步，使此功能更进一步。

例如，考虑一个集成应用程序，该应用程序在文件出现在目录中时读取文件，并更新数据库。此功能使我们可以配置事务提交或回滚时要采取的不同操作。

我们还将展示即使没有真正的交易资源，如何可以配置这些“后处理”操作。

警告

该功能不会使固有的非事务性资源（例如简单文件系统）成为事务性的；相反，它仅允许我们将文件系统（或其他）活动与事务同步。

内部构造

Spring事务同步功能的基本组件是一个实现对象TransactionSynchronization 。这样的对象具有框架在同步期间调用的许多回调（示例包括afterCommit ， afterRollback ）。

Spring Integration 2.2引入了TransactionSynchronizationFactory 。该工厂负责创建一个TransactionSynchronization每个邮件的对象。提供了默认实现，该实现允许作为事务同步的一部分针对消息执行SpEL表达式。

组态

让我们继续文件系统示例。为了启用事务同步，我们只需添加对TransactionSynchronizationFactory在元素，并定义工厂以及我们希望在事务同步期间评估的SpEL表达式。

<int-file:inbound-channel-adaper ...>
    <poller fixed-rate="1000">
        <transactional synchronization-factory="syncFactory"/>
    </poller>
</int-file:inbound-channel-adapter>

<int:transaction-synchronization-factory id="syncFactory">
	<int:after-commit expression="payload.renameTo('/successful/' + payload.name" />
	<int:after-rollback expression="payload.renameTo('/failed'/ + payload.name" />
</int:transaction-synchronization-factory>

如您所见，我们将重命名原始入站文件，并将该文件放到另一个目录中，具体取决于关联事务是提交还是回滚。可以选择将评估结果发送到channel使用transaction-synchronization-factory的子元素上的'channel'属性定义。

（注意：“ before-commit”元素也可用，也可以用来影响提交是否实际完成）。

伪交易

从上面可以看出，我们现在有了一个方便的机制，可以在事务提交之前或之后或回滚之后执行一些操作。但是，如果我们的流程根本不涉及交易资源，该怎么办？例如： file:inbound-adapter<-poller->transformer->ftp:outbound adapter 。

假设我们要根据转换文件的ftp文件传输的成功或失败来重命名输入文件。为了适应这种用例，我们引入了PseudoTransactionManager 。该类实现PlatformTransactionManager仅出于允许使用上述同步技术的目的，即使不涉及实际交易也是如此。只需添加元素与以前一样，为它提供了对PseudoTransactionManager （或使用默认的bean id'transactionManager'）。

结论

使用Spring Integration 2.2，用户现在可以在流程完成后（成功或不成功）执行适当的操作，通常会影响原始输入源。这可以与某些其他事务资源上的事务同步，甚至在不涉及实际事务的情况下也可以同步。
