脚本化：VMware的JavaScript编辑器

工程
安迪·克莱门特
2012年10月11日

脚本代码编辑器的第一个版本已于本周在github上发布： https://github.com/scripted-editor/scripted 。

尽管Scripted是通用代码编辑器，但最初的重点是构建出色的JavaScript编辑体验。脚本本身完全以JavaScript和HTML / CSS构建。它是基于浏览器的编辑器，可在开发人员的计算机上本地运行，并使用Node.js实例来提供编辑器代码和执行编辑器操作。运行Scripted的唯一前提条件是您已安装了最新版本的Node.js （我们现在正在测试Node 0.8.11）。脚本化可重复使用Eclipse Orion中的编辑器组件。

为什么要创建脚本？


脚本化是一些内部原型设计和调查工作的结果，这些工作探索了未来工具的不同策略。探索这个空间的驱动因素实际上有两个：

首先，我们看到许多用户选择不使用IDE，而是选择使用更简单的轻量级编辑器（vim，Sublime，textmate）。开发人员通常拥有一套非常熟悉的工具来完成常见任务（例如，命令行git），并且不需要通过其他用户界面来学习如何使用这些工具。这些开发人员希望这些工具能够立即启动，并在操作过程中保持极高的响应速度。然而，目前，当选择放弃IDE时，他们似乎也正在放弃他们习惯于提供出色的内容帮助，快速导航和早期错误指示的那些好处。脚本提供了轻便，快捷的功能，还支持开发人员无法缺少的关键IDE功能-诸如内容辅助和对通用模块系统的理解之类的功能。这些是脚本化的重点。

其次，我们看到Cloud IDE的普及以及Cloud工作区的概念正在增加。开发人员连接到某个远程系统以进行开发工作，通常这些工具（例如Cloud9 IDE和eXo Cloud IDE ）提供基于浏览器的编辑体验。用户工作区位于远程系统上。这种设置对于某些团队来说可以很好地工作，尽管根据我们的经验，我们仍然需要离线开发模式，因为开发人员尚未100％地连接到Internet，而且很难使他们进入放弃“完全控制”权并远程托管其文件。在遵循基于浏览器的编辑模型并在本地托管服务器时，Scripted提供了一些现在可以满足开发人员需求的功能，但将来也可以通过简单地远程部署服务器来使用云工作区。

从根本上讲，我们认为许多关键领域缺少许多现有的JavaScript工具，并且鉴于我们在语言工具方面的背景，我们希望了解是否可以构建一个轻量级的工具来满足这些需求。
 

专注于JavaScript


JavaScript继续流行。不再仅仅用于客户端编程，像Node.js这样的框架使它可以用于完整的端到端解决方案。这就是为什么我们选择将JavaScript视为Scripted的第一要务。当然，还有一些我们也感兴趣的相关语言（例如，CoffeeScript，最近宣布的TypeScript），但目前的共同点是JavaScript。我们希望为一种语言提供出色的体验，而不是为多种语言提供非凡的体验。
 

功能集



  • 快速启动，轻巧。

  • JavaScript，HTML和CSS的语法高亮显示。

  • 错误和警告：

    • 集成了JSLint以在JavaScript代码上提供错误/警告标记。

    • AMD和CommonJS模块分辨率：存在基本分辨率，其中未解决的引用将被标记为错误。



  • 内容辅助：

    • HTML，CSS的基本内容辅助

    • 对于JavaScript，内容辅助由类型推断引擎驱动，该引擎知道AMD / CommonJS模块的依赖性，并且还使用JSDoc注释来帮助其理解代码。



  • 悬停：将鼠标悬停在JavaScript标识符上将显示推断出的类型签名。

  • 导航：在标识符（推理者已识别）上按F8键，编辑器将导航到声明。这也适用于模块标识符（例如，在define（）子句中）

  • 格式：JSbeautify已集成

  • 侧面板：可以在主编辑器旁边打开侧面板-当前可用于托管第二个编辑器。

  • 与外部命令的键绑定：编辑器中的键绑定可以调用外部命令（less，mvn等）


Wiki文档中有关于这些功能的更多详细信息 。

 

使用脚本开发脚本


脚本化是100％JavaScript，HTML和CSS。因此，它是使用脚本编辑器的理想代码库。对于其他工具项目，我们通常不在日常工作中使用这些工具。相反，我们开发它们供他人使用。在VMware，我们使用脚本化来开发脚本化-并没有比开发人员自己不断发现bug更快地修复错误！

 

脚本化入门


github登陆页面包含一个入门视频：

但是基本步骤如下：


  1. 确保您已安装节点

  2. 从此处获取最新的打包版本（0.2.0）：
    scripted_v0.2.0.zip 

  3. 解压缩

  4. 确保bin文件夹中的脚本可通过以下方式执行（如果在linux / mac上）：
    chmod 755 bin / *

  5. 将bin文件夹添加到您的路径：

    Mac / Linux
    导出PATH = / bin：$ PATH

    赢得：

    设置PATH = \ bin;％PATH％
     


  6. 开始使用它，就像使用“ scr”或“ scripted”启动“ vi”一样启动它

    scr foo.js



脚本化脚本将在启动时尝试推断您的项目根目录。它通过在层次结构中搜索关闭的.git或.project文件来实现。如果没有，它将在单文件模式下工作。要告诉Scripted根目录在哪里，您可以在根目录创建一个简单的（空） .scripted文件。脚本编写者需要知道其根源，因为某些操作（如内容辅助，依赖项解析，搜索）当然是在项目的上下文中发生的。
 

技术

如前所述，服务器端技术是Node.js，但这实际上是很少的服务器代码。推理引擎的基础是我们编写的用于分析模块依赖性的一些服务器端JS。

在客户端，我们不想重新发明编辑器技术，因此选择使用Eclipse Orion中的编辑器。这提供了一种很好的快速编辑体验，这对于在“全食”中使用过编辑器的人来说都是非常熟悉的-它具有许多相同的行为和键绑定。我们在内容辅助等功能方面所做的工作将尽可能地归还给Orion项目。任何需要解析的JavaScript都会通过我们的Esprima解析器的可恢复派生传递。当开发人员积极地编辑文件时，代码通常处于未完成状态，因此，即使有错误也可以返回良好的AST的可恢复解析器非常重要。

 

接下来是什么？


我们目前处于早期阶段（版本0.2.0），我们的未来计划包括：

  • 更加智能的推理，可以提供更好的内容辅助和更轻松的导航。

  • 侧面板的更多窗格。当前只有一个编辑器窗格，但我们打算包括搜索结果窗格，文档，git信息窗格，也许还包括代码预览和模拟代码执行窗格。Scripted的意图是尝试并在可能的情况下自动进行管理，因此屏幕上的所有内容均与手头任务相关。

  • 简单的插件系统。

  • 调试。探索与Chrome开发工具和节点检查器等工具的集成。


 

我们决定尽早开源以获取反馈。如果您想帮助我们设计编辑器，请加入讨论。有一个脚本化的google开发小组来讨论它，还有一个jira issuetracker来记录错误，增强功能请求和对现有问题进行投票，以确保对它们进行优先排序。如果您想自己开始在代码库上进行黑客攻击，我们绝对欢迎提交文件-有关更多信息，请参见github页面。
