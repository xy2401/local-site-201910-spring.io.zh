Spring MVC 3.2预览版：引入Servlet 3，异步支持

工程
罗森·斯托扬切夫（Rossen Stoyanchev）
2012年5月7日

最后更新时间为2012年11月5日（Spring MVC 3.2 RC1）

总览

Spring MVC 3.2引入了基于Servlet 3的异步请求处理。这是涵盖此新功能的几篇博客文章中的第一篇，并提供了上下文，以了解您将如何以及为何使用它。

早期版本的主要目的是寻求反馈。自从3.2 M1版本发布以来，我们在这里和JIRA中都收到了很多。感谢所有尝试并发表评论的人！发生了许多变化，仍然有时间获得更多反馈！

乍看上去

从编程模型的角度来看，新功能看似简单。控制器方法现在可以返回一个java.util.concurrent.Callable异步完成处理。然后，Spring MVC将调用Callable在一个单独的线程的帮助下TaskExecutor 。这是之前的代码段：


// Before
@RequestMapping(method=RequestMethod.POST)
public String processUpload(final MultipartFile file) {
    // ...
    return "someView";
}

// After
@RequestMapping(method=RequestMethod.POST)
public Callable<String> processUpload(final MultipartFile file) {

  return new Callable<String>() {
    public Object call() throws Exception {
      // ...
      return "someView";
    }
  };
}

控制器方法也可以返回DeferredResult （Spring MVC 3.2中的新类型）以完成Spring MVC未知的线程中的处理。例如，对JMS或AMQP消息，Redis通知等做出反应。这是另一个代码片段：


@RequestMapping("/quotes")
@ResponseBody
public DeferredResult<String> quotes() {
  DeferredResult<String> deferredResult = new DeferredResult<String>();
  // Add deferredResult to a Queue or a Map...
  return deferredResult;
}


// In some other thread...
deferredResult.setResult(data);
// Remove deferredResult from the Queue or Map

上述示例引发了许多问题，我们将在后续文章中详细介绍。现在，我将围绕这些功能提供一些背景信息。

Web应用程序中异步性的动机

Web应用程序中异步性的最基本动机是处理需要较长时间才能完成的请求。可能是缓慢的数据库查询，对外部REST API的调用或其他一些受I / O约束的操作。这样长的请求可能很快耗尽Servlet容器线程池并影响可伸缩性。

在某些情况下，您可以在后台作业完成处理后立即返回客户端。例如，发送电子邮件，启动数据库工作等，其他都代表了即发即弃的情况，可以通过Spring的@Async支持来处理，或者通过将事件发布到Spring Integration通道中，然后返回客户可以使用的确认ID查询结果。

在其他情况下，如果需要结果，则需要将处理与Servlet容器线程解耦，否则我们将耗尽其线程池。Servlet 3提供了这样的支持，其中Servlet（或Spring MVC控制器）可以指示响应应在退出Servlet容器线程后保持打开状态。

为此，Servlet 3 Web应用程序可以调用request.startAsync()并使用返回的AsyncContext继续从其他一些单独的线程写入响应。同时，从客户端的角度来看，请求仍然看起来像任何其他HTTP请求-响应交互。它只需要更长的时间就可以完成。以下是事件的顺序：

  1. 客户发送请求
  2. Servlet容器分配一个线程并在其中调用一个Servlet。
  3. servlet调用request.startAsync() ，保存AsyncContext ，并返回
  4. 容器线程一直退出，但响应保持打开状态
  5. 其他线程使用保存的AsyncContext完成回应
  6. 客户收到回复

Servlet异步支持当然还有很多。您可以找到各种 示例和文章 ，但以上内容总结了您需要了解的基本的最低概念。下一篇文章介绍了异步请求处理的第二个动机-浏览器需要实时接收信息更新。
