Spring Security 3.2 M1亮点，Servlet 3 API支持

工程
罗伯·温奇
2012年12月17日

上周，我宣布了Spring Security 3.2 M1的发布，其中包含改进的Servlet 3支持。在本文中，我将介绍3.2 M1版本中发现的一些更令人兴奋的功能。具体来说，我们将研究以下新的Spring Security功能：

  • 并发支持
  • Servlet 3 API集成
  • Servlet 3异步支持
  • Spring MVC异步集成


并发支持

您可能会问：“在以Servlet 3为主题的发行版中，并发支持是做什么的？“原因是并发支持为该版本中的所有其他功能提供了基础。尽管并发支持由Servlet 3集成使用，但它还可以用作构建块，以在任何应用程序中支持并发和Spring Security。现在让我们看一下Spring Security的并发支持。

DelegatingSecurityContextRunnable

Spring Security并发支持中最基本的构建块之一是DelegatingSecurityContextRunnable 。它包装了一个委托Runnable为了初始化SecurityContextHolder具有指定的SecurityContext为代表。然后，它调用委托Runnable确保清除SecurityContextHolder然后。的DelegatingSecurityContextRunnable看起来像这样：

public void run() {
  try {
    SecurityContextHolder.setContext(securityContext);
    delegate.run();
  } finally {
    SecurityContextHolder.clearContext();
  }
}

虽然非常简单，但可以无缝传输SecurityContext从一个Thread到另一个。这很重要，因为在大多数情况下， SecurityContextHolder对每个行为Thread基础。例如，您可能已经使用Spring Security的支持来保护您的一项服务。您现在可以轻松转移SecurityContext当前Thread到Thread调用安全服务。下面是如何执行此操作的示例：


Runnable originalRunnable = new Runnable() {
  public void run() {
    // invoke secured service
  }
};

SecurityContext context = SecurityContextHolder.getContext();
DelegatingSecurityContextRunnable wrappedRunnable =
    new DelegatingSecurityContextRunnable(originalRunnable, context);

new Thread(wrappedRunnable).start();

上面的代码执行以下步骤：

  • 创建一个Runnable这将调用我们的安全服务。请注意，它不知道Spring Security
  • 获得SecurityContext我们希望从SecurityContextHolder并初始化DelegatingSecurityContextRunnable
  • 使用DelegatingSecurityContextRunnable创建一个Thread
  • 开始Thread我们创造了

由于创建一个DelegatingSecurityContextRunnable与SecurityContext来自SecurityContextHolder有一个快捷方式构造函数。以下代码与上面的代码相同：


Runnable originalRunnable = new Runnable() {
  public void run() {
    // invoke secured service
  }
};

DelegatingSecurityContextRunnable wrappedRunnable =
    new DelegatingSecurityContextRunnable(originalRunnable);

new Thread(wrappedRunnable).start();

我们拥有的代码易于使用，但仍然需要了解我们正在使用Spring Security。在下一节中，我们将研究如何利用DelegatingSecurityContextExecutor隐藏我们正在使用Spring Security的事实。

DelegatingSecurityContextExecutor

在上一节中，我们发现使用DelegatingSecurityContextRunnable ，但这并不理想，因为我们必须知道Spring Security才能使用它。让我们来看看DelegatingSecurityContextExecutor可以使我们的代码免受使用Spring Security的任何了解。

的设计DelegatingSecurityContextExecutor与...非常相似DelegatingSecurityContextRunnable除了它接受委托执行器而不是委托Runnable 。您可以在下面查看如何使用它的示例：


SecurityContext context = SecurityContextHolder.createEmptyContext();
Authentication authentication = 
    new UsernamePasswordAuthenticationToken("user","doesnotmatter", AuthorityUtils.createAuthorityList("ROLE_USER"));
context.setAuthentication(authentication);

SimpleAsyncTaskExecutor delegateExecutor =
    new SimpleAsyncTaskExecutor();
DelegatingSecurityContextExecutor executor =
    new DelegatingSecurityContextExecutor(delegateExecutor, context);

Runnable originalRunnable = new Runnable() {
  public void run() {
    // invoke secured service
  }
};

executor.execute(originalRunnable);

该代码执行以下步骤：

  • 创建SecurityContext用于我们的DelegatingSecurityContextExecutor 。请注意，在此示例中，我们仅创建了SecurityContext用手。但是，无论在哪里或如何获得SecurityContext （即，我们可以从SecurityContextHolder如果我们想要）。
  • 创建一个负责执行提交的委托委托人Runnable s
  • 最后，我们创建一个DelegatingSecurityContextExecutor负责包装任何Runnable通过一个DelegatingSecurityContextRunnable 。然后通过包装Runnable给委托执行人。在这种情况下，相同SecurityContext将用于每个Runnable提交给我们DelegatingSecurityContextExecutor 。如果我们正在运行需要由特权较高的用户运行的后台任务，那就很好。

此时，您可能会问自己：“这如何屏蔽我的代码，使其不了解Spring Security？“而不是创建SecurityContext和DelegatingSecurityContextExecutor在我们自己的代码中，我们可以注入一个已经初始化的实例DelegatingSecurityContextExecutor 。


@Autowired
private Executor executor; // becomes an instance of our DelegatingSecurityContextExecutor

public void submitRunnable() {
  Runnable originalRunnable = new Runnable() {
    public void run() {
      // invoke secured service
    }
  };
  executor.execute(originalRunnable);    
}

现在，我们的代码没有意识到SecurityContext被传播到Thread ，然后执行originalRunnable，然后清除SecurityContextHolder。在此示例中，使用同一用户执行每个Thread 。如果我们想使用以下用户怎么办SecurityContextHolder在我们调用时executor.execute(Runnable) （即当前登录的用户）来处理originalRunnable？这可以通过删除SecurityContext来自我们的论点DelegatingSecurityContextExecutor构造函数。例如：


SimpleAsyncTaskExecutor delegateExecutor = new SimpleAsyncTaskExecutor();
DelegatingSecurityContextExecutor executor =
    new DelegatingSecurityContextExecutor(delegateExecutor);

现在任何时候executor.execute(Runnable)被执行SecurityContext首先由SecurityContextHolder然后那个SecurityContext用于创建我们的DelegatingSecurityContextRunnable 。这意味着我们正在执行我们的Runnable与用于调用executor.execute(Runnable)码。

Servlet 3 API集成

Spring Security支持Servlet API集成已有相当一段时间。但是，直到3.2 M1才支持Servlet 3中添加的新方法。在本节中，我们将讨论Spring Security集成的每种方法。如果您希望实际操作，可以使用Gradle插件将Spring Security导入Spring Tool Suite并运行servletapi示例应用程序 。

HttpServletRequest.authenticate（HttpServletRequest，HttpServletResponse）

Spring Security现在与HttpServletRequest.authenticate（HttpServletRequest，HttpServletResponse）集成在一起。简而言之，我们可以使用此方法来确保对用户进行身份验证。如果未通过身份验证，则将使用配置的AuthenticationEntryPoint来请求用户进行身份验证（即，重定向到登录页面）。

HttpServletRequest.login（String，String）

Spring Security现在与HttpServletRequest.login（String，String）集成。用户可以使用此方法通过Spring Security验证用户名和密码。如果身份验证失败，则ServletException包装了原始的Spring Security AuthenticationException将被抛出。这意味着如果您允许ServletException传播Spring Security的ExceptionTranslationFilter将为您处理。或者，您可以抓住ServletException并自己处理。

HttpServletRequest.logout（）

Spring Security现在通过调用已配置的与HttpServletRequest.logout（）集成LogoutHandler实现。通常，这意味着SecurityContextHolder将被清除， HttpSession将无效，将清除所有“记住我”身份验证，依此类推。但是，已配置LogoutHandler具体实现将取决于您的Spring Security配置。重要的是要注意HttpServletRequest.logout()已被调用，您仍然负责写出响应。通常，这将涉及重定向到欢迎页面。

AsyncContext.start（Runnable）

确保您的凭据将被传播到新的AsynchContext.start（Runnable）方法Thread 。使用Spring Security新增的并发支持，Spring Security可以覆盖AsyncContext.start(Runnable)确保当前SecurityContext用于处理Runnable 。


Servlet 3异步支持

Spring Security现在支持Servlet 3，异步请求。那么您如何使用它呢？

第一步是确保已更新web.xml以使用3.0模式，如下所示：


<web-app xmlns="http://java.sun.com/xml/ns/javaee"
  xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
  xsi:schemaLocation="http://java.sun.com/xml/ns/javaee http://java.sun.com/xml/ns/javaee/web-app_3_0.xsd"
  version="3.0">

</web-app>

接下来，您需要确保已设置springSecurityFilterChain来处理异步请求。


<filter>
  <filter-name>springSecurityFilterChain</filter-name>
  <filter-class>
    org.springframework.web.filter.DelegatingFilterProxy
  </filter-class>
  <async-supported>true</async-supported>
</filter>
<filter-mapping>
  <filter-name>springSecurityFilterChain</filter-name>
  <url-pattern>/*</url-pattern>
  <dispatcher>REQUEST</dispatcher>
  <dispatcher>ASYNC</dispatcher>
</filter-mapping>

而已！现在，Spring Security将确保您的SecurityContext也会在异步请求上传播。

那么，发生了什么变化？Spring Security中的内部重构将确保您的SecurityContext在另一个响应上提交时不再清除Thread ，导致用户似乎已注销。此外，您可以使用Spring Security并发支持和Spring Security的AsyncContext.start(Runnable)集成可帮助您处理Servlet请求。


Spring MVC异步集成

将SecurityContext与Callable的关联

从技术上讲，Spring Security与WebAsyncManager集成。的SecurityContext用于处理Callable是个SecurityContext存在于SecurityContextHolder在调用startCallableProcessing时。

正如Rossen在先前的博客文章中所展示的那样，Spring Web MVC 3.2具有出色的Servlet 3异步支持 。如果没有其他配置，Spring Security将自动设置SecurityContext到Thread执行一个Callable由您的控制器返回。例如，以下方法将自动具有Callable与执行SecurityContext当Callable已创建：


@RequestMapping(method=RequestMethod.POST)
public Callable<String> processUpload(final MultipartFile file) {
 
  return new Callable<String>() {
    public Object call() throws Exception {
      // ...
      return "someView";
    }
  };
}

没有与DeferredResult由控制器返回。这是因为DeferredResult由用户处理，因此无法自动与其集成。但是，您仍然可以使用并发支持来提供与Spring Security的透明集成。
