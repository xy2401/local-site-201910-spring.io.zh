通过里程碑1偷窥Reactor-Core 3.2

发布
西蒙·巴斯莱（SimonBaslé）
一月16，2018

问候语Reactive Spring社区！

尽管该团队仍在3.1线上进行大量工作，但我们也想给社区一个机会，让我们一窥未来的3.2线。

特别值得一提的是，该大礼包的定价为3.2.0。RELEASE是我们一直所说的“错误模式”，“继续模式”或最近更正式的“错误策略”的补充。

这个主意

确实很简单：如果可以从在操作符中执行的用户代码异常中恢复出来，使序列继续下去 ，该怎么办？

让我们以一个示例为例，并假设您具有以下方法：

public Flux<Integer> divide100By(Flux<Integer> dividers) {
  return dividers.map(div -> 100 / div);
}

万一dividers源发出0在某个时候，结果Flux会立即以onError一个信号ArithmeticException 。

假设来源是， Flux.range(0, 10) ，仍然可以映射9个完全有效的值。

您如何做到这一点，以使这样的短暂异常（仅该异常）被忽略，并有机会进一步处理有效值？

当前的解决方法

在Reactor 3.1中，有一种解决方法可以通过使用flatMap为每个元素创建内部序列，然后将错误恢复操作符应用于这些粒度序列：

public Flux<Integer> divide100By(Flux<Integer> dividers) {
  return dividers.flatMap(div -> (1)
      Mono.just(100 / div) (2)
          .doOnError(e -> { (3)
              if (e instanceof ArithmeticException) process(e); (4)
          })
          .onErrorResume(ArithmeticException.class, e -> Mono.empty()) (5)
      );
}

  1. 代替map我们用flatMap ，产生一个小的内部Mono对于每个值

  2. 那Mono基本上是旧的map操作...

  3. …添加了错误恢复。

  4. 首先，我们确保要处理（例如日志） ArithmeticException （仅这些）在“恢复”之前

  5. 然后我们用onErrorResume与一个Mono.empty()有效地忽略结果序列中的异常

可以，但是写起来有点麻烦（尽管compose和transform可以帮助使这类代码相互化）。而且，我们从单一map运算符flatMap与几个内部运算符。

flatMap具有比map ，因为需要协调多个来源。即使像操作员融合之类的事情减少了该成本，它仍然存在。

引入错误策略

如果要进一步减少此类处理的开销，困难在于我们现在必须在每个操作员的实现级别上进行工作。

必须以某种方式使链中的每个运算符都知道应该捕获异常，但不能通过异常传播onError ，但处理方式有所不同。这是一个很大的变化，而且是横向的！

请注意，这在概念上听起来像是filter ，但有错误。像一个filter这意味着操作员在其中一个操作之后继续处理其源onNext引发异常的情况下，还应从其源中至少再请求一个元素。

尽管可以将其隔离到特殊的执行路径，但对于操作员而言，这仍然是一个复杂的核心实现更改。

然后是API的问题：将它放置为构造函数参数，否则会产生额外的重载，这确实很繁琐。 Flux带有``错误恢复''布尔值...我们真的需要将方法中的方法数量加倍吗？ Flux支持该功能的API？

所幸没有：从3.1开始，我们有了Context ，这是将此类信息传播给链中每个（Reactor）操作员的好方法。

这就是我们采取错误策略功能的道路：

  • 支持仅添加到特定的运算符 （ map ， filter ， flatMap ， handle仅举几例...）。这些具有特殊的javadoc标记来记录该事实。

  • 该功能已针对给定激活Flux通过在其上放置一个特殊的键Context

  • 每个支持的运算符在其中都有一条特殊的路径onNext检查该密钥的实现，如果找到，将更改其处理错误的方式。

  • 该功能通过errorStrategyContinue() API

  • 它可能更精细：可以过滤可以从中恢复的异常，还可以为此类恢复的异常设置自定义处理程序。

重要
 要记住的一件事是，因为这是通过Context ，该功能遵循与以下相同的传播规则Context 。例如，它将在内部序列中激活flatMap 。如果不希望这样，请使用errorStrategyStop()在flatMap中返回默认行为（这不会逃避flatMap的范围）。它还向后传播，激活之前的运算符errorStrategyXXX 。

这是我们先前的示例在错误策略中的作用：

public Flux<Integer> divide100By(Flux<Integer> dividers) {
  return dividers.map(div -> 100 / div) (1)
      .errorStrategyContinue(ArithmeticException.class, (2)
          (error, value) -> process(error)); (3)
}

  1. 回到简单map

  2. 我们只能从ArithmeticException

  3. 我们将此类异常传递给内部处理程序（请注意，我们也可以访问导致异常的原始值，如果有的话）

用里程碑1进行测试

我们刚刚发布了一个3.2.0.M1里程碑[ 1 ] ，主要包含错误策略功能，我们希望您对其进行测试🙇

注意
 这是一个横向变化，即使您不打算使用它，也可以使用工件来进行测试，以验证是否未明确使用它，这很有价值。 errorStrategyContinue() ，则您不应看到任何行为上的变化（因为该功能包含在特定的执行路径中）。

为了获得里程碑，请添加repo.spring.io/milestone仓库到您的Maven或Gradle构建配置并获取reactor-core3.2.0.M1工件：

<dependency>
  <groupId>io.projectreactor</groupId>
  <artifactId>reactor-core</artifactId>
  <version>3.2.0.M1</version>
</dependency>

特别是对于该功能的API，目前还没有任何问题。因此，如果您有任何反馈意见，请通过在GitHub上发布问题或让我们在Gitter上讨论该功能来告诉我们。

同时，祝您编程愉快！

1 。PS：此里程碑早于3.1.3发布。RELEASE仍在开发中，因此请注意，它不包含3.1.3和后续3.1.x版本中的所有修补程序。
