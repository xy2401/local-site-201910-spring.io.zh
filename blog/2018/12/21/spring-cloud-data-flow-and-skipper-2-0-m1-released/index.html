Spring Cloud Data Flow和Skipper 2.0 M1发布

发布
马克·波拉克
十二月21，2018

Spring Cloud Data Flow团队很高兴宣布发布2.0 M1遵循在本地 ， Cloud Foundry和Kubernetes上运行的入门指南 。

手拉手是2.0 M1 Spring Cloud Skipper的发行版已更新，以依赖于Spring Boot 2.1。参考指南中的“ 入门”部分是开始踢轮胎的最佳位置。

这是数据流的重点

  • 流部署始终委托给Skipper

  • 在所有受支持的平台上运行的单个服务器

  • 更新到Boot 2.1

  • 标准化OAuth2和OpenID Connect以获得安全性

  • 更新内部组件以使用JPA

  • 界面改进

  • 任务/作业执行和性能改进

流部署始终委托给Skipper

1.x系列Data Flow Server允许Streams由Data Flow Server直接部署或委托给Skipper。这导致了两种操作模式，“经典”和“船长”。现在只有一种通过Skipper部署流的选项，它为流应用程序提供滚动升级和降级功能。数据流服务器像以前一样部署任务。

在所有受支持的平台上运行的单个服务器

我们已经将Local，Cloud Foundry和Kubernetes服务器整合为一台服务器，无需根据要运行的平台选择不同的jar或docker镜像。每个平台的入门步骤基本相同。一项重大变化是配置任务的方式，因为我们正在开放跨不同平台启动任务的功能。

更新到Boot 2.1

Data Flow Server 2.0版本现在基于Spring Boot 2.1。通过这种根本的转变，我们花了一些时间解决技术问题并采用了Spring Boot的新功能。在迈向Data Flow 2.0 GA的过程中，我们将继续解决技术难题和一些其他内部重构问题。

标准化OAuth2和OpenID Connect以获得安全性

通过采用OAuth2和OpenID connect作为默认的安全性实现，已经在提高安全性方面进行了大量工作。传统的安全选项已删除。令牌授权，生命周期管理，吊销和续订都是该功能集的一部分，我们很高兴为用户带来的开发人员体验。

更新内部组件以使用JPA

从Spring XD到Data Flow 1.x，我们保留了一些“键值”之类的表，并通过Spring JDBC API对其进行管理。现在，我们已更新为使用Spring Data JPA，并将在未来的里程碑版本中引入Flyway进行模式管理。

界面改进

仪表板的改进工作会定期进行。Angular已升级到7.1.1，并根据用户反馈在作业的执行页面内添加了作业重新启动按钮。

任务/作业执行和性能改进

感谢社区成员Nicolas Widart提供了有关任务执行及其相关性能问题的详尽的错误报告 。

添加了一个新端点，以避免破坏对现有REST资源的更改，该更改现在已在客户端工具的幕后使用。有了这些更改，查询任务/批处理执行历史记录的速度几乎快了10倍，这在批处理作业中有数百个步骤时提供了帮助。

接下来是什么

2.0 M2版本将全面改进对Analytics（分析）的支持-迁移为使用Micrometer及其受支持的后端，而不是当前的基于Redis的功能。Analytics资讯主页也将被淘汰，并且可以使用更多的生产级工具，例如Grafana。度量标准收集也将发生类似的变化，服务器和现成的应用程序将通过Mirometer库进行配置，以用于一些流行的监视平台。Spring Cloud Metrics Collector将不再使用。任务支持将得到一些改进，从而使Data Flow Server可以跨平台启动任务。我们还在一个新的Data Flow网站上工作，该网站将提供更好的体验，以帮助您了解如何将Data Flow用于各种用例。

另请注意，Spring Cloud Data Flow的1.x系列将从2.0 GA发布之日起十二个月后停止维护。2.，0 GA暂定于2019年2月发布，因此1.x线将达到通用支持终止状态。
