响应式编程和关系数据库

工程
马克·帕鲁奇
十二月07，2018

命令性代码在软件吞噬世界的同时，以传入请求的速度吞噬线程。这篇文章讨论了在JVM上进行反应式编程的假设以及这对集成意味着什么，特别是关系数据库。

提出此职位的动机是，反应式编程的采用不断增加，而一些主要的构建模块尚不可用，特别是问题：关系数据库如何？

什么是反应式编程

关于什么是反应式编程以及与反应式系统的比较，有很多答案。我将响应式编程视为一种编程模型，该模型通过创建对资源的可用性和可处理性做出反应的事件驱动的非阻塞功能管道来促进可伸缩性和稳定性。延迟执行， 并发性和异步性只是底层编程模型的结果。

仅当整个堆栈都是反应性的，并且所有参与的组件（应用程序代码，运行时容器，集成）都遵守延迟执行，非阻塞API和数据流的流化性质时，反应式编程的全部好处才会生效–基本上遵循基本假设。

虽然可以将非反应性组件引入以功能反应性风格编写的应用程序，但最终结果是可伸缩性和稳定性影响，实际预期收益降低了。在最坏的情况下，运行时行为几乎没有差异。但是，反应式编程有助于提高代码的可读性。

如果我们查看反应性生态系统，我们将发现几个框架，库和集成。他们每个人都有其特定的优势。无论是通用方法还是在特定的反应框架下，都可以很好地涵盖许多功能领域。因此，让我们讨论关系数据库集成。

关系数据库和反应式

关系数据库流行是众所周知的，大概大多数企业项目都严重依赖关系数据库的使用。无论如何，最常被问到的问题是：我们什么时候才能获得用于反应性关系数据库集成的API？

Java使用JDBC作为与关系数据库集成的主要技术。JDBC具有阻塞性–减轻JDBC的阻塞性没有任何明智的选择。有关如何使调用非阻塞的第一个想法是将JDBC调用卸载到Executor （通常Thread池）。尽管此方法有些奏效，但它具有一些缺点，而忽略了反应式编程模型的好处。

线程池要求运行线程（毫不奇怪）。响应式运行时通常使用与CPU内核数量匹配的有限数量的线程。附加线程会增加开销，并减少线程限制的影响。另外，JDBC调用通常堆积在队列中，一旦线程被请求饱和，池将再次阻塞。因此，JDBC现在不是一种选择。

响应式数据库工作

有几个独立的驱动程序，例如Reactiverse的react-pg-client 。这些驱动程序带有特定于供应商的API，并不真正适合广泛采用。客户集成商将需要提供其他层来公开通用API。无法轻松将新的驱动程序插入客户端库。相比之下，拥有标准的API可以实现可插拔性，同时将客户端与特定于数据库的解决方案脱钩，这对所有人来说都是巨大的价值。

甲骨文宣布推出ADBA计划 ，该计划旨在通过使用期货为Java中的异步数据库访问提供标准化的API。ADBA中的所有工作仍在进行中，ADBA背后的团队很高兴获得反馈。一群Postgres人员正在研究可用于首次实验的Postgres ADBA驱动程序 。PgNio是Postgres的另一个异步驱动程序，已开始尝试使用ADBA。

ADBA的可用性未知。绝对不是Java 12附带的。坦率地说，目前尚不知道ADBA计划在其中首次发布的Java版本。

以下代码段显示了ADBA的用法INSERT和SELECT声明：

DataSource ds = dataSource();
CompletableFuture<Long> t;

try (Session session = ds.getSession()) {

  Submission<Long> submit = session
  .<Long>rowCountOperation(
    "INSERT INTO legoset (id, name, manual) " +
    "VALUES($1, $2, $3)")
    .set("$1", 42055, AdbaType.INTEGER)
    .set("$2", "Description", AdbaType.VARCHAR)
    .set("$3", null, AdbaType.INTEGER)
    .apply(Result.RowCount::getCount)
    .submit();

  t = submit.getCompletionStage().toCompletableFuture();
}

t.join();

CompletableFuture<List<Map<String, Object>>> t;
try (Session session = ds.getSession()) {

  Submission<List<Map<String, Object>>> submit = session
    .<List<Map<String, Object>>> rowOperation(
      "SELECT id, name, manual FROM legoset")
    .collect(collectToMap()) // custom collector
    .submit();
  t = submit.getCompletionStage().toCompletableFuture();
}

t.join();

注意collectToMap(…)以应用程序提供的函数为例，该函数将结果提取为所需的返回类型。

TL; DR，没有可用的反应式API来访问关系数据库。

R2DBC来救援！

缺乏标准的API和驱动程序不可用， Pivotal的团队开始研究反应性关系API的想法，该想法非常适合于反应性编程目的。他们提出了R2DBC ， R2DBC代表反应式关系数据库连接。到目前为止，R2DBC是一个孵化器项目，旨在评估可行性并开始讨论驱动程序供应商是否对支持反应性/非阻塞/异步驱动器感兴趣。

到目前为止，有三种驱动程序实现：

  • PostgreSQL的
  • H2
  • Microsoft SQL服务器

R2DBC随附API规范（ r2dbc-spi ）和客户（ r2dbc-client ），从而使SPI可用于应用程序。

以下代码片段显示了R2DBC SPI的用法INSERT和SELECT声明：

ConnectionFactory connectionFactory = null;

Mono<Integer> count = Mono.from(connectionFactory.create())
  .flatMapMany(it ->
    it.createStatement(
    "INSERT INTO legoset (id, name, manual) " +
    "VALUES($1, $2, $3)")
      .bind("$1", 42055)
      .bind("$2", "Description")
      .bindNull("$3", Integer.class)
      .execute())
  .flatMap(io.r2dbc.spi.Result::getRowsUpdated)
  .next();

Flux<Map<String, Object>> rows = Mono.from(connectionFactory.create())
  .flatMapMany(it -> it.createStatement(
    "SELECT id, name, manual FROM legoset").execute())
  .flatMap(it -> it.map((row, rowMetadata) -> collectToMap(row, rowMetadata)));

虽然上面的代码有点笨重，但R2DBC还附带了一个客户端库项目，用于更人性化的用户API。 R2DBC SPI不能直接使用，而是可以通过客户端库使用。

用R2DBC Client重写的相同代码是：

R2dbc r2dbc = new R2dbc(connectionFactory);

Flux<Integer> count = r2dbc.inTransaction(handle ->
  handle.createQuery(
    "INSERT INTO legoset (id, name, manual) " +
    "VALUES($1, $2, $3)")
    .bind("$1", 42055)
    .bind("$2", "Description")
    .bindNull("$3", Integer.class)
    .mapResult(io.r2dbc.spi.Result::getRowsUpdated));

Flux<Map<String, Object>> rows = r2dbc
  .inTransaction(handle -> handle.select(
    "SELECT id, name, manual FROM legoset")
  .mapRow((row, rowMetadata) -> collectToMap(row, rowMetadata));

注意collectToMap(…)以应用程序提供的函数为例，该函数将结果提取为所需的返回类型。

Spring Data团队创建Spring Data R2DBC作为孵化器，旨在通过数据库客户端提供反应性API并支持反应性存储库。用Spring Data R2DBC重写的示例代码为：

DatabaseClient databaseClient = DatabaseClient.create(connectionFactory);

Mono<Integer> count = databaseClient.execute()
  .sql(
    "INSERT INTO legoset (id, name, manual) " +
    "VALUES($1, $2, $3)")
  .bind("$1", 42055)
  .bind("$2", "Description")
  .bindNull("$3", Integer.class)
  .fetch()
  .rowsUpdated();

Flux<Map<String, Object>> rows = databaseClient.execute()
  .sql("SELECT id, name, manual FROM legoset")
  .fetch()
  .all();

R2DBC及其生态系统仍处于起步阶段，并要求进行实验和反馈以收集用例，并查看反应性关系数据库集成是否有意义。

光纤上的JDBC

让我们谈谈技术的结合。尽管JDBC和其他技术公开了阻塞的API（主要是由于等待I / O），但正在制作Project Loom 。织机介绍Fibers作为一种轻量级抽象，它将阻塞API变成非阻塞API。一旦调用到达阻塞的API，就可以通过堆栈切换来实现。所以底层Fiber尝试继续使用阻塞API的先前流程。

的Fiber执行模型大大减少了所需的本机线程数。结果是更好的可伸缩性和非阻塞行为–通过将阻塞调用卸载到Fiber支持Executor 。这里我们需要的是一个适当的API，该API允许在Fiber的实现。

结论

反应式编程和关系数据库的未来将如何发展？老实说，我不知道。如果我尝试进行有根据的猜测，则可以看到Project Loom和基于光纤的Executor与完善的JDBC驱动程序结合在一起，可以成为行业中潜在的游戏规则改变者。使用Java的加速发布节奏，这可能并不遥远。

ADBA的目标是将Java Standard运行时包含进来，根据目前的时间表，我预计Java标准运行时将不早于Java 17，即2021年的某个时候。

与现在可用的R2DBC相比。它带有驱动程序和客户端，并允许进行实验性使用。R2DBC的一个很好的副作用是，它公开了完全可响应的API，同时又独立于基础数据库引擎。随着版本的发布，您无需猜测Project Loom，也不必等待三年的时间来测试API。 R2DBC现在有可能。
