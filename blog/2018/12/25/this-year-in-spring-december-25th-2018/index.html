今年是有趣的一年，这一年见证了现有实践以非凡的新方式进行了改进。这是我会记得2018年的事情。

  • 响应式编程我们在此列表的其他项目中谈到了此方面的特定方面，但是，如果没有其他的话，2018年将是响应式的一年。在2018年，我们交付了所有有意义的集成的GA版本。在2018年，我们发布了Spring Boot 2和Spring Cloud Finchley，它们聚合了其他项目，如Spring Data Kay，Spring Security 5和Spring Framework 5。这些发行版在适当的情况下为阻塞和同步世界中的现有用例带来了奇偶校验。也是这一年，我们开始着手提供仅反应式的期权。 R2DBC和RSocket之类的东西超越了可能的广告，它们更好地基于响应式编程。
  • RSocket系统集成非常困难。整合很难。HTTP是用于文档检索的强大协议，但是它缺少每个开发人员都必须在HTTP谓词（REST）之上重新构想的应用程序级语义。它适用于请求-响应体系结构，但缺乏处理其他消息交换模式的能力，例如请求-响应，即发即忘和流（在请求，响应或两者中）。我们知道反应流规范为处理集成提供了一个很好的思维模型。在其中，边界被假定为异步和解耦的。组件通过背压概念支持流量控制。此外，HTTP是文本编码的。当然，您可以进行二进制数据处理，但是您需要对其进行编码。RSocket是一种新的二进制协议，内置了对来自Netflix和Facebook的人们的反应流规范的支持。它支持上述所有消息交换模式。由于它是一个协议，因此存在支持不同平台和语言（包括JavaScript，C ++和Java）的不同客户端库。Java实现自然基于Project Reactor 。我们在tentpole SpringOne Platform 2018大会上宣布了即将到来的Spring Framework 5.2版本中对RSocket的正式支持。
  • R2DBC当人们查看反应式编程时，我们总是会遇到一个问题：“我应该使用JDBC吗？”或“我可以使用JDBC吗？”或者，更直率地说，如果没有反应性JDBC选项，“我什至应该为反应性编程而烦恼吗？”答案总是令人沮丧，非常悲观：如果您需要在流中的任何一点阻塞以处理规模，您将不会获得反应式编程的夸大规模优势。如果最终扩展了新线程来扩展与JDBC的交互，那么您就限制了系统的扩展潜力。但这并不是说无法实现反应式SQL。它不是JDBC。对于那些其驱动程序本机支持异步IO和反应式编程的数据库供应商，即将出现的选择可能为JDBC提供一种非常有趣的替代方法。一种选择是R2DBC ，我们在tentpole会议SpringOne Platform 2018上宣布了这一点。集成分为三层：首先是较低级别的R2DBC SPI，在其之上是内置的数据库绑定。此时，存在针对H2，Microsoft SQL Server和PostgreSQL的绑定。在此级别上，您会发现DatabaseClient ，或多或少等于JdbcTemplate ，这使得使用底层R2DBC尽可能方便。然后，最后，在谚语堆栈的顶部是Spring Data R2DBC。我也在上周的Spring Tips系列中介绍了所有这些内容以及更多内容！
  • 如今 ， KNative Serverless这个词本身几乎毫无意义， 如今风靡一时。它描述了这个想法，我们应该让平台处理越来越多的将软件投入生产的工作。在功能即服务平台中，货币单位是功能。这是平台根据请求调用的。平台负责处理路由，调度，扩展和扩展等所有其他事情。一个功能的输出可以是另一功能的输入。这种组合意味着可以发生有趣的事情。可以将功能临时连接在一起以产生结果。此外，功能可以附加到事件源，在功能运行的任何平台上对功能生态系统中的事件做出响应。在这里，他们作为集成代码的能力变得显而易见。我们在Pivotal于2017年宣布了Riff项目 。正如我们最初设想的那样，它是建立在Kubernetes上的功能即服务平台。但这只是第一步。在我们在SpringOne Platform 2017活动中介绍它的几个月后，Riff团队开始与Google合作开发KNative 。KNative是Kubernetes的进一步抽象。它提供了内置原语，可以想象像Project Riff之类的东西。这正是我们所做的。在2018年，谷歌和Pivotal在Google NEXT上宣布了KNative，而我与曾荫权在Google Next上的联合演示中成为第一位做KNative演示的非Google员工。KNative不仅对Project Riff来说意义重大，而且对整个Kubernetes生态系统也意义重大，我很高兴看到它的去向。
  • Spring Cloud Supremacy今年巩固了Spring Cloud在面向开发人员的分布式系统工具包中的作用。Spring Cloud核心本身就变得被动了。它引入了像Spring Cloud Gateway这样的新特性。它还看到了针对Google Cloud的 Spring Cloud和针对Microsoft Azure的Spring Cloud的GA版本的发布。它还看到了阿里巴巴的Spring Cloud的首次亮相（这是明天的Spring Tips系列 ！）今年，我们比以往任何时候都看到大型组织转向微服务，甚至是像Netflix这样的先锋技术公司，它们已经集成了Spring Cloud， 今年在Spring Cloud上的收入翻了一番 。

确实是一年！明年已经有望变得更大，更好！这是2019年的包装。我们将于明年（2019年1月1日）见。

祝大家新年快乐！
