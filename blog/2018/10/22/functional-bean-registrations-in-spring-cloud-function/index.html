Spring Cloud Function has a couple of new features in 2.0 (still in milestone phase), and possibly the most dramatic is the ability to go "fully functional". This is made possible by changes in Spring Boot 2.1 together with Spring Framework 5.1, and it means a different way of thinking about bean definitions in Spring applications, but also significant improvements in startup performance.

AWS Cost Savings

It’s always good to start with a picture, especially if it tells a story. Here’s a graph that shows the improvement in Spring Cloud Function 2.0 over 1.0, comparing the cost of cold starts in AWS:

The x-axis is memory in MB, and the y-axis is cost of a cold start in GBsec. The most dramatic effect is for low memory containers, where the cost is almost 4 times lower with 2.0. The "Custom" function is even faster (10x over Spring Cloud Function 1.0) - it’s a custom AWS runtime using Spring Cloud Function with functional beans. The origin of the improvement is in dramatically shorter startup times, which in turn come from using the functional form of bean definitions in the application. Spring Cloud功能中的功能Bean注册

工程
戴夫·瑟
十月22，2018

Spring Cloud Function在2.0中有几个新功能（仍处于里程碑阶段），可能最引人注目的是“完全运行”的功能。通过在Spring Boot 2.1和Spring Framework 5.1中进行更改，可以实现这一点，这意味着在Spring应用程序中以不同的方式考虑bean定义，但也大大提高了启动性能。

AWS节省成本

从图片开始总是好事，特别是当它讲述一个故事时。下图显示了Spring Cloud Function 2.0在1.0之上的改进，并比较了AWS中冷启动的成本：

x轴是以MB为单位的内存，y轴是以GBsec为单位的冷启动成本。最显着的效果是对于低内存容器，其成本几乎是2.0的4倍。“自定义”功能甚至更快（比Spring Cloud Function 1.0快10倍）-它是使用带有功能Bean的Spring Cloud Function的自定义AWS运行时 。改进的根源在于大大缩短了启动时间，而启动时间又来自于在应用程序中使用bean定义的功能形式。如果您需要介绍的话，Josh做了一段关于功能bean注册的视频 （在YouTube上）。现在，让我们仔细看看它在Spring Cloud Function中的工作方式。

将功能与传统Bean定义进行比较

这是1.0版的Spring Cloud Function应用程序， @Configuration和@Bean声明风格：

@SpringBootApplication
public class DemoApplication {

  @Bean
  public Function<String, String> uppercase() {
    return value -> value.toUpperCase();
  }

  public static void main(String[] args) {
    SpringApplication.run(DemoApplication.class, args);
  }

}

您可以在AWS Lambda中运行它（例如），方法是将它与所有依赖项绑定在一起，然后将其上传到Amazon。项目中还支持Azure函数和Apache OpenWhisk 。其他无服务器提供程序，例如Oracle Fn和Riff ，维护它们自己的绑定。

您还可以通过以下方式在自己的HTTP服务器中运行上述应用程序： spring-cloud-function-starter-web在类路径上。运行main方法将公开可用于ping的终结点uppercase功能：

$ curl localhost:8080 -d foo
FOO

1.0中的Web适配器是使用Spring MVC实现的，因此您需要一个Servlet容器。在Spring Cloud Function 2.0中，您还可以使用Webflux，默认服务器为netty（即使您愿意，即使您仍可以使用Servlet容器），也可以将其包含在内。 spring-cloud-starter-function-webflux依赖。功能相同，并且两者都可以使用用户应用程序代码。

但是在2.0中，用户应用程序代码可以重铸为“功能”形式，如下所示：

@SpringBootConfiguration
public class DemoApplication
    implements ApplicationContextInitializer<GenericApplicationContext> {

  public static void main(String[] args) {
    FunctionalSpringApplication.run(DemoApplication.class, args);
  }

  public Function<String, String> uppercase() {
    return value -> value.toUpperCase();
  }

  @Override
  public void initialize(GenericApplicationContext context) {
    context.registerBean("demo", FunctionRegistration.class,
        () -> new FunctionRegistration<>(uppercase())
            .type(FunctionType.from(String.class).to(String.class)));
  }

}

主要区别在于：

  • 主班是ApplicationContextInitializer 。

  • 的@Bean方法已转换为对context.registerBean()

  • 的@SpringBootApplication已被替换为@SpringBootConfiguration表示我们没有启用Spring Boot自动配置，但仍将该类标记为“入口点”。

  • 的SpringApplication从Spring Boot已替换为FunctionalSpringApplication来自Spring Cloud Function（它是一个子类）。

您在Spring Cloud Function应用程序中注册的业务逻辑bean是类型FunctionRegistration 。这是一个包装，其中包含函数以及有关输入和输出类型的信息。在里面@Bean可以反射性地导出信息的应用程序形式，但是在功能性bean注册中，除非我们使用FunctionRegistration 。

替代使用ApplicationContextInitializer和FunctionRegistration是使应用程序本身实现Function （要么Consumer要么Supplier ）。示例（与上述等效）：

@SpringBootConfiguration
public class DemoApplication implements Function<String, String> {

  public static void main(String[] args) {
    FunctionalSpringApplication.run(DemoApplication.class, args);
  }

  @Override
  public String apply(String value) {
    return value.toUpperCase();
  }

}

添加一个单独的，独立类型的类也将起作用Function并在SpringApplication使用另一种形式的run()方法。最主要的是，泛型类型信息可在运行时通过类声明获得。

如果添加，该应用程序将在其自己的HTTP服务器中运行spring-cloud-starter-function-webflux （由于尚未实现嵌入式Servlet容器的功能形式，因此目前无法与MVC启动器一起使用）。该应用程序还可以在AWS Lambda或Azure Functions中正常运行，并且启动时间的改善是惊人的（如上图所示）。这是另一张图中的启动时间（y轴上的启动时间以秒为单位）：

测试功能应用

Spring Cloud Function 2.0还具有一些用于集成测试的实用程序，Spring Boot用户将非常熟悉。例如，这是包装以上应用程序的HTTP服务器的集成测试：

@RunWith(SpringRunner.class)
@FunctionalSpringBootTest
@AutoConfigureWebTestClient
public class FunctionalTests {

	@Autowired
	private WebTestClient client;

	@Test
	public void words() throws Exception {
		client.post().uri("/").body(Mono.just("foo"), String.class)
                    .exchange().expectStatus().isOk()
                          .expectBody(String.class).isEqualTo("FOO");
	}

}

该测试几乎与您要编写的测试相同@Bean同一应用的版本-唯一的区别是@FunctionalSpringBootTest注释，而不是常规@SpringBootTest 。所有其他作品，例如@AutowiredWebTestClient是标准的Spring Boot功能。

主流Spring Boot应用程序中的功能Bean定义

Spring Boot可以在功能性bean注册中正常工作-Spring Cloud Function在Spring Boot上构建并运行-但是Spring Boot中一些最有用的功能（自动配置）均以非功能性样式编码。与整个Spring Boot相比，大多数Spring Cloud Function应用程序的范围相对较小，因此我们能够轻松地使其适应这些功能Bean定义。如果您超出了有限的范围，则可以通过切换回来扩展Spring Cloud Function应用程序@Bean样式配置，或使用混合方法。将相同的功能扩展到Spring Boot生态系统的其余部分将需要更长的时间，但这是我们正在积极努力的工作。请试用Spring Cloud Function 2.0，如有需要，请向我们提供一些反馈-GA版本即将发布。
