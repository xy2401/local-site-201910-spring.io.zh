春赋的演变

工程
塞巴斯蒂安·德勒兹（SébastienDeleuze）
十月02，2018

我借此机会在SpringOne平台之间短暂停留，在那里我首次谈到了Spring Fu和Kotlinconf ，以概述该项目的发展，总结当前状态并分享下一步的可能。

6月初，我宣布了一个名为Spring Fu的新实验项目，目的是尝试一种新型API，以使用Kotlin DSL和功能配置来配置Spring应用程序。

今天，我很自豪地宣布一个新的实验项目：Spring Fu。这是一个@Kotlin微框架，可以轻松地使用功能性API（而不是注释）创建轻量级的Spring驱动的应用程序。我们正在寻找反馈。 https://t.co/R15wJ1gD8K pic.twitter.com/ScljoPZ8rW

—SébastienDeleuze（@sdeleuze） 2018年6月8日

我必须承认，我并不期望随之而来的大量反馈，我要感谢Spring社区的热烈欢迎。此后，我一直在从事该项目，以将基于原始Spring Framework API的第一个POC转变为新的Spring功能特性的孵化器。

Kofu配置

现在基于Spring启动基础设施科特林DSL，被称为甲府（ 柯为和tlin 福 nctional）。它允许使用Kotlin DSL和lambda代替注释来配置Spring Boot应用程序，并且具有以下特征：
-通过Kotlin DSL进行显式配置
-基于功能性使用方式的Spring Boot基础架构
-未基于类路径检测启用任何功能
-声明式和程序化
-更快的启动和更低的内存消耗
-最少的反射和注释使用
-纯lambda，无CGLIB代理

配置有Kofu配置的典型Spring Boot应用程序如下所示：

val app = application {
  import(beans)
  listener<ApplicationReadyEvent> {
    ref<UserRepository>().init()
  }
  properties<SampleProperties>("sample")
  server {
    port = if (profiles.contains("test")) 8181 else 8080
    mustache()
    codecs {
      string()
      jackson {
        indentOutput = true
      }
    }
    import(::routes)
  }
  mongodb {
    embedded()
  }
}

val beans = beans {
  bean<UserRepository>()
  bean<UserHandler>()
}

fun routes(userHandler: UserHandler) = router {
  GET("/", userHandler::listView)
  GET("/api/user", userHandler::listApi)
  GET("/conf", userHandler::conf)
}

fun main() = app.run()

为了使用它，您“只是”必须添加一个org.springframework.fu:spring-boot-kofu依赖于Spring Boot 2.1应用程序。作为当前版本号0.0.2建议这样做，请注意，目前该API尚不稳定，不适合生产，并且范围仅限于Spring Boot支持的子集。

但是，您可以随意使用它，发送反馈并尝试这种配置Spring Boot应用程序的新方法，该方法可以正常运行，可以通过IDE自动完成并记录下来 。

Kofu不会比自动配置更好或更差，只是有所不同。我相信这可能是Spring Boot吸引那些更喜欢更明确的配置模型的开发人员以及来自其他背景（例如Kotlin，Go，Node或Ruby）的开发人员的一种方式。

Jafu配置

最初仅限于科特林，我收到来自兴趣在这个明确的DSL方法以及Java开发人员的主要反馈之一，所以我曾在一个Java当量，结束了这个Jafu（JA为VA和福 nctional）DSL。

public class JafuApplication {

  public static SpringApplication app = application(app -> {
    app.beans(beans -> {
      beans.bean(SampleService.class);
      beans.bean(SampleHandler.class);
    });
    app.server(server -> server.router(router -> {
      SampleHandler sampleHandler = app.ref(SampleHandler.class);
      router.GET("/", sampleHandler::hello);
      router.resources("/**", new ClassPathResource("static/"));
    }));
  });

  public static void main (String[] args) {
    app.run(args);
  }
}

目前这只是一个POC，但我打算在不久的将来与Kofu达到功能对等，并并行开发2个DSL。缺少类型安全的构建器 ， 可靠的 类型参数或扩展机制将使Jafu比Kofu更加冗长且扩展性较差，但是尽管有这些限制，但我发现Jafu非常好用。

对于仅对与功能Bean注册基础结构的使用相关的性能提高感兴趣的用户而言，值得注意的是， Dave Syer当前正在试验一些解决方案，这些解决方案可以利用基于以下功能的常规Spring Boot应用程序利用功能Bean注册效率：注释。

将Spring应用程序作为本机可执行文件运行

GraalVM是Oracle开发的一种新虚拟机，除其他功能外，它还可以通过Substratevm将JVM字节码编译为本机可执行文件。

Spring Framework 5.1为GraalVM本机映像提供了一些初始支持 ，但我们确实是故事的开始。GraalVM团队需要解决Dave提出的各种问题，以使一切正常工作，而生态系统需要适应这个具有不同约束和特征的新平台。

但是GraalVM团队正在听取我们的反馈，最近几个月来，Spring应用程序支持取得了长足的进步。已经可以将带有Kofu配置的基本Spring Boot反应式应用程序编译为几乎可以立即运行的本机可执行文件！

协程支持

如前所述，Spring Fu的主要目标是孵化将集成到当前顶级项目（如Spring Framework，Spring Data和Spring Boot）中的功能。

Spring Fu当前正在孵化对Spring WebFlux和Spring Data的协程支持，以便能够以更命令式的方式利用Spring Reactive堆栈。这主要针对希望利用此类堆栈的可伸缩性而不需要反应式API的所有功能的开发人员。

class UserRepository(private val mongo: CoroutinesMongoTemplate) {
	suspend fun count(): Long = mongo.count<User>()
	suspend fun findAll(): List<User> = mongo.findAll<User>()
	suspend fun findOne(id: String): User? = mongo.findById<User>(id)
	suspend fun deleteAll() = mongo.remove<User>()
	suspend fun save(user: User): User? = mongo.save(user)
}

请注意，尽管协程从Kotlin 1.3开始被认为是最终的，但仍然存在重大遗漏之处kotlinx-coroutines因为它不提供任何类型来处理冷流。我们将需要这种缺少的抽象，以便能够使用协程API公开我们的反应性基础，有关更多详细信息，请参阅kotlinx.coroutines＃254 。

注意，我们应该能够在协程和反应堆类型之间传递上下文，以允许使用功能强大的用例，例如反应式安全性和事务。

Konrad Kaminski通过他的出色的spring-kotlin-协程项目贡献了最初的Spring coroutines支持，不久将加入Spring Fu与我一起进行此功能。
