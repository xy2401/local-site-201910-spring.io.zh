On behalf of the team, I am pleased to announce the general availability of Spring Cloud Data Flow 1.3 across a range of platforms

Follow the Getting Started guides for Local Server, Cloud Foundry, and Kubernetes

Release Highlights

Stream updates and rollback

A streaming data pipeline orchestrated as a series of microservice applications has always been the core value of Spring Cloud Data Flow’s design. In Data Flow 1.3 we have provided the ability to update sources, processors, and sinks independently without having to undeploy and redeploy the entire stream.

The stream update and rollback functionality is implemented by delegating the deployment process to a new Spring Cloud project called Skipper. Skipper is a lightweight Spring Boot application, purpose-built to fill this feature gap in Data Flow. Skipper defines a package format, much like helm or brew and can also deploy/undeploy applications to multiple cloud platforms: Local, Cloud Foundry, and Kubernetes. It uses the same Spring Cloud Deployer libraries that have been part of Data Flow since the beginning. Recent presentations at SpringOne 2017 introduces Skipper and the integration with Data Flow in more depth.

When deploying a Stream, Data Flow creates Skipper package describing the Stream and the applications that are part of the Stream definition. Skipper then deploys the applications to the desired platform. When requesting a stream update, only the application or applications that need to be changed are automatically redeployed. A simple strategy managed by a Spring Statemachine instance performs the update or rollback steps.

Data Flow includes new stream commands to make upgrade and rollback operations.

dataflow:>app register --name transform --type processor --uri maven://com.eg:transformer:0.0.1

dataflow:>stream create mystream --definition "jdbc | transform | mongodb"

dataflow:>app register --name transform --type processor --uri maven://com.eg:transformer:0.0.2

dataflow:>stream update mystream --properties “version.transform=0.0.2”

dataflow:>stream rollback mystream

In this series of commands, the stream is deployed using version 0.0.1 of the transformer. The jdbc and mongodb source and sink are already registered. The stream is then updated to use version 0.0.2 of the transformer. Only the transform application is updated, with version 0.0.2 being deployed and version 0.0.1 being undeployed. The jdbc and mongodb applications are left as-is. The rollback command does the opposite, bringing the stream back to the state with version 0.0.1 of the transformer.

Note: To use Data Flow and Skipper, Data Flow’s feature toggle for Skipper must be enabled in both the Data Flow Server and shell.

Spring Cloud Data Flow 1.3 GA发布

发布
冈纳·希尔特（Gunnar Hillert）
2018年2月1日

我很荣幸地代表团队宣布Spring Cloud Data Flow 1.3在各种平台上的普遍可用性

遵循本地服务器 ， Cloud Foundry和Kubernetes的入门指南

发布要点

流更新和回滚

精心策划为一系列微服务应用程序的流数据管道一直是Spring Cloud Data Flow设计的核心价值。在Data Flow 1.3中，我们提供了独立更新源，处理器和接收器的功能，而不必取消部署和重新部署整个流。

通过将部署过程委托给一个名为Skipper的新Spring Cloud项目来实现流更新和回滚功能。Skipper是一个轻量级的Spring Boot应用程序，旨在填补Data Flow中的这一功能空白。Skipper定义了一种包格式，非常类似于helm或brew并且还可以将应用程序部署/取消部署到多个云平台：本地，Cloud Foundry和Kubernetes。自开始以来，它就使用与数据流相同的Spring Cloud Deployer库。SpringOne 2017上的最新演讲更深入地介绍了Skipper以及与Data Flow的集成 。

部署Stream时，Data Flow会创建Skipper程序包，以描述Stream和属于Stream定义的应用程序。然后，Skipper将应用程序部署到所需平台。请求流更新时，仅自动重新部署一个或多个需要更改的应用程序。一个由Spring Statemachine实例管理的简单策略执行更新或回滚步骤。

数据流包括新的流命令以进行升级和回滚操作。

dataflow:>app register --name transform --type processor --uri maven://com.eg:transformer:0.0.1

dataflow:>stream create mystream --definition "jdbc | transform | mongodb"

dataflow:>app register --name transform --type processor --uri maven://com.eg:transformer:0.0.2

dataflow:>stream update mystream --properties “version.transform=0.0.2”

dataflow:>stream rollback mystream

在这一系列命令中，使用转换器的0.0.1版部署流。jdbc和mongodb源和接收器已经注册。然后，将流更新为使用转换器的0.0.2版。仅更新了transform应用程序，已部署版本0.0.2和未部署版本0.0.1。jdbc和mongodb应用程序保持mongodb 。rollback命令执行相反的操作，将流返回到转换器0.0.1版的状态。

注意：要使用Data Flow和Skipper，必须在Data Flow Server和外壳程序中同时启用Skipper的Data Flow 功能切换 。

JavaDSL

DataFlowTemplate类一直是以编程方式部署流和任务的主力军。但是，它是一个相当低级的API。我们添加了一个新的流利样式API，以创建，部署或启动流，该API易于使用，并且可以跨多个流重用StreamApplication实例。

StreamApplication source = new StreamApplication("http")
        .addProperty("server.port", 9900);

StreamApplication processor = new StreamApplication("filter")
        .addProperty("expression", "payload=='good'");

StreamApplication sink = new StreamApplication("log");

Stream simpleStream = streamBuilder.name("simpleStream")
        .source(source)
       	.processor(processor)
        .sink(sink)
       	.create()
        .deploy();

使用Stream实例，您可以询问流的状态，取消部署或销毁该流。

参考指南的Java DSL部分提供了有关DSL的更完整介绍，而Spring Data Flow Samples存储库提供了一个示例，供您开始在项目中使用它。

角4

数据流仪表板已更新，以利用Angular 4并与Pivotal UI样式保持一致。主要重点是域模型类与JSON直接字符串的一致使用。这样可以对应用程序的状态进行更细粒度的控制，例如，从分页列表切换到详细信息页面并返回时。使用Compodoc 的UI也有改进的文档 。

扇入和扇出可视化

扇入是指多个源都将数据发送到同一消息传递目标时。扇出是指在运行时确定消息传递目的地。该视频显示了具有这些拓扑的流的实际用户界面。

有一个新的UI控件可以直接从特定节点中分支出来以“点击流”，而另一个新的UI控件可以管理命名目的地 。这些新功能使构建复杂的流拓扑更加容易。

应用程序注册表改进

将应用程序注册为Maven工件的用户现在可以利用“更新策略”功能来覆盖和刷新Spring Cloud Data Flow的内部Maven缓存。举例来说，在发展中，可以连续通过设置Maven构件的决心快照版本的update-policy=always ，这将迫使的流或批处理/任务应用程序的最新版本下载这是在DSL /仪表板使用。

根据用户反馈，使用http资源注册的应用程序将始终被下载而不进行缓存。这促进了开发生命周期中更新托管在Web服务器上的应用程序uberjar的代码而不是名称的开发生命周期。

在Skipper模式下， 可以注册多个应用程序版本 。部署流时使用默认版本。您可以使用新命令app default设置默认版本。但是，在流中升级应用程序版本时，必须首先在数据流中注册它。

外壳改进

此版本为流和任务/批处理名称以及其他元数据添加了“自动补全”。不用再猜测了-一切都在TAB之外！请查看以下截屏视频，以了解有关高级Shell功能，技巧和窍门的更多信息。

功能赛跑者

通过使用功能运行器应用程序，可以为SCDF中的运行功能提供初始支持。使用Spring Cloud Function应用程序创建流时，您传入函数的类名和jar位置。

dataflow:> stream create foo --definition "http | function-runner --function.className=com.example.functions.CharCounter --function.location=file:///home/john/myfunction.jar | log"

您可以使用一个示例来试用此功能。由于不需要明确的用户调用简化春季云功能的部署function-runner是我们的路线图。

应用领域

MQTT源和MQTT接收器应用程序可以与IoT设备进行交互。

TensorFlow处理器可以帮助进行实时预测模型评估。查看对它的Twitter情感模型预测，以了解一个想法。

为了改进数据科学功能，现在还提供Python-HTTP和Python-Jython处理器。

新的bit.ly链接为摄氏。SR1和克拉克。GA可以批量导入并在SCDF中注册所有开箱即用的应用程序。

由于不需要明确的用户调用简化春季云功能的部署function-runner是我们的路线图。

用于PCF的Spring Cloud Data Flow Tile

在过去的几个月中，Spring Cloud Data Flow的Cloud Foundry磁贴一直处于关闭状态。我们已经对客户和现场反馈进行了迭代，并将正式从BETA升级到1.0 GA状态。此版本将自动配置（包括指标收集器，跳过程序，数据库和消息代理）以及Cloud Foundry中的端到端OAuth / SSO集成自动化。还有许多其他增值，敬请关注有关Pivotal Network中平铺页面的更集中的讨论， 文档和指导。

Kubernetes的舵图

合并pull-request后， Spring Cloud Data Flow的掌舵图将更新为最新的1.3 GA版本。使用此图表，可以使用以下helm命令自动配置最新版本的SCDF及其配套组件（度量收集器，跳过程序，数据库和消息代理）。

helm repo add incubator http://storage.googleapis.com/kubernetes-charts-incubator
helm repo update
helm install --name scdf incubator/spring-cloud-data-flow --set rbac.create=tru
