Spring Cloud Data Flow 1.7 M1发布

发布
马克·波拉克
九月18，2018

Spring Cloud Data Flow团队很高兴宣布发布1.7 M1 。遵循本地服务器 ， Cloud Foundry和Kubernetes的入门指南 。

这是亮点

  • 改进的用户界面

  • 流应用DSL

  • 审计追踪

  • 并发任务启动限制

  • 流和任务验证

  • 强制升级流

改进的用户界面

用户界面具有全新的外观。导航已从选项卡移至左侧导航系统。这样可以增加使用Flo Designer创建流的屏幕空间，并且可以通过最小化左侧导航来获得更多屏幕空间。有一个快速搜索功能，可以搜索所有不同的数据流类别。添加了其他颜色和整体主题更改，以使UI看起来更生动。更深入的核心是，路由管理得到了改进，并且我们使用BrowserStack / SauceLabs增加了端到端测试的覆盖范围。

流应用DSL

并不是所有的用例都可以通过建立线性流水线来解决，其数据流从源流到处理器再流到接收器，并且只有一个目标连接每个应用程序。一些用例需要具有多个输入和输出的应用程序集合。使用用户定义的绑定接口在Spring Cloud Stream中支持此拓扑，但在Data Flow中不支持此拓扑。在Kafka Streams应用程序中通常会有多个输入和输出。

此外，使用Spring Cloud Stream应用程序并不能解决所有用例。可以仅使用Spring Integration来编写将同步请求/答复消息发送到Kafka或RabbitMQ应用程序的http网关应用程序。

在这些情况下，数据流无法假设从一个应用程序到另一个应用程序的数据流，因此无法像使用Stream Pipeline DSL一样设置应用程序的目标属性。

为了解决这些用例，我们引入了流应用DSL 。该DSL使用comma ，而不是pipe symbol ，以指示数据流不应配置应用程序的绑定属性。相反，开发人员需要设置适当的部署属性以“连接”应用程序。使用EIP Cafe示例的域的DSL 示例是

dataflow:> stream create --definition "orderGeneratorApp, baristaApp, hotDrinkDeliveryApp, coldDrinkDeliveryApp" --name myCafeStream

DSL中列出的应用程序需要注册为的位置--type app 。

在此流中， baristaApp有两个输出目的地打算由hotDrinkDeliveryApp和coldDrinkDeliveryApp分别。部署流时，请设置目标属性，以使目标与所需的数据流匹配，例如：

app.baristaApp.spring.cloud.stream.bindings.hotDrinks.destination=hotDrinksDest
app.baristaApp.spring.cloud.stream.bindings.coldDrinks.destination=coldDrinksDest
app.hotDrinkDeliveryApp.spring.cloud.stream.bindings.input.destination=hotDrinksDest
app.coldDrinkDeliveryApp.spring.cloud.stream.bindings.input.destination=coldDrinksDest

审计追踪

为了帮助回答“谁在何时何地做了什么？引入了审核跟踪，以存储所采取的与应用程序注册，时间表，流和任务有关的操作。对于应用程序和计划，将审核创建和删除操作。对于流，将审核创建，删除，部署，取消部署，更新和回滚。对于任务，将对创建，启动和销毁进行审核。审核信息可在UI中查询。即将在Shell中访问审核信息。

并发任务启动限制

Spring Cloud Data Flow允许您强制执行最大数量的并发运行任务，以防止计算资源饱和。可以通过设置spring.cloud.dataflow.task.maximum-concurrent-tasks属性。预设值为20。您还可以通过REST端点检索当前同时执行的任务数/tasks/executions/current 。一个新的tasklauncher-dataflow应用程序利用此功能仅在并发任务数少于最大任务数时启动任务。该功能也是正在开发的新FTP摄取示例应用程序的中心。面向数据密集型应用程序的云原生模式提供了一个秘密峰会

流和任务验证

新的Shell命令stream validate和task validate将验证流或任务应用程序资源有效且可访问。这样可以避免在部署时获取异常。即将使用UI进行验证。

强制升级流

用力！ 升级流时，您现在可以使用该选项--force即使没有更改应用程序或部署属性，也可以部署当前已部署应用程序的新实例。如果应用程序本身在启动时（例如从Spring Cloud Config Server）获取配置信息，则需要这种行为。您可以使用选项指定要强制升级的应用程序--app-names 。如果未指定任何应用程序名称，则所有应用程序将被强制升级。您可以指定--force和--app-names选项以及--properties要么--propertiesFile选项。
