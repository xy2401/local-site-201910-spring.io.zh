Spring Data Lovelace有什么新功能？

工程
奥利弗·德罗博姆（Oliver Drotbohm）
九月27，2018

随着Spring Data Lovelace上周以其普遍可用的版本发布，是时候简要介绍一下我们添加的新功能了。发行列车具有很多功能。在此博客文章中，我将介绍更一般的文章。以下博客文章介绍了特定于商店的高级新闻：

  • Jens Schauder介绍Spring Data JDBC

  • Jens再次讨论如何使用Spring Data JDBC对聚合建模

  • Christoph Strobl在MongoDB上

  • Mark Paluch在Apache Cassandra和Redis的Spring Data上

阅读那些吗？好了，那么让我们开始更一般的项目。

改进了不可变类型的对象映射

借助Spring Data Lovelace，我们有机会对大多数NoSQL模块使用的对象映射功能进行了改进。除了微小的方面，我们一直支持不可变的实体：在数据库交互过程中需要更改的值，例如自动生成的标识符或乐观的锁定版本属性，总是在幕后进行了更改。最新版本已对此进行了更改。

为了支持这些属性，我们现在需要您提供一种凋枯方法（遵循with…(…)命名约定），或者-在Kotlin世界中-公开专用copy(…)方法，以所有属性为参数。以下示例使用withId方法：

class Person {

  private final Long id;
  private final String firstname, lastname;

  static Person of(String firstname, String lastname) {
    return new Person(null, firstname, lastname);
  }

  private Person(Long id, String firstname, String lastname) {
    // Assign parameters to fields
  }

  Person withId(Long id) {
    return new Person(id, this.firstname, this.lastame);
  }

  // …
}

使用Lombok，我们可以执行以下操作：

@Value(staticConstructor = "of")
class Person {

  @Wither Long id;
  String firstname, lastname;

  // …
}

在以下情况下使用声明的构造函数Person实例是从数据库中读取的。首先将新实体持久存储到商店时，标识符属性的凋谢方法起作用。确保您确实使用了从存储库的返回的结果save(…)方法，因为这将是与您最初交给该方法的实例不同的实例。

我们使用了这方面的工作，通过在使用不可变类型时避免对属性进行几次不必要的迭代，从而提高了某些商店实现中实际对象转换的性能。有关更多详细信息，请参见参考文档的新增部分 。

支持Vavr的Try Monad

默认情况下，Spring Data存储库传播在执行查询方法期间发生的错误（作为运行时异常）。支持编写Java代码的更具功能性的风格的支持者通常在其应用程序中使用Vavr。我们已经支持Vavr的集合类型作为存储库返回类型已有相当长的一段时间了。Spring Data Lovelace现在增加了对Try monad，以便查询方法执行中发生的错误导致Failure要返回的实例，而不是引发异常。

interface PersonRepository extends Repository<Person, Long> {

  Try<Option<Person>> findByLastnameContaining(String lastname);
}

如果可以找到一个以上的人，则该方法将失败lastname符合给定条件。用Try用作返回类型时，客户端组件现在可以从异常情况中恢复，如下所示：

repository.findByLastnameContaining("e")
  .recover(it -> Match(it).of(
    Case(instanceOf(IncorrectResultSizeDataAccessException.class), Optional.empty())
));

尽管在这个人为的微小示例中看起来有些费解，但是它是一种非常强大的机制，可以在可能引发异常的代码上继续映射潜在的可用结果。请注意，将事务存储与此功能结合使用时，您必须自己管理事务回滚，因为实际上不再抛出异常，并且不会触发以下事务的事务回滚@Transaction带注释的方法。对它的支持正在发展。确保在Spring Framework中遵循此票证 ，甚至对它进行投票。

在行动中看到的Vavr支持，检查出的例子在我们的规范春数据仓库的例子。

延迟的JPA资料库初始化

启动Spring Data JPA应用程序需要一个EntityManagerFactory被引导。这通常会消耗整个应用程序的启动时间。为了解决这个问题，Spring Framework已经引入了引导功能EntityManagerFactory在后台线程中，因此，让其他bean并行初始化。不幸的是，Spring Data存储库通常至少对某种类型的Spring bean具有传递性依赖。反过来，它们的初始化已经与工厂进行了交互，这无需进一步的步骤即可导致其他组件的初始化受阻。这极大地限制了该新功能的效果。

从Spring Data Lovelace开始，您现在可以将存储库定义为以惰性模式引导，该模式会自动将Spring Data存储库的所有注入点声明为惰性。这将导致Spring Framework创建代理并将延迟实际存储库初始化，直到与它们第一次交互。换句话说，可以依次创建存储库客户端和这些客户端的客户端，而不必等待EntityManagerFactory完成引导程序。这意味着可以将大量的Spring bean初始化并行化到JPA引导程序。

初始化的延迟还会导致任何形式的验证被延迟，直到（例如）第一个命中实际使用存储库的代码路径的请求。如果仍要确保在应用程序开始接收请求之前已完全初始化存储库，请使用延迟的引导程序模式。此模式的工作方式与惰性模式相同，不同之处在于，当容器完成自举时，它将显式触发存储库中所有存储库的初始化。 ApplicationContext 。

在这些模式之间切换的最简单方法是设置Spring Boot属性spring.jpa.repositories.bootstrap-mode到lazy要么deferred 。这也可以确保延迟初始化EntityManagerFactory配置正确。另外，请务必检查出添加到参考文档中的新部分。

在Spring Data REST中控制HTTP方法公开

最后，我们看一看Spring Data REST中一段时间以来所请求的功能。在Spring Data REST中，默认情况下，由声明的CRUD方法决定是否公开哪种资源类型支持哪种HTTP方法。但是，在某些情况下，单个存储库方法的存在会导致启用多种资源-HTTP-方法组合。例如， save(…)存储库中公开的方法导致POST在收集资源以及PUT用于更新和创建项目资源。

Spring Data Lovelace有了新的ExposureConfiguration （可从RepositoryRestConfiguration ），可以对应该公开哪些方法进行精细控制：

ExposureConfiguration config = restConfiguration.getExposureConfiguration();

config.withItemExposure((metadata, httpMethods)
  -> httpMethods.disable(HttpMethod.PATCH));
config.forDomainType(User.class).disablePutForCreation();

如您所见，配置选项有两种形式：

  • 全局（首先显示），它允许通过在容器上使用lambda来控制集合，项目和关联资源的暴露ResourceMapping和ConfigurableHttpMethods 。

  • 特定于类型的一个，如下所示。某些设置详细信息，例如禁用（默认情况下启用）通过使用创建实体的功能PUT显式配置方法支持请求。
