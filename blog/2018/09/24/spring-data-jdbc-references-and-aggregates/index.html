Spring Data JDBC，参考和聚合

工程
延斯·肖德（Jens Schauder）
九月24，2018

在我以前的博客文章中 ，我描述了如何设置和使用Spring Data JDBC。我还描述了使Spring Data JDBC比JPA更易于理解的前提。一旦考虑了引用，这将变得很有趣。作为第一个示例，请考虑以下域模型：

class PurchaseOrder {

  private @Id Long id;
  private String shippingAddress;
  private Set items = new HashSet<>();

  void addItem(int quantity, String product) {
    items.add(createOrderItem(quantity, product));
  }

  private OrderItem createOrderItem(int quantity, String product) {

    OrderItem item = new OrderItem();
    item.product = product;
    item.quantity = quantity;
    return item;
  }
}
class OrderItem {
  int quantity;
  String product;
}

此外，考虑定义如下的存储库：

interface OrderRepository extends CrudRepository {

  @Query("select count(*) from order_item")
  int countItems();
}

如果您创建包含商品的订单，则可能希望所有商品都将保留下来。而这正是发生的情况：

@Autowired OrderRepository repository;

@Test
public void createUpdateDeleteOrder() {

  PurchaseOrder order = new PurchaseOrder();
  order.addItem(4, "Captain Future Comet Lego set");
  order.addItem(2, "Cute blue angler fish plush toy");

  PurchaseOrder saved = repository.save(order);

  assertThat(repository.count()).isEqualTo(1);
  assertThat(repository.countItems()).isEqualTo(2);
  …

另外，如果删除PurchaseOrder ，则其所有项目也将被删除。再次，就是这样。

  …
  repository.delete(saved);

  assertThat(repository.count()).isEqualTo(0);
  assertThat(repository.countItems()).isEqualTo(0);
}

但是，如果我们考虑语法上相同但语义上不同的关系该怎么办？

class Book {
  // …
  Set authors = new HashSet<>();
}

书籍绝版后，您将其删除。所有的作者都走了。当然不是您想要的，因为有些作者可能还写了其他书。现在，这没有任何意义。还是呢？我认为是的。

为了理解为什么这样做有意义，我们需要退后一步，看看哪些存储库实际存在。这与一个又一个反复出现的问题密切相关： 在JPA中，每个表是否应该有一个存储库？

正确和权威的答案是“否”。存储库将持久保存并加载聚合。聚集是形成一个单元的对象簇，应该始终保持一致。同样，它应该始终保持在一起（并加载）。它具有一个称为聚合根的对象，这是唯一可以接触或引用聚合内部的对象。聚合根是传递给存储库以持久化聚合的内容。

这就提出了一个问题：Spring Data JDBC如何确定集合中的哪些部分，什么不是？答案很简单：通过遵循非临时引用，您可以从聚合根获得的所有内容都是聚合的一部分。

考虑到这一点， OrderRepository的行为非常OrderRepository 。 OrderItem实例是聚合的一部分，因此被删除。 相反， Author实例不属于Book汇总，因此不应删除。因此，不应仅仅从Book类中引用它们。

问题解决了。好吧，……不是真的。我们仍然需要存储和访问有关Book和Author之间的关系的信息。答案可以再次在域驱动设计（DDD）中找到，该建议建议使用ID代替直接引用。这适用于各种多对多关系。

如果多个集合引用同一个实体，则该实体不能成为引用该实体的那些集合的一部分，因为它只能是一个集合的一部分。因此，任何多对一和多对多关系都必须仅通过引用id来建模。

如果应用此方法，则可以实现多个目标：

  1. 您清楚地指出了聚合的边界。

  2. 您还可以完全分离（至少在应用程序的域模型中）所涉及的两个聚合。

  3. 可以通过不同的方式在数据库中表示这种分离：

    1. 保留数据库的正常状态，包括所有外键。这意味着您必须确保以正确的顺序创建和保留聚合。

    2. 使用延迟约束，这些约束仅在事务的提交阶段进行检查。这可以实现更高的吞吐量。它还将最终一致性的版本编码为“最终”与事务结束相关的版本。这也允许引用不存在的聚合，只要它仅在事务期间发生即可。这对于避免仅为了满足外键和非null约束而避免大量基础结构代码可能很有用。

    3. 完全删除外键，以确保最终的一致性。

    4. 将引用的聚合持久保存在其他数据库中，甚至可能没有SQL存储。

无论您采取何种分离措施，即使由Spring Data JDBC强制实施的最小分离方案也都鼓励应用程序模块化。另外，如果您尝试迁移真正的整体式10年历史的应用程序，那么您将了解其价值。

使用Spring Data JDBC，您可以像这样对多对多关系进行建模：

class Book {

  private @Id Long id;
  private String title;
  private Set authors = new HashSet<>();

  public void addAuthor(Author author) {
    authors.add(createAuthorRef(author));
  }

  private AuthorRef createAuthorRef(Author author) {

    Assert.notNull(author, "Author must not be null");
    Assert.notNull(author.id, "Author id, must not be null");

    AuthorRef authorRef = new AuthorRef();
    authorRef.author = author.id;
    return authorRef;
  }
}
@Table("Book_Author")
class AuthorRef {
  Long author;
}
class Author {
  @Id Long id;
  String name;
}

请注意额外的类（ AuthorRef ），它表示Book汇总中有关作者的知识。它可能包含有关作者的其他汇总信息，然后这些信息实际上将在数据库中重复。考虑到作者数据库可能与书本数据库完全不同，这带来了很多事情。

另外，请注意，作者集是一个私有字段， AuthorRef实例的实例化是在私有方法中进行的。因此，聚合之外的任何内容都无法直接访问它。Spring Data JDBC绝对不需要这样做，但是DDD鼓励这样做。该域将像这样使用：

@Test
public void booksAndAuthors() {

  Author author = new Author();
  author.name = "Greg L. Turnquist";

  author = authors.save(author);

  Book book = new Book();
  book.title = "Spring Boot";
  book.addAuthor(author);

  books.save(book);

  books.deleteAll();

  assertThat(authors.count()).isEqualTo(1);
}

总结一下：Spring Data JDBC不支持多对一或多对多关系。为了对它们建模，请使用ID。这鼓励域模型的干净模块化。它还消除了一整套问题，必须解决这些问题并学习推理出是否可能进行这种映射。

用类似的思路，避免双向依赖。集合内部的引用从集合根到元素。集合之间的引用由一个方向上的ID表示。另外，如果需要反向浏览，请在存储库中使用查询方法。这样就可以清楚地知道哪个聚合负责维护参考。
