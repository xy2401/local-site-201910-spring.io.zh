动手使用Spring Vault 2.1

工程
马克·帕鲁奇
九月24，2018

Spring Vault 2.1已经面世。我想通过这篇文章深入了解即将发布的版本中的更改和功能。

团队一直在研究一系列新功能：

  • 扩展基于基础架构的身份验证以支持Google Cloud IAM和Azure托管服务身份

  • 集成Vault的版本化键值后端

  • 包装API支持

  • Java 11兼容性

Spring Vault支持HashiCorp Vault版本0.5到0.11。您可以在GitHub上找到Spring Vault和Spring Cloud Vault 示例库 。现在，让我们深入了解Spring Vault 2.1功能！

Google Cloud身份验证

在0.8.1版中，保险柜引入了对Google Cloud的身份验证支持。GCP身份验证使用GCP的IAM服务来使用以下身份验证工作流程之一：

  • 使用服务帐户凭据进行IAM登录以生成签名令牌

  • 使用GCP的元数据服务进行GCE登录以检索签名的令牌

Spring Vault支持两种方法。使用IAM身份验证需要设置凭据（通过环境或使用凭据文件），而GCE身份验证将平台用作身份提供者，因此GCE身份验证对初始设置的要求较少。

IAM登录

IAM身份验证使用Google的IAM服务，通过使用来自Google凭据的签名来生成签名的令牌。签名的令牌将传递到Vault以验证令牌。此身份验证方法要求通过以下方式提供凭据： GOOGLE_APPLICATION_CREDENTIALS环境变量或通过配置GcpIamAuthenticationOptions 。 GcpIamAuthentication使用Google的服务API SDK（ google-api-services-iam ）与IAM进行交互。客户端配置可能类似于以下代码示例：

@Configuration
class VaultConfiguration extends AbstractVaultConfiguration {

  @Override
  public ClientAuthentication clientAuthentication() {

    try {

      GcpIamAuthenticationOptions options = GcpIamAuthenticationOptions.builder()
        .role("my-role")
        .credential(GoogleCredential.getApplicationDefault())
        .build();
      return new GcpIamAuthentication(options, restOperations());

    } catch (IOException e) {
      throw new IllegalStateException(e);
    }
  }

  // …
}

默认情况下，IAM身份验证方法从凭据中导出项目ID和服务帐户ID。如果要代表特定服务帐户进行身份验证，也可以配置特定值。

有关更多信息，请参见参考文档 。

GCE登录

GCE（Google Compute Engine）身份验证工作流程适用于VM实例。它使用元数据服务（计算元数据）来获取签名令牌。签名的身份将传递给Vault以验证VM实例。客户端配置可能类似于以下代码示例：

@Configuration
class VaultConfiguration extends AbstractVaultConfiguration {

  @Override
  public ClientAuthentication clientAuthentication() {

    GcpComputeAuthenticationOptions options = GcpComputeAuthenticationOptions
        .builder().path("my-role").build();

    return new GcpComputeAuthentication(options, restOperations());
  }

  // …
}

有关更多信息，请参见参考文档 。

Azure身份验证

在版本0.10.0中，Vault引入了对Azure的身份验证支持。在Azure虚拟机上运行的应用程序可以通过使用托管服务身份针对Vault进行身份验证。可以为不需要预备凭据的虚拟机激活托管服务身份（MSI）。

Spring Vault从Azure实例元数据服务（IMDS）获取MSI凭据。保管箱需要其他详细信息（subscriptionId，资源组名称，VM名称）来执行身份验证。默认情况下，这些值也是从IMDS获取的，并与身份令牌一起传递到Vault。可能的客户端配置可能类似于以下代码示例：

@Configuration
class VaultConfiguration extends AbstractVaultConfiguration {

  @Override
  public ClientAuthentication clientAuthentication() {

    AzureMsiAuthenticationOptions options = AzureMsiAuthenticationOptions.builder()
        .role("my-role").build();
    return new AzureMsiAuthentication(options, restOperations());
  }

  // …
}

有关更多信息，请参见参考文档 。

键值API

Vault随版本0.10引入了其键-值后端的版本化变体（在数个版本之前也称为通用机密后端）。此更改引入了另一个后端类型，该后端类型的操作（列表，获取，放置，删除）具有相似的外部API，但实现不同的API。

为了统一版本控制和非版本控制的API访问，我们引入了通用API VaultKeyValueOperations 。 VaultKeyValueOperations公开了您不希望与版本控制元数据进行交互的通用功能，可以使用与版本无关的样式使用这些功能。以下示例显示了如何使用它：

VaultOperations vaultOperations = …
VaultKeyValueOperations operations = vaultOperations.opsForKeyValue("secret", KeyValueBackend.unversioned());

Map<String, Object> secret = new HashMap<>();
secret.put("key", "value");
secret.put("ttl", "5");

operations.put("key", secret);

operations.put("key", new Person(…));

VaultResponseSupport<Person> person = operations.get("key", Person.class);

上面显示的变体忽略了版本控制细节，即使它们是由目标机密后端提供的也是如此。您可以获取版本化的键值API以与版本进行交互。 VaultVersionedKeyValueOperations公开特定于版本的操作，例如特定秘密版本的检索或比较并设置。看下面的例子：

VaultOperations vaultOperations = …
VaultVersionedKeyValueOperations operations = vaultOperations.opsForVersionedKeyValue("versioned");

Map<String, Object> secret = new HashMap<>();
secret.put("key", "value");
secret.put("ttl", "5");

Metadata metadata = operations.put("key", secret);

Versioned<Map<String, Object>> versioned = operations.get("key", Version.from(42));

Map<String, Object> update = new HashMap<>();
update.put("key", "new-key");
update.put("ttl", "5");

Versioned<Map<String, Object>> compareAndSet = Versioned.create(secret, versioned.getVersion());

operations.put("key", compareAndSet);

operations.delete("key", Version.from(42));

对版本化机密的请求和响应将其内容与版本化元数据一起包装在Versioned对象以附加版本控制上下文。

包装API支持

Vault的核心概念是包装响应并返回令牌以获取实际的响应主体。现在，专用的API支持响应包装VaultWrappingOperations 。包装支持允许查找包装的响应。您可以阅读这些内容并重新包装内容。 VaultWrappingOperations不支持Spring Vault API调用的响应包装。您可以使用RestTemplate通过VaultOperations.doWithSession(…)如果您需要创建包装的响应，则直接使用。

以下示例概述了VaultWrappingOperations使用：

VaultOperations vaultOperations = …
VaultWrappingOperations operations = vaultOperations.opsForWrapping();

VaultToken wrappingToken = VaultToken.of(…);

// Metadata encapsulated TTL and Creation Time
WrappedMetadata lookup = operations.lookup(wrappingToken);

// Read the response as generic Map
VaultResponse response = operations.read(wrappingToken);

// Read the response applying a type hint.
VaultResponseSupport<SocialSecurityNumber> response = operations.read(
        wrappingToken, SocialSecurityNumber.class);

// You can also wrap user-supplied data and get a token in return
Map<String, String> map = Collections.singletonMap("key", "value");
WrappedMetadata metadata = wrappingOperations.wrap(map, Duration.ofSeconds(100));

Java 11兼容性

Spring Vault完全支持Java 11作为其下一个长期支持版本。实际上，如果您正在为JDK 11开发，我们强烈建议您将Spring Vault版本升级到2.1版本（与Spring Framework 5.1结合使用），因为它将为您提供Spring库的无警告体验。所有HTTP客户端集成都具有JDK 11支持。
