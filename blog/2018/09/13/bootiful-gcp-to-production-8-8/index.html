可怜的GCP：量产！（8/8）

工程
乔什·朗（Josh Long）
九月13，2018

嗨，春季粉丝！在这个简短的8部分系列文章中，我们将介绍Google Cloud Platform的Spring Cloud集成，称为Spring Cloud GCP。 Spring Cloud GCP代表了Google和Pivotal的共同努力，致力于在使用Google Cloud Platform时为Spring Cloud开发人员提供一流的体验。Pivotal Cloud Foundry用户将可以更轻松地与GCP服务代理集成 。我用Google Cloud Developer Advocate和我的好友Ray Tsang的投入编写了这些文章。您还可以在我们的Google Next 2018会话Bootiful Google Cloud Platform中观看 Spring Cloud GCP的演练。谢谢哥们儿！与往常一样， 如果您有任何反馈意见 ， 我希望能收到您的来信 。

该系列有八个职位。这些都是：

  • Bootiful GCP：针对Google Cloud Platform的Spring Cloud入门（1/8）
  • Bootiful GCP：使用Spring Cloud GCP进行关系数据访问（2/8）
  • Bootiful GCP：使用Spanner（3/8）进行全球一致的数据访问
  • Bootiful GCP：与Google Cloud Pub / Sub（4/8）集成
  • Bootiful GCP：使用Spring Cloud GCP Runtime Config（5/8）进行运行时配置
  • 引导性GCP：通过Spring Cloud GCP Stackdriver Trace（6/8）支持可观察性
  • 引导性GCP：使用Spring Cloud GCP连接到其他GCP服务（7/8）
  • 可怜的GCP：量产！（8/8）

在研究这些示例时，我们依赖于Google Cloud SDK和Spring Cloud GCP的默认身份验证以及Google Cloud SDK和gcloud CLI的本地安装。 Spring Cloud GCP的自动配置可配置DefaultCredentialsProvider ，以寻找某种方式进行身份验证。到目前为止，在我们的本地计算机上一切正常，因为我们运行了交互式身份验证提示，并在Google Cloud网站上确认了授权机构与此应用程序的链接。但是，如果您想在CI环境中运行测试或将应用程序部署到云平台怎么办？

在这些环境中运行时，我们需要一种方法来传达我们的授权。我们可以使用服务帐户在Google Cloud上执行此操作。在此说明该过程 。要点是，您将需要尽可能狭窄地描述您的应用程序可以使用Google Cloud服务帐户拥有的特权，然后将这些凭据呈现到一个文件中，然后我们将其提供给该应用程序。

PROJECT_ID=$(gcloud config list --format 'value(core.project)')


NAME=gcp-service-app


gcloud iam service-accounts create $NAME


gcloud projects add-iam-policy-binding $PROJECT_ID  --member "serviceAccount:${NAME}@${PROJECT_ID}.iam.gserviceaccount.com" --role "roles/owner"


gcloud iam service-accounts keys create ${NAME}.json --iam-account ${NAME}@${PROJECT_ID}.iam.gserviceaccount.com

  • 名称是任意的。您可以使用反映应用程序性质的名称。

  • 创建服务帐户并为其命名

  • 在我们的服务绑定中添加一个角色- roles/owner 。您会做得更好，可以更精细地分配角色。

  • 生成文件gcp-service-app.json ，其中包含密钥。

这个过程的结果应该是一个.json文件， gcp-service-app.json ，在本地机器上。

警告

该.json文件是个秘密！不要共用或失去它的轨道！

您需要告诉Spring Cloud应用程序在哪里可以找到该证书。您可以将其作为文件位置或以Base 64编码的String传递给Spring Cloud GCP。让我们做后者。我们可以使用spring.cloud.gcp.credentials.encoded-key来传递刚生成的文件的Base64编码内容。

这是一个秘密值，在一个环境与另一个环境之间会有所不同。我们不希望它在src/main/resources/application.properties被检入版本控制，即使我们这样做，我们仍然需要维护不同的特定于环境的值。Spring Boot让我们在运行应用程序时提供替代。我们可以使用application.properties或application.yml文件。我们可以将它们存储在特定于环境的Runtime Config中。在运行应用程序时，我们可以将它们提供为-D参数或环境变量。所以，可以很容易地说java -Dspring.cloud.gcp.credentials.encoded-key=…​ -jar ..提供一个值，要么贡献一个新值应用程序或将覆盖现有值src/main/resources/application.properties 。

环境变量是一个更自然适合像Cloud Foundry的或Heroku的平台，在这里我们并不一定要加强对应用程序的运行方式 ，在其控制java咒语。Buildpack平台为我们做到了。让我们看一下如何在本地运行应用程序。

运行.sh

#!/bin/bash


export SPRING_CLOUD_GCP_CREDENTIALS_ENCODED_KEY=$( cat $GCP_SERVICE_ACCOUNT_KEY_FILE | base64 -w0  )
export SPRING_CLOUD_GCP_PROJECT_ID=$(gcloud config list --format 'value(core.project)')

mvn clean spring-boot:run
  • 运行此脚本时， SPRING_CLOUD_GCP_CREDENTIALS_ENCODED_KEY在运行应用程序之前设置环境变量SPRING_CLOUD_GCP_CREDENTIALS_ENCODED_KEY ，以指向您的服务帐户.json文件的位置。

在我的机器上，我能够像这样运行该脚本： GCP_SERVICE_ACCOUNT_KEY_FILE=/home/jlong/keys/gcp-service-account.json ./run.sh 。.json文件的本地路径很可能会有所不同。

您可以根据需要为特定CI环境转换这些环境变量。您的应用程序经过测试和集成后，就可以投入生产！对我来说，生产是Cloud Foundry（同样在Google Cloud之上运行）。这是一个将应用程序部署到经过身份验证的Cloud Foundry实例的脚本。

deploy.sh。

#!/usr/bin/env bash

mvn -DskipTests=true clean package


cf d -f $APP_NAME


cf push --no-start --random-route -p $JAR $APP_NAME
cf set-env $APP_NAME SPRING_CLOUD_GCP_CREDENTIALS_ENCODED_KEY "$( cat $GCP_SERVICE_ACCOUNT_KEY_FILE | base64 -w0 )"
cf set-env $APP_NAME SPRING_CLOUD_GCP_PROJECT_ID $(gcloud config list --format 'value(core.project)')


cf restart $APP_NAME

  • 警告！该脚本删除现有应用程序。您不需要，但是可以确保一切都被重置：)

  • 在这里，我们推送应用程序，给它一个随机的路由，并告诉Cloud Foundry 不要启动它。然后，我们为应用程序指定环境变量。

  • 最后，既然我们已经完成了对应用程序的保护，就可以启动它。

小费

在run.sh和deploy.sh我们都使用base64 -w0在Base64中对.json服务帐户文件进行编码。我们中的一个在Linux上运行，另一个在OS X上运行。- -w0操作数可确保在Linux上针对GNU base64实用程序运行时，Base64编码的文件不会被硬包装，并且是一行连续的文本。
