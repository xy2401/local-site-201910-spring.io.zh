用于Redis和Apache Cassandra的Spring Data Lovelace有哪些新功能？

工程
马克·帕鲁奇
九月26，2018

这篇博客文章解释了针对Apache Cassandra和Redis的 Spring Data Lovelace中的新功能和值得注意的功能。确保还查看有关Spring Data Lovelace for MongoDB的新功能的博客文章。 。

Spring Data Lovelace上周刚刚发布了其普遍可用的版本，现在该简要介绍一下我们添加的新功能。发行列车具有很多功能。

在此博客文章中，我将介绍Apache Cassandra和Redis 。

Apache Cassandra的Spring数据

在此发行版中，我们改进了Cassandra特定类型的数据访问，引入了对生命周期事件的支持，改善了Java和Kotlin使用的编程体验，并进行了其他各种改进。让我们看一下此版本如何帮助您改善对Cassandra的数据访问。

映射和元组优化

Map和Tuple数据类型是Cassandra中的特定类型，允许在单个列中存储多个值。以前，我们在映射实体中以原始形式支持这两种类型，这意味着您只能使用具有原始键和值的映射。对于元组，您只能使用TupleValue ，即原始的Cassandra驱动程序类型，无需进一步的映射甚至不支持模式。

在此版本中，我们增加了对Map和Tuple类型的属性的映射和转换支持。地图现在可以包含非基本键和值，并且转换层将应用可能已注册的转换器。

请考虑以下类型：

@UserDefinedType
class Manufacturer {

  String name;

  // getters/setters omitted
}

@Table
class Supplier {

  Map<Manufacturer, List<String>> acceptedCurrencies;

  // getters/setters omitted
}

Manufacturer是映射的用户定义类型，地图用作键。值表示为List的字符串。我们现在可以重构代码以使用适当的Currency类型（例如java.util.Currency ）。为此，我们提供了String和Currency并通过注册这些CassandraCustomConversions 。以下示例显示了如何执行此操作：

enum StringToCurrencyConverter implements Converter<String, Currency> {
  INSTANCE;

  @Override
  public Currency convert(String source) {
    return Currency.getInstance(source);
  }
}

enum CurrencyToStringConverter implements Converter<Currency, String> {

  INSTANCE;

  @Override
  public String convert(Currency source) {
    return source.getCurrencyCode();
  }
}

@Configuration
class MyCassandraConfiguration {

  public CassandraCustomConversions cassandraCustomConversions() {
    return new CassandraCustomConversions(
      Arrays.asList(StringToCurrencyConverter.INSTANCE, CurrencyToStringConverter.INSTANCE));
  }
}

注册转换器后，我们可以继续使用Currency在Supplier类型以使用值对象而不是基元，如以下示例所示：

@Table
class Supplier {

  Map<Manufacturer, List<Currency>> acceptedCurrencies;

  // getters/setters omitted
}

元组不能真正用于Apache Cassandra的Spring Data的早期版本中。直接使用元组Row互动和TupleType检索以创建适当的元组值。因此，我们决定提供映射的元组类型，如以下示例所示：

@Table
class Supplier {

  List<Dependance> dependances;

  // getters/setters omitted
}

@Tuple
class Dependance {

  @Element(0) String address;
  @Element(1) String city;
  @Element(2) Currency currency;

  // getters/setters omitted
}

映射的元组带有注释@Tuple ，并且元组的各个组成部分引用（通过使用@Element(…) ）到元组内的序数索引。转换器检查加载的元组，并将它们映射到属于域模型的常规Java类。您不再需要与之互动TupleType和TupleValue直接-尽管您仍然可以-但是您可以使用类型安全的方法表示元组值。映射的元组受益于转换器的各种映射功能，并且可以引用已注册自定义转换器的类型。

对地图和元组的支持还包括模式生成，以通过从域模型中派生类型来快速设置模式。

有关更多详细信息，请参见我们的映射元组示例 。

生命周期事件

Cassandra映射框架现在包括几个org.springframework.context.ApplicationEvent您的应用可以通过在ApplicationContext 。在对象进行转换之前对其进行拦截（这会将您的域对象转换为Statement ），您可以注册的子类AbstractCassandraEventListener覆盖了onBeforeSave方法。调度事件后，将在域对象进入转换器之前调用并传递您的侦听器。以下示例显示了如何使用onBeforeSave ：

public class BeforeConvertListener extends AbstractCassandraEventListener<Person> {
  @Override
  public void onBeforeSave(BeforeSaveEvent<Person> event) {
    // does some auditing manipulation, set timestamps, whatever
  }
}

在Spring ApplicationContext中声明这些bean会使事件在分派时被调用。

以下回调方法存在于AbstractCassandraEventListener ：

  • onBeforeSave ：已拨入CassandraTemplatesave 在将行插入或保存到数据库之前进行操作。

  • onAfterSave ：已拨入CassandraTemplatesave在数据库中插入或保存行之后的操作。

  • onBeforeDelete ：已拨入CassandraTemplatedelete删除数据库中的行之前的操作。

  • onAfterDelete ：已拨入CassandraTemplatedelete删除行的数据库后操作。

  • onAfterLoad ：已拨入CassandraTemplateselect和selectOne从数据库中检索该行之后的方法。

  • onAfterConvert ：已拨入CassandraTemplateselect和selectOne从数据库检索的行之后的方法被转换为POJO。

仅针对根级别类型发出生命周期事件。用作实体根内部属性的复杂类型不受事件发布的约束。

请参阅我们的生命周期事件示例 。

Kotlin扩展

Spring Data公开了接受目标类型以查询结果或将结果值投影到其上的方法。Kotlin用自己的类型表示类（ KClass ），这可能会成为尝试获取Java类类型的障碍。

Apache Cassandra的Spring Data附带了一些扩展，这些扩展为使用泛型或直接接受KClass接受类型参数的方法添加了重载，如以下示例所示：

operations.getTableName<Person>()

operations.getTableName(Person::class)

operations.find<Person>().as<Contact>
  .matching(query(where("firstname").isEqualTo("luke"))).all();

有关更多详细信息，请参见我们的Cassandra Kotlin使用示例。

流利的模板API

的CassandraOperations与Apache Cassandra进行更底层的交互时，接口是核心组件之一。它提供了广泛的方法，可满足从批处理和结果流到CRUD操作的需求。您可以为每个方法找到多个重载。其中大多数涵盖了API的可选或替代部分，例如CQL查询， Statement或查询依据Query 。

FluentCassandraOperations为以下常用方法提供了更狭窄的界面CassandraOperations并提供了更易读，流畅的API。入口点（ insert(…) ， query(…) ， update(…)以及其他）根据要运行的操作遵循自然的命名模式。从入口点开始，API旨在仅提供上下文相关的方法，从而导致终止方法调用实际的Cassandra对方。

考虑一个查询示例：

List<Person> all = operations.query(Person.class)
  .inTable("people")
  .all();

此查询查询people所有行的表格，并将结果映射到Person类型。省略inTable(…)从实体类型派生表名。

下一个示例使用投影和查询：

List<Contact> all = operations.query(Person.class)
  .as(Contact.class)
  .matching(query(where("firstname").is("luke")))
  .all();

该查询使用的表Person映射类型并将项目结果（DTO或接口投影）投影到Contact 。查询本身是通过使用来自Person类型。您可以在检索单个实体和检索多个对象之间切换List或Stream通过终止方法： first() ， one() ， all() ， 要么stream() 。

流利的API是类型安全的，并且中间对象是不可变的。您可以准备查询的基础部分，并继续执行更具体的执行，如以下示例所示：

TerminatingSelect<Contact> select = operations.query(Person.class)
  .as(Contact.class)
  .matching(query(where("firstname").is("luke")))

Contact contact = select.first();
long count = select.count();

有关更多详细信息，请参见我们的Kotlin示例 。

已为Apache Cassandra模块的Spring Data找到了其他一些增强功能，因此请确保查阅参考文档中的“ 新功能”部分，以了解有关反应式切片查询和存在/计数预测的更多信息。

Spring Data Redis

Spring Data Redis的此发行版对各个主题进行了改进，而这些主题不适用于2.0发行版。它们中的大多数消除了Redis集群使用周围的粗糙边缘。核心主题是：

  • 连接改善

  • 改进Redis群集的使用

  • 框架中的各种改进

连接改进

Redis支持多种操作模式：独立，具有复制的独立，具有或不具有复制的Redis Sentinel，Redis群集。我们已经介绍了独立，Redis Sentinel和Redis群集模式。到目前为止，缺少的部分已从副本中读取。此版本引入了对跨多种Redis操作模式的副本读取的支持。下面的示例演示如何使用此新功能：

LettuceClientConfiguration clientConfiguration = LettuceClientConfiguration.builder()
  .readFrom(ReadFrom.NEAREST)
  .build();

RedisSentinelConfiguration endpoint = new RedisSentinelConfiguration()
  .master("my-master")
  .sentinel("sentinel-host1", 26379)
  .sentinel("sentinel-host2", 26379);

LettuceConnectionFactory factory = new LettuceConnectionFactory(endpoint, clientConfiguration);

指定ReadFrom使您可以在发出只读命令时选择特定的节点类型，例如GET要么SMEMBERS 。您可以使用Lettuce的预定义设置之一，也可以创建一个新的ReadFrom战略。 ReadFrom在所有可用副本的设置中都考虑使用：Redis Sentinel，Redis Cluster和静态主/副本设置（例如AWS ElastiCache），这使我们有了下一个改进。

您可以将AWS ElastiCache或任何其他静态主/副本设置（即，将Redis与一个或多个专用副本一起使用）与Spring Data Redis和Lettuce一起从副本节点读取。在以前的版本中，您只能使用主节点。看一下以下配置代码片段：

LettuceClientConfiguration clientConfiguration = LettuceClientConfiguration.builder().readFrom(ReadFrom.NEAREST).build();

RedisStaticMasterSlaveConfiguration endpoint = new RedisStaticMasterSlaveConfiguration("my-master-host", 6379)
  .node("my-replica-host1", 6379)
  .node("my-replica-host2", 6379);

LettuceConnectionFactory factory = new LettuceConnectionFactory(endpoint, clientConfiguration);

在此代码中，我们配置LettuceConnectionFactory使用多个节点而不实际指定角色。生菜本身确定各个主机的角色，并根据其角色使用节点。

该类别的最后一个改进是通过Unix域套接字使用本地连接。Unix域套接字或IPC（进程间通信）套接字是一个数据通信端点，用于在同一主机操作系统上运行的进程之间交换数据。与命名管道一样，Unix域套接字支持传输与TCP相比可靠的字节流。由于Unix域套接字通信仅在内核内发生，因此通信绕过网络，通常具有改进的性能配置文件。

要使用Unix域套接字，您需要使用Lettuce并为Netty添加本机扩展（ netty-transport-native-epoll在Linux上运行或netty-transport-native-kqueue在MacOS上运行时）。以下示例通过套接字配置与Redis的通信：

RedisSocketConfiguration endpoint = new RedisSocketConfiguration("/var/run/redis");

LettuceConnectionFactory factory = new LettuceConnectionFactory(endpoint);

Redis集群优化

此发行版对使用Lettuce驱动程序的Redis Cluster连接的连接处理进行了改进。以前的版本没有共享与Redis Cluster的底层Lettuce连接，这表现为性能下降，因为新连接始终会建立新的群集连接。发出多个命令时，此行为会产生影响，因为每个命令基本上都使用一个新命令RedisConnection 。

默认情况下，现在为Redis Cluster连接启用了本地连接共享。其他使用模式（例如Redis Standalone）已经在以前的版本中使用连接共享。以下示例显示了如何创建LettuceConnectionFactory具有共享的本机连接：

RedisClusterConfiguration clusterConfiguration = new RedisClusterConfiguration(…);

LettuceConnectionFactory factory = new LettuceConnectionFactory(clusterConfiguration);
factory.setShareNativeConnection(true);

某些操作（例如阻塞操作）需要专用连接，以不影响将在同一本地连接上运行的其他进程。如果您的应用程序严重依赖于阻止Redis命令，则可以为Redis群集连接启用缓冲池以缓冲连接的创建。启用池化是客户端配置方面。启用池化后， LettuceConnectionFactory将池应用于已配置的Redis使用方案。您可以使用LettucePoolingClientConfiguration作为启用池的入口点，如以下示例所示：

LettucePoolingClientConfiguration clientConfiguration = LettucePoolingClientConfiguration.builder().poolConfig(…).build();
RedisClusterConfiguration clusterConfiguration = new RedisClusterConfiguration(…);

LettuceConnectionFactory factory = new LettuceConnectionFactory(clusterConfiguration, clientConfiguration);

随着介绍ReadFrom设置和简化的集群连接处理，我们现在可以通过使用SCAN命令。在幕后，驱动程序维护一个有状态的游标，该游标使您可以遍历在群集中拥有键的所有主/副本节点。使用连接的scan(…)该方法可为您提供与在Redis Standalone设置中使用该方法相同的体验，如以下示例所示：

Cursor<byte[]> scan = clusterConnection.keyCommands()
  .scan(ScanOptions.scanOptions().match("foo*").build());
scan.forEachRemaining(key -> …);

键空间扫描还为所有Redis操作模式提供了响应式变体。呼唤scan(…)在反应式Redis Template API上返回Flux键。所结果的Flux具有背压意识，并将需求转化为SCAN如果有足够的需求来扫描整个键空间，则调用。如果满足需求，它将停止扫描。以下示例构造了这样一个Flux ：

Flux<String> scan = redisTemplate.scan(ScanOptions.scanOptions().match("something*").build());

Redis存储库改进

此版本附带对Redis存储库的按示例查询支持。通过示例查询是一种具有简单界面的用户友好查询技术。它允许动态查询创建，并且不需要您编写包含字段名称的查询。按示例查询的性质不需要查询语言，因为实际查询源自Example宾语。您现在可以定义一个Example查询存储在Redis哈希中的索引值。Redis仓库可以实现QueryByExampleExecutor片段以继承Query by Example方法。看一下以下代码片段：

interface PersonRepository extends CrudRepository<Person, String>, QueryByExampleExecutor<Person> {
}

PersonRepository repository = …;

Person eddard = new Person("eddard", "stark");
Person tyrion = new Person("tyrion", "lannister");
Person robb = new Person("robb", "stark");
Person jon = new Person("jon", "snow");
Person arya = new Person("arya", "stark");

repository.saveAll(Arrays.asList(eddard, tyrion, robb, jon, arya));

List<Person> result = repository.findAll(Example.of(new Person(null, "stark")));

这段代码插入了一堆Person对象。的Example对象定义一个探针，仅设置姓氏。查询引擎创建一个查询，该查询仅包含非空字段（默认情况下），查询其对象lastname是stark 。

有关更多详细信息，请参见按示例查询示例 。

Redis存储库现在支持类型别名，您可以通过使用以下方式注释域类来使用@TypeAlias 。默认情况下，Redis中的类型提示使用完全限定的类名。您可以应用别名来自定义类型名称并减少Redis内存使用。

以下示例保留了一个实例Person ：

package com.acme;

@TypeAlias("person")
class Person {
  // …
}

此代码导致类型提示（ person ）代替com.acme.Person 。用于在Redis中存储实体的相应命令如下：

HMSET "person:19315449-cda2-4f5c-b696-9cb8018fa1f9" "_class" "person" "id" "19315449-cda2-4f5c-b696-9cb8018fa1f9"

Redis模块中还发现了其他一些增强功能，因此请确保查看参考文档中的“ 新功能”部分，以了解有关键空间扫描，反应式发布/订阅和新命令的更多信息。
