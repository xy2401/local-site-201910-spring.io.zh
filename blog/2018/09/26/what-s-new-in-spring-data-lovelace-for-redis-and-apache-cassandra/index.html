Redis和Apache Cassandra的Spring Data Lovelace有什么新功能？

工程
马克·帕鲁奇
九月26，2018

这篇博客文章解释了针对Apache Cassandra和Redis的 Spring Data Lovelace中的新功能和值得注意的功能。确保还查看有关Spring Data Lovelace for MongoDB的新功能的博客文章。 。

Spring Data Lovelace上周刚刚发布了其普遍可用的版本，现在该简要介绍一下我们添加的新功能。发行列车具有很多功能。

在此博客文章中，我将介绍Apache Cassandra和Redis 。

Apache Cassandra的Spring数据

在此发行版中，我们改进了Cassandra特定类型的数据访问，引入了对生命周期事件的支持，改善了Java和Kotlin使用的编程体验，并进行了其他各种改进。让我们看一下此版本如何帮助您改善对Cassandra的数据访问。

映射和元组优化

Map和Tuple数据类型是Cassandra中的特定类型，允许在单个列中存储多个值。以前，我们在映射实体中以原始形式支持这两种类型，这意味着您只能使用具有原始键和值的映射。对于元组，你只能使用TupleValue ，原卡桑德拉驱动型，无需进一步映射甚至模式支持。

在此版本中，我们增加了对Map和Tuple类型的属性的映射和转换支持。地图现在可以包含非基本键和值，并且转换层将应用可能已注册的转换器。

请考虑以下类型：

@UserDefinedType
class Manufacturer {

  String name;

  // getters/setters omitted
}

@Table
class Supplier {

  Map> acceptedCurrencies;

  // getters/setters omitted
}

Manufacturer是映射的用户定义类型，映射用作键。值表示为字符串List 。现在，我们可以重构代码以使用适当的Currency类型（例如java.util.Currency ）。为此，我们在String和Currency之间提供了转换器，并通过CassandraCustomConversions注册了它们。以下示例显示了如何执行此操作：

enum StringToCurrencyConverter implements Converter {
  INSTANCE;

  @Override
  public Currency convert(String source) {
    return Currency.getInstance(source);
  }
}

enum CurrencyToStringConverter implements Converter {

  INSTANCE;

  @Override
  public String convert(Currency source) {
    return source.getCurrencyCode();
  }
}

@Configuration
class MyCassandraConfiguration {

  public CassandraCustomConversions cassandraCustomConversions() {
    return new CassandraCustomConversions(
      Arrays.asList(StringToCurrencyConverter.INSTANCE, CurrencyToStringConverter.INSTANCE));
  }
}

在注册了转换器之后，我们可以继续使用并在Supplier类型中使用Currency来处理值对象而不是基元，如以下示例所示：

@Table
class Supplier {

  Map> acceptedCurrencies;

  // getters/setters omitted
}

元组不能真正用于Apache Cassandra的Spring Data的早期版本中。使用元组需要直接的Row交互和TupleType检索来创建适当的元组值。因此，我们决定提供映射的元组类型，如以下示例所示：

@Table
class Supplier {

  List dependances;

  // getters/setters omitted
}

@Tuple
class Dependance {

  @Element(0) String address;
  @Element(1) String city;
  @Element(2) Currency currency;

  // getters/setters omitted
}

映射的元组使用@Tupleannotation，并且元组的各个组件（通过使用@Element(…) ）引用其在元组中的序号索引。转换器检查加载的元组，并将它们映射到属于域模型的常规Java类。尽管您仍然可以，但您不再需要直接与TupleType和TupleValue进行交互，但是您可以使用类型安全的方法来表示元组值。映射的元组受益于转换器的各种映射功能，并且可以引用已注册自定义转换器的类型。

对地图和元组的支持还包括模式生成，以通过从域模型中派生类型来快速设置模式。

请参阅我们的例子为映射元组进一步的细节。

生命周期事件

Cassandra映射框架现在包括几个org.springframework.context.ApplicationEvent事件。要在对象经历转换过程（将您的域对象转换为Statement ）之前对其进行拦截，可以注册AbstractCassandraEventListener的子类，该子类重写onBeforeSave方法。调度事件后，将在域对象进入转换器之前调用并传递您的侦听器。以下示例显示如何使用onBeforeSave ：

public class BeforeConvertListener extends AbstractCassandraEventListener {
  @Override
  public void onBeforeSave(BeforeSaveEvent event) {
    // does some auditing manipulation, set timestamps, whatever
  }
}

在Spring ApplicationContext中声明这些bean会使事件在分派时被调用。

以下回调方法存在于AbstractCassandraEventListener ：

  • onBeforeSave ： 在将行插入或保存到数据库之前，在CassandraTemplate save操作中调用。

  • onAfterSave ：在数据库中插入或保存行后，在CassandraTemplate save操作中调用。

  • onBeforeDelete ：在delete数据库中的行之前调用CassandraTemplate delete操作。

  • onAfterDelete ：在delete数据库中的行之后，在CassandraTemplate delete操作中调用。

  • onAfterLoad ：从数据库中检索行之后 ，在CassandraTemplate调用select和selectOne方法。

  • onAfterConvert ：在将从数据库中检索到的行转换为POJO 之后 ，在CassandraTemplate调用select和selectOne方法。

仅针对根级别类型发出生命周期事件。用作实体根内部属性的复杂类型不受事件发布的约束。

请参阅我们的生命周期事件示例 。

Kotlin扩展

Spring Data公开了接受目标类型以查询结果或将结果值投影到其上的方法。Kotlin用自己的类型（ KClass ）表示类，在尝试获取Java Class类型时可能会遇到障碍。

Apache Cassandra的Spring Data附带了一些扩展，这些扩展为使用泛型或直接接受KClass接受类型参数的方法添加了重载，如以下示例所示：

operations.getTableName()

operations.getTableName(Person::class)

operations.find().as
  .matching(query(where("firstname").isEqualTo("luke"))).all();

有关更多详细信息，请参见我们的Cassandra Kotlin使用示例。

流利的模板API

与Apache Cassandra进行更底层的交互时， CassandraOperations接口是核心组件之一。它提供了广泛的方法，可满足从批处理和结果流到CRUD操作的需求。您可以为每个方法找到多个重载。其中大多数涵盖了API的可选或替代部分，例如CQL查询， Statement或Query 。

FluentCassandraOperations提供的常用方法更窄的接口CassandraOperations并且提供了更可读的，流利API。入口点（ insert(…) ， query(…) ， update(…)等）将根据要运行的操作遵循自然的命名模式。从入口点开始，API旨在仅提供上下文相关的方法，从而导致终止方法调用实际的Cassandra对应对象。

考虑一个查询示例：

List all = operations.query(Person.class)
  .inTable("people")
  .all();

该查询查询people表中的所有行，并将结果映射到Person类型。省略inTable(…)从实体类型获得表名。

下一个示例使用投影和查询：

List all = operations.query(Person.class)
  .as(Contact.class)
  .matching(query(where("firstname").is("luke")))
  .all();

此查询使用在其上的表Person类型上映射和项目结果（DTO或接口投影） Contact 。通过使用“ Person类型中的字段名称来映射查询本身。您可以通过终止方法： first() ， one() ， all()或stream()在获取单个实体和将多个对象作为List或Stream获取之间进行切换。

流利的API是类型安全的，并且中间对象是不可变的。您可以准备查询的基础部分，并继续执行更具体的执行，如以下示例所示：

TerminatingSelect select = operations.query(Person.class)
  .as(Contact.class)
  .matching(query(where("firstname").is("luke")))

Contact contact = select.first();
long count = select.count();

有关更多详细信息，请参见我们的Kotlin示例 。

已为Apache Cassandra模块的Spring Data找到了其他一些增强功能，因此请确保查阅参考文档中的“ 新功能”部分，以了解有关反应式切片查询和存在/计数预测的更多信息。

Spring数据Redis

Spring Data Redis的此发行版对各个主题进行了改进，而这些主题不适用于2.0发行版。它们中的大多数消除了Redis集群使用周围的粗糙边缘。核心主题是：

  • 连接改善

  • 改进Redis群集的使用

  • 框架中的各种改进

连接改进

Redis支持多种操作模式：独立，带复制的独立，带或不带复制的Redis Sentinel，Redis群集。我们已经介绍了独立，Redis Sentinel和Redis群集模式。到目前为止，缺少的部分已从副本中读取。此版本引入了对跨多种Redis操作模式的副本读取的支持。下面的示例演示如何使用此新功能：

LettuceClientConfiguration clientConfiguration = LettuceClientConfiguration.builder()
  .readFrom(ReadFrom.NEAREST)
  .build();

RedisSentinelConfiguration endpoint = new RedisSentinelConfiguration()
  .master("my-master")
  .sentinel("sentinel-host1", 26379)
  .sentinel("sentinel-host2", 26379);

LettuceConnectionFactory factory = new LettuceConnectionFactory(endpoint, clientConfiguration);

通过指定ReadFrom可以在发出只读命令（例如GET或SMEMBERS时选择特定的节点类型。您可以使用Lettuce的预定义设置之一，也可以创建新的ReadFrom策略。在所有可用副本的设置中都考虑使用ReadFrom ：Redis Sentinel，Redis Cluster和静态Master / Replica设置（例如AWS ElastiCache），这使我们有了下一个改进。

您可以将AWS ElastiCache或任何其他静态主/副本设置（即，将Redis与一个或多个专用副本一起使用）与Spring Data Redis和Lettuce一起从副本节点读取。在以前的版本中，您只能使用主节点。看一下以下配置代码片段：

LettuceClientConfiguration clientConfiguration = LettuceClientConfiguration.builder().readFrom(ReadFrom.NEAREST).build();

RedisStaticMasterSlaveConfiguration endpoint = new RedisStaticMasterSlaveConfiguration("my-master-host", 6379)
  .node("my-replica-host1", 6379)
  .node("my-replica-host2", 6379);

LettuceConnectionFactory factory = new LettuceConnectionFactory(endpoint, clientConfiguration);

在此代码中，我们将LettuceConnectionFactory配置为使用多个节点而不实际指定角色。生菜本身确定各个主机的角色，并根据其角色使用节点。

该类别的最后一个改进是通过Unix域套接字使用本地连接。Unix域套接字或IPC（进程间通信）套接字是用于在同一主机操作系统上运行的进程之间交换数据的数据通信端点。与命名管道一样，Unix域套接字支持传输与TCP相比可靠的字节流。由于Unix域套接字通信仅在内核内发生，因此通信绕过网络，通常具有改进的性能配置文件。

要使用Unix域套接字，您需要使用Lettuce并为Netty添加本机扩展（在Linux上运行时为netty-transport-native-epoll ，在MacOS上运行时为netty-transport-native-kqueue ）。以下示例通过套接字配置与Redis的通信：

RedisSocketConfiguration endpoint = new RedisSocketConfiguration("/var/run/redis");

LettuceConnectionFactory factory = new LettuceConnectionFactory(endpoint);

Redis集群优化

此发行版对使用Lettuce驱动程序的Redis Cluster连接的连接处理进行了改进。以前的版本没有共享与Redis Cluster的底层Lettuce连接，这表现为性能下降，因为新连接始终会建立新的群集连接。发出多个命令时，此行为会产生影响，因为每个命令基本上都使用一个新的RedisConnection 。

默认情况下，现在为Redis Cluster连接启用了本地连接共享。其他使用模式（例如Redis Standalone）已经在以前的版本中使用连接共享。以下示例显示如何使用共享的本机连接创建LettuceConnectionFactory ：

RedisClusterConfiguration clusterConfiguration = new RedisClusterConfiguration(…);

LettuceConnectionFactory factory = new LettuceConnectionFactory(clusterConfiguration);
factory.setShareNativeConnection(true);

某些操作（例如阻塞操作）需要专用连接，以不影响将在同一本地连接上运行的其他进程。如果您的应用程序严重依赖于阻止Redis命令，则可以为Redis群集连接启用缓冲池以缓冲连接的创建。启用池化是客户端配置方面。启用池后， LettuceConnectionFactory池应用于已配置的Redis使用方案。您可以使用LettucePoolingClientConfiguration作为启用池的入口点，如以下示例所示：

LettucePoolingClientConfiguration clientConfiguration = LettucePoolingClientConfiguration.builder().poolConfig(…).build();
RedisClusterConfiguration clusterConfiguration = new RedisClusterConfiguration(…);

LettuceConnectionFactory factory = new LettuceConnectionFactory(clusterConfiguration, clientConfiguration);

通过引入ReadFrom设置和简化的集群连接处理，我们现在可以使用SCAN命令来支持集群范围的键空间扫描。在幕后，驱动程序维护一个有状态的游标，使您可以遍历在群集中拥有键的所有主/副本节点。使用连接的scan(…)方法可为您带来与在Redis Standalone设置中使用该连接时相同的体验，如以下示例所示：

Cursor scan = clusterConnection.keyCommands()
  .scan(ScanOptions.scanOptions().match("foo*").build());
scan.forEachRemaining(key -> …);

键空间扫描还为所有Redis操作模式提供了响应式变体。在反应式Redis Template API上调用scan(…)将返回Flux键。产生的Flux是背压感知和需求转化成SCAN ，如果有足够的需求来扫描整个密钥空间调用。如果满足需求，它将停止扫描。下面的示例构造这样的Flux ：

Flux scan = redisTemplate.scan(ScanOptions.scanOptions().match("something*").build());

Redis存储库改进

此版本附带对Redis存储库的按示例查询支持。通过示例查询是一种具有简单界面的用户友好查询技术。它允许动态查询创建，并且不需要您编写包含字段名称的查询。通过示例查询的性质不需要查询语言，因为实际查询是从Example对象派生的。现在，您可以定义一个Example以查询存储在Redis哈希中的索引值。Redis存储库可以实现QueryByExampleExecutor片段，以通过Example方法继承Query。看一下以下代码片段：

interface PersonRepository extends CrudRepository, QueryByExampleExecutor {
}

PersonRepository repository = …;

Person eddard = new Person("eddard", "stark");
Person tyrion = new Person("tyrion", "lannister");
Person robb = new Person("robb", "stark");
Person jon = new Person("jon", "snow");
Person arya = new Person("arya", "stark");

repository.saveAll(Arrays.asList(eddard, tyrion, robb, jon, arya));

List result = repository.findAll(Example.of(new Person(null, "stark")));

此代码插入了一堆Person对象。Example对象定义一个探针，仅设置姓氏。查询引擎创建一个查询，该查询仅包含非空字段（默认情况下）查询lastname为stark对象。

有关更多详细信息，请参见按示例查询示例 。

Redis存储库现在支持类型别名，您可以通过使用@TypeAliasannotation域类来使用@TypeAlias 。默认情况下，Redis中的类型提示使用完全限定的类名。您可以应用别名来自定义类型名称并减少Redis内存使用。

以下示例将持久保存Person的实例：

package com.acme;

@TypeAlias("person")
class Person {
  // …
}

此代码导致使用类型提示（ person ）代替com.acme.Person 。用于在Redis中存储实体的相应命令如下：

HMSET "person:19315449-cda2-4f5c-b696-9cb8018fa1f9" "_class" "person" "id" "19315449-cda2-4f5c-b696-9cb8018fa1f9"

Redis模块中还发现了其他一些增强功能，因此请确保参考参考文档中的“ 新功能”部分，以了解有关键空间扫描，反应式发布/订阅和新命令的更多信息。
