Spring Cloud Function 2.0和Azure函数

工程
戴夫·瑟
九月25，2018

Spring Cloud Function从1.0版开始就支持Microsoft Azure Functions ，但是在最新的2.0版本中（仍处于里程碑阶段），我们决定对编程模型进行一些更改。本文介绍了这些更改对用户而言意味着什么，并提供了此更改的背景知识。 Spring团队中的工作很有趣，并且与Microsoft的人员合作，为用户提供了两种技术的最佳结合。

Java的Azure函数

微软已经在Azure Functions中支持Java了一段时间，它使开发人员可以轻松地编写和部署Java代码，这些Java代码以无服务器的方式连接到Windows中的各种平台服务（事件，数据库，存储，HTTP网关等）。 Azure。它带有基于注释的编程模型，该模型将函数实现放入Java方法中。因此，您编写了一个方法并使用@FunctionName对其进行注释，该方法将成为Azure函数。目前有很多基于Maven插件的工具，这些工具可驱动Azure命令行，并可用于构建功能，在本地运行和调试并将其部署到云中。Azure网站上有一个快速入门指南 ，可以帮助您安装和运行所有必备组件，并且《 开发人员指南》中有关于Azure功能如何工作的更详细的文档。

注释还将功能方法参数和返回类型与部署时使用的服务联系在一起。例如，如果要在部署时绑定到HTTP网关，请使用@HttpTrigger ：

@FunctionName("uppercase")
public Bar execute(
    @HttpTrigger(name = "req", methods = { HttpMethod.GET, HttpMethod.POST },
        authLevel = AuthorizationLevel.ANONYMOUS) Foo foo,
    ExecutionContext context) {
  return new Bar(foo.getValue());
}

在此示例中，我们接受传入的HTTP POST请求，Azure将其主体绑定到类型为Foo的POJO。我们将Foo转换为Bar然后将其返回到HTTP响应中的调用方。

HTTP触发器是Azure Functions中最受欢迎的前5种集成，但基于事件的触发器和基于存储或基于数据库的触发器则更为流行。完整列表可在“ 触发器和绑定”文档中找到-有一个表，您可以在其中单击特定的绑定或触发器，并带您到参考页，在该页上有所有语言（包括Java）的代码示例。

这是另一个使用Azure Event Hub作为输入并使用Cosmos DB作为输出的示例。这个例子在github中 ：

@FunctionName("uppercase")
public Bar execute(
  @EventHubTrigger(name = "data", eventHubName = "events",
        connection = "TRANSACTIONS_EVENT_HUB_CONNECTION_STRING")
    Foo data,
    @CosmosDBOutput(name = "document", databaseName = "inventory",
        collectionName = "messages",
        connectionStringSetting = "PRODUCT_ITEMS_DOCUMENTDB_CONNECTION_STRING",
        createIfNotExists = true)
    OutputBinding<Bar> document,
    final ExecutionContext context) {
  return document.setValue(new Bar(foo.getValue()));
}
注意
 如果传入的JSON无法转换为函数输入类型（在这种情况下为Foo ），您将看到Azure失败，并出现令人困惑的no such method错误。如果看到可以将@FunctionName方法更改为String输入并查看数据，以确保该数据可绑定到所需的输入类型。

批注通过间接指向功能部署中配置的环境变量的方式携带连接凭证信息。通过Azure Functions Maven插件在生成pom.xml进行所有配置。例如：

<plugin>
  <groupId>com.microsoft.azure</groupId>
  <artifactId>azure-functions-maven-plugin</artifactId>
  <configuration>
    <resourceGroup>${functionResourceGroup}</resourceGroup>
    <appName>${functionAppName}</appName>
    <region>${functionAppRegion}</region>
    <appSettings>
            <property>
              <name>FUNCTIONS_EXTENSION_VERSION</name>
              <value>beta</value>
            </property>
            <property>
              <name>TRANSACTIONS_EVENT_HUB_CONNECTION_STRING</name>
            <value>${TRANSACTIONS_EVENT_HUB_CONNECTION_STRING}</value>
            </property>
            <property>
              <name>PRODUCT_ITEMS_DOCUMENTDB_CONNECTION_STRING</name>
              <value>${PRODUCT_ITEMS_DOCUMENTDB_CONNECTION_STRING}</value>
            </property>
            <property>
              <name>MSDEPLOY_RENAME_LOCKED_FILES</name>
              <value>1</value>
            </property>
    </appSettings>
  </configuration>
  <executions>
    <execution>
            <id>package-functions</id>
            <goals>
              <goal>package</goal>
            </goals>
    </execution>
  </executions>
</plugin>

在这种情况下，环境变量名称将插件配置链接到函数绑定声明。例如， @EventHubTrigger具有一个connection属性，该属性将在运行时从TRANSACTIONS_EVENT_HUB_CONNECTION_STRING环境变量填充。插件使用具有相同名称的本地环境变量（请注意${}占位符）来远程配置它，开发人员或CI进程负责在运行时进行设置。

您自己的个人连接字符串是机密，可以在Azure仪表板中找到-单击相关资源时，通常会有一个“ Connection Strings链接（或类似链接），您可以将其复制并粘贴到本地进程中（例如，在您在本地运行，但不签到源代码管理。例如，您可以使用如下所示的setup-env.sh脚本：

export PRODUCT_ITEMS_DOCUMENTDB_CONNECTION_STRING="AccountEndpoint=https://..."
export TRANSACTIONS_EVENT_HUB_CONNECTION_STRING="Endpoint=sb://..."

并在终端会话开始时提供一次。

该示例的pom.xml中还有其他一些插件声明。它们都是重要的，但基本上是样板-您应该能够复制它们并在所有Azure Function应用程序中重复使用相同的配置。

弹簧云功能

当应用程序开发人员声明类型为java.util. Spring bean时，Spring Cloud Function旨在支持类似的无服务器用例java.util.Function 。与普通的Java函数相比，在Azure上使用Spring Cloud Function的优点是（原则上）实际的业务逻辑代码可移植到其他平台，并且它是现有Spring用户熟悉的编程模型。而且，Spring的所有常规好处都适用：依赖注入以及与许多其他Java库的全面集成。

上面两个示例的等效项都是一个@Bean ：

@Bean
public Function<Foo, Bar> uppercase() {
  return foo -> new Bar(foo.getValue().toUpperCase());
}

在Spring Cloud Function的1.0版中，用户必须手动将Microsoft批注映射到JSON部署描述符，然后将其手动打包到具有正确平台布局的存档中。该过程很脆弱（但独立于Azure Java编程模型）。

在2.0版中，此操作仍然有效，但是我们选择了更明确地支持使用Azure批注。因此，现在我们有了一个基类，应用程序开发人员可以用Azure批注扩展和装饰基类。上面的示例与@Bean完全相同，并且上面的execute方法之一将插入到Spring Cloud处理程序的子类中。例：

public class UppercaseHandler extends AzureSpringBootRequestHandler<Foo, Bar> {
  @FunctionName("uppercase")
    @HttpTrigger(name = "req", methods = { HttpMethod.GET,
        HttpMethod.POST }, authLevel = AuthorizationLevel.ANONYMOUS) Foo foo,
    ExecutionContext context) {
  return super.handle(foo, context);
}

注意，基类AzureSpringBootRequestHandler是通用的，带有用于输入和输出的类型参数。您必须将输入类型与传入事件数据进行匹配，这些事件数据将以JSON形式呈现，并在Spring要做任何事情之前由Azure使用Jackson进行转换。在基类中有2种实用程序方法，一种（ handle ）返回响应对象，一种（ handleOutput ）接受OutputBinding并将其绑定到用户Function的输出。

注意
 基类是纯模板，仅用作Spring函数与无服务器平台服务绑定的外部表示。例如，如果您在其他平台上运行，或者通过Spring Cloud Function Web适配器在本地运行，则将忽略Azure绑定。将来可能会用接口声明代替它-Azure平台当前不允许这样做，但是我们正在与Microsoft团队合作。

有多种配置选项可驱动Azure函数的运行时行为。最重要的（和唯一的强制性）一个是MAIN_CLASS ，这是主要的@SpringBootApplication类携带的声明Function （或Functions ）。您可以将其指定为环境变量，也可以将其指定为应用程序jar清单中的Main-Class条目。只要您的应用程序具有仅具有一个功能的主类，就无需执行其他任何操作。在示例应用程序中，我们使用清单定义主类：

<plugin>
  <groupId>org.apache.maven.plugins</groupId>
  <artifactId>maven-jar-plugin</artifactId>
  <configuration>
    <archive>
      <manifest>
        <mainClass>example.FunctionApplication</mainClass>
      </manifest>
    </archive>
  </configuration>
</plugin>

如果您的应用程序具有多个Function bean，则可以通过@FunctionName批注将它们映射到Azure函数@FunctionName名称（或更确切地说，在FunctionCatalog中的名称）与函数名称匹配。这样，您可以创建一个Azure Function应用程序，它是一组功能的单个部署工件。如果愿意，还可以使用任意@FunctionName并通过环境变量FUNCTION_NAME或application.properties的function.name配置Spring Cloud Function名称。
