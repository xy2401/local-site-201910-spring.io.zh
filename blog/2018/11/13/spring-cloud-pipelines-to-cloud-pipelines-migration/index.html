从Spring Cloud Pipelines到Cloud Pipelines的迁移

发布
马尔辛·格热兹扎克（Marcin Grzejszczak）
十一月13，2018

Spring Cloud Pipelines是一个GitHub项目，旨在解决以下问题：

  • 创建通用部署管道

  • 传播良好的测试和部署实践

  • 减少将功能部署到生产中所需的时间。

第一次提交发生在2016年8月31日。从那时起，我们就获得了有关建议的部署管道及其具体可视化的大量反馈。在过去的两年中，我们成功构建的最重要的功能是：

  • 固执己见的部署管道设置

  • 管道脚本，用于验证项目的向后兼容性并允许零停机时间部署

  • 支持PHP，.NET，NodeJS和JVM（Maven和Gradle）项目

  • Cloud Foundry的部署选项

  • Kubernetes的部署选项

  • 通过Ansible的部署选项

  • 使用Jenkins Job DSL在Jenkins中进行管道可视化

  • 使用Jenkinsfile在Jenkins中进行管道可视化

  • Concourse中的管道可视化

我们想宣布下一个1.0.0。M9版本的Spring Cloud Pipelines，这也是当前形式的最新版本。

（春季）云管道

由于我是Spring Cloud团队的成员，因此我们决定将其置于Spring Cloud的保护下。除此之外，该项目与Spring Cloud无关，更不用说Spring了。

这就是为什么我们决定将项目分成多个部分，将其重命名，然后将其放入一个单独的GitHub组织中的原因。

欢迎使用云管道！

Spring Cloud Pipelines有一个新家和一个新名称。这些项目现在位于GitHub的Cloud Pipelines组织之下。结果，该项目被分为以下几部分：

  • 脚本 ：包含部署管道的核心逻辑。您可以调用此存储库recipes 。每个配方都包含部署管道的步骤。文档在这里 。

  • Concourse ：使用Concourse完成的管道可视化。文档在这里 。

  • Jenkins ：使用Jenkins完成管道可视化。文档在这里 。

  • 项目搜寻器 ：从存储库管理服务中获取数据的抽象。

  • Pipeline Base ：CI服务器使用的Docker基本映像。

在将项目迁移到新组织之后，我们决定将其标记为“已完成” 。我们已经在Spinnaker上工作了一段时间，并计划投资使Cloud Pipelines配方能够在Spinnaker上运行以进行回滚测试和合同测试。对于其他重叠领域，例如部署策略，我们想依靠Spinnaker的抽象。当然，我们将促进围绕该项目的任何讨论以及与问题，请求和发布相关的琐事，但是我们可以放心地说，该项目将由社区完全推动和维护。

如果您有兴趣将项目从Spring Cloud Pipelines迁移到Cloud Pipelines，则应查看迁移指南：

  • 脚本迁移指南

  • Jenkins迁移指南

为什么选择大三角帆？

Spinnaker是一个开源的多云连续交付平台，该平台始于Netflix，但现在涉及来自Google，亚马逊，Pivotal等许多公司的更广泛的贡献者社区。Spring团队和Pivotal Cloud R＆D的共同努力导致了Spinnaker中Cloud Foundry的支持。

Spinnaker使我们可以访问更广泛的受支持的云提供商，而无需重新发明这些交互。

迄今为止，有关Cloud Pipelines中零停机时间部署和回滚测试的工作揭示了持续交付平台（如Spinnaker）的关键价值，该平台可单独维护已部署环境中的资产清单。

  • 在Cloud Pipelines中，当前的生产系统状态未存储。为了减轻这个问题，我们使用Git标签存储状态。但是，在进行手动部署或回滚时，很容易破坏这一点。此外，生产系统状态可能会跨越多个版本（跨云提供商或区域），并且通常无法还原为可以存储在标签中的单个值。一个执行例如回滚测试的Spinnaker管道可以自由使用Spinnaker的系统状态知识将回滚测试与目标区域中的生产版本相互关联。

  • Cloud Pipelines中提供的部署选项是有限的。Spinnaker概括了跨支持的云提供商的蓝/绿，滚动推送，零停机时间和自动canary分析部署功能。

更具体地说，考虑蓝色/绿色部署。Spinnaker支持蓝色/绿色，同时保留任意数量的先前版本服务器组。在以下示例中，从应用程序的当前版本开始，您只需执行一个操作即可回滚到该应用程序的最新版本。我们随时可以选择销毁V022，

有效地使其无法回滚。这样的事情可能会在部署活动的带外发生（例如释放容量），从而使无状态系统无法应对更改。
