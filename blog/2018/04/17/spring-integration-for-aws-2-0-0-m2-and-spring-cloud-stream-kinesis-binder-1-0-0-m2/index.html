适用于AWS 2.0.0的Spring Integration。M2和Spring Cloud Stream Kinesis Binder 1.0.0。M2

发布
阿尔特姆·比兰（Artem Bilan）
2018年4月17日

亲爱的春季社区！

我很高兴宣布支持Amazon Web Services的 Spring项目的里程碑发布： 适用于AWS的Spring Integration 2.0.0.M2和适用于AWS Kinesis的Spring Cloud Stream Binder 1.0.0.M2 。

这两个里程碑在Spring Milestone存储库中都可用，并且可以作为Maven依赖项使用：


    org.springframework.integration
    spring-integration-aws
    2.0.0.M2



    org.springframework.cloud
    spring-cloud-stream-binder-kinesis
    1.0.0.M2

使用第二个时不需要包括第一个。AWS Kinesis Binder也可传递地包括AWS Kinesis和AWS DynamoDB Java SDK依赖项。

在社区通过StackOverflow问题，Gitter聊天，错误报告和功能请求提供了宝贵的反馈之后，这些里程碑是团队和一些第三方贡献者的努力。所以，谢谢大家！

这是自先前宣布的里程碑1以来的更改：

  • 的KinesisMessageDrivenChannelAdapter当AWS Client中发生间歇性网络错误时，已改进了容错能力。

  • 运动学Checkpointer已改进为基于ConcurrentMetadataStore以获得更好的并发交互，尤其是在集群环境中-消费群体。

  • 标头映射机制已针对SqsMessageHandler和SnsMessageHandler -的SqsHeaderMapper和SnsHeaderMapper分别作为现成的实现。

  • 由于AWS Kinesis不提供任何标头符号，因此我们通过将标头嵌入 Kinesis实现了变通方法Record身体。为此， KinesisMessageHandler和KinesisMessageDrivenChannelAdapter应该配置OutboundMessageMapper和InboundMessageMapper ， 分别。开箱即用的Spring集成是一个不错的选择EmbeddedJsonHeadersMessageMapper可以在两侧用于通过AWS Kinesis承载任何所需的消息头。

  • 的KinesisMessageDrivenChannelAdapter还提供了在以下情况下转换记录的逻辑： ListenerMode.batch用来。

  • 的KinesisMessageChannelBinder已根据Spring Cloud Stream和Spring Integration AWS中的最新更改和修复进行了更新。

  • 也是新的rawRecords引入了侦听器模式。默认情况下，将Kinesis Binder配置为record发出消息的方式包含有效载荷为byte[]用于Spring Cloud Stream的下游转换。的batch模式现在会产生一个List和这个新rawRecords模式产生一条带有有效负载的消息List -到业务模型的所有转换逻辑都取决于目标中的最终用户@StreamListener方法。

像往常一样，通过适当的交流渠道欢迎任何反馈，功能性想法，批评家，错误报告和问题！
