Spring Boot 2.0中的属性绑定

工程
菲尔·韦伯
2018年3月28日

从Spring Boot的第一个版本开始，就可以通过使用@ConfigurationProperties注解。也可以用不同的形式指定属性名称。例如， person.first-name ， person.firstName和PERSON_FIRSTNAME可以全部互换使用。我们称此功能为“轻松绑定”。

不幸的是，在Spring Boot 1.x中，“松弛绑定”显得有些放松。确切定义绑定规则是什么以及何时可以使用特定格式非常困难。我们还开始获得有关问题的报告，这些问题很难用我们的1.x实施来解决。例如，在Spring Boot 1.x中，无法将项目绑定到java.util.Set 。

因此，在Spring Boot 2.0中，我们着手重新设计绑定发生的方式。我们添加了几个新的抽象，并且我们开发了一个全新的绑定API。在此博客文章中，我们介绍了一些新的类和接口，并描述了为什么要添加它们，它们做什么以及如何在自己的代码中使用它们。

财产来源

如果您已经使用Spring一段时间了，那么您可能对Environment抽象。这个界面是一个PropertyResolver使您可以从某些基础层解析属性PropertySource实现。

Spring Framework 提供PropertySource常见事物的实现，例如系统属性，命令行标志和属性文件。Spring Boot以对大多数应用程序有意义的方式自动配置这些实现（例如，加载application.properties ）。

配置属性源

而不是直接使用现有的PropertySource绑定接口，Spring Boot 2.0引入了新的ConfigurationPropertySource接口。我们引入了一个新接口，为我们提供了一个逻辑上的位置，以实现以前是活页夹一部分的宽松的绑定规则。

该接口的主要API非常简单：

ConfigurationProperty getConfigurationProperty(ConfigurationPropertyName name);

还有一个IterableConfigurationPropertySource实现的变体Iterable这样您就可以发现源中包含的所有名称。

你可以适应 Spring Environment至ConfigurationPropertySources通过使用以下代码：

Iterable<ConfigurationPropertySource> sources =
	ConfigurationPropertySources.get(environment);

如果您需要它，我们还提供了一个简单的方法MapConfigurationPropertySource实施。

配置属性名称

事实证明，如果将放宽的属性名称限制在一个方向，则更容易实现。无论属性如何在基础源中表示，都应始终使用规范形式访问代码中的属性。

的ConfigurationPropertyName类强制执行这些规范的命名规则，这些规则基本上可以归结为“使用小写的kebab-case名称”。

因此，例如，您应该在代码中引用一个属性为person.first-name即使person.firstName要么PERSON_FIRSTNAME用于基础源。

来源支持

如您所料， ConfigurationProperty从传回的物件ConfigurationPropertySource封装实际的属性值，但也可以包含可选的Origin宾语。

一个Origin是Spring Boot 2.0中引入的新接口，可让您查明加载值的确切位置。有一些Origin实现，也许是最有用的TextResourceOrigin 。这提供了详细信息Resource加载的值以及该行的行号和列号。

对于.properties和.yaml文件，我们已经编写了自定义加载程序，用于在文件加载时跟踪来源。已经对几种现有的Spring Boot功能进行了改造，以利用原始信息。例如，活页夹验证异常现在显示了无法绑定的值和来源。故障分析器显示错误的方法如下：

活页夹API

的Binder类（在org.springframework.boot.context.properties.bind ）让您选一个或多个ConfigurationPropertySource并从他们身上绑定一些东西。更确切地说， Binder需要一个Bindable并返回一个BindResult 。

可绑定

一种Bindable可能是现有的Java bean，类类型或复杂的ResolvableType （比如一个List ）。这里有些例子：

Bindable.ofInstance(existingBean);
Bindable.of(Integer.class);
Bindable.listOf(Person.class);
Bindable.of(resovableType);

Bindable也用于携带注释信息，但是您通常不必担心。

绑定结果

联编程序不是直接返回绑定的对象，而是返回称为a的内容。 BindResult 。类似于Java 8的方式Streams返回Optional ， 一种BinderResult表示可能已绑定或未绑定的内容。

如果尝试获取未绑定对象的实际结果，则会引发异常。我们还提供了一些方法，可让您在未绑定任何内容或map不同的类型：

var bound = binder.bind("person.date-of-birth",
	Bindable.of(LocalDate.class));

// Return LocalDate or throws if not bound
bound.get();

// Return a formatted date or "No DOB"
bound.map(dateFormatter::format).orElse("No DOB");

// Return LocalDate or throws a custom exception
bound.orElseThrow(NoDateOfBirthException::new);

格式化和转换

最ConfigurationPropertySource实现将其基础值公开为字符串。当。。。的时候Binder需要将源值转换为其他类型，它将委托给Spring的ConversionService API。如果您需要调整值的转换方式，则可以自由使用格式器注释，例如@NumberFormat要么@DateFormat 。

Spring Boot 2.0还引入了一些新的批注和转换器，它们对于绑定特别有用。例如，您现在可以转换诸如4s至Duration 。看着那（这org.springframework.boot.convert包以获取详细信息。

BindHandler

有时，您可能需要在绑定时实现其他逻辑，并且BindHandler界面提供了一种不错的方法。每BindHandler拥有onStart ， onSuccess ， onFailure和onFinish可以实现以覆盖行为的方法。

Spring Boot提供了许多处理程序，主要是为了支持现有的@ConfigurationProperties捆绑。例如， ValidationBindHandler可以用来申请Validator绑定对象的验证。

@ConfigurationProperties

如本博文开头所述， @ConfigurationProperties从一开始就一直是Spring Boot的功能。最有可能@ConfigurationProperties将继续是大多数人执行绑定的方式。

尽管事实上我们已经重写了整个绑定过程，但是大多数人似乎在升级Spring Boot 1.5应用程序时没有太多问题。只要您遵循迁移指南中的建议 ，您就会发现一切正常。如果在升级应用程序时确实发现问题，请在GitHub问题跟踪器上报告它们，并提供一个小样本来重现该问题。

未来的工作

我们计划继续开发Binder在Spring Boot 2.1中，我们要支持的第一个功能是不可变的配置属性。如果当前需要getter和setter的配置属性可以改用基于构造函数的绑定，那就太好了：

public class Person {

	private final String firstName;
	private final String lastName;
	private final LocalDateTime dateOfBirth;

	public Person(String firstName, String lastName,
			LocalDateTime dateOfBirth) {
		this.firstName = firstName;
		this.lastName = lastName;
		this.dateOfBirth = dateOfBirth;
	}

	// getters

}

我们认为构造函数绑定也可以与Kotlin数据类一起很好地工作。如果您有兴趣跟踪此功能的进度，请订阅问题＃8762 。
