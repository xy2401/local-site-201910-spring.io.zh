使用Spring Boot 2.0测试自动配置

工程
Madhura Bhave
2018年3月7日

自动配置是Spring Boot最强大的功能之一。自动配置类的测试通常遵循相同的模式。大多数测试会使用受测试的自动配置类来启动ApplicationContext ，并且根据测试，还会加载其他配置来模拟用户行为。这种模式的重复出现会在代码库中增加很多重复。

Spring Boot 2.0提供了一套新的测试助手，可以轻松配置ApplicationContext来模拟自动配置测试场景。以下示例将ApplicationContextRunner配置为测试UserServiceAutoConfiguration ：

private final ApplicationContextRunner contextRunner = new ApplicationContextRunner()
    .withConfiguration(AutoConfigurations.of(UserServiceAutoConfiguration.class));

由于在测试类份额类似的配置大多数测试中， ApplicationContextRunner是最好的，因为它被设置了一些常见的配置测试类的领域。

每个测试都可以使用所需的配置和属性进一步自定义ApplicationContext 。

@Test
public void someTest() {
    this.contextRunner
        .withPropertyValues("user.my.property=test")
	    .withUserConfiguration(MyConfiguration.class)
	    .run(...);
}

除了调用UserServiceAutoConfiguration ，上面的示例还注册MyConfiguration并将user.my.property属性设置为test 。

ApplicationContextRunner透明地复制Spring Boot使用的配置初始化顺序（先扫描用户配置，然后根据定义的顺序自动配置）。

支持AssertableApplicationContext ，它在ApplicationContext上提供AssertJ样式的断言。您还可以链接多个断言，如以下示例所示：

@Test
public void someTest() {
    this.contextRunner.run((context) -> {
        assertThat(context).hasSingleBean(MyBean.class);
        assertThat(context).getBeanNames(UserRule.class).hasSize(2);
    });
}

断言也可以在未能开始检查失败原因的ApplicationContext上使用。无论如何，上下文的生命周期不再需要由测试来管理，即上下文自动关闭。

对于需要WebApplicationContext测试，可以使用WebApplicationContextRunner或ReactiveWebApplicationContextRunner 。

使用@ConditionalOnClassannotation的类路径上是否存在特定的Class ，也会影响自动配置。ApplicationContextRunner ，您可以测试在运行时不存在给定Class时发生的情况。Spring Boot附带有FilteredClassLoader ，跑步者可以轻松使用。在以下示例中，我们断言，如果不存在UserService ，则会自动禁用自动配置：

@Test
public void serviceIsIgnoredIfLibraryIsNotPresent() {
    this.contextRunner
        .withClassLoader(new FilteredClassLoader(UserService.class))
        .run((context) -> assertThat(context)
                .doesNotHaveBean("userService"));
}
