DM Server上的Spring Integration

工程
伊文·富尔德
2009年2月27日

介绍

在这篇博客文章中，我将向您展示如何使用Spring Integration和dm Server创建一个松耦合和可伸缩的应用程序。使用OSGi的附加好处将使我们能够在运行时更改应用程序的行为，当然，我们也会从中获得一些乐趣。首先，我将快速强调设计可同时使用的应用程序的原因，然后，我将描述将OSGi软件包与Messaging集成的不同策略。在此过程中，您将瞥见我们的工具和dm Server功能。如果您已经下载并安装了最新的SpringSource Tool Suite和dm Server，则应该可以自己执行此操作。您无需遵循示例代码即可，但是如果您感兴趣，可以在Spring Integration沙箱中找到它。

同步故障

应用程序中的同步调用可以正常运行，但是无法扩展。之前已经有很多文档记录了，所以我给您快速回顾一下，然后继续。当您进行同步调用时，调用线程必须阻塞直到调用完成。如果您碰巧使用该方法进行I / O，则阻塞线程正在浪费CPU周期。如果您有一个大规模的并行Web应用程序，那么这不是一个太大的问题，因为您可以增加正在使用的线程池的最大大小。但是，如果您需要通过一次调用就充分利用服务器的CPU潜力，那么您就不走运了。由于一个线程在一个内核上运行，因此拥有更多内核的单线程程序效率将更低。

救援消息

自己进行并发编程很有趣，但这并不是最容易的事情，您可能希望今天早些时候完成工作。那就是Spring Integration可以为您提供帮助的地方。它使您可以对所需的并发处理进行所有控制，而不必对并发编程的低级细节负责。

OSGi有何帮助？

如果没有OSGi，则可以完全防止同步故障。它使生活变得更加美好，您不应该错过它。每次进行少许更改后，进行完全重新部署并重新启动jvm不会很有趣。还有另一件事使OSGi和消息传递的组合非常有趣。如果企业应用程序变大，则必须在某个时候进行模块化。在同一台传统服务器中运行多个模块是灾难的根源（也称为jar hell）。因此，在传统架构中，人们在许多情况下过早地向外扩展。这使得设置非常差。不同的节点必须通过网络相互通信，并且它们不能共享cpu和内存等资源。这使体系结构效率低下并严重影响性能。您不仅在浪费一台服务器上某些内核上的周期，而且现在还在多个节点上重复相同的低效率操作。为了增加侮辱性伤害，在通常的本地方法调用已足够的地方，网络等待时间会增加I / O等待时间。

使用OSGi，您不必大惊小怪，您可以在同一个jvm中运行不同的包，这样它们之间的调用就可以像普通方法调用一样快速，而不必担心在同一类路径上部署不同团队的jar会产生可怕的jar hell 。拥有一个可以在运行时更改其行为的应用程序，可以与多个团队合作，并且可以帮助您将所需的硬件减少到一台强大的服务器上。听起来很甜，所以让我们开始吧。

制备

如果您希望在博客中关注实际的代码示例，或者自己独立编写示例，则需要具备适当的工具（和代码）。要遵循博客的想法，您不需要这样做。我将在此处包括必要的代码示例。本文无意作为详细的操作方法，因此，如果您想继续学习，则必须自己进行编码，或者仅使用示例。为此，请按照下列步骤操作。

  1. 下载并安装SpringSource Tool Suite和dm Server （这与解压缩文件一样容易）。如果愿意，还可以将Eclipse与dm Server工具和Spring IDE结合使用。
  2. 使用默认名称配置dm服务器并启动它
    1. 打开服务器视图，右键单击它并选择新建
    2. 在搜索窗口中键入Spring，然后选择SpringSource dm Server 1.0
    3. 指向解压缩的dm服务器目录
  3. 更新您的软件包存储库并安装Spring Integration Core 1.0.1和Commons Logging 1.1.1
    1. 双击新创建的服务器，服务器管理控制台将打开
    2. 选择存储库选项卡，然后单击“更新本地捆绑和库存储库索引”
    3. 搜索Spring Integration并下载核心捆绑包
    4. 启动服务器
  4. 从Spring Integration沙箱导入项目
    1. 将项目检出到本地工作目录
    2. 在STS中，选择文件->导入->现有项目到工作区，然后指向工作目录

到消息传递包的异步切换

在本节中，我们将使用依赖于Spring Integration的单个捆绑包，并将我们的工作交给它。该捆绑包将隐藏所有消息传递详细信息，并且如果您只有一小部分要做大量工作的系统，那么这可能正是您所需要的。

我将展示的示例应用程序是为中世纪小镇开发的。在这些城镇中，曾经有一个镇民，走在大街上，大声疾呼当地新闻和公告。有问题的小镇（希望保持匿名）已被镇上的工会强迫改善其哭泣的劳动力的工作条件，并且由于预算限制，他们希望同时减少人员。

他们得出的结论是，实现这些目标的唯一方法是用自动化系统代替镇压机的大部分体力劳动。镇长只需在办公室舒适的地方将其消息传递到该系统，然后该消息就会分发给镇上的所有公民。市政厅已经跟踪了所有市民，因此将这种分配逻辑放置在城镇市政厅中将是最容易的。

Town Crier捆绑包
决定不要再培训镇级操作员以使用新系统，因此他仍将调用镇级cry（）方法，只需要进行多次操作即可覆盖镇上的每个部分该镇了。公告的分发将由镇上的新系统透明地进行。Towncrier捆绑包只有一个类TownCrier.java，其中注入了应该哭泣的Town。


@Component
public class TownCrier {
	private static final Log logger = LogFactory.getLog(TownCrier.class);

	@Autowired //Spring will take care of the wiring as usual
	private Town town;

	private ScheduledThreadPoolExecutor schedule;

	@PostConstruct
	public void start() {
		logger.info("Starting feeder");
		schedule = new ScheduledThreadPoolExecutor(1);

		TimerTask task = new TimerTask() {
			public void run() {
				logger.info("Crying the time");
				town.cry("Oyez! Oyez! Oyez! It is now " + new Date());
			}
		};
		schedule.scheduleAtFixedRate(task, 1, 1, TimeUnit.SECONDS);
	}
}

如果您知道Spring Integration，那么您将不胜枚举此代码，并想使用入站通道适配器和轮询器代替，但是我们在这里只是在模拟旧代码，所以请耐心等待。

为了帮助Spring引用将在另一个捆绑包中的城镇，我们将使用Spring DM。如果您之前从未使用过OSGi和Spring DM，那么现在是阅读入门的好时机。我们只需在osgi-context.xml中添加OSGi参考元素。


<osgi:reference id="collectorService"
	interface="com.springsource.samples.integration.osgi.town.input.Town" />

最后，依赖于MANIFEST.mf中从中导出服务的城镇捆绑包。

Import-Package: com.springsource.samples.integration.osgi.town.input,
 org.apache.commons.logging;version="[1.1.1,1.1.1]",
 javax.annotation

同样，完整的代码在Spring Integration沙箱中

城镇捆绑
城镇（捆绑）已完全接受消息传递。他们正在使用Spring Integration来照顾他们的消息传递和并发性。它们还定义了乡镇居民和市民所需的输入和输出接口（稍后将提到）。

Town接口具有cry方法，我们希望将其实现公开给Town Crier。我们可以让Spring Integration为我们创建一个实现，该实现采用string参数并将其放在包装在Message中的通道上。这是使用网关完成的。


<gateway id="inputGateway" default-request-channel="announcements"
	service-interface="com.springsource.samples.integration.osgi.town.input.Town" />

<publish-subscribe-channel id="announcements" />


public interface Town {
	void cry(String string);
}

由于Town界面只有一个方法，我们不需要告诉Spring Integration要调用哪个方法，因此@Gateway在这里是可选的。

在运行时监听

镇上的公民将在市政厅注册，但不知道消息传递。Citizen可以使用onCry（String string）方法收听哭声，但是对Spring Integration没有任何依赖。在某些情况下，您可能想运行没有Spring Integration依赖项的捆绑软件，但要使用Spring Integration在事件驱动的体系结构中启用它。

该示例在CityHall上的Town捆绑包中使用了服务激活器，该服务激活器负责拆解消息并将其推送给市民：


@ServiceActivator
public void onCry(String announcement) {
	for (Citizen consumer : citizens) {
		consumer.onCry(announcement);
	}
}

为了允许公民进行自我注册，CitizenRegistry作为OSGi服务公开。


<osgi:service id="citizenRegistry" ref="cityHall"
	interface="com.springsource.samples.integration.osgi.town.output.CitizenRegistry" />

公民登记服务还由CityHall实施，以维护向其分发公告的公民收藏。

在抄写员包中，实现了一个公民示例。它通过OSGi引用引用了CitizenRegistry。


<osgi:reference id="citizenRegistry"
	interface="com.springsource.samples.integration.osgi.town.output.CitizenRegistry" />

抄写员在初始化期间将自己注册到公民注册表，然后记录所有被推送的公告。

这样，只需更改城镇捆绑包的代码即可开始使用新系统，而市民捆绑包可以保持不变。当然，可能有充分的理由让多个分发包知道正在进行的消息传递并直接共享频道。在下一节中，我们将对此进行研究。

将频道公开为OSGi服务

假设该镇也与中央政府打交道。政府以政府捆绑形式派遣代表到镇上。该代表直接了解公告频道，因此无需使用网关或服务激活器将其屏蔽。
该镇将announcementsChannel作为OSGi服务公开：


<osgi:service id="announcementsChannel" ref="announcements"
	interface="org.springframework.integration.channel.SubscribableChannel" />

为了将政府法令发送到城镇的公告频道，我们在示例中的政府捆绑包中有一个网关，但这可以是任何端点。重要的是要引用政府捆绑中的城镇捆绑渠道：


	<osgi:reference id="announcementsChannel"
		interface="org.springframework.integration.channel.SubscribableChannel" />

现在，您可以将其用作另一个捆绑包中的普通频道：


<si:service-activator input-channel="announcementsChannel"
	ref="nationalArchive" />

<si:gateway
	service-interface="com.springsource.samples.integration.osgi.government.GovernmentDecreeGateway"
	id="decreeer" default-request-channel="announcementsChannel" />

为了简单起见，该示例实现了非常简单的本地终结点，但是一旦通道桥接了捆绑，就不会阻止您使用Spring Integrations适配器退出jvm。

如何在现实世界中应用？

如果消息负载是中央银行或美联储的利率变化，小镇是内部银行应用程序的中心消息系统，公民是个人交易服务（例如外汇，债券交易，期权和认股权证，等等）。每个都需要模块化，因为


  • 他们由不同的团队经常更换

  • 定期添加更多模块

  • 每个模块需要根据新利率采取不同的措施

假设政府捆绑成为银行的内部借贷/信贷部门（它自己对其他人的借贷利率可能会根据美联储的利率而有所不同-可以建立在不同的体系结构上）。我不是您的财务专家，也不是您所在领域的专家，因此，我不会假定告诉您此技术解决方案的确切应用方式。将来，我们会将样本重构为一个更现实的域，并将其添加到样本中以给您一些想法。如果您有特殊要求，请务必在此处评论。

结论

要设计可伸缩的应用程序，您需要考虑同步切换的问题。消息传递可以解决许多这些问题。为了使大型应用程序最好在单个jvm上运行以获得最佳性能，OSGi将会是一个引人注目的选择。将两者与Spring Integration和dm Server混合使用既强大又简单。在本文中，您已经了解了如何：


  • 使用网关通过捆绑包进行入站呼叫，而没有Spring Integration依赖项

  • 使用服务激活器对捆绑包进行出站呼叫不依赖于Spring Integration

  • 公开一个SubscribableChannel以使消息可以通过使用Spring Integration的捆绑包进行轮询

  • 公开一个MessageChannel以使消息被使用Spring Integration的包推送


另一个消息传递框架和OSGi容器也可以进行相同的设置，但是此示例的简单性提供了一个很好的示例，将很难被击败。是的，这是一个挑战，继续接受诱饵，您知道您想…
