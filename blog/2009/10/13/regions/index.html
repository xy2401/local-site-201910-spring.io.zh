史蒂夫·鲍威尔
2009年10月13日

（2009年10月15日更新）
从里程碑M5开始，dm Server 2.0使用region来将内核与用户应用程序隔离。这意味着内核实现对于应用程序和应用程序管理几乎是完全不可见的。

同样在里程碑M5中，完全删除了对克隆的支持。它们之间的区域隔离和范围划分的计划为克隆旨在解决的最常见问题提供了更简单，更易管理的解决方案。

在以下两节中，我概述了这些更改以及进行更改的原因。

区域新闻


区域就像OSGi框架一样，是安装，解析和运行应用程序的地方。

dm内核创建一个用户区域，在其中运行应用程序，所有应用程序（包括dm Server提供的应用程序-Splash，Admin，Web和托管存储库）都部署到用户区域中 。

内核捆绑包未安装在用户区域中 （除了一些用于区域管理的区域）。支持内核的必要功能在OSGi框架中运行，但是用户区应用程序看不到它，除了通常提供的服务。这些应用程序与内核实现隔离 。

这种隔离有很多好处：例如，内核和用户的应用程序不再需要使用相同版本的Spring Framework。（实际上，内核并没有安装所有的Spring Framework，也不需要安装。）如果更新了内核，则需要升级或调整应用程序以适应这种情况的可能性就大大降低了。因此，内核实现更加稳定和具有弹性，并且应用程序更有可能在发行版之间进行内核升级。

欢乐地区



  • 使用命令行管理程序或管理控制台，您只会看到用户区域 -内核实现捆绑包不可见，从而简化了管理视图。[注：2009年10月15日添加：在用户区域中仍然可以看到一些内核捆绑包，因为内核已经在其中安装了它们用于区域管理。

  • 该设计具有很强的扩展能力，可以在将来的版本中容纳多个用户区域。管理区域的开销很小且固定—实现是可伸缩的。

  • 用户区域的基本初始化是可配置的，并且完全独立于内核。

  • 区域的实现是完全通用的，它完全不依赖于Spring或Spring DM。如果需要，可以根据需要配置完全不支持Spring的用户区域。


发送克隆


Rob于4月在dm服务器路线图中描述了克隆。克隆是分离的两个应用程序之间的依赖关系的方式，否则将不应该被共享， 或尝试不能共享份额的文物要么份额文物。

例如，可能不希望共享具有静态状态的捆绑软件，并且能够拥有此类捆绑软件的副本（克隆）意味着不同的应用程序可以使用不同的克隆（应用程序作用域机制允许我们在操作系统中安装多个副本。框架）。在另一种情况下，当较低级别的包通过公共中介（通常是Spring框架的一部分）被依赖项固定时，克隆中介将允许克隆依赖于不同的情人级别的绑定，从而将克隆释放给满足原始中介不能满足的传递约束。

在dm Server 2.0的里程碑版本（最多M4）中，提供了克隆支持（手动和自动）。我们从这种实现中获得了很多反馈和经验，并且作为一个实验，它被证明是富有成效的。但是，时间和经验表明，该产品尚未准备就绪，因此从M5中删除了克隆，因此不会成为最终2.0版本的一部分。区域隔离以及范围界定的计划可用于管理克隆旨在解决的最常见问题。这些解决方案更容易理解， 更易于在生产中提供支持。

一个细心的克隆


删除克隆的最重要原因是自动克隆太脆弱且不可预测：

  • 它在工作时非常出色，但总的来说可能会花费很长的时间（失败或成功）。

  • 这是非常难以听懂克隆发生及其原因，并不能保证相同克隆的解决方案将获得每次对同一制造品的部署。


这些不是大多数系统管理员想要的用于生产系统的属性（尽管有时可以很方便地进行开发）。在生产中，我们需要一个可预测的配置，每次都可以依靠它以相同的方式工作（否则很快就会失败）。

手动克隆相对稳定，但即使在这里也存在问题：


  • 在固定情况下（上面已概述），可能需要克隆很多中介。

  • 事先确定需要哪些克隆并非易事。（正是这一困难使自动克隆很难有效地完成。）


两种克隆解决方案还取决于作用域，并且这些解决方案不适用于无范围的应用程序。

最后，克隆实现必须包括Spring依赖项作为特例，以便有效地工作。Spring DM扩展程序还引入了一些特殊情况：这些情况对于其他扩展程序和其他库根本无法一概而论。它太特定于Spring。

总体而言，尽管克隆和自动克隆的一些成就令人印象深刻，但该功能仅不稳定到无法在现场支持，因此出于整体质量和稳定性的考虑而将其删除。

最重要的问题可以通过区域隔离和显式作用域解决。

高音和低音


尽管可能有人会哀悼克隆支持的传递（虽然很有趣，但不是为了调试），但是区域隔离和现有范围内的应用程序（和计划）为常见的依赖问题提供了更简单，更可靠的解决方案。

固定问题几乎都是通过Spring框架产生的，因此用户区域的隔离为大多数这些问题提供了解决方案。

其他常见情况可以通过适当使用范围内的应用程序来解决，这实际上是管理服务器中安装的伪像的更可控制和可支持的方式。范围计划使创建所需范围变得容易，并且随着框架上下文的变化，分辨率连线在生产期间保持稳定。
