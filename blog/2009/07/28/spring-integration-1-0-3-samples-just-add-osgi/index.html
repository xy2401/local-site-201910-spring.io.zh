Spring Integration 1.0.3示例：仅添加OSGi

工程
奥列格·朱拉库斯基（Oleg Zhurakousky）
2009年7月28日

介绍

上周，马克·费舍尔（Mark Fisher）向您介绍了新发行的Spring Integration 1.0.3附带的新的经过重组和简化的Spring Integration示例 ，到目前为止，反馈是非常积极的。
除了重组和简化现有样本外，我们还介绍了一些新样本，目的是演示在基于OSGi的平台上运行Spring Integration的一些好处。
在此博客中，使用非常琐碎但功能强大的示例，我们将探讨Spring Integration和OSGi一起用于解决当今企业的动态特性时的一些好处。

Spring Integration是基于POJO的轻量级可嵌入消息传递框架，具有松散耦合的编程模式 l，旨在简化基于Enterprise Integration Patterns的异构系统的集成 ，而无需单独的类似ESB的引擎或专有的开发和部署环境。另一方面， OSGi是一种范式，它允许人们从称为OSGi Bundles的独立模块组成松散耦合的系统 。从一组独立开发的模块组成系统可能不是一个新的范例，我们（希望）已经做了很多年了。话虽如此，OSGi模块化真正优势的实现不是来自其静态打包模型，而是来自于了解其部署和运行时动态以及如何很好地应对当今业务流程的动态。

因此，让我们看一个简单的示例，它是基于消息交换和OSGi动态的集成如何相互补充，从而实现功能强大且动态的系统。

Spring Integration样本随Spring Integration一起提供。您也可以从此处独立下载它们。为简单起见，这些示例是作为SpringSource Tool Suite （STS）项目开发的，同时利用dm Server工具与SpringSource dm Server （基于OSGi和Spring的Enterprise Java平台）进行快速集成。但是，由于这些示例是基于OSGi的项目，因此可以在任何经过正确配置的OSGi平台上充分运行。

样品


这两个样本建立在非常简单和熟悉的生产者/消费者概念上。
第一个osgi-inbound捆绑包是一个生产者，它使您可以生成将发送到消息通道的消息。第二个包osgi-outbound是使用者，它将使用osgi-inbound包放置在通道上的消息，并将消息写入文件。

这个非常简单而琐碎的概念演示了基于Spring Integration提供的消息传递模型的两个系统之间的松耦合集成。
但是，在现实世界中，我们在尝试将两个系统集成时必须面对的问题之一是此类系统的独立生命周期，其中它们的行为发生变化 ，新系统的添加和/或旧系统的寿命终止是正常现象。通常，此类更改不仅需要更改代码，还需要在服务器完全重启后重新部署整个整体（例如EAR，WAR）应用程序。
Spring集成的POJO编程模型非常适合解决这些系统的松散耦合特性，因为对一个系统的更改很少会影响另一个系统。但是，其生命周期动态如何？
让我们假设关于（必须在我们的小样本范围内）文件必须写入何处或如何写入的要求已更改。“哪里”和“如何”都是osgi出站捆绑包的责任。
在正常情况下，对osgi-outbound捆绑包功能的任何更改都将要求对系统进行完全刷新（即，重新部署整个系统并重新启动服务器）。当您的系统仅由两个捆绑软件组成时，这可能不是一个大问题。但是，如果超过两个呢？您是否准备仅由于必须写入消息的目录已更改或引入了应记录每条传入消息的新子系统而重建和重新部署打包为WAR或EAR的整个系统？

这是OSGi及其服务层，最重要的是OSGi服务动态性提供了很大的帮助。因此，让我们以上述样本需求为基础，看看如何根据扩展的当前样本实现这些需求。首先，让我们查看osgi-inbound捆绑包的Application Context配置：

<osgi:service id="inboundService" ref="inboundChannel"
interface="org.springframework.integration.channel.SubscribableChannel"/>

<integration:publish-subscribe-channel id="inboundChannel"/>

<integration:gateway id="inboundGateway"
service-interface="org.springframework.integration.samples.osgi.inbound.InboundGateway"
default-request-channel="inboundChannel"/>

如您所见，一个非常简单的配置定义了一个网关代理，该网关代理允许以POJO方式发送消息，该消息将被存入配置为发布-订阅通道的入站通道。但是，更令人感兴趣的是，此通道已作为服务通过元素导出到OSGi服务注册中心，从而允许在当前和将来的捆绑软件之间建立松散耦合但动态的合作模型。

我们还回顾一下osgi-outbound捆绑软件的Application Context配置。

<osgi:reference id="filesIn"
interface="org.springframework.integration.channel.SubscribableChannel"/>

<file:outbound-gateway id="filesOut"
request-channel="filesIn"
directory="${java.io.tmpdir}/spring-integration-samples/output"
delete-source-files="true"/>

此捆绑包的目的是在捆绑包出现后立即动态订阅由osgi-inbound捆绑包播发的频道，并将消息写入到由osgi-inbound捆绑包配置的目录中的文件中元素，因此考虑了OSGi服务层提供的动态。
如您所见，通过osgi-inbound捆绑包导出的' inboundChannel'OSGi服务现在通过元件。
现在，由OSGi服务前置的入站通道已准备好在运行时动态接受和/或丢失订户。由于OSGi服务的动态性，我们还可以更新配置或完全重新设计/重新实现osgi-outbound捆绑包，而不会影响系统的生产部分（osgi-inbound）。因此，让我们继续更改要写入文件的目录（确保dm Server仍在运行……实际上忘记在本博客期间将其停止）

开放OSGi的出站项目- > 的src - > META-INF - > Spring - > OSGi的outbound.xml并添加子目录的'目录'的配置（在这种情况下它的'富'）

保存文件。在几秒钟内，您将看到已重新部署osgi-outbound捆绑包
打开OSGi或服务器控制台，然后像以前一样发送另一条消息，然后查看新文件是否已写入您刚刚指定的目录。希望是;）

结论


请记住，尽管在这个琐碎的示例中，我们仅处理两个捆绑包，但这些类型的生产者/消费者捆绑包本身也可以用作系统其他部分的网关，并且这些部分自然是动态的。例如，假设您要接收有关正在写入的文件的电子邮件通知。使用SI和OSGi，您唯一需要做的就是创建另一个捆绑包，将子系统的该部分表示为osgi-inbound捆绑包广告的渠道服务的使用者 ，并且在不需要时将其取消订阅，通过停止或取消部署此使用者而不影响系统的其余部分。实际上，请继续尝试开发另一个捆绑包，作为osgi-inbound的使用者 ，或者发挥您的想象力，开发另一个捆绑包，该捆绑包将对osgi-outbound捆绑包所写的文件有所帮助。
