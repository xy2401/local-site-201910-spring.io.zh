使用Yourkit查找内存泄漏

工程
戴夫·瑟
2009年7月5日

今天，我在Yourkit方面拥有非常棒的经验， 以至于我想写一个快速的插件。自从我在愤怒中使用它已经有两年了，即使那时它也是我能找到的最好的工具，但是现在它确实非常光滑。我没有对市场进行详尽的调查，这并不是练习的目的：我只是想要一种工具来解决问题。

这是我今天的故事；沮丧，然后是烦躁，然后是最后的满足。我在Spring Batch中怀疑有内存泄漏，因此需要快速对其进行跟踪。背后的故事是我已经看到大量的内存泄漏，但是我已经有相当长的一段时间没有面对煤层了。这些天我住在STS中 （有时也梦想成真），所以我需要在IDE中运行良好的工具。我尝试了两种工具，但这仅仅是因为第一种选择不起作用。我尝试过的两个是TPTP和Yourkit 。

我用JUnit编写了一个简单的集成测试，并使用这两个工具对其进行了概要分析。从内存分析中我可以看出，在过程结束时，堆中也可能存在JobExecution类型的对象，那么我所需要知道的是谁在保留对它们不应该的引用？似乎是一个简单的问题。

由于Eclipse是STS的核心，因此我想看看Eclipse性能监视工具（TPTP）是否比上次尝试的要好。经过半天的混乱，简短的回答是“不是”。长版本是没有安装（在Ubuntu Ibex上）晦涩的过期共享库就根本无法工作，即使运行，在内存分析中也几乎没有有用的信息来跟踪泄漏。Google可以帮助您解决第一个问题（我不知道为什么TPTP文档在该领域如此差劲），但不能解决第二个问题。我可以从TPTP中获取足够的信息以查看是否存在泄漏，但是我仍然不知道泄漏的来源。问题在于报告仅告诉您堆中的对象是什么以及最初分配它们的代码行，这在上游太远了，以至于在过程结束时不知道谁对它们有引用。结论：TPTP比几年前好了多个数量级，但实际上仍然没有用。

因此，我继续学习Yourkit，知道它可以轻松解决此问题，但对安装和工具感到担忧。我发现Eclipse工具现在非常出色（上次我发现它不见了），甚至更好的是，经过大约5分钟的工作，我发现了内存泄漏的根源。还不错。如果您想了解杀手级功能，那就很简单：浏览堆和所有对象引用的能力。它也有助于在过程结束时自动创建快照，因此您可以编写简单的单元测试，从IDE中对其进行概要分析，然后单击两次即可将堆加载到Yourkit UI中。

如果您想知道，Spring Batch问题很小（并且仅影响快照发行版），并且很容易解决。

我知道SpringSource的Mark和Filip等人正在使用Yourkit（例如，请参阅此博客条目 ），我问的每个人都非常喜欢它。但是，如果我们在文章和咨询活动中提到它，那么它往往会与许多其他内容一起被掩埋，因此，我希望这篇小文章能使Yourkit突如其来。
