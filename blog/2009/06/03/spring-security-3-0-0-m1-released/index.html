Spring Security 3.0.0。M1发布

工程
卢克·泰勒
2009年6月3日

我们高兴地宣布，Spring Security 3.0的第一个里程碑现已可供下载 。也可以通过Maven里程碑资料库（ http://maven.springframework.org/milestone）获得该版本。与Spring 3.0一样，这是第一个发行版本，需要最低的JDK 1.5运行并且还需要Spring 3.0，因此您应该掌握Spring 3.0.0。M3版本（如果您尚未使用它的话）。那么，此版本中有哪些新功能和更改内容？

基于表达式的访问控制


该版本是第一个包含基于Spring-EL的授权支持预览的版本。现在，您既可以在方法注释中使用表达式，也可以在Web安全中使用表达式。与熟悉的属性和基于投票者的机制相比，这开辟了许多新的可能性。一个简单的例子可能是一个好的开始。这是一个使用安全名称空间的Web应用程序：


  <http use-expressions="true">
     <intercept-url pattern="/secure/**" access="hasRole('ROLE_SUPERVISOR') and hasIpAddress('192.168.1.0/24')" />
     ...
  </http>;

内置的hasRole（'ROLE_SUPERVISOR'）表达式没什么用，它只是检查当前用户的权限列表，如果他们具有给定的角色，则返回true。但是我们添加了一个额外的表达式来指定发出请求的IP地址必须在subnet / mask参数定义的范围内。这可能会或可能不会实际使用，具体取决于您的网络设置，但它说明了基于表达式的方法的强大功能。以前包含在“访问” XML属性中的安全属性列表已由布尔表达式替换。如果评估结果为“ true”，则将授予访问权限。如果评估结果为“假”，则访问将被拒绝。

@Pre和@Post批注


方法安全性比我们刚刚针对Web请求看到的简单允许或拒绝要复杂得多。为了为具有方法安全性的表达式的使用提供更全面的支持，我们引入了四个新的批注，其中可以包含在方法调用之前和之后应用的表达式属性。为了支持它们， global-method-security名称空间元素上有一个新属性：


    <global-method-security pre-post-annotations="enabled"/>

最明显有用的注释是@PreAuthorize ，它控制是否可以实际调用方法。例如（来自“联系人”示例应用程序）：


    @PreAuthorize("hasRole('ROLE_USER')")
    public void create(Contact contact);

这意味着只有角色为“ ROLE_USER”的用户才能访问。这里没有新内容。但是关于：


    @PreAuthorize("hasPermission(#contact, 'admin')")
    public void deletePermission(Contact contact, Sid recipient, Permission permission);

在这里，我们实际上是将方法参数用作表达式的一部分，以确定当前用户是否对给定的联系人具有“管理员”权限。hasPermission（）表达式通过应用程序上下文链接到Spring Security ACL模块中（有关如何实现的信息，请参阅Contacts示例配置）。您可以按名称作为表达式变量访问任何方法参数，前提是您的代码已编译了调试信息。表达式中提供了任何Spring-EL功能，因此您也可以访问参数的属性。例如，如果您想要一种特定的方法仅允许访问其用户名与联系人的用户名匹配的用户，则可以编写


    @PreAuthorize("#contact.name == principal.name)")
    public void doSomething(Contact contact);

在这里，我们访问另一个内置表达式，它是从安全上下文中获取的当前Spring Security Authentication对象的“主要”。您也可以使用表达式名称“ authentication”直接访问Authentication对象。也可以在方法调用之后使用@PostAuthorize批注执行授权。要从方法访问返回值，请在表达式中使用内置名称“ returnObject”。

筛选


您可能已经知道，Spring Security还支持集合和数组的过滤，现在可以使用表达式来实现。这通常在方法的返回值上执行。例如：


    @PreAuthorize("hasRole('ROLE_USER')")
    @PostFilter("hasPermission(filterObject, 'read') or hasPermission(filterObject, 'admin')")
    public List getAll();

当与@PostFilter批注一起使用时，Spring Security遍历返回的集合并删除提供的表达式为false的所有元素。名称“ filterObject”是指集合中的当前对象。您也可以使用@PreFilter进行方法调用之前的过滤，尽管这可能是不太常见的要求。语法是一样的，但是如果有多个参数是集合类型，则必须使用此批注的“ filterTarget”属性按名称选择一个。

代码库重组


在3.0之前的版本中，Spring Security的大多数代码都包含在spring-security-core jar中。多年来，随着添加了更多功能，在代码库本身以及第三方库中跟踪依赖关系变得越来越困难。例如，对于用户而言，很难确定框架中特定功能集所需的核心Maven pom.xml中列出的依赖项。此外，自从2003年该框架起源于Acegi Security以来，原始的包结构和类名一直存在，当时仅支持一些基本的身份验证机制。随着代码数量的增加和功能集的扩展，此程序包结构已开始显示其时代。

Spring Security 2.0.4包结构


上图显示了由Structure101生成的2.0.4版本中的core，core-tiger，cas-client和acl jars的高级打包图。您不必一定是代码结构方面的专家，就可以意识到这里存在一些问题。程序包中有很多循环引用，没有明确的整体依赖关系结构。跨jar边界拆分软件包时还存在一些问题，这可能会导致OSGi问题。随着Spring Security的发展，代码结构的这种脆弱性可能会导致维护开销，因此决定重新构建3.0版本的代码，以为我们的未来开发提供稳定的基础。让我们看一下事物的组织方式。

工程罐


我们要做的第一件事是将核心分成几个罐子。spring-security-core jar现在仅包含基本身份验证和访问控制代码，并且更加干净。例如，它不依赖于LDAP或servlet API，并且现在有用于Web特定代码和LDAP的单独jar。我们还将名称空间解析代码分解为一个单独的jar，因为它依赖于其他大多数jar，并且不会公开任何可能直接在应用程序中使用的公共API。仅当在应用程序上下文XML文件中使用Spring Security名称空间配置时，才需要使用它。下表显示了主要的项目jar。

其他杂项更改


重命名的班级


代码库中的某些类名称也开始显示其年龄。例如， AbstractProcessingFilter现在是AbstractAuthenticationProcessingFilter ，它最常见的具体子类AuthenticationProcessingFilter现在是UsernamePasswordAuthenticationProcessingFilter ，因为框架中现在有很多过滤器可以处理身份验证请求。现在其对应的AuthenticationEntryPoint是LoginUrlAuthenticationEntryPoint 。晦涩难懂的名称ObjectDefinitionSource已重命名为SecurityMetadataSource ，并且AbstractSecurityInterceptor上的相应属性已类似地重命名。实现类和子接口也已类似地重命名。HttpSessionContextIntegrationFilter现在是SecurityContextPersistenceFilter，并且具有可插拔策略来控制其存储安全上下文的方式–不必存储在HttpSession中 。

身份验证成功或失败的重定向/转发


AbstractProcessingFilter的另一个问题是，在成功或失败的身份验证之后，控制浏览器重定向到目标位置的不同方法数量很多。使用了各种属性和策略。在Spring Security 3.0.0中。M1，已由两个单独的策略AuthenticationSuccessHandler和AuthenticationFailureHandler代替，它们分别负责处理目标。请查看这些接口的Javadoc，并查看问题SEC-745以获取更多信息。

参考手册和网站更新


显然，3.0参考手册仍在开发中：-)，但我们一直在进行一些更新。命名空间和技术概述各章应提供框架的合理概述。请随时通过问题跟踪器提交任何想法或补丁。当然，如果发现任何错误，也是如此。

项目网站也已更新。常见问题解答有一些新问题，并且有一个新页面，其中包含演示视频和在线文章的链接。请看一下，让我们知道您是否想在那里查看（外部安全链接页面已经在开发中）。

结论

Spring Security 3仍然有很多工作要做，但是我们希望您能试用此里程碑版本并提供有关新表达语言支持的反馈。您可以在Jira ChangeLog中找到迄今为止的更改的完整列表，还可以在此处找到当前项目路线图的链接。

社区论坛是询问有关使用Spring Security或开始讨论新功能的最佳场所。另外，如果发现不对劲，可以提出《 吉拉问题》 。

我们希望您喜欢使用Spring Security 。
