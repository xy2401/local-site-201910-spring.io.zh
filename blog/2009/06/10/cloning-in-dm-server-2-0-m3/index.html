在dm Server 2.0 M3中克隆

工程
格林·诺明顿
2009年6月10日

克隆是dm Server 2.0中的功能，可将某些捆绑包和库复制到路线图中描述的范围内的应用程序（即PAR或范围内的计划）中。

在过去的sprint中，dm Server中对克隆的支持一直在稳步发展。M1中存在基本机制 ：可能会触发克隆：


  • 通过指定伪指令共享手动进行：==克隆导入库或导入捆绑

  • 当范围限定的应用程序由于使用约束冲突而无法解析时，将自动执行此操作。

从那以后，对代码进行了一些整理，针对克隆的Spring Framework 的常见情况添加了主要的性能优化，添加了日志消息以指示已克隆了哪些捆绑软件，改进了跟踪，并修复了一些错误。

我们注意到手动克隆是一种相对安全的操作，因为它完全在用户的控制之下。但是，自动克隆始终是推测性的。它是由OSGi解析器故障驱动的，尤其是违反使用限制的情况（如先前博客所述 ）。克隆无法避免某些使用约束违规，但是直到尝试自动克隆并且仍然存在使用违规时，我们才能知道。这可能涉及运行解析器和克隆一个或多个捆绑软件的多次迭代。有时，特别是如果克隆了Spring Framework 包，则还必须克隆Spring DM扩展器束和一些相关的机器。

在最坏的情况下，所有这些处理可能要花费几秒钟甚至几分钟。因此，我们添加了一个配置选项以允许关闭自动克隆。此选项的一种用途是开发一个启用了自动克隆的应用程序。如果必须使用自动克隆来解决该应用程序，但性能不可接受，则可以使用手动克隆（并且可以禁用自动克隆）。

在接下来的两个冲刺中，dm服务器团队的目标是交付M3。在此之前，我们希望您对社区（即您）更喜欢此配置选项的默认值提供反馈。您是否要默认启用或禁用自动克隆？

默认情况下启用它的好处是，自动克隆支持将倾向于获得更多的使用，并且任何剩余的错误更有可能被充实。默认情况下禁用具有的好处是，除非用户选择加入，否则他们不会面对自动克隆的相对复杂性和可能的性能成本。我希望至少在开发的这个阶段默认启用它，但是我还不确定在我们最终发布2.0时默认应该是什么。

除了M3之外，克隆支持将被重构到新的部署器体系结构中，但在另一场合会更多。
