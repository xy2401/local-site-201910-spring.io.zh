STS 2.0中的OSGi开发工具

工程
克里斯蒂安·杜普依斯（Christian Dupuis）
2009年3月5日

昨天，我们发布了即将发布的SpringSource Tool Suite版本2.0的最终版本。RC版本可从STS产品页面上用于Windows，Mac OS X和Linux。

STS 2.0配备了用于开发Spring应用程序的新的生产力工具，例如快速修复，键入时验证和更正，项目和bean创建向导以及Visual Spring Configuration Editor等。请继续关注有关这些新功能的更多信息。

此外，STS 2.0附带了用于基于OSGi的应用程序开发的工具，有经验的人以及OSGi的新采用者可能会觉得很有趣。我想使用此博客简要介绍这些功能。

现有工具

概括地说，让我们首先概述自带 STS的SpringSource dm Server和开源dm Server工具发布以来可用的OSGi开发工具：

  • dm Server集成：STS将dm Server集成到Eclipse WTP中，使您可以管理集成的dm Server实例（启动/停止/调试）的生命周期以及部署OSGi捆绑包，PAR和传统的JEE WAR。如果用户更改了代码，则dm Server集成还可以透明地处理已部署工件的刷新和更新操作。
  • 项目向导：要创建OSGi捆绑包和PAR项目，STS提供了两个项目向导，这些向导可自动设置项目结构和所有必需的Eclipse元数据。对于传统的WAR，仍可以使用WTP动态Web项目向导。
  • 清单编辑和验证：STS为MANIFEST.MF文件提供内容辅助，超链接和验证，以使管理OSGi元数据的任务尽可能容易。常见问题可以通过使用集成的快速修复程序自动解决（例如，下载缺少的依赖项或正确的版本范围）。
  • 类路径管理：STS将从MANIFEST.MF中获取OSGi依赖项元数据，并创建一个应用了正确可见性规则的类路径容器。正在针对本地dm Server捆绑包和库存储库解决依赖性。可以将仅测试依赖项添加到测试源文件夹的META-INF文件夹中名为TEST.MF的文件中。
  • 仓库浏览器：STS集成了SpringSource Enterprise Bundle仓库以提供对可用企业库的集成访问。这使得下载和安装第三方依赖项包非常容易。

STS 2.0的新功能

STS 2.0提供了新的工具，可基于导入/导出包关系和服务使用情况来分析包并可视化其依赖性。可访问的信息类似于用户通常从OSGi运行时的管理外壳（例如Equinox控制台）以文本形式获取的信息。但是，这些新工具的好处在于，您可以将相关信息集成到开发环境中，并且可以使用可视化工具来导航和检查包元数据和连线。无需离开IDE即可打开telnet会话。

以下屏幕截图（单击放大）显示“服务器编辑器”的“捆绑包概述”页面。该页面列出了所有已安装的捆绑软件，并以清单形式提供对捆绑软件元数据的访问，以及从OSGi运行时可访问的所有接线信息。专用部分显示了导出和导入的软件包及其各自的使用者和提供者，以及已注册的OSGi服务，包括其属性和使用者。

显示的信息是从正在运行的dm Server中检索到的，因此，如果安装，停止或更新了捆绑软件，则会立即反映出更改。在捆绑包概述或Equinox控制台中导航特定捆绑包的依赖关系可能会非常麻烦，因为两者都无法可视化整个依赖关系图。因此，为了分析布线问题，用户需要从一个束到另一个束进行迭代。

通过允许您直观地浏览依赖关系图，新的捆绑包依赖关系图使此任务非常容易。该图可以显示软件包或服务的依赖性。另外，可以更改图中显示的最大依赖深度。

我建议您自己尝试一下，因为从静态屏幕快照中可以想象到图表运行起来会更加有趣。

最后但并非最不重要的一点是，一些用户喜欢留在他们习惯的环境中。这就是为什么我们也将文本控制台集成到STS中的原因。服务器控制台（见下文）支持与Equinox控制台完全相同的命令，但在顶部提供命令完成和命令历史记录。

结论

突出显示的功能不是STS 2.0的唯一附加功能，而应提供有关STS为OSGi开发提供的功能的见解。随着本月下旬我们与Google Analytics（分析）的联系更加紧密，我们将发布涵盖其他功能区域的更多信息。

不过，这是对您的Spring或OSGi项目尝试STS 2.0 RC的正确时机，并让我们知道您的想法。请报告您在产品的JIRA或社区论坛中可能发现的任何问题。
