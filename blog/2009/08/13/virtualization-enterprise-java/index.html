虚拟化和企业Java

工程
阿德里安·科耶尔（Adrian Colyer）
2009年8月13日

如果您想在战略层面上了解VMware最近宣布的收购SpringSource的意义，可以参考很多好资料，包括Steve Herrod的博客（VMware的CTO） ， Rod Johnson的评论 ， Paul Maritz的媒体和分析师电话 以及 Darryl Taft在eWeek中颇有见地的文章 。

在这篇文章中，我将在技术层面上更加关注这一切的含义，以使您了解可以期待的各种功能。

首先，让我重申一下，对于我们的开源项目和SpringSource产品而言， 没有任何改变 。除了将来我们将有更多机会为它们添加令人兴奋的新功能这一事实外，没有任何改变。Spring 3.0即将发布，我们刚刚发布了 里程碑4 。 dm Server 朝着 2.0版本 迈进了快速的步伐 ，而即将推出的tc Server 版本 ，我们还有很多很棒的东西。在 对Groovy Eclipse的工具支持 正在产生的利息群众，Grails是推迈向 1.2版本 ，和令人兴奋的事情在我们的Spring项目发生。所有这些将继续进行。

蓝图的力量


每个Spring支持的应用程序都有所谓的“ 应用程序蓝图 ”。此蓝图包含组成应用程序的所有组件（bean）的描述，如何配置和连接它们，如何与周围环境交互以及应如何处理诸如安全性和事务之类的跨领域问题。蓝图在运行时由BeanDefinitions集合表示。BeanDefinitions来自各种来源，包括XML定义，批注，JavaConfig，Groovy DSL以及可以插入Spring的任何其他配置机制。Spring容器的工作是获取您为应用程序指定的蓝图，并“确保如此”。应用程序蓝图使Spring对您的应用程序的运行情况具有高度的了解。我们将回到那个想法……

在生产环境中部署Spring支持的应用程序时会发生什么？在典型情况下，需要配置和连接多个协同工作的组件。例如， 跨一组 tc服务器 实例 的 http服务器 负载平衡 ，这些实例又与在主/从设置中配置的数据库通信。这些（中间件）组件构成了应用程序的逻辑层（现在使用术语“大型”应用程序）。在实际部署中，逻辑层将映射到物理层（例如，您可以将数据库和应用程序服务器部署在同一台计算机或不同的计算机上）。最初发明此术语时，物理层实际上是物理层。但是如今，您的物理层当然可以是虚拟的，而这些虚拟机又被映射到物理资源上……..仍然与我在一起？？？


就像我们有一个描述蓝图驱动的应用程序的组件以及它们如何装配的应用程序蓝图一样 ， 部署蓝图可以描述给定部署方案的组件-有哪些组件，如何连接和配置它们以及如何处理诸如安全性和（反）亲和性等跨领域问题。首先，可以在目录中捕获一些常见的部署模式（例如我之前提供的tc Server服务器场示例）。随着时间的推移，您可以想象一个运营团队使用自己的自定义蓝图扩展该目录，以进行应用程序部署。

从部署蓝图到vApp


将应用程序投入生产应该很简单，包括开发具有相关应用程序蓝图的应用程序，选择适合该应用程序样式的部署蓝图（例如，Web应用程序，批处理，集成等），然后单击“部署”。您已经在实践中看到了该模型的早期示例，例如， CloudFoundry 支持Amazon EC2上的Web应用程序部署蓝图。

VMware vSphere 包含对称为vApp的概念的支持。 vApp是“包含一个或多个虚拟机的逻辑实体，它使用行业标准的开放虚拟化格式来指定和封装多层应用程序的所有组件以及与之关联的操作策略和服务级别。”

vApp是体现部署蓝图的理想打包单位。您的数据中心和公共 vCloud 可以支持相同的vApp 。vApp还可以公开配置属性-运营商在部署vApp时会为这些属性提供值。

从 dm Server 开始 （请注意即将发布的2.0.0中的更多详细信息。M5版本），我们使通过vApp属性配置中间件成为可能。这使操作员可以在部署vApp时覆盖端口和其他配置设置，而无需了解有关虚拟机或内部中间件组件的配置的任何信息。该功能也扩展到了中间件组件之外，您还可以配置在部署时从操作员指定的vApp属性中获取的应用程序属性（将由Spring注入依赖项）。


可以通过多种有趣的方式组合这些功能，但让我选择两种我认为可以说明此处潜力的方法： 平台即服务（PaaS）模型 ；以及应用设备模型 。

在平台即服务模型中，您的数据中心或注册为vCloud服务提供商的多个供应商中的任何一个都提供了可供选择的部署蓝图目录。在PaaS的意义上，每一个都可以视为平台，您可以在该平台上部署应用程序。您选择要部署到的平台，代表您配置相应的vApp（也许通过Web前端，可以让您指定蓝图公开的任何vApp属性），然后将应用程序工件上传到您已配置并正在运行的平台实例。对于使用 Grails 或 Roo 构建的应用程序 ，我们甚至可以进一步了解您的应用程序的结构，可以直接从Grails（或Roo）命令行通过插件提供部署蓝图选择和工件上传。考虑一下这种模型将开放的此类应用程序的托管机会！

在应用程序设备模型中，开发或运营团队选择一个初始部署蓝图，创建相应vApp的实例，然后将应用程序工件安装到正在运行的系统中。到目前为止，这看起来就像PaaS模型。但是接下来发生的事情是不同的。虚拟机（现已安装了应用程序工件）打包为新的vApp，并且每个部署上可能更改的任何特定于应用程序的属性（例如，如果vApp依赖于外部数据库，则为数据库URL和密码）配置为vApp属性。因此，现在整个应用程序以及运行该应用程序所需的一切都打包为一个vApp（应用程序设备），可以作为一个单元进行配置（并控制版本）。然后，将应用程序投入生产就像部署vApp一样简单-不会出错，所有内容都经过预先打包和测试。

智能配置


希望到现在为止，您已经开始看到与vApp模型相关的部署蓝图如何简化从开发到生产的过程。但是，这种方法不仅使事情变得更快，更简单，而且还使配置更智能。

在没有Spring知识的情况下，vApp只是vSphere可以在其可用的物理资源上配置的虚拟机的集合。但是，当通过应用程序蓝图和部署蓝图的知识完成配置时，事情会变得更加有趣。现在，我们突然对应用程序和中间件组件以及它们之间的连接方式有了一些了解，并且我们可以优化虚拟基础架构来支持它。例如：


  • vSphere可以自动创建 vShield区域，以便只能从Web服务器访问应用程序服务器节点，并且只能公开访问Web服务器节点。

  • vSphere可以自动设置反关联性组，这样就不会在与从数据库相同的物理硬件上配置数据库主数据库

  • vSphere可以根据蓝图暗示的预期流量模式优化网络配置

  • …


此外：

  • 可以将蓝图内置在 Hyperic HQ管理服务器中 ，该代理在vApp中的所有虚拟机上运行。由于我们了解蓝图，因此可以自动创建适当的总部组（例如，将所有tc服务器作为一个逻辑资源进行管理），自动添加清单以及默认情况下适当的控制措施和警报设置。无需手动安装和配置任何这些。


智能运行时管理


智能配置很棒，但还不止于此。 vSphere包含完善的机制，可在运行时优化虚拟机和数据中心的使用。迄今为止，这些机制在不了解虚拟机试图支持的应用程序的情况下就已经运行。当您将Hyperic HQ的应用程序见解与vCenter的虚拟基础架构见解相结合时，就可以创建一个组合的应用程序运行状况和管理模型，从而基于应用程序SLA优化运行时。例如，如果HQ检测到来自tc Server节点的响应时间已接近SLA中指定的上限，并且这与表明CPU或内存是瓶颈的指标相关联，则可以采取一些纠正措施，包括分配更多的物理使用 vMotion 将虚拟机 资源转移到现有机器上的虚拟机上，使用 vMotion 将虚拟机迁移到功能更强大的硬件上，或者旋转其他tc Server虚拟机来分担负载。

关于扩展（或缩减），扩展点只是扮演该角色的服务器的部署蓝图“ 1..n”（或“ 3..8”，或您决定的任何内容）中的另一元数据。指定了这些之后，只需让Hyperic HQ和vCenter协同工作即可为您管理和优化服务器的数量（即使不需要关闭临时关闭的物理机也可以节省能源成本）-全部基于您指定的应用程序SLA和虚拟基础结构SLA。

触手可及


尽管在此博客文章中，我主要侧重于实际部署到生产中，但该技术在开发过程中也非常有用。想象一下，拥有一个虚拟机目录，该目录代表您的应用程序必须在其中运行的所有不同环境（不同的浏览器，操作系统，应用程序服务器等），并且能够从以下任何一种直接启动和测试您的应用程序你骑。想象一下，当QA期间发生难以重现的错误时，能够快照一组虚拟机的状态，然后让开发人员直接从 SpringSource Tool Suite 内部启动并分析这些VM的副本 。想象一下，能够在具有代表性的生产环境中进行基本的规模和性能测试，而不必麻烦地进行实际设置。想像…

来吧…


另一波创新浪潮，另一个行业破坏点。在SpringSource，这些是我们热爱的挑战，也是我们蓬勃发展的挑战。来吧！
