最新Cloud Foundry中的新功能

工程
查尔斯·李
2009年11月23日

我们最近更新了CloudFoundry.com 。在此更新中，我们引入了一些令人兴奋的新功能，这些功能为将来的开发铺平了道路。我想借此机会描述和解释这些功能，并为它们的产生提供一些背景知识。我们很高兴您能试用它们并向我们提供反馈。

成本追踪


尽管云的按使用付费模型可以减少支出，但跟踪实际使用情况和成本很重要，这样它才不会导致意外的高额账单。以前，Cloud Foundry会将其当前的部署支出率告知用户。这对于了解计量费率很有用，但是对于应用程序部署的实际成本并没有提供清晰的了解或统计信息。Cloud Foundry的最新更新为成本跟踪提供了新的历史功能。对于正在运行的部署，Cloud Foundry将显示到目前为止的成本。对于不再活动的部署，将显示在整个生命周期内运行该部署的总成本。


除了防止每月的账单使您感到意外之外，这在您需要将每个应用程序的成本与预算进行比较时特别有用。如果您的预算是通过将业务目标与花费的美元相关联而得出的，那么Cloud Foundry将清楚地向您显示您是否已实现目标。随着时间的推移，我们希望该功能变得更加准确（当前不包括网络I / O费用），并扩展成本与其他运营指标之间的统计相关性，从而为您提供更好的定位和计划工具。

可用区


由于需要在指定的可用区域中启动EC2中的保留实例，因此许多用户要求为部署指定可用区域的能力。预留实例是预付费的，其成本约为按需实例的三分之一。Cloud Foundry现在提供了指定可用性区域的功能，并且在配置部署时，选择输入位于区域下方。


注意：由于Amazon尚未提供检索账单明细的便捷方法，因此成本跟踪未反映使用预留实例的成本节省。

维护页面


如果您需要禁用Java Web应用程序进行维护，则习惯上是显示一个保险杠页面（又称维护页面），以通知用户该不可用。Cloud Foundry现在仅需一个按钮即可提供此功能。首先，当您添加或编辑应用程序时，您会注意到一个新字段： Maintenance Page 。这是您在维护模式下要显示的静态页面（来自应用程序的根上下文）。部署后，您可以通过转到“部署详细信息”页面并单击“开始维护”以显示维护页面，从而在维护和正常操作之间切换；然后单击“结束维护”以恢复为正常运行的应用程序。这是提供完整的应用程序生命周期管理支持的一小步。我们上个月介绍了与开发人员工具的集成，我们将分析和确定在应用程序的开发和部署到云中应支持的各个生命周期阶段/阶段。

克隆部署


使用相同的配置复制环境有很多原因，现在Cloud Foundry通过“部署详细信息”页面上的“克隆”按钮提供了一种克隆部署的简单方法。与维护功能一起升级应用程序时，基于现有部署创建重复部署特别有用。当您的部署利用EBS卷时，将启用“克隆”按钮。系统将提示您为克隆命名，所有配置将被复制并在新部署中启动。启动克隆的部署后，您可以通过单击“应用程序”部分中的“重新部署”按钮来部署应用程序的不同版本或完全不同的应用程序。


升级应用程序的活动部署可能涉及一系列步骤：

  1. 将经过测试的应用程序上传到Cloud Foundry

  2. 在当前部署上启用维护页面

  3. 克隆部署

  4. 运行SQL脚本以迁移数据库架构

  5. 重新部署应用程序的较新版本

  6. 重新配置弹性IP，以便从静态IP地址提供新部署

  7. 停止旧的部署


通过克隆功能，可以利用现有的生产部署配置来平滑且可靠地升级正在运行的应用程序。

附加指标和图表


在此最新版本的Cloud Foundry beta中，我们通过扩展收集的指标数量以及添加用户界面功能来改进了监视功能。首先，添加了Hyperic SIGAR文件系统指标，以监视VM根磁盘和数据库存储上的关键磁盘使用情况信息。现在，这些指标在实例详细信息区域中可见。
接下来，客户端请求的数量已添加为度量。Cloud Foundry会收集此数据并将其转换为基于速率的每分钟指标，从而为您提供应用程序使用情况的度量。该指标在改进的部署详细信息部分中显示为图表，其中列出了合计值（最大值，最小值和平均值）。


这只是提供动态监视显示的开始，该显示提供有用且相关的操作指标和数据。随着时间的推移，我们将添加更多有关性能，运行状况和资源消耗的指标。
