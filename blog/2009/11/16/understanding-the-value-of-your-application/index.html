了解您的应用程序的价值

工程
查理斯
2009年11月16日

我们都喜欢使用最新，最先进的技术来构建应用程序，但是最终，我们将获得交付实现业务目标的应用程序的报酬。我们将每个开发迭代都集中在如何最好地实现这些目标上，但是我们如何知道最新的迭代是否实现了目标呢？最新版本是否影响了应用程序交付的现有业务目标？

SpringSource通过使用Spring仪表，显示趋势的监视工具和报告工具（可将这些数据导出到管理团队）为您提供对应用程序的可见性，从而提供此功能。

仪器化Spring Framework


仪表Spring Framework 是一个商业版本中的Spring Framework ，仪表化管理和监控。只需使用已检测的jar代替Spring Framework jar，即可将应用程序中的许多Spring Bean自动导出到JMX进行管理，并使用AOP自动监视其性能。

所有的org.springframework。随该产品一起分发的.instrumented jars是开放源代码Spring Framework中对应产品的简单编译时编织版本。例如， org.springframework.beans.instrumented jar文件是随开源Spring Framework一起分发的org.springframework.beans jar的直接替代。

通过建议AbstractApplicationContext的refresh方法，插桩的Spring Framework自动发现应用程序。如果所检测的组件知道如何管理和监视通过AbstractApplicationContext创建的任何bean，则它将被发现为托管资源。在ApplicationContext之外创建了一些组件，这些组件也将被自动发现。关闭AbstractApplicationContext时 ，将取消部署其关联的托管资源。

一旦托管资源被自动发现，Spring Framework Instrumentation就会自动创建代表发现的每个应用程序资源的JMX ModelMBean ，并将这些ModelMBean注册到自动检测到的MBeanServer中。这些MBean包含代表被管理资源的度量或属性的预定属性，以及提供对被管理资源的运行时控制的操作。

插装的Spring Framework 使用编译时编织的方面和现有Spring Framework 代码的挂钩的组合来监视其大部分托管资源。插装的Spring Framework 还使用Spring AOP代理来监视应用程序中定型组件（标为@ Controller ， @Transactional ， @ Service ， @ Repository或@Component ）的方法执行。

我们不仅将Instrumented Spring Framework 限于核心Spring Framework ，而且在Spring Web Flow ， Spring Security和Spring Web Services上提供此功能。Spring 3.0还引入了附加注释，以允许高级度量捕获。我们希望将来的博客中会更详细地介绍这一点。

使用Hyperic HQ进行监控


Hyperic HQ管理，监视和控制大型IT环境-涉及数百到数千台计算机。从单个应用程序，应用程序服务器，操作系统，数据库和网络接口转换为成千上万的托管资源。

总部自动发现由Hyperic的系统信息收集器（ SIGAR ）提供支持，该系统是一个跨平台的API，用于收集软件清单数据。总部代理使用SIGAR发现广泛的系统信息，包括CPU速度，RAM大小，操作系统版本，主机名和IP地址。自动发现适用于所有主要平台-Unix，Linux，Windows，Mac OSX，Solaris，HP-UX和AIX。

与所有SpringSource产品一样，Hyperic HQ建立在开源上。由于此博客着重于高级报告，最终将获得经理可以理解的出色报告，因此我将使用Hyperic Enterprise Edition。您可以在以下位置获取有关Hyperic Enterprise Edition和Hyperic HQ open source的区别的完整详细信息。 http://www.hyperic.com 。

如果您使用的是Instrumented Spring Framework，它将自动在您的应用程序中发现内部指标。这包括有关Spring Core，Spring DAO，Spring ORM，Spring JEE集成，Spring MVC的内部指标，以及您已通过插桩的Spring Framework 注释的应用程序指标。
