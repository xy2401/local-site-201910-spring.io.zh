SpringSource DM服务器路线图

工程
罗伯·哈罗普
2009年4月1日

dm Server用户对接下来的几个发行版中会有什么提出了很多疑问。在此博客文章中，我将概述路线图的主要功能。我们遵循Scrum惯例，因此您可以期望从sprint的输出中看到相当频繁的里程碑，并且我们可以灵活地处理新要求和优先级的变化。

共享存储库

共享存储库使您可以在一个中央位置来管理可在dm Server实例中安装的工件。然后可以在其存储库链中的任何位置将这些共享的存储库添加到dm Server配置中。

共享存储库只是一个dm Server节点，它管理文件系统存储库并使它可供其他节点使用。这些存储库托管节点称为dm存储库节点 。共享存储库的数量及其顺序不固定-您可以自由选择对您的环境有意义的任何存储库结构。

客户端dm服务器节点使用REST API与存储库节点进行通信。该REST API将被记录在案并可用，以便第三方工具可以与共享存储库基础结构集成。我们想象一些场景，通过这些场景可以创建工具来管理存储库，而其他场景则可以在第三方存储库产品之上实现REST API。

我们将升级企业捆绑软件存储库的软件，使其作为完整的dm存储库运行，并具有REST API访问权限。

存储库和资源调配增强

dm Server设置系统已升级，以处理通用的工件抽象，而不是专门处理捆绑软件。这样做的主要原因是dm Server 2.0将支持许多不同的部署单元，而不仅仅是捆绑包和WAR文件。

存储库基础结构已得到扩展，可以通过单个存储库管理不同类型的工件。然后，您的dm Server节点可以查询存储库以获取其所需的所有工件。

计划文件

dm Server中对PAR文件的支持使您可以将捆绑软件作为单个逻辑单元组合在一起。在PAR内，将捆绑包和服务放置在一个范围内 ，该范围将它们与系统的其余部分隔离开。这种范围确定可确保捆绑包相互连接，并优先于范围之外的服务查看彼此的服务。范围界定还可以防止应用程序代码泄漏到全局范围或其他应用程序的范围内。

计划文件（或简称为计划）概括了PAR，以支持：

  • 捆绑包以外的文物
  • 可选范围
  • 工件生命周期的控制

计划还消除了PAR所需的额外打包步骤的需要-计划仅引用存储库中的其他工件：

<plan name="greenpages" version="1.0.0" scoped="true" atomic="true">
  <artifact type="bundle" name="greenpages.db" version="${plan.version}"/>
  <artifact type="bundle" name="greenpages.app" version="${plan.version}"/>
  <artifact type="bundle" name="greenpages.jpa" version="${plan.version}"/>
  <artifact type="bundle" name="greenpages.web" version="${plan.version}"/>
</plan>

在此示例中，您可以看到一个名为greenpages的计划，其版本为1.0.0 ，引用了4个分发包。

计划的scoped属性控制是否应将计划中的工件安装到计划特定的范围中。禁用作用域时，工件将部署到全局范围中，并且可供所有其他工件访问。

原子属性控制是否应将计划中工件的生命周期联系在一起。使计划成为原子意味着将计划中单个工件的安装，启动，停止和卸载事件升级为计划中的所有工件。同样，在原子计划中，dm Server可以防止工件处于不一致状态。例如，如果一个工件无法启动，则dm Server将停止计划中的所有工件。

计划内工件声明的顺序很重要。为了支持排序约束，dm Server将按照计划中定义的顺序将生命周期事件广播到工件。这使您可以创建在启动过程中具有排序依赖性的捆绑软件，而不必担心如何控制部署时间排序。

计划概括了当今dm Server中可用的许多构造。PAR等效于有范围的原子计划，而库等效于无范围的非原子计划。子系统的dm Server内部构造只是简单的原子计划。

配置设置

dm Server 2.0将支持的最令人兴奋的新部署工件之一是配置文件。配置文件只是一个属性文件，可以使用OSGi Config Admin服务在运行时使用。

配置文件具有完整的生命周期支持，可以在运行时进行更新和刷新。您的应用程序可以订阅对其配置文件进行刷新的通知，并进行相应的调整。

与计划结合使用时，配置文件为管理应用程序的外部配置数据提供了一种出色的机制。考虑以下JDBC配置文件：


jdbc.url=jdbc:h2:tcp://localhost/~/greenpages
jdbc.user=greenpages
jdbc.password=pass
jdbc.driverClassName=org.h2.Driver

可以在版本1.0下将该配置文件发布为名称为greenpages.jdbc.dev的资源库。在我们的计划中，我们可以参考以下配置文件：

<plan name="greenpages.dev" version="1.0.0" scoped="true" atomic="true">
  <artifact type="properties" name="greenpages.jdbc.dev" version="${plan.version}"/>
  <artifact type="bundle" name="greenpages.db" version="${plan.version}"/>
  <artifact type="bundle" name="greenpages.app" version="${plan.version}"/>
  <artifact type="bundle" name="greenpages.jpa" version="${plan.version}"/>
  <artifact type="bundle" name="greenpages.web" version="${plan.version}"/>
</plan>

在这里，一个名为greenpages.dev的计划引用了greenpages.jdbc.dev配置文件。现在，我们可以构造一个引用了另一个配置文件的计划greenpages.prod 。

为了减少这种情况下所需的重复量，计划本身就是存储库中的工件，因此可以从其他计划中引用

<plan name="greenpages.bundles" version="1.0.0" abstract="true">
  <artifact type="bundle" name="greenpages.db" version="${plan.version}"/>
  <artifact type="bundle" name="greenpages.app" version="${plan.version}"/>
  <artifact type="bundle" name="greenpages.jpa" version="${plan.version}"/>
  <artifact type="bundle" name="greenpages.web" version="${plan.version}"/>
</plan>

<plan name="greenpages.dev" version="1.0.0" scoped="true" atomic="true">
  <artifact type="properties" name="greenpages.jdbc.dev" version="${plan.version}"/>
  <artifact type="plan" name="greenpages.bundles" version="${plan.version}"/>
</plan>

<plan name="greenpages.prod" version="1.0.0" scoped="true" atomic="true">
  <artifact type="properties" name="greenpages.jdbc.prod" version="${plan.version}"/>
  <artifact type="plan" name="greenpages.bundles" version="${plan.version}"/>
</plan>

这里被称为greenpages.bundles计划是由greenpages.prod和greenpages.dev计划引用。宣布greenpages.bundles计划为抽象计划，阻止其独自部署到dm Server中。

构建增强

我们最近宣布了Bundlor工具 ，该工具可简化在Maven，Ant和Eclipse中创建OSGi清单的过程。我们将与dm Server 2.0一起发布Bundlor的许多增强功能，包括：

  • 从Maven和Ivy文件自动检测版本
  • 支持bnd样式的版本范围表达式
  • 批量处理

自动版本检测使Bundlor可以使用Maven或Ivy文件中的信息来确定软件包或软件包依赖项的版本。当前，此信息包含在TEMPLATE.MF文件或外部属性文件中。

使用版本范围表达式，可以将输入版本扩展到版本范围。当映射到软件包依赖项时，这允许将Maven或Ivy文件中定义的版本扩展到范围。

批量捆绑处理将允许将大量JAR自动转换为捆绑。JAR之间的关系用于推断版本信息，因此您可以减少在模板中编写的代码。

模块化Web应用程序

我们计划未来最激动人心的功能之一就是对模块化Web应用程序的支持。当前，在dm Server 1.0中，可以将应用程序的大多数层拆分为多个包，但是Web层往往会带来问题。静态资源解析，会话共享以及需要由多个贡献者组成UI的问题使此功能成为普通OSGi的问题。

使用dm Server中的模块化Web应用程序支持，您将能够将Web内容拆分为多个捆绑包，称为Web片段 。应用程序中的所有Web片段共享相同的ServletContext会话状态。每个Web片段都被分配来处理应用程序URL空间的子部分，这与WAR应用程序完成Web上下文映射非常相似。例如，考虑下图：

此图表示单个Web应用程序Spring Travel，该应用程序映射到/ springtravel上下文路径。Spring Travel应用程序由三个Web片段组成，每个Web片段都映射到/ springtravel的子路径。例如， flights Web片段服务请求发送到/ springtravel / flights / * 。

Web片段可以动态添加到应用程序中，并且应用程序可以在安装和卸载这些片段时进行调整。

模块化Web应用程序最有趣的功能是支持UI组合。Web主捆绑包可以定义Web应用程序的基本模板，布局，样式和图像。然后，Web片段可以将部分内容贡献给现有页面或整个新页面，以适合母版提供的模板之一。

我们仍在准确定义此UI模型的外观，但是我希望最初我们能够将JSF作为定义模板和贡献的一种方式。

分布式管理

AMS团队将扩展对dm Server的支持，以允许在一组dm Server节点上执行以下操作：

  • 工件安装/卸载
  • 神器开始/停止
  • 神器刷新
  • 存储库发布/删除

克隆

克隆是我们引入的一项功能，旨在解决高级用户面临的许多棘手问题。

克隆解决的第一个问题是共享具有静态状态的分发包的问题。静态与定义静态字段的类耦合。在OSGi中，使用具有静态状态的类的捆绑软件不会获得该类的自己版本。作为此问题的示例，请考虑在应用程序中使用Log4J。

Log4J将其配置状态存储在静态变量中。当您将应用程序部署为WAR文件时，这不太可能出现问题。Log4J将部署在WEB-INF / lib中，并且您的应用程序将拥有自己的Log4J类的私有副本。但是，在OSGi中，您可能有许多捆绑包共享同一Log4J捆绑包，因此共享相同的Log4J类。

克隆旨在解决的第二个问题是固定 。固定可防止您安装依赖于同一第三方捆绑软件但版本不同的两个捆绑软件。当您要安装的两个捆绑包的依赖关系图部分重叠，并且不重叠的部分打破了使用约束时，就会发生固定。考虑下图中的情况：

您可以看到Bundle B和EclipseLink 1.0.1之间的依赖关系是可以满足的，但是Bundle B无法连接到Spring 2.5.6 。原因是Spring对所有org.eclipse.persistence包都有使用约束。该约束阻止Bundle B连接到Spring 2.5.6 和 EclipseLink捆绑包，除非它连接到与Spring 2.5.6相同的EclipseLink捆绑包。有关使用的更多信息，请参见Glyn的简介博客和我的关于使用失败的诊断博客。

为了解决这些问题，克隆提供了创建捆绑包的托管克隆并将此克隆绑定到应用程序范围的功能。使用Import-Bundle引用捆绑包时，可以手动调用克隆。您可以使用它为您的应用程序捆绑包创建Log4J的克隆：

Bundle-ManifestVersion: 2.0
Bundle-SymbolicName: app.bundle
Bundle-Version: 1.0.0
Import-Bundle: com.springsource.org.apache.log4j;bundle-version="1.3.4";sharing:=clone

改进的日志记录和跟踪支持

关于dm Server 1.0中的日志记录和跟踪支持，我们有很多反馈，而在2.0中，我们将解决用户提出的最紧迫的问题。

对于dm Server 2.0，我们将对所有跟踪进行全面检查，使其在SLF4J之上运行，并在其下面提供LogBack实现。另外，我们的日志记录框架将被重新构造为SLF4J之上的薄层。

我们将提供一系列LogBack（可能还有Log4J）附加程序，这些附加程序会将用户和应用程序特定的跟踪路由到dm Server跟踪文件中。这里最令人感兴趣的可能是有选择地选择要编写特定于应用程序的跟踪的束的功能。同样重要的是要注意，使用特定于应用程序的跟踪时，将根据dm Server中的当前工作单元动态选择应用程序目标。这意味着相同的共享库可以根据其用法将其跟踪写入不同的应用程序跟踪文件。

我很兴奋的一项功能是支持从存储库部署LogBack配置。将配置安装到dm Server后，捆绑软件可以选择绑定到该配置作为其日志记录配置源。这提供了一种简单的机制，可以并排支持多种不同的日志配置，并可以在多个捆绑软件中共享日志配置。

企业OSGi

今年应该会看到专门针对企业使用的OSGi规范的第一版。我们积极参与此过程，我们希望dm Server 2.0成为运行Enterprise OSGi应用程序的最佳场所。我们将支持：

  • RFC66-OSGi的Web容器
  • RFC119-分布式OSGi
  • RFC124-蓝图服务（从Spring Dynamic Modules中产生）
  • RFC139-OSGi的JMX接口
  • RFC142-JNDI和OSGi集成

RFC124的参考实现由SpringSource根据Spring动态模块中完成的工作创建。我们还将使用dm Server 1.0中已经存在的Web应用程序支持构建RFC66的参考实现。

管理外壳

管理外壳提供对dm Server管理功能的安全命令行访问。使用管理外壳，您可以：

  • 管理已配置的存储库
  • 安装/卸载工件
  • 启动/停止工件
  • 更新/刷新工件
  • 查询工件依赖图
  • 查询运行服务
  • 诊断依赖性解决问题

管理外壳是完全可插入的，允许轻松添加自定义命令。

Tomcat配置

由于需求旺盛，我们将从dm服务器特定的配置Tomcat的机制切换回使用Tomcat自己的配置文件格式。
