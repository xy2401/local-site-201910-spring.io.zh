Bundlor版本扩展和属性替换

工程
克里斯托弗·弗罗斯特
2009年5月12日

介绍

最近发布了Bundlor的M3里程碑（ 论坛公告 ）。此里程碑增加了对属性替换和版本扩展的支持。这篇博客文章解释了如何使用这些新功能来提高生成的清单的质量。

财产替代

现在可以使用Bundlor将任何属性值替换为清单模板。


Bundle-Name: ${name}
Bundle-Description: Test bundle using new version of Kernel at ${com.springsource.kernel}
Import-Template: com.springsource.kernel.*;version=“${com.springsource.kernel}”

该语法允许您为以下属性指定属性占位符${name}和${com.springsource.kernel}并在运行时用实际值替换它们。这些值的传递方式特定于使用哪个Bundlor前端。

在命令行中指定属性

从命令行运行Bundlor时，它将使用所有可用的属性作为系统属性，其中不包括任何环境变量。命令行脚本将通过传递的任何变量-D因此以下内容将提供${com.springsource.kernel}属性值为'2.0.0。发行”和${name}属性的值为“内核测试包”。

bundlor.sh \
        transform \
        --bundle ./org.springframework.integration.jar \
        --manifest ./template.mf \
        --outputfile ./target/org.springframework.integration.jar \
        -Dcom.springsource.kernel=2.0.0.RELEASE \
        -Dname="Kernel test bundle"

使用Maven指定属性

要将属性从Maven构建传递到Bundlor，您应该在Maven项目文件的project元素下定义一个新的（或添加到现有的）properties元素。


    ${project.name}
    2.0.0.RELEASE

这将使用替换值运行Bundlor ${com.springsource.kernel}的'2.0.0。RELEASE”和替换值${name} Maven项目名称的名称。

用Ant指定属性

从Ant调用Bundlor任务时，可以在传递给Ant属性集的任务上提供额外的属性，该属性将传递给Bundlor。


    bundlePath="${basedir}/org.springframework.integration.jar"
    outputPath="${basedir}/target/org.springframework.integration.jar"
    bundleVersion="2.0.0.BUILD-${timestamp}"
    manifestTemplatePath="${basedir}/template.mf">

这将运行Bundlor并提供替换值${com.springsource.kernel}的'2.0.0。RELEASE”和替换值${name} Ant项目的名称。

版本扩充

属性替换可以在清单模板的值中的任何位置进行。它甚至可以用来替代“导入模板”版本。就其本身而言，导入中的单个版本不是很有用，因为它具有限制性，但是自动将版本扩展到版本范围很有用。可以在冒号后给出版本扩展模式，该冒号将应用于版本以生成版本范围。

Import-Template: com.springsource.kernel.*;version="${com.springsource.kernel:[=.=.=.=, +1.0.0)}",
 org.apache.commons.logging.*;version="${org.apache.commons.logging:[=.=.=.=, =.=.+1)}"

版本扩展适用于具有合法OSGi版本号的变量。根据该版本号，扩展分为四个部分，主要，次要，微型和限定符。如果提供的属性不是有效的OSGi版本，则Bundlor将失败，并出现异常，指出提供的属性不是有效的版本字符串。

此版本号的前三段可以按如下所示在扩展中替换：
=等于变量中的值
[+/-] n通过此量调整变量的值
ñ将该值替换为变量中的一个。通常只用于输入0
第四段（限定符）只能采用以下替换：
=等于变量中的值
任何合法的限定符值。将此值替换为变量中的一个


因此，对于上面的示例，生成的版本范围${com.springsource.kernel}而给定的1.2.0版本为[1.2.0，2.0.0）。这意味着将使用从1.2.0到（但不包括）2.0.0的最高版本。对于${org.apache.commons.logging}而给定的1.4.0版本将生成的版本范围为[1.4.0，1.4.1），这要严格得多，因为它将只允许更改版本1.4.0的限定符，而不会允许使用1.4.1.xxx及更高版本。

命名

如果某个模式将用于多个导入，则可以使用一个名称来指定它并使用多次。模板清单中使用新的标头来指定命名的版本扩展模式。

Import-Template: 
 org.apache.commons.codec.*;version="${org.apache.commons.codec:apache}",
 org.apache.commons.logging.*;version="${org.apache.commons.logging:apache}",
 org.hibernate.*;version="${org.hibernate:hibernate}"
 org.myorg.*;version="${org.myorg:(=.=.=.=, =.+1.0.=]}"

Version-Patterns: 
 apache;pattern="[=.=.=.=, +1.0.0)",
 hibernate;pattern="[=.=.=.=, =.=.+1)"

这显示了定义了多个模式并将其用于多个导入。apache模式将提供一个版本范围，从提供的版本一直到但不包括下一个主要版本。休眠模式将提供所提供的版本范围，但不包括下一个微型版本。还有一个使用内联版本扩展模式定义的导入，可以与在同一模板中很好用的命名模式一起使用。嵌入式模式显示了更不寻常的用法。它将创建一个版本范围，从（但不包括）所提供的版本，直至并包括具有相同限定符的下一个次要版本。

结论

一起使用这些技术，可以生成可靠的清单。正确使用Version Expansion将确保生成的清单具有足够的限制性，以阻止不必要的接线，但又具有足够的灵活性，以允许进行较小的更改。在编写模板清单时，应为每个相关包采取务实的方法，因为并非所有包都使用相同的方法进行版本控制。
