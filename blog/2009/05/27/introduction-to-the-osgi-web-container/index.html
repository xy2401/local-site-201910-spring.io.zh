OSGi Web容器简介

工程
罗伯·哈罗普
2009年5月27日

更新：添加了针对Git的版本控制说明。

在过去的几个月中，我一直与Subbarao Meduri，Graham Charters，Hal Hildebrand和OSGi企业专家组的其他人一起研究RFC66 Web容器规范。Web容器规范定义了如何以标准方式在OSGi服务平台上部署WAR文件。

这对我们来说非常有趣，因为dm Server已经支持WAR文件近18个月了，我们很高兴能够开发标准模型。作为最终用户，您将能够在OSGi上部署WAR文件，而无需依赖于专有API或功能。

SpringSource负责编写Web容器参考实现，并且在过去的几周中，我一直在努力。在此博客中，我将描述Web容器规范的主要功能，并讨论开始使用RI代码所需的步骤。不允许发布规范文档，但我将概述最重要的几点。

目前，RI没有二进制发行版，但是从源代码构建开始确实很容易。

Web容器的主要功能

Web容器支持的最有趣的功能是：

  • 安装WAR文件
  • Web应用程序捆绑包（WAB）的安装
  • 使用扩展程序的Web应用程序生命周期控制
  • 使用URL参数控制配置属性

安装WAR文件

对我来说，Web容器最令人兴奋的功能是无需更改代码即可直接将WAR文件部署到OSGi的功能。（使用JNDI的WAR文件必须在未更改代码的情况下使用，因为JNDI当前不属于规范的一部分。我们正在努力解决此问题，因此我不希望这种限制永远存在。）

要将WAR文件安装到OSGi中，只需在调用BundleContext.installBundle或使用平台的控制台时将webbundle：前缀到文件位置即可。例如，在Equinox中，我可以这样做：


install webbundle:file:formtags.war

当然，这需要在OSGi平台上安装Web容器实现。

Web容器在WAR上执行各种转换，以确保该容器具有符号名称，版本和必要的导入才能使其工作。Web容器还将更新包的类路径，以添加WEB-INF / classes和WEB-INF / lib中的每个JAR文件。

安装Web应用程序捆绑包（WAB）

如果您不想依靠Web容器来转换WAR文件，则可以完全跳过转换阶段。简单地，您自己添加适当的清单标题并直接安装捆绑包，而无需使用webbundle协议。

WAB不能在捆绑类路径上具有WEB-INF下的任何条目。这是为了防止您的任何应用程序类在Web应用程序中显示为资源。WAB可能还会出现其他限制，但这些限制尚未最终确定。

使用扩展器进行生命周期控制

Web容器使用扩展器模型来控制Web应用程序的生命周期。Web应用程序在启动相应的捆绑软件时启动，并在该捆绑软件停止时停止。

在Equinox中，这意味着我可以直接从控制台启动和停止Web应用程序。例如，如果上面的formtags.war捆绑包的捆绑包ID为50：


start 50
… Tomcat output …
stop 50

使用URL参数控制配置

可以通过将某些受支持的配置属性附加到安装URL来控制WAR文件的配置。

例如，要控制Web应用程序的上下文路径，可以添加Web-ContextPath选项：


install webbundle:file:formtags.war?Web-ContextPath=ftags

或控制生成的Bundle-SymbolicName标头：


install webbundle:file:formtags.war?Bundle-SymbolicName=ftags.bundle

Web容器和dm服务器

在Web容器RI的代码大多取自并通过DM服务器的代码启发，但是，RI 并不要求完全DM服务器。我们将使用Web Container RI替换dm Server中的当前Web代码，并且还将采用Web Container作为在OSGi上构建Web应用程序的推荐方法。

参考实现简介

Web容器RI使用Tomcat作为servlet容器实现。RI本身由四个束组成。核心捆绑包包含与servlet容器无关的所有代码。Tomcat捆绑软件包含Tomcat特定的代码。tomcat.fragment捆绑包包含默认的Tomcat配置，并作为片段附加到Tomcat Catalina捆绑包。扩展程序包包含所有扩展程序行为，如果您希望手动管理Web应用程序生命周期，可以将其删除。

构建参考实现

要开始使用RI，您需要从源代码构建它。可以直接从Git获取源：


git clone git://git.springsource.org/osgi-web-container/osgi-web-container.git

签出代码后，您可以使用以下命令对其进行构建和测试：


cd build-web-container
ant clean test collect

如果您的测试失败，请提出JIRA 。成功构建后，您可以运行Web容器并安装一些WAR文件。

运行参考实现

我正在使用PAX Runner运行Web容器RI。我的PAX Runner配置使用内置的二进制文件和常春藤管理的依赖项来运行Web容器。

在build-web-container目录中，您将找到一个名为Runner.bundles的文件。此文件可用于指示PAX Runner安装Web容器所需的所有捆绑软件：


pax-run –platform=equinox –snapshot runner.bundles

此命令使用PAX Runner启动Equinox。–snapshot标志告诉PAX Runner下载Equinox的最新流稳定版本-Equinox实现了最新的已知OSGi 4.2规范。

Equinox启动后，发出ss命令来验证捆绑软件是否已安装，您应该会看到大约有45个捆绑软件已安装并正在运行。

安装应用程序

我正在使用dm Server的FormTags WAR示例进行测试。您可以从这里下载。

如果我尝试直接安装FormTags应用程序，那么事情将无法按预期进行：


install webbundle:file:formtags.war
Bundle id is 48
start 48

运行启动后，出现错误，提示org.xml.sax包中某个类的ClassNotFoundException 。这里的问题是，默认情况下，仅对WAR文件进行4次导入： javax.servlet ， javax.servlet.http ， javax.servlet.jsp和javax.servlet.tagext 。

使用Import-Package URL参数解决此问题相对容易：


uninstall 48
install webbundle:file:/Users/robharrop/tmp/formtags.war?Import-Package=org.xml.sax,org.xml.sax.helpers,javax.xml.parsers,org.w3c.dom
Bundle id is 49
start 49

在这里，我指定了FormTags应用程序所需的额外包导入。这次，此应用程序成功启动，可以从Web浏览器中访问 http：// localhost：8080 / formtags 。

在dm Server中，WAR文件为系统捆绑包导出的每个软件包获取一个导入，这意味着可以自动使用常见的系统类型。我认为这是一个有用的功能，我很想知道您是否同意或是否希望手动控制对系统软件包的访问。
