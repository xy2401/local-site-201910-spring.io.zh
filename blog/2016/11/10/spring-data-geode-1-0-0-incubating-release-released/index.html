Spring Data Geode 1.0.0。发行发行

工程
约翰·布鲁姆
2016年11月10日

我非常荣幸地代表Spring和Apache Geode社区宣布发布适用于Apache Geode 1.0.0-incubating的Spring Data 。

您可以通过在应用程序Maven POM或Gradle构建文件中包括以下依赖项来从Maven Central中获取信息。

马文


  org.springframework.data
  spring-data-geode
  1.0.0.INCUBATING-RELEASE

摇篮

compile 'org.springframework.data:spring-data-geode:1.0.0.INCUBATING-RELEASE'

包括spring-data-geode依赖关系将传递所有必需的Apache Geode工件，因此您可以立即开始构建使用Apache Geode的 Spring应用程序。

注意：

我通过删除再次更改了版本限定符APACHE-GEODE资格和简化为INCUBATING-RELEASE 。Apache Geode毕业后， INCUBATING资格也会下降，版本号将变成major.minor.maint.[M#|RC#|RELEASE] 。

什么是新的

两者均为Spring Data Geode 1.0.0。出于某些原因， INCUBATING-RELEASE和Apache Geode 1.0.0-incubating发行版很重要。

首先，这标志着Apache软件基金会 （ASF）内部的Apache Geode的第一个正式GA版本 。这是向前迈出的一大步，不仅标志着Geode的成熟，它起源于Pivotal GemFire十多年的生产经验，而且还加速了它作为ASF内部顶级项目（TLP）的毕业。

但是，还不止这些！

安全！

此版本还通过引入新的 集成安全框架 （ 此处包含一些技术细节），对Apache Geode的安全模型进行了重大更改，该框架不仅包括安全传输（即SSL），而且还包括身份验证和授权。

这非常重要，因为Apache Geode是为数不多的无需企业许可即可提供安全性的OSS IMDG选项之一！

有关此新功能的最好的事情之一是，它是一个允许插入不同安全提供程序的框架 。开箱即用的Geode是建立在Apache Shiro之上的， Apache Shiro提供了一种熟悉且健壮的方式来不仅为Geode，而且为您的应用程序配置安全性。

如何保护Apache Geode

如果没有Spring（Data Geode） ， Apache Geode提供了自己的配置安全性选项。

一种选择是实现Apache Geode SecurityManager接口并设置相应的Geode security-manager （系统）属性为全限定的类名。一个例子可以在这里看到。

但是，使用属性引用FQCN会严重限制您配置FQCN的方式。 SecurityManager在托管环境或测试环境中。根据我的反馈，将在以后的Geode版本中解决此问题 。

另一种选择是使用Apache Geode的 security-shiro-init （System）属性，以指定位于Apache Shiro支持的指定资源路径中的INI配置文件 。但是，这有两个方面的限制。

首先， Apache Geode仅支持classpath:目前的资源说明符（Geode工程团队也正在处理 ）。其次，无论多么标准 ，都必须学习另一种配置文件格式，这比XML，IMO更好。

当然， Apache Shiro通过提供this来尝试减轻在Spring上下文中运行时的痛苦。但是，仍然有太多的样板配置逻辑尚待改进。

如何使用Spring保护Apache Geode（Data Geode）

本着使Apache Geode 快速且易于使用的精神（请参阅我的上一篇博文 ），我一直与Geode工程团队密切合作，以改进初始设计，并使Integrated Security真正成为IBM的一流公民。通过使用许多由Spring Framework和Spring Boot流行的基本API和框架设计概念， Spring Data Geode 。

所以，我给你新的 @EnableSecurity SDG的新基于注释的配置模型中的注释。您可以通过多种方式使用注释来配置Apache Geode的安全功能。

SecurityManager类名称参考

您仍然可以引用Geode SecurityManager使用其完全限定的类名来实现

package example;

class ExampleSecurityManager 
    implements org.apache.geode.security.SecurityManager {
  ...
}

@CacheServerApplication(name = "ClassNameExample")
@EnableSecurity(securityManagerClassName = "example.ExampleSecurityManager")
class ExampleApplication {
  ...
}

可以在SDG Contacts Application RI中找到更详细的示例。

但是，您必须提供一个默认的无参数构造函数，并且您的Geode SecurityManager实施将负责在构建时加载所有安全认证/授权详细信息；不太理想。

SecurityManager代理实现

另一种选择是创建一个Proxy实施Geode SecurityManager接口，该接口委托给实际的基础Geode SecurityManager在Spring容器或其他托管环境（例如Pivotal CloudFoundry）中配置并由其注入。

一个这样的Proxy实现可以被视为在RI 这里 ，并且被配置如下...

@CacheServerApplication(name = "ProxyExample")
@EnableSecurity(securityManagerClassName = 
  "example.app.geode.security.SecurityManagerProxy", 
  useBeanFactoryLocator = true)
class ExampleApplication {

    ...

    @Bean
    JdbcSecurityRepository securityRepository(JdbcTemplate template) {
      return new JdbcSecurityRepository(template);
    }

    @Bean
    SimpleSecurityManager securityManager(
        SecurityRepository securityRepository) {

      return new SimpleSecurityManager(securityRepository);
    }
}

的SecurityMangerProxy由Apache Geode在缓存初始化期间构建。Spring容器将找到SimpleSecurityManager bean定义并将其注入到SecurityManagerProxy 。

的SecurityManagerProxy通过利用Spring的另一个功能BeanFactoryLocator来工作，如参考指南 （及此处 ）所述，SDG使用BeanFactoryLocator来配置和自动装配在Spring容器外部构造和初始化的对象，例如Apache Geode 。

这在可能已在Geode的本机中定义了应用程序对象（例如CacheLoader ）的情况下很有用cache.xml配置，并且需要与bean自动连接（例如DataSource ）定义在Spring容器中。这也适用于在Geode（系统）属性中引用的对象，例如SecurityManager 。

的SecurityManagerProxy必须扩展 LazyWiringDeclarableSupport类，从而启用Proxy由Spring容器使用BeanFactoryLocator一旦Geode构造了对象。实际上非常光滑。

你可以看到完整的示例配置在RI 这里 。这也需要useBeanFactoryLocator可以在Geode Server的Spring Boot应用程序类上将属性设置为true ，在上面的示例中也是如此。

Apache Shiro INI配置文件

也许您不想将您的应用程序代码不必要地耦合到Geode的专有类和接口，例如SecurityManager 。也许您只想充分利用Apache Shiro的安全框架。

一种方法是创建一个Apache Shiro INI配置文件 ，并在@EnableSecurity像这样的注解...

@CacheServerApplication(name = "ProxyExample")
@EnableSecurity(shiroIniResourcePath = "my-shiro.ini")
class ExampleApplication {
    ...
}

同样， Apache Shiro INI文件必须位于类路径上。由于目前的Apache Geode 限制 ，因此无法使用其他资源说明符（例如file:要么url: ）。

这个完整的示例配置可以在这里看到。

Apache Shiro领域

但是，作为应用程序开发人员，您真正想做的就是定义Apache Shiro Realms作为春豆Spring容器来访问你的应用程序，以确保阿帕奇的Geode和春天有所需的安全性元数据做的所有工作。

好吧，SDG也可以为您做到这一点。例如…

@CacheServerApplication(name = "RealmExample")
@EnableSecurity
class ExampleApplication {

    @Bean
    PropertiesRealm shiroRealm() {
      PropertiesRealm propertiesRealm = new PropertiesRealm();
      propertiesRealm.setResourcePath("classpath:shiro.properties");
      propertiesRealm.setPermissionResolver(new GeodePermissionResolver());
      return propertiesRealm;
    }
  }

而已;这就是您所需要的。

注意Shiro PropertiesRealm使用GeodePermissionResolver解决Geode权限。另外，您可以选择指定您选择的任何资源路径。您不限于classpath:只要。

您还可以自由定义任何内容Realms Shiro提供的（例如JDBC，JNDI，LDAP等）应用程序用来访问其安全元数据。

如果您定义多个Shiro Realm ，您甚至可以使用Spring的 @Order上的注释Realm bean定义，像这样……

@CacheServerApplication(name = "OrderedMultiRealmExample")
@EnableSecurity
class ExampleApplication {

    @Bean
    @Order(1)
    IniRealm iniRealm() {
      IniRealm iniRealm = new IniRealm("classpath:partial-shiro.ini");
      iniRealm.setPermissionResolver(new GeodePermissionResolver());
      return iniRealm;
    }

    @Bean
    @Order(2)
    PropertiesRealm propertiesRealm() {
      PropertiesRealm propertiesRealm = new PropertiesRealm();
      propertiesRealm.setResourcePath("classpath:partial-shiro.properties");
      propertiesRealm.setPermissionResolver(new GeodePermissionResolver());
      return propertiesRealm;
    }
}

Realm排序是Apache Shiro 身份验证序列中使用的身份验证策略的重要因素。

您可以看到使用Shiro的多个示例配置Realms在RI 这里 。
