Default publish-subscribe semantics

The most important change in Spring Cloud Stream 1.0.0.M4 is the way in which applications interact with each other. We have opted for a default publish-subscribe model, in which each application that receives messages from a given destination will receive a copy of the message. This is a better fit for the processing model of stream applications, where intermediate topics act as data hubs for the various intersecting data streams, also making scenarios such as tapping easier and more efficient. This feature goes hand in hand with the next, which is consumer groups.

Spring Cloud Stream 1.0.0。M4现在可用

发布
马里乌斯·波哥维奇（Marius Bogoevici）
2016年2月6日

我很高兴代表团队宣布Spring Cloud Stream 1.0.0的发布。M4 。新版本进行了一些重大更改和增强，它定义了核心抽象和原语，我们认为这些抽象和原语对于分布式实时数据处理应用程序的开发至关重要。列举其中一些：

默认的发布-订阅语义

Spring Cloud Stream 1.0.0中最重要的更改。M4是应用程序相互交互的方式。我们选择了默认的发布-订阅模型，在该模型中，每个从给定目标接收消息的应用程序都将收到消息的副本。这更适合流应用程序的处理模型，在该模型中，中间主题充当各种相交数据流的数据集线器，还使轻拍等场景变得更加高效。此功能与下一个消费者群体紧密相关 。

用于负载平衡和分区的使用者组

当然，如果给定应用程序有多个实例，我们希望它们充当竞争者或分区使用者-消息应发送到一个且仅一个并行运行的实例。现在可以在Spring Cloud Stream 1.0.0中完成此操作。M4通过使用消费群体 。输入绑定可以在运行时指定使用者组，例如spring.cloud.stream.bindings.input.group=myGroup 。如果有多个应用程序这样做，它们将成为同一组的一部分，并且消息将通过负载平衡或分区策略（由不同的配置集控制）在它们之间分发。这个概念受到Kafka的启发，但是不同的活页夹实现有不同的方法来提供此功能，这是框架中的一流概念。

绑定器SPI的简化

虽然不对普通用户开放，但Binder SPI是Spring Cloud Stream的重要组成部分，可确保框架保持灵活性和可扩展性。在此版本中，我们已经采取了简化 SPI的基本步骤，除已提供的Kafka，Rabbit和Redis外，如有必要，使开发人员可以更轻松地创建自己的实现。（为公平起见，我们计划在RC发布之前对Binder SPI进行更多更改）。

Kafka装订器胶印重置和启动胶印控制

Kafka活页夹现在支持在开始时通过spring.cloud.stream.binder.kafka.resetOffsets属性，允许应用程序通过spring.cloud.stream.binder.kafka.startOffset可以设置为earliest要么latest偏移量。后者可用于控制新启动的应用程序的起始偏移量。

健康指标支持

现在，Spring Cloud Stream通过暴露粘合剂的运行状况指示器来反映中间件连接的状态，从而充分利用了Spring Boot的应用程序运行状况管理支持。Rabbit和Redis当前支持此功能，如果活页夹使用的连接不同于Boot提供的默认连接，则此功能特别重要。

完整的增强和修复列表可以在GitHub中查看。
