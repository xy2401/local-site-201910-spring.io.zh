Spring Data Couchbase 2.0

工程
乔什·朗（Josh Long）
2016年2月10日

这是Couchbase的 SimonBASLÉ的跨帖子博客。您可以在twitter（ @simonbasle ）或github上找到他。在开发人员门户上了解有关Couchbase和Couchbase Java SDK的更多信息。再次感谢Simon，非常感谢您和您的团队！ -乔什

Spring Data Couchbase 2.0是对原始Spring Data Couchbase 1.4.x连接器的重写。它基于Couchbase Java 2.2 SDK，并大量使用了新的查询语言N1QL（在Couchbase Server 4.0中引入），为Spring Data用户提供了更多功能。

第一个里程碑已于去年8月发布，然后发布了候选版本，此后实施了附加功能（和错误修正），现在可以向公众发布GA版本。

让我们快速浏览一下发生了什么变化（用⭐到⭐⭐⭐符号表示我们认为每个功能的出色程度和重要性😄）：

Spring Data Couchbase 2.0的新功能

Spring Data Couchbase的1.x版本与其2.x版本之间的主要区别是：

  • 配置元素更接近Couchbase现实：环境，群集，存储桶（可能允许您创建CouchbaseTemplate ，每个都连接到不同的存储桶，甚至连接到不同的群集！）
  • 支持自定义存储库方法不再总是使用视图来完成，（默认情况下）现在通过N1QL完成，这更加灵活并且需要较少的服务器端维护。
  • 使用视图的自定义方法已进行了一些修改，以更好地遵循Spring Data哲学。这稍微降低了灵活性，但是实现是从方法名称生成的（通过“查询派生”）。
  • 现在，您可以使用视图对数据进行地理空间查询（如果超出3维，则可以进行多维查询）。

当然，您仍然可以使用CouchbaseTemplate而不是CouchbaseRepository接口来访问较低级别的API，甚至可以从SDK中访问基础Bucket 。

通过N1QL的存储库方法

⭐⭐⭐

Couchbase 4.0中最大的新功能是N1QL ，这是一个可在JSON文档上使用的SQL扩展（因此它向SQL添加了与JSON相关的特性）。

这对于Spring Data中的Repository模式和查询派生尤其有用，因为绝大多数查询派生关键字都可以轻松转换为N1QL。

N1QL现在是存储库方法的默认后备Couchbase功能。如果要在执行的查询中显式显示，也可以选择使用@Query接口。

public interface UserRepository extends Repository {

  User findByUsernameEquals(String username);

  List findByUsernameContains(String contains);

  @Query //optional for N1QL query derivation but more explicit
  List findByAgeBetween(int minAge, int maxAge);
}

通过视图的存储库方法

⭐⭐

此版本的一个重大变化是，现在，基于视图的存储库查询（又名自定义存储库方法）更符合Spring Data的哲学。还必须使用@View(viewName="something")显式注释它们。

这意味着Couchbase特定的任何内容都不应泄漏到您的存储库界面中。相反，您可以做的是对大多数查询使用查询派生机制。

查询派生在很小的程度上也是可行的，在视图支持的方法中可以接受一些关键字。

public interface UserRepository extends Repository {

  @Override
  @View(designDocument = "user", viewName = "customFindAllView")
  Iterable findAll();

  @View(viewName = "customFindByNameView")
  User findByUsernameIs(String lowKey);

  @View(viewName = "customFindByNameView")
  List findByUsernameBetween(String lowKey, String highKey);
}

使用View的reduce函数

⭐

以前不支持的另一项新功能是执行reduce函数（如果有的话）。现在，为了执行它，您只需在@Viewannotation中将reduce标志设置为true。

如果对您有意义，您也可以在方法的前面加上“计数”而不是“查找”（即您实际上使用了“计数”减少功能）。

请注意，Couchbase中的reduce函数可以是除预先存在的_count之外的其他函数，甚至可以返回长于JsonObject ，例如内置_stats 。

同样，在方法名称中添加变体“ topX”或“ firstX”将导致在请求上设置其他限制（例如， findFirst5ByLastName将列表限制为5个结果）。

配置一致性，自己撰写

⭐⭐⭐

使用异步填充的辅助索引（例如视图和GSI）（支持N1QL的新的辅助索引引擎）时，经常出现的一件事是需要立即读取先前写入操作中的修改。

这意味着只要数据仍在被索引过程中，view / N1QL就不应响应，因此这会牺牲一些性能以利于一致性。

相反（Spring Data Couchbase的当前默认设置）是通过接受要返回的陈旧数据来提高性能。

通过围绕一致性概念提供一个小的抽象，我们添加了全局语义，用于配置框架通过查询派生构造的所有查询（基于视图或基于N1QL）。

这是通过重写AbstractCouchbaseConfiguration的getDefaultConsistency()方法来完成的。Consistency是一个枚举，让您选择READ_YOUR_OWN_WRITES ， STRONGLY_CONSISTENT ， UPDATE_AFTER和EVENTUALLY_CONSISTENT 。请参阅官方文档，以获取有关它们如何正常工作以及在查询时的影响的更多信息。

您还可以通过使用标签。

从GA开始，存储库中的CRUD方法现在也考虑了默认配置的一致性。

更改存储的JSON中的类型信息字段

⭐

一些用户报告了Spring Data和Couchbase Mobile方面的问题，而Sync Gateway拒绝了包含带下划线前缀的字段的文档。

这对于Spring Data是有问题的，因为默认情况下它将类型信息存储在_class字段中：(

解决方案是允许通过配置修改该类型信息字段的名称。您可以通过重写这样做typeKey()的方法AbstractCouchbaseConfiguration 。例如，您可以使用常量MappingCouchbaseConverter.TYPEKEY_SYNCGATEWAY_COMPATIBLE （即“ javaClass ”）。

此字段是生成的N1QL查询用来仅过滤与存储库实体相对应的文档的字段。

在N1QL派生查询中支持Pageable / PageRequest

⭐⭐

使用N1QL，对于通过查询派生生成的查询，现在支持Pageable和Sort参数。

  • 支持PagingAndSortingRepository基于N1QL。
  • 添加两个依赖N1QL进行分页和/或排序的findAll方法。使用默认配置的一致性。

使用空间视图进行地理空间和多维查询

⭐⭐⭐

使用坐标查询Couchbase！如果您的实体具有Point （或x和y ）位置，则可以使用以下方法找到它：

  • 一个边界框： findByLocationWithin(Box area)
  • 一个圆： findByLocationWithin(Circle area) ， findByLocationWithin(Point center, Distance radius) 。
  • 一个多边形： findByLocationWithin(Polygon area) ， findByLocationWithin(Point[] polygon) 。
  • 一个距离findByLocationNear(Point near, Distance maxDistance) 。

作为服务器上的边界框近似值，可以快速执行圆形和类似多边形的查询，然后在呈现结果之前，框架会消除误报。

您可以利用Couchbase空间视图的多维方面为查询添加额外的维度（例如，在城市中营业至深夜的商店…）。

public interface DimensionalPartyRepository extends CrudRepository {

  @Dimensional(designDocument = "partyGeo", spatialViewName = "byLocation")
  List findByLocationNear(Point p, Distance d);

  @Dimensional(designDocument = "partyGeo", spatialViewName = "byLocation")
  List findByLocationWithin(Box boundingBox);

  @Dimensional(designDocument = "partyGeo", spatialViewName = "byLocation")
  List findByLocationWithin(Polygon zone);
  
  @Dimensional(designDocument = "partyGeo", spatialViewName = "byLocationAndAttendees", dimensions = 3)
  List findByLocationWithinAndAttendeesGreaterThan(Polygon zone, double minAttendees);
}

注意：如果要重用注释，也可以这样做（也适用于@View和@Query ）：

public interface DimensionalPartyRepository extends CrudRepository {

  //define your own meta-annotation
  @Dimensional(designDocument = "partyGeo", spatialViewName = "byLocation", dimensions = 2)
  @Retention(RetentionPolicy.RUNTIME)
  @interface IndexedByLocation { }
  
  //use it :)
  @IndexedByLocation
  List findByLocationNear(Point p, Distance d);

  @IndexedByLocation
  List findByLocationWithin(Box boundingBox);

  @IndexedByLocation
  List findByLocationWithin(Polygon zone);
  
  //here we use a variation with 3 dimensions, so we need to revert to @Dimensional
  @Dimensional(designDocument = "partyGeo", spatialViewName = "byLocationAndAttendees", dimensions = 3)
  List findByLocationWithinAndAttendeesGreaterThan(Polygon zone, double minAttendees);
}

内联N1QL @Query现在具有SpEL支持

⭐⭐⭐

内联查询可以使用SpEL表示法来：

  1. 确保正确的选择和过滤应用于语句以构造和返回实体：使用#{#n1ql.selectEntity}生成SELECT ... FROM ...子句和#{#n1ql.filter}将在WHERE子句限制查询到正确的实体。
  2. 计算值或从Spring上下文中配置的外部SpEL值提供检索数据。

存储库“主”索引创建可以自动触发

⭐⭐

⚠️ 重要：在开发/测试过程中被认为是一种帮助，在生产中不鼓励使用

为了确保在开发环境或预生产环境中激活给定存储库中实体的N1QL索引，可以使用@N1qlPrimaryIndexed （启用存储桶范围的自由格式查询）和@N1qlSecondaryIndexed （仅对索引进行注释）对其进行注释。对应于该实体的类型，通过使用SpEL产生的WHERE子句类似于文件#{#n1ql.filter}

此外，对于CRUD操作后盾视图可以自动通过注释与信息库中创建@ViewIndexed （你需要提供设计文件名称，其对应于实体的简单类名首字母小写）。

还必须通过在AbstractCouchbaseConfiguration重新定义indexManager bean来选择启用此功能。

使用单行投影时，现在支持简单的返回类型（基本类型和String ）

⭐⭐

这特别针对具有聚合函数（如COUNT(*) ， AVG(field)等COUNT(*)内联N1QL查询。查询必须返回具有单个投影的单行。

在N1QL内联查询中支持命名参数

⭐⭐

使用命名参数或位置参数，但不能同时使用。命名参数的语法为$paramName ，要求每个方法参数都使用@Param("paramName")进行注释。

其它功能

⭐

其他功能包括：

  • 修复Bean命名问题，以使Spring Data Couchbase创建的所有bean都以“ couchbase ”为前缀，以避免与其他存储区发生冲突。
  • 现在支持更改所有存储库的基类（遵循通用Spring Data文档中记录的过程）
  • 如果索引过期，则从CouchbaseTemplate find方法中删除已删除的文档
  • 到期时间可以在@Document上设置为long + timeUnit

RC1的一些错误修正和改进也已实现。

文献资料

⭐⭐⭐

文档也得到了改进，添加了面向Couchbase的示例，这些示例涉及如何向存储库添加自定义方法的实现，如何更改所有存储库的基类，如何在内联查询中处理SpEL，等等。

关于Spring Cache的说明

Spring Cache支持已从Spring Data存储库中移出。它仍然存在，我们计划对其进行改进。您现在可以在github上的Couchbase 存储库中找到它，但是它应该很快重新集成Spring官方项目系列。
