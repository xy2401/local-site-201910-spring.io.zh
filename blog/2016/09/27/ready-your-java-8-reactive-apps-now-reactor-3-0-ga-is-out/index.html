立即准备您的Java 8 Reactive应用程序，Reactor 3.0 GA即将发布！

发布
斯蒂芬·马尔蒂尼（Stephane Maldini）
九月27，2016

经过一年多的工作 ， 多个里程碑并根据大量反馈进行了微调，我很高兴宣布Reactor 3正式上市。您会发现Reactor Core 3.0.2。在Maven Central上发布。

什么是Reactor 3？

Reactor 3为基于Java 8的应用程序提供了强大而高效的反应式编程模型。该模型以Reactor 2和RxJava 1的经验为基础，并引入了一种流畅的方式来编写异步可用于背压的事件处理。Spring Framework 5使用Reactor 3来构建并最终公开完整的反应式故事 。

其设计依赖于支持事件处理托管的可扩展执行模型。通常，只有在明确要求时，Reactor才会在事件流阶段之间跳转线程。例如，诸如列表访问或有效负载转换之类的内存中操作通常不需要线程边界。如果操作生产者或接收者可能需要一些时间，则期望用户使用Flux#publishOn / Mono#publishOn要么Flux#subscribeOn / Mono#subscribeOn并选择要运行的调度程序 。或者，如果用户正在合并许多结果Publisher与merge或concat一样 ，Reactor将以线程安全的方式隐式处理各种潜在的线程边界。实际上，由Publisher至少可能由生产线程和接收线程遍历。

所有这些都是根据Reactive Streams规范定义的行为实现的。Reactor致力于使规范成为图书馆作家或Spring开发人员的商品，并带有预制的运算符，您可以在流和/或异步方案中应用这些运算符。

单声道和助焊剂

流量定义是Reactor所谓的Flux或Mono链，具体取决于流入定义流的体积。这些类型实现反应式流Publisher在每个阶段都可以通用。为什么有两种反应类型呢？因为基数对Reactor很重要。一种Flux将观察0到N个项目，最终成功或失败。一种Mono将观察0或1项， Mono最多提示0个项目。

让我们看一下以下阻塞API：

interface BlockingUserRepository {
     User get(String id);
     List<User> findAll();
     void save(User data) throws RepositoryException;
     List<User> findAllByUsernameLike(String s);
}

使用普通的反应流Publisher ，我们将获得以下合同：

interface ReactiveUserRepository {
     Publisher<User> get(String id);
     Publisher<User> findAll();
     Publisher<Void> save(Publisher<? extends User> source);
     Publisher<User> findAllByUsernameLike(String s);
}

但是使用Reactor，我们可以保留预期基数的语义证据：

interface ReactorUserRepository {
     Mono<User> get(String id);
     Flux<User> findAll();
     Mono<Void> save(Publisher<? extends User> source);
     Flux<User> findAllByUsernameLike(String s);
}

由于两者Mono和Flux实行Publisher我们可以轻松地将它们的任何引用作为响应数据源传递，同时返回带有Mono流利的API：

// ReactorUserRepository userRepository;

userRepository.save(Mono.fromCallable(() -> new User("thomas")))
              .doOnSuccess(res -> success())
              .subscribe();

userRepository.save(Flux.just(new User("bob"), new User("robert")))
              .doOnSuccess(res -> success())
              .subscribe();

请记住Flux和Mono适用于可能需要时间的数据生产者。为了管理重入和线程安全，操作员有时必须在执行的流程中增加一些开销。尽管如此，效率仍然是我们关注的重点，我们从引擎贡献者David Karnok那里获得定期报告。当前，Reactor 3是JVM上最高效的反应式库之一。除了这些直接相关的基准测试之外，我们现在还从RxJava 2社区反馈中受益，因为它在概念上源自相同的smith和forge： Reactive Streams Commons 。

下一步是什么 ？

我们将在接下来的几周内开发3.0.3，并与Spring 5，CloudFoundry的最新需求以及Reactive Streams Commons的最新研究保持同步。

我们将优先处理：

  • 新的测试支持：原计划对Reactor 3提供测试支持，但是最初的反馈提出了一些用户体验问题。我们现在正在努力交付缺失的那部分。同时，用户可以根据需要轻松地从我们的测试中复制隔离的TestSubscriber 。

  • 指导：当Reactor 3变得越来越流行时，我们仍在内部或外部与许多人类交互，具有Rx知识的高级用户以及Sebastien Deleuze撰写的快速教程进行合作。您将在本文末尾找到更多资源，但是我们已经开始建立一些端到端的场景，我们认为这些场景具体，有价值，并将帮助制定正式的参考反应堆指南。

反应堆IPC

IPC代表进程间通信，而Reactor IPC是一项持续不断的倡议，旨在回答“如何以反应式流方式摆脱JVM”的问题。我们正在与Reactor Kafka ， Reactor Aeron和Reactor Netty一起进行一组初始实施。实际上，合同的重新设计和Reactor Kafka / Netty的工作现在正在发生很多事情，这些工作支持了一些新的Spring反应故事。IPC伞的目的不是创建新的Web或消息框架，而是可以在其上构建反应性驱动程序应用程序或库。他们将给定的运行时输入/输出转换为Flux和Mono要么Subscriber ，将反应性背压传播到IO接入层。在接下来的几个月中，将会有很多关于这些计划的新闻。

学分

RxJava 1将主流的Reactive带入了JVM，其完整的功能性Rx代数已成为我们所遵循的工业标准。它启发了我们使用Reactive Streams实现的改变游戏规则的规范，并将JVM主要参与者（如Netflix，Oracle，Pivotal，Typesafe，Red Hat等）汇集在一起。
