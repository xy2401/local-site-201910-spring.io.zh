提供了用于Spring Integration 1.2 Milestone 2的Java DSL

发布
阿尔特姆·比兰（Artem Bilan）
九月15，2016

我很高兴地宣布，用于 Spring 集成的Java DSL 1.2 M2现在可用！

首先，我要感谢所有在StackOverflow上提出问题，提出拉取请求，提供反馈或只是问问题的人。特别感谢自上一个里程碑以来的早期采用者1 。在他们的帮助下，我们改进并修复了运行时流注册中的一些问题。

神器org.springframework.integration:spring-integration-java-dsl:1.2.0.M2可在Milestone仓库中使用 。因此，请尝试一下，不要犹豫，提出GH问题以获取任何反馈！

当前迭代的一些亮点：

JPA支持

经过许多社区的要求，我们终于推出了Jpa工厂和相应的“ IntegrationComponentSpec”为Spring Integration JPA组件提供流利的API：

@Autowired
private EntityManagerFactory entityManagerFactory;

@Bean
public IntegrationFlow pollingAdapterFlow() {
    return IntegrationFlows
            .from(Jpa.inboundAdapter(this.entityManagerFactory)
                    .entityClass(StudentDomain.class)
                    .maxResults(1)
                    .expectSingleResult(true),
                e -> e.poller(p -> p.trigger(new OnlyOnceTrigger())))
            .channel(c -> c.queue("pollingResults"))
            .get();
}

@Bean
public IntegrationFlow updatingGatewayFlow() {
    return f -> f
            .handle(Jpa.updatingGateway(this.entityManagerFactory),
                    e -> e.transactional(true))
            .channel(c -> c.queue("persistResults"));
}

@Bean
public IntegrationFlow retrievingGatewayFlow() {
    return f -> f
            .handle(Jpa.retrievingGateway(this.entityManagerFactory)
                    .jpaQuery("from Student s where s.id = :id")
                    .expectSingleResult(true)
                    .parameterExpression("id", "payload"))
            .channel(c -> c.queue("retrieveResults"));
}

中流交易支持

受到Spring Integration JPA组件的事务支持配置的复杂性的“启发”（实际上是编程的TransactionalInterceptor ），我们已经介绍了TransactionInterceptorBuilder 。此外，我们提供TransactionHandleMessageAdvice这允许从任何端点开始整个子流的事务，而不仅仅是handleRequestMessage与常规情况一样ConsumerEndpointSpec.advice() 。实际上，主要技巧是通过HandleMessageAdvice ，最近在Spring Integration Core中引入，它是一个标记界面，用于区分针对handleRequestMessage仅或从当前开始的流量MessageHandler 。为了方便起见.transactional()方法已添加到ConsumerEndpointSpec 。

分散收集支持

现在， Scatter-Gather EI模式具有自己的Java DSL API：

@Bean
public IntegrationFlow scatterGatherFlow() {
    return f -> f
      .scatterGather(scatterer -> scatterer
         .applySequence(true)
         .recipientFlow(m -> true,
                     sf -> sf.handle((p, h) -> Math.random() * 10))
         .recipientFlow(m -> true,
                     sf -> sf.handle((p, h) -> Math.random() * 10))
         .recipientFlow(m -> true,
                     sf -> sf.handle((p, h) -> Math.random() * 10)),
      gatherer -> gatherer
         .releaseStrategy(group ->
                group.size() == 3 ||
                      group.getMessages()
                          .stream()
                          .anyMatch(m -> (Double) m.getPayload() > 5)),
      scatterGather -> scatterGather
        	 .gatherTimeout(10_000));
}

哪里scatterer只是一个RecipientListRouter ， gatherer -一个AggregatingMessageHandler和最后Consumer接受选项ScatterGatherHandler 。

更多路由器改进

的.routeToRecipients() API现在为收件人提供了更多配置变体：

.routeToRecipients(r -> r
    .recipient("foo-channel", "'foo' == payload")
    .recipient("bar-channel", m ->
        m.getHeaders().containsKey("recipient")
            && (boolean) m.getHeaders().get("recipient"))
    .recipientFlow("'foo' == payload or 'bar' == payload or 'baz' == payload",
        f -> f.<String, String>transform(String::toUpperCase)
            .channel(c -> c.queue("recipientListSubFlow1Result")))
    .recipientFlow((String p) -> p.startsWith("baz"),
        f -> f.transform("Hello "::concat)
            .channel(c -> c.queue("recipientListSubFlow2Result")))
    .recipientFlow(new FunctionExpression<Message<?>>(m ->
                                   "bax".equals(m.getPayload())),
        f -> f.channel(c -> c.queue("recipientListSubFlow3Result")))
    .defaultOutputToParentFlow())

以前的.route()下一个操作员.channel()在里面IntegrationFlow作为一个defaultOutputChannel的Router 。根据用户的经验，做出这样无条件的决定听起来并不合理。我们重做.route()使它与标准对齐AbstractMessageRouter行为。的.defaultOutputChannel()和.defaultSubFlowMapping()已添加以利用default逻辑Router 。要回退到以前的行为.defaultOutputToParentFlow()正如您所注意到的.routeToRecipients()以上示例。

查看提交历史 1.2.0.M2版本以获取更多信息。并始终阅读JavaDocs以了解您使用的API！

下一步

我们期待版本的第一个（希望最后一个）发行候选版本1.2在经过几周的Spring Integration 4.3.2和Spring Boot 1.4.1的采用之后。尽快spring-integration-java-dsl将移至Spring Integration Core 5.0和Java 8代码库。目前1.2仍将支持该版本，但仅用于错误修复。
