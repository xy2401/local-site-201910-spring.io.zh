Spring Cloud Stream 布鲁克林。M1可用

发布
马里乌斯·波哥维奇（Marius Bogoevici）
2016年8月26日

我很高兴代表团队宣布发布Spring Cloud Stream布鲁克林发布火车的第一个里程碑。 Spring Cloud Stream 布鲁克林。M1可在Spring Milestone存储库中使用 ，有关其功能的详细说明，请参见参考文档 。此处提供发行说明，其中包含有关迁移路径的重要信息。

从整体到发布

Spring Cloud Stream 布鲁克林。M1继承了Spring Cloud Stream 1.0。命名方案的更改反映了项目的结构更改，特别是从将核心组件和活页夹实现包含在一起的整体结构切换到更加分散的结构。在新结构中，核心和活页夹实现是独立的项目，具有各自的发布节奏。发布系列BOM将发布组件聚合在一起并管理其版本。

这种方法的好处是双重的。一方面，它允许更快地为各个实现添加新功能和修复。另一方面，它降低了创建和开发新活页夹的障碍，随着它们的成熟，只需将其添加到发布列的BOM中，它们就可以自己成为将来发布列的一部分。最后，它仅适合于以微服务开发为目标的项目，而不是一个整体！

以下组件是布鲁克林的一部分。M1放行列车：

  • Spring Cloud Stream 1.1.0。M1（核心组件）
  • Spring Cloud Stream Kafka Binder 1.1.0。M1
  • Spring Cloud Stream Rabbit活页夹1.1.0。M1

让我们看看新版本带来了什么。

有什么新东西？

Spring Cloud Stream 布鲁克林。M1对现有组件进行了一些重大升级，并引入了针对编程模型，应用程序互操作性和整体开发人员体验的新功能。

Apache Kafka 0.9新的消费者支持

Apache Kafka Binder已升级为使用基于Spring Kafka 1.0.x和Spring Integration Kafka 2.0.x的Kafka新的使用者库（版本0.9引入），当前支持Apache Kafka 0.9。新客户端库的使用增加了一些好处：

  • 使用Apache Kafka Binder的应用程序不必具有spring.cloud.stream.instanceIndex组。新的使用者库将负责将分区分配给给定使用者组的所有实例，并在新实例加入或退出时进行重新平衡-特别适用于在运行时扩展使用者应用程序。对于仍希望使用静态分区分配方案的用户，我们将在最终版本中支持该方案（ spring.cloud.stream.instanceIndex在这种情况下仍然需要）。
  • 通过SSL和Kerberos支持受保护的客户端 。

Spring Kafka和Spring Integration Kafka的抽象也将使添加对Apache Kafka 0.10的支持也变得更加容易，并且期望最终版本也将支持Kafka 0.10。

响应式编程支持（使用Java 8）

除了Spring Integration应用程序模型和@StreamListener此版本包含在1.0版中，引入了对基于Project Reactor的 反应式API的支持。此功能需要使用Java 8。

应用程序可以添加spring-cloud-stream-reactive模块作为依赖项，从而可以直接将反应类型用作编程抽象。在数据处理的背景下，功能和反应式编程模型非常有吸引力，因为反应式合成API具有声明性和表达性，这使开发人员可以专注于做什么而不是怎么做。另一方面，虽然单独处理消息是企业集成的经典范例，但在流处理方面，开发人员还需要考虑处理连续的入站消息流，并描述仅在这种情况下才有意义的操作，如开窗-按时间或计数。

这是一个反应式处理器的示例，用于计数单词（用于确定最近5秒钟内每秒最受欢迎的标签）。

@StreamListener
@Output(Processor.OUTPUT)
public Flux<WordCount> count (@Input(Processor.Input) Flux<String> flux) {
  return flux.window(ofSeconds(5), ofSeconds(1))
    .flatMap(window ->
      window.groupBy(word -> word)
        .flatMap(group -> group.reduce(0, (count,word) -> count + 1)
          .map(count -> new WordCount(group.key(), count))));
}

此外，使用反应性API可以与其他反应性组件（例如反应性Web控制器）集成。要获得有关春季即将到来的反应式支持的更全面概述，请观看Stephane Maldini和Rossen Stoyanchev 在Spring One Platform 2016上的主题演讲，Spring One Platform上的其他Reactive演讲之一，或阅读Dave Syer的博客系列 。

Avro序列化和架构演变支持

Spring Cloud Stream 布鲁克林。M1还增加了对Avro和架构演变的支持。从此版本开始，应用程序可以包括spring-cloud-stream-schema模块，其中包含MessageConverters使用Apache Avro。

Apache Avro序列化器既支持固定模式 ，又与模式注册表动态交互 。您只需添加以下内容，即可使您的应用程序与架构注册表进行交互@EnableSchemaRegistryClient到您的应用程序并设置application/*+avro内容类型在您的出站通道上，以便以Apache Avro格式发送数据。这样，发布者应用程序将为其发送的消息注册架构，并将有关主题和版本的元数据传递给使用者。基于此，使用者可以从注册表中检索编写者的架构，并反序列化接收到的消息，即使事先不知道该架构也是如此。

这是微服务演进的重要功能，因为它允许系统的不同组件在不破坏现有组件的架构和数据格式的情况下升级或更改其架构和数据格式。

该发行版包括一个架构注册表服务器和一个通用架构注册表客户端 。还提供Confluent模式注册表的模式注册表客户端的实现。

特别感谢Vinicius Carvalho贡献了模式发展支持！

接下来是什么 ？

在接下来的几周中，我们将继续开发布鲁克林发布火车，以期产生候选发布车。按照里程碑的惯例，在RC之前可能需要对API进行一些更改。在最终版本发布之前，以下是一些预期的附加功能：

  • 通过简单的1.1版本的Spring Kafka库的直接替换来添加对Apache Kafka 0.10的支持；
  • 增加对反应性粘合剂（包括反应性生产者和消费者，例如Reactor Kafka项目引入的反应性生产者和消费者）的支持；
  • 增加对使用Kafka Stream API开发应用程序的支持;

而且，一如既往，我们欢迎您提供反馈：在GitHub ， Stack Overflow或Twitter上 。
