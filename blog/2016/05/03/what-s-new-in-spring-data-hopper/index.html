Spring Data Hopper有什么新功能？

工程
克里斯多夫·斯特罗布尔
2016年5月3日

正如我们刚刚发布的Spring数据发布火车Hopper 的GA版本一样 ，让我们更深入地研究火车上13个模块附带的更改和功能。发行版依赖项中的一个非常根本的变化是将Spring Framework 4.2（当前为4.2.5）升级为基准。这是为即将发布的4.3框架做的准备。我们还借此机会将Querydsl集成升级到4.x（当前为4.1），这需要对非常核心的抽象进行一些重大更改。除此之外，Hopper还包含许多重要的主要版本更改模块。

这些升级主要是由底层商店驱动程序和实现的主要版本变化驱动的，这些变化需要反映在这些模块公开的API的潜在重大更改中。其中一些模块（例如Spring Data Neo4j和Spring Data Couchbase）已经在发行版之外看到了一个新的主要发行版，现在已重新集成到其中。

请欢迎（返回）：

  • 基于Couchbase 2.2的Spring Data Couchbase 2.1
  • 基于Elasticsearch 2.2的Spring Data Elasticsearch 2.0
  • 基于Neo4J OGM 2.0的Spring Data Neo4j 4.1
  • 基于Solr 5.5的Spring Data Solr 2.0

除了这些升级外，团队还一直在开发一系列新功能。

  • 使用@AliasFor可组合注释
  • 示例查询
  • 存储库查询方法的投影
  • Redis集群和存储库支持
  • MongoDB的$lookup聚合和批量操作
  • 在Gemfire和Redis中使用Spring 4.3时，同步缓存查找
  • Querydsl 4支持

我想在博文的其余部分中花一些点介绍这些功能，以便为您提供有关它们的更详细的概述。

示例查询

Spring Data存储库抽象允许通过Querydsl执行查询方法和灵活的谓词已经有一段时间了。就是说，长期以来一直要求能够提供部分设置的域类型实例作为对存储库的探针以返回与该特定探针匹配的所有实体的功能。Hopper在Spring Data Commons中引入了对此示例查询机制的一般支持，并在JPA和MongoDB模块中实现了API的实现（更多内容将在以后介绍）。

示例查询API包含四个基本部分：

  • 探针实例，它是域模型的实例，仅部分填充了字段。
  • 一个可选的ExampleMatcher ，它包含有关如何匹配特定字段， null ，一般情况下的String的详细信息和策略。
  • 一个Example ，由探针和ExampleMatcher 。
  • 您的存储库将另外实现的QueryByExampleExecutor接口，该接口提供采用Example的方法，类似于QueryDslPredicateExecutor 。

在最普通的情况下，只需在域类型上设置要查询的值，然后将该示例移交给存储库即可。

interface PersonRepository extends CrudRepository<Person, Long>,
  QueryByExampleExecutor<Person> { … }

Example<Person> example = Example.of(new Person("Jon", "Snow"));
Iterable<Person> result = repository.findAll(example);

默认情况下，给定值按原样匹配，在查询创建期间会忽略null 。您可以通过提供ExampleMatcher来自定义常规或单个字段的处理，从而更好地控制匹配过程。

ExampleMatcher exampleSpec = new ExampleMatcher()
  .withMatcher("firstname", endsWith())
  .withMatcher("lastname", startsWith().ignoreCase());

上面的规范创建了例如。上述示例中的JPA谓词like(firstname, "%Jon")和like(lower(expression), "snow%") 。

有更多可用的选项。因此，请查看Spring Data JPA和MongoDB的参考文档。

关于存储库查询方法的投影

投影的概念Spring Data REST在Evans发布火车中附带了一个称为投影的功能。借助Hopper，我们增加了对JPA和MongoDB查询方法的支持，以在存储库级别使用投影。投影是域模型的自定义视图，在这种情况下，是从查询方法返回的。

@Entity
public class Person {

  @Id @GeneratedValue
  private Long id;
  private String firstName, lastName;

  @OneToOne
  private Address address;
}

@Entity
public class Address {

  @Id @GeneratedValue
  private Long id;
  private String street, state, country;
}

将Person数据firstName限制为firstName和lastName需要专用的DTO类。使用投影，您只需定义一个带有您要公开的属性（getter方法）的接口，然后将投影接口用作查询方法的返回类型：

interface NoAddresses {

  String getFirstName();

  String getLastName();
}

interface PersonRepository extends CrudRepository<Person, Long> {
  NoAddresses findByFirstName(String firstName);
}

投影是一种强大的模式，可根据现有模型构造调整后的视图。NoAddresses是一个封闭的投影，因为它不包含任何动态计算值的方法（请参阅下文中的更多信息）。封闭式投影使我们能够优化查询执行，因为仅从数据存储中查询了公开的属性。因此，在上述情况下，实际执行的查询在语义上等效于select u.firstName, u.lastName from User u where u.firstName = ?1 。然后，将返回的元组包装到一个代理中，该代理返回与声明的访问器相对应的值。

但是，投影也可以用于丰富数据模型。您可以标注暴露与性能@Value利用规划环境地政司表情，露出综合性能。

interface FullNameAndCountry {

  String getFirstName();

  String getLastName();

  @Value("#{target.firstName} #{target.lastName}")
  String getFullName();

  @Value("#{target.address.country}")
  String getCountry();

  @Value("#{@mybean.someMethod(target)}")
  String getSomeCalculatedValue()
}

请注意，我们如何利用目标实例的属性，遍历甚至不在顶层公开的嵌套属性，甚至调用其他Spring Bean上的方法，并将目标移交给它进行高级计算。在这种情况下，不会应用查询优化，因为为该接口创建的代理将需要访问原始目标实例。

请参阅Spring Data JPA和Spring Data MongoDB参考文档，以获取有关如何在查询方法中使用投影的更多详细信息。

Redis集群

对Redis群集的支持在具有群集功能的现有Redis驱动程序之上提供了高级API。集群支持基于与非集群通信完全相同的构造块。RedisClusterConnection的扩展， RedisConnection处理与Redis群集的通信，并将错误转换为Spring的DataAccessException层次结构。您会发现与使用Spring Data Redis时所使用的功能没有太大区别。

Redis群集的行为不同于单节点Redis甚至是Sentinel监视的主从环境。这是由将密钥映射到分布在节点上的16384个插槽之一的自动分片引起的。因此，涉及多个键的命令必须断言所有键都映射到完全相同的插槽，以避免跨插槽执行错误。

RedisClusterConnection提供了与单个插槽或节点通信的API，但在与集群交互时保留了预期的行为。它负责执行涉及多个密钥，插槽或群集节点的命令，因此连接到所需的节点并收集结果，因此，例如KEYS命令不仅返回单个节点的匹配密钥，而且还返回所有匹配项的累积列表集群中的密钥。

在Spring Data Examples GitHub存储库的Redis Cluster模块以及参考文档中可以找到更多信息以及如何设置Spring Data Redis与集群一起使用的完整示例。

Redis存储库

借助Redis存储库，Hopper在Redis上附带了Spring Data存储库抽象的实现，以便您可以执行基本的CRUD操作并执行派生的查询方法。它允许您无缝地将域对象转换并存储到Redis哈希中，应用自定义映射策略并使用二级索引。让我们看一个示例域类型，一个存储库和必要的Spring配置，以引导Redis存储库启动。

@RedisHash("persons")
class Person {

  @Id String id;
  String firstname;
  @Indexed String lastname;
  Address address;
}

interface PersonRepository extends CrudRepository<Person, String> {
  List<Person> findByLastname(String lastname);
}

@Configuration
@EnableRedisRepositories
class ApplicationConfig {

  @Bean
  RedisConnectionFactory connectionFactory() {
    return new JedisConnectionFactory();
  }

  @Bean
  RedisTemplate<?, ?> redisTemplate() {
    return new RedisTemplate<byte[], byte[]>(connectionFactory());
  }
}

请注意，在firstname上使用@Indexed批注允许在派生的查询方法中使用该属性。这也适用于嵌套或嵌入式对象。有关自定义对象映射策略，到期时间和侦听器以及存储对象引用的更多信息，请参考Spring Data Redis参考文档 。

Spring Data REST

对于Spring Data REST的一个常见请求是使用聚合的唯一属性的值来构成所公开项目资源的URI。想象一个非常简单的实体Country ，您想要在URI中为其指定唯一名称。

@Entity
class Country {

  @Id @GeneratedValue
  private Long id;
  private String name;
}

为了完成这项工作，Spring Data REST的RepositoryRestConfiguration现在允许通过专用API自定义实体查找。为URI使用不同的属性实际上需要定义两件事：要使用的属性以及存储库上的查询方法，用于将该属性的值映射回其实例。如果您使用的是Java 8，则注册看起来像这样简单：

@Component
public class SpringDataRestCustomization extends RepositoryRestConfigurerAdapter {

  @Override
  public void configureRepositoryRestConfiguration(
    RepositoryRestConfiguration config) {

    config.withCustomEntityLookup().
      forRepository(CountryRepository.class,
        Country::getName, CountryRepository::findByName);
  }
}

如您所见，我们在此处使用方法句柄来定义来回映射步骤，以供基础结构使用。当然，这里显示的方法也可以在Java 6上使用，还有一个替代方法。有关此内容的详细信息，请确保您在GitHub上查看了专用于此的示例 。

查询类型

域模型通常包含作为值对象的类型，但实际上代表专用值集中的特定值。上面示例的Country类实际上属于该类别。因为我们需要管理超级值集，所以这里有一个存储库。如果还应该允许它通过REST管理集合，则也需要导出存储库。由于存储库通常指示正在管理的聚合，因此，Spring Data REST处理该场景的默认方法是在遇到Country实例的任何地方都提供指向关联资源的链接。Hopper发布火车添加了声明所谓的查找类型的方法，Spring Data REST随后为其提供呈现形式的内联的单个属性，并注册相应的Jackson Deserializer，以确保将该属性值转换回该值的实例。 PUT和POST请求的类型。

假定包含Country实例的资源的原始表示形式：

{
  "zipCode" : "…",
  "_links" {
    "country" : { "href" : "…" }
  }
}

如果您现在继续将Country注册为查找类型，如下所示：

@Component
public class SpringDataRestCustomization extends RepositoryRestConfigurerAdapter {

  @Override
  public void configureRepositoryRestConfiguration(
    RepositoryRestConfiguration config) {

    config.withCustomEntityLookup().
      forLookupRepository(CountryRepository.class).
      withIdMapping(Country::getName).
      withLookup(CountryRepository::findByName);
  }
}

表示将更改为此，但仍在模型中保留实体语义：

{
  "zipCode" : "…",
  "country" : "Germany"
}

可组合的注释

升级到Spring 4.2作为框架基准，使我们能够提供增强的选项，以组成您自己的注释。我们将基础基础架构添加到了Spring Data Commons模块中，并对JPA，MongoDB和Redis的实现进行了调整，以使您能够利用这些更改。

假设您在许多这样的地方一起使用Spring Data JPA批注@Modifying和@Query ：

@Modifying
@Query("update #{#entityName} u set u.active = ?1 where u.id in ?2")
void updateUserActiveState(boolean activeState, Integer... ids);

@Modifying
@Query
@Retention(RetentionPolicy.RUNTIME)
public @interface ModifyingQuery {

  @AliasFor(annotation = Query.class, attribute = "value")
  public String query();
}

@ModifyingQuery(query =
  "update #{#entityName} u set u.active = ?1 where u.id in ?2")
void updateUserActiveState(boolean activeState, Integer... ids);

或者只是玩一点，然后将现有注释转换为您喜欢的语言 。
