带有Kotlin，Spring Boot和PostgreSQL的地理空间Messenger

工程
塞巴斯蒂安·德勒兹（SébastienDeleuze）
2016年3月20日

继我在Kotlin的第一篇博客文章之后 ，今天我要介绍我为即将举行的Spring I / O 2016会议演讲“使用Kotlin和Spring Boot开发地理空间Web服务”开发的新Spring Boot + Kotlin应用程序。

处理本机数据库功能

该应用程序的目标之一是了解如何像在NoSQL世界中一样利用本机数据库功能。在这里，我们要使用PostGIS提供的地理空间支持， PostGIS是PostgreSQL的空间数据库扩展程序。 本机JSON支持也可能是一个很好的用例。

该Geospatial Messenger示例应用程序可在GitHub上以两种方式获得：
-的master分支使用Exposed ，这是一个Kotlin SQL库，具有由JetBrains创建的类型安全API。可以将其与Query DSL SQL或jOOQ进行比较，但提供了惯用的Kotlin API，并且不需要代码生成。
-的spring-data-jdbc-repository分支正在使用spring-data-jdbc-repository ，一个允许使用Spring Data的社区项目PagingAndSortingRepository带有原始SQL查询的API，无需JPA。我正在使用这个Jakub Jirutka叉子 ，它是Tomasz Nurkiewicz原始项目的改进版本。

Spring Data JPA + Hibernate Spatial变体会很有趣，因此随时可以通过请求请求贡献它；-) Kotlin Query DSL支持也很不错，但是目前不支持（请对此问题发表评论） 。在此博客文章中，我将重点介绍Exposed变体。

地理空间Messenger代码之旅

领域模型

通过这两个Kotlin类，可以轻松描述我们的领域模型：

class Message(
    var content  : String,
    var author   : String,
    var location : Point? = null,
    var id       : Int?   = null
)

class User(
    var userName  : String,
    var firstName : String,
    var lastName  : String,
    var location  : Point? = null
)

SQL模式

使用Exposed，我们可以使用易于使用的类型安全的SQL API描述表的结构（自动完成，重构和易于出错）：

object Messages : Table() {
    val id       = integer("id").autoIncrement().primaryKey()
    val content  = text("content")
    val author   = reference("author", Users.userName)
    val location = point("location").nullable()
}

object Users : Table() {
    val userName  = text("user_name").primaryKey()
    val firstName = text("first_name")
    val lastName  = text("last_name")
    val location  = point("location").nullable()
}

有趣的是，Exposed不原生支持PostGIS功能，例如几何类型或地理空间请求。这就是Kotlin扩展的亮点，并允许使用几行代码来添加此类支持，而无需使用扩展类：

fun Table.point(name: String, srid: Int = 4326): Column<Point>
  = registerColumn(name, PointColumnType())

infix fun ExpressionWithColumnType<*>.within(box: PGbox2d) : Op<Boolean>
  = WithinOp(this, box)

储存库

更新 ：我们现在可以使用“ 暴露的” @Transactional支持 ！交易管理只需配置@EnableTransactionManagement注释和PlatformTransactionManager豆在Application类。

我们的存储库也很短而且非常灵活，因为即使您使用复杂的存储库，它们也可以编写任何类型的SQL请求WHERE类型安全的SQL API子句。

请注意，由于我们使用的是Spring Framework 4.3， 因此不再需要指定@Autowired此类单构造函数类中的注解 。

interface CrudRepository<T, K> {
    fun createTable()
    fun create(m: T): T
    fun findAll(): Iterable<T>
    fun deleteAll(): Int
    fun findByBoundingBox(box: PGbox2d): Iterable<T>
    fun updateLocation(userName:K, location: Point)
}

interface UserRepository: CrudRepository<User, String>

@Repository
@Transactional // Should be at @Service level in real applications
class DefaultUserRepository(val db: Database) : UserRepository {

    override fun createTable() = SchemaUtils.create(Users)

    override fun create(user: User): User {
        Users.insert(toRow(user))
        return user
    }

    override fun updateLocation(userName:String, location: Point) = {
        location.srid = 4326
        Users.update({Users.userName eq userName})
            { it[Users.location] = location }
    }

    override fun findAll() = Users.selectAll().map { fromRow(it) }

    override fun findByBoundingBox(box: PGbox2d) =
            Users.select { Users.location within box }
                 .map { fromRow(it) }

    override fun deleteAll() = Users.deleteAll()

    private fun toRow(u: User): Users.(UpdateBuilder<*>) -> Unit = {
        it[userName] = u.userName
        it[firstName] = u.firstName
        it[lastName] = u.lastName
        it[location] = u.location
    }

    private fun fromRow(r: ResultRow) =
        User(r[Users.userName],
             r[Users.firstName],
             r[Users.lastName],
             r[Users.location])
}

控制器

控制器也非常简洁，即将使用Spring Framework 4.3 @GetMapping / @PostMapping批注只是方法的特定快捷方式@RequestMapping注释：

@RestController
@RequestMapping("/user")
class UserController(val repo: UserRepository) {

    @PostMapping
    @ResponseStatus(CREATED)
    fun create(@RequestBody u: User) { repo.create(u) }

    @GetMapping
    fun list() = repo.findAll()

    @GetMapping("/bbox/{xMin},{yMin},{xMax},{yMax}")
    fun findByBoundingBox(@PathVariable xMin:Double,
                          @PathVariable yMin:Double,
                          @PathVariable xMax:Double,
                          @PathVariable yMax:Double)
            = repo.findByBoundingBox(
                        PGbox2d(Point(xMin, yMin), Point(xMax, yMax)))

    @PutMapping("/{userName}/location/{x},{y}")
    @ResponseStatus(NO_CONTENT)
    fun updateLocation(@PathVariable userName:String,
                       @PathVariable x: Double,
                       @PathVariable y: Double)
            = repo.updateLocation(userName, Point(x, y))
}

客户端是使用OpenLayers映射库开发的纯HTML + Javascript应用程序（有关更多详细信息，请参见index.html和map.js ），该应用程序对您进行了地理定位 ，并通过“服务器发送事件”来创建发送给/从其他用户接收/接收的地理信息。

最后但并非最不重要的一点是，感谢令人敬畏的Spring REST docs项目，对REST API进行了全面的测试和记录，有关更多详细信息，请参见MessageControllerTests和index.adoc 。

结论

开发该应用程序给我的主要印象是，它有趣，高效，并具有SQL API和Kotlin类型系统提供的高度灵活性和安全性以及null安全性 。最终的Spring Boot应用程序是一个18 MBytes独立的可执行jar，具有较低的内存消耗（该应用程序可以与-Xmx32m !!!）。使用Spring REST文档也很有趣，再次展示了Kotlin不错的Java互操作性。

我遇到的一些痛点（ 数组注释属性 ， Java 8 Stream支持 ， 完全可调用的参考支持 ）计划在Kotlin 1.1中修复。暴露的库还很年轻，需要成熟，但是从我的角度来看，它是有前途的，并且展示了如何将Kotlin用于构建类型安全的DSL API（ 该HTML类型安全的构建器也是一个很好的例子）。

并请记住，官方支持的Spring Data项目与Kotlin很好地配合使用，如我之前的博文中 spring-boot-kotlin-demo项目中所示。
