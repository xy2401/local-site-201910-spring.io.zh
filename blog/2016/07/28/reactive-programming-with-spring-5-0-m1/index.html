Spring 5.0 M1的反应式编程

工程
罗森·斯托扬切夫（Rossen Stoyanchev）
2016年7月28日

正如Juergen在他的Spring Framework 5 M1 发布公告中提到的那样，我们的Spring Reactive计划已被合并到Spring Framework中，以保留所有的贡献和超过一年的全部历史。

它是什么？

简而言之，反应式编程是关于无阻塞，事件驱动的应用程序，该应用程序以少量线程进行扩展，并带有背压作为关键成分，旨在确保生产者不会压倒消费者。Reactive Streams规范 （也在Java 9中采用）使能够跨不同提供商的层和库传达需求。例如，写给客户端的HTTP连接可以传达其可用性，以一直向上游写入数据存储库，从而从数据库中获取数据，从而在HTTP客户端速度慢的情况下，存储库也可能变慢甚至暂停。有关反应式编程的更广泛介绍，请参阅Dave Syer的多部分系列文章“关于反应式编程的说明” 。

从命令式逻辑切换到非阻塞式逻辑时的一个实际挑战是组合异步逻辑的能力，而又不会迷失在“回调地狱”中。我们需要的API的一个很好的例子是CompletionStage和Stream Java 8中的API。然而Stream是真正为Collections构建的，不适用于无限或延迟敏感的序列，例如我们在非阻塞I / O和事件驱动的应用程序中经常遇到的序列。Reactor及其即将发布的3.0 GA版本是Reactive Streams的实现，它扩展了Reactive Streams Publisher与Flux和Mono API类型提供了类似于Java 8 Stream的声明性成分API，但功能更广泛，与ReactiveX模式更可比。有关此内容的更多信息，请参见Sebastien Deleuze的“了解反应类型” 。

这箱子里有什么东西？

Spring Framework 5包含响应式流和Reactor，以用于其自身的响应式使用以及其许多核心API。 M1版本提供与JSON（Jackson）和XML（JAXB）之间的反应性序列化和反序列化，该反应性Web框架支持@Controller编程模型和反应式WebClient 。支持微服务，分散/收集，数据摄取等的输入和输出流方案变得容易。

下面是一个控制器，该控制器以完全非阻塞和反应性的方式从远程服务器获取并传输数据：

@GetMapping("/accounts/{id}/alerts")
public Flux<Alert> getAccountAlerts(@PathVariable Long id) {

  return this.repository.getAccount(id)
      .flatMap(account ->
          this.webClient
              .perform(get("/alerts/{key}", account.getKey()))
              .extract(bodyStream(Alert.class)));
}

什么是支持此的反应堆？Spring Web Reactive生活在新环境中spring-web-reactive现有的模块旁边（并很受欢迎！）驻留在spring-webmvc模块。这两个模块共享许多算法和机制，但实际上不能共享任何代码。这是因为Spring Web Reactive在Reactive Streams HTTP适配器层上运行，该适配器层在HTTP运行时一直是完全非阻塞和反应式的。因此，尽管Spring MVC是为Servlet容器构建并运行的，但Spring Web Reactive也可以在非Servlet运行时（例如Netty和Undertow）上运行

Spring Web反应式与Spring Web MVC

此时，Spring Framework 团队如何看待您可能想知道的这两个框架，我们建议您使用什么？首先，我们的目标是在理性范围内保持最大的一致性。两者之间没有什么根本不兼容的地方@Controller编程模型和反应方式。这是关于在支持该模型的情况下发生的一切的原因，因此在表面上除了完全支持诸如Flux和Mono和Observable和Single来自RxJava的输入和输出。

Spring Web Reactive是否比Spring MVC更好？Spring Framework反应式支持的最大价值主张以及我们独特的定位是，我们不会将现有的应用程序抛在后面。在Spring 5中，传统的Spring MVC可以在任何Servlet 3.1堆栈上运行，包括Java EE 7服务器。对于Spring Web Reactive，我们毫不妥协地支持Tomcat，Jetty，Undertow和Netty，并且还可以适应任何Servlet 3.1容器。我们计划在共享算法和机制方面继续Spring MVC和Spring Web Reactive之间的协同作用，以支持相同的编程模型。Spring MVC方面的改进请求或错误报告将使Spring Web Reactive受益，反之亦然。

这意味着开发人员可以根据自己的目的选择更好的选择。如果有人告诉你同步或阻塞是邪恶的，那就换种方式。并非如此，实际上这是一个权衡。命令式逻辑易于编写且易于调试。当然，它不能很好地扩展规模或效率，但这就是权衡的地方。在很多情况下，当务之急是必须满足当务之急，而在其他情况下则必须采用反应性和非阻塞性。在微服务场景中，您甚至可以为每个服务选择实现样式，所有这些都在相同的一致编程模型内进行。
