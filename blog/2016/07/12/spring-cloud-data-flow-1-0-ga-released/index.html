Spring Cloud Data Flow 1.0 GA发布

发布
马克·波拉克
2016年7月12日

我很高兴代表团队宣布Spring Cloud Data Flow的1.0 GA版本！

注意
 开始使用此新版本的一个好方法是遵循参考文档的“ 入门”部分 。它使用在计算机上运行的Data Flow服务器，并为每个应用程序部署一个新进程。

Spring Cloud Data Flow（SCDF）是现代运行时针对数据微服务的编排服务。SCDF允许您描述可以由长期流传输应用程序或短期任务应用程序组成的数据管道，然后将它们部署到您今天可能已经在使用的平台运行时中，例如Cloud Foundry，Apache YARN，Apache Mesos和Kubernetes。我们提供了广泛的流和任务应用程序，因此您可以立即开始开发针对用例的解决方案，例如数据提取，实时分析和数据导入/导出。

流是使用受Unix管道语法启发的DSL定义的。例如，将执行从http端点执行数据提取并写入Apache Cassandra数据库的流的DSL定义为http | cassandra 。相应地，该DSL的每个元素都映射到一个Spring Boot微服务应用程序，该应用程序专注于使用Spring Cloud Stream编程模型的数据处理。通过此编程模型，您可以专注于处理应用程序的输入和输出，而SCDF配置这些输出和输入如何映射到消息传递中间件上，这就是应用程序如何通信的方式。Spring Cloud Stream中的绑定器抽象支持多个消息代理。目前，RabbitMQ和Kafka可用于生产。消费者组和数据分区在Spring Cloud Stream中也受支持，可以在部署流时进行配置。

Unix的哲学非常酷，即“编写可以完成一件事情的程序并做好它。”，“编写程序以协同工作”和“编写程序以处理文本流，因为这是一个通用接口”。以及SCDF中的微服务架构和Spring Cloud Stream绑定程序。

今天我们也宣布发布

  • Spring Cloud Data Flow的Apache YARN Server 1.0 GA

  • Spring Cloud Data Flow的Kubernetes Server 1.0 GA

  • Spring Cloud Data Flow的Cloud Foundry Server 1.0 M4

对Apache Mesos的支持正在开发中。我们也很高兴看到社区对其他运行时的贡献，例如OpenShift 。您可以在我们的参考手册中找到有关SCDF架构的更多信息。

此版本中的重要功能包括：

  • 将数据管道描述为各个应用程序的有向图的Stream DSL 。

  • DSL对命名目的地的支持，使您可以使用流定义中任何“管道”中的事件。这被称为点击流 。您还可以合并多个流的输出 。

  • 部署清单可让您定义单个应用程序（CPU，磁盘，内存）的资源使用情况以及应用程序实例计数以及如何对数据进行分区 。部署时，您还可以传递任意应用程序属性 。

  • 支持将应用程序打包为Spring Boot uber-jar或Docker映像。

  • 支持将使用Spring Cloud Stream构建的数据微服务部署到可处理无限量数据的长寿命Stream应用程序，并将Spring Cloud任务部署到可处理有限数据集然后终止的应用程序。反过来，这些基于Spring Boot构建。

  • 具有Tab键补全功能的Shell应用程序，用于创建，部署和监视流和任务。

  • HTML5 仪表板 ，使您可以创建，部署和监视已部署的流和任务。

  • Flo for Spring Cloud Data Flow： 用于Stream定义的可视设计器 ，还支持可脚本转换处理器，该处理器接受Ruby，Groovy，Python或Javascript代码作为运行时计算逻辑。

  • 支持基本的HTTP和OAuth 2.0 身份验证 。

  • “ NoSql”风格的实时分析使用服务器上具有HTTP端点的字段值和聚合计数器来访问计数器值。计数器数据由Redis支持。

  • 使用Spring Initializr简化流应用程序的创建 。

  • Spring Cloud Stream应用程序支持RabbitMQ和Kafka 0.8

  • 任务和流应用程序启动器 ，可用于自定义我们提供的许多源，处理器，接收器和任务应用程序。

  • Spring Boot应用程序属性的白名单提供了Shell / UI信息，以显示一组首选的引导属性，以显示代码完成和应用程序信息。

Spring Cloud Data Flow从以前的项目Spring XD演变而来，开发了大约一年，该项目的目标相似，以简化流和批处理应用程序的开发。我们从这种经验中学到了很多，Sabby Anandan在这篇博客文章中对此进行了很好的描述。

一项重大的体系结构更改是用可插入的Deployer Service Provider接口替换了我们自己的应用程序运行时。尽管在Spring Cloud Data Flow 1.0 GA中花费了很多工程时间来进行这种体系结构转换，但我们现在处于非常有利的位置，可以继续在此基础上增加更高的价值，而不必花费时间开发核心运行时特征。以下是团队集体思想的一些想法：

  • 通过将流或任务应用程序的组件视作“仅是应用程序”，我们可以利用许多其他Spring Cloud项目（例如Spring Cloud Sleuth）来收集分布式应用程序中的响应时间。

  • 与Spinnaker集成以处理应用程序的连续部署/升级的责任，因为Spinnaker“以应用程序交易”为货币单位，并且可以使用诸如响应时间之类的数据来做出自动决策以升级到应用程序的新版本。

  • 多语言部署，我们希望部署的不仅仅是Java Spring Boot应用程序。由于许多数据科学团队都使用Python开发需要实时评估的模型，因此我们将首先考虑部署Python应用程序。

  • 从Spring XD重新带回Task DSL和UI Designer。

由于Spring Cloud Data Flow与Spring Cloud Stream和Spring Cloud Task的发布生命周期脱钩，因此随着这些项目发布新功能，SCDF可以立即使用它们。值得一提的Spring Cloud Stream令人兴奋的功能包括对Project Reactor和Kafka Streams API的支持，以及对Kafka 0.9，Google Cloud Pub / Sub，Azure Event Hub和JMS的绑定支持。对于Spring Cloud Task，计划支持Cloud Foundry上的最新Task功能。查看这两个项目的路线图以获取详细信息

有关功能，错误修复和改进的完整列表，请参阅封闭的1.0 RELEASE GitHub问题 。
