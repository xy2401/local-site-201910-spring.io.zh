How did we get here?

Spring Cloud Task, is a step in an evolution of data applications in the Spring portfolio. As we announce this release, it may be helpful to look at how we got to this point.

It began with Spring Batch

Built upon the Spring portfolio, the evolution of this type of finite workload really began with Spring Batch. One of the main workload specific extensions of the Spring framework, Spring Batch provided functionality for performing finite amounts of processing in an enterprise friendly way. With multiple processing models (Tasklet vs Chunk) as well as scalability options that allowed for optimizing data processing both within a single JVM and using multiple JVMs. Spring Batch has served as the standard bearer of finite processing on the JVM for many years.

Spring Cloud Task 1.0.1。现在可以发布

发布
迈克尔·米内拉
2016年7月11日

我们很高兴地宣布发布Spring Cloud Task 1.0.1。发布。这是建议用于Spring Cloud Task的GA版本，可从Maven Central以及其他通常位置获得。

我们是怎么来到这里的？

Spring Cloud Task是Spring产品组合中数据应用程序发展的一步。当我们宣布此版本时，看看我们如何达到这一点可能会有所帮助。

它从Spring Batch开始

建立在Spring产品组合的基础上，这种有限工作负载的演变实际上始于Spring Batch 。作为Spring Framework 主要针对工作负载的扩展之一，Spring Batch提供了以企业友好的方式执行有限数量的处理的功能。具有多种处理模式（ Tasklet vs Chunk）以及可扩展性选项，这些选项可优化单个JVM内和使用多个JVM的数据处理。多年来，Spring Batch一直是JVM上有限处理的标准载体。

介绍Spring XD

Spring XD是向Spring Cloud Task演进的下一步。Spring XD提供了在分布式环境中协调Spring Batch作业的功能。但是，Spring XD在提供用于批处理作业编排的强大工具的同时，为我们确定了对比工作抽象的工具的需求。在查看大数据世界（及以后）中的用例时，有许多不需要批处理作业的开销，因此将模型强制应用于流程是没有意义的。这让我们开始思考...

看一下现代运行时

将Spring XD重新配置为Spring Cloud Data Flow也为我们提供了一些见解。当查看诸如Cloud Foundry ， Mesos ， YARN和Kubernetes之类的现代应用程序运行时时 ，我们发现它们大多数为短命的应用程序提供了便利。但是，用户与他们进行交互，跟踪其结果等方式在运行时之间是不同的。在Cloud Foundry上， Diego提供了任务的概念，但其结果只是短暂的。在YARN上，可能会或可能不会配置作业历史记录服务器。作业完成后，Kubernetes会保持豆荚。

提供一种以独立于平台的方式构造有限数据微服务的统一方法，以及提供操作和功能特性，将非常有用……

这使我们进入了Spring Cloud Task

Spring Cloud Task诞生于这样的想法，即某些功能需要在企业环境中执行，而这些功能不需要长时间运行的应用程序。

spring-cloud-task提供以下功能：

  • 一个简单的编程模型 -Spring Cloud Task建立在Spring Boot之上，并为其提供附加功能。通过使用@EnableTask注释，您无需任何其他代码即可获得Spring Cloud Task的功能。
  • 任务存储库 -Spring Cloud Task提供了以独立于平台的方式监视和记录任务结果的功能。因此，无论您使用的是裸机，Cloud Foundry，Kubernetes，Mesos等，还是用于操作任务的方式都相同。
  • 扩展点 -Spring Cloud Task提供了使用侦听器扩展任务的功能。这使您可以在任务前后以及出现问题时执行逻辑。
  • 与Spring Batch集成-通过将Spring Batch作业作为Spring Cloud Task运行，可以在运行时通过Spring Cloud Data Flow对其进行动态管理。Spring Cloud Task可以让批处理作业作为独立运行的任务或通过Spring Cloud Data Flow编排的任务运行，而不是让任务在等待执行的容器中侦听，从而在环境中提供更大的弹性。Spring Cloud Task还提供了一个PartitionHandler允许启动工人作为分区作业的任务。
  • 与Spring Cloud Stream集成 -Spring XD提供了启动作业以及通过消息监视作业的功能。Spring Cloud Task为开发人员带来了相同的功能，而不受XD容器的影响。借助对类路径的正确依赖关系，Spring Cloud Task能够使任务和作业（作为任务运行）发出信息性消息，以及通过Spring Cloud Stream通过消息启动任务。

1.0.1中有什么新功能。发布？

1.0.0中提供了上述所有主要功能。RC1版本，可以在这里阅读。此版本中的不同之处在于，已对依赖项进行了重构，因此不再包括spring-cloud-task-core以及所需的任何其他任务模块，您现在可以仅包含spring-cloud-task-starter依赖性和其他想要获得预期功能的spring模块。

例如，如果开发人员想要编写作为任务运行的批处理作业，而不需要spring-cloud-task-core ， spring-cloud-task-batch和spring-boot-starter-batch在您的pom中，现在您只需要spring-cloud-task-starter和spring-boot-starter-batch 。批处理功能将自动为您引导（具有与所有Boot autoconfig一样的简单替代选项）。

从这里到哪里？

Spring Cloud Task现在是GA，我们很高兴收到您的来信。我们的短期重点是改善Spring Cloud Task与Spring Cloud Data Flow及其支持的运行时的互操作性。但是，我们期待您在下面的评论中，在StackOverflow和Twitter上在线的其他地方以及下个月的Spring One Platform上收到您的意见！
