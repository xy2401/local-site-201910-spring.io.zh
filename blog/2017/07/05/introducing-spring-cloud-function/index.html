介绍Spring Cloud功能

工程
马克·费舍尔
2017年7月5日

Spring Cloud Function是一个新项目，具有以下高级目标：

  • 通过功能促进业务逻辑的实现。
  • 将业务逻辑的开发生命周期与任何特定的运行时目标脱钩，以便相同的代码可以作为Web端点，流处理器或任务运行。
  • 支持跨无服务器提供程序的统一编程模型，以及独立运行（本地或在PaaS中）的能力。
  • 在无服务器提供程序上启用Spring Boot功能（自动配置，依赖项注入，指标）。

就像Spring一直提倡基于普通旧Java对象（POJO）的编程模型一样，Spring Cloud Function也提倡基于普通旧函数的编程模型。所谓“核心接口”，是指java.util.function包： Function ， Consumer和Supplier 。

这些类型的实现可以通过启用的类路径扫描显式或隐式地注册为Bean。 @FunctionScan 。参数和/或返回类型可以选择使用Reactor的Flux ，这是反应式流Publisher 。这实现了与其他Reactive Streams组件（甚至是基于其他实现的组件，例如RxJava 2）的互操作性，并且为该处理模型带来了诸如非阻塞IO和反压之类的响应功能（有关更多信息，请参阅Project Reactor ） 。每当参数和/或返回类型不是Flux ，Spring Cloud Function将它们包装起来，以便函数可以通过Flux 。对于简单的一次处理项目用例，您可以使其简单：

public class Greeter implements Function<String, String> {
  public String apply(String name) {
    return "Hello " + name;
  }
}

但是，如果您需要通过窗口化或归约运算来实现将数据集作为处理单位进行处理的功能，则可以使用Flux类型：

public static class WordCount
    implements Function<Flux<String>, Flux<Map<String, Integer>>> {
  public Flux<Map<String, Integer>> apply(Flux<String> phrases) {
    return phrases.window(3)
      .flatMap(f -> f.flatMap(phrase -> Flux.fromArray(phrase.split("\\W")))
      .reduce(new HashMap<String, Integer>(),
        (map, word) -> { map.merge(word, 1, Integer::sum); return map; }));
  }
}

依靠函数类型还可以轻松地编写函数，例如：

twistAndShout = twist.andThen(shout);

当然，也可以使用lambda定义函数，例如：

Function<String, String> shout =  s -> s.toUpperCase() + “!”;

实际上，Spring Cloud Function支持将基于字符串的lambda动态地编译到函数实例中。在原型设计或添加一些琐碎的转换逻辑时，这尤其有用，因为今天通常使用Spring Expression Language。

您可能会问，为什么Spring需要推广此模型，因为您可以轻松创建Function ， Consumer和Supplier无论如何还是实例。知道答案涉及控制反转并不奇怪。多年来，好莱坞原则描述了从基本依赖注入到Spring普遍使用模板模式的所有内容：“不要叫我们，我们会叫你”。的Flux上面提到的自适应实际上是控制反转的一个示例，但更重要的是将业务逻辑与部署配置文件分离。在这种情况下，业务逻辑是指功能，而部署概要文件可以是REST应用程序，流处理应用程序或有限任务。Spring Cloud Function为这些类型中的每种类型提供JAR，并且在每种情况下都使用自动配置的FunctionCatalog来定位Functions ， Consumers和Suppliers在ApplicationContext 。

例如部署Greeter以上作为REST端点只需要添加“弹簧云功能的web”依赖所示功能，如在此可以看出POM 。该插件还包括Spring Boot Maven插件，以便构建生成可执行的JAR：

./mvnw clean install
java -jar greeter/target/greeter-0.0.1-SNAPSHOT.jar

然后可以使用curl调用它：

$ curl -H "Content-Type: text/plain" :8080/greeter -d World
Hello World

同样，要将功能部署为流处理器，只需添加“ spring-cloud-function-stream”依赖关系，该依赖关系又将基于Spring Cloud Stream构建。正如Spring Cloud Stream提供的Binder抽象消除了定义通道适配器的需要一样，Spring Cloud Function也消除了声明诸如服务激活器，变形器甚至组件等组件的需要。 @StreamListener Spring Cloud Stream委派给的注释方法。“ spring-cloud-function-stream” JAR本身提供了所有这些。这是将控制反转提升到另一个层次的又一案例。

在本系列博客的第2部分中，我们将提供有关如何Suppliers ， Functions和Consumers可以在下一版本的Spring Cloud Data Flow中使用。基本思想是，只要需要提供一些自定义逻辑，就可以实现简单的功能。这是一个自以为是的模型的完美示例，在该模型中，您不仅不需要提供样板，而且框架无论如何都要更好。例如，您将能够仅注册功能-内联或打包为JAR（而不是Spring Cloud Stream应用程序），然后在依靠Spring Cloud Data Flow为您包装功能的同时引用DSL中的功能：

mySupplier | myFunction | myConsumer

部署配置文件甚至扩展到无服务器（又称“功能即服务”）提供程序的领域，例如AWS Lambda和Apache OpenWhisk（以及一旦提供Java支持的Azure函数和Google Cloud Functions）。在本博客系列的第3部分中，我们将深入探讨该主题，但现在您可以细读AWS Lambda适配器和Apache OpenWhisk适配器的文档。即将发布的博客还将介绍与基于Kubernetes的无服务器框架（例如Fission）的集成。

除了将业务逻辑和基础结构分离之外，各种部署配置文件JAR和FaaS适配器还促进了可移植性。开发人员可以完全隔离地实现功能，包括仅涉及输入和输出参数的单元测试。然后可以将该功能与相关性打包在一起，使其可以在目标环境中运行，从独立的REST应用到Spring Cloud Data Flow或FaaS提供程序。

这将我们带到了此入门博客的最后。“无服务器”一词引起了很大的反响，并且几乎总是伴随着这样的解释：“当然仍然有服务器，但是您不必考虑它们。”因此，尽管我们将拒绝引入“无框架”一词，但相同的概念确实可以应用于框架。在上面的Spring Cloud Data Flow示例中，函数开发人员无需考虑框架，甚至不需要生成依赖项中包含任何框架代码的构件。同样的想法适用于FaaS适配器。基本上，我们正在将“控制反转”推向可以将好莱坞原则转变为：“不依赖我们，我们依赖您”的地步。在好莱坞，这可能无法很好地解决问题，但是对于开发人员而言，这意味着您可以编写一个函数，将其打包在JAR中，然后注册该函数以与各种端点或适配器一起使用。一如既往，Spring遵循Alan Kay雄辩的原则：“简单的事情应该简单。复杂的事情应该是可能的。”在即将发布的博客文章中，我们将介绍一些由于Spring Cloud Function而可能实现的更复杂的事情，但是我们永远不会忘记保持简单的事情简单化。
