流更新和回滚

精心策划为一系列微服务应用程序的流数据管道一直是Spring Cloud Data Flow设计的核心价值。在1.3.0中。M3提供了独立更新源，处理器和接收器的能力，而无需取消部署和重新部署整个流。

流更新功能是通过将部署过程委派给名为Skipper的新Spring Cloud项目来实现的。在此博客中介绍的Spring Cloud Skipper是一台独立的服务器，可将Spring Boot应用程序部署到多个云平台。它还跟踪应用程序版本，应用程序属性以及已部署的一个或多个应用程序的部署属性，以便可以根据更新请求来计算对这些属性中的任何一个的更改。

当请求更新Stream中的应用程序时，仅更新需要更改的一个或多个应用程序。执行简单的红色/黑色更新，并且设计依赖于与新应用程序相关联的Spring Boot的运行状况端点。通过跟踪部署，Skipper还可以回滚到流的先前版本。

以下新的数据流命令显示了将日志接收器从版本1.1.0更新到版本1.2.0，然后回滚到1.1.0的基本生命周期。假定日志接收器的版本1.1.0和1.2.0均已部署为Maven工件。请注意，在对日志接收器进行的所有更改中，HTTP源应用程序始终保持部署状态。

数据流：>应用程序注册-名称http-类型源--uri maven：//org.springframework.cloud.stream.app：http-source-rabbit：1.2.0。释放数据流：>应用程序寄存器-名称日志-类型接收器-uri maven：//org.springframework.cloud.stream.app：log-sink-rabbit：1.1.0。RELEASE数据流：>流创建--name httptest-定义“ http --server.port = 9000 | log”数据流：>流跳过器部署--name httptest数据流：>流跳过器更新--name httptest --properties版本。日志= 1.2.0。释放数据流：>流跳过器回滚--name httptest

Spring Cloud Data Flow 1.3.0。M3发布

发布
马克·波拉克
2017年11月29日

我们很高兴宣布1.3.0。M3版本的Spring Cloud Data Flow及其相关项目生态系统。

本地服务器： 入门指南

发布要点

流更新和回滚

该文档的入门和带队长流部分详细介绍了整个过程。

JavaDSL

作为使用Shell的一种替代方法，新添加的JavaDSL提供了一种编程方法来使用流利的API创建，部署或启动流。现在，您不必通过将脚本文件传递到Shell来将流部署到生产环境中，而现在可以编写应用程序并在版本控制下连同其他代码工件一起对其进行管理。

JavaDSL提供两种样式来创建StreamDefinitions。

  • “定义”样式保留了使用管道的感觉，并在外壳中过滤了文本DSL。设置流名称后，使用定义方法选择此样式，例如

StreamDefinition streamDefinition = Stream.builder（dataFlowOperations）.name（“ ticktock”）.definition（“ time | log”）.create（）

  • “流利的”样式允许您通过传入StreamApplication的实例将源，处理器和接收器链接在一起，例如

StreamApplication timeSource =新的StreamApplication（“ time”）; StreamApplication logSink =新的StreamApplication（“ log”）; StreamDefinition streamDefinition = Stream.builder（dataFlowOperations）.name（“ ticktock”）.source（“ timeSource”）.sink（“ logSink”）;

尽管此方法乍一看似乎与定义样式没有明显不同，但它允许您通过在应用程序上下文中将它们声明为Spring Bean，从而在多个流中重用StreamApplication实例。

使用StreamDefinition实例，然后可以部署流。

流stream = streamDefinition.deploy（）;

使用Stream实例，您可以询问流的状态，取消部署或销毁该流。

参考指南的Java DSL部分提供了DSL的更完整介绍，Spring Data Flow Samples存储库提供了一个工作示例，供您开始在自己的项目中使用。

基于HTTP资源的应用解析

以前，任何注册为HTTP资源的应用程序都会下载一次，然后由Data Flow服务器缓存。从理论上讲，这可以优化部署时间，因为不会多次下载现有应用程序，但实际上，HTTP资源通常用于开发自定义流应用程序的早期阶段。在这种操作方式下，即使应用程序代码会更改，包含版本号的工件的文件名也不会更改。然后，看到新的代码运行，将需要您增加版本号，这在开发的早期阶段是不直观的。

为了适应这种用例，数据流服务器现在总是下载基于HTTP的资源。为确保磁盘空间不会未经检查而增长，LRU缓存用于删除下载的资源。此功能还允许您通过HTTP资源引用从Maven存储库下载SNAPSHOT命名的工件，并始终获取最新代码。总之，这进一步简化了开发经验。特别是在积极迭代业务逻辑时。

仪表板/浮动

我们很高兴地宣布，此版本为SCDF的仪表板带来了功能奇偶性。较旧的UI堆栈支持的所有功能都已移植并升级到Angular 4.0堆栈。流，任务，作业，分析，运行时和应用程序选项卡功能齐全。用于流和组合任务的Spring Flo设计器在操作和功能上均已完成。出于好奇，我们还以NPM软件包的形式发布了Spring Flo的核心库 。

版本兼容性

该版本带来了Spring Boot 1.5.8的兼容性，并且底层Spring Cloud基础架构已更新为Edgware。发布。有关更多详细信息，请查看1.3.0。M3发行说明。

展望未来，我们的目标是在几周内发布1.3.0 RC1，然后在2018年1月底发布通用版本。
