Spring Security 5.0.0。RC1发布

发布
罗伯·温奇
2017年11月1日

我很高兴代表社区宣布Spring Security 5.0.0的发布。RC1。此版本解决了150多个问题。以下是此版本的重点内容：

  • WebFlux安全亮点

    • ReactiveSecurityContextHolder

    • 添加了响应式CSRF支持

  • OAuth 2.0亮点

    • OAuth 2.0客户端支持

    • OAuth2AuthorizedClient /服务

    • OAuth 2.0支持的下一步是什么？

  • 核心亮点

    • 密码存储已更新

    • 添加UnboundId LDAP内存支持

    • 允许对ObjectIdentity.getIdentifier（）使用非数字（例如UUID）值

ReactiveSecurityContextHolder

以前，Spring Security使用ServerWebExchange.getPrincipal()作为被认证者的真实来源。经过身份验证的用户被复制到Reactor的上下文中以支持方法安全性，该方法使用Reactor上下文作为事实来源。有多种事实来源显然不是理想的。

Spring Security现在使用Reactor的Context作为验证谁的真实来源。仍然可以从以下位置访问用户ServerWebExchange.getPrincipal() ，但该值也来自Reactor的Context。

您可以读写SecurityContext使用Reactor的上下文ReactiveSecurityContextHolder 。例如，这演示了如何检索当前登录用户的消息。

Authentication authentication =
    new TestingAuthenticationToken("user", "password", "ROLE_USER");

Mono<String> messageByUsername = ReactiveSecurityContextHolder.getContext()
  .map(SecurityContext::getAuthentication)
  .map(Authentication::getName)
  .flatMap(this::findMessageByUsername)
  // In a WebFlux application the `subscriberContext`
  // is automatically setup using `ReactorContextWebFilter`
  .subscriberContext(ReactiveSecurityContextHolder.withAuthentication(authentication));

StepVerifier.create(messageByUsername)
	.expectNext("Hi user")
	.verifyComplete();

与this::findMessageByUsername定义为：

Mono<String> findMessageByUsername(String username) {
	return Mono.just("Hi " + username);
}

OAuth 2.0客户端支持

已经进行了许多更新和完善，以完善新的OAuth 2.0登录功能。我们很高兴在接下来的几周内将其发布给Spring社区。

除了更新之外，下面还添加了一些新功能：

  • 用于OAuth 2.0登录的Spring Boot 2.0自动配置

  • ID令牌验证

  • 使用以下措施防止对UserInfo Response进行令牌替换攻击sub索赔验证

  • 通过的隐式赠款支持ImplicitGrantConfigurer

OAuth2AuthorizedClient /服务

的OAuth2AuthorizedClient代表授权客户 。当最终用户（资源所有者）已向客户端授予访问其受保护资源的权限时，该客户端被视为“授权” 。此类的目的是将一个OAuth2AccessToken到ClientRegistration （客户）和资源所有者 ，即Principal授予授权的最终用户。

的主要作用OAuth2AuthorizedClientService是要管理OAuth2AuthorizedClient实例。从开发人员的角度来看，它提供了查找OAuth2AccessToken与客户端关联，以便可以用来发起对资源服务器的请求。

的OAuth2AuthorizedClientService可以注册为@Bean在里面ApplicationContext （尽管不是必需的），以便开发人员可以查找OAuth2AccessToken与客户相关联。

例如：

@Controller
public class GoogleCalendarController {

	@Autowired
	private OAuth2AuthorizedClientService authorizedClientService;

	@RequestMapping("/calendar")
	public String calendar(OAuth2AuthenticationToken authentication) {
		OAuth2AuthorizedClient authorizedClient =
			this.authorizedClientService.loadAuthorizedClient(
				"google", authentication.getName());

		OAuth2AccessToken accessToken = authorizedClient.getAccessToken();

		// ...

		return "calendar";
	}
}

OAuth 2.0支持的下一步是什么？

HandlerMethodArgumentResolver（s）

我们将提供一个HandlerMethodArgumentResolver对于OAuth2AuthorizedClient和OAuth2AccessToken 。

替代使用OAuth2AuthorizedClientService直接，您将能够解决OAuth2AuthorizedClient要么OAuth2AccessToken作为一个论点@Controller方法参数。

资源服务器支持

很快我们将开始计划功能日志以提供对OAuth 2.0资源服务器角色的支持，因此请继续关注。

密码存储已更新

密码存储已进行了大修，以提供更安全的默认设置以及迁移密码存储方式的能力。默认值PasswordEncoder现在是DelegatingPasswordEncoder ，这是一个非被动的更改。此更改可确保默认情况下现在使用BCrypt对密码进行编码，允许以旧格式验证密码，并允许将来升级密码存储。

构造DelegatingPasswordEncoder

您可以使用以下方法轻松构造实例PasswordEncoderFactories 。

PasswordEncoder passwordEncoder =
    PasswordEncoderFactories.createDelegatingPasswordEncoder();

或者，您可以创建自己的自定义实例。例如：

String idForEncode = "bcrypt";
Map encoders = new HashMap<>();
encoders.put(idForEncode, new BCryptPasswordEncoder());
encoders.put("noop", NoOpPasswordEncoder.getInstance());
encoders.put("pbkdf2", new Pbkdf2PasswordEncoder());
encoders.put("scrypt", new SCryptPasswordEncoder());
encoders.put("sha256", new StandardPasswordEncoder());

PasswordEncoder passwordEncoder =
    new DelegatingPasswordEncoder(idForEncode, encoders);

{noop}password (2)
{pbkdf2}5d923b44a6d129f3ddf3e3c8d29412723dcbde72445e8ef6bf3b508fbf17fa4ed4d6b99ca763d8dc (3)

密码编码

的idForEncode传入构造函数确定PasswordEncoder将用于编码密码。在里面DelegatingPasswordEncoder我们在上面构建，这意味着编码的结果password将被委托给BCryptPasswordEncoder并以{bcrypt} 。最终结果如下所示：

{bcrypt}$2a$10$dXJ3SW6G7P50lGmMkkmwe.20cQQubK3.HZWzG3YB1tlRy.fqvM/BG

密码匹配

匹配是基于id以及id到PasswordEncoder在构造函数中提供。我们的“ 密码存储格式”示例提供了一个有效的示例。默认情况下，调用结果matches(CharSequence, String)带有密码和id未映射（包括空ID）将导致IllegalArgumentException 。可以使用以下方式自定义行为DelegatingPasswordEncoder.setDefaultPasswordEncoderForMatches(PasswordEncoder) 。

通过使用id我们可以匹配任何密码编码，但是使用最现代的密码编码对密码进行编码。这很重要，因为与加密不同，密码哈希被设计为没有简单的方法来恢复明文。由于无法恢复明文，因此很难迁移密码。虽然用户迁移很简单NoOpPasswordEncoder ，我们选择默认情况下将其包括在内，以简化入门经验。

迁移到Spring Security 5

如果您没有使用显式PasswordEncoder还是依靠旧的核心PasswordEncoder您将需要迁移。

迁移现有密码

如果您的密码以明文形式存储，则升级哈希与获取明文密码并对其进行编码一样容易。

String encoded = passwordEncoder.encode(plainTextPassword);

如果您的密码以其他格式存储，则无法更新哈希。要迁移这些密码，您必须确定密码存储的算法，并为所有密码加上前缀{id} 。例如，如果密码用sha256 ：

97cde38028ad898ebc02e690819fa220e88c62e0699403e94fff291cfffaf8410849f27605abcbc0

确保所有密码都带有前缀{sha256}喜欢：

{sha256}97cde38028ad898ebc02e690819fa220e88c62e0699403e94fff291cfffaf8410849f27605abcbc0

虽然此密码不是以安全格式存储的，但它确实允许其他密码以安全格式存储。我们还可以要求用户更改密码，以更新存储在其中的哈希值。

对于精明的读者而言，很明显，您也可以通过在纯文本密码前添加前缀来迁移纯文本密码{noop} 。例如，密码为

password

您可以简单地在密码前面加上{noop}喜欢：

{noop}password

这将起作用，但是它不是安全的，因此不建议在生产环境中使用。

PasswordEncoder和SaltSource

如果您使用的是旧的而已弃用的PasswordEncoder从本质上讲，此内容已被删除，因为它要求用户提供盐并使用SaltSource （也已删除）。每一个PasswordEncoder核心实现已通过Javadoc中的迁移说明迁移到了新的crypto API。一个示例是MessageDigestPasswordEncoder 。

恢复为以前的行为（不安全）

在不安全的情况下，用户可以通过提供以下操作来恢复以前的行为： NoOpPasswordEncoder作为一个@Bean 。如果应用程序利用AuthenticationManagerBuilder的NoOpPasswordEncoder必须明确提供给AuthenticationManagerBuilder 。例如，如果您有：

auth
	.inMemoryAuthentication()
		.withUser("user").password("password").roles("USER");

您可以使用以下方法恢复为以前的行为：

auth
	.inMemoryAuthentication()
		.passwordEncoder(NoOpPasswordEncoder.getInstance())
		.withUser("user").password("password").roles("USER");

入门经验

如果您要编写演示或样本，那么花一些时间来哈希用户密码会比较麻烦。有一些便利机制可以简化此过程，但这仍然不适合生产。

User user = User.withDefaultPasswordEncoder()
  .username("user")
  .password("password")
  .roles("user")
  .build();
System.out.println(user.getPassword());
// {bcrypt}$2a$10$dXJ3SW6G7P50lGmMkkmwe.20cQQubK3.HZWzG3YB1tlRy.fqvM/BG

如果要创建多个用户，则也可以重复使用buider。

UserBuilder users = User.withDefaultPasswordEncoder();
User user = users
  .username("user")
  .password("password")
  .roles("USER")
  .build();
User admin = users
  .username("admin")
  .password("password")
  .roles("USER","ADMIN")
  .build();

这会散列存储的密码，但是密码仍会在内存和已编译的源代码中公开。因此，对于生产环境它仍然不被认为是安全的。对于生产，您应该在外部对密码进行哈希处理。

自动密码迁移（未来）

用DelegatingPasswordEncoder用户验证后，我们将能够更新密码格式。我们可以将默认方法添加到PasswordEncoder返回一个类型（即PasswordMatch ），说明密码是否匹配。如果密码匹配并且使用旧格式， PasswordMatch也会有一个成员，该成员的密码采用最新编码。当Spring Security看到有关新格式的建议时，它可以使用API更新用户密码的格式。
