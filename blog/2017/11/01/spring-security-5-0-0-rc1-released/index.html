Spring Security 5.0.0。RC1发布

发布
罗伯·温奇
2017年11月1日

我很高兴代表社区宣布Spring Security 5.0.0的发布。RC1。此版本解决了150多个问题。以下是此版本的重点内容：

  • WebFlux安全亮点

    • ReactiveSecurityContextHolder

    • 添加了响应式CSRF支持

  • OAuth 2.0亮点

    • OAuth 2.0客户端支持

    • OAuth2AuthorizedClient /服务

    • OAuth 2.0支持的下一步是什么？

  • 核心亮点

    • 密码存储已更新

    • 添加UnboundId LDAP内存支持

    • 允许对ObjectIdentity.getIdentifier（）使用非数字（例如UUID）值

ReactiveSecurityContextHolder

以前，Spring Security使用ServerWebExchange.getPrincipal()作为经过身份验证的真相的来源。经过身份验证的用户被复制到Reactor的上下文中以支持方法安全性，该方法使用Reactor上下文作为事实来源。有多种事实来源显然不是理想的。

Spring Security现在使用Reactor的Context作为验证谁的真实来源。仍然可以从ServerWebExchange.getPrincipal()访问该用户，但是该值也来自Reactor的Context。

您可以使用ReactiveSecurityContextHolder读写SecurityContext到Reactor的上下文。例如，这演示了如何检索当前登录用户的消息。

Authentication authentication =
    new TestingAuthenticationToken("user", "password", "ROLE_USER");

Mono messageByUsername = ReactiveSecurityContextHolder.getContext()
  .map(SecurityContext::getAuthentication)
  .map(Authentication::getName)
  .flatMap(this::findMessageByUsername)
  // In a WebFlux application the `subscriberContext`
  // is automatically setup using `ReactorContextWebFilter`
  .subscriberContext(ReactiveSecurityContextHolder.withAuthentication(authentication));

StepVerifier.create(messageByUsername)
	.expectNext("Hi user")
	.verifyComplete();

将this::findMessageByUsername定义为：

Mono findMessageByUsername(String username) {
	return Mono.just("Hi " + username);
}

OAuth 2.0客户端支持

已经进行了许多更新和完善，以完善新的OAuth 2.0登录功能。我们很高兴在接下来的几周内将其发布给Spring社区。

除了更新之外，下面还添加了一些新功能：

  • 用于OAuth 2.0登录的Spring Boot 2.0自动配置

  • ID令牌验证

  • 使用sub声明验证来防止对UserInfo Response的令牌替换攻击

  • 通过ImplicitGrantConfigurer支持ImplicitGrantConfigurer 式授予

OAuth2AuthorizedClient /服务

OAuth2AuthorizedClient代表授权客户 。当最终用户（资源所有者）已向客户端授予访问其受保护资源的权限时，该客户端被视为“授权” 。这个类提供一个相关联的目的OAuth2AccessToken到ClientRegistration （客户端）和资源所有者 ，谁是Principal终端用户表示授予的授权。

OAuth2AuthorizedClientService的主要作用是管理OAuth2AuthorizedClient实例。从开发人员的角度来看，它提供了查找与客户端关联的OAuth2AccessToken的功能，以便可以将其用于发起对资源服务器的请求。

OAuth2AuthorizedClientService可以在ApplicationContext注册为@Bean （尽管不是必需的），以便开发人员可以查找与客户端关联的OAuth2AccessToken 。

例如：

@Controller
public class GoogleCalendarController {

	@Autowired
	private OAuth2AuthorizedClientService authorizedClientService;

	@RequestMapping("/calendar")
	public String calendar(OAuth2AuthenticationToken authentication) {
		OAuth2AuthorizedClient authorizedClient =
			this.authorizedClientService.loadAuthorizedClient(
				"google", authentication.getName());

		OAuth2AccessToken accessToken = authorizedClient.getAccessToken();

		// ...

		return "calendar";
	}
}

OAuth 2.0支持的下一步是什么？

HandlerMethodArgumentResolver（s）

我们将为OAuth2AuthorizedClient和OAuth2AccessToken提供HandlerMethodArgumentResolver的实现。

作为替代使用OAuth2AuthorizedClientService直接，你就能够解决的OAuth2AuthorizedClient或OAuth2AccessToken作为参数传递给@Controller方法参数。

资源服务器支持

不久我们将开始计划功能日志以提供对OAuth 2.0资源服务器角色的支持，因此请继续关注。

密码存储已更新

密码存储已进行了大修，以提供更安全的默认设置以及迁移密码存储方式的能力。默认的PasswordEncoder现在是DelegatingPasswordEncoder ，这是一个非被动的更改。此更改可确保默认情况下现在使用BCrypt对密码进行编码，允许以旧格式验证密码，并允许将来升级密码存储。

构造DelegatingPasswordEncoder

您可以使用PasswordEncoderFactories轻松构建实例。

PasswordEncoder passwordEncoder =
    PasswordEncoderFactories.createDelegatingPasswordEncoder();

或者，您可以创建自己的自定义实例。例如：

String idForEncode = "bcrypt";
Map encoders = new HashMap<>();
encoders.put(idForEncode, new BCryptPasswordEncoder());
encoders.put("noop", NoOpPasswordEncoder.getInstance());
encoders.put("pbkdf2", new Pbkdf2PasswordEncoder());
encoders.put("scrypt", new SCryptPasswordEncoder());
encoders.put("sha256", new StandardPasswordEncoder());

PasswordEncoder passwordEncoder =
    new DelegatingPasswordEncoder(idForEncode, encoders);

密码的一般格式为：

{noop}password (2)
{pbkdf2}5d923b44a6d129f3ddf3e3c8d29412723dcbde72445e8ef6bf3b508fbf17fa4ed4d6b99ca763d8dc (3)

密码编码

传递给构造函数的idForEncode确定将使用哪个PasswordEncoder编码密码。在上面我们构造的DelegatingPasswordEncoder ，这意味着编码password的结果将委托给BCryptPasswordEncoder并以{bcrypt} BCryptPasswordEncoder 。最终结果如下所示：

{bcrypt}$2a$10$dXJ3SW6G7P50lGmMkkmwe.20cQQubK3.HZWzG3YB1tlRy.fqvM/BG

密码匹配

匹配是基于id和id到构造函数中提供的PasswordEncoder的映射完成的。我们的“ 密码存储格式”示例提供了一个有效的示例。默认情况下，使用密码和未映射的id （包括null ID）调用matches(CharSequence, String)结果将导致IllegalArgumentException 。可以使用DelegatingPasswordEncoder.setDefaultPasswordEncoderForMatches(PasswordEncoder)定义此行为。

通过使用id我们可以匹配任何密码编码，但是使用最现代的密码编码对密码进行编码。这很重要，因为与加密不同，密码哈希被设计为没有简单的方法来恢复明文。由于无法恢复明文，因此很难迁移密码。尽管用户迁移NoOpPasswordEncoder ，但我们默认选择包含它，以使入门体验更简单。

迁移到Spring Security 5

如果您没有使用显式的PasswordEncoder或依靠旧的核心PasswordEncoder ，则需要迁移。

迁移现有密码

如果您的密码以明文形式存储，则升级哈希与获取明文密码并对其进行编码一样容易。

String encoded = passwordEncoder.encode(plainTextPassword);

如果您的密码以其他格式存储，则无法更新哈希。要迁移这些密码，必须确定密码存储的算法，并在所有密码前加上{id} 。例如，如果密码用sha256散列：

97cde38028ad898ebc02e690819fa220e88c62e0699403e94fff291cfffaf8410849f27605abcbc0

确保所有密码都以{sha256}为前缀，例如：

{sha256}97cde38028ad898ebc02e690819fa220e88c62e0699403e94fff291cfffaf8410849f27605abcbc0

虽然此密码不是以安全格式存储的，但它确实允许其他密码以安全格式存储。我们还可以要求用户更改密码，以更新存储在其中的哈希值。

对于精明的读者来说，很显然，您也可以通过在纯文本密码前加上{noop}前缀来迁移纯文本密码。例如，密码为

password

您可以使用{noop}作为密码的前缀，例如：

{noop}password

这将起作用，但是它不是安全的，因此不建议在生产环境中使用。

PasswordEncoder和SaltSource

如果您在核心中使用旧的且已弃用的PasswordEncoder ，则已将其删除，因为它要求用户提供盐并使用SaltSource （也已删除）。核心中的每个PasswordEncoder实现都已使用Javadoc中的迁移说明迁移到了新的crypto API。一个示例是MessageDigestPasswordEncoder 。

恢复为以前的行为（不安全）

尽管不安全，但用户可以通过提供NoOpPasswordEncoder作为@Bean来恢复以前的行为。如果应用程序利用AuthenticationManagerBuilder ， NoOpPasswordEncoder必须将NoOpPasswordEncoder明确提供给AuthenticationManagerBuilder 。例如，如果您有：

auth
	.inMemoryAuthentication()
		.withUser("user").password("password").roles("USER");

您可以使用以下方法恢复为以前的行为：

auth
	.inMemoryAuthentication()
		.passwordEncoder(NoOpPasswordEncoder.getInstance())
		.withUser("user").password("password").roles("USER");

入门经验

如果您要编写演示或样本，那么花一些时间来哈希用户密码会比较麻烦。有一些便利机制可以简化此过程，但这仍然不适合生产。

User user = User.withDefaultPasswordEncoder()
  .username("user")
  .password("password")
  .roles("user")
  .build();
System.out.println(user.getPassword());
// {bcrypt}$2a$10$dXJ3SW6G7P50lGmMkkmwe.20cQQubK3.HZWzG3YB1tlRy.fqvM/BG

如果要创建多个用户，则也可以重复使用buider。

UserBuilder users = User.withDefaultPasswordEncoder();
User user = users
  .username("user")
  .password("password")
  .roles("USER")
  .build();
User admin = users
  .username("admin")
  .password("password")
  .roles("USER","ADMIN")
  .build();

这会散列存储的密码，但是密码仍在内存和已编译的源代码中公开。因此，对于生产环境它仍然不被认为是安全的。对于生产，您应该在外部对密码进行哈希处理。

自动密码迁移（未来）

使用DelegatingPasswordEncoder我们将能够在用户验证身份后更新密码格式。我们可以向PasswordEncoder添加一个默认方法，该方法返回一个类型（即PasswordMatch ），该类型说明密码是否匹配。当密码匹配并且使用旧格式时， PasswordMatch也将具有一个成员，该成员具有最新的密码编码。当Spring Security看到有关新格式的建议时，它可以使用API更新用户密码的格式。
