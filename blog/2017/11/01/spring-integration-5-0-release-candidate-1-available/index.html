Spring Integration 5.0 Release Candidate 1可用

发布
阿尔特姆·比兰（Artem Bilan）
2017年11月1日

我很荣幸地代表Spring Integration小组宣布Spring Integration 5.0版本的Release Candidate 1（ 5.0.0.RC1 ） 现在可用。

可以从Milestone资料库下载：

repositories {
    maven { url 'http://repo.spring.io/libs-milestone' }
}

compile "org.springframework.integration:spring-integration-core:5.0.0.RC1"

此版本包含20个JIRA （和一些GitHub问题），以及错误修复和许多新功能。该功能的一些重点RC1 ，因为先前宣布的里程碑7 ：

  • 现在，由Java DSL解析器填充的组件已注册为BeanDefinitions在应用程序上下文中，感谢Spring框架中的新引入5.0Supplier基于程序的Bean注册。这种方法有助于我们避免一些用于单例注册和初始化的样板代码。另外这个BeanDefinition在某些用例中，注册可能对选择应用程序上下文中的特定组件很有用。实际上， Spring Cloud Function项目中使用的正是java.util.function.*豆扫描。

  • 的IntegrationFlows.from(Class serviceInterface)现在已经过载了版本beanName论点。这恰好成为覆盖网关的生成网关代理的Bean名称。 [flowId].gateway值：

@Bean
public IntegrationFlow uppercaseFlow() {
return IntegrationFlows.from(MessageFunction.class, "uppercase")
        .transform(String::toUpperCase)
        .get();
}

这在上面提到的Spring Cloud Function中再次有用，当我们通过其bean名称将该函数作为微服务的入口点公开时。另外，通过手动注册的集成流程，这也很方便IntegrationFlowContext ，当我们为“ IntegrationFlow 。

  • HTTP入站端点（包括WebFlux）现在可以正确处理ResponseEntity在回复消息中：

@Bean
public WebFluxInboundEndpoint errorInboundEndpoint() {
    WebFluxInboundEndpoint endpoint = new WebFluxInboundEndpoint();
    RequestMapping requestMapping = new RequestMapping();
    requestMapping.setPathPatterns("/error");
    endpoint.setRequestMapping(requestMapping);
    endpoint.setRequestChannelName("errorServiceChannel");
    return endpoint;
}

@ServiceActivator(inputChannel = "errorServiceChannel")
public ResponseEntity processHttpRequest() {
    return new ResponseEntity<>("<500 Internal Server Error,{}>",
                                       HttpStatus.INTERNAL_SERVER_ERROR);
}

这样，我们就可以将Spring Web的Spring Integration Channel Adapters与Spring MVC和Spring WebFlux中的现有行为对齐。

  • 除了升级到最新的Spring Web Services 3.0和Smack 4.2.1 ，我们已经修复了Spring Integration Web Services网关中DOM POX消息的处理。

  • 现在一些领导者发起者可能会发出OnFailedToAcquireMutexEvents ，当他们无法获取共享资源的锁时。它经常发生，因此请检查LockRegistryLeaderInitiator#setPublishFailedEvents()想要查询更多的信息。

  • 的JdbcChannelMessageStore现在可以提供自定义MessageGroupPreparedStatementSetter对于某些非标准消息存储结构或与默认Java序列化行为不同的消息。

  • 的ClientStompEncoder已经引入以正确填充SEND我们使用时的STOMP框架StompSubProtocolHandler从客户端。

特别感谢一些社区成员对框架的持续积极贡献；扫描提交并给他们一些荣誉！

有关更改的完整列表5.0 ，另请参阅参考手册中的“新增功能”一章。

我们预计将在几周内发布GA-赶在春季启动之前2.0 GA ！
