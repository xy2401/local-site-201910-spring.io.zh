Spring Framework 5 Kotlin API的功能方式

工程
塞巴斯蒂安·德勒兹（SébastienDeleuze）
2017年8月1日

更新：另请参见Spring Fu实验项目 。

自从我们最初的公告以来（社区非常欢迎！）作为Spring Framework 5中Kotlin官方支持的一部分 ，我们将继续努力，以配合最近对Spring WebFlux的改进而进一步增强Kotlin支持。

为了演示这些功能以及如何将它们一起使用，我创建了一个新的具有spring-kotlin功能的演示应用程序，该应用程序是在Kotlin中开发的独立的Spring WebFlux应用程序，具有Mustache模板渲染，JSON REST webservices和Server-发送事件流功能。请不要犹豫，在预期于9月发布Spring Framework 5之前，向我们发送反馈和建议。

程序引导

Spring WebFlux和Reactor Netty允许对应用程序进行编程引导，因为它们本来就是设计为作为嵌入式Web服务器运行的。开发Spring Boot应用程序时显然不需要这样做，但是对于在微服务架构或其他受限环境中使用自定义引导进行紧密部署的单元而言，这将非常有用。

class Application {
	
  private val httpHandler: HttpHandler
  private val server: HttpServer
  private var nettyContext: BlockingNettyContext? = null
  
  constructor(port: Int = 8080) {
    val context = GenericApplicationContext().apply {
        beans().initialize(this)
        refresh()
    }
    server = HttpServer.create(port)
    httpHandler = WebHttpHandlerBuilder.applicationContext(context).build()
  }

  fun start() {
    nettyContext = server.start(ReactorHttpHandlerAdapter(httpHandler))
  }
	
  fun startAndAwait() {
    server.startAndAwait(ReactorHttpHandlerAdapter(httpHandler),
        { nettyContext = it })
  }
	
  fun stop() {
    nettyContext?.shutdown()
  }
}

fun main(args: Array) {
  Application().startAndAwait()
}

Spring的新Kotlin DSL的功能Bean定义

Spring Framework 5引入了一种使用lambdas注册bean的新方法。它非常高效，不需要任何反射或CGLIB代理（因此Reactive应用程序不需要kotlin-spring插件），并且非常适合Java 8或Kotlin之类的语言。您可以在此处对Java和Kotlin语法进行概述。

在spring-kotlin-functional中 ，bean在包含bean定义的Beans.kt文件中声明。DSL在概念上声明了Consumer通过一个干净的声明性API，该API允许您处理配置文件和Environment以自定义如何注册Bean。该DSL还允许通过if表达式， for循环或任何其他Kotlin构造对bean进行自定义注册逻辑。

beans {
  bean()
  bean()
  bean("webHandler") {
    RouterFunctions.toWebHandler(
      ref().router(),
      HandlerStrategies.builder().viewResolver(ref()).build()
    )
  }
  bean("messageSource") {
    ReloadableResourceBundleMessageSource().apply {
      setBasename("messages")
      setDefaultEncoding("UTF-8")
    }
  }
  bean {
    val prefix = "classpath:/templates/"
    val suffix = ".mustache"
    val loader = MustacheResourceTemplateLoader(prefix, suffix)
    MustacheViewResolver(Mustache.compiler().withLoader(loader)).apply {
      setPrefix(prefix)
      setSuffix(suffix)
    }
  }
  profile("foo") {
    bean()
  }
}

在此示例中， bean()通过构造函数使用自动装配，而ref()是applicationContext.getBean(Routes::class.java)的快捷方式。

Spring和Reactor API的零安全

Kotlin的主要功能之一是null安全性 ，它允许在编译时处理null值，而不是在运行时遇到著名的NullPointerException 。这通过干净的可空性声明，表达“值或无值”的语义使您的应用程序更安全，而无需像Optional那样支付包装费用。（Kotlin允许使用具有可为空值的函数构造；请查阅有关Kotlin空安全性的全面指南 。）

尽管Java不允许在其类型系统中表示null安全性，但我们通过对工具友好的注释为Spring API引入了一些null安全性 ：包级别的@NonNullApiannotation声明非null是默认行为，并且我们在特定参数或返回值可以为null地方显式放置@Nullableannotation。我们已经为整个Spring Framework API完成了这项工作（是的，这是一项巨大的努力！），其他项目，例如Spring Data，也开始利用它。Spring注释使用JSR 305元注释（处于休眠状态的JSR，但受IDEA，Eclipse，Findbugs等工具支持）进行元注释，以向Java开发人员提供有用的警告。

在Kotlin方面，杀手级功能是-从Kotlin 1.1.51版本开始-Kotlin 可以识别这些注释，以便为整个Spring API提供null安全性。这意味着在使用Spring 5和Kotlin时，您的代码中永远不应包含NullPointerException ，因为编译器将不允许这样做。您需要使用-Xjsr305=strict编译器标志，以在Kotlin类型系统中考虑这些注释。

用于Spring WebFlux的Kotlin DSL的功能路由

而不是使用@RestController和@RequestMapping @RestController ， spring-kotlin-functional通过专用的Kotlin DSL使用WebFlux功能API。

router {
  accept(TEXT_HTML).nest {
    GET("/") { ok().render("index") }
    GET("/sse") { ok().render("sse") }
    GET("/users", userHandler::findAllView)
  }
  "/api".nest {
    accept(APPLICATION_JSON).nest {
      GET("/users", userHandler::findAll)
    }
    accept(TEXT_EVENT_STREAM).nest {
      GET("/users", userHandler::stream)
    }		
  }
  resources("/**", ClassPathResource("static/"))
}

像bean DSL一样，功能性路由DSL允许基于自定义逻辑和动态数据以编程方式注册路由（对于开发CMS或电子商务解决方案非常有用，其中大多数路由都依赖于通过后台创建的数据）。

路由通常指向负责通过可调用引用基于HTTP请求创建HTTP响应的处理程序。这是UserHandler ，它利用Spring Framework 5直接在Spring JAR中提供的Kotlin扩展来避免使用Kotlin的类型化参数避免众所周知的类型擦除问题。Java中的相同代码将需要其他Class或ParameterizedTypeReference参数。

class UserHandler {
	
  private val users = Flux.just(
      User("Foo", "Foo", LocalDate.now().minusDays(1)),
      User("Bar", "Bar", LocalDate.now().minusDays(10)),
      User("Baz", "Baz", LocalDate.now().minusDays(100)))
	
  private val userStream = Flux
      .zip(Flux.interval(ofMillis(100)), users.repeat())
      .map { it.t2 }

  fun findAll(req: ServerRequest) =
      ok().body(users)

  fun findAllView(req: ServerRequest) =
      ok().render("users", mapOf("users" to users.map { it.toDto() }))
	
  fun stream(req: ServerRequest) =
      ok().bodyToServerSentEvents(userStream)
	
}

注意，使用Spring WebFlux和服务器端模板渲染（此应用程序中的髭须）创建“服务器发送事件”端点非常容易。

使用WebClient，Reactor Test和JUnit 5轻松进行测试

Kotlin允许在反引号之间指定有意义的测试函数名称，从JUnit 5.0 RC2开始，Kotlin测试类可以使用@TestInstance(TestInstance.Lifecycle.PER_CLASS)来启用测试类的单个实例化，从而允许在非静态方法上使用@BeforeAll和@AfterAllannotation，这非常适合Kotlin。现在，由于具有带有junit.jupiter.testinstance.lifecycle.default = per_class属性的junit-platform.properties文件，还可以将默认行为更改为PER_CLASS 。

class IntegrationTests {
	
  val application = Application(8181)
  val client = WebClient.create("http://localhost:8181")
	
  @BeforeAll
  fun beforeAll() {
    application.start()
  }
	
  @Test
  fun `Find all users on JSON REST endpoint`() {
    client.get().uri("/api/users")
        .accept(APPLICATION_JSON)
        .retrieve()
        .bodyToFlux()
        .test()
        .expectNextMatches { it.firstName == "Foo" }
        .expectNextMatches { it.firstName == "Bar" }
        .expectNextMatches { it.firstName == "Baz" }
        .verifyComplete()
  }

  @Test
  fun `Find all users on HTML page`() {
    client.get().uri("/users")
        .accept(TEXT_HTML)
        .retrieve()
        .bodyToMono()
        .test()
        .expectNextMatches { it.contains("Foo") }
        .verifyComplete()
  }

  @Test
  fun `Receive a stream of users via Server-Sent-Events`() {
    client.get().uri("/api/users")
        .accept(TEXT_EVENT_STREAM)
        .retrieve()
        .bodyToFlux()
        .test()
        .expectNextMatches { it.firstName == "Foo" }
        .expectNextMatches { it.firstName == "Bar" }
        .expectNextMatches { it.firstName == "Baz" }
        .expectNextMatches { it.firstName == "Foo" }
        .expectNextMatches { it.firstName == "Bar" }
        .expectNextMatches { it.firstName == "Baz" }
        .thenCancel()
        .verify()
  }
	
  @AfterAll
  fun afterAll() {
    application.stop()
  }
}
