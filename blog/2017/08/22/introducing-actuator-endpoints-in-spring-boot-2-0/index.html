在Spring Boot 2.0中介绍执行器端点

工程
斯特凡·尼科尔（StéphaneNicoll）
2017年8月22日

Spring Boot 2给Actuator带来了重要的变化，我很高兴代表团队为您简要介绍即将发生的事情2.0.0.M4 。

处理主要的新版本使我们有机会重新审视并改进一些公共合同。我们很快感觉到端点基础结构就是其中之一：当前，Actuator中的Web端点仅受Spring MVC支持（不支持JAX-RS）。同样，创建一个新的端点来暴露一些操作需要编写很多样板：您需要编写一个主要端点，即Spring MVC扩展（作为@RestController ），JMX MBean和必要的自动配置。从Spring Boot 2开始，对“反应式”执行器的支持已成为显而易见的要求，也带来了一些新的挑战。

端点基础架构

Spring Boot 2带来了全新的端点基础结构，该基础结构使您能够以独立于技术的方式定义一个或多个操作，并支持Spring MVC，Spring WebFlux和Jersey！Spring Boot 2将提供对Jersey的本地支持，并且只要可以通过编程方式注册资源的方式，就可以轻松地为另一个JAX-RS实现编写适配器。

让我们用现有的新API来说明/application/loggers MVC端点 ，使您可以在运行时控制日志记录配置。该端点具有三个操作：

  • 列出当前配置的主要操作。

  • 更狭窄的操作以按名称提供记录器的配置

  • 用于更新特定记录器配置的写操作。

从Spring Boot 2开始，端点就是这样的：

@Endpoint(id = "loggers")
public class LoggersEndpoint {

    @ReadOperation
    public Map<String, Object> loggers() { ... }

    @ReadOperation
    public LoggerLevels loggerLevels(@Selector String name) { ... }

    @WriteOperation
    public void configureLogLevel(@Selector String name, LogLevel configuredLevel) { ... }

}

新的@Endpoint注释将这种类型声明为具有强制性唯一ID的终结点。正如我们稍后将看到的那样，将自动推断出一系列属性。不需要其他代码即可在以下位置公开此终结点/applications/loggers或作为org.springframework.boot:type=Endpoint,name=Loggers JMX MBean。让我们找出原因。

Web端点

该端点公开了三个操作：

  • GET上/application/loggers ：所有记录器的配置（因为它没有“ selector”参数）：

  • GET上/application/loggers/{name} ：命名记录器的配置（使用name @Selector ）。

  • POST上/application/loggers/{name} ：更新命名记录器的配置。此操作有一个附加参数需要解决，因为它不用于进一步选择要更新的资源。在这种情况下，带有configuredLevel JSON属性是预期的，类似于：

{
    "configuredLevel": "WARN"
}

读取操作的其他参数将从请求属性中自动映射。

JMX MBean

不需要其他代码即可将该端点公开为JMX MBean。因为典型的JMX客户端（例如JConsole ）无法访问第三方库，任何非核心类型都会自动翻译。特别是， LogLevel是在字符串中作为字符串公开的枚举configureLogLevel JMX操作。

跨领域功能

基础架构使我们能够提供诸如端点缓存之类的跨领域功能。目前，我们提供了一种缓存主操作结果的方法（即loggers上面的方法）。端点安全性也在进行中。

扩展名

如果您需要以特定于技术的方式表达某些内容，则可以为其编写扩展名。一个示例是运行状况终结点，该终结点需要根据计算得出的值来更改响应的HTTP状态代码Health 。的health端点几乎是您所期望的 ：

@Endpoint(id = "health")
public class HealthEndpoint {

    @ReadOperation
    public Health health() { ... }

}

的HealthWebEndpointExtension 覆盖主读取操作以产生一个WebEndpointResponse比较起来Health 。这使Web扩展程序有机会为响应提供Web特定的属性：

@WebEndpointExtension(endpoint = HealthEndpoint.class)
public class HealthWebEndpointExtension {

    private final HealthEndpoint delegate;

    public HealthWebEndpointExtension(HealthEndpoint delegate) { ... }

    @ReadOperation
    public WebEndpointResponse<Health> getHealth() {
        Health health = this.delegate.health();
        Integer status = // get http status based on current health
        return new WebEndpointResponse<>(health, status);
    }

}

组态

每个端点都会在端点处自动获得一个专用的配置名称空间。 。的spring-boot-configuration-processor已更新为自动检测@Endpoint类：

要配置端点，实际上所需要做的就是将其公开为@Bean 。有一个新的@ConditionalOnEnabledEndpoint确保不会根据当前配置创建（或公开）端点：

@Bean
@ConditionalOnBean(LoggingSystem.class)
@ConditionalOnMissingBean
@ConditionalOnEnabledEndpoint
public LoggersEndpoint loggersEndpoint(LoggingSystem loggingSystem) {
	return new LoggersEndpoint(loggingSystem);
}

因此，例如，如果endpoints.loggers.enabled=false存在于环境中，条件将不匹配，并且端点将完全不暴露。

下一步是什么？

因为我们现在有了集中式的基础架构，所以根据运行时环境，我们可以变得更聪明。如果您要通过WebFlux运行Actuator，则不一定要假设一切都是非阻塞的。实际上，不会返回Publisher （即Mono要么Flux ）将在专用线程池上运行，这样它们就不会阻止响应式操作。这项工作应该在下周初登陆大师。

我们还正在为执行器制定一些无功特定合同，即无功Health可以执行可用的端点HealthIndicators以反应方式：

public interface ReactiveHealthIndicator {

    Mono<Health> health();

}

我们也正在考虑从R / W模型迁移到CRUD模型（即支持删除和创建操作），如果您有兴趣，请观看＃10023 。

如果您想尝试使用这种新的端点基础结构，我们将很高兴收到您的来信。首先，使用Spring Boot在start.spring.io上生成一个应用程序2.0.0.BUILD-SNAPSHOT 。
