Spring Cloud Data Flow 1.3.0。M1发布

发布
冈纳·希尔特（Gunnar Hillert）
2017年8月7日

我很高兴代表团队宣布Spring Cloud Data Flow 1.3.0的第一个里程碑版本。

本地服务器： 入门指南 。

可用性是我们团队关注的重点之一，在1.3版本中，我们旨在通过升级的客户端技术堆栈，改进的工具和文档来实现仪表板的现代化。重点是提高可读性，用户体验和整体性能。

角4

Spring Cloud Data Flow的第一个里程碑版本包含新的基于Angular 4的Dashboard UI的初始版本。旧的基于AngularJS 1.x的UI，特别是工具和辅助框架/库选择，开始显示出它们的年龄。

因此，我们决定对整个堆栈进行现代化改造，并利用Angular 4提供的功能。此外，我们正在使仪表板的样式与Pivotal UI保持一致 。

主要的改进是TypeScript的使用 ，它是JavaScript的超集。与Angular 4结合使用，TypeScript非常适合Spring开发人员，并减少了服务器端代码和客户端代码之间的摩擦。

熟悉Spring（Web）的开发人员将欣赏TypeScript的类型支持，对类和接口的支持，以及Angular 4对TypeScript 装饰器 （Think注释）的使用以及基于构造函数的依赖注入。

从根本上讲，我们使用Angular CLI来构建和管理项目，从而极大地减少了基础架构样板代码。在这方面，Angular CLI将Spring Boot注入了您的客户端开发过程中。

更好的文档

作为UI现代化过程的一部分，我们还希望提供更好的文档。在M2版本中，我们应该使用JSDoc拥有近100％的文档覆盖率。然后，我们使用Compodoc作为分析和创建最终文档的工具。最终，每个CI构建都将生成并将源代码文档推送到：

http://cloud.spring.io/spring-cloud-dataflow-ui/

测试范围

遵循其他Spring Cloud项目的脚步，我们使用Codecov跟踪Spring Cloud Data Flow UI的单元测试范围。可以在此处找到代码覆盖率报告的链接。Spring Cloud Data Flow UI项目的README提供了一个图标，该图标显示当前覆盖率以及指向当前覆盖率报告的链接。

这项工作仍在进行中，因此预计M2版本会覆盖率急剧增加。此外，我们计划在下一个里程碑版本中纳入全面的E2E测试，并确保Dashboard在所有现代浏览器上均可使用。

国家管理

新仪表板的主要重点是正确使用适当的域模型类。在旧的UI中，我们基本上会将返回的JSON数据直接传递到UI模板代码中，使其难以维护正确的状态。有了定义明确的域模型和有状态服务，我们现在可以对应用程序的状态进行更细粒度的控制，例如，从分页列表过渡到详细信息页面并返回时。

快取清除

在旧版Dashboard中，我们依赖RequireJS来提供模块支持。但是，这破坏了AngularJS 1.x的生产捆绑，从而导致缓存无效化问题。现在，我们通过Angular CLI（下面的Webpack ）使用了适当的生产捆绑，这将缓存清除散列添加到生成的文件中。

怎么了

尽管我们已经完成了“ about”，“ apps”，“ runtime”，“ stream”和“ task”选项卡的重构和移植，但我们非常希望获得反馈。随意打开GitHub问题，其中包含您的发现，错误和功能要求。在即将发布的里程碑版本中将提供“流和组合任务设计器”，“工作”和“分析”选项卡。

是否曾经想编辑流，对流/任务数据管道进行版本控制，升级或回滚流/任务数据管道中的应用程序？正在积极开发一种新的灵活模型来补充Spring Cloud Data Flow中的连续交付故事。请继续关注并期待下一个里程碑版本！

反馈很重要。请在StackOverflow和GitHub上与我们联系，以获取问题和功能要求。我们也欢迎您的贡献！我们对改善Spring Cloud Data Flow生态系统的任何帮助表示赞赏。
