$ diff -q spring-data-gemfire spring-data-geode

工程
约翰·布鲁姆
2017年10月26日

问候春季社区和Pivotal GemFire / Apache Geode用户-

我经常被问到的一个问题是：“ Spring Data GemFire和Spring Data Geode有什么区别？”

既然Spring Data Geode是Spring Data Release Train的一部分 ，从Kay开始（请参阅官方发布公告以获取更多详细信息），现在终于可以在一个开放的论坛上回答这个问题了。

为了帮助回答这个问题，我为Spring Data GemFire和Spring Data Geode （现在统称为SDG ^ 2）设置了两个简单的目标：

  1. 首先，也是最重要的是，允许用户无缝地将Spring Data Geode与Spring Data GemFire互换，反之亦然。

  2. 接下来，帮助用户将基于Spring Boot ，Pivotal GemFire或Apache Geode的应用程序从本地开发环境迁移到托管环境（例如Pivotal CloudFoundry） ，而几乎不需要更改任何代码。

让我们更仔细地看一下每个目标及其含义。

可互换的依赖

第一个目标就是选择 。

这只是意味着用户可以从spring-data-geode至spring-data-gemfire并且希望不必在使用Pivotal GemFire或Apache Geode的Spring Boot应用程序中更改一行代码。

只需从此更改依赖项声明 ...

马文
<dependency>
  <groupId>org.springframework.data</groupId>
   <artifactId>spring-data-geode</artifactId>
   <version>2.0.0.RELEASE</version>
</dependency>
摇篮

compile 'org.springframework.data:spring-data-geode:2.0.0.RELEASE'

为此...

马文
<dependency>
  <groupId>org.springframework.data</groupId>
   <artifactId>spring-data-gemfire</artifactId>
   <version>2.0.0.RELEASE</version>
</dependency>
摇篮

compile 'org.springframework.data:spring-data-gemfire:2.0.0.RELEASE'

并且，您完成了！

甚至可以从spring-data-gemfire回到spring-data-geode再次，一切都正常。整个Pivotal GemFire代码库及其所有功能都在Apache Geode的开源版本中。

在功能和行为上，它们是相同的。唯一可区别的是，您获得了一组不同的传递依赖项，即GemFire与Geode。尝试与其他IMDG一起说，尤其是在OS产品和企业级解决方案之间。

包装和类别名称

可是等等！包和类名呢？ 它们在Pivotal GemFire和Apache Geode中没有区别吗？ 您是否会更改包和类的名称 ，例如org.springframework.data.gemfire.GemfireTemplate在Spring Data GemFire中org.springframework.data.geode.GeodeTemplate在Spring Data Geode中 ？

没有;绝对不是！

作为示例，请考虑以下GemFire / Geode接口， org.apache.geode.cache.GemFireCache 。

在Pivotal GemFire 9.0及更高版本中，界面为org.apache.geode.cache.GemFireCache ， 使用org.apache.geode包名称空间。在Apache Geode中，相同的接口是org.apache.geode.cache.GemFireCache仍使用“ GemFire”名称的位置。其他类和接口也是如此。

除了迁移原因之外，这对于互操作性而言实际上非常重要，尤其是在GemFire / Geode的分发层中，专有消息在客户端和服务器之间以及集群中的对等成员之间发送。

版本号

Spring Data GemFire 2.0（ Kay ） 基于 Pivotal GemFire 9.1.1，后者本身基于Apache Geode 1.2.0。

Spring Data Geode 2.0（ Kay ） 基于 Apache Geode 1.2.1。

如此有效，SDG ^ 2库可以互换，而无需更改应用程序代码。

现在， Spring Data Geode正式成为Spring Data Release Train的一部分 ， Spring Data Geode和Spring Data GemFire都在版本号上保持一致，并且从此以后它们的版本号将保持不变。

小费
 我们建议您使用Spring Data BOM文件，即org.springframework.data:spring-data-releasetrain:Kay-RELEASE （如Spring Boot在其依赖项声明和版本中所做的那样），或从Spring Boot启动器父POM继承 。或者，您可以使用Spring IO Platform ，该平台提供了经过精心设计和协调的一组依赖项，这些依赖项已经过测试，可以一起使用。

从本地环境到托管环境

第二个目标是通过利用Cloud Native软件设计模式并以Spring方式采用云就绪解决方案（例如Pivotal GemFire）来帮助用户走向云。

早在2003年，我的一位同事曾对我说：“ 思考全球，打造本地 ”。他当时很讽刺，但是我刚加入公司，正在从我的IDE询问如何在本地构建，运行和测试我们的企业Java应用程序。

当时，我们的团队正忙于集中所有开发活动。这部分是由于设置开发人员的本地开发环境，在本地安装IBM WebSphere变得非常复杂，因为我们的应用程序必须在WebSphere上运行。

好吧，事实证明，他的评论实际上是对的！

至关重要的是，运营合同不会损害开发，尤其是在开发过程中，因为它们可能随时都会改变。拥有CI / CD的主要原因之一是弥合这一差距。

在开发过程中，开发人员需要完全控制其环境以实现应用程序的要求。能够使用轻量级进程构建，调试，测试，调试和分析应用程序对于敏捷性至关重要。随着从创建到生产的整个过程，CI / CD对于及时响应变化至关重要。但是，任何事情都不应妨碍开发人员交付有效代码的能力，而这始于IDE内部。

从一开始， Spring就一直与开发人员的生产力有关，为开发人员提供正确的框架和工具，以快速，可靠且高质量地解决任何企业应用程序问题。而现在，考虑到诸如Spring Boot和Spring Cloud之类的技术 ，路径已经很清楚了，这全都是关于“ Cloud Native ”的。这不只是常识性的开发，而是明智的开发。

因此，目标2的主要目的是采用相同的概念，并分别使用Spring Data GemFire或适用于Pivotal GemFire的Spring Data Geode或Apache Geode将其应用于数据。通过使用SDG的新的基于注释的配置方法以及用户的Spring Boot应用程序中对Pivotal GemFire / Apache Geode的自动配置支持 ，用户是否在本地使用Apache Geode还是在PCC / SSC （由Pivotal GemFire支持）中都没关系在PCF的全球范围内，用户几乎不需要更改代码就能获得一致的体验。

这意味着更少的惊喜，而更多地关注实际问题，从而为最终用户带来价值。毕竟，这是框架或工具的重点，它可以简化过程并处理样板，而这些样板在必要时对最终用户而言不是必需的。

摘要

那么区别是什么呢？

绝对没有！至少，即使在不同情况下，实践中也不应存在明显差异。

我希望这篇文章可以帮助人们从根本上理解和理解本文试图实现的范围，它可以扩展到Spring Data GemFire和Spring Data Geode的功能，行为和表面上的差异之外。它还包括以最简单，最简单的方式从开始到交付，在生产中一直具有一致的经验。

与往常一样，无论是通过在JIRA中提交票证，提交PR还是在StackOverflow中简单地提出问题，反馈都将受到赞赏。

重要
 确保今年进入SpringOne平台 。计划中的内容太多了，需要学习的新东西，尤其是在新的Reactive Spring上。这将是一个古老的事件。
