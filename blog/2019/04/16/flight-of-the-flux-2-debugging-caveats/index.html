在反应世界中进行调试

从命令式，阻塞性模式切换到反应性，非阻塞性模式带来了好处，但同时也带来了一些警告。其中之一就是调试经验。这是为什么？

主要是因为您已经学会了依靠良好的旧stacktrace ，但是突然之间，由于反应式编程的异步方面，此无价之宝的价值变得远远不如之前。但是，这并不特定于反应式编程：一旦引入异步代码，便会在程序中计划调度的代码与异步执行的代码之间创建边界。

用香草异步代码演示问题

让我们举一个例子ExecutorService和一个Future （此处没有Reactor代码）：

	private static void imperative() throws ExecutionException, InterruptedException {
		final ScheduledExecutorService executor =
				Executors.newSingleThreadScheduledExecutor();

		int seconds = LocalTime.now().getSecond();
		List<Integer> source;
		if (seconds % 2 == 0) {
			source = IntStream.range(1, 11).boxed().collect(Collectors.toList());
		}
		else if (seconds % 3 == 0) {
			source = IntStream.range(0, 4).boxed().collect(Collectors.toList());
		}
		else {
			source = Arrays.asList(1, 2, 3, 4);
		}

		executor.submit(() -> source.get(5))  //line 76
		        .get();
	}

如果我们寻找与上述代码等效的Reactor，我们可以提出以下内容：

		int seconds = LocalTime.now().getSecond();
		Mono<Integer> source;
		if (seconds % 2 == 0) {
			source = Flux.range(1, 10)
			             .elementAt(5);
		}

			             .elementAt(5);

		else {
			source = Flux.just(1, 2, 3, 4)

		source.subscribeOn(Schedulers.parallel())

如前所述，这是因为在汇编和订阅之间，代码中还有一个附加的“边界”。跟踪仅返回到订阅点（此处是block() ）:-(

因此，在异步环境中，将堆栈跟踪用于分析和调试目的更加困难，在Reactor中甚至更难（因为它是异步的，并且具有汇编与订阅的默认延迟方法）。但是幸运的是，库中有一些工具可以尝试缓解这一情况。

使事情变得更好

回到经典：log

请记住，当您将命令性代码与print陈述？它可能不像启动步进调试器那样酷，但是有时它是您需要的快速而肮脏的解决方案。

在Reactor中，您拥有log()操作员：

  • 它记录反应流信号： onNext ， onComplete ， onError （ 甚至 onSubscribe ， cancel和request ！）
  • 您可以将其调整为仅将这些信号的一部分列入白名单
  • 您可以选择一个特定的Logger以及

简而言之， log是一种快速而肮脏的解决方案，可让您轻松概览一下序列中正在发生的事情。在开发过程中自由使用它，并且可以为每个名称指定一个“名称” log呼吁区别对待。

使用log(String)可以转移到提示导致错误的来源：

使用调试模式丰富堆栈跟踪

Reactor中可用的另一种方法是尝试获取运行时堆栈跟踪中的程序集信息。

这可以通过以下方式激活所谓的“调试模式”来完成： Hooks类：

Hooks.onOperatorDebug();

它有什么作用？它使每个运算符实例化（又称程序集）捕获堆栈跟踪并保留以备后用。

如果onError到达一个运算符，它将将该程序集堆栈跟踪附加到onError的Throwable （作为抑制Exception ）。结果，当您看到stacktrace时，您将获得运行时和程序集的更完整的图片。

启用调试模式后，在我们之前的示例中，我们将能够看到采用了哪个汇编路径以及实际处理了哪个源：

	private static void hook() {
		Hooks.onOperatorDebug();
		try {
			int seconds = LocalTime.now().getSecond();
			Mono<Integer> source;
			if (seconds % 2 == 0) {
				source = Flux.range(1, 10)
				             .elementAt(5); //line 149
			}
			else if (seconds % 3 == 0) {
				source = Flux.range(0, 4)
				             .elementAt(5); //line 153
			}
			else {
				source = Flux.just(1, 2, 3, 4)
				             .elementAt(5); //line 157
			}

			source.block(); //line 160
		}
		finally {
			Hooks.resetOnOperatorDebug();
		}
	}

降低成本checkpoint

使用的一个缺点Hooks.onOperatorDebug()是它对应用程序中使用的每个运算符进行汇编stacktrace捕获。填充单个stacktrace是一项昂贵的操作，因此，不用说这会对性能产生重大影响。因此，仅在开发环境中建议这样做。

幸运的是，如果您确定代码库中容易产生这种源歧义的部分，则可以降低成本。

通过使用checkpoint()操作员可以仅在代码库中的特定点激活程序集跟踪捕获。如果使用以下命令给检查点一个唯一且有意义的名称，您甚至可以完全不用填充堆栈跟踪信息checkpoint(String) ：

	private static void checkpoint() {
		int seconds = LocalTime.now().getSecond();
		Mono<Integer> source;
		if (seconds % 2 == 0) {
			source = Flux.range(1, 10)
			             .elementAt(5)
			             .checkpoint("source range(1,10)");
		}
		else if (seconds % 3 == 0) {
			source = Flux.range(0, 4)
			             .elementAt(5)
			             .checkpoint("source range(0,4)");
		}
		else {
			source = Flux.just(1, 2, 3, 4)
			             .elementAt(5)
			             .checkpoint("source just(1,2,3,4)");
		}

		source.block(); //line 186
	}
