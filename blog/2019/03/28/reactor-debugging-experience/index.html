反应堆调试经验

工程
谢尔盖·埃格罗夫（Sergei Egorov）
2019年3月28日

在Project Reactor团队中，我们认为您所依赖的库的调试经验与功能集或性能一样重要。

今天，我们很高兴宣布Reactor系列中的两个新实验项目！

BlockHound-一个新孩子

新手最常见的错误之一是阻止应该只运行非阻塞代码的Java线程（例如， Schedulers.parallel() ）。
这是最有害的问题之一，因为您可能阻止不相关的处理，甚至造成死锁！

考虑以下代码：

Flux.range(0, Runtime.getRuntime().availableProcessors() * 2)
        .subscribeOn(Schedulers.parallel())
        .map(i -> {
            CountDownLatch latch = new CountDownLatch(1);

            Mono.delay(Duration.ofMillis(i * 100))
                .subscribe(it -> latch.countDown());

            try {
                latch.await();
            } catch (InterruptedException e) {
                throw new RuntimeException(e);
            }

            return i;
        })
        .blockLast();

运行此代码需要多长时间？1秒？10秒？
如果我告诉您它将永远不会退出并造成死锁怎么办？原因如下：

  1. 我们创造N * 2信号，在哪里N是我们的JVM可以使用的CPU数量。
  2. 我们订阅使用Schedulers.parallel -仅限于N线程。
  3. 对于每个信号，我们为并行调度程序调度另一个任务（ Mono.delay隐式使用Schedulers.parallel ）。
  4. 我们的逻辑假设延迟将很快得到处理，并且锁存器最终将解除阻塞。
  5. 但是，所有N个线程将等待它们的闩锁，并且延迟任务将永远不会执行！

即使您没有阻塞所有线程而仅阻塞某些线程，也阻止了其他不相关的任务的进行。最可能的结果是性能将下降。

当您将旧的阻塞代码迁移到反应性方法时，该问题尤其明显。当您的函数具有相同的颜色时，即使是最有经验的代码审查员也可能不会发现阻塞调用！

这就是为什么我们创建BlockHound的原因-Java代理可以检测来自非阻塞线程的阻塞调用。
与其他解决方案不同，它会检测原始方法（甚至是本机方法！）而且即使反射也无法调用阻塞方法！

现在，如果我们按照docs中的说明将其添加到我们的应用中 ，我们将得到以下异常：

java.lang.Error: Blocking call! sun.misc.Unsafe#park

  at reactor.BlockHound$Builder.lambda$install$8(BlockHound.java:254)

注意await来电Unsafe#park在内部等待逻辑。我们不希望我们的线程被停放或阻塞，而BlockHound可以保护我们免受此伤害！

如果您想了解实施细节，请阅读其工作原理页面。
TL; DR：它包装原始方法，并仅向它们添加两个方法调用。

您可以在测试中或在质量检查/过渡环境中运行它，而不会损失性能。Holy Atom，鉴于开销很低，您甚至可以尝试在生产环境中运行它！ :)

BlockHound可与Project Reactor和RxJava 2一起使用，您可以编写自己的集成 。

Reactor Debug Agent-生产就绪的组件回溯

由于功能性编程方面的原因，调试反应性代码有时可能具有挑战性：您无需声明要对数据执行什么操作，而是声明数据应如何在系统中流动。这意味着声明和执行在不同的时间发生。

您可以在Simon的精彩文章中阅读有关它的更多信息：
https://spring.io/blog/2019/03/06/flight-of-the-flux-1-assembly-vs-subscription

在Reactor中，我们将其称为“组装时间”和“执行时间”。在组装期间，您可以通过调用“设计”管道myFlux.map(i -> i * 2).filter(5 % i == 1).single()和其他运营商。稍后，此“管道定义”将用于处理由myFlux 。但是，发生错误时会发生什么？

你们中有些人可能已经知道Hooks.onOperatorDebug() 。这是一个非常有用的挂钩reactor-core 。它从此转换堆栈跟踪：

如您所见，启用调试模式后，我们可以清楚地识别发生错误的组装操作。它就像一个堆栈跟踪，但是（因为执行与程序集分离）是一个反向跟踪。

您可能会想：“太好了，现在我要在生产中使用它！”-我们也是如此。但是当你使用Hooks.onOperatorDebug() ，我们必须在组装时进行重量级堆栈遍历，才能在每次致电运营商时捕获呼叫站点，例如.map(...) ，即使您的代码永远不会抛出错误！这是由于Java中缺少呼叫站点跟踪，因此唯一的替代方法是new Exception().getStackTrace()要么StackWalker （在Java 9+中）。

显然，我们不能在生产中使用这种方法，因此我们为此做了一个工具！

ReactorDebugAgent来自Reactor-Tools项目的Java代理可以帮助您调试应用程序中的异常而无需支付运行时成本（与Hooks.onOperatorDebug() ）。

⚠️该项目处于孵化阶段，将来可能会或可能不会成为独立项目或https://github.com/reactor/reactor-core的模块。

它通过字节码转换来转换链，例如：

Flux.range(0, 5)
       .single()

至：

Flux flux = Flux.range(0, 5);
flux = Hooks.addCallSiteInfo(flux, "Flux.range\n foo.Bar.baz(Bar.java:21)"));
flux = flux.single();
flux = Hooks.addCallSiteInfo(flux, "Flux.single\n foo.Bar.baz(Bar.java:22)"));

要启用它，您需要首先初始化代理：

ReactorDebugAgent.init();

Since️由于实现将在加载类时对您的类进行检测，因此放置它的最佳位置是在您的所有其他内容之前main(String[])方法：

public static void main(String[] args) {
    ReactorDebugAgent.init();
    SpringApplication.run(Application.class, args);
}
