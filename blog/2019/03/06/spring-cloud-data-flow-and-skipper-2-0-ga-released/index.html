Spring Cloud Data Flow和Skipper 2.0 GA发布

发布
马克·波拉克
三月06，2019

Spring Cloud Data Flow团队很高兴宣布发布2.0数据流。遵循在本地 ， Cloud Foundry和Kubernetes上运行的入门指南 。

手拉手是2.0发布Spring Cloud Skipper 。如果要与数据流分开使用Skipper，则参考指南中的“ 入门”部分是最好的起点。

这是数据流的重点

  • 流部署始终委托给Skipper

  • 在所有受支持的平台上运行的单个服务器

  • 针对多个平台启动任务

  • 界面改进

  • 标准化OAuth2和OpenID Connect以获得安全性

  • 改进指标并监视已部署的应用程序

  • 使用千分尺更新分析

  • 数据库迁移支持

  • 更新到Boot 2.1

  • 更新内部组件以使用JPA

  • 任务/作业执行和性能改进

流部署始终委托给Skipper

1.x系列Data Flow Server允许Streams由Data Flow Server直接部署或委托给Skipper。这导致了两种操作模式，“经典”和“船长”。现在，只有一种通过Skipper部署流的选项，它可以部署到所有受支持的平台，并且还为长期流应用程序提供滚动升级和降级功能。从结构上讲，任务是像以前一样由数据流服务器部署的，但是现在可以跨不同的平台部署。有关该功能的更多信息，请参见下文。

在所有受支持的平台上运行的单个服务器

我们已经将Local，Cloud Foundry和Kubernetes服务器整合为一台服务器，无需根据要运行的平台选择不同的jar或docker镜像。每个平台的入门步骤基本相同。一项重大更改是配置任务的方式，请参见下文，因为我们现在支持跨不同平台启动任务的功能。与以前的版本一样，可以将长期存在的应用程序部署在不同的平台上。

针对多个平台启动任务

启动任务后，您可以指定执行任务的平台。以前，此功能仅可用于流和应用程序部署。通过Data Flow服务器，您可以配置多个Kubernetes和Cloud Foundry任务平台。当您要针对多个平台协调数据管道，但要从一个中央位置管理所有数据管道时，此功能特别有用。

例如，如果您要在Cloud Foundry上运行数据流org1/space1 ，您可以在中启动任务org2/space2或Kubernetes集群的名称空间中。同样，如果您正在Kubernetes上运行Data Flow namespace ，您可以在中启动任务namespace2或成org1/space1在Cloud Foundry上。如果在本地运行Data Flow Server，则可以指定不同的本地任务平台（例如具有不同的JVM属性），以替代使用任务部署属性。该文档描述了如何为Local ， Cloud Foundry和Kubernetes配置多个任务平台

用户界面改进

  • UI支持针对后端平台集合启动任务。

  • UI / UX的构建具有流和任务启动的一致体验。

  • 基于另一个重要的功能集，现在可以根据数据流中定义的精细角色自动启用/禁用仪表板功能。

  • Grafana启动图标本地集成在仪表板的“流”和“运行时”页面中。借助针对可用的Micrometer后端配置的Data Flow，只需单击一下鼠标，即可监视流管道指标，例如消息速率，错误计数和延迟。

  • 现在支持一个新按钮以回滚到流的先前版本。

  • 分析标签已被删除，并已替换为连接千分尺的Grafana仪表板的链接。

  • Angular已升级到7.2.4

  • 根据用户反馈，在作业的执行页面内添加了作业重新启动按钮。

  • UI和数据流服务器现在支持按日期搜索审核记录。

标准化OAuth2和OpenID Connect以获得安全性

通过采用OAuth2和OpenID Connect作为默认的安全性实现，已经进行了大量的工作来提高安全性。传统的安全选项已删除。基于令牌的授权，授予密码的类型验证和LDAP集成是一些选项，并且在UAA作为后端的支持下始终得到支持。

添加了用于控制流/任务部署操作的新粒度角色 。粒度使与预期操作的一致性更好，并且客户端工具（Shell和Dashboard）也自动适应该操作。

有一个示例应用程序显示了如何使用轻量级目录访问协议（LDAP）安全性支持的Cloud Foundry用户帐户和身份验证 （UAA）服务器来实现Spring Cloud Data Flow。还提供了文档和示例代码，显示了如何配置DataFlowTemplate与OAuth2.0一起使用 。报告了最近CVE的从属库已更新。

改进指标并监视已部署的应用程序

Data Flow 2.0引入了一种新的体系结构，用于收集和显示Streams的应用程序指标。删除了Data Flow 1.4中引入的Data Flow Metrics Collector。

这种新的体系结构基于在已部署的应用程序中使用Micrometer库将度量发送到流行的监视系统，然后使用Grafana可视化度量。爱因斯坦发布者入门系列的每个应用程序都包含用于Prometheus，InfluxDB和DataDog的Micrometer库。

我们提供了有关如何开始使用Prometheus和InfluxDB作为笔记本电脑本地运行的Data Flow监视系统的说明。还有关于在Kubernetes上运行Data Flow时如何开始使用Prometheus作为监视系统的说明 。

还提供了从数据流UI到Grafana仪表板的链接。还提供了两个Grafana仪表板，一个显示了更多以应用程序为中心的视图，另一个显示了更多基于流的视图。这是基于Prometheus的Grafana仪表板的屏幕截图，显示了流中应用程序的消息速率。

应用程序和流的选择器允许您向下导航到感兴趣的特定区域。

这是一个简短的视频，显示了正在进行的Stream监视。

使用千分尺更新分析

在Data Flow 2.0中，Redis计数器的使用已由Micrometer库提供的计数器代替。爱因斯坦发布的应用程序入门系列现在提供了计数器处理器和计数器接收器 。twitter Analytics示例应用程序已更新为使用新的计数器实现，并且提供了Grafana仪表板以显示Data Flow Analytics UI先前显示内容的等效视图。

这是一个简短的视频，展示了正在使用的新分析功能。

数据库迁移支持

使用一套基于Docker的集成测试 ，对所有支持的数据库中的Flyway迁移代码进行了改进和测试 。

更新到Boot 2.1

Data Flow Server 2.0版本现在基于Spring Boot 2.1。通过这种根本的转变，我们花了一些时间解决技术问题并采用了Spring Boot的新功能。在迈向Data Flow 2.0 GA的过程中，我们将继续解决技术难题和一些其他内部重构问题。

更新内部组件以使用JPA

从Spring XD到Data Flow 1.x，我们保留了一些“键值”之类的表，并通过Spring JDBC API对其进行管理。现在我们更新为使用Spring Data JPA。

任务/作业执行和性能改进

感谢社区成员Nicolas Widart提供了有关任务执行及其相关性能问题的详尽的错误报告 。

添加了一个新端点，以避免破坏对现有REST资源的更改，该更改现在已在客户端工具的幕后使用。有了这些更改，查询任务/批处理执行历史记录的速度几乎快了10倍，这在批处理作业中有数百个步骤时提供了帮助。

接下来是什么

虽然我们为2.1考虑了一些功能，但接下来的几个月重点将放在文档，入门指南，视频和整体用户体验上。还计划建立一个新的Data Flow站点。

另请注意，从此2.0 GA发布日期（2020年3月7日）起，Spring Cloud Data Flow 1.x系列将停止维护十二个月。
