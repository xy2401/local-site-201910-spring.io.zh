如果您缺少对反应式流的介绍和Reactor的基本概念，请访问该站点的学习部分和参考指南 。

事不宜迟，让我们进入：

装配时间

当您第一次了解JVM上的响应 式流和响应式编程时，您首先了解的是Publisher和Subscriber ：一个产生数据，另一个消耗数据。简单吧？此外，似乎Publisher 将数据推送到Subscriber 。

但是，当使用诸如Reactor（或RxJava2）之类的Reactive Streams库时，您很快就会遇到以下问题：

订阅之前什么都不会发生

有时，您可能会读到两个库都实现了“推挽混合模型”。等一下拉 ？

我们将回到它，但是要理解该句子，您首先需要意识到，默认情况下，Reactor的反应类型是lazy 。

调用方法Flux要么Mono （ 操作员 ）不会立即触发该行为。取而代之的是Flux （要么Mono ）返回，您可以在其上继续编写其他运算符。因此，您创建了一个运算符链 （或一个运算符非循环图），它表示您的异步处理管道 。

此声明阶段称为组装时间 。

让我们举一个示例，其中客户端应用程序向服务器发出HTTP请求，并期望使用HttpResponse：

Mono<HttpResponse> httpSource = makeHttpRequest();
Mono<Json> jsonSource = httpSource.map(req -> parseJson(req));
Mono<String> quote = jsonSource.map(json -> json.getString("quote"));
//at this point, no HTTP request has been made

可以使用fluent API简化此操作：

Mono<String> quote = makeHttpRequest()
    .map(req -> parseJson(req))
    .map(json -> json.getString("quote"));

声明完管道后，有两种情况：要么通过Flux / Mono将处理管道表示为另一段代码，或者触发管道。

前者表示您返回的代码Mono可能会应用其他运算符，从而导致派生新管道。由于操作员创建了新实例（就像洋葱一样），因此您自己Mono不会被突变，因此可以对其进行多次修饰，结果会大不相同：

//you could derive a `Mono<String>` of odd-length strings vs even-length ones
Mono<String> evenLength = quote.filter(str -> str.length() % 2 == 0);
Mono<String> oddLength = quote.filter(str -> str.length() % 2 == 1);

//or even a `Flux<String>` of words in a quote
Flux<String> words = quote.flatMapMany(quote -> Flux.fromArray(quote.split(" ")));

//by this point, none of the 3 "pipelines" have triggered an HTTP request

与一个比较CompletableFuture ，这本质上并不懒惰：一旦您引用了CompletableFuture ，表示处理已经在进行中...

考虑到这一点，让我们研究一下如何触发反应式管道。

订阅时间

到目前为止，我们已经组装了一个异步管道 。也就是说，我们已实例化Flux和Mono通过使用运算符的变量，这导致其他Flux / Mono行为像洋葱一样分层。

但是数据还没有开始流经每个声明的管道。

这是因为数据流的触发不是管道的声明，而是对其的订阅 。记得：

订阅之前什么都不会发生

订阅是指“好的，该管道代表数据的转换，我对数据的最终形式很感兴趣”。最常见的方法是致电Flux.subscribe(valueConsumer, errorConsumer) 。

感兴趣的信令通过运营商链向后传播，直到源运营商， Publisher实际上会产生初始数据：

makeHttpRequest() //<5>
    .map(req -> parseJson(req)) //<4>
    .map(json -> json.getString("quote")) //<3>
    .flatMapMany(quote -> Flux.fromArray(quote.split(" "))) //<2>
    .subscribe(System.out::println, Throwable::printStackTrace); //<1>
  1. 我们赞成的话Flux ，表明我们要将每个单词打印到控制台（并打印任何错误的堆栈跟踪）
  2. 兴趣已传达给flatMapMany步…
  3. …将其向上标志到json map步…
  4. …然后提出要求map步…
  5. …最终到达makeHttpRequest() （我们将考虑其来源）

此时，源被触发。它以适当的方式生成数据：在这里它将向产生JSON的端点发出HTTP请求，然后发出HTTP响应。

从那时起，我们处于执行时间 。数据已开始流过管道（从上到下的顺序，或者从上游到下游 ）：

  1. 的HttpResponse被发射到parseJson map
  2. 提取JSON正文并将其发送给getString map
  3. 提取报价并将其传递给flatMapMany
  4. 的flatMapMany将报价分成单词并分别发出每个单词
  5. 中的值处理程序subscribe通知每个单词，将它们打印到控制台，每行一个

希望可以帮助您了解汇编时间与订阅/执行时间之间的区别！

冷vs热

在解释了差异并介绍了这一口头禅之后，可能是引入异常的好时机：laughing：

在您订阅之前什么都不会发生… 直到有事情发生

冷

到目前为止，我们一直在处理Flux和Mono消息称感冒Publisher 。正如我们所解释的，这些Publishers是懒惰的，仅当有Subscription 。此外，他们为每个人重新生成数据Subscription 。

在我们的HTTP响应示例中Mono ，则将为每个订阅执行HTTP请求：

Mono<String> evenLength = quote.filter(str -> str.length() % 2 == 0);
Mono<String> oddLength = quote.filter(str -> str.length() % 2 == 1);
Flux<String> words = quote.flatMapMany(quote -> Flux.fromArray(quote.split(" ")));

evenLength.subscribe(); //this triggers an HTTP request
oddLength.subscribe(); //this triggers another HTTP request
words.subscribe(); //this triggers a third HTTP request

附带说明，某些运营商的行为暗示了多个订阅。例如retry发生错误时重新订阅其来源（ onError信号），而repeat做了同样的onComplete信号。

因此，对于像HTTP请求这样的冷源，类似retry将重新执行该请求，从而允许从例如服务器瞬时错误中恢复。

热

热门Publisher另一方面，它并不那么明确：它不一定需要Subscriber开始抽取数据。不必每次每个新生成专用数据Subscriber要么。

为了说明这一点，让我们介绍一个新的冷发布者示例，然后我们将展示如何将该冷发布者转变为热门发布者：

Flux<Long> clockTicks = Flux.interval(Duration.ofSeconds(1));

clockTicks.subscribe(tick -> System.out.println("clock1 " + tick + "s");

Thread.sleep(2000);

clockTicks.subscribe(tick -> System.out.println("\tclock2 " + tick + "s");

打印：

clock1 1s
clock1 2s
clock1 3s
    clock2 1s
clock1 4s
    clock2 2s
clock1 5s
    clock2 3s
clock1 6s
    clock2 4s

我们可以把clockTicks通过调用将其变成热门share() ：

Flux<Long> coldTicks = Flux.interval(Duration.ofSeconds(1));
Flux<Long> clockTicks = coldTicks.share();

clockTicks.subscribe(tick -> System.out.println("clock1 " + tick + "s");

Thread.sleep(2000);

clockTicks.subscribe(tick -> System.out.println("\tclock2 " + tick + "s");

而是产生以下结果：

clock1 1s
clock1 2s
clock1 3s
    clock2 3s
clock1 4s
    clock2 4s
clock1 5s
    clock2 5s
clock1 6s
    clock2 6s

您会看到两个订阅现在共享相同的时钟滴答。 share()通过将源多播元素转换为新的将冷到热Subscribers ，但是只有这些新订阅之后发出的元素 。以来clock2 2秒后订阅，错过了早期发射1s和2s 。

因此，热门发布者可以减少懒惰，即使他们通常至少需要初步的Subscription触发数据流。

结论

在本文中，我们了解了实例化a和Flux /链接运算符（又名Assembly time ），触发它（又称Subscription time ）并执行它（又称Execution time ）。

因此，我们了解到Flux和Mono大多是懒惰的（又名冷Publisher ）： 在您订阅它们之前，没有任何反应 。

最后，我们了解了Flux和Mono ，被称为热门Publisher ，其行为略有不同，并且比较懒惰。

在下一期中，我们将了解为什么这三个阶段在您作为开发人员调试基于反应堆的代码的方式上有重大不同。
