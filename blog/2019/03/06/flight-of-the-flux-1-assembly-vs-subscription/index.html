工程

三月06，2019

它源自我的《 Flight of the Flux演讲，我发现该内容更适合于博客文章格式。

发布其他帖子时，我将使用链接更新下表，但这是计划的内容：

如果你错过了介绍无流和电抗器的基本概念，伸头网站的学习部分和参考指南 。

事不宜迟，让我们进入：

装配时间

当您第一次了解JVM上的响应 式流和响应式编程时，您首先了解的是Publisher和Subscriber之间的高级关系：一个产生数据，另一个消耗数据。简单吧？此外，似乎Publisher 将数据推送到Subscriber 。

但是，当使用诸如Reactor（或RxJava2）之类的Reactive Streams库时，您很快就会遇到以下问题：

订阅之前什么都不会发生

有时，您可能会读到两个库都实现了“推挽混合模型”。等一下拉 ？

我们将回到它，但是要理解该句子，您首先需要意识到，默认情况下，Reactor的反应类型是lazy 。

调用一个方法Flux或Mono （ 运营商 ）不会立即触发的行为。而是返回一个新的Flux （或Mono ）实例，在该实例上您可以继续组成其他运算符。因此，您创建了一个运算符链 （或一个运算符非循环图），它表示您的异步处理管道 。

此声明阶段称为组装时间 。

让我们以一个示例为例，其中客户端应用程序向服务器发出HTTP请求，并期望使用HttpResponse：

Mono httpSource = makeHttpRequest();
Mono jsonSource = httpSource.map(req -> parseJson(req));
Mono quote = jsonSource.map(json -> json.getString("quote"));
//at this point, no HTTP request has been made

可以使用fluent API简化此操作：

Mono quote = makeHttpRequest()
    .map(req -> parseJson(req))
    .map(json -> json.getString("quote"));

声明完管道后，有两种情况：或者将代表处理管道的Flux / Mono传递到另一段代码，或者触发管道。

前者意味着您返回Mono的代码可能会应用其他运算符，从而导致派生新的管道。由于操作员创建了新实例（就像洋葱一样），因此您自己的Mono不会发生变异，因此可以对其进行多次修饰，结果会大不相同：

//you could derive a `Mono` of odd-length strings vs even-length ones
Mono evenLength = quote.filter(str -> str.length() % 2 == 0);
Mono oddLength = quote.filter(str -> str.length() % 2 == 1);

//or even a `Flux` of words in a quote
Flux words = quote.flatMapMany(quote -> Flux.fromArray(quote.split(" ")));

//by this point, none of the 3 "pipelines" have triggered an HTTP request

将其与本质上并不懒惰的CompletableFuture进行比较：一旦您引用了CompletableFuture ，就意味着处理已经在进行中……

考虑到这一点，让我们研究一下如何触发反应式管道。

订阅时间

到目前为止，我们已经组装了一个异步管道 。也就是说，我们已经通过使用operator实例化了Flux和Mono变量，这导致其他Flux / Mono的行为像洋葱一样分层。

但是数据还没有开始流经每个声明的管道。

这是因为数据流的触发不是管道的声明，而是对其的订阅 。记得：

订阅之前什么都不会发生

订阅是指“好的，该管道代表数据的转换，我对数据的最终形式很感兴趣”。最常见的方法是调用Flux.subscribe(valueConsumer, errorConsumer) 。

该感兴趣的信令通过运营商的链向后传播，直到源操作中， Publisher ，实际上产生了初期数据：

makeHttpRequest() //<5>
    .map(req -> parseJson(req)) //<4>
    .map(json -> json.getString("quote")) //<3>
    .flatMapMany(quote -> Flux.fromArray(quote.split(" "))) //<2>
    .subscribe(System.out::println, Throwable::printStackTrace); //<1>
  1. 我们订阅单词Flux ，表明我们要将每个单词打印到控制台（并打印任何错误的堆栈跟踪）
  2. 兴趣已flatMapMany给flatMapMany步骤…
  3. …将其向上标志到json map步骤的链map …
  4. …然后是请求map步骤…
  5. ......终于到达makeHttpRequest()我们会考虑我们的消息来源）

此时，源被触发。它以适当的方式生成数据：在这里它将向产生JSON的端点发出HTTP请求，然后发出HTTP响应。

从那里，我们在执行时间 。数据已开始流过管道（按照从上到下的顺序，或者从上游到下游 ）：

  1. 所述HttpResponse发射到parseJson map
  2. 它提取JSON正文并将其发送到getString map
  3. 提取报价并将其传递给flatMapMany
  4. flatMapMany将引号拆分为多个单词，并分别发出每个单词
  5. 通知每个subscribe的值处理程序每个单词，并将其打印到控制台，每行一个

希望可以帮助您了解汇编时间与订阅/执行时间之间的区别！

冷与热

在解释了差异并介绍了这一口头禅之后，可能是引入异常的好时机：laughing：

在您订阅之前什么都不会发生… 直到有事情发生

冷

到目前为止，我们一直在处理称为Cold Publisher的Flux和Mono来源。正如我们已经解释的那样，这些Publishers是惰性的，仅在有Subscription时才生成数据。此外，它们为每个单独的Subscription重新生成数据。

在我们的HTTP响应Mono示例中，将为每个订阅执行HTTP请求：

Mono evenLength = quote.filter(str -> str.length() % 2 == 0);
Mono oddLength = quote.filter(str -> str.length() % 2 == 1);
Flux words = quote.flatMapMany(quote -> Flux.fromArray(quote.split(" ")));

evenLength.subscribe(); //this triggers an HTTP request
oddLength.subscribe(); //this triggers another HTTP request
words.subscribe(); //this triggers a third HTTP request

附带说明，某些运营商的行为暗示了多个订阅。例如，在发生错误（ onError信号）的情况下， retry重新预订其源，而repeat对onComplete信号执行相同的操作。

因此，对于像HTTP请求这样的冷源，诸如retry类的东西将重新执行该请求，从而允许从例如服务器瞬时错误中恢复过来。

热

另一方面， 热门Publisher并不那么明确：它不一定需要Subscriber才能开始抽取数据。它也不一定每个新Subscriber都重新生成专用数据。

为了说明这一点，让我们介绍一个新的冷发布者示例，然后我们将展示如何将该冷发布者转变为热门发布者：

Flux clockTicks = Flux.interval(Duration.ofSeconds(1));

clockTicks.subscribe(tick -> System.out.println("clock1 " + tick + "s");

Thread.sleep(2000);

clockTicks.subscribe(tick -> System.out.println("\tclock2 " + tick + "s");

打印：

clock1 1s
clock1 2s
clock1 3s
    clock2 1s
clock1 4s
    clock2 2s
clock1 5s
    clock2 3s
clock1 6s
    clock2 4s

我们可以通过调用share()将clockTicks源变成热门的源：

Flux coldTicks = Flux.interval(Duration.ofSeconds(1));
Flux clockTicks = coldTicks.share();

clockTicks.subscribe(tick -> System.out.println("clock1 " + tick + "s");

Thread.sleep(2000);

clockTicks.subscribe(tick -> System.out.println("\tclock2 " + tick + "s");

而是产生以下结果：

clock1 1s
clock1 2s
clock1 3s
    clock2 3s
clock1 4s
    clock2 4s
clock1 5s
    clock2 5s
clock1 6s
    clock2 6s

您会看到两个订阅现在共享相同的时钟滴答。 share()通过将源多播元素分配给新的Subscribers来将冷到热转换，但仅将这些新订阅之后发出的元素传递给新的Subscribers 。由于clock2在2秒后订阅，因此错过了早发射1s和2s 。

因此，即使热门发布者通常通常至少需要初始Subscription才能触发数据流，他们也可以较为轻松。

结论

在本文中，我们了解了实例化Flux /链接运算符（即Assembly time ），触发它（即Subscription time ）和执行它（即Execution time ）之间的区别。

因此，我们了解到Flux和Mono大多是懒惰的（又称Cold Publisher ）： 只有订阅它们后， 什么都不会发生 。

最后，我们了解了Flux和Mono的另一种风味，称为热门Publisher ，其行为有所不同，并且不那么懒惰。

在下一期中，我们将了解为什么这三个阶段在您作为开发人员调试基于反应堆的代码的方式上有重大不同。
