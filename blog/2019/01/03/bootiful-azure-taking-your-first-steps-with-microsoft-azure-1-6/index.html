这是6部分系列的第1部分，在星期一和星期四有新帖子，向Spring开发人员介绍Microsoft Azure。如果没有Microsoft的Asir Vedamuthu Selvasingh，Yitao Dong，Bruno Borges，Brian Benz和Theresa Nguyen的支持，我不可能将这些放在一起。您可以在Github上找到该系列的代码。在阅读有关任何反馈或问题的文章时，请在Twitter（@starbuxman）上打我。您还可以在我的Spring Tips（@SpringTipsLive）系列Bootiful Azure中了解有关Microsoft Azure的更多信息。

嗨，春季粉丝！在本系列中，我将介绍Spring对Microsoft Azure的支持，重点介绍平台启用的，可用于Spring应用程序使用的有趣技术。我想一如既往地强调，您应该利用Microsoft Azure，它可以提供与众不同的体验。Microsoft Azure和Google Cloud以及Amazon Web Services和Alicloud都提供基础设施服务，例如自定义消息队列，存储，数据库等，它们提供了引人入胜的，有时甚至比更多库存标准商品和开源替代产品更好的替代产品。所有平台都有使它们以一种或另一种方式领先于众的事物。这些东西一定有引力 ;任何人都可以运行MySQL或Redis的地方，很少人可以运行Google Cloud Spanner或Microsoft CosmosDB。使用righ工具完成工作！如果您需要它，请使用CosmosDB。

如果您要进行的工作具有足够的通用性，可以将其与特定平台解除捆绑，那么您应该绝对以尽可能不受限制的方式使用它。尽可能保持通用。是否要运行Java或Node.js进程？使用像Kubernetes这样的容器协调器。或者，如果您喜欢自己，并且希望快速投入生产，请使用平台即服务，例如Heroku或Cloud Foundry，它们的平台是应用程序，而不是底层容器。这些技术使您可以根据应用程序二进制文件进行思考，并快速交付生产。它们也是开源的，具有庞大的使用基础和生态系统，因此它们是最简单的解决方案，也是最强大的解决方案。从技术上讲，在这种情况下，使用通用工具也是最佳选择。使用Cloud Foundry（只需在Microsoft Azure上安装并部署Pivotal Cloud Foundry服务）或Microsoft Azure上的Kubernetes之类的工具开始就非常简单。

数据库是一个特别棘手的命题。如果您投资于专有数据库（例如Google的Spanner），则应该期望如果有一天想要将数据从该平台上移出将非常困难。前Riak首席技术官Dave McCrory谈到了数据重力的想法-数据池激发了应用到该池中或从中提取的应用程序生态系统的概念。这些应用程序越有用，就会有更多的人使用它们，并且人们越有可能建立这些集成。Salesforce就是一个很好的例子。其数据是粘性的 。Salesforce CRM是丰富而广泛的，但绝不是唯一功能齐全的CRM。它所具有的比任何竞争产品都更多的是伙伴集成的生态系统，此后人们开始依赖该系统。这些合作伙伴的整合使Salesforce成为加州风格的酒店：您可以办理登机手续，但永远都不能离开。因此，很显然，使用最佳工具来完成工作，但要意识到，除了具有可观的管理标记之外，这些工具还具有可选成本。

设定

让我们看一下Microsoft Azure本身。您需要登录到门户网站以获得本系列中将介绍的各种服务的相关配置密钥。门户是您应该添加书签的东西 。通常，当您查看门户网站上与所使用服务有关的部分时，会看到一个名为“密钥”或“配置密钥”或类似名称的部分。不幸的是，这在整个服务中并不是100％一致的。

在本系列中，我们将专注于利用平台的优势-微软通过Azure平台提供差异化优势的地方。您可以使用来实现我们将在本文中介绍的大部分内容az命令行工具。由于它是用Python编写的，因此可以很容易地安装在多个操作系统上。请查阅此文档，以获取有关如何安装Microsoft Azure CLI的更多信息 ， az 。

就是这样假设您有一个Azure帐户，并且az准备使用CLI，就可以开始编码了。Azure没有平台范围的身份验证概念。您对每项服务进行身份验证。这样一来，就可以轻松地使用每个单独的服务，但这意味着，从整体上讲，您比在使用Google Cloud时要更担心身份验证。这并不是说要引入更多的工作。借助Google Cloud，您必须经常明确地“选择加入”特定服务，然后才能使用它。所以：我想是六个，另一个六个。

如您所料，您可以使用Spring Boot和Spring Initializr来启动新项目。选择Azure Support除了您要使用的其他功能。另外，如果要将Microsoft Azure依赖项引入现有应用程序，则可以将Maven物料清单（BOM）工件手动添加到Spring Boot项目的Maven或Gradle构建中。这就是您为Maven要做的事情。

    <dependencyManagement>
        <dependencies>
            <dependency>
                <groupId>com.microsoft.azure</groupId>
                <artifactId>azure-spring-boot-bom</artifactId>
                <version>${azure.version}</version>
                <type>pom</type>
                <scope>import</scope>
            </dependency>
        </dependencies>
    </dependencyManagement>

就我而言，Maven属性azure.version ，设置为2.0.5 。

从这里开始，我将使用Ivy语法介绍Maven存储库依赖项坐标： groupId ： artifactId ： version 。有时， version不需要，因此我将省略它，因为BOM将为我们管理这些版本。

认证方式

如果您想使用az安装在本地计算机上的CLI，您需要运行az login 。否则，您可以从Azure门户中启动Microsoft Azure Cloud Shell。它会给您两个选择：Bash或Powershell。

资源组

登录后，您需要配置资源组。资源组是资源的命名集合。这是一件方便的事情，因为您可以站起所有与此Spring和Microsoft Azure简介相关的资源，将每个资源与该资源组相关联，然后在完成所有操作后销毁该资源组，级联删除与其关联的所有内容。

这使计费，生命周期管理等变得非常方便！每个资源组都与一个区域关联。Microsoft Azure有很多地区可以在其中运行服务。截至撰写本文时， 全球有54个地区 ，超过任何其他云提供商。

这是创建资源组的方式， bootiful在里面West US 2美国的位置。我们将在后续示例中使用它。

az group create --name bootiful --location "West US 2"

您可以运行以下命令来迭代所有可能的位置： az account list-locations 。您会注意到，在发出命令时，我们需要在创建时将资源与特定位置相关联，通常是-l要么--location 。这可能很乏味，因此指定默认位置，然后在随后的调用中将其忽略是很有帮助的，如下所示： az configure --defaults location=westus 。

通过运行来确认一切正常az configure 。

Cloud Foundry服务经纪人

您可以使用Cloud Foundry服务代理快速启动各种Microsoft Azure服务，然后将它们绑定到您的应用程序。该服务代理使运行在Cloud Foundry上的应用程序，尤其是在Microsoft Azure之上运行，这是针对Microsoft Azure服务的应用程序最简单的生产途径。
