Migration Details

The entire 15+ year history of every Spring Framework issue, and every comment, has been imported into GitHub. There is a lot to consider in such a move, so let’s take a tour and go over some details.

Links

If you have a link to an existing issue, e.g. https://jira.spring.io/browse/SPR-16751, you’ll be redirected to the corresponding GitHub issue. If you actually mean to go to the Jira issue, append the query parameter redirect=false. On the GitHub side, the imported issue has a link back to its Jira issue origin.

Spring框架从Jira迁移到GitHub的问题

工程
罗森·斯托扬切夫（Rossen Stoyanchev）
一月15，2019

Spring框架已将其整个问题历史从Jira迁移到GitHub。这篇博客文章的目的是为您提供有关此举的背景信息和详细信息。

迁移详情

每个Spring Framework问题的全部15年历史以及每个评论都已导入到GitHub。这样的举动有很多要考虑的地方，所以让我们一起浏览一下一些细节。

链接

如果您具有指向现有问题的链接，例如https://jira.spring.io/browse/SPR-16751 ，您将被重定向到相应的GitHub问题。如果您确实要去解决Jira问题，请附加查询参数redirect=false 。在GitHub端，导入的问题具有指向其Jira问题起源的链接。

吉拉（Jira）发行密钥以文本形式显示，例如“SPR-16751”已替换为GitHub 问题参考 ，其优点是可以在双方问题的时间表中插入链接。另一个好处是，当您将鼠标悬停在链接上时，可以使用迷你预览。

Jira向其他以文本形式显示的Spring项目颁发密钥，例如“DATAJPA-813”，已被链接替换。例如，请参阅＃18558以及指向Spring Data JPA的链接， ＃20904带有 Spring Data MongoDB的链接，以及＃16906带有 Spring Integration Extensions的链接。

链接到其他GitHub项目问题，并拉取显示在文本中的请求，可自动受益。迁移之后，所引用问题的时间轴中会包含指向Spring框架的链接，您可以通过鼠标悬停来预览链接。例如，请参阅＃21362到Spring Boot的链接， ＃20849到JUnit的链接， ＃20256到Jackson的链接。

链接到GitHub上任何项目的提交和源代码，也可以获得自动的好处。例如，请参见问题＃18463中的源代码范围链接。

吉拉详情

每个导入的问题都会在其说明的下半部分显示来自Jira的信息。这个想法是，来自Jira的所有信息都可以在GitHub上找到，而您不必在两者之间来回走动。您可能会看到以下一项或多项信息（如果有）：

  • 受影响的版本
  • 参考网址
  • 附件
  • 子任务和相关问题
  • 拉取请求并提交引用
  • 反向版本
  • 投票人数和观看者人数

请注意，投票和观看订阅无法转移到GitHub。即使spring-issuemaster拥有全部特权，它也只能投票一次。因此，请访问GitHub问题并重新应用响应，并订阅以接收特定问题的更新。

标签

一些Jira字段已转换为GitHub问题标签：

吉拉菲尔德 标签
发行类型 "type: *"
状态 "status: *"
解析度 "status: *"
零件 "in: *"

还为导入的问题应用了两个额外的标签：

标签 描述
"has: votes-jira" 导入的问题票数超过10
"has: backports" 反向版本的问题

我们利用这次机会简化了Jira字段值，例如，Jira中的25个组件值对应于5个"in: *" GitHub上的标签。的"status: *"和"type: *"标签也已通过额外的修改。

我们选择的标签与Spring Boot中使用的标签对齐。Boot团队已经给了他们的过程和标签很多想法，我们知道许多人会赞赏它的一致性。请参阅完整的标签集 。

在Jira中，许多字段和标签都是可修改的。在GitHub中，只有贡献者可以添加或删除标签。这是很合理的。记者只描述问题，而贡献者则将问题归类。开发人员和贡献者都可以使用标签进行搜索。

修复版本

一个Jira问题可以有多个修复版本，而一个GitHub问题只能有一个目标里程碑。这似乎是一个缺点，但肉眼看不到的还多，而且约束条件迫使我们考虑进行一些有意义的调整。

以修复版本“ 4.3.19”，“ 5.0.9”和“ 5.1 RC3”的SPR-17226为例。尽管在所有3个版本中均解决了该问题，但无需污染当时仍在开发中的“ 5.1 RC3”发行说明。相反，我们可以说它已在5.0.9中修复并反向移植到4.3.19，因此确保修复已传播到仍在开发中的下一个版本是供贡献者关注的问题。这就是它在GitHub问题＃21759上的显示方式，我们过去是否可以进行此调整？可以，但是Jira对多个修复版本的支持并没有迫使我们考虑。

这是一个很小的调整，将产生更干净的发行说明，但也会影响我们提交修复程序的方式。当母版上正在开发下一个版本时，我们将首先将修补程序应用于当前的生产分支，然后将其向前合并到母版，然后从较早的生产分支中进行挑选。而不是将修补程序应用于母版，然后从生产分支中挑选。结果是更清晰的提交历史记录，因为前向合并有助于git感知或进行相关更改。

至于反向移植，由于一个问题只能有一个目标里程碑，因此我们必须创建单独的问题来表示修订的反向移植。在同一示例中，选择“ 5.0.9”作为目标里程碑是值得的，因为我们只会遇到一个backport问题。如果选择“ 5.1 RC3”，则需要两个backport问题。为了让您感觉到这种变化带来的巨大变化，可以说Spring框架从第一天起就出现在GitHub Issues上。如果使用后一种方法，那么今天将有大约2500个反向端口问题。如果使用前者，则大约为1000。

对于历史性的移植，我们为每个版本创建了一个发行人。其中大约有45个 。展望未来，我们将针对所有向后移植的修复程序逐一解决问题，这些问题将自动创建，并且主要用于发布跟踪。至于讨论，大多数对话应在主要问题上进行，而反向移植问题将仅用于特定于反向移植的讨论。

标记

毫无疑问，加价是迁移过程中最大，最痛苦的部分。跟踪问题已有15年的历史，反映了编程风格的重大变化，进而决定了注释中显示的内容。

例如，开始时在注释中粘贴了许多XML，Markdown将其视为HTML块 ，这导致标记根本不显示。当然如果那些被包围{code:xml}...{code} ，它看起来还不错，但是在那个时候标记并不常用，而且XML代码片段还是出现了，这并没有引起问题，因此无法正确迁移。

还有很多其他复杂的操作，例如花括号的转义，以避免单行距的影响，或星号的转义，以防止它们消失以加粗字体。我将为您保留细节。可以说我们付出了很多努力来确保标记转换的质量相当高。

要强调的一个具体问题是在纯文本中（即代码块外部）使用“ @”。这些是触发通知的GitHub上的用户提及。你可能会惊讶的是@Bean ， @Configuration ， @Component是实际的GitHub的用户。诸如@ andy，@ arjen，@ brian之类的名字引用与GitHub用户名冲突在某些时候也很常见，所有这些在导入17,000多个带有注释的问题中都是非常麻烦的。这就是为什么我们会小心翼翼地逃脱它们。展望未来，在创建新问题或评论时，请成为一名优秀的GitHub公民并使用反引号，例如`@ Foo`（是的， https：//github.com/foo确实存在）。

背景

我使用和喜欢吉拉很久了。迁移到GitHub Issues的想法并没有立即出现。相比之下，这似乎太基础了。尽管我必须承认自从此举以来，GitHub Issues的确在我身上成长，但完全使我回过头来与逐个特征的比较关系不大。我要暗示的是更大的力量在起作用。

GitHub是几乎每个开源项目（包括每个 Spring项目）的所在地，可以合理地期望所有用户都拥有GitHub凭据。因此，如今，期望开发人员为他们依赖或想要报告问题的每个开源项目的问题跟踪程序维护单独的登录已经变得站不住脚了。

然后，可以找到并置源代码和问题的好处。我之前提到过很多，例如在单个项目中以及在GitHub上的所有项目中，跨问题，拉取请求，源代码和提交的自动链接引用 。提及和通知任何GitHub用户的能力。所有这些都是非常强大的优势，而孤立的问题跟踪根本无法实现。我怀疑有人想回到开源项目在不同地方托管的时代。问题跟踪也是如此。

并置源代码和问题有更深，不太明显的好处。GitHub将问题视为平等并拉取请求。为它们分配了相同顺序的编号。它们看起来相同（描述，注释，标签和目标里程碑）。它们无区别地出现在发行说明中。拉取请求不过是附带提交的问题。

从历史上讲，在Spring框架中，我们一直为每个拉取请求坚持一个Jira问题。我们也不喜欢这种负担，但是我们需要为所有问题提供一个单一的记录位置。由于这种分歧的局面，人们一直不太清楚在拉取请求下应该讨论什么以及在吉拉问题中应占多少。

向前发展，这不再是问题。我们期望一个问题或一个请求请求，而不是两者兼而有之。如果您需要先进行讨论（我们鼓励这样做） ，然后创建问题，然后再提交请求，则PR将取代该问题。两者仍然保持链接，并且没有丢失。对话只是跟随动作。

标记问题不容忽视。毫无疑问，Wiki标记对于代码相关的讨论很痛苦。我每天都使用它好几年了。我已经习惯了，但是有些事情很难，需要太多的努力。这里提醒您在代码段中显示大括号和星号等常见内容：
{{/endpoint/\{server-id\}/\{session-id\}/\{transport/\*\}}} 。

毫无疑问，Markdown更容易与代码相关的注释。它需要较少的输入，并且仅在格式化代码时起作用，因为它更简单并且不会与代码中常见的符号冲突。从一开始我就很明显，因为我也并行使用了GitHub和Markdown多年了。我不知道为什么Jira仍然不支持Markdown。需要明确的是，这不是决定性因素。这只是您以后要学会与之共处的那些事物之一，可以成为改变的额外动力。

最后但并非最不重要的一点是，今天大多数开发人员通过Spring Boot使用Spring，而Spring Boot一直使用GitHub Issues。仅从这个角度来看，Spring框架有足够的动机进行迁移，因为Spring Boot不会迁移到Jira，而这将是为Spring用户提供更一致的体验的唯一其他方法。

实际迁移

尽管做了很多准备工作，但是没有什么比实际迁移的日子要好了。我们使用了GitHub的非官方导入API ，据记录该API不会触发任何通知。在测试过程中，我们没有发现任何问题。实际迁移开始后，有关每个问题和评论的通知开始涌入。

我们开始使用每个可用的渠道来获得GitHub支持。对我们来说幸运的是，他们也注意到了。他们怎么可能不呢？据我估计，在我们中断之前导入的2,600个问题一定产生了数千万封电子邮件，不足为奇地导致通知中断。

一天后，在GitHub支持更正了问题并关闭了Spring Framework项目的所有通知以确保安全后，我们走上了一条通畅的路径，可以在8到9个小时内导入所有问题。又花了几个小时才能再次通过所有问题和评论，用GitHub参考号替换Jira问题密钥，然后又花了几天的时间来检查和清理标记转换问题。

所有这些工作现在已经完成，很高兴宣布我们已经在GitHub Issues上营业。
