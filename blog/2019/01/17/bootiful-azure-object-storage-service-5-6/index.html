现在让我们谈谈更多……平凡。理想情况下，您甚至不会经常考虑所有事情。应用程序通常具有存储要求：它们可能需要存储上载的用户内容（图片或文档之类的二进制数据），生成的工件（如PDF文件，视频，音乐等）。他们可能希望存储日志。不难想到应用程序可能想要持久存储的内容。

这些应用程序可以使用文件系统，例如本地计算机上的文件系统或网络连接的文件系统（例如NFS（网络文件系统）） 。我将使用NFS笼统地指代任何网络连接的文件系统，例如Samba，NFS（那些通过NFS协议安装的文件系统）本身，或旧选项（例如DAC，FAL等）。NFS选项通常为文件提供类似于文件系统的接口以分层格式。NFS选项很有趣，因为它们就像我们都熟悉的文件系统一样，每个人都知道如何使用文件系统，对吗？当然，一切看起来都像Windows资源管理器中表达的层次结构吗？还是macOS Finder？除了，他们看起来彼此都不像...

想一想，当我说NFS选项就像文件系统时，我不得不说“喜欢”。一些支持比其他支持更多细微的权限模型。一些支持对附加到树中文件和目录的元数据进行编码和复制。支持为不同的操作提供不同的速度保证；某些文件系统可能会针对读取和写入进行优化。有些可能会优化目录遍历。客户端对文件读写的看法因所使用的客户端而异。（写操作是否立即在所有复制节点上保持一致？）

即使我们假定要写入的所有文件系统都是POSIX友好的，您也可以使用int open(const char *path, int flags)要么java.io.File -对于我们的客户而言可能并非如此。如果客户端不“说”文件系统，而是希望以其他方式处理数据怎么办？如果客户端只能说HTTP，该怎么办？还是要说说Bittorrent来更有效地整合点对点网络的下载内容？

由于这些原因以及其他原因，Amazon Web Services引入了S3，即简单云存储服务（了解吗？“ S”乘以3？“ S3”？）此后，这已成为所有其他云供应商都需要支持的主流标准。对于Microsoft Azure，对象存储服务（OSS）可以提供类似S3的体验。它不是POSIX文件系统。您可以直接使用它的API，就像我们在这里将要使用的那样 ，但是也可以使用S3Proxy通过AWS S3客户端代理对OSS的写入 ，其中有无数！Azure的对象存储服务确实很无聊 ，而这正是处理诸如文件状数据的持久卷之类的基本内容时想要的。它甚至提供了一个名为Microsoft Azure Storage Explorer的独立浏览器， 该浏览器可以运行（是！）。在Linux，Macintosh和Windows上。该独立浏览器使您可以查询OSS存储以及 CosmosDB数据。为了方便起见？您当然可以使用az CLI或API本身。我们将对Java API使用Spring集成。

配置Azure对象存储服务

您必须创建一个Azure Object Storage Service帐户（ bootiful ）分配给现有资源组bootiful 。然后，创建一个存储容器（ files ）分配给刚创建的存储帐户。这是一个示例脚本。

#!/usr/bin/env bash

rg=$1
accountname=bootiful

az storage account create --name ${accountname} --resource-group ${rg}
az storage container create -n files --account-name ${accountname}

使用以下命令确定连接到应用程序所需的连接字符串。

az storage account show-connection-string --resource-group bootiful --name bootiful

请注意连接字符串，以备后用。

将Azure对象存储服务引入您的Spring应用程序

加com.microsoft.azure ： azure-storage-spring-boot-starter到应用程序的构建文件。确保您已为azure.storage.connection-string属性。

我们将在应用程序的目录中读取猫图像的字节src/main/resources目录，然后将这些字节作为“块博客”写入对象存储服务。您可以通过其他接口与OSS进行对话，但是出于我们的目的，很自然地将其视为“容器”（事物的逻辑分组，几乎像S3术语中的目录或存储桶）和“ blob”的集合。”Blob本质上是一个文件，具有名称和与之关联的元数据。以下示例所做的全部工作就是将猫照片的字节存储到一个名为的容器中files以随机名称开头cat-并以.jpg 。

package com.example.bootifulazure;

import com.microsoft.azure.storage.CloudStorageAccount;
import com.microsoft.azure.storage.StorageException;
import com.microsoft.azure.storage.blob.CloudBlobContainer;
import com.microsoft.azure.storage.blob.CloudBlockBlob;
import lombok.extern.log4j.Log4j2;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.boot.context.event.ApplicationReadyEvent;
import org.springframework.context.event.EventListener;
import org.springframework.core.io.Resource;
import org.springframework.stereotype.Component;

import java.io.InputStream;
import java.net.URISyntaxException;
import java.util.UUID;

@Log4j2
@Component
class ObjectStorageServiceDemo {

    private final CloudStorageAccount cloudStorageAccount;
    private final Resource resource;
    private final CloudBlobContainer files;

    ObjectStorageServiceDemo(
        CloudStorageAccount csa,
        @Value("classpath:/cat.jpg") Resource cat) throws URISyntaxException, StorageException {
        this.resource = cat;
        this.cloudStorageAccount = csa;
        this.files = this.cloudStorageAccount
            .createCloudBlobClient()
            .getContainerReference("files");
    }

    @EventListener(ApplicationReadyEvent.class)
    public void demo() throws Exception {
        CloudBlockBlob blockBlobReference = this.files.getBlockBlobReference("cat-" + UUID.randomUUID().toString() + ".jpg");
        try (InputStream in = this.resource.getInputStream()) {
            blockBlobReference.upload(in, this.resource.contentLength());
            log.info("uploaded blockblob to " + blockBlobReference.getStorageUri());
        }

    }
}

Microsoft Azure特定的位不那么琐碎。我们获得对容器的引用，然后对其进行写入，然后注销该资源的可寻址URI。多么平凡 ！这正是您在诸如文件系统之类的计算系统原语中想要的。应该是平凡的 。坦白说，我对使用Java 7的try-with-resources语法感到非常满意。 AutocloseableInputStream参考！
