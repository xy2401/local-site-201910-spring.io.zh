安全配置

我们从本地机器的舒适性中轻松轻松地开发了该应用程序，并在需要时插入了相关的配置值。麻烦的是，这些值通常是非常敏感的值，因此不应在未加密的静态文件系统上保留这些值。为此有很多好的解决方案。您当然可以部署Spring Cloud Config Service本身。如果您在Microsoft Azure（或其他）上运行Pivotal Cloud Foundry ，则建议使用此方法，因为它是部署和工作的唯一方法。您当然可以部署Hashicorp Vault并使用Spring Cloud Vault 。或者，您可以将键和值存储在Microsoft 自己的Key Vault Service中 。为了有效地使用Key Vault，您需要配置Microsoft Active Directory，亲爱的读者，虽然我不希望您或您所爱的人对我有什么兴趣，但是对我却很感兴趣。

Microsoft活动目录

在过去20多年的帮助组织开发软件的过程中，我看到nary很少使用LDAP，即使不使用Microsoft Active Directory。当然，我希望谷歌企业应用套件获得在企业中更多的牵引力，但我们很清楚：Active Directory是在企业IT 的普遍标准。这是一种生活方式，即使您和我在我们理想的应用程序开发世界中可能不必经常担心所有这些。并且有充分的理由！它为商务专业人员集成了整个Windows桌面体验。这是Office365故事的跳动的心脏，也是组织自身组织和自我构建的方式。想知道您是否获得了晋升吗？检查活动目录！是否想知道有人坐在哪里？检查活动目录！是否要强制重置密码？维护企业范围的审核日志？开除某人 ？检查活动目录！您可能以为您的组织运行Active Directory，但要明确一点：它运行您的组织。

Active Directory是Directoy服务器。它提供了用户，组织等等的树。它充当Microsoft CRM，Microsoft SQL Server，Microsoft Office甚至Microsoft Windows本身等技术的身份管理器。您可以在Active Directory中描述用户，他们的权限和角色等等。这使我们回来了。Microsoft Azure为您运行Active Directory！您可以直接从平台导入和配置Microsoft Active Directory安装的所有相关信息。甚至还有一个Spring Boot启动器，用于将Microsoft Azure连接到您的OAuth授权的Spring Boot和Spring Security驱动的应用程序 。您可以自己部署和管理Microsoft SQL Server或Microsoft Active Directory之类的东西吗？当然。但是， 你应该吗 ？

应用洞察

随着横向扩展和扩展更多的微服务，您将引入越来越多的移动部件，而能够观察数据在系统中从一个节点到另一个节点的移动就变得越来越重要。在这里，针对Spring应用程序的Microsoft Application Insights集成（目前至少与针对Microsoft Azure的主要Spring集成是分开提供的）使使用它非常容易！添加com.microsoft.azure ： applicationinsights-spring-boot-starter ： 1.1.0-BETA到您的构建文件。

您可以编写Application Insights订阅的脚本，如下所示：

#!/usr/bin/env bash

rg=$1
appname=${rg}-appinsights

az resource create \
    --resource-group $rg \
    --resource-type "Microsoft.Insights/components" \
    --name $appname \
    --location "South Central US" \
    --properties '{"ApplicationId":"bootiful","Application_Type":"web"}'

这将创建一个订阅并将其绑定到可bootiful资源组。使用以下命令访问生成的键（假定范围内相同的$rg和$appname变量）：

az resource show -g $rg -n $appname --resource-type "Microsoft.Insights/components" --query properties.InstrumentationKey

然后，您需要指定azure.application-insights.instrumentation-key并为您的应用程序指定spring.application.name名称。公平地说，您应该这样做。而已！重新启动您的应用程序，通过应用程序中的HTTP终结点驱动一些流量（甚至只是在localhost运行），然后登录到Microsoft Application Insights仪表板并观察实际操作！

云铸造

您不应运行无法付费的软件。关键人物詹姆斯·沃特斯（James Watters）经常谈论“价值线以下”的工作，以及技术人员和管理人员应专注于给定组织的价值线以上的工作的想法。普通组织有足够的问题。运行像MySQL或Kafka这样的商品软件不应该包括在其中。进行无法与您的企业脱颖而出的事情不应该包括在其中。它会无限地成本较低，在足够的时间，花钱找人来运行，不管它是什么，你如果别人可以做一个称职的工作。理想情况下，最终运行该软件的人都应该对良好运行这些东西有既得利益。除了配置SSL或调试MySQL复制问题外，大多数组织的任务都是其他。让别人替您这样做实际上总是比较便宜。

如果您尝试做一些标准的事情，那么您绝对应该依靠标准化的工具。当您可以使用诸如Cloud Foundry或Kubernetes之类的工具时，为什么使用云供应商的锁定软件来运行基于Java或Node的应用程序，因为它们的租用池更大，成本几乎为零，并且易于使用-等于或优于云供应商提供的产品？

这是Microsoft与Pivotal建立如此良好关系的重要原因。企业客户了解，对于某些工作负载，公有云已成定局，但他们不想被锁定。选择性很有价值。能够转移到另一个平台很有价值。敏捷-应对变化的能力-十分宝贵 。我们一直都在看这个；对于某些工作负载类型，Google Cloud Platform的价格比首次发布时的Amazon Web Services便宜。一些公共云供应商在您可能希望位于的区域中提供了可用性区域，而其他区域则没有。越来越多的组织开始了解混合云或mutlicloud策略，因为他们知道他们的工作量会变化，可用性需求也会变化。这些组织希望尽可能降低对各种云基础架构进行操作，保护，培训和部署的成本。

在这里，Pivotal Cloud Foundry是自然的选择。它使组织能够可靠地部署到许多云平台中的任何一个，并且在需要时可以轻松轻松地部署到特定于平台的服务。这是微软在2017年和2018年将Pivotal评为年度最佳合作伙伴的原因之一。Pivotal Cloud Foundry是我们经Apache 2许可的开源Cloud Foundry项目的发行版，有助于在Microsoft Azure上提高资源使用率，并且以可移植的方式做到这一点，同时仍可提供宝贵而独特的资源。

我全力以赴使用Microsoft Azure之类的东西来简化基础架构的工作，Microsoft可以提供与众不同的体验。但是运行Java进程还是Node.js进程？坚持使用事实上的标准基础架构，例如Cloud Foundry或Kubernetes。将Cloud Foundry部署在Microsoft Azure之上是相当琐碎的，一旦部署，在其中部署Spring Boot应用程序将变得很简单。它通常和cf push -p my.jar一样简单。当然 ，您将需要Cloud Foundry CLI。

在Cloud Foundry的世界中， 服务代理是HTTP API，Cloud Foundry平台与该HTTP API进行交互以管理某物的提供和生命周期。它可以是MySQL之类的数据库，也可以是Apache Kafka之类的消息队列。它可能是Active Directory安装或CosmosDB连接。可能是任何事情。Microsoft和Pivotal团队携手合作，以确保在Microsoft Azure上运行Pivotal Cloud Foundry的用户的服务代理选项尽可能方便地支持tentpole服务 。此列表包括Microsoft Azure服务，例如Azure存储，Azure Redis缓存，Azure服务总线，Azure事件中心，Azure SQL数据库（SQL Server，PostgreSQL和MySQL）和故障转移组，以及Azure CosmosDB。

您可以通过在CLI上发出cf marketplace命令来检查Cloud Foundry服务目录。它会向您显示所有相关服务，然后您可以选择设置服务实例及其相关计划。计划描述了您可以从服务代理资源中预期的特定服务级别，并且自然是针对所提供的每个服务代理的。

假设您已将一个Spring Boot应用程序部署到Cloud Foundry，并使用了逻辑名称myapp ：

> cf push -p my.jar --no-start myapp

假设您想为您的应用程序提供Microsoft SQL Server实例，并且希望能够逻辑上将该数据库引用为mydb 。您的会话可能如下所示。以下命令将使用通过内联JSON或.json文件路径指定的参数来提供SQL Server实例，并为其分配逻辑名称mydb 。

> cf create-service azure-sqldb basic mydb -c '{"sqlServerName": "Bootiful"}'

该服务很快将在应用程序内提供并可用。您只需要将服务绑定到您的应用程序，以便可以将相关的连接信息作为环境变量注入到应用程序的环境中。

cf bind-service myapp mydb
cf restart myapp

现在，您正在运行的Spring Boot应用程序将具有一个环境变量VCAP_SERVICES ，其内容将包括刚刚提供的服务的连接信息。看起来像这样：

在Spring Boot中，您可以使用扁平化的属性访问语法来引用这些属性，例如： vcap.services.mydb.credentials.jdbcUrl 。这里的一种常见模式是在Spring配置文件处于活动状态的情况下在云中运行应用程序。说， cloud ？这样，您可以在application-cloud.properties cloud.properties下的代码中放置一个配置文件，并且当应用程序在Cloud Foundry中启动时将加载该属性文件。您可以将默认的本地配置放在application-default.properties 。所以，当春天开始启动，没有配置文件中指定它会加载在配置application-default.properties 。在Azure上的Cloud Foundry中运行时，它将在application-cloud.properties加载配置。因此，您可以将以下内容添加到application-default.properties文件中。

spring.datasource.url=${vcap.services.mydb.credentials.jdbcUrl}
