Spring Data Moore有哪些新功能？

工程
克里斯多夫·斯特罗布尔
十月08，2019

Spring Data Moore附带16个模块，已完成700多张票。它包括整个产品组合中的大量改进和新功能，并且重点关注三个主要主题：Reactive，Kotlin和Performance。该版本增加了诸如声明式反应式事务和协程/流程支持等功能，并提供了高达60％ *更快的查找器方法。

让我们从Moore的一些Reactive功能开始。

声明式，被动交易

Lovelace版本以封闭式的方式引入了对响应式交易的早期支持，为改进留有余地。以下清单显示了该样式：

Lovelace中的反应式交易（使用MongoDB）
public Mono doSomething(Long id) {

  return template.inTransaction().execute(txTemplate -> {

    return txTemplate.findById(id)
      .flatMap(it -> start(txTemplate, it))
      .flatMap(it -> verify(it))
      .flatMap(it -> finish(txTemplate, it));

  }).next();
}

在前面的代码段中，必须通过显式调用来启动事务inTransaction()在闭包内使用一个事务感知模板，调用next()最后把返回的Flux变成一个Mono满足方法签名，即使findById(…)已经只发出一个元素。

显然，这不是进行被动交易的最直观的方法。因此，让我们看看使用声明式反应式事务支持的相同流程。与Spring的交易支持一样 ，您需要一个组件来为您处理交易。对于被动交易， ReactiveTransactionManager目前由MongoDB和R2DBC模块提供。以下清单显示了这样的组件：

@EnableTransactionManagement
class Config extends AbstractReactiveMongoConfiguration {

  // …

  @Bean
  ReactiveTransactionManager mgr(ReactiveMongoDatabaseFactory f) {
    return new ReactiveMongoTransactionManager(f);
  }
}

从那里，您可以使用@Transactional并依靠基础架构来启动，提交和回滚事务流，以通过Reactor Context处理生命周期。这使您可以将代码从Lovelace转换为以下清单，从而无需使用范围模板和多余的闭包Flux至Mono转型：

摩尔中的声明式反应式事务（使用MongoDB）
@Transactional
public Mono doSomething(Long id) {

  return template.findById(id)
    .flatMap(it -> start(template, it))
    .flatMap(it -> verify(it))
    .flatMap(it -> finish(template, it));
}

响应式Elasticsearch存储库

可以在社区模块之一中找到对反应式系列的另一个显着补充，Spring Data Elasticsearch现在提供了基于完全反应式Elasticsearch REST客户端而构建的反应式模板和存储库支持，而该客户端又是基于Spring的。 WebClient 。

客户端通过在Java High-Level REST Client附近公开一个熟悉的API，为日常搜索操作提供一流的支持，并在需要时进行必要的削减。模板和存储库API的组合使您可以根据需要无缝过渡到响应式而不会迷失方向。以下清单显示了如何配置Elasticsearch以使用反应式客户端：

反应性Elasticsearch
class Config extends AbstractReactiveElasticsearchConfiguration {

  // …

  @Bean
  public ReactiveElasticsearchClient reactiveClient() {
    return ReactiveRestClients.create(localhost());
  }
}

@Autowired
ReactiveElasticsearchTemplate template;

//…

Criteria criteria = new Criteria("topics").contains("spring")
    .and("date").greaterThanEqual(today())

Flux result = template.find(new CriteriaQuery(criteria), Conference.class);

反应式查询

谈到在过渡中迷路： Querydsl （←普通HTTP / NO HTTPS）提供了一种出色的方式来定义多个数据存储的类型安全查询，并且已经有相当长的一段时间支持非反应性数据访问。为了在响应式场景中提供支持，我们添加了响应式执行层，可让您运行Predicate支持的查询。的ReactiveQuerydslPredicateExecutor ，添加到资源库界面后，将提供所有入口点，如以下示例所示：

反应式查询
interface SampleRepository extends …, ReactiveQuerydslPredicateExecutor<…> {
  // …
}

@Autowired
SampleRepository repository;

// …
Predicate predicate = QCustomer.customer.lastname.eq("Matthews");
Flux result = repository.findAll(predicate);

支持Kotlin协程和MongoDB标准API DSL

沿着Moore不断增强的反应性支持，我们延续了从Lovelace Release开始的Kotlin故事。特别是，我们通过以下提供的方法为Kotlin协程和流程提供了几种扩展： awaitSingle()和asFlow() 。以下方法使用awaitSingle()方法：

Kotlin协程支持
val result = runBlocking {
  operations.query()
   .matching(query(where("lastname").isEqualTo("Matthews")))
   .awaitSingle()
}

社区贡献了使用Kotlin语言功能的另一项重大改进，为Spring Data MongoDB标准API添加了类型安全的查询DSL。这使您可以转换诸如query(where("lastname").isEqualTo("Matthews"))改为以下符号：

Kotlin类型安全查询
val people = operations.query()
  .matching(query(Person::lastname isEqualTo "Matthews"))
  .all()

性能提升

在设计所有这些新功能的同时，我们还花了一些时间来研究当前实现的潜在瓶颈，并找到了一些需要改进的地方。这包括摆脱Optional ，可以在许多地方捕获lambda和流执行，添加缓存并避免不必要的查找操作。最终，基准测试表明，JPA单属性查找器方法的吞吐量提高了近60％，例如findByTitle(…) 。

这很棒，值得花时间！但是，我想清楚一点，所有基准测试都使用无尘室方案，这些方案可以避免任何开销。如果将它们移至更真实的场景（例如，通过将内存中的H2数据库替换为实际的可用于生产的数据库），则结果看起来会大相径庭，因为性能限制因素转移到了网络交互，查询执行和结果上传输。改进仍然可见，但通常降至个位数百分比。基准可在此GitHub存储库中找到。

新实体回调API

我们还改进了现有的挂钩，以通过在持久性操作期间远离当前对象来拦截实体的生命周期。 ApplicationEvent基于方法的更直接的交互模型。的EntityCallback API引入了对不可变类型的更好支持，提供了运行时保证，还无缝集成到了响应流中。当然，我们仍然支持和发布ApplicationEvents ，但我们强烈建议切换到EntityCallbacks何时应更改已处理实体。

在以下示例中， BeforeConvertCallback通过使用修改给定的不可变实体wither分配一个id到实体的副本，然后将其返回，并在下一步中转换为商店特定的表示形式：

EntityCallback API
@Bean
BeforeConvertCallback beforeConvert() {

  return (entity, collection) -> {
    return entity.withId(…);
  }
}

除了与ApplicationEvents （可以使用AsyncTaskExecutor ，在执行操作时几乎保持打开状态）， EntityCallback API保证在实际事件触发之前立即调用。即使在反应流中。以下清单显示了它的工作方式：

响应式实体回调API
@Bean
ReactiveBeforeConvertCallback beforeConvert() {

  return (entity, collection) -> {
    return Mono.just(entity.withId(…));
  }
}

支持Redis流

说到流，Spring Data Redis现在支持Redis流，该流几乎与反应流无关，而是一种新的Redis仅追加数据结构，该模型对日志进行建模，其中每个条目都包含一个id（通常是一个时间戳加上一个序列号）和多个键/值对。与通常的可疑对象（例如添加到日志中并从中读取日志）一起，Spring Data Redis提供了允许无限监听和处理添加到日志中的条目的容器。它像tail -f但对于Redis Stream。以下示例显示了Redis流侦听器：

Redis Streams侦听器
@Autowired
RedisConnectionFactory factory;

StreamListener> listener =
  (msg) -> {

    // … msg.getId()
    // … msg.getStream()
    // … msg.getValue()
  };

StreamMessageListenerContainer container = StreamMessageListenerContainer.create(factory));

container.receive(StreamOffset.fromStart("my-stream"), listener);

的StreamMessageListenerContainer在前面的示例中，读取了以下所有现有条目my-stream并收到有关新添加的通知。对于收到的每条消息， StreamListener被调用。单个容器可以接收来自多个流的消息。

当然，反应式基础结构最好使用类似流的结构，如以下示例所示：

StreamReceiver receiver = // …

receiver.receive(StreamOffset.fromStart("my-stream"))
  .doOnNext(msg -> {
      // …
  })
  .subscribe();

JPA存储过程的多个输出参数

在JPA方面，现在有了微小的改进，您可以拥有多个OUT存储过程的参数，这些参数在Map 。以下示例显示了如何执行此操作：

JPA存储过程的输出参数
@NamedStoredProcedureQuery(name = "User.s1p", procedureName = "s1p",
  parameters = {
    @StoredProcedureParameter(mode = IN, name = "in_1", type = …),
    @StoredProcedureParameter(mode = OUT, name = "out_1", type = …),
    @StoredProcedureParameter(mode = OUT, name = "out_2", type = …)})
@Table(name = "SD_User")
class User { … }

interface UserRepository extends JpaRepository<…> {

  @Procedure(name = "User.s1p")
  Map callS1P(@Param("in_1") Integer arg);
}

JPA声明的所有out参数@StoredProcedureParameter注释最终将在Map由存储库查询方法返回。

关于存储库方法的声明性MongoDB聚合

使用MongoDB，复杂的数据处理是通过聚合来完成的， 聚合为Spring提供了特定的（流利的）API，并对操作和表达式进行了抽象。但是，Stackoverflow告诉我们人们倾向于在命令行上进行聚合，然后将其转换为Java代码。该翻译原来是一个主要的痛点。
所以我们借此机会介绍了@Aggregation作为在存储库方法中运行聚合的直接方法。以下示例显示了如何执行此操作：

声明式MongoDB聚合
interface OrderRepository extends CrudRepository {

  @Aggregation("{ $group : { _id : '$cust_id', total : { $sum : '$amount' }}}")
  List totalByCustomer(Sort sort);

  @Aggregation(pipeline = {
    "{ $match : { customerId : ?0 }}",
    "{ $count : total }"
  })
  Long totalOrdersForCustomer(String customerId);
}

像它的亲戚一样@Query注解， @Aggregation支持参数替换，并在查询方法参数提供的情况下将排序添加到聚合中，如前面的示例所示。我们甚至更进一步，为返回简单类型的方法（例如上一个示例中的totalOrdersForCustomer方法）提取单属性文档值。的$count在这种情况下，阶段返回一个文档，例如{"total" : 101 }通常需要映射到任一org.bson.Document或相应的域类型。但是，由于该方法声明Long作为返回类型，我们检查结果文档并从那里提取/转换值，从而消除了对专用类型的需求。
