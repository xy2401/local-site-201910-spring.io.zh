简化Spring Cloud发布流程

发布
斯宾塞·吉布
七月24，2019

Spring Cloud继续证明非常受欢迎，并且在最近几年中，许多IaaS提供商已经提供了与其技术的集成，并加入了发布培训。这通常涉及进入spring-cloud GitHub org并在org.springframework.cloud Maven groupid中发布。随着包含的项目数量的增加，它变得有点笨拙，我们希望退后一步，回顾一下该模型提供的利弊，并提出一条使所有相关项目受益的更好的发展道路。

成为发布流程一部分的最大缺点可能是，它消除了项目维护人员的控制水平。由于核心Spring Cloud团队负责发布，因此维护人员无法按照他们所集成的技术的时间表进行工作。或者，即使他们确实设法使服务刷新，在发布下一个发行版本之前，发行火车也不会使用它。该模型还意味着，通常项目维护者无法直接访问关键统计信息，例如其项目的Maven下载编号。

另外，似乎很容易成为发行培训的一部分，但实际上，许多好处是任何项目都可以使用的，而不是与培训或在spring-cloud GitHub org中有任何关系：

  • Spring Cloud团队定期与外部项目合作，以帮助对其进行审核，并就如何最佳地与Spring Cloud集成提供建设性反馈。包含在发布系列中并不会或多或少会发生这种情况。（如果需要反馈，请联系。）
  • 有一种看法认为，这可以使在start.spring.io上获得难以捉摸的复选框变得更加容易。这不是真的。对于如何确保开发一个适合于start.spring.io的项目，存在一些指导原则 ，但是有关包含的决策不仅仅遵循命名方案。我们是否发现社区正在拥抱与该项目集成的技术？该项目是否建立了一个充满活力和可持续发展的社区，从而确保了长寿和健康的代码库？是否有足够的文档，指南等说明如何使用该项目？指南中的任何内容都不需要包含在Spring GitHub组织中。
  • Spring团队会定期制作涉及第三方项目的内容，重点介绍关键版本或增强功能。为此，无需成为发布流程的一部分。
    https://spring.io网站显然包含涵盖主要Spring Cloud项目的内容，但还包含涵盖某些第三方项目的内容。这可能是显示如何使用该技术的指南，也可能是现有项目页面之一上的条目。参见https://spring.io/projects/spring-cloud ，它涉及许多项目-其中一些在培训中，而另一些不在。

出于这些原因，Spring Cloud团队决定采用一种简化发布流程的模型，使我们能够在保持所有可观收益的同时解决弊端。提议是IaaS提供者将在自己的GitHub组织中托管和维护其代码。这适用于当前发布系列中的所有现有集成。Spring Cloud Azure从未参加过培训，Spring Cloud Alibaba团队从孵化器组织毕业后已经开始采用这种新模型。

实际上这意味着什么？

从Spring Cloud GitHub组织中删除这些项目中的某些项目以及发布火车，并不是关于有效开发或支持政策发生变化的声明。这些团队将与Spring Cloud版本保持同步。如果有的话，此更改可以使支持模型更加清晰：如果项目托管在IaaS提供者的GitHub组织中，则它们显然会提供支持。

新模型确实意味着组ID可能会更改，并且工件ID可能也会更改。软件包名称也可能会更新以反映这些更改。开发人员还需要在项目中明确包含这些依赖关系，而不是通过Spring Cloud BOM通过托管版本继承它们。

Spring Boot团队具有适当命名模块的准则 （自动配置和启动器）。这些不是针对Spring Cloud严格执行的，但是遵守准则可能意味着依赖该方案构建的任何内容都将“正常工作”（例如，IDE中的内容辅助工具）。

按照这种模式，阿里巴巴团队将从spring-cloud-incubator迁移到自己的GitHub org， 网址为https://github.com/alibaba/spring-cloud-alibaba ，我们将在本周继续与他们合作支持Spring Cloud Greenwich的重要版本。SR2和Finchley。SR4。此版本将包括对Spring Cloud Gateway的Sentinel支持，并使Nacos服务发现和Spring-Cloud-Config一起使用。注意这一点！

您可以期望看到其他提供者在将来的主要Spring Cloud火车发行期间有所变化，因为主要发行是进行此类重大更改的正确时机。

请注意，这仅是描述此时主要Spring Cloud列车的情况和计划；它不会影响其他项目（例如，Spring Cloud Stream活页夹）。

我们希望上面的推理能够说明我们要使用该策略实现的目标。与往常一样，我们乐于接受反馈-请让我们知道您的想法。在继续提供与我们在代码审查和项目推广方面与我们紧密合作的所有好处的同时，给予项目所有者充分的灵活性来构建和发布更适合其文化的软件，应该使所有人受益。
