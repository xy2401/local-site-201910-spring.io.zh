适用于Apache Geode和Pivotal GemFire 1.1.0的Spring Boot。发布＆1.2.0。M1发布！

发布
约翰·布鲁姆
2019年8月23日

我很高兴代表Spring和Apache Geode社区宣布发布适用于Apache Geode和Pivotal GemFire （SBDG）的Spring Boot。 1.1.0.RELEASE和1.2.0.M1发布。

SBDG 1.1.0.RELEASE在Maven Central中可用。

SBDG 1.2.0.M1在Spring Milestone存储库中可用。

新增功能1.2.0.M1

适用于Apache Geode和Pivotal GemFire （SBDG）的Spring Boot 1.2.0.M1是将SBDG与最新的GemFire / Geode和Spring钻头对齐的基准版本：

  • Spring Framework 5.2.0。RC1

  • Spring Boot 2.2.0。M5

  • Spring Data Moore-RC2 / 2.2.0。RC2

  • 春季会议Corn-M3 / 2.2.0。M3

使用Spring Data Moore，它还可以引入：

  • Apache Geode 1.9.0

  • 关键GemFire 9.8.3

往回看

适用于Apache Geode和Pivotal GemFire （SBDG）的Spring Boot 1.1覆盖了很多地面。在此发行版中，我们包含了许多重要的更改和功能，以帮助开发人员保持生产力并解决重要的相关问题。

让我们回顾一些重点和值得一提的地方。

新的依赖

SBDG 1.1是根据：

  • Spring Framework 5.1.9。发布

  • Spring Boot 2.1.7。发布

  • Spring Data Lovelace-SR10 / 2.1.10。发布

  • Spring Session Bean-SR7 / 2.1.7。发布

使用Spring Data Lovelace，它可以实现：

  • Apache Geode 1.6.0

  • 关键GemFire 9.5.4

有关版本的更多详细信息，请参见SBDG 版本兼容性列表 。

支持缓存用例和模式

SBDG 1.1现在包括对3种最常见的缓存设计模式的专门支持：

  • 后备缓存以及指南和示例代码可帮助您应用此模式。

  • 内联缓存 ，结合Spring数据存储库的强大功能，还提供了指南和示例代码，以帮助您应用此模式。

  • Near Caching以及指南和示例代码可帮助您应用此模式。

  • 此外，我们介绍了一种最常见的缓存用例HTTP会话状态缓存 ， 它又包括指南和示例代码 。

很难争辩，随着云环境中微服务的激增，缓存将成为用于Cloud-Native开发的最常见和最重要的模式之一。

从关系数据库访问数据时，不再仅需要缓存来提高性能。当微服务成为您的应用程序用来访问信息的新数据源时，缓存将变得越来越普遍且至关重要。

这是因为网络天生就容易发生故障，并且每当发生网络跳变时都会造成重大的延迟损失。对于高负载的应用程序，这可能是致命的，从而导致各种级联故障（例如，内存问题）。目的是尽可能减少不必要的网络调用，这非常容易做到，尤其是在读取比例远远超过写入比例时。

缓存（HTTP）会话状态将继续是最常见的缓存形式。幸运的是，对于所有类型的会话（HTTP， WebSocket和WebSession （响应））来说， Spring Session的强大功能（特别是Apache Geode和Pivotal GemFire的Spring Session）来管理Spring Boot应用程序的Session状态非常简单。另外， Spring Session支持多种后备存储（ JDBC ， Redis ， MongoDB ， Hazelcast ，当然还有GemFire / Geode ），如果/当您的需求发生变化时，切换供应商变得很简单。

混合云部署

与SBDG 1.1 ，您现在可以将Spring Boot应用程序部署到Pivotal CloudFoundry（PCF） ，但仍将这些应用程序连接到外部数据源。

当必须保护数据并将其保留在本地时，混合云安排很常见。对于试图迁移到云并使堆栈现代化的公司来说，这也是非常常见的中间步骤。混合部署使公司能够采用增量方法来采用云，而不是全有或全无。

因此，SBDG 提供了对在PCF中运行Spring Boot应用程序的一流支持 ，但仍将这些应用程序连接到外部管理的独立Apache Geode或Pivotal GemFire集群。

在相关说明中，除了混合云部署外，SBDG还支持：

  • 连接到多个Pivotal Cloud缓存实例

  • 关键云缓存实例定位

进行切换

SBDG 1.1 （甚至是1.1之前的版本）一直专注于开发人员的生产力 ：帮助用户尽可能轻松 ， 快速 ， 可靠地 启动和运行 ！

该策略的一部分植根于帮助用户和客户：

  1. 从开源 （ Apache Geode ）切换到商业 （ Pivotal Cloud Cache ）

  2. 从非托管 （ 独立，外部 ）迁移到托管环境（ Pivotal Cloud Foundry ）

  3. 几乎 没有 代码或配置更改；它应该工作！

实际上，双向移动同样重要。从托管回到独立环境，从商业回到开源 。再次，它应该工作！

这就是转换的全部内容...... ``选择 ''，特别是``您的'' 选择 。

下一步是什么

现在所有焦点都转移到了SBDG 1.2 。我们将继续在几个领域进行投资。

  • 首先， 开发人员的生产率不可高估。

从技术上讲这意味着什么？

我们计划引入一个新的注释， @EnableClusterAware ，与您一起声明时@SpringBootApplication类，将使您能够从本地开发上下文（例如，IDE）迁移到客户端/服务器，而无需担心连接细节和集群配置。SBDG将为您解决！这将使环境之间的“ 切换 ”（例如DEV＆TEST / STAGING）更加容易，尤其是当您在调试和测试代码时快速迭代开发时。

  • 目标运行时用户支持 （请参阅问题＃44 ）

此外，我们将使PCF上下文中的开发人员能够在运行应用程序时将具有特定角色的用户分配给他们已部署的Spring Boot应用程序。这对于某些必须以只读模式运行应用程序的UC非常重要，其中应用程序只能读取数据，而不能修改数据。

  • 多站点缓存

为了补充Look-Aside ， Inline和Near Caching模式 ，我们将提供另一本指南以及示例代码，以展示多站点功能，应用于缓存的WAN拓扑的功能，以及如何帮助进一步提高应用程序的可用性和弹性。体系结构（考虑主动-主动 ， 主动-被动 ， 可用区等）。

  • 包含在Spring Initializer中

SBDG 1.2位正在进入start.spring.io 。在不久的将来，您将能够在开发Spring Boot应用程序时选择Apache Geode作为选项，这将方便地为您提供spring-geode-starter依赖性。
