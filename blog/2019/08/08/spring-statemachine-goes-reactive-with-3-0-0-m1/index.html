Spring Statemachine在3.0.0版本中开始响应。M1

发布
珍妮·瓦尔基亚拉赫蒂
八月08，2019

我很高兴地宣布Spring Statemachine 3.0.0的第一个里程碑版本。M1和这些话，我可以说Statemachine正在变得被动。

Statemachine本身不需要为自己的执行而做出反应，但是一旦机器脱离其受控环境执行诸如Actions和Guards之类的用户定义的逻辑，就无法保证这些功能不会被阻塞。

当世界的另一端变得更加被动时，那些使用数据库的持久性功能的用户将受益于被动更改。基本上，每次我们需要走出机器世界的一步时，我们都可能会遇到阻塞IO操作的情况。

2.x系列将在可预见的将来出现，其中3.x将受益于新变化。实际上，这取决于用户向2.x系列添加了多少新功能。

在2.x机器中，执行完全基于Spring TaskExecutor API。这些执行将机器从一种状态转移到另一种状态。在执行过程中，进入或退出状态时会执行各种用户级别的操作。守卫可以用来保护各种过渡路径，我们无法控制它们的作用。 TaskScheduler API通常与处于状态的触发器计时器和执行操作一起使用。

因为一切都基于TaskExecutor和TaskScheduler用户能够将它们切换到其他内容，以提供更多的并行执行和调度，这在处理需要并行执行的状态机区域时通常是必需的。所有这些使内部线程模型变得相对复杂，并且多年来已经花费了数小时来跟踪各种比赛条件。

现在，所有内容实际上都基于Reactor及其功能，这种内部线程模型变得更加可靠。我们真的不再需要使用锁定了，因为Reactor可以保证我们自己的流程执行简单。

我们有一种新的方法可以通过使用Mono和Flux消息。回来Flux然后获得更丰富的结果特定事件发生了什么。例如从返回StateMachineEventResult可以检查事件是否被接受或推迟，以及哪个地区处理了该事件。

Flux<StateMachineEventResult<S, E>> sendEvent(Mono<Message<E>> event);
Flux<StateMachineEventResult<S, E>> sendEvents(Flux<Message<E>> events);

在旧的不推荐使用的api中，您使用了类似以下内容：

boolean accepted = machine.sendEvent("EVENT");

现在什么都没有发生，直到您订阅了返回的Flux StateMachineEventResult 。

Message<String> event = MessageBuilder.withPayload("EVENT").build();

machine.sendEvent(Mono.just(event))
  .doOnComplete(() -> {
    System.out.println("Event handling complete");
  })
  .subscribe();

同样的故事继续着动作，只是一个Function服用StateContext然后返回一个Mono 。

public interface ReactiveAction<S, E>
  extends Function<StateContext<S, E>, Mono<Void>> {}

与警卫相同，期望您需要返回Boolean与一个Mono 。

public interface ReactiveGuard<S, E>
  extends Function<StateContext<S, E>, Mono<Boolean>> {}

现在是尝试并为下一个里程碑提供反馈的好时机。我要感谢所有社区的贡献，因为某些重构工作相对耗费人力。
