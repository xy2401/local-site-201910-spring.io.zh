Spring Batch和Spring Integration的实际使用

工程
戴夫·瑟
2010年2月15日

Spring Batch和Spring Integration的用户普遍存在一些担忧，我们被问到了很多有关它们如何组合在一起的问题。Spring Batch管理员 1.0.0。M2是最近发布的，它大量使用了Spring Integration，因此它是查看某些特定用例的良好工具，而这正是我们计划在本文中进行的工作。

Spring批处理集成

1.0.0的一部分。M2版本是Spring Batch集成模块，最近从Spring Batch迁移而来，并通过Batch Admin获得了新家。Spring Batch Integration中实现或演示了许多Batch-Integration交叉用例。之所以要使用新版本，是因为Batch Admin使用了Batch Integration的许多功能，因此调整这些项目的发布周期更加有意义。

Spring Batch管理员

Spring Batch Admin是SpringSource的一个开源项目。它旨在为开发人员提供一个Web UI和用于构建自己的UI以便与Spring Batch作业进行交互（启动，停止，调查失败原因等）的工具。就计划功能1.0而言，最新的里程碑版本已经相当完善，但是如果您有任何想法或贡献，请访问论坛和问题跟踪器，并参与社区活动。

开箱即用的目标运行时是servlet容器（例如SpringSource tc Server）的单个实例，并且在该容器中，系统使用零配置或无配置运行。但是我们希望能够支持基本用例的自定义和扩展，包括将部署扩展到服务器集群，并且事实证明Spring Integration是许多扩展点的关键。

批处理和集成相结合

Spring Batch和Spring Integration之间的界线并不总是很清楚，但是有一些准则可以遵循。原则上，这些是：考虑粒度，并应用通用模式。本文介绍了其中一些常见的模式。更多信息在Spring Batch Integration和Spring Batch Admin中实现（并且可能是以后的文章的主题）。

将消息添加到批处理过程中可以实现操作的自动化，以及关键问题的分离和制定策略。例如，一条消息可能触发作业执行，然后可以通过多种方式公开消息的发送。或者，当作业完成或失败时，可能会触发消息发送，而这些消息的使用者可能会遇到与应用程序本身无关的操作问题。

反过来也行得通：消息传递也可以嵌入到作业中，但这超出了本文的范围。例如：通过通道读取或写入要处理的项目。

以下是使用Spring Integration和Spring Batch Integration在Batch Admin中实现的一些用例。

模式：消息触发器

Spring Integration的优点在于将消息生产者和消息使用者之间的关注点分离开来，其中一个很好的具体示例是消息触发Job执行的能力。在这种情况下，使用者是完全通用的，并且是围绕标准Spring Batch JobLauncher的非常薄的包装器（此处的代码来自Spring Batch Integration JobLaunchingMessageHandler ）：

@ServiceActivator
public JobExecution launch(JobLaunchRequest request) {

    Job job = request.getJob();
    JobParameters jobParameters = request.getJobParameters();

    return jobLauncher.run(job, jobParameters);
    
}

从上面的代码片段中可以看到，包装程序非常薄，几乎不值得一提，但是它的优点是它具有非常清晰明显的责任感，并且很容易进行隔离测试。所述JobLaunchRequest目的是一种用于输入参数到JobLauncher一个特殊的包装，从而使它们可以形成在Spring集成一个消息的有效载荷。

JobLaunchingMessageHandler连接到Spring Batch Admin中的MessageChannel （在Manager jar /META-INF/bootstrap/integration/launch-context.xml中 ）：

<service-activator input-channel="job-requests">
    <beans:bean class="org.springframework.batch.integration.launch.JobLaunchingMessageHandler">
        <beans:constructor-arg ref="jobLauncher" />
    </beans:bean>
</service-activator>

那就是这种集成模式的消费者方面。这是一种本地方法，因为它不适用于远程调用，因为JobLaunchRequest是有意不可序列化的 （因为Job不可）。

要在本地启动Job ，我们需要做的就是创建一个生产者，并使用它将JobLaunchRequest发送到job-requests频道。Batch Admin Manager模块中有一个集成测试可以做到这一点，但是集成方法的真正功能是能够对请求进行战略制定并使其来自各种不同的生产者。

模式：频道重用

可以通过多种方式将消息发送到Batch Admin中的作业请求通道。要打破本地调用并远程公开作业，所需要做的就是使传入的请求以某种其他形式适应JobLaunchRequest ，Spring Integration使得此操作非常容易。这是我们称为通道重用的模式的基本方案。例如：

HTTP：浏览器

Spring Integration HTTP适配器模块可用于通过HTTP接受输入消息：

<http:inbound-channel-adapter name="/job-requests" channel="job-launches" 
    request-mapper="bodyInboundRequestMapper" view="reload-job-executions" />

可以在Batch Admin Manager模块（ META-INF / servlet / integration-servlet.xml ）中找到此代码片段。它公开了一个端点URL http：//.../batch/job-requests ，我们可以通过在浏览器中提交表单来发送作业执行请求。

原则上，请求可以采用我们喜欢的任何形式，因为我们可以在此适配器的下游和JobLaunchingMessageHandler的上游转换消息。在Spring Batch Admin中，转换是由另一个POJO消息处理程序（ StringToJobLaunchRequestAdapter ）在适配器的输出上完成的。

HTTP：命令行

上面使用的同一HTTP适配器可用于从UN * X命令行远程启动作业。这是使用Spring Integration HTTP适配器的一种非常不错的方法：您可以使用简单的shell脚本自动执行很多操作。例如，如果应用程序是通过名为“ staging”的作业在本地部署的，则此方法有效：

$ echo staging[input.file=foo] | curl -v -d @- -H "Content-Type: text/plain" \
  http://localhost:8080/springone-web-demo/batch/job-requests

使用一个参数（ input.file = foo ）启动名为“ staging”的作业，其中foo是要读取为输入的文件的绝对路径。作业配置有以下项目读取器：

<bean id="reader" class="org.springframework.batch.item.file.FlatFileItemReader" scope="step">
    <property name="linesToSkip" value="1" />
    <property name="lineMapper">
        <bean class="org.springframework.batch.item.file.mapping.PassThroughLineMapper"/>
    </property>
    <property name="resource" value="#{jobParameters[input.file]}" />
</bean>

（此代码段不在Spring Batch Admin示例中，而是在我们在Spring One Americas 2009中提供的演示中。）

文件轮询

Spring Integration可以轮询目录中的文件（使用文件适配器模块）。生成的消息仅需要针对工作请求通道进行调整。Spring Batch Admin在一个简单的消息处理程序（ FileToJobLaunchRequestAdapte ）中执行此操作：

public JobLaunchRequest adapt(File file) throws NoSuchJobException {
    JobParameters jobParameters = new JobParametersBuilder().addString(
            "input.file", file.getAbsolutePath()).toJobParameters();
    return new JobLaunchRequest(job, jobParameters);
}

这个简单的POJO方法被声明为@ServiceActivator （它可能是@Transformer ），因此可以在JobLaunchingMessageHandler之前的消息处理链中插入它，以将File转换为JobLaunchRequest 。

重新开始

失败的作业通常可以在Spring Batch中重新启动，并且可以通过Spring Batch Admin UI中的Web浏览器使用此功能。它也可以在命令行上使用，或者可以将消息发送到Spring Integration通道（称为job-restarts）的任何人使用 ：

<channel id="job-restarts" />
    <service-activator input-channel="job-restarts" output-channel="job-requests">
    <beans:bean class="org.springframework.batch.admin.integration.JobNameToJobRestartRequestAdapter">
        <beans:property name="jobLocator" ref="jobRegistry" />
        <beans:property name="jobExplorer" ref="jobExplorer" />
    </beans:bean>
</service-activator>

该通道所需的全部是作业名称，它已作为HTTP入站端点公开，因此可以从UN * X命令行执行此操作：

$ echo staging | curl -v -d @- -H "Content-Type: text/plain" \
  http://localhost:8080/springone-web-demo/batch/job-restarts

重试

如果作业重复失败并出现可恢复的错误（例如在调用远程服务时发生超时或网络故障），则可能希望自动重新启动。重试可以使用Spring Batch的某些功能在较低的级别进行处理，但是重试整个作业需要对运行时进行一些操作。既然job-requests通道正在接受启动请求，这可以通过Spring Integration轻松完成。为此的端点将充当筛选器，以查找已知可重试的作业中的失败条件，然后充当重新启动转换程序（如上述示例）。因此，像这样的链将起作用：

<chain input-channel="input-files" output-channel="job-requests" 
        xmlns="http://www.springframework.org/schema/integration">
    <filter>
        <bean class="...RetryableJobExecutionFilter" 
            xmlns="http://www.springframework.org/schema/beans">
            <property name="pattern" value="(&amp;s).*TimeoutException.*" />
        </bean>
	</filter>
    <service-activator>
        <bean class="org.springframework.batch.admin.integration.FileToJobLaunchRequestAdapter" 
            xmlns="http://www.springframework.org/schema/beans">
            <property name="job" ref="job1" />
        </bean>
    </service-activator>
</chain>

RetryableJobExecutionFilter可以这样实现

public boolean isRetryable(JobExecution jobExecution) {
    boolean retryable = false;
    for (StepExecution stepExecution : jobExecution.getStepExecutions()) {
        if (stepExecution.getStatus().isLessThan(BatchStatus.STOPPED)) {
            continue;
        }
        if (stepExecution.getExitStatus().getExitDescription().matches(pattern)) {
            retryable = true;
            break;
        }
    }
    return retryable;
}

这个例子在我们的Spring One演示中；尽管可以轻松实现所需的任何特定过滤器，但它不在Spring Batch Admin中。

输入文件上传

通过Spring Batch Admin UI直接支持将文件上传到应用程序。不建议使用HTTP POST上传大文件，主要是因为应用程序必须将内容缓冲在内存中，但这对于上传中小型数据集以供Spring Batch处理是一个很好的功能。

该示例应用程序实际上并未使用Spring Integration中的文件轮询器（但是，如上所述，它可用于希望对其进行配置的客户端）；而是在文件上传后使用直接消息触发器。策略是使Manager模块上载文件，然后将消息发送到发布-订阅通道（ input-files ）。

可以使用输入文件的任何作业都只需让上游组件订阅该通道，然后在感兴趣的情况下传递该文件。在此示例中，通过按文件的父目录名称过滤文件来完成此操作：

<chain input-channel="input-files" output-channel="job-requests" 
        xmlns="http://www.springframework.org/schema/integration">
    <filter expression="payload.parent.name=='sample'" />
    <service-activator>
        <bean class="org.springframework.batch.admin.integration.FileToJobLaunchRequestAdapter" 
            xmlns="http://www.springframework.org/schema/beans">
            <property name="job" ref="job1" />
        </bean>
    </service-activator>
</chain>

如果输入文件具有父目录（可以在Web UI中设置）“样本”，则将其通过管道传送到服务激活器中，该服务激活器将其转换为JobLaunchRequest并发送给JobLaunchingMessageHandler进行处理， 如前所述 。

模式：POJO消息处理

按照Spring应用程序的最佳传统，在Batch Admin Manager中通过简单的POJO和基于接口的组件将文件发送到输入文件通道。有一个FileService接口和一个本地实现，该接口使用一个临时目录来封送通过HTTP传入的文件。文件上传后，服务便会通过具有自定义界面的简单消息传递网关将其发送给服务：

public interface FileSender {
    
    void send(File file);

}

该接口没有实现（单元测试中的存根除外），因为Spring Integration可以提供以下一种：

<gateway id="fileSender" 
    service-interface="org.springframework.batch.admin.service.FileSender"
    default-request-channel="input-files" />

<beans:bean class="org.springframework.batch.admin.service.LocalFileService">
    <beans:property name="fileSender" ref="fileSender" />
</beans:bean>

配置文件上传

Spring Batch Admin允许用户上传Spring配置文件，以便从UI启动和管理作业。这对于在运行时重新参数化作业非常有用，例如在运行一组性能测试以衡量各种性能调整的效果时，例如在步骤中更改提交间隔。

为了接受整理文件进行输入，我们使用消息通道，以便可以通过多种不同的输入方法重复使用它。这些配置进入一个称为job-configuration的通道：

<service-activator input-channel="job-configurations" output-channel="job-registrations">
	<beans:bean class="org.springframework.batch.admin.integration.JobConfigurationResourceLoader">
		<beans:property name="jobRegistry" ref="jobRegistry" />
	</beans:bean>
</service-activator>

这里的服务激活器仅接受Spring 资源并将其视为配置文件：加载ApplicationContext ，对其进行扫描以查找Job组件并将它们注册在提供的注册表中。一旦进入注册表，就可以如上所述从UI的主Jobs菜单中启动作业，也可以通过Job-requests通道启动作业 。

就像输入文件一样，配置文件可以通过HTTP发送，在这种情况下，可以作为文件附件或纯文本参数，也可以通过文件轮询。轮询用例是在Manager模块中实现的，因此值得一看以了解其工作原理。在META-INF / bootstrap / integration / configuration-context.xml中，我们找到以下内容：

<file:inbound-channel-adapter directory="target/config" channel="job-configuration-files"
	filename-pattern=".*\.xml">
	<poller max-messages-per-poll="1">
		<cron-trigger expression="5/1 * * * * *" />
	</poller>
</file:inbound-channel-adapter>

适配器将轮询目录（此处出于演示目的而硬编码为“ target / config”，但将在实际应用程序中进行参数化）并查找名称以“ .xml”结尾的文件。当匹配该模式的文件到达时，将其发送（作为java.io。File ）到job-configuration-files通道。消息从那里转换，以便文件成为资源 ，并且可以将其发送到作业配置通道。

模式：信息性消息

一旦开始使用Spring Integration消息来驱动许多应用程序功能，能够利用消息流来提供信息或报告通常会很有用。例如，在作业开始，停止（完成或失败）时发送消息会很有用。使用Spring Integration的MessagePublishingInterceptor可以轻松做到这一点。在Spring Batch Admin Manager中，拦截器配置为发送作业执行消息：


<aop:config>
	<aop:advisor advice-ref="jobMessagePublishingInterceptor" pointcut="execution(* *..Job+.execute(..))" />
</aop:config>

<bean id="jobMessagePublishingInterceptor" class="org.springframework.integration.aop.MessagePublishingInterceptor"
	xmlns="http://www.springframework.org/schema/beans">
	<constructor-arg index="0">
		<bean class="org.springframework.batch.admin.integration.TrivialExpressionSource" p:payload="#args[execution]" />
	</constructor-arg>
	<property name="defaultChannel" ref="job-operator" />
</bean>

每次执行Job时 ，AOP顾问程序都会将参数值（ JobExecution ）传递给Job-operator通道。感兴趣的各方然后可以订阅该频道并获取有关最近执行的消息的信息。除了将这些消息记录在控制台上并在UI中列出它们之外，Spring Batch Admin不会对这些消息进行任何处理。在Spring Batch Admin之上构建自己的应用程序的客户可能会发现有用的消息，它们可以将操作的结果通知操作员或报告系统。

设置信息消息可能会带来打开新应用程序功能的副作用：上面描述的Job retry功能是通过将端点挂接到侦听作业操作员通道的方法而用于Spring One的。
