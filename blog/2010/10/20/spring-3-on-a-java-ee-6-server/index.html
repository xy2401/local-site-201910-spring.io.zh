Java EE 6服务器上的Spring 3

工程
于尔根·霍勒（Juergen Hoeller）
2010年10月20日

Spring on Java EE 6-可行的组合？当您要部署Java EE 6服务器时，是否可以轻松继续使用Spring？在数小时前开始的今年的SpringOne会议版本上，我将再次展示有关Spring和EE 6的会议：现在重点介绍GlassFish 3作为实际可用的工具（到目前为止，生产环境中唯一可用的）EE 6服务器。作为预览，以下是该演示文稿中的四个主要注意事项……

1。像GlassFish 3这样的Java EE 6服务器是Spring 3的良好运行时环境

GlassFish 3提供了许多现成的中间件：Servlet 3.0，JSF 2.0，JPA 2.0以及传统的EE服务（如JTA和JMS）-非常适合在基于Spring的应用程序中使用。GlassFish还捆绑了一个JSR-303验证提供程序，JSR-330 API，JSR-250通用注释：以及与Spring 3.0一起使用的关键API。通过这种组合，Spring作为框架和服务器作为中间件平台的重叠是最小的：基本上仅是EJB和CDI组件模型（仅构成GlassFish 3代码库的5％），Spring为该模型提供了替代编程模型。 ，同时与下面的相同平台服务集成。

2。即使在Java EE 6服务器上，选择Spring也是完全自然的事情

在GlassFish 3上，许多开发人员将继续选择，例如，基于JSF的Wicket或GWT，基于EclipseLink的Hibernate以及基于EJB / CDI的Spring。这些决定将受到样式方面的考虑以及体系结构要求的驱动，尤其是所有这些都由可移植性问题驱动。所有这些框架都不是服务器的核心部分，因此必须与应用程序一起部署，但这几乎不会成为决定性因素。FWIW，它们在“重量”方面甚至处于同一水平：将Hibernate 3.6 jar的大小与Spring 3.0的大小进行比较（约4 MB）。

3。您是否愿意将目标环境限制在Java EE 6服务器上？

今天仅提供GlassFish 3作为最终版本。JBoss 6仍处于里程碑5； WebSphere 8处于早期测试阶段。另一方面，Spring 3在Tomcat 5、6和7以及J2EE 1.4服务器（例如WebSphere 6.1）和Java EE 5服务器（例如WebSphere 7）上运行。而且，如果EE 6服务器恰好在生产中，那么Spring 3就是“好公民”，它将使用尽可能多的EE 6服务器功能。除此之外，基于Spring 3的应用程序可以部署到云平台上，例如Google App Engine和VMforce…

4。世界正在超越Java EE 6，并且可以预见，也将超越Java EE 7。

看云，看NoSQL，看分布式缓存... Java EE 6规范捕捉了2009年初的最新技术-Spring 3.0也是如此。但是，自EE 6发布以来，替代趋势就移到了前排。Spring 3.1 / 3.2将在2011年朝着这个方向发展：全新的编程模型功能将可立即在现有部署平台（例如Tomcat 6或WebSphere 7）上使用，而无需通过服务器升级）。凭借这种固有的敏捷性，Spring将比缓慢的Java EE周期继续领先数年。
