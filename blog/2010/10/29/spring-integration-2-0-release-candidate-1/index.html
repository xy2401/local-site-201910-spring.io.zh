Spring Integration 2.0版本候选1

工程
马克·费舍尔
2010年10月29日

我们很高兴地宣布Spring Integration 2.0的第一个候选版本！
下载 | 参考手册 | Java文档

我以为我会借此机会提供一般性的“新内容”？指南。实际上，有太多的新功能和改进无法在一篇文章中涵盖所有功能，但我将重点介绍其中的一些亮点。随着我们接近2.0 GA版本，我们将发布更多博客。目前，此帖子大致基于我和Oleg上周在SpringOne上发表的会议 。该演示文稿大部分是由演示驱动的，并且该代码可在我们的Git存储库中找到 。

为了在此处提供一些结构，我将逐步介绍许多类别的功能……

Spring 3.0启用的增强功能

Spring Integration 2.0直接在Spring 3.0上构建。实际上，RC1版本是在上周发布的Spring 3.0.5的基础上构建的。在这里，我们将看到通过对基础框架的重大升级而启用的一些最显着的功能。

Spring Expression Language

现在，您可以在转换器，路由器，过滤器，拆分器，聚合器，服务激活器以及Spring Integration核心名称空间的许多其他元素中使用SpEL表达式。例如，如果您可以基于有效负载对象中的简单属性评估“过滤器”规则，则只需执行以下操作：


<filter input-channel="numbers" expression="payload > 0" output-channel="positives"/>

在许多情况下，“表达式”属性可以代替对“引用”和“方法”以及引用所指向的POJO的使用。另一方面，即使您确实需要在Spring上下文中的bean上执行一个方法，简单的“ method”属性也可能不够。例如，您可能需要控制多个参数传递给该方法。尽管方法参数上的@Payload和@Header批注提供了一个选项，但有时最好将此类详细信息保留在配置中而不是代码中。这是一个使用SpEL BeanResolver策略而不是'ref'的示例：


<router input-channel="accounts"
    expression="@accountService.getAccountType(payload.accountId, payload.address.country)"/>
ConversionService和Converter

您可以定义一个名为“ integrationConversionService”的ConversionService Bean，然后注册您想要的任何Converter （必要时包括您自己的自定义实现）。在Spring Integration中有两个地方可以使用ConversionService。为了进行急切的转换，可以在元素上添加“数据类型”属性， 并提供要在该通道上允许的类的全限定名称。这是一个例子：


<channel id="datatypeChannel" datatype="example.Foo"/>

<beans:bean id="integrationConversionService"
            class="org.springframework.context.support.ConversionServiceFactoryBean">
    <beans:property name="converters">
        <beans:bean class="conversion.StringToFooConverter"/>
    </beans:property>
</beans:bean>

在评估SpEL表达式期间，“ integrationConversionService”做出了重要贡献。每个表达式都是根据EvaluationContext评估的，并且该上下文知道“ integrationConversionService”。在Spring Integration中，我们依赖于SpEL进行所有方法调用（不仅在使用“表达式”属性时，甚至在“参考”和“方法”配置中）。这意味着您可以利用绑定到Message实例的方法调用的值的运行时类型转换。例如，如果您正在调用FooService但在Message有效负载中传递Bar对象，则只需向“ integrationConversionService”注册BarToFooConverter。此示例还显示了一种注册转换器的更简洁的方法。通过新的“ converter”元素使用名称空间支持，您甚至不需要显式定义“ integrationConversionService” bean：


<!-- FooService.process(Foo) will be invoked, even though the Message payload is a Bar instance -->
<service-activator input-channel="in" ref="fooService" method="process" output-channel="out"/>

<converter>
    <beans:bean class="example.BarToFooConverter"/>
</converter>
TaskScheduler和触发器

Spring Integration 1.0具有自己的TaskScheduler实现以及基于时间间隔和基于cron的Trigger实现。既然Spring 3.0提供了TaskScheduler和Trigger，我们就可以依靠它们了。实际上，我们能够从Spring Integration方面删除大量代码，同时将其中一些代码贡献回底层框架（例如，轻量级的CronTrigger实现）。任何轮询使用者都可以配置一个显式子元素（否则，它将依赖于单个默认轮询器配置）。这些轮询器元素接受以下属性之一：“固定延迟”，“固定速率”或“ cron”。实际上，您甚至可以提供Spring的Trigger接口的自定义实现，然后使用“ trigger”属性作为参考。这是带有cron表达式的轮询器的示例：


<file:inbound-channel-adapter directory="/some/path">
    <poller cron="*/10 * 9-17 * * MON-FRI"/>
</file:inbound-channel-adapter>
RestTemplate和HttpMessageConverter

我们的出站HTTP适配器现在委托给Spring的RestTemplate来执行HTTP请求并处理其响应。这意味着HttpMessageConverter策略扮演着核心角色。我们默认情况下启用了多个转换器，但是您可以完全自定义列表和/或添加自己的实现。实际上，我们在HTTP入站适配器中也使用了相同的HttpMessageConverter策略。我们利用RestTemplate的另一个不错的功能是使用URI占位符。以下HTTP出站消息传递网关示例演示了自定义HttpMessageConverter和URI占位符的用法。实际上，在URI中替换的值是在运行时针对消息有效负载的SpEL评估的结果：


<http:outbound-gateway id="trafficService" 
        url="http://example/traffic/{zipCode}" 
        request-channel="requestChannel"
        reply-channel="responseChannel" 
        http-method="GET"
        message-converters="trafficConverter"
        expected-response-type="example.Traffic">
    <http:uri-variable name="zipCode" expression="payload.address.zipcode"/>
</http:outbound-gateway>

企业集成模式添加

同样在2.0中，我们增加了对Hohpe和Woolf的Enterprise Integration Patterns一书中描述的更多模式的支持。

讯息记录

通过在应用程序上下文中启用消息历史记录，将在每个消息中添加标头。该标题跟踪所有经过的组件，包括每个通道和端点的名称以及该遍历的时间戳。对于异步应用程序交互和跨多个线程的流很常见的消息传递应用程序，监视和审核可能是一个重大挑战。即使仅在开发时启用，此简单的标头对于解决该挑战也非常有用。切换功能很简单。只需在配置中添加或删除元素。默认情况下，它将跟踪每个通道和端点的历史记录，但是可以使用简单的名称模式进行微调：


<message-history tracked-components="*Service, foo*"/>
留言库

消息存储提供了一种方法，可以为单个事务中可能花费太长时间的任何过程保留消息。例如，通过使用支持MessageStore的新消息通道，您可以具有缓冲和事务处理行为，而不必依赖其他中间件。您将需要使用支持事务持久性的MessageStore实现，但是有很多灵活性。我们为许多常见的RDBMS提供JDBC实现以及DDL，并且已经为基于GemFire的实现提供了原型。随着Spring Data项目的不断发展，您可以在此领域看到很多选项的增加。使用共享存储的另一个好处是，多个进程可能使用相同的组件运行。换句话说，您可以分发分散收集配置（使用拆分器，聚合器以及介于两者之间的某些组件），但是只要这些聚合器各自共享相同的MessageStore，就可以分配工作负载。


<aggregator input-channel="in" ref="aggregator"
    message-store="messageStore" output-channel="out"/>

<int-jdbc:message-store id="messageStore" data-source="dataSource"/>
索赔检查

Claim Check模式背后的想法是，您可以将Message负载交换为“ claim ticket”，反之亦然。这样可以减少带宽和/或避免在跨通道发送消息时的潜在安全问题。您可以将其视为“按引用传递”，而不是典型的“按值传递”语义。我们的实现直接建立在上面刚刚提到的消息存储支持上。一对“声明检入”和“声明检出”转换器应共享对同一消息存储的引用。为了方便起见，将首先考虑一个名为“ messageStore”的bean。如有必要，您可以通过'message-store'属性提供明确的引用。


<claim-check-in input-channel="payloadsIn" output-channel="ticketsOut"/>

<claim-check-out input-channel="ticketsIn" output-channel="payloadsOut"/>
控制总线

使用控制总线，您可以使用消息传递来管理和监视端点和通道。实际上，这是一种相当通用的机制，您可以在其中发送一条消息，该消息的有效载荷实际上是SpEL表达式，可以针对集成应用程序中的某些组件进行评估。例如，您可以将有效负载“ @ somePoller.stop（）”发送到控制总线的输入通道。为了启用此功能，只需添加元素：


<control-bus input-channel="controlChannel"/>

新的通道适配器和消息传递网关

我们在Spring Integration 2.0中添加了几个新的通道适配器和消息传递网关。对于这些已经在文档中很好介绍的配置，这里不再提供每个配置的示例配置，而是简单地提供一个到Spring Integration Reference Manual中相关部分的链接。

  • JDBC
  • JMX
  • TCP / UDP
  • XMPP

通用航空参考书中将介绍其他一些内容，但是我将在此处提供简要说明和/或相关链接。Josh Long在最近的博客中详细介绍了FTP和SFTP适配器。

我们还添加了RSS / Atom Feed阅读支持。这是一个例子：


<feed:inbound-channel-adapter channel="newsChannel" url="http://example/news/rss.xml"/>

每个人的最爱：Twitter适配器。我们支持入站和出站状态更新以及直接消息。这是一个简单的例子。如您所见，它启用了启用OAuth的Twitter Connection实例的配置：


<twitter:outbound-update-channel-adapter channel="tweets"
            twitter-connection="twitterConnection"/>

<twitter:twitter-connection id="twitterConnection"
            consumer-key="${twitter.oauth.consumerKey}"
            consumer-secret="${twitter.oauth.consumerSecret}"            
            access-token="${twitter.oauth.accessToken}"
            access-token-secret="${twitter.oauth.accessTokenSecret}"/>

其他增加

动态Groovy脚本

与上面显示的SpEL示例一样，您实际上可以将Groovy脚本用于任何转换器，过滤器，路由器，分离器等。这些脚本非常简单，可以利用绑定到脚本的“有效载荷”和“标题”运行时的执行上下文。例如，以下Groovy脚本可以定义为'LengthRouter.groovy'：


return payload.length() > 100 ? 'long' : 'short'

然后，您可以在路由器元素（或任何其他核心元素类型）中引用它。您可以提供“刷新检查延迟”，以便在运行时提取脚本中的更改：


<router input-channel="strings">
    <int-groovy:script location="example/LengthRouter.groovy" refresh-check-delay="10000"/>
</router>

当然，您可以做的比Groovy脚本还要多。通常，Groovy脚本选项在SpEL和POJO之间提供了很好的中间立场。

地图变形金刚

这些对称的转换器将有效载荷对象与Map进行转换，Map中的键可以通过SpEL保持“扁平”属性路径（例如“ customer.address.city”）。


<object-to-map-transformer input-channel="objectsIn" output-channel="mapsOut"/>

<map-to-object-transformer input-channel="mapsIn" output-channel="objectsOut"/>
JSON变形金刚

这些对称的转换器将有效载荷对象与JSON转换。他们使用Jackson库，如果您需要自定义行为，则可以引用'object-mapper'。


<object-to-json-transformer input-channel="objectsIn" output-channel="jsonOut"/>

<json-to-object-transformer input-channel="jsonIn" output-channel="objectsOut"/>
序列化变压器

这些对称的转换器将有效载荷对象与字节数组进行相互转换。尽管这些转换器在2.0中并不完全是新的，但它们现在确实委托给一对新的策略接口。默认策略是标准Java序列化。但是，您可以选择提供“序列化器”（或“反序列化器”的“反序列化器”）属性值以引用任何序列化器（或反序列化器）实现。这些是Spring 3.0.5中可用的新策略接口。在JDBC MessageStore和TCP / UDP适配器中使用了相同的策略。再一次，随着Spring Data项目的发展，您可以期待看到许多新的实现。


<payload-serializing-transformer input-channel="objectsIn" output-channel="bytesOut"
            serializer="someCustomSerializer"/>

<payload-deserializing-transformer input-channel="bytesIn" output-channel="objectsOut"
            deserializer="someCustomDeserializer">
