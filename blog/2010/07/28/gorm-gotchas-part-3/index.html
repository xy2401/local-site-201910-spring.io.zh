GORM陷阱（第3部分）

工程
彼得·莱德布鲁克
2010年7月28日

很高兴听到人们发现这些文章很有用，因此很高兴在本系列中添加其他文章。这次，我将再次讨论关联，但重点是它们何时加载到内存中。

2010年8月2日更新我添加了更多关于渴望通过一对多关系进行获取的信息，因为您需要注意一些问题。

懒惰很酷

人们了解GORM关系的第一件事是默认情况下它们是延迟加载的。换句话说，当您从数据库中获取域实例时，将不会加载其任何关系。相反，GORM仅在实际使用关系时才加载它。

通过考虑上一篇文章中的示例，让我们更具体一点：

class Location {
    String city
}

class Book {
    String title

    static constraints = {
        title(blank: false)
    }
}

class Author {
    String name
    Location location

    static hasMany = [ books: Book ]
}

如果获取一个Author实例，则无需执行另一个查询就可以使用的唯一信息就是作者的姓名。当我们尝试获取关联的位置或书籍时，将启动更多查询以获取我们所需的额外数据。

这确实是唯一明智的默认选项，尤其是对于具有较长关联链的复杂模型而言。如果默认为急切获取，则可能仅通过获取单个实例即可最终从数据库中获取一半的数据。

但是，此选项并非没有成本。我将研究懒惰关联的三种副作用，以便您了解它们的含义，可以识别症状并解决由这些副作用引起的任何问题。

代理人

关联的延迟加载涉及一些魔术。毕竟，您不希望上面的location属性返回null ，对吗？因此，Hibernate使用代理和自定义集合类来提供对延迟加载的集合和关联的透明访问-您不必担心它们尚未存储在内存中。通常，这些代理在隐藏幕后工作方面做得很好，但有时实现会泄漏。

例如，请考虑以下域模型：

class Pet {
    String name
}

class Dog extends Pet {
}

这是一个非常简单的继承层次结构，因此不会有任何令人讨厌的惊喜。现在，假设我们在数据库中有一个Dog实例，其ID为1。您认为以下代码会发生什么？

def pet = Pet.load(1)
assert pet instanceof Dog

凭直觉，这应该起作用。毕竟，ID为1的宠物是Dog 。那么为什么断言失败呢？load（）方法不是从数据库中获取基础实例，而是返回一个代理，该代理按需执行所需的查询，例如，当您尝试访问id以外的属性时。该代理是Pet而不是Dog的动态子类，因此instanceof检查失败。即使从数据库中加载了实例，它仍然会失败！以图表形式：

将Pet.load（）更改为Dog.load（）将解决此问题，因为代理将成为Dog的动态子类。您还可以通过用get（）替换load（）使其工作，因为后者的实现会自动解包代理并返回基础的Dog实例。实际上，Grails会在其他许多情况下努力执行此自动展开操作，因此您不太可能遇到这个问题。这就是诸如此类的原因之一，当您这样做时会感到惊讶。

还有另一种可能会引起心痛的情况，尽管这种情况很少见。假设您还有另一个类Person ，它与Pet的关系如下：

class Person {
    String name
    Pet pet
}

pet关系是惰性的，因此当您获得Person实例时， pet属性将是代理。通常，GORM对您隐藏了此内容，但请检查以下行为：

def p = Person.get(1)
assert p.pet instanceof Dog
assert Pet.get(1) instanceof Dog
assert Pet.findById(1) instanceof Dog
assert Pet.list()[0] instanceof Dog

假设我们有一个Person实例和一个Pet实例（即Dog） ，并假设这两个实例通过pet属性关联，则前三个断言将成功，而后一个则不会。摆脱其他代码行，然后断言将成功。?？

这种行为无疑令人困惑，但是其根源在于Hibernate会话。当您从数据库中检索Person时 ，其pet属性是一个代理。该代理存储在会话中，并代表ID为1的Pet实例。现在，Hibernate会话保证无论您在单个会话中检索多少次特定的域实例，Hibernate都会返回完全相同的对象。因此，当我们调用Pet.get（1）时 ， Hibernate为我们提供了代理。相应断言成功的原因是GORM自动解包代理。对于findBy *（）和只能返回单个实例的任何其他查询，也会发生同样的情况。

但是，GORM不会为list（） ， findAllBy *（）和其他可返回多个结果的查询的结果解开代理。因此， Pet.list（）[0]返回给我们解包后的代理实例。如果未首先获取Person ， Pet.list（）将返回真实实例：这次代理不在会话中，因此查询没有义务返回它。

您可以通过两种方式保护自己免受此问题的影响。首先，您可以使用动态instanceOf（）方法代替instanceof运算符。它在所有GORM域实例上都可用，并且可识别代理： Pet.get（1）.instanceOf（Dog） 。其次，使用def而不是静态域类类型声明变量，否则您可能会看到类强制转换异常。所以，而不是

Person p = Person.get(1)
Dog dog = Pet.list()[0]    // Throws ClassCastException!

使用

def p = Person.get(1)
def dog = Pet.list()[0]

使用这种方法，即使使用代理，您仍然可以访问Dog特有的任何属性或方法。

必须说，GORM在使开发人员免受代理攻击方面做得非常出色。它们仅很少泄漏到您的应用程序代码中，尤其是在较新版本的Grails中。尽管如此，有些人还是会遇到问题，因此了解症状是什么以及为什么会出现很有用。

我在上一个示例中展示了会话的行为与延迟加载相结合如何可以产生一些有趣的结果。这种组合还存在一个更常见的错误： org.hibernate。LazyInitializationException 。

延迟加载和会话

正如我已经提到的那样，当您有一个延迟加载的关系时，如果您希望以后再浏览该关系，则Hibernate必须执行一个额外的查询。在正常情况下，这不是问题（除非您担心性能），因为Hibernate透明地进行操作。但是，如果您尝试在另一个会话中访问该关系会怎样？

假设您已在控制器操作中加载了ID为1的Author实例，并将其存储在HTTP会话中。在这一点上，尚无代码触及书籍收藏。在下一个请求时，用户将转到一个与此控制器操作相对应的URL：

class MyController {
    def index = {
        if (session.author) {
            render "Author ${session.author.name} has written the books: ${session.author.books*.title}"
        else {
            render "No author in session"
        }
    }
    ...
}

这里的意图是，如果我们的HTTP会话包含一个author变量，则该操作将呈现该作者的书名。除非在这种情况下，否则不会。而是抛出LazyInitializationException 。

问题在于Author实例就是我们所说的分离对象 。它已在一个Hibernate会话中加载，但随后在请求结束时关闭了该会话。关闭对象的会话后，该对象将分离，并且您无法访问该对象上的任何属性，这将导致查询。

“但是我的行动是公开的，所以为什么会有问题呢？”我听到你哭了。这是个好问题。不幸的是，这是一个新的Hibernate会话，它对我们的Author实例一无所知。只有将对象显式附加到新会话后，您才能访问其惰性关联。有几种技术可以做到这一点：

def author = session.author

// Re-attach object to session, but don't sync the data with the database.
author.attach()

// Re-attach object, but merge any changes with the data in the database.
// You *must* use the instance returned by the merge() method.
author = author.merge()

attach（）方法在自检索到分离对象以来数据库中不太可能更改域实例的情况下很有用。如果这些数据可能已更改，那么您必须要小心。查看Grails参考指南 ，获取有关merge（）和refresh（）行为的信息。

现在，如果获得LazyInitializationException ，您将知道这是因为您的域对象未附加到Hibernate会话。尽管我将介绍另一种解决问题的方法，但是您还将对如何解决问题有个好主意。在此之前，我想看看延迟初始化的另一个经典副作用：N + 1选择问题。

N + 1个选择

让我们回到本文前面的author / book / location示例。假设我们在数据库中有四个作者，我们运行以下代码：

Author.list().each { author ->
    println author.location.city
}

将执行多少个查询？答案是五个：一个获取所有作者，然后每个作者获取一个对应的位置。这就是N + 1选择问题，编写受其困扰的代码非常容易。乍看之下，上面的示例看上去毫无害处。

在开发过程中，这并不是真正的问题，但是在将应用程序部署到生产环境时，执行如此多的查询会损害应用程序的响应能力。因此，最好在向最终用户开放应用程序之前分析数据库的使用情况。最简单的方法是在grails-app / conf / DataSource.groovy中启用Hibernate登录，以确保所有查询都记录到stdout中：

dataSource {
    ...
    loggingSql = true
}

当然，您可以基于每个环境启用它。另一种方法是使用特殊的数据库驱动程序（例如P6Spy）来拦截查询并将其记录下来。

那么如何避免这些额外的查询呢？通过渴望而不是懒惰地获取关联。这种方法还解决了我提到的其他与延迟加载有关的问题。

渴望

GORM允许您基于每个关系覆盖默认的延迟加载行为。例如，我们可以将GORM配置为始终通过以下映射将作者的位置和作者一起加载：

class Author {
    String name
    Location location

    static hasMany = [ books: Book ]

    static mapping = {
        location fetch: 'join'
    }
}

在这种情况下，不仅作者会加载该位置，而且还会使用SQL连接在同一查询中对其进行检索。所以这段代码：

Author.list().each { a ->
    println a.location.city
}

只会导致一个查询。您也可以使用lazy：false选项代替fetch：'join'，但这将导致额外的查询来加载位置。换句话说，该关联已被加载，但是带有单独的SQL选择。大多数时候，您可能会希望使用fetch：'join'来最大程度地减少执行查询的次数，但是有时这可能是更昂贵的方法。这实际上取决于您的模型。

还有其他选择，但我在这里不再赘述。如果您想了解更多，它们在Grails用户指南的 5.3.4和5.5.2.8节中有完整的文档说明（尽管我会等待Grails的1.3.4版本发布，该版本将进行一些重要的文档更新）。

在域类映射中配置快速加载的不利之处在于，关联将始终被快速加载。但是，如果您仅偶尔需要该信息怎么办？任何只想显示作者姓名的页面都会被不必要地拖慢，因为还必须加载该位置。对于这样的简单关联而言，成本可能较低，但对于集合而言，成本会更高。这就是为什么您还可以选择根据每个查询急于加载关联的原因。

查询是上下文相关的，因此它们是指定是否应热切加载特定关联的理想场所。假设我们已恢复到“ 作者”的默认行为，现在我们希望获取所有作者并显示其城市。在这种情况下，我们显然希望在获得作者时检索位置。这是如何做：

Author.list(fetch: [location: 'join']).each { a ->
    println a.location.city
}

我们要做的就是在查询中添加一个fetch参数，该参数带有关联名称映射-> fetch模式。如果代码还显示了作者书籍的标题，则我们也将书籍关联添加到地图中。动态查找器支持完全相同的提取选项：

Author.findAllByNameLike("John%", [ sort: 'name', order: 'asc', fetch: [location: 'join'] ]).each { a->
    ...
}

我们还可以使用条件查询实现相同的目的：

def authors = Author.withCriteria {
    like("name", "John%")
    join "location"
}

以上所有内容也适用于一对多关系，但是您需要考虑一些其他注意事项。

一对多的热切装载

我在上面说过，当您渴望获取关联时，通常会希望使用联接，但是这种经验法则不适用于一对多关系。要了解原因，请考虑以下查询：

Author.list(max: 2, fetch: [ books: 'join' ])

极有可能只返回一个Author实例。那可能不是您期望或想要的行为。那是怎么回事？

Hibernate在后台使用左外部联接为每个作者获取书籍。这意味着您将获得重复的Author实例：与作者相关联的每本书都有一个。如果那里没有max选项，那么您将不会看到这些重复项，因为GORM会将其删除。但是麻烦是在删除重复项之前将max选项应用于结果。因此，在上面的示例中，Hibernate仅返回两个结果，这两个结果可能具有相同的作者。然后，GORM删除重复项，最后得到一个Author作者实例。

域类映射配置和条件查询都会发生此问题。实际上，条件查询默认情况下不会从结果中删除重复项！对于这种混淆，只有一种明智的解决方案：对于一对多关系，始终使用“选择”模式。例如，在域映射中使用lazy：false ：

class Author {
    ...
    static hasMany = [ books: Book ]

    static mapping = {
        location fetch: 'join'
        books lazy: false
    }
}

在查询中，根据您使用的是动态查找器还是条件查询，使用适当的设置：

import org.hibernate.FetchMode

Author.list(fetch: [ books: 'select' ])

Author.withCriteria {
    fetchMode "books", FetchMode.SELECT
}

是的，您最终将获得一个额外的查询来获取集合，但这只是一个查询，您可以获得一致性和简单性。如果您发现确实需要减少查询数量，则可以随时使用HQL。

除了一对多的情况外，在GORM中渴望获取是很简单的，如果您遵循对一对多使用“选择”获取模式的原则，则同样适用于那些。主要工作是对应用程序的数据库访问进行性能分析，以确定应该在哪里急切地获取关联，或者特别是通过联接来获取关联。只是提防过早的优化！

包起来

如您所见，延迟加载关联会引发各种问题，尤其是与Hibernate会话结合使用时。尽管存在这些问题，延迟加载还是一项重要功能，对于对象图而言，它仍然是明智的默认设置。一旦您了解了容易出现的问题，就可以轻松识别它们，并且通常也很容易解决。而且，如果没有什么想到的，那么您总是可以退一步，明智地使用急切的加载方式。

综上所述，随着Grails版本号的增加，用户遇到这些问题的可能性越来越小。当您考虑Hibernate幕后发生的事情时，这是一个非常令人印象深刻的技巧！
