与Grails重用Hibernate / JPA域模型

工程
彼得·莱德布鲁克
2010年8月26日

这是一个普通的故事。您已经有一个数据库，现在想以最少的麻烦为其添加一个Web应用程序。Grails完成任务了吗？当然是这样！更好的是，如果您已经具有基于JPA或Hibernate配置文件的域模型，则可以重用它，并且仍然可以从惯用的强大GORM功能中受益。

重用现有域模型的第一步是在应用程序的类路径上获取类文件。您可以通过将它们打包到一个JAR文件中并将其包含在应用程序中（通常通过将其放在“ lib”目录中）或将源文件放在“ src / java”目录中来实现。在应用程序之间共享模型时，JAR文件是更好的方法，但是您需要一个多项目构建系统，以确保所有项目的JAR始终是最新的。可能值得付出额外的努力，并且Gradle和Maven都具有用于构建Grails项目的插件。或者，如果您的Grails应用程序是唯一使用该模型的项目，则应将源代码保留在“ src / java”中。

一旦决定了如何合并域模型，就需要告诉Grails。具体操作方式取决于您使用的是Hibernate配置文件还是JPA批注。

XML配置文件

Hibernate XML映射文件是老派的，但是它们具有将所有映射信息保存在一个地方的优点。要从您的Grails应用程序中使用它们，只需创建文件grails-app / conf / hibernate / hibernate.cfg.xml ，内容如下：

<?xml version='1.0' encoding='UTF-8'?>
<!DOCTYPE hibernate-configuration PUBLIC
        "-//Hibernate/Hibernate Configuration DTD 3.0//EN"
        "http://hibernate.sourceforge.net/hibernate-configuration-3.0.dtd">
<hibernate-configuration>
    <session-factory>
        <mapping resource="org.example.Book.hbm.xml"/>
        <mapping resource="org.example.Author.hbm.xml"/>
        ...
    </session-factory>
</hibernate-configuration>

确保添加一个每个Hibernate映射文件的元素，然后将这些映射文件放入grails-app / conf / hibernate中 。令人惊讶的是，这就是您要做的！那么带注释的模型呢？

JPA批注

与Hibernate映射文件一样，您必须创建grails-app / conf / hibernate / hibernate.cfg.xml文件。这次，您需要做更多的工作，因为您必须添加一个每个域类的条目：

<?xml version='1.0' encoding='UTF-8'?>
<!DOCTYPE hibernate-configuration PUBLIC
        "-//Hibernate/Hibernate Configuration DTD 3.0//EN"
        "http://hibernate.sourceforge.net/hibernate-configuration-3.0.dtd">
<hibernate-configuration>
    <session-factory>
        <mapping class="org.example.Book"/>
        <mapping class="org.example.Author"/>
        ...
    </session-factory>
</hibernate-configuration>

我给你的乐趣并不多，但至少你只需要做一次。

如果您使用的是Grails 1.1.x或更早版本，则还有另一项任务要执行：您必须通过将以下内容添加到grails-app / conf / DataSource.groovy来告知Grails这是基于注释的模型：

dataSource {
    configClass = "org.codehaus.groovy.grails.orm.hibernate.cfg.GrailsAnnotationConfiguration"
    ...
}

您不必在Grails 1.2+上执行此操作，因为GrailsAnnotationConfiguration是这些版本中的默认设置。下次运行Grails应用程序时，您将有权访问所有带注释的域类。

注意从1.3.2版开始，您已经可以使用Grails命令create-hibernate-cfg-xml为您生成Hibernate配置文件。节省键入/复制粘贴！

从GORM的角度来看，领域模型中仍然存在一个明显的空白：约束。您无法在Java类中定义它们，但是如果没有它们，GORM验证将毫无用处。幸运的是，您可以通过添加包将约束元数据附加到类。DomainClass Constraints.groovy脚本转换为“ src / java”。例如，如果您有一个域类org.example。LibraryBook ，那么您应该添加文件src / java / org / example / LibraryBookConstraints.groovy 。在此脚本内，您可以定义约束块，如下所示：

constraints = {
    name(blank: false)
    age(min: 0, max: 120)
    ...
}

您可能想知道为什么脚本进入“ src / java”目录。这是个好问题。Grails要求源代码形式的约束脚本，因此您需要将其放置在导致Grails在WAR中包括.groovy文件（而非已编译的类）的位置。 src / java目录是唯一满足此要求的位置。

在您的Grails应用程序中合并Java域模型确实非常简单。您可能会失去标准GORM域类的优雅和自定义映射，但是仍然可以使用动态查找器，条件查询，验证等。脚手架又如何？Grails将竭尽全力支持您的域模型，但这并不能保证。只要模型简单明了，脚手架就可以正常工作。重用域模型也是尝试Grails的一种好方法，因为您不必从头开始创建模型。

请注意，如果您只是想访问旧数据库而没有现有的域模型，那么几乎可以肯定的是，将标准GORM域类与自定义映射一起使用会更好。用户指南中记录了这两种方法： 休眠映射和定制映射DSL 。

最后一件事：如果您在自己的Grails项目中使用过Java域模型，那么我很想听听您的经验。任何可以帮助其他想要走这条路的用户的事情，将不胜感激！
