使用Grails 1.3管理插件

工程
彼得·莱德布鲁克
2010年5月18日

长期以来，管理Grails依赖项仅意味着将它们放置在应用程序的lib目录中。然后是Grails 1.2和依赖项DSL：您最终可以声明依赖项，并让Grails自动下载它们并使它们可用于您的应用程序。大！

现在，Grails 1.3将依赖项DSL带到了插件领域。

所以呢？


到目前为止，用户经常会在插件系统中遇到两个问题：

  1. 设置合适的Subversion服务器充当Grails插件存储库并不容易；和

  2. 您无法控制插件将哪些依赖项带入您的应用程序。


其中的第一个并不是特别适用于个人开发人员，但对于可能限制访问互联网或希望对“最新”版本的库和插件进行更多控制的公司而言，这是一笔更大的交易。如果他们可以使用Nexus或Artifactory这样的存储库管理器，那就好多了。

至于依赖项，某些插件包括不需要的库（甚至更糟）会破坏您的应用程序。使用依赖性DSL，您可以明确排除有问题的库。

这些就是为什么此更改对您很重要的一些原因。那么，您如何使用新功能？

依赖两步


为了正确演示Grails依赖管理的实用性，我们必须建立一个本地存储库。对于本文，我们将使用Artifactory。您同样可以使用Nexus或任何其他工具来创建兼容Maven的存储库。

我假设Artifactory已经安装并正在运行-有关进入此阶段的详细信息，请访问其网站，尤其是“一分钟Artifactory”截屏 。至此，我们有了一个本地存储库，但是它尚未提供任何工件。让我们修复它。

我将添加“ db-util”插件和“ commons-digester”库。您可以尝试使用任何可用的插件或JAR。只需导航至Artifactory的“部署”选项卡（已使用用户名“ admin”，密码“ password”登录），选择适当的文件并上传。我抓起文件：

 $USER_HOME/.ivy2/cache/commons-digester/commons-digester/jars/commons-digester-2.0.jar
 $USER_HOME/.grails/1.2.1/plugins/db-util-0.4.zip

并通过Artifactory UI添加了它们：

请注意，UI允许您指定上载的目标存储库。我为“ db-util”插件选择了“ plugins-releases-local”，为“ commons-digester”选择了“ libs-releases-local”。另外，在提交插件工件之前，我将其groupId设置为“ org.grails.plugins”，并将其artifactId设置为“ db-util”（默认值均为“ grails-db-util”）。“ org.grails.plugins” groupId是Grails假设的用于插件依赖性的组。还请注意artifactId如何不包含“ grails-”前缀。

这两个工件现在可以在本地任何Grails应用程序中使用。现在，我们需要在Grails应用程序中配置存储库和依赖项。

依赖DSL


接下来的部分出奇的简单。我们需要做两件事：

  1. 配置Grails从我们的Artifactory资源库下载插件和JAR；和

  2. 指定我们的依赖项。


与Grails 1.2一样，此操作在grails-app / conf / BuildConfig.groovy中完成 。以下是示例Grails应用程序中的相关部分：

grails.project.dependency.resolution = {
    ...
    log "warn"
    repositories {
        grailsPlugins()
        grailsHome()
        mavenRepo "http://localhost:8081/artifactory/libs-releases-local/"
        mavenRepo "http://localhost:8081/artifactory/plugins-releases-local/"
        grailsCentral()
    }
    plugins {
        build "org.grails.plugins:db-util:0.4"
    }
    dependencies {
        compile "commons-digester:commons-digester:2.0"
    }
}

因此，在库部分添加由Artifactory的服务器暴露每一个“仓库”一mavenRepo条目。请注意，在这种情况下，我们同时添加了“ libs-releases-local”存储库和“ plugins-releases-local”存储库。

DSL的新增功能是“ 插件”部分，该部分（毫不奇怪）用于声明Grails插件。它的语法与“ 依赖项”部分完全相同。因此，“ db-util”在插件下声明，“ commons-digester”在dependencies下声明。

那就是你要做的。如果现在执行grails run-app ，则会看到类似以下的输出：

安装了“ db-util”插件，并且在类路径中包含了“ commons-digester” JAR。简单。

如果遇到依赖项解析问题，则可以通过将BuildConfig.groovy中的日志“警告”行替换为日志“ info”甚至日志“ debug”来拨号记录日志 。

当然，有时您会希望托管自己的插件或公共插件的自定义/修补版本。现有的release-plugin命令仅适用于旧式Subversion存储库，那么您该怎么办？

将插件发布到兼容Maven的存储库


Grails核心在这里无济于事，但随后并不需要。有一个Grails插件专用于发布Grails应用程序和兼容Maven的存储库的插件 ： Maven Publisher插件 。

将其安装在您的插件项目中，将以下条目添加到您的BuildConfig.groovy文件中：

grails.project.dependency.distribution = {
    remoteRepository(id: "pluginSnapshots", url: "http://localhost:8081/artifactory/plugins-snapshots-local/") {
        authentication username: "admin", password: "password"
    }
}

然后运行
grails maven-deploy –repository=pluginSnapshots
瞧！您的插件将以groupId为org.grails.plugins部署到本地Artifactory的“ plugins-snapshots-local”存储库。请注意，您为–repository参数传递的值应与BuildConfig.groovy中grails.project.dependency.distribution闭包中定义的远程存储库的“ id”相匹配。

您可以对部署进行微调，例如通过将pom.xml文件与元素或通过在插件描述符中添加groupId属性，但是基本步骤都很简单。

剩下要解释的是依赖性信息的保存位置。例如，正常工件（例如JAR库）的依赖项存储在元数据文件（如pom.xml）中。Grails插件不同。它们的JAR依赖项位于Dependencies.groovy文件中的zip包中。至于插件依赖性，您可以像以前一样在插件描述符的dependsOn字段中定义它们。然后将它们转换为插件生成的pom.xml文件（与存储库中的插件zip文件一起存储）中的依赖项声明。

真正的全部就是它。现在，您可以提供自己的易于管理的Maven兼容存储库，这样您的团队成员就不必在每次需要插件或依赖项时都可以访问Internet。您还可以通过控制可用的Grails插件版本来为项目注入一些一致性。您可以轻松地修补公共插件，并在等待修复程序将其发布为正式版本时将这些版本提供给本地团队。
