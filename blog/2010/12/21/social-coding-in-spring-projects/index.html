春季项目中的社交编码

工程
基思·唐纳德
2010年12月21日

去年，新的Spring项目在多个领域启动，包括社交 ， 移动 ， 数据和集成 。我已经这样做了将近7年 ，老实说，这对我来说从来没有像现在这样令人兴奋。我之所以这样想，是因为我们的社区了解在您之前奠定的基础上提高门槛的重要性。这就是为什么我们能够如此迅速地前进，这证明了Juergen Hoeller领导的核心开发团队的素质 。

我非常兴奋的一件事是，我们看到的社区贡献越来越多。传统上，这些是通过JIRA补丁发布的，但是诸如Github和Gitorious之类的现代社交编码平台开辟了新的机遇。在此博客文章中，我想介绍一个新的贡献模型，该模型使您能够为自己喜欢的Spring项目做出贡献，并直接与核心开发团队合作。

增强对可能性的认识

让我想到这个话题的是Jettro Coenradie 最近的一篇博客 ，内容涉及Spring Mobile。在他的帖子中，Jettro提出了一个新功能的好主意。他甚至花时间整理了一个展示其价值的样本。我当时想着“ 太棒了！ ”，如果Jettro被授权将其功能回馈给该项目，这不会动摇。Jettro将在下一个正式版本中拥有他的代码，从而从中受益，而社区将因拥有其以前没有的实用功能而受益。借助社交编码，今天这一切都是可能的，让社区意识到这一过程是我们的责任。

“社交编码”平台的兴起

在过去的两年中，我们看到了社交编码平台（例如Github ， Gitorious ， Bitbucket和Launchpad）的兴起。之四，Github上是最流行的超过百万托管的项目，其中包括一些 著名 的人 。这些平台的秘密之处在于它们结合了分布式版本控制 （ Distributed Version Control）的原理，该原理使共享变更变得容易，并具有社交网络功能以培养授权的软件开发社区。

因此，假设像Jettro一样，我是一个需要对Spring项目进行改进的用户。在此之前，我最好的办法是签出源代码，在本地进行更改，生成补丁文件，并将补丁附加到问题上 。由于我不是项目提交者，因此我自己无法提交任何更改。我也无法创建一个分支，该分支使我可以立即进行多项改进。如果发现我的补丁程序需要改进，那么我将被迫进入痛苦的过程，即生成并附加另一个补丁程序文件。

现代社交编码平台提供了出色的工作流程。 首先，我不必成为项目提交者即可进行更改。我有权分叉项目的存储库，创建本地主题分支，然后开始黑客入侵。我将更改提交到存储库中，准备好后，我要求核心开发团队将更改从我的分支拉到master分支。该工作流程使您可以正确地处理重要的事情：代码，并避免与JIRA问题和补丁文件相关的官僚主义。

授权的贡献者工作流程可以如下所示：

Spring项目团队成员，例如Juergen Hoeller，会收到您的拉取请求，然后集成您的更改：

良好的社交编码平台提供了支持此工作流程的有用功能，例如交互式Diff审阅者，审阅讨论功能，参与者许可协议（CLA）处理者和事件通知程序。

赋权贡献的一个例子

对于Spring Mobile 1.0.0。M2版本，我亲自经历了这个工作流程，以贡献Jettro最初建议的改进。在下一节中，我将重播该经验，以便您可以将其作为自己授权的贡献的示例。

由于Spring Mobile托管在SpringSource Gitorious实例上 ，因此此示例中的某些内容特定于Gitorious。通常，社交编码平台非常相似。如果存在重大差异，我将予以注意。

步骤1：分叉存储库

我要做的第一件事是创建自己的Spring Mobile存储库分支。Gitorious使用术语“克隆”而不是派生，这是通过单击存储库仪表板上的“克隆存储库”按钮完成的：

步骤2：在本地克隆货叉

接下来，我将fork克隆到文件系统上的本地目录中：

git clone --recursive [email protected]:~kdonald/spring-mobile/kdonalds-spring-mobile.git

请注意，URL如何引用我的Gitorious用户目录中的存储库位置。作为比较，Github的URL格式相似，但略有不同：

git clone --recursive [email protected]:kdonald/spring-mobile.git

步骤3：链接到上游存储库

接下来，我将本地分支连接到上游Spring Mobile存储库。这是一个可选步骤，但通常建议这样做，因为当更改被推送到上游存储库时，它允许我保持最新状态。

git remote add upstream git://git.springsource.org/spring-mobile/spring-mobile.git
git fetch upstream

步骤4：为您的工作创建一个主题分支

接下来，我创建了一个主题分支以进行改进。主题分支为我的更改提供了专用的工作空间，并允许我的fork保留可重复使用的主服务器的干净镜像。我将要实现的功能的名称命名为branch site-switcher ：

git checkout -b site-switcher

第5步：修改，提交和推送您的更改

接下来，我实现了该功能，以逻辑的，迭代的方式在本地提交了我的工作。我花了好几次迭代才能得出我满意的完整实现，其中包括新代码，测试和文档。最后，我将4次提交推送到了我的fork。

./gradlew eclipse build <!-- import into Eclipse and hack, hack, hack... -->
git commit -m "logical commit 1"
git commit -m "logical commit 2"
git commit -m "logical commit 3"
git commit -m "logical commit 4"
git push origin site-switcher

步骤6：发送拉取请求

接下来，我向开发团队发送了请求请求，请求他们将我的更改集成到主存储库中。在执行此操作之前，我确保所做的更改可以无冲突地应用于master分支的当前状态：

git checkout master
git fetch upstream
git merge upstream/master
get checkout site-switcher
get rebase master

要创建拉取请求，我从我的叉子的仪表板上选择了“请求合并”：

Gitorious使用术语“合并请求”而不是Github流行的“拉取请求”。它们的含义完全相同，Gitorious和Github都提供了使流程变得微不足道的表单工作流程。在表单上，我首先提供了我对核心开发团队所做更改的描述：

接下来，我表明我希望我的主题分支中的所有提交都合并到master分支中，然后单击“创建合并请求”以发送请求：

发送后，将通知开发团队，并且新的，打开的“合并请求”将显示在主存储库的仪表板上：

每个拉或合并请求都分配有一个URL ，您可以在其中查看合并状态，查看差异并讨论更改。

整合授权捐款的示例

此时，核心开发团队的工作就在这里。他们有责任审查和集成您的更改，并在合理的时间范围内以社区的最大利益进行更改。在本节中，我将通过说明典型的集成工作流程来继续该示例。

步骤1：审查和测试

首先，我对更改进行了快速的代码回顾。我通过使用Gitorious的差异查看器来完成此操作，该查看器使我能够从Web浏览器中查看更改并可以选择对它们进行评论。Github提供了类似的功能。

接下来，我创建了一个本地分支，该分支为我提供了一个专用的工作区来引入更改并测试它们：

git checkout -b kdonald-site-switcher master
git pull git://git.springsource.org/spring-mobile/spring-mobile.git refs/merge-requests/1
git log --pretty=oneline --abbrev-commit master..kdonald-site-switcher
./gradlew build

在Gitorious上，每个合并请求都从目标存储库中获得一个分支，该分支还可用于推送从审阅中发现的其他改进。在Github上，您只需直接从贡献者的topic分支中拉出：

git pull https://github.com/kdonald/spring-mobile.git site-switcher

步骤2：合并

完成审查后，我将更改合并到master分支中：

git checkout master
git merge kdonald-site-switcher
git push origin master

在Gitorious上，我现在必须将“合并请求”的状态更新为“已关闭”，表示已完成（Github将在合并后自动关闭拉取请求）。贡献者将被通知，剩下的就是最后的清理工作。

步骤3：清理

在集成方面，我只是删除本地评论分支。由于更改现已合并，因此不再需要：

git branch -D kdonald-site-switcher

回到贡献端，我将叉子与上游主服务器同步以引入合并的更改。这使我的叉子与不断发展的上游存储库保持同步：

git checkout master
git fetch upstream
git merge upstream/master
git log

我删除了主题分支，因为我已经完成了这项工作：

git branch -D site-switcher
git push origin :site-switcher

而已！我的更改现已集成，我现在是Spring Mobile的官方@author，并且保存了我的提交历史！在等待我的变更被集成的同时，我还可以并行地进行其他改进，每个改进都在一个专门的主题分支中进行。我还进行了快速截屏 ，向社区展示了该新功能的价值。

GitHub与其他

在下一部分中，我想简要介绍一下社交编码平台的比较方式以及SpringSource当前如何使用它们。

鉴于Github的流行，不可避免地将所有其他社交编码平台与之进行比较。我最近在Hudson邮件列表上阅读了一篇帖子，其中提到了这样的情况：对于开发人员而言，“拥有GitHub帐户几乎与拥有Twitter句柄或Gmail地址一样普遍”。我看到 雇主在筛选求职者时使用 Github配置文件作为差异化因素。各个平台的特定功能实际上非常相似 。Github与众不同的显着优势是它的知名度和领导地位 。这对于专注于构建多样化，授权社区的开源项目特别有吸引力。

SpringSource本身当前运行一个内部Gitorious实例 ，该实例托管许多项目，包括Spring Integration ， Security ， Roo ， Mobile和Social 。Gitorious的优点是可以免费托管您自己的实例，而这正是我们在此处所做的，将这些项目托管在我们自己的基础架构上。

SpringSource最近也已在Github注册为组织 ，新的Spring Data项目以及Batch ， AMQP和Grails都托管在该组织中。

总的来说，我希望越来越多的Spring项目能够支持DVCS和社交编码的发展，对此我感到非常兴奋。我希望我们将继续支持Gitorious和Github，最终您可能会看到从Github可以访问的所有内容（直接或通过镜像）。我很想听听您对您希望看到的内容以及您是否偏爱特定平台的反馈。

摘要

我希望这篇文章可以帮助您为社区贡献的卓越典范打开眼界。我鼓励您充分利用它，并成为一名受权的Spring开发人员，与我们的核心开发团队合作，共同解决您在职业中日常使用的项目。我们非常希望代表Spring项目团队与你们中的许多人建立新的关系，以使整个Spring社区受益。成为开发人员真是令人兴奋！
