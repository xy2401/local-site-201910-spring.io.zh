SpringSource使用RabbitMQ进行云消息传递

工程
罗德·约翰逊
2010年4月13日

我很高兴地宣布 ，VMware的SpringSource部门已经收购了Rabbit Technologies Ltd.，该公司是流行的开源云消息传递技术RabbitMQ背后的公司。此次收购将增强我们的中间件产品组合，并加快我们的云计划。

云中的消息传递演进


随着组织越来越多地在云环境中构建和部署应用程序，支持这种新模型的基础架构也在不断发展。需要一种新型的轻型，可靠，可伸缩和便携式消息传递系统，以支持将用户请求路由到适当的资源，无论它们位于何处。RabbitMQ是该领域的领导者，并展示了围绕云消息传递的重大创新。

RabbitMQ非常适合云计算，主要有以下三个原因：


  • 开放 ：RabbitMQ是开放源代码，具有开放和活跃的社区，并且基于开放标准。开放性将是成功的云平台的关键因素，因为如果事实证明，迁移到云将成为供应商锁定的另一种形式，那么它将失去很多吸引力。

  • 灵活的可扩展性 ：RabbitMQ能够以多种不同方式进行扩展，不仅是技术水平扩展，还包括跨跨服务扩展直至地理上不同扩展的联合消息传递。

  • 基于协议 ：RabbitMQ是基于协议而不是基于API的消息传递系统。这意味着消息传递客户端不与供应商安装的库绑定在一起，这增加了应用程序的可移植性，并允许来自各种不同平台的客户端消息传递。这种方法还允许RabbitMQ支持多种协议，从而增加了其在异构环境（如云）中的实用性。

实际上，Rabbit在云场景中的适用性已经在实践中得到了证实，它已经被云服务提供商和构建私有云的组织（例如NASA Nebula）采用 。

这对您意味着什么？

如果您是RabbitMQ社区的成员，这是个好消息，并且对您的技术选择进行了进一步的验证。RabbitMQ将继续是开源的，并以与以前相同的方式进行分发。社区可以期望看到对这项杰出技术的投资增加，这将导致开源版本的重大改进。RabbitMQ社区也可以期望看到SpringSource对其他开源社区（如基于Spring，Grails，Groovy和基于Apache的技术）的同样奉献和支持。

我们热爱并致力于开发人员使用Rabbit的各种语言和平台（Java之外）。消息技术的部分价值在于其跨平台的覆盖范围。

如您所料，我们将为Spring社区构建特别引人注目的体验。直接使用Spring或通过Grails使用Spring的企业开发人员可以期望与RabbitMQ作为消息传递系统进行一流的集成。这将为开发人员在结构化应用程序方面提供更大的灵活性和更多选择，同时确保他们开发的应用程序可以更轻松地从开发人员桌面迁移到虚拟或物理数据中心，再迁移到云中。

这个Spring集成将包含两个元素：


  • 低级集成，类似于Spring的JMS支持。这将包括RabbitTemplate，它提供简单的发送和接收操作，并委托给MessageConverter以支持POJO消息传递模型。除了模板之外，我们还通过MessageListenerContainer和MessageListenerAdapter为“消息驱动的POJO”提供支持，这与它们在Spring的JMS支持中的命名非常相似。当然，我们将管理低级资源以及事务。所有这些都将适用于Java和.NET，并且我们正在设计高级AMQP层，以便将来可以支持多种实现。

  • Spring Integration支持。 这将包括用于将Spring Integration Message Channels连接到发送方的AMQP交换器和接收方的AMQP队列的信道适配器。这些适配器建立在上述低级支持的基础上，它们进一步抽象了编程模型，以便最终用户可以在纯POJO服务层之上依赖Spring Integration配置。

后者特别重要。Spring Integration是最令人兴奋的Spring项目之一，它提供了Spring POJO编程模型的自然扩展，以处理企业集成模式。我们看到社区和客户对Spring Integration的兴趣激增，并且正在显着增加对其开发的投资。与Spring Dependency Injection提供的抽象方法相同，该方法避免了使用低级API（例如JNDI），简化应用程序代码并将其与部署问题分离的需求，我相信，即使不是大多数，异步模式也更好。与直接使用消息传递API和协议相比，使用Spring Integration在更概念性的级别上进行表达。

Spring Integration使开发人员无需实现侦听器接口或编写依赖模板的代码，即可利用同步和异步消息传递的优势。它支持直接在Spring Framework上构建的企业集成模式 （Hohpe / Woolf）。开发人员可以使用声明性的，配置驱动的方法来解决诸如转换，路由，拆分和聚合之类的集成问题，同时保持与基于POJO的业务逻辑的清晰隔离，该逻辑可以轻松地进行隔离测试。Spring Integration的消息通道抽象提供了进程内和进程间消息传递之间的一致性。它还将应用程序代码与任何特定的传输或协议解耦。这非常符合Spring的普遍哲学，即可移植性不需要更改代码并且技术选择不应该导致锁定。通过将RabbitMQ添加到传输选项中，开发人员将有更多选择，并且能够立即在RabbitMQ和任何其他受支持的传输之间建立桥梁。

我参加这个博客聚会有点晚了。媒体，博客圈和推特（一直是一个热门话题）已经开始关注SpringSource / Rabbit。（这提醒我：最近，我在Twitter上的活动比在我的博客上的活动要活跃得多，我鼓励您在springrod上关注我。）

但是，让我成为第一个欢迎RabbitMQ社区加入SpringSource家族并鼓励SpringSource社区看看如何开始使用RabbitMQ的人。
