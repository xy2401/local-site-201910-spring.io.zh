Spring 3.1 M1：介绍@Profile

工程
克里斯·比姆斯
2011年2月14日

介绍

在我之前宣布Spring 3.1 M1的文章中，我讨论了使用Spring时应用的新bean定义配置文件功能。 使用XML配置容器。今天，我们将介绍新的@Profileannotation，并了解在使用@Configuration类而不是XML时如何应用相同的功能。在此过程中，我们将介绍一些设计@Configuration类的最佳实践。

调出@Configuration

对于那些不熟悉@Configuration类的人，您可以将它们视为与Spring等效的纯Java语言。 XML文件。之前，我们已经在博客中介绍了 此 功能集 ，并且参考文档对此进行了很好的介绍 。如果需要简介或复习，则可能需要重新访问这些资源。

正如我们将在此后的文章中所看到的那样，Spring 3.1中的@Configuration方法已经引起了很多关注，以便对其进行完善，并使之成为那些希望在不使用XML的情况下配置其应用程序的人真正的一流选择。今天的帖子将仅介绍这些增强功能之一：新的@Profileannotation。

与上一篇文章一样，我编写了一个简单的示例，您可以在其中进行后续操作并自己尝试一下。你可以在找到它 https://github.com/cbeams/spring-3.1-profiles-java以及所有设置细节都在README中。该示例包含上一篇文章中介绍的基于XML的配置，以及com.bank.config.xml和com.bank.config.code包中的@Configuration类。每个软件包都已经复制了IntegrationTests JUnit测试用例；这应该可以帮助您比较和对比引导容器的两种样式。

从XML到@Configuration

让我们潜入吧！我们的任务很简单：采用前面显示的基于XML的应用程序并将其移植为@Configuration样式。我们在上一篇文章中以XML配置开始，如下所示：

这很容易移植到@Configuration类中：

src/main/com/bank/config/code/TransferServiceConfig.java


@Configuration
public class TransferServiceConfig {

	@Bean
	public TransferService transferService() {
		return new DefaultTransferService(accountRepository(), feePolicy());
	}

	@Bean
	public AccountRepository accountRepository() {
		return new JdbcAccountRepository(dataSource());
	}

	@Bean
	public FeePolicy feePolicy() {
		return new ZeroFeePolicy();
	}

	@Bean
	public DataSource dataSource() {
		return new EmbeddedDatabaseBuilder()
			.setType(EmbeddedDatabaseType.HSQL)
			.addScript("classpath:com/bank/config/sql/schema.sql")
			.addScript("classpath:com/bank/config/sql/test-data.sql")
			.build();
	}
}

注意： EmbeddedDatabaseBuilder是基础组件 XML中最初使用的元素。正如你所看到的，它是一个内使用相当方便@Bean方法。

此时，基于@Configuration的单元测试将通过绿色条：

src/test/com/bank/config/code/IntegrationTests.java


public class IntegrationTests {

	@Test
	public void transferTenDollars() throws InsufficientFundsException {

		AnnotationConfigApplicationContext ctx = new AnnotationConfigApplicationContext();
		ctx.register(TransferServiceConfig.class);
		ctx.refresh();

		TransferService transferService = ctx.getBean(TransferService.class);
		AccountRepository accountRepository = ctx.getBean(AccountRepository.class);

		assertThat(accountRepository.findById("A123").getBalance(), equalTo(100.00));
		assertThat(accountRepository.findById("C456").getBalance(), equalTo(0.00));

		transferService.transfer(10.00, "A123", "C456");

		assertThat(accountRepository.findById("A123").getBalance(), equalTo(90.00));
		assertThat(accountRepository.findById("C456").getBalance(), equalTo(10.00));
	}

}

上面使用了AnnotationConfigApplicationContext ，它允许直接注册@Configuration和其他@Component AnnotationConfigApplicationContext类。这为我们提供了一种无字符串且无类型安全的容器配置方式。没有XML，这很好，但是在这一点上，我们的应用程序遇到了我们在第一篇文章中看到的相同问题：将应用程序部署到生产环境中时，一个独立的数据源将毫无意义。需要从JNDI查找。

没问题让我们将基于嵌入式和基于JNDI的数据源分解为它们自己的专用@Configuration类：

src/main/com/bank/config/code/StandaloneDataConfig.java


@Configuration
@Profile("dev")
public class StandaloneDataConfig {

	@Bean
	public DataSource dataSource() {
		return new EmbeddedDatabaseBuilder()
			.setType(EmbeddedDatabaseType.HSQL)
			.addScript("classpath:com/bank/config/sql/schema.sql")
			.addScript("classpath:com/bank/config/sql/test-data.sql")
			.build();
	}

}

src/main/com/bank/config/code/JndiDataConfig.java


@Configuration
@Profile("production")
public class JndiDataConfig {

	@Bean
	public DataSource dataSource() throws Exception {
		Context ctx = new InitialContext();
		return (DataSource) ctx.lookup("java:comp/env/jdbc/datasource");
	}

}

至此，我们已经在自己的@Profileannotation的@Configuration类中声明了两个不同的DataSource bean。与XML一样，这些类及其中的@Bean方法将根据当前处于活动状态的Spring概要文件被跳过或处理。但是，在看到实际效果之前，我们首先需要完成重构。我们已经拆分了两个可能的DataSource bean，但是如何在TransferServiceConfig引用它们的方法-特别是它的accountRepository()方法呢？我们有两个选择，并且都开始于了解@Configuration类是@Autowired注入的候选对象。这是因为最后@Configuration对象在容器中被当作“只是另一个Spring bean”来管理。让我们来看看：

src/main/com/bank/config/code/TransferServiceConfig.java


@Configuration
public class TransferServiceConfig {

	@Autowired DataSource dataSource;

	@Bean
	public TransferService transferService() {
		return new DefaultTransferService(accountRepository(), feePolicy());
	}

	@Bean
	public AccountRepository accountRepository() {
		return new JdbcAccountRepository(dataSource);
	}

	@Bean
	public FeePolicy feePolicy() {
		return new ZeroFeePolicy();
	}

}

通过使用上面的@Autowiredannotation，我们已经要求Spring容器为我们注入DataSource类型的Bean，而不管它是在XML中，在@Configuration类中还是在其他地方声明的。然后，在accountRepository()方法中，将简单地引用注入的dataSource字段。这是在@Configuration类之间实现模块化的一种方法，从概念上讲，这与两个类之间的ref样式引用没有区别在不同XML文件中声明的元素。

重构的最后一步是更新单元测试以不仅引导TransferServiceConfig ，而且引导我们的DataSource bean的JNDI和独立的@Configuration变体：

src/test/com/bank/config/code/IntegrationTests.java


public class IntegrationTests {
	@Test
	public void transferTenDollars() throws InsufficientFundsException {
		AnnotationConfigApplicationContext ctx = new AnnotationConfigApplicationContext();
		ctx.getEnvironment().setActiveProfiles("dev");
		ctx.register(TransferServiceConfig.class, StandaloneDataConfig.class, JndiDataConfig.class);
		ctx.refresh();

		// proceed with assertions as above ...
	}
}

现在，我们所有的@Configuration类都可以在引导时对容器使用，并且基于活动的配置文件（在这种情况下为“ dev”），@ @Profileannotation的类及其bean将被处理或跳过。快速说明一下，您可以避免列出上面的每个@Configuration类，而告诉AnnotationConfigApplicationContext只需扫描整个.config包，即可.config检测出我们所有的类。这大致相当于基于通配符（例如**/*-config.xml ）加载Spring XML文件：


		AnnotationConfigApplicationContext ctx = new AnnotationConfigApplicationContext();
		ctx.getEnvironment().setActiveProfiles("dev");
		ctx.scan("com.bank.config.code"); // find and register all @Configuration classes within
		ctx.refresh();

但是，您选择注册@Configuration类，此时我们的任务已完成！我们已经从Spring移植了我们的配置 XML到@Configuration类，并使用AnnotationConfigApplicationContext从这些类直接引导容器。

进一步改善@Configuration类结构

一切在我们的应用程序中都可以正常运行，并且JUnit栏为绿色，但是仍有改进的空间。回忆一下如何将DataSource bean @Autowired到TransferServiceConfig ？这种运作良好，但它并不十分清楚，其中 bean是从哪里来的。如上所述，它可以来自XML或任何其他@Configuration类。我将在下面介绍的技术引入了面向对象的配置，并应进一步实现我们的目标，即拥有一种基于Java的自然配置-可以充分利用IDE的功能。

如果我们考虑StandaloneDataConfig和JndiDataConfig ，那么它们实际上是两个同类，因为它们都声明了具有以下签名的方法：


		public DataSource dataSource();

似乎缺少的只是一个统一两者的接口。让我们介绍一个-我们很快就会在下面看到原因：

src/main/com/bank/config/code/DataConfig.java


interface DataConfig {
	DataSource dataSource();
}

当然还要更新两个@Configuration类以实现此新接口：


@Configuration
public class StandaloneDataConfig implements DataConfig { ... }

@Configuration
public class JndiDataConfig implements DataConfig { ... }

这对我们有什么好处？就像我们将DataSource Bean直接@Autowired直接插入TransferServiceConfig ， 我们也自己注入@Configuration实例 。让我们看一下实际情况：

src/main/com/bank/config/code/TransferServiceConfig.java


@Configuration
public class TransferServiceConfig {

	@Autowired DataConfig dataConfig;

	// ...

	@Bean
	public AccountRepository accountRepository() {
		return new JdbcAccountRepository(dataConfig.dataSource());
	}

	// ...
}

这使我们可以使用IDE在代码库中进行完全导航。以下屏幕截图显示了在调用dataConfig.dataSource()时按CTRL-T以获得“快速层次结构”悬停的结果：

现在很容易问这个问题：“ DataSource bean在哪里定义的？”并将答案限制在实现DataConfig一组类型上。如果我们尝试以对Java开发人员尽可能熟悉和有用的方式进行操作，那也不错。

@Profile更高级用法

值得一提的是，就像许多Spring注释一样，@ @Profile可以用作元注释 。这意味着您可以定义自己的自定义annotation，并用@Profile标记它们，Spring仍会检测到@Profileannotation的存在，就好像直接声明了它一样。


package com.bank.annotation;

@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Profile("dev")
pubilc @interface Dev {
}

这使我们可以使用新的自定义@Devannotation来标记@Component类，而无需使用Spring的@Profile ：


@Dev @Component
public class MyDevService { ... }

或者，从上面的示例中，用@Dev标记我们的StandaloneDataConfig @Dev可以工作：


@Dev @Configuration
public class StandaloneDataConfig { ... }

摘要

XML和@Configuration样式完全支持Spring 3.1的bean定义配置文件功能。无论您喜欢哪种样式，我们希望您都能找到有用的个人资料。保持反馈的到来，因为它将直接影响即将到来的3.1 M2。在下一篇文章中，我们将更深入地研究Spring的新Environment抽象及其在管理应用程序中的配置属性方面的帮助。敬请关注！
