Spring 3.1 M1：介绍@Profile

工程
克里斯·比姆斯
2011年2月14日

介绍

在我先前宣布Spring 3.1 M1的文章中，我讨论了使用Spring时应用的新bean定义配置文件功能。 使用XML配置容器。今天我们将介绍新的@Profile注释，并查看在使用时如何应用相同功能@Configuration类而不是XML。在此过程中，我们将介绍一些最佳设计实践@Configuration类。

调出@Configuration

对于那些不熟悉@Configuration类的人，可以将它们视为与Spring等效的纯Java语言。 XML文件。之前，我们已经在博客中介绍了 此 功能集 ，并且参考文档对此进行了很好的介绍 。如果需要简介或复习，则可能需要重新访问这些资源。

正如我们将在本博文及后续博文中看到的那样，我们对@Configuration Spring 3.1中提供了一种新方法，以使其更完善，并使其成为那些希望在不使用XML的情况下配置其应用程序的人的头等选择。今天的帖子将仅介绍以下增强功能之一： @Profile注解。

与上一篇文章一样，我编写了一个简单的示例，您可以在其中进行后续操作并自己尝试一下。你可以在找到它 https://github.com/cbeams/spring-3.1-profiles-java以及所有设置细节都在README中。该示例包含上一篇文章中介绍的基于XML的配置，以及@Configuration类，分别位于com.bank.config.xml和com.bank.config.code包中。的IntegrationTests每个软件包的JUnit测试用例已被复制；这应该可以帮助您比较和对比引导容器的两种样式。

从XML到@Configuration

让我们潜入吧！我们的任务很简单：采用前面显示的基于XML的应用程序并将其移植到@Configuration样式。我们在上一篇文章中以XML配置开始，如下所示：


<beans xmlns="http://www.springframework.org/schema/beans"
	xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
	xmlns:jdbc="http://www.springframework.org/schema/jdbc"
	xsi:schemaLocation="...">

	<bean id="transferService" class="com.bank.service.internal.DefaultTransferService">
		<constructor-arg ref="accountRepository"/>
		<constructor-arg ref="feePolicy"/>
	</bean>

	<bean id="accountRepository" class="com.bank.repository.internal.JdbcAccountRepository">
		<constructor-arg ref="dataSource"/>
	</bean>

	<bean id="feePolicy" class="com.bank.service.internal.ZeroFeePolicy"/>

	<jdbc:embedded-database id="dataSource">
		<jdbc:script location="classpath:com/bank/config/sql/schema.sql"/>
		<jdbc:script location="classpath:com/bank/config/sql/test-data.sql"/>
	</jdbc:embedded-database>
</beans>

这很容易移植到@Configuration类：

src/main/com/bank/config/code/TransferServiceConfig.java


@Configuration
public class TransferServiceConfig {

	@Bean
	public TransferService transferService() {
		return new DefaultTransferService(accountRepository(), feePolicy());
	}

	@Bean
	public AccountRepository accountRepository() {
		return new JdbcAccountRepository(dataSource());
	}

	@Bean
	public FeePolicy feePolicy() {
		return new ZeroFeePolicy();
	}

	@Bean
	public DataSource dataSource() {
		return new EmbeddedDatabaseBuilder()
			.setType(EmbeddedDatabaseType.HSQL)
			.addScript("classpath:com/bank/config/sql/schema.sql")
			.addScript("classpath:com/bank/config/sql/test-data.sql")
			.build();
	}
}

注意： EmbeddedDatabaseBuilder是基础的组成部分 XML中最初使用的元素。如您所见，在@Bean方法。

至此，我们的@Configuration基于基础的单元测试将通过绿色条：

src/test/com/bank/config/code/IntegrationTests.java


public class IntegrationTests {

	@Test
	public void transferTenDollars() throws InsufficientFundsException {

		AnnotationConfigApplicationContext ctx = new AnnotationConfigApplicationContext();
		ctx.register(TransferServiceConfig.class);
		ctx.refresh();

		TransferService transferService = ctx.getBean(TransferService.class);
		AccountRepository accountRepository = ctx.getBean(AccountRepository.class);

		assertThat(accountRepository.findById("A123").getBalance(), equalTo(100.00));
		assertThat(accountRepository.findById("C456").getBalance(), equalTo(0.00));

		transferService.transfer(10.00, "A123", "C456");

		assertThat(accountRepository.findById("A123").getBalance(), equalTo(90.00));
		assertThat(accountRepository.findById("C456").getBalance(), equalTo(10.00));
	}

}

AnnotationConfigApplicationContext在上面使用，可以直接注册@Configuration和别的@Component -带注释的类。这为我们提供了一种无字符串且无类型安全的容器配置方式。没有XML，这很好，但是在这一点上，我们的应用程序遇到了我们在第一篇文章中看到的相同问题：将应用程序部署到生产环境中时，独立的数据源将变得毫无意义。需要从JNDI查找。

没问题让我们将基于嵌入式和基于JNDI的数据源分解为专用的数据源。 @Configuration类：

src/main/com/bank/config/code/StandaloneDataConfig.java


@Configuration
@Profile("dev")
public class StandaloneDataConfig {

	@Bean
	public DataSource dataSource() {
		return new EmbeddedDatabaseBuilder()
			.setType(EmbeddedDatabaseType.HSQL)
			.addScript("classpath:com/bank/config/sql/schema.sql")
			.addScript("classpath:com/bank/config/sql/test-data.sql")
			.build();
	}

}

src/main/com/bank/config/code/JndiDataConfig.java


@Configuration
@Profile("production")
public class JndiDataConfig {

	@Bean
	public DataSource dataSource() throws Exception {
		Context ctx = new InitialContext();
		return (DataSource) ctx.lookup("java:comp/env/jdbc/datasource");
	}

}

至此，我们宣布了两个不同的DataSource自己的豆子@Profile -带注释@Configuration类。与XML一样，这些类和@Bean根据当前处于活动状态的Spring配置文件，将跳过或处理其中的方法。但是，在看到实际效果之前，我们首先需要完成重构。我们将两种可能分开DataSource豆，但我们如何从内部引用它们TransferServiceConfig –特别是accountRepository()方法？我们有两种选择，而且两种选择都始于了解@Configuration课程是@Autowired注射。这是因为最后@Configuration对象在容器中被当作“只是另一个Spring bean”来管理。让我们来看看：

src/main/com/bank/config/code/TransferServiceConfig.java


@Configuration
public class TransferServiceConfig {

	@Autowired DataSource dataSource;

	@Bean
	public TransferService transferService() {
		return new DefaultTransferService(accountRepository(), feePolicy());
	}

	@Bean
	public AccountRepository accountRepository() {
		return new JdbcAccountRepository(dataSource);
	}

	@Bean
	public FeePolicy feePolicy() {
		return new ZeroFeePolicy();
	}

}

随着使用@Autowired在上面的注释中，我们已经要求Spring容器注入类型为的bean DataSource对我们来说，无论它在哪里声明–在XML中， @Configuration类，否则。然后在accountRepository()方法，注入dataSource字段仅被引用。这是一种实现@Configuration类，并且在概念上没有不同ref两者之间的样式引用在不同XML文件中声明的元素。

重构的最后一步是不仅将单元测试更新为引导TransferServiceConfig ，还有JNDI和独立@Configuration我们的变体DataSource豆：

src/test/com/bank/config/code/IntegrationTests.java


public class IntegrationTests {
	@Test
	public void transferTenDollars() throws InsufficientFundsException {
		AnnotationConfigApplicationContext ctx = new AnnotationConfigApplicationContext();
		ctx.getEnvironment().setActiveProfiles("dev");
		ctx.register(TransferServiceConfig.class, StandaloneDataConfig.class, JndiDataConfig.class);
		ctx.refresh();

		// proceed with assertions as above ...
	}
}

现在我们所有的@Configuration在启动时容器可以使用类，并且基于活动的配置文件（在这种情况下为“ dev”）， @Profile -带注释的类及其Bean将被处理或跳过。快速说明，您可以避免列出每个@Configuration上面的课，而不是告诉AnnotationConfigApplicationContext简单地扫描整个.config包，一举发现我们所有的班级。这与基于通配符加载Spring XML文件（例如， **/*-config.xml ）：


		AnnotationConfigApplicationContext ctx = new AnnotationConfigApplicationContext();
		ctx.getEnvironment().setActiveProfiles("dev");
		ctx.scan("com.bank.config.code"); // find and register all @Configuration classes within
		ctx.refresh();

但是，您选择注册您的@Configuration上课，到此我们的任务就完成了！我们已经从Spring移植了我们的配置 XML到@Configuration类，并使用以下命令直接从这些类引导容器AnnotationConfigApplicationContext 。

进一步改善@Configuration类结构

一切在我们的应用程序中都可以正常运行，并且JUnit栏为绿色，但是仍有改进的空间。回想一下DataSource豆原为@Autowired进入TransferServiceConfig ？这种运作良好，但它并不十分清楚，其中 bean是从哪里来的。如上所述，它可以来自XML或任何其他@Configuration类。我将在下面介绍的技术引入了面向对象的配置，并应进一步实现我们的目标，即拥有一种基于Java的自然配置–可以充分利用IDE的功能。

如果我们考虑一下StandaloneDataConfig和JndiDataConfig ，它们实际上是两个相同的类，因为它们都声明了具有以下签名的方法：


		public DataSource dataSource();

似乎缺少的只是一个统一两者的接口。让我们介绍一个-我们很快就会在下面看到原因：

src/main/com/bank/config/code/DataConfig.java


interface DataConfig {
	DataSource dataSource();
}

当然更新两个@Configuration实现此新接口的类：


@Configuration
public class StandaloneDataConfig implements DataConfig { ... }

@Configuration
public class JndiDataConfig implements DataConfig { ... }

这能给我们带来什么？就像我们@Autowired的DataSource豆直接入TransferServiceConfig ， 我们也要注入@Configuration实例本身 。让我们看一下实际情况：

src/main/com/bank/config/code/TransferServiceConfig.java


@Configuration
public class TransferServiceConfig {

	@Autowired DataConfig dataConfig;

	// ...

	@Bean
	public AccountRepository accountRepository() {
		return new JdbcAccountRepository(dataConfig.dataSource());
	}

	// ...
}

这使我们可以使用IDE在代码库中进行完全导航。以下屏幕截图显示了调用CTRL-T时的结果dataConfig.dataSource()获得“快速层次结构”悬停：

现在很容易问这个问题： DataSource定义的bean？”并将答案限于一组实现的类型DataConfig 。如果我们尝试以对Java开发人员尽可能熟悉和有用的方式进行操作，那也不错。

更高级的使用@Profile

值得一提的是，就像许多Spring注释一样， @Profile可以用作元注释 。这意味着您可以定义自己的自定义注释，并用标记它们@Profile ，而Spring仍会检测到@Profile注释，就好像它是直接声明的一样。


package com.bank.annotation;

@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Profile("dev")
pubilc @interface Dev {
}

这使我们可以标记我们的@Component具有新习惯的类@Dev注释，而不是要求使用Spring的@Profile ：


@Dev @Component
public class MyDevService { ... }

或者，从上述示例中， StandaloneDataConfig与@Dev也会工作：


@Dev @Configuration
public class StandaloneDataConfig { ... }
