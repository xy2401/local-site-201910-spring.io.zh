Spring 3.1 M1：引入FeatureSpecification支持

工程
克里斯·比姆斯
2011年2月17日

更新： FeatureSpecification Spring Framework 3.1 M2中删除了此博客文章中描述的功能，以支持@Enable*注释。有关更多信息，请参见3.1 M2公告 。

介绍

在 本 系列的早期，我谈到了新的@Profile注释可以与@Configuration类，以利用Spring的bean定义配置文件 。今天，我们将介绍Spring 3.1中基于代码的配置格局的全新功能： FeatureSpecification类及其相关支持。

我整理了一个示例项目，以配合这篇文章。在找到它 https://github.com/cbeams/spring-3.1-featurespec并按照README中的说明进行操作。

从至@Bean

支持@Configuration Spring 3.0中添加的类实质上提供了一种用Java而不是XML编写bean定义的机制。例如：

/src/main/com/bank/config/xml/transfer-service-config.xml


<beans>
	<bean id="accountRepository" class="com.bank.repository.internal.JdbcAccountRepository">
		<constructor-arg ref="dataSource"/>
	</bean>

	<!-- 'dataSource' and other bean definitions -->
</beans>

可以翻译成以下@Configuration类：

/src/main/com/bank/config/xml/TransferServiceConfig.java


@Configuration
public class TransferServiceConfig {

	@Bean
	public AccountRepository accountRepository() {
		return new JdbcAccountRepository(dataSource());
	}

	@Bean
	public DataSource dataSource() {
		// create, configure and return the DataSource ...
	}
}

XML配置将在以下位置引导：


new GenericXmlApplicationContext("com/bank/config/xml/transfer-service-config.xml");

的@Configuration将以类似方式进行引导，但是将类作为输入：


new AnnotationConfigApplicationContext(TransferServiceConfig.class);

这种方法行之有效，但仅限于表达单个bean定义-迄今为止，尚无与Spring的XML名称空间等效的支持。 @Configuration世界。Spring的XML名称空间，例如和允许以非常简洁的方式进行大量的配置。例如，正如许多用户所知道的那样，启用Spring的注释驱动的事务管理在XML中是一味的：

/src/main/com/bank/config/xml/transfer-service-config.xml


<tx:annotation-driven transaction-manager="txManager"/>

在幕后，当tx:annotation-driven元素被处理后，代表您创建了许多支持的基础结构级Bean定义，您无需直接关心这些定义，这是一件好事–它可以正常工作 ，确保您的任何Spring Bean都标有@Transactional有一个TransactionInterceptor 。在Spring 3.1 M1之前，为了在@Configuration类，您将不得不自己声明每个这些基础设施bean @Bean方法。这是一些人实际上已经走下的路，我认为可以肯定地说这不是一种首选方法。为了避免这种额外的工作，Spring 3.0提供了@ImportResource注释允许@Configuration当需要名称空间时，可以有选择地“插入” XML配置的类：

/src/main/com/bank/config/code/TransferServiceConfig.java


@Configuration
@ImportResource("com/bank/config/xml/tx-config.xml")
public class TransferServiceConfig {
	// ...
}

在上面的示例中， tx-config.xml将包含宣言。这个混合模型可以工作很高兴，但是同样，在XML和代码之间拆分配置并不是完全理想。所需要的是一种允许在代码中指定所有应用程序配置的方法。

从至TxAnnotationDriven

要了解FeatureSpecification类，最好先考虑一下Spring的许多XML名称空间的性质。考虑以下：

这些元素中的每一个都指定 Spring容器功能的配置：组件扫描功能，注释驱动的事务管理功能等。 FeatureSpecification类允许用户完全在代码中配置Spring容器的这些相同功能。

让我们来看看FeatureSpecification在行动中， 至TxAnnotationDriven举个例子：


<beans>
    <tx:annotation-driven transaction-manager="txManager"/>
</beans>

变成：

/src/main/com/bank/config/code/TxFeature.java


@FeatureConfiguration
class TxFeature {

	@Feature
	public TxAnnotationDriven txAnnotationDriven(PlatformTransactionManager txManager) {
		return new TxAnnotationDriven(txManager);
	}

}

@FeatureConfiguration类由Spring容器处理，期望@Feature返回的方法FeatureSpecifation诸如TxAnnotationDriven 。

@FeatureConfiguration类被设计为与@Configuration类。引导容器时，两者可以互换使用：


ApplicationContext ctx =
	new AnnotationConfigApplicationContext(TransferServiceConfig.class, TxFeatures.class);
TransferService transferService = ctx.getBean(TransferService.class);
// ...

而且可能@Import另一个：


@Configuration
@Import(TxFeature.class)
public class TransferServiceConfig { ... }

要么：


@FeatureConfiguration
@Import(TransferServiceConfig.class)
public class TxFeature { ... }

@FeatureConfiguration类可能包含一个或多个@Feature方法。例如，我们可以在同一类中配置组件扫描和注释驱动的事务管理：


@FeatureConfiguration
class MyFeatures {

	@Feature
	public TxAnnotationDriven tx(PlatformTransactionManager txManager) {
		return new TxAnnotationDriven(txManager);
	}

	@Feature
	public ComponentScanSpec scan() {
		return new ComponentScanSpec("com.bank.service");
	}

}

@Feature容器在引导时调用方法，并且FeatureSpecification返回的对象将被处理，以配置相关功能的实际内部。如果是TxAnnotationDriven ，配置了代理创建，注册了事务拦截器Bean，等等。如果是ComponentScanSpec ， 一种ClassPathBeanDefinitionScanner配置有指定的值，然后运行以检测和注册标有的类@Component或其他Spring构造型注释。

@Feature方法可以接受参数以获取对Spring Bean的引用：


	@Feature
	public TxAnnotationDriven tx(PlatformTransactionManager txManager) {
		return new TxAnnotationDriven(txManager);
	}

为了让容器提供参数，必须有一个类型为Spring的bean PlatformTransactionManager在其他地方声明，可能在@Configuration类。“参数自动装配”的概念在框架中并不陌生。例如，这类似于@InitBinder和@RequestMapping方法在Spring MVC中有效。引用实际Spring bean实例的能力通过避免使用基于字符串的bean名称来保持类型安全。

努力状况

第一个里程碑版本已提供以下FeatureSpecification实现：

  • ComponentScanSpec
  • TxAnnotationDriven
  • MvcAnnotationDriven
  • MvcResources
  • MvcViewControllers
  • MvcDefaultServletHandler

您会看到许多Mvc* Rossen即将在本系列文章中使用的类型，涵盖了Spring 3.1 M1中对Spring MVC的增强。

请注意，并非所有的Spring XML名称空间元素都是相同的。有些像已经提到的那些用于配置容器功能，而有些像以及一切名称空间只是用于XML的便捷实用工具，而在没有任何特殊支持的情况下，可以很容易地在代码中完成。因此，不要期望Spring XML元素和FeatureSpecification类型。仅将那些有意义的元素翻译为代码。

开放模型

就像用户可以定义自己的Spring XML名称空间并注册自己的一样NamespaceHandler和BeanDefinitionParser实现，自然是整个Feature*模型也一样开放。我会参考Javadoc的完整详细信息，但是只要您可以编写自己的文档就足够了FeatureSpecification键入并轻松使用它们@Feature方法。

对于那些已经实现了自定义名称空间的人， BeanDefinitionParsers ，您会很感兴趣的是，在该框架内，我们已经重构了现有的BeanDefinitionParsers委托给FeatureSpecifications内部。这极大地简化了解析器的实现，并确保了跨XML和@Configuration样式。我们鼓励您在适当的时候考虑进行同样的重构。

给快乐的XML用户的注释

尽管Spring继续基于代码的配置向前发展，但必须一遍又一遍地指出，我们绝不会消除，弃用或以其他方式鼓励用户逐步淘汰使用XML来配置容器。Spring XML仍然很流行，并且许多用户对此感到满意，尤其是考虑到SpringSource Tool Suite提供的用于编写bean配置文件的高级工具选项。 @Configuration - 现在@FeatureConfiguration –类为那些希望尽可能多地使用Java的人提供了另一种一流的配置方法。因此，请放心，Spring XML不会无处可寻-实际上，在Spring 3.1 M1中，我们通过添加嵌套功能对XML给予了足够的关注元素c:命名空间可以更方便地进行构造函数注入，并且cache: nampespace配置Spring的新缓存功能。
