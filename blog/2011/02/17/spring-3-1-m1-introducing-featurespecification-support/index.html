Spring 3.1 M1：引入FeatureSpecification支持

工程
克里斯·比姆斯
2011年2月17日

更新：本博客中描述的FeatureSpecification功能在Spring Framework 3.1 M2中已被删除，转而使用@Enable*annotation。有关更多信息，请参见3.1 M2公告 。

介绍

在 本 系列的前面，我谈到了如何将新的@Profileannotation与@Configuration类结合使用，以利用Spring的bean定义配置文件 。今天，我们将研究Spring 3.1中基于代码的配置领域的全新功能： FeatureSpecification类及其相关支持。

我整理了一个示例项目，以配合这篇文章。在找到它 https://github.com/cbeams/spring-3.1-featurespec并按照README中的说明进行操作。

从到@Bean

在Spring 3.0中添加的对@Configuration类的支持实质上提供了一种用Java而不是XML编写bean定义的机制。例如：

/src/main/com/bank/config/xml/transfer-service-config.xml

可以转换为以下@Configuration类：

/src/main/com/bank/config/xml/TransferServiceConfig.java


@Configuration
public class TransferServiceConfig {

	@Bean
	public AccountRepository accountRepository() {
		return new JdbcAccountRepository(dataSource());
	}

	@Bean
	public DataSource dataSource() {
		// create, configure and return the DataSource ...
	}
}

XML配置将在以下位置引导：


new GenericXmlApplicationContext("com/bank/config/xml/transfer-service-config.xml");

@Configuration将类似地被引导，但是将类作为输入：


new AnnotationConfigApplicationContext(TransferServiceConfig.class);

这种方法很好用，但仅限于表达单个bean定义-迄今为止， @Configuration世界中还没有与Spring的XML名称空间等效的支持。Spring的XML名称空间，例如和允许以非常简洁的方式进行大量的配置。例如，正如许多用户所知道的那样，启用Spring的注释驱动的事务管理在XML中是一味的：

/src/main/com/bank/config/xml/transfer-service-config.xml

在幕后，处理tx:annotation-driven元素时，代表您创建了许多支持的基础结构级Bean定义，您无需直接关心这些定义，这是一件好事–它可以正常工作 ，确保任何标有@Transactional Spring bean都带有TransactionInterceptor代理。在Spring 3.1 M1之前，为了在@Configuration类中模拟此功能，您必须将每个基础结构Bean声明为单独的@Bean方法。这是一些人实际上已经走下的路，我可以肯定地说这不是首选的方法。为了避免这种额外的工作，Spring 3.0提供了@ImportResourceannotation，以允许@Configuration类在需要名称空间时有选择地“拉入” XML配置：

/src/main/com/bank/config/code/TransferServiceConfig.java


@Configuration
@ImportResource("com/bank/config/xml/tx-config.xml")
public class TransferServiceConfig {
	// ...
}

在上面的示例中， tx-config.xml将包含宣言。这个混合模型可以工作很高兴，但是同样，在XML和代码之间拆分配置并不是完全理想。所需要的是一种允许在代码中指定所有应用程序配置的方法。

从到TxAnnotationDriven

要了解FeatureSpecification类，最好首先考虑一下Spring的许多XML名称空间的性质。考虑以下：

这些元素中的每一个都指定 Spring容器功能的配置：组件扫描功能，注释驱动的事务管理功能等FeatureSpecification类允许用户完全在代码中配置Spring容器的这些相同功能。

让我们看看一个FeatureSpecification情况，从以TxAnnotationDriven为例：

变成：

/src/main/com/bank/config/code/TxFeature.java


@FeatureConfiguration
class TxFeature {

	@Feature
	public TxAnnotationDriven txAnnotationDriven(PlatformTransactionManager txManager) {
		return new TxAnnotationDriven(txManager);
	}

}

Spring容器会使用@FeatureConfiguration类来处理@Feature方法， @Feature方法会返回FeatureSpecifation类型，例如TxAnnotationDriven 。

@FeatureConfiguration类被设计为@Configuration类的伴侣。引导容器时，两者可以互换使用：


ApplicationContext ctx =
	new AnnotationConfigApplicationContext(TransferServiceConfig.class, TxFeatures.class);
TransferService transferService = ctx.getBean(TransferService.class);
// ...

一个可以@Import另一个：


@Configuration
@Import(TxFeature.class)
public class TransferServiceConfig { ... }

要么：


@FeatureConfiguration
@Import(TransferServiceConfig.class)
public class TxFeature { ... }

@FeatureConfiguration类可以包含一个或多个@Feature方法。例如，我们可以在同一类中配置组件扫描和注释驱动的事务管理：


@FeatureConfiguration
class MyFeatures {

	@Feature
	public TxAnnotationDriven tx(PlatformTransactionManager txManager) {
		return new TxAnnotationDriven(txManager);
	}

	@Feature
	public ComponentScanSpec scan() {
		return new ComponentScanSpec("com.bank.service");
	}

}

容器在引导时调用@Feature方法，并处理返回的FeatureSpecification对象，以配置相关功能的实际内部结构。在TxAnnotationDriven的情况下，配置了代理创建，注册了事务拦截器Bean，等等。对于ComponentScanSpec ，将使用指定的值配置ClassPathBeanDefinitionScanner ，然后运行该ClassPathBeanDefinitionScanner ，以检测和注册标有@Component或其他Spring构造型注释的类。

@Feature方法可以接受参数以获取对Spring Bean的引用：


	@Feature
	public TxAnnotationDriven tx(PlatformTransactionManager txManager) {
		return new TxAnnotationDriven(txManager);
	}

为了由容器提供参数，必须在其他地方（可能在@Configuration类中）声明一个类型为PlatformTransactionManager的Spring bean。“参数自动装配”的概念对于框架而言并不陌生。例如，这类似于Spring MVC中@InitBinder和@RequestMapping方法的工作方式。通过避免使用基于字符串的Bean名称，可以引用实际的Spring Bean实例，从而保持了类型安全。

请注意，并非所有的Spring XML名称空间元素都是相同的。有些像已经提到的那些用于配置容器功能，而有些像还有一切名称空间只是用于XML的便捷实用工具，而在没有任何特殊支持的情况下，可以很容易地在代码中完成。因此，不要期望Spring XML元素和FeatureSpecification类型之间是一对一的映射。仅将那些有意义的元素翻译为代码。

开放式

就像用户可以定义自己的Spring XML名称空间并注册自己的NamespaceHandler和BeanDefinitionParser实现一样，整个Feature*模型自然也一样开放。我将参考Javadoc的完整细节，但足以说明您可以编写自己的FeatureSpecification类型并在@Feature方法中轻松使用它们。

对于已经实现自定义名称空间和BeanDefinitionParsers ，您会很感兴趣地知道，在框架内，我们已经重构了现有的BeanDefinitionParsers以便在内部委托给FeatureSpecifications 。这极大地简化了解析器的实现，并确保跨XML和@Configuration样式重用关键的bean注册逻辑。我们鼓励您在适当的时候考虑进行同样的重构。

给快乐的XML用户的注释

尽管Spring继续基于代码的配置向前发展，但必须一遍又一遍地指出，我们绝不会消除，弃用或以其他方式鼓励用户逐步淘汰使用XML来配置容器。Spring XML仍然很受欢迎，并且许多用户对此感到满意，特别是考虑到SpringSource Tool Suite提供的用于编写bean配置文件的高级工具选项。@Configuration –现在是@FeatureConfiguration –类为喜欢尽可能多地使用Java的人员提供了另一种一流的配置方法。因此，请放心，Spring XML不会无处可寻–实际上，在Spring 3.1 M1中，我们通过添加嵌套功能对XML给予了足够的关注元素， c:名称空间（用于更方便的构造函数注入）和cache: nampespace（用于配置Spring的新缓存功能）。
