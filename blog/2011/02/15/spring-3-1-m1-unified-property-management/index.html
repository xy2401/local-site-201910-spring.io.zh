Spring 3.1 M1：统一属性管理

工程
克里斯·比姆斯
2011年2月15日

在本系列的前两篇 文章中，我描述了bean定义配置文件功能，以及它与Spring 3.1 M1中新增的Environment抽象的关系。今天，我们将研究Environment的第二个方面–它如何帮助简化配置属性管理的问题。

了解财产来源

Spring的Environment抽象提供了对属性源可配置层次结构的搜索操作。为了全面解释，请考虑以下内容：


ApplicationContext ctx = new GenericApplicationContext();
Environment env = ctx.getEnvironment();
boolean containsFoo = env.containsProperty("foo");
System.out.println("Does my environment contain the 'foo' property? " + containsFoo);

在上面的代码片段中，我们看到了一种高级方式来询问Spring是否为当前环境定义了'foo'属性。为了回答这个问题， Environment对象在一组PropertySource对象上执行搜索。PropertySource是对任何键-值对源的简单抽象，并且Spring的DefaultEnvironment配置有两个PropertySource对象-一个代表JVM系统属性的集合（例如System.getProperties() ），另一个代表系统环境的集合变量（例如System.getenv() ）[1]。这意味着，如果在运行时存在“ foo”系统属性或“ foo”环境变量，则对env.containsProperty(“foo”)的调用将返回true 。

执行的搜索是分层的。默认情况下，系统属性优先于环境变量，因此，如果在调用env.getProperty(“foo”)同时在两个地方都设置了'foo'属性，则系统属性值将'win'并优先返回在环境变量上。

最重要的是，整个机制是可配置的。也许您有一个想要集成到此搜索中的自定义属性源。没问题–只需实现并实例化您自己的PropertySource并将其添加到当前Environment的PropertySources集中：


ConfigurableApplicationContext ctx = new GenericApplicationContext();
MutablePropertySources sources = ctx.getEnvironment().getPropertySources();
sources.addFirst(new MyPropertySource());

在上面的代码中， MyPropertySource已在搜索中以最高优先级添加。如果它包含'foo'属性，它将在任何其他PropertySource的任何'foo'属性之前被检测并返回。MutablePropertySources API公开了许多方法，这些方法可以精确地控制属性源集。浏览Javadoc以获取全部详细信息。

利用财产来源

既然您了解了属性源的基本知识以及它们与Environment的关系，那么您可能想知道，作为Spring应用程序的开发者，所有这些与您之间的关系如何。让我们考虑几个场景，看看它们如何结合在一起。

在Spring 3.1之前的版本中，占位符的值只能根据JVM系统属性或环境变量来解析元素[2]。情况不再如此。由于Environment抽象已集成到整个容器中，因此很容易通过它路由占位符的解析。这意味着您可以按照自己喜欢的任何方式配置解析过程：更改搜索系统属性和环境变量的优先级，或者将其完全删除；将您自己的财产来源添加到适当的组合中。

从Spring 3.1开始， 不再注册PropertyPlaceholderConfigurer ，而是注册PropertySourcesPlaceholderConfigurer [3]。该组件仍将使用datasource.properties文件来替换上面的${database.*}占位符，但是如果在文件中找不到属性，则该组件将退回到当前Environment的PropertySources集。同样，这给了您更多的控制权。在进行此更改之前，唯一的后备选项是系统属性和环境变量。

在Web应用程序中处理属性源

到目前为止，我们已经看到了如何在独立的应用程序中访问和操作属性源，在该应用程序中，可以通过编程方式访问ApplicationContext 。然而，实际上，许多Spring应用程序都是web应用程序，其中Spring的ContextLoaderListener为您管理ApplicationContext 。由于这个原因，我们引入了ApplicationContextInitializer接口及其伴侣， contextInitializerClasses servlet上下文参数。看一看：

web.xml



    contextInitializerClasses
    com.bank.MyInitializer



public class MyInitializer implements ApplicationContextInitializer {
	public void initialize(ConfigurableWebApplicationContext ctx) {
		PropertySource ps = new MyPropertySource();
		ctx.getEnvironment().getPropertySources().addFirst(ps);
		// perform any other initialization of the context ...
	}
}

实现和注册ApplicationContextInitializer提供了一种在刷新之前与应用程序上下文进行交互的简单方法。这是操作属性源的理想场所，但您也可以调用setConfigLocations(…)或设计为在refresh()之前调用的任何其他方法。

摘要

Spring的Environment抽象提供了一个配置配置文件和属性的位置 。如先前文章所述，配置文件确定应为给定的部署上下文注册哪些bean定义；这篇文章中描述的属性支持提供了对任何属性源的一致抽象，从而在整个应用程序配置中提供了更灵活的属性访问和占位符解析。

在本系列的下一篇文章中，我们将研究Spring 3.1如何通过FeatureSpecification支持使基于100％基于Java（无XML）的应用程序配置成为现实-这是自@Configuration类支持引入的自然发展春季3.0。

脚注

[1]：这些默认属性源存在于DefaultEnvironment ，可在独立应用程序中使用。DefaultWebEnvironment填充了其他默认属性源，包括servlet配置和servlet上下文参数。同样， DefaultPortletEnvironment可以访问portlet配置和portlet上下文参数作为属性源。两者都可以选择启用JndiPropertySource 。有关详细信息，请参见Javadoc 。

[2]：因为处理一定发生之前的元素BeanFactoryPostProcessors被调用，这意味着即使PropertyPlaceholderConfigurer这里忍不住。由于Environment及其PropertySources集是在刷新容器之前配置的，因此占位符在可以针对Environment解决元素，而没有任何生命周期问题。

[3]：在某些情况下， 仍然会注册一个PropertyPlaceholderConfigurer 。在spring-context模式的3.1版本中，已从property-placeholder元素中删除了system-properties-mode属性。这是因为在PropertySources // Environment感知的世界中，此属性不再有意义。但是，如果您是根据Spring 3.1构建的，但仍使用spring-context-3.0.xsd模式并设置了system-properties-mode属性， 将恢复为注册PropertyPlaceholderConfigurer以便遵循此设置的确切语义。无论如何，这种方法都可以保持向后兼容性。
