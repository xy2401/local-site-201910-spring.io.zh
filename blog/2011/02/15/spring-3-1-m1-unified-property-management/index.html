Spring 3.1 M1：统一属性管理

工程
克里斯·比姆斯
2011年2月15日

在本系列的前两篇 文章中，我描述了bean定义配置文件功能，以及它与Environment Spring 3.1 M1中新增的抽象。今天，我们将看看Environment –它如何帮助简化对配置属性管理的关注。

了解财产来源

春天的Environment抽象提供了对属性源可配置层次结构的搜索操作。为了全面解释，请考虑以下内容：


ApplicationContext ctx = new GenericApplicationContext();
Environment env = ctx.getEnvironment();
boolean containsFoo = env.containsProperty("foo");
System.out.println("Does my environment contain the 'foo' property? " + containsFoo);

在上面的代码片段中，我们看到了一种高级方式来询问Spring是否为当前环境定义了'foo'属性。为了回答这个问题， Environment对象对一组进行搜索PropertySource对象。一种PropertySource是对任何键值对来源的简单抽象，而Spring的DefaultEnvironment配置有两个PropertySource对象–一个代表一组JVM系统属性的对象System.getProperties() ）和代表系统环境变量集（一个System.getenv() ）[1]。这意味着，如果在运行时存在“ foo”系统属性或“ foo”环境变量，则对env.containsProperty(“foo”)将返回true 。

执行的搜索是分层的。默认情况下，系统属性优先于环境变量，因此，如果在调用的同时在两个地方都设置了'foo'属性， env.getProperty(“foo”) ，系统属性值将“赢得”并优先于环境变量返回。

最重要的是，整个机制是可配置的。也许您有一个想要集成到此搜索中的自定义属性源。没问题–只需实现并实例化您自己的实例PropertySource并将其添加到PropertySources目前Environment ：


ConfigurableApplicationContext ctx = new GenericApplicationContext();
MutablePropertySources sources = ctx.getEnvironment().getPropertySources();
sources.addFirst(new MyPropertySource());

在上面的代码中， MyPropertySource已在搜索中以最高优先级添加。如果它包含'foo'属性，它将被检测并在其他任何'foo'属性之前返回PropertySource 。的MutablePropertySources API公开了许多方法，可以对属性源集进行精确操作。浏览Javadoc以获取全部详细信息。

在Spring 3.1之前的版本中，占位符的值只能根据JVM系统属性或环境变量来解析元素[2]。情况不再如此。因为Environment抽象集成在整个容器中，很容易通过它路由占位符的解析。这意味着您可以按照自己喜欢的任何方式配置解析过程：更改搜索系统属性和环境变量的优先级，或者将其完全删除；将您自己的财产来源添加到适当的组合中。

在Web应用程序中处理属性源

到目前为止，我们已经了解了如何在独立的应用程序中访问和操作属性源，在该应用程序中，可以通过编程方式访问ApplicationContext 。但是实际上，许多Spring应用程序都是webapp，其中ApplicationContext由Spring的为您管理ContextLoaderListener 。因此，我们引入了ApplicationContextInitializer接口及其伴侣contextInitializerClasses servlet上下文参数。看一看：

web.xml



    contextInitializerClasses
    com.bank.MyInitializer



public class MyInitializer implements ApplicationContextInitializer {
	public void initialize(ConfigurableWebApplicationContext ctx) {
		PropertySource ps = new MyPropertySource();
		ctx.getEnvironment().getPropertySources().addFirst(ps);
		// perform any other initialization of the context ...
	}
}

实施和注册ApplicationContextInitializer提供了一种在刷新之前与应用程序上下文进行交互的简单方法。这是处理财产来源的理想场所，但您也可以致电setConfigLocations(…)或设计为在之前调用的任何其他方法refresh() 。

摘要

春天的Environment抽象提供了一个配置配置文件和属性的位置 。如先前文章所述，配置文件确定应为给定的部署上下文注册哪些bean定义；这篇文章中描述的属性支持提供了对任何属性源的一致抽象，从而在整个应用程序配置中提供了更灵活的属性访问和占位符解析。

脚注

[1]：这些默认属性源存在于DefaultEnvironment ，用于独立应用程序。 DefaultWebEnvironment用其他默认属性源（包括servlet配置和servlet上下文参数）填充。 DefaultPortletEnvironment同样，可以访问portlet config和portlet上下文参数作为属性源。两者都可以选择启用JndiPropertySource 。有关详细信息，请参见Javadoc 。

[2]：因为处理元素必定发生在BeanFactoryPostProcessors被调用，这意味着即使PropertyPlaceholderConfigurer不禁在这里。因为Environment及其一组PropertySources在容器刷新之前配置，占位符在元素可以针对Environment没有任何生命周期问题。

[3]：在某些情况下， 仍然会注册一个PropertyPlaceholderConfigurer 。在3.1版中spring-context模式， system-properties-mode属性已从中删除property-placeholder元件。这是因为此属性在PropertySources -/ Environment意识的世界。但是，如果您针对Spring 3.1进行构建，但仍使用spring-context-3.0.xsd模式并设置system-properties-mode属性， 将恢复为注册PropertyPlaceholderConfigurer为了尊重此设置的确切语义。在任何情况下，此方法都会保持向后兼容性。
