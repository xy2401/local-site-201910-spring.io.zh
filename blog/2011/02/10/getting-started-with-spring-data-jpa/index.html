Spring Data JPA入门

工程
奥利弗·德罗博姆（Oliver Drotbohm）
2011年2月10日

当我们刚刚发布Spring Data JPA项目的第一个里程碑时，我想向您快速介绍其功能。您可能知道，Spring框架提供了构建基于JPA的数据访问层的支持。那么，Spring Data JPA对基础支持又增加了什么呢？为了回答这个问题，我想从使用普通JPA + Spring实现的示例域的数据访问组件开始，并指出仍有待改进的地方。完成之后，我将重构实现以使用Spring Data JPA功能来解决这些问题领域。在Github上可以找到示例项目以及重构步骤的逐步指南。

域


为简单起见，我们从一个很小的知名域开始：我们有具有Account的Customer 。

@Entity
public class Customer {

  @Id
  @GeneratedValue(strategy = GenerationType.AUTO)
  private Long id;

  private String firstname;
  private String lastname;

  // … methods omitted
}

@Entity
public class Account {

  @Id
  @GeneratedValue(strategy = GenerationType.AUTO)
  private Long id;

  @ManyToOne
  private Customer customer;

  @Temporal(TemporalType.DATE)
  private Date expiryDate;

  // … methods omitted
}

该Account有一个有效期，我们将在以后使用。除此之外，类或映射没有什么特别的-它使用纯JPA注释。现在让我们看一下管理Account对象的组件：

@Repository
@Transactional(readOnly = true)
class AccountServiceImpl implements AccountService {

  @PersistenceContext
  private EntityManager em;

  @Override
  @Transactional
  public Account save(Account account) {

    if (account.getId() == null) {
      em.persist(account);
      return account;
    } else {
      return em.merge(account);
    }
  }

  @Override
  public List findByCustomer(Customer customer) {

    TypedQuery query = em.createQuery("select a from Account a where a.customer = ?1", Account.class);
    query.setParameter(1, customer);

    return query.getResultList();
  }
}

我特意为类*Service命名，以避免名称冲突，因为我们在开始重构时将引入存储库层。但是从概念上讲，这里的类是存储库而不是服务。那么，我们这里实际上有什么？

该类带有@Repositoryannotation，以启用从JPA异常到Spring的DataAccessException层次结构的异常转换。除此之外，我们使用@Transactional来确保save(…)操作正在事务中运行，并允许为findByCustomer(…)设置readOnly flag（在类级别findByCustomer(…) 。这会导致持久性提供程序内部以及数据库级别的一些性能优化。

因为我们希望使客户摆脱在EntityManager上调用merge(…)还是persist(…) ，所以我们使用Account的id -field来决定是否将Account对象视为新对象。当然，可以将这种逻辑提取到一个公共的超类中，因为我们可能不想为每个特定于域对象的存储库实现重复此代码。查询方法也非常简单：我们创建查询，绑定参数并执行查询以获得结果。几乎直截了当，以至于可以将实现代码视为样板，因为有点想象力，它可以从方法签名派生而来：我们期望一个Account List ，查询与方法名称非常接近，我们只需绑定方法参数。如您所见，还有改进的空间。

Spring数据存储库支持

在开始重构实现之前，请注意，示例项目包含测试案例，可以在重构过程中运行这些测试案例以验证代码是否仍然有效。现在让我们看看如何改善实施。

Spring Data JPA提供了一个存储库编程模型，该模型以每个受管域对象的接口开头：

public interface AccountRepository extends JpaRepository { … }

定义此接口有两个目的：首先，通过扩展JpaRepository我们在我们的类型中获得了一堆通用的CRUD方法，该方法允许保存Account ，将其删除等等。其次，这将允许Spring Data JPA存储库基础结构扫描该接口的类路径并为其创建Spring Bean。

要让Spring创建一个实现此接口的bean，您需要做的就是使用Spring JPA命名空间并使用适当的元素激活存储库支持：

这将扫描com.acme.repositories所有软件包，以获取扩展JpaRepository接口，并JpaRepository创建一个Spring Bean，该bean由SimpleJpaRepository的实现提供SimpleJpaRepository 。让我们迈出第一步，重构一下AccountService实现，以使用我们新引入的存储库接口：

@Repository
@Transactional(readOnly = true)
class AccountServiceImpl implements AccountService {

  @PersistenceContext
  private EntityManager em;

  @Autowired
  private AccountRepository repository;

  @Override
  @Transactional
  public Account save(Account account) {
    return repository.save(account);
  }

  @Override
  public List findByCustomer(Customer customer) {

    TypedQuery query = em.createQuery("select a from Account a where a.customer = ?1", Account.class);
    query.setParameter(1, customer);

    return query.getResultList();
  }
}

重构之后，我们只需将对save(…)的调用委托给存储库即可。默认情况下，如果存储库实现的id -property为null则该存储库实现会将其视为新实体，就像您在前面的示例中看到的那样（请注意，如有必要，您可以对该决定进行更详细的控制）。另外，我们可以摆脱方法的@Transactionalannotation，因为Spring Data JPA存储库实现的CRUD方法已经使用@Transactional进行了注释。

接下来，我们将重构查询方法。让我们对查询方法遵循与保存方法相同的委托策略。我们在存储库接口上引入了一种查询方法，并将我们的原始方法委托给该新引入的方法：

@Transactional(readOnly = true) 
public interface AccountRepository extends JpaRepository {

  List findByCustomer(Customer customer); 
}

@Repository
@Transactional(readOnly = true)
class AccountServiceImpl implements AccountService {

  @Autowired
  private AccountRepository repository;

  @Override
  @Transactional
  public Account save(Account account) {
    return repository.save(account);
  }

  @Override
  public List findByCustomer(Customer customer) {
    return repository.findByCustomer(Customer customer);
  }
}

让我在这里添加有关事务处理的快速注释。在这个非常简单的例子，我们可以删除@Transactional从注释AccountServiceImpl类完全是作为存储库的CRUD方法是事务性和查询方法标有@Transactional(readOnly = true)在仓库界面了。当前设置最好将服务级别的方法标记为事务性的（即使在这种情况下不需要），因为在查看服务级别时会清楚地知道操作在事务中进行，这是最好的。除此之外，如果修改了服务层方法以对存储库方法进行多次调用，那么所有代码仍将在单个事务内执行，因为存储库的内部事务将简单地加入从服务层开始的外部事务。参考资料中详细记录了存储库的事务处理行为和进行调整的可能性。

尝试再次运行测试用例，看看它是否有效。停止，我们没有为findByCustomer(…)提供任何实现，对吗？这是如何运作的？

查询方法

当春天JPA的数据为创建的Spring bean实例AccountRepository接口它会检查它定义的所有查询方法，并导出为他们每个人的查询。默认情况下，Spring Data JPA将自动解析方法名称并从中创建查询。该查询是使用JPA标准API实现的。在这种情况下， findByCustomer(…)方法在逻辑上等效于JPQL查询，请select a from Account a where a.customer = ?1 。分析方法名称的解析器支持相当多的关键字，例如And ， Or ， GreaterThan ， LessThan ， Like ， IsNull ， Not等。如果愿意，还可以添加OrderBy子句。有关详细的概述，请查阅参考文档 。这种机制为我们提供了一种查询方法编程模型，就像您习惯于Grails或Spring Roo一样。

现在，让我们假设您要明确使用的查询。为此，您可以在实体或您的orm.xml中的注释中声明遵循命名约定（在本例中为Account.findByCustomer ）的JPA命名查询。或者，您可以使用@Queryannotation存储库方法：

@Transactional(readOnly = true)
public interface AccountRepository extends JpaRepository {

  @Query("")
  List findByCustomer(Customer customer); 
}

现在，我们使用到目前为止已经看到的功能，对CustomerServiceImpl进行之前/之后的比较：

@Repository
@Transactional(readOnly = true)
public class CustomerServiceImpl implements CustomerService {

  @PersistenceContext
  private EntityManager em;

  @Override
  public Customer findById(Long id) {
    return em.find(Customer.class, id);
  }

  @Override
  public List findAll() {
    return em.createQuery("select c from Customer c", Customer.class).getResultList();
  }

  @Override
  public List findAll(int page, int pageSize) {

    TypedQuery query = em.createQuery("select c from Customer c", Customer.class);

    query.setFirstResult(page * pageSize);
    query.setMaxResults(pageSize);

    return query.getResultList();
  }

  @Override
  @Transactional
  public Customer save(Customer customer) {

    // Is new?
    if (customer.getId() == null) {
      em.persist(customer);
      return customer;
    } else {
      return em.merge(customer);
    }
  }

  @Override
  public List findByLastname(String lastname, int page, int pageSize) {

    TypedQuery query = em.createQuery("select c from Customer c where c.lastname = ?1", Customer.class);

    query.setParameter(1, lastname);
    query.setFirstResult(page * pageSize);
    query.setMaxResults(pageSize);

    return query.getResultList();
  }
}

好的，让我们创建CustomerRepository并首先消除CRUD方法：

public interface CustomerRepository extends JpaRepository { … }

  }

  public Customer save(Customer customer) {
    return repository.save(customer);
  }

  @Override
  public List findByLastname(String lastname, int page, int pageSize) {

    TypedQuery query = em.createQuery("select c from Customer c where c.lastname = ?1", Customer.class);

    query.setParameter(1, lastname);
    query.setFirstResult(page * pageSize);
    query.setMaxResults(pageSize);

    return query.getResultList();
  }
}

到目前为止，一切都很好。现在剩下的是两种用于处理常见情况的方法：您不想访问给定查询的所有实体，而只访问它们的一页（例如，页面大小为10的第1页）。现在，使用两个整数适当地限制了查询，以解决此问题。这有两个问题。这两个整数实际上一起代表一个概念，这里不再赘述。除此之外，我们还返回一个简单的List因此我们会丢失有关实际数据页面的元数据信息：这是第一页吗？是最后一个吗？一共有几页？Spring Data提供了一个抽象，它包含两个接口： Pageable （用于捕获分页请求信息）和Page （用于捕获结果以及元信息）。因此，让我们尝试将findByLastname(…)添加到存储库接口，并重写findAll(…)和findByLastname(…) ，如下所示：

@Transactional(readOnly = true) 
public interface CustomerRepository extends JpaRepository {

  Page findByLastname(String lastname, Pageable pageable); 
}

@Override 
public Page findAll(Pageable pageable) {
  return repository.findAll(pageable);
}

@Override
public Page findByLastname(String lastname, Pageable pageable) {
  return repository.findByLastname(lastname, pageable); 
}

确保根据签名更改来调整测试用例，但是它们应该可以正常运行。这归结为两件事：我们有支持分页的CRUD方法，查询执行机制也知道Pageable参数。在这个阶段，包装类实际上已经过时了，因为客户端可以直接使用我们的存储库接口。我们摆脱了整个实现代码。

摘要

在本博客文章的过程中，我们将为存储库编写的代码量减少为使用3种方法和一行XML的两个接口：

@Transactional(readOnly = true) 
public interface CustomerRepository extends JpaRepository {

    Page findByLastname(String lastname, Pageable pageable); 
}

@Transactional(readOnly = true)
public interface AccountRepository extends JpaRepository {

    List findByCustomer(Customer customer); 
}

我们具有类型安全的CRUD方法，内置的查询执行和分页功能。很酷的事情是，这不仅适用于基于JPA的存储库，而且还适用于非关系数据库。几天后，第一个支持此方法的非关系数据库将是MongoDB，作为Spring Data Document版本的一部分。您将获得与Mongo DB完全相同的功能，我们也在努力支持其他数据库。我们还将在即将发布的博客文章中逐步探讨其他功能（例如，实体审核，自定义数据访问代码的集成）。
