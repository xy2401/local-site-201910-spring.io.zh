Spring 3.1 M1：缓存抽象

工程
Costin Leau
2011年2月23日

Spring Framework 3.1 M1中添加的主要功能之一是用于透明地将缓存应用于Spring应用程序的通用缓存抽象。就像事务支持一样，缓存抽象允许一致使用各种缓存解决方案，而对代码的影响最小。

目的

缓存通常用于通过以更快的方式透明地为频繁访问的数据提供服务（例如，从本地内存而不是从网络提供数据）来提高应用程序性能。你们中的许多人已经知道是否使用缓存了：大多数ORM / JPA框架都提供了专用的缓存功能（也称为2级缓存）。但是，Spring 3.1 M1引入了一种通用的缓存机制，可以将其应用于任何Java类，方法或库：可以将其与现有的缓存基础结构结合使用，以在不具有此类支持的情况下（例如JDBC）向API添加缓存，或者仅向改进了缓慢，耗时且耗费资源的方法的性能。

认识@Cacheable ， @CacheEvict和SpEL

让我们看看缓存任意方法需要做什么：

@Cacheable("books")
public Book findBook(ISBN isbn) {...}

通过使用@Cacheableannotation标记该方法，我们告诉容器findBook方法由缓存条目books 。也就是说，每次调用该方法时，都会使用方法参数（在本例中为isbn参数）作为键执行高速缓存查找。如果找到一个值，将返回该值，并跳过方法执行。但是，如果找不到该键，则该方法将照常执行，并将其结果存储在缓存中，因此，下次调用该方法时，可以返回结果，而无需实际执行（昂贵或慢速）方法。

实际上，并非所有方法都只有一个参数，或者更糟糕的是，这些参数不适合用作缓存键-例如，采用上述方法的一种变体：

public Book findBook(ISBN isbn, boolean checkWarehouse, boolean includeUsed)

在这种情况下，可以使用Spring 3 Spring Expression Language或SpEL挑选合适的参数，浏览对象树

// use property 'rawNumber' on isbn argument as key
@Cacheable(value="book", key="#isbn.rawNumber")
public Book findBook(ISBN isbn, boolean checkWarehouse, boolean includeUsed)

或即时计算密钥，甚至无需编写任何代码即可调用任意方法：

// get the key by calling someType#hash(isbn)
@Cacheable(value="book", key="T(someType).hash(#isbn)")
public Book findBook(ISBN isbn, boolean checkWarehouse, boolean includeUsed)

另外，可以指定何时或是否应该进行缓存：是应该检查缓存还是应该完全忽略缓存，并且该方法可以正常执行。由开发人员决定标准是什么：可以是任何大小，从键大小或类型到一天中的时间或任意方法的结果：SpEL支持所有这些：

// cache only names shorter then 32 chars
@Cacheable(value="book", condition="#name.length < 32")
public Book findBook(String name)

// do not cache on weekends
@Cacheable(value="book", condition="!T(TimeUtils).isWeekend()")
public Book findBook(String name)

缓存抽象还支持通过@CacheEvictannotation逐出缓存条目或整个缓存。要在高速缓存无效时将其逐出（例如，由于已更新高速缓存的数据），可以使用以下方法：

// evict all cache entries
@CacheEvict(value = "books", allEntries=true)
public void loadBooks(InputStream batch)

缓存抽象与其他缓存（例如JPA 2级缓存）相比如何？

通常，只要开发人员注意任何域重叠，两种缓存机制就可以愉快地共存。以JPA 2级缓存为例，可以将它用于通过JPA进行数据访问，同时将Spring缓存用于Web层或远程服务调用。如果可以的话，可以在两种机制之间重用支持的缓存，从而走得更远。
