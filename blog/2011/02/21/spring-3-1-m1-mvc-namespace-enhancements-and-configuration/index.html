Spring 3.1 M1：MVC命名空间增强和@Configuration

工程
罗森·斯托扬切夫（Rossen Stoyanchev）
2011年2月21日

在描述 Spring 3.1 M1 功能的系列文章的第5部分中，我将重点介绍Web应用程序。在上半年中，我将讨论对MVC XML名称空间的增强。然后，我将展示如何使用所有Java配置创建等效的MVC命名空间。最后，我提到了在3.1 M2中可以预期的一些与Servlet 3.0相关的配置更改。

MVC命名空间的改进

Spring MVC 3.0提供了一个自定义的MVC名称空间。命名空间的核心– 元素，使用注释的控制器方法配置了处理请求所需的一切。更重要的是，尽管它设置了一系列默认值，并且还必须处理类型转换，格式设置，验证，请求和响应主体的读写等。

随着时间的流逝，许多用户请求对上述默认配置的各个方面进行更多控制，并且我们在3.1 M1版本中解决了许多此类请求。

格式器注册

我们将从注册Converters和Formatters ，这是通过提供您自己的ConversionService实例如下：


<mvc:annotation-driven conversion-service="..." />

对于自定义格式器，您可以将其子类化FormattingConversionServiceFactoryBean并在代码中注册格式化程序。从3.1 M1开始，您可以注册Formatter和AnnontationFormatterFactory使用setter声明式地输入：


<mvc:annotation-driven conversion-service="conversionService" />

<bean class="org.springframework.format.support.FormattingConversionServiceFactoryBean">
	<property name="formatters">
		<list>
			<bean class="org.example.EmailFormatter"/>
			<bean class="org.example.PhoneAnnotationFormatterFactory"/>
		</list>
	</property>
</bean>

您仍然可以选择注册Converters和Formatters在代码中。这是通过FormatterRegistrar此版本中引入的界面。这是一个例子：


public class FinancialInstrumentFormatterRegistry implements FormatterRegistrar {

	public void registerFormatters(FormatterRegistry registry) {
		// Register custom Converters and Formatters here...
	}

}

这就是你的FormatterRegistrary可以插入：


<bean class="org.springframework.format.support.FormattingConversionServiceFactoryBean">
	<property name="formatterRegistrars">
		<list>
			<bean class="org.example.FinancialInstrumentFormatterRegistrar"/>
		</list>
	</property>
</bean>

所以，什么时候应该使用FormatterRegistrar而不是格式化程序的设置程序？一种FormatterRegistrar当您需要从一个地方注册一个特定格式类别的多个相关转换器和格式器时，此功能很有用。另一种情况是注册一个Formatter在除其自身的通用类型之外的其他字段类型下建立索引， 或者可能注册一个Formatter从一个Printer / Parser对。一个实际的好例子FormatterRegistrar实现是JodaTimeFormatterRegistrar在Spring Framework 中，所以在那里看看。

最后一个选项FormattingConversionServiceFactoryBean是关闭默认功能Formatter完全通过注册registerDefaultFormatters旗。

HttpMessageConverters的注册

从3.1 M1开始，您可以注册HttpMessageConverters通过一个子元素mvc:annotation-driven 。例如：


<mvc:annotation-driven>
	<mvc:message-converters>
		<bean class="com.google.protobuf.spring.http.ProtobufHttpMessageConverter"/>
		<bean class="org.springframework.http.converter.json.MappingJacksonHttpMessageConverter">
			<property name="prefixJson" value="true"/>
		</bean>
	</mvc:message-converters>
</mvc:annotation-driven>

默认情况下，以这种方式提供的消息转换器列表优先于MVC名称空间寄存器中的消息转换器。例如，上面我们添加了一个自定义转换器， ProtobufHttpMessageConverter我们还提供了Spring MVC的实例MappingJacksonHttpMessageConvert根据应用需求定制。

如果您不想默认注册任何消息转换器，请使用register-defaults的属性元件。

注册自定义WebArgumentResolvers

一种WebArgumentResolver ，如果您以前从未见过，则用于解析自定义参数@RequestMapping方法。Spring Mobile项目有一个SitePreferenceWebArgumentResolver 。它解决SitePreference指示用户是要页面的移动版还是完整版的方法参数类型。从Spring 3.1 M1开始，您可以通过MVC命名空间注册自定义参数解析器：


<mvc:annotation-driven>
	<mvc:argument-resolvers>
		<bean class="org.springframework.mobile.device.site.SitePreferenceWebArgumentResolver"/>
	</mvc:argument-resolvers>
</mvc:annotation-driven>

自定义MessageCodesResolver

列表中的最后一个是提供自定义功能MessageCodesResolver ：


<mvc:annotation-driven message-codes-resolver="messageCodesResolver" />

<bean id="messageCodesResolver" class="org.example.CustomMessageCodesResolver" />

MVC名称空间可以完成许多其他事情。上面的列表应该有助于涵盖最常见的用例，以增加灵活性，但是如果您认为我们还错过了其他重要的用例，请告诉我们。

在文章的这一部分中，我将使用一个现有的示例应用程序：您可能会从Keith Donald的先前文章中熟悉许多人所熟悉的mvc-showcase ，并将其XML配置替换为基于Java的配置。这样可以比较之前和之后的代码和配置。

生成的示例应用程序可在spring-3.1-mvc-java-config进行签出。您可以直接在GitHub上浏览源代码，也可以按照README指令在本地获取代码。

我们的第一步是修改web.xml以指向我们基于Java的配置，并指定ApplicationContext用于处理该配置的类型。以下是相关的web.xml片段：


<servlet>
	<servlet-name>appServlet</servlet-name>
	<servlet-class>org.springframework.web.servlet.DispatcherServlet</servlet-class>
	<init-param>
		<param-name>contextClass</param-name>
		<param-value>
			org.springframework.web.context.support.AnnotationConfigWebApplicationContext
		</param-value>
	</init-param>
	<init-param>
		<param-name>contextConfigLocation</param-name>
		<param-value>
			org.springframework.samples.mvc.config.MvcFeatures
			org.springframework.samples.mvc.config.MvcBeans
		</param-value>
	</init-param>
</servlet>

接下来我们将创建MvcFeatures和MvcBeans在〜.config包中。 MvcFeatures贡献@Feature方法和是我们的主要兴趣点：


@FeatureConfiguration
class MvcFeatures {

	@Feature
	public MvcAnnotationDriven annotationDriven(ConversionService conversionService) {
		return new MvcAnnotationDriven().conversionService(conversionService)
			.argumentResolvers(new CustomArgumentResolver());
	}

	// ...

}

上面的代码片段等效于此XML名称空间配置：


<mvc:annotation-driven conversion-service="conversionService">
	<mvc:argument-resolvers>
		<bean class="org.springframework.samples.mvc.data.custom.CustomArgumentResolver"/>
	</mvc:argument-resolvers>
</mvc:annotation-driven>

如你看到的MvcAnnotationDriven使用便捷的链接方法API提供与XML元素相同的选项。另请注意，我们已经声明了ConversionService方法参数。该参数按类型自动接线并注入。它的声明可以在MvcBeans ：


@Configuration
public class MvcBeans {

	@Bean
	public ConversionService conversionService() {
		DefaultFormattingConversionService bean = new DefaultFormattingConversionService();
		bean.addFormatterForFieldAnnotation(new MaskFormatAnnotationFormatterFactory());
		return bean;
	}

	// ...
	
}

注意使用DefaultFormattingConversionService而不是熟悉的FormattingConversionServiceFactoryBean通常在XML配置中使用。前者给我们相同的默认值Converter和Formatter注册为后者，但更适合与Java配置一起使用-它提供了一个简单的构造函数，而不是FactoryBean Spring在使用XML时调用的生命周期初始化方法。

其余部分MvcFeatures声明的等价于 ， 和元素：


@FeatureConfiguration
class MvcFeatures {

	// ...

	@Feature
	public MvcResources resources() {
		return new MvcResources("/resources/**", "/resources/");
	}

	@Feature
	public MvcViewControllers viewController() {
		return new MvcViewControllers("/", "home");
	}

	@Feature
	public ComponentScanSpec componentScan() {
		return new ComponentScanSpec("org.springframework.samples").excludeFilters(
				new AnnotationTypeFilter(Configuration.class), 
				new AnnotationTypeFilter(FeatureConfiguration.class));
	}
}

有两件事值得注意。一个是MvcViewControllers需要使用链式方法调用定义任意数量的视图控制器。第二个是在componentScan()为了防止的方法MvcFeatures和MvcBeans从两次注册-一次由AnnotationConfigWebApplicationContext然后第二次通过组件扫描。

为了完整起见，这是MvcBeans ：


@Configuration
public class MvcBeans {

	// ...

	@Bean
	public InternalResourceViewResolver jspViewResolver() {
		InternalResourceViewResolver bean = new InternalResourceViewResolver();
		bean.setPrefix("/WEB-INF/views/");
		bean.setSuffix(".jsp");
		return bean;
	}

	@Bean
	public MultipartResolver multipartResolver() {
		return new CommonsMultipartResolver();
	}
}

最后一步是删除位于下面的Spring XML配置/WEB-INF/spring 。

摘要

我们到了。我们已经使用所有基于Java的Spring配置引导了Web应用程序。现在@FeatureConfiguration和@Feature已经介绍了，您可以期待看到更多FeatureSpecification实现，以替代自定义XML名称空间。我更喜欢Java配置的最终结果，但这当然并不意味着我现有的应用程序需要切换。这都是关于选择。如果您更喜欢XML的声明性，并且使用IDE在Spring XML配置中对类和方法名称进行代码完成的IDE，那么也可以使用XML名称空间。

正如最近在Spring 3.1简介网络研讨会中听到的那样，在Spring 3.1的里程碑2中，您可以期望看到Servlet 3.0支持，包括不带XML的Web应用程序设置（即没有web.xml），并结合了AnnotationConfigWebApplicationContext和本次演示的环境抽象以及本系列博客的先前文章。

我们希望您喜欢这些功能并发现它们有用。请让我们知道你的想法。
