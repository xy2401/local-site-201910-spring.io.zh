将Cloud Foundry Services与Spring结合使用：第2部分-自动重新配置

工程
拉姆尼瓦斯·拉达德（Ramnivas Laddad）
2011年11月4日

如果您观看了有关Cloud Foundry启动事件的视频，则会看到我们部署了从Spring Web Flow示例下载的Spring Travel应用程序，将MySQL服务绑定到该应用程序，并将该应用程序拖放到STS中的Cloud Foundry服务器上， 而没有在应用程序本身中进行单行更改 。由于将应用程序配置为使用本地数据库，这怎么可能？这就是自动重新配置起作用的时候。

Cloud Foundry努力使您的初始投资降低。除了美元和美分以外，真正的投资还来自开发人员开始投入的时间。当您要开始使用Cloud Foundry时，自动重新配置是一种减少初始投资的机制。在此博客中，我们探讨了它如何与Spring应用程序一起使用（grails应用程序使用相同的底层机制，因此行为相同）。

利用依赖注入进行自动重新配置

您的应用程序由业务逻辑以及与数据库和消息传递等服务的交互组成。在典型的Spring应用程序中，您可以利用依赖项注入（DI）为使用的每个服务创建bean，并将这些bean注入需要访问这些服务的其他bean中。

让我们看一个典型的使用关系数据库的Spring应用程序，该数据库将数据源bean定义为：


<bean class="org.apache.commons.dbcp.BasicDataSource" id="dataSource">
    <property name="driverClassName" value="com.mysql.jdbc.Driver"/>
    <property name="url" value="jdbc:mysql://localhost:3306/inventory-db"/>
    <property name="username" value="myuser"/>
    <property name="password" value="mypass"/>
</bean>

我们可以将诸如用户名和密码之类的属性外部化到一个单独的文件中，但是我们嵌入值以关注于自动重新配置机制。

然后可以将此bean注入其他bean（在这种情况下，注入JPA实体管理器）：


<bean class="org.springframework.orm.jpa.LocalContainerEntityManagerFactoryBean" id="entityManagerFactory">
    <property name="persistenceUnitName" value="persistenceUnit"/>
    <property name="dataSource" ref="dataSource"/>
</bean>

我们可以很容易地观察到：数据库URL指向localhost上的数据库，用户名设置为“ myuser”，密码设置为“ mypass”。当您将此应用程序推送到Cloud Foundry并绑定MySQL或Postgres服务时，该服务的URL不会是jdbc:mysql://localhost:3306/inventory-db用户名或密码就不会那么容易了！因此，如果没有其他机制，此类应用程序将在启动时失败。这是自动重新配置机制起作用的地方。自动重新配置机制利用DI来检查Spring应用程序上下文，查找与服务相对应的bean，并使用基于绑定到应用程序的服务创建的bean替换每个服务。结果是用户应用程序可以在本地部署和Cloud Foundry中运行，而无需进行任何更改。

下表显示了自动重新配置要寻找的Bean类型。

服务类型替换豆类型
MySQL，Postgresjavax.sql.DataSource
雷迪斯org.springframework.data.redis.connection.RedisConnectionFactory
MongoDBorg.springframework.data.document.mongodb.MongoDbFactory
兔子MQorg.springframework.amqp.rabbit.connection.ConnectionFactory

自动重新配置背后的基本机制使用BeanFactoryPostProcessor在创建bean之前检查应用程序上下文，并根据绑定到应用程序的服务将匹配类型的现有bean与等效bean交换。对于关系数据库，它还会重新配置JPA实体管理器工厂或Hibernate会话工厂，以调整所使用的方言。

在部署过程中暂存您的应用程序时，Cloud Foundry将进行两项修改：


  1. 它将向您的应用程序添加一个附加jar，其中包含BeanFactoryPostProcessor和相关资源。请注意，用于自动重新配置的jar也附带了cloudfoundry-runtime的版本。但是，这些类通过阴影机制重定位到其他程序包。这使您的应用程序可以使用不同版本的cloudfoundry-runtime，而不会发生任何冲突。

  2. 它将修改web.xml以更新构成Spring应用程序上下文的文件，以添加BeanFactoryPostProcessor对此。

局限性

服务的自动重新配置仅在以下情况下起作用：


  1. 给定服务类型可能恰好只有一个服务。例如，您可以仅将一个关系数据库服务（MySQL或Postgres）绑定到应用程序。

  2. 可能恰好只有一个匹配类型的bean。例如，您可能只有一个DataSource bean在应用程序上下文中。

如果应用程序没有遵循这些限制，则自动重配置机制将不会发生。在这种情况下，您将需要使用具有或不具有Spring 3.1配置文件支持的下一个博客中描述的名称空间。

自动重配置机制需要典型的Spring应用程序。如果您的应用程序上下文很复杂，则可能无法正常工作。在这种情况下，您可以选择不进行自动重新配置，如下所述。

选择退出自动重新配置

在某些情况下，您可能希望退出自动重配置。例如，您可能有一个内存中的关系数据库，该数据库不应绑定到Cloud Foundry服务。Cloud Foundry提供了几种选择退出自动重新配置机制的方法。


  1. 部署应用程序时，将框架选择为“ JavaWeb”。这会将您的应用程序视为不是Spring应用程序。这是一个很难的选择，因为您的应用程序将保持不变（不会将jar添加到您的应用程序中，而web.xml将保持不变）。这也意味着本博客系列后面讨论的配置文件功能将不适用于此类应用程序。

  2. 使用任何创建代表服务的bean的元素。目前包括 ， ， ， 和 。如果应用程序直接基于这些命名空间元素（例如， CloudMongoDbFactoryBean ），则选择退出将生效。Cloud Foundry使用此机制选择退出，因为应用程序要么希望具有自动重新配置行为，要么干脆完全控制服务创建。我们没有看到自动重新配置某些服务并手动执行其他服务的价值。

结论

Cloud Foundry中的自动重新配置功能是入门的绝佳方法。随着应用程序的成熟或需要绑定到多个服务，您可能需要更好地控制服务连接对象。这是支持命名空间。在本系列的下一个博客中，Thomas Risberg将解释如何使用它。托马斯拿走。
