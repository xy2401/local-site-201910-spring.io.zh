调整Spring数据存储库

工程
奥利弗·德罗博姆
2011年7月27日

自从我们发布Spring Data JPA 1.0 GA以来只有几天，这是Spring Data项目发布的第一个主要版本，在我们的Spring Data Commons模块中实现了存储库抽象的实现。存储库抽象包括三个主要部分：定义存储库接口，公开CRUD方法和添加查询方法。在Spring Data JPA的第一篇博客文章中详细讨论了添加查询方法。但是，定义存储库接口和公开CRUD方法在早期的博客文章中引发了很多问题。这就是为什么现在将仔细研究它们，讨论用户可用的选项的原因。

声明一个Spring Data Repository接口

可以通过两种方式完成存储库接口的声明：使用批注或扩展标记接口：


@RepositoryDefinition(domainClass=Customer.class, idClass=Long.class)
public interface CustomerRepository {
  // declare query methods
}

public interface AccountRepository extends Repository<Account, Long> {
  // declare query methods
}

选择哪种风格几乎取决于品味。希望尽可能减少代码依赖的纯粹主义者可能会坚持使用基于注释的方法，但基于继承的方法实际上与其余编程模型更加一致。无论选择哪种方法，Spring Data基础架构都将了解如何实现这些接口中定义的存储库方法。

CRUD方法

您可能想做的下一步是公开CRUD方法。为此，我们提供了两个单独的接口： CrudRepository和PagingAndSortingRepository后者扩展了前者。顾名思义CrudRepository公开了基本的CRUD方法，例如T save(T entity) ， T findOne(ID id)和void delete(T entity) 。 PagingAndSortingRepository公开了可以进行分页的方法，例如Page findAll(Pageable pageable) 。要将这些方法公开给客户，只需扩展这些接口，如下所示：

public interface CustomerRepository extends PagingAndSortingRepository<Customer, Long> {
  // declare query methods here
}

微调存储库接口

扩展前面显示的界面对于快速入门来说是很整洁的，但是有一个很大的缺点：您向不受您控制的客户端公开了一组预定义的操作，直到现在几乎全部还是没有。似乎没有办法只公开读取操作，而完全隐藏状态更改操作。

为避免此问题，我们允许您为存储库定义自定义基本接口，在其中可以有选择地公开喜欢的CRUD方法：


@NoRepositoryBean
public interface ReadOnlyRepository<T, ID extends Serializable> extends Repository<T, ID> {
  T findOne(ID id);
  Iterable<T> findAll();
  Iterable<T> findAll(Sort sort);
  Page<T> findAll(Pageable pageable);
}

这里需要考虑一些关键的事情：首先，中间接口也必须是一个Spring Data Repository接口，这意味着它要么必须带有注释。 @RepositoryDefinition或扩展Repository如此处所示。您还应该使用以下注释接口@NoRepositoryBean防止带有正式类型参数的接口T和ID由Spring Data类路径扫描获取。最后（可能也是最关键的部分）是方法签名必须与CrudRepository要么PagingAndSortingRepository 。您的实际存储库界面将如下所示：

public interface CustomerRepository extends ReadOnlyRepository<Customer, Long> {
  // declare query methods here
}

当支持为该接口创建的代理的存储库实现实现您的自定义中间接口中声明的方法时，我们可以将对这些调用的路由路由到实现中。您甚至可以在共享基本接口中声明共享查询方法：

@NoRepositoryBean
public interface NamedRepository<T, ID extends Serializable> extends ReadOnlyRepository<T, ID> {
  List<T> findByName(String name);
}

public interface CustomerRepository extends NamedRepository<Customer, Long> { … }

public interface PersonRepository extends NamedRepository<Person, Long> { … }

假设Customer和Person分享name属性（不一定通过继承）。因为查询将自动从方法名称中派生。甚至可以通过简单地声明JPA命名查询来手动定义对两个具体存储库执行的查询Customer.findByName和Person.findByName 。使用这种方法，您可以轻松地为您的应用程序场景量身定制基础接口。

技术特定的接口

这就给我们留下了何时使用技术特定的基本接口（例如JpaRepository 。第一个原因可能只是快速入门的一个原因： List基于读取的方法（例如List findAll() ）哪些开发人员可能更习惯使用JPA以及从其超级接口派生的所有CRUD方法。我们还会在这里公开一些JPA特定的方法，因为某些客户端可能需要调用它们。实际上，我们建议不要向客户公开技术细节，但有时这只是对理论的实用主义。所有这些专业都可以通过自定义基本接口公开（重新声明read方法，返回List ，公开JPA特定的方法），因此您几乎可以选择在特定情况下最省力的方法。

因此，如果您想开始使用Spring Data的存储库抽象，请随时访问我们在Github上针对JPA和Mongo的示例项目。
