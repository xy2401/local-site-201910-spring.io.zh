Spring 3.1 M2：Spring MVC增强功能

工程
罗森·斯托扬切夫（Rossen Stoyanchev）
2011年6月13日

这篇文章重点介绍Spring 3.1 M2中Spring MVC的新增功能。这里是主题：

  • MVC名称空间的基于代码的等效项。
  • 可自定义的@MVC处理。
  • 编程模型改进。

简要提醒一下，此处讨论的功能正在Greenhouse项目中运行。

Spring MVC的基于代码的配置

正如Chris在上周五的博客文章中所指出的那样，XML名称空间大大减少了配置，但同时也降低了透明度，有时还降低了灵活性。MVC命名空间适用于这种情况，它支持多种自定义设置，但不支持所有可用的自定义设置。这意味着您可以使用它，也可以保留它。我们相信基于代码的配置可以解决此问题，并提供从简单到高级的路径。

让我们从这个简单，熟悉的代码段开始：


<mvc:annotation-driven />

尽管使用带注释的控制器不是必需的， 做很多有用的事情-它检测到JSR-303（Bean验证）实现的存在并配置数据绑定，如果Jackson JSON库可用，则添加一个JSON消息转换器，以及一些其他可以保存的东西相当多的配置。

现在，让我们将其与基于代码的配置相匹配：


@Configuration
@EnableWebMvc
public class WebConfig {
}

这里@EnableWebMvc进口@Configuration与之相称的课程 。就如此容易。

下一步是在也许提供一个FormattingConversionService ，或者添加可能配置消息转换器的子元素，或者使用其他MVC名称空间元素（例如 ， 等

让我们看看如何在基于代码的配置中完成所有这些工作：


@Configuration
@EnableWebMvc
public class WebConfig extends WebMvcConfigurerAdapter {

    @Override
    public void addFormatters(FormatterRegistry registry) {
        // register converters and formatters...
    }

    @Override
    public void configureMessageConverters(List<HttpMessageConverter<?>> converters) {
        // add message converters...
    }

    @Override
    public void configureInterceptors(InterceptorConfigurer configurer) {
        configurer.addInterceptor(new AccountExposingHandlerInterceptor());
    }

    @Override
    public void configureResourceHandling(ResourceConfigurer configurer) {
        configurer.addPathMapping("/resources/**").addResourceLocation("/resources/");
    }

    // more @Override methods ... 

}

请注意，这次我们也将其分类WebMvcConfigurerAdapter ，这是一个方便的基类，只不过是空的方法实现WebMvcConfigurer接口，它定义用于自定义Spring MVC配置的配置回调。 @Configuration检测到实现此接口的类，并有机会应用自定义。就是这样。

上面提供的与MVC命名空间的奇偶校验可以说是更加透明的 。您可以使用熟悉的Java IDE快捷方式来探索WebMvcConfigurer界面和@Bean进口方法@Configuration类。

那更高级的自定义呢？好吧，这是我们可以使用的MVC名称空间。对于基于代码的配置，在下一个（RC1）版本中，你就可以把上面的例子是 ，滴导入的配置（即删除@EnableWebMvc ）并切换到包含以下内容的基类@Bean您可以覆盖的方法。这意味着您可以使用基于Spring MVC代码的配置，知道可以进行任何级别的自定义-通过简单的回调机制或通过直接从提供实际配置的类扩展而来。

这是看一下web configuration在温室。

可自定义的@MVC处理

@MVC编程模型非常成功，可以实现灵活的控制器方法签名。然而，您中的许多人都要求底层的@MVC支持类更可定制。作为回应，我们推出了一组新的@MVC支持类，它们为您提供了更多功能并为我们提供了更好的基础。

在注释之前，Controller是处理端点。通过注释，单个控制器方法成为具有其自己的请求映射的端点。遵循这个逻辑HandlerMapping不应仅限于选择控制器，而应选择控制器方法 。因此，我们添加了一个HandlerMethod抽象和AbstractHandlerMethodMapping用于可以选择HandlerMethod 。

这些是围绕的新@MVC支持类。 HandlerMethod抽象：

  • RequestMappingHandlerMapping
  • RequestMappingHandlerAdapter
  • ExceptionHandlerExceptionResolver

结果，我们现在在处理程序映射中有了一个选择点，处理程序适配器确切地知道选择了哪个处理程序方法，其他组件也是如此。例如，你们中的许多人都要求在处理程序方法的调用周围进行拦截，该方法现已关闭。另一个不那么明显的结果是，只要映射在某些其他方式（例如HTTP方法）上有所不同，就可以在不同的控制器之间自由映射同一URL 。

除了请求映射之外，执行控制器方法还需要解析方法参数（ @RequestParameter ， @ModelAttribute ，等等）并处理返回值（ @ResponseBody ， @ModelAttribute等）。新的@MVC支持类公开了一种可插拔机制，其中HandlerMethodArgumentResolver和HandlerMethodReturnValueHandler可以插入以解析每个方法参数并处理每个返回值。您对此具有完全控制权-您可以设计自己的参数类型和返回值类型，也可以自定义内置参数的处理。有关详细信息，请参见后续文章。

要尝试新的@MVC支持类，只需升级到Spring 3.1 M2。MVC名称空间和@EnableWebMvc配置它们。或者，如果您有自己的配置，只需交换以下内容：

  • DefaultAnnotationHandlerMapping ->RequestMappingHandlerMapping
  • AnnotationMethodHandlerAdapter ->RequestMappingHandlerAdapter
  • AnnotationMethodExceptionResolver ->ExceptionHandlerExceptionResolver

关于兼容性的说明 ：现有支持类将继续可用。但是，我们建议您继续进行切换。例如，下一部分中所有编程模型的改进都仅以这种方式提供。新的类在大多数情况下应该是可替代的，但是有两个值得注意的区别。一个，您不能合并任何现有的AbstractUrlHandlerMapping类型（例如SimpleUrlHandlerMapping ）与新的处理程序适配器一起使用，该适配器期望HandlerMethod而不是处理程序。二，当两个时，您不能依赖方法名称@RequestMapping方法与请求相等匹配。

编程模型改进

本节列出了新@MVC支持类中引入的编程模型改进。

1。宣告@PathVariable现在，参数会自动添加到模型中。例如：


@RequestMapping("/develop/apps/edit/{slug}")
public String editForm(@PathVariable String slug, Model model) {
	model.addAttribute("slug", slug);
    // ...
}

替换为：


@RequestMapping("/develop/apps/edit/{slug}")
public String editForm(@PathVariable String slug, Model model) {
    // model contains "slug" variable
}

2。重定向字符串支持URI模板，该模板使用模型中的变量扩展（包括已声明的@PathVariables ）。例如：


@RequestMapping(
    value="/groups/{group}/events/{year}/{month}/{slug}/rooms", 
    method=RequestMethod.POST) 
public String createRoom(
    @PathVariable String group, @PathVariable Integer year, 
    @PathVariable Integer month, @PathVariable String slug) {
    // ...	
    return "redirect:/groups/" + group + "/events/" + year + "/" + month + "/" + slug;
}

替换为：


@RequestMapping(
    value="/groups/{group}/events/{year}/{month}/{slug}/rooms", 
    method=RequestMethod.POST) 
public String createRoom(
    @PathVariable String group, @PathVariable Integer year, 
    @PathVariable Integer month, @PathVariable String slug) {
    // ...	
    return "redirect:/groups/{group}/events/{year}/{month}/{slug}";
}

3。数据绑定中支持URI模板变量。例如：


@RequestMapping("/people/{firstName}/{lastName}/SSN")
public String find(Person person, 
                   @PathVariable String firstName, 
                   @PathVariable String lastName) {
    person.setFirstName(firstName);
    person.setLastName(lastName);
    // ...
}

替换为：


@RequestMapping("/people/{firstName}/{lastName}/SSN")
public String search(Person person) {
    // person.getFirstName() and person.getLastName() are populated
    // ...
}

4。可通过以下方式指定消耗性和生产性媒体类型@RequestMapping 。例如：


@RequestMapping(value="/pets", headers="Content-Type=application/json")
public void addPet(@RequestBody Pet pet, Model model) {
    // ...
}

替换为：


@RequestMapping(value="/pets", consumes="application/json")
public void addPet(@RequestBody Pet pet, Model model) {
    // ...
}

除了更短之外，如果URL匹配但输入媒体类型不匹配，则上述方法还会返回NOT_ACCEPTABLE（406）。

5，对于可生产的媒体类型，这是：


@Controller
@RequestMapping(value = "/pets/{petId}", headers="Accept=application/json")
@ResponseBody
public Pet getPet(@PathVariable String petId, Model model) {    
    // ...
}

替换为：


@Controller
@RequestMapping(value = "/pets/{petId}", produces="application/json")
@ResponseBody
public Pet getPet(@PathVariable String petId, Model model) {    
    // ...
}

如果URL匹配但可接受的媒体类型不匹配，则上面的代码会返回NOT_SUPPORTED_MEDIA_TYPE（415）。
