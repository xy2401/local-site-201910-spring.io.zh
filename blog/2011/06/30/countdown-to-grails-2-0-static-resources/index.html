倒计时Grails 2.0：静态资源

工程
彼得·莱德布鲁克
2011年6月30日

Web应用程序通常严重依赖我们所谓的静态资源，例如Javascript，CSS和图像文件。在Grails应用程序中，它们被放入项目的Web应用程序目录中，然后从HTML进行引用。例如，

将创建指向文件web-app / css / main.css的链接 。一切都很简单。您甚至可能认为当前的支持已足以满足任何人的需求。您还要做什么？

那是个很好的观点。答案取决于应用程序的复杂性，但让我们从上面的示例CSS链接开始。为什么我们必须输入 ？仅通过查看扩展名，我们就知道该资源是一个CSS文件。我们也知道应该使用上述元素将CSS文件链接到HTML页面中。因此，基本上，我们为Grails应该能够处理的事情做了很多打字工作。

“可以，”我听到您说，“但是我们可以为CSS链接添加标签。”的确可以，Grails 2将包含一个标记，用于智能地选择资源的适当链接类型。但是现在考虑生产：将CSS和Javascript文件捆绑在一起不是一个好主意吗？压缩它们呢？（ 顺便说一句 ，雅虎提供了一个名为ySlow的工具，它将为您提供有关此类优化的提示）。那么我们如何实现这种优化呢？您可能会临时执行此操作，但是这样做会限制您的选择，因为您无法定义资源之间的关系。我们真正想要的是一种在中心位置声明资源并指示应处理哪些资源并将其捆绑在一起的方法。

如果我还没有说服您有更好的方法来管理静态资源，那么请考虑使用Grails插件。其中许多提供自己的静态资源。实际上，某些插件彼此提供相同的资源。例如，较旧版本的YUI，Bubbling和Grails UI插件都包含YUI库。我们真的需要整个Javascript库的多个副本吗？当然不是。部分问题可以通过插件间的依赖关系来解决，但不能单独解决，因为插件无法说出其某些资源依赖于另一个插件中的特定资源。最后，由开发人员来确保所有适当的资源（包括传递依赖项）都包含在每个页面中。这可能涉及到一些试验和错误，以确保所有需要的资源链接到页面，并以正确的顺序。

幸运的是，我们可以做得更好。输入资源插件 。

声明性资源

通过仅安装一个插件，您可以开始在可重用模块中声明您的静态资源（JavaScript，CSS等）。然后，您可以定义模块之间的依赖关系，以便Grails准确知道模块所需的资源，包括传递依赖关系中指定的资源。另外一个好处是，模块可以确保始终在页面中以正确的顺序声明资源，并且没有重复项。这样，模块可以减轻静态资源管理的大部分麻烦。

声明模块

您可以在应用程序和插件中声明资源模块。有两种方法可以执行此操作，但是最常见的方法是向项目添加一个或多个专用工件。对于应用程序，可能是grails-app / conf / ApplicationResources.groovy 。作为另一个示例，YUI插件具有grails-app / conf / YuiPluginResources.groovy 。这些文件的基本结构如下所示：

modules = {
    core {
        resource url: 'js/core.js', disposition: 'head'
        resource url: 'js/ui.js'
        resource url: 'css/main.css'
        resource url: 'css/branding.css'
        resource url: 'css/print.css', attrs: [media: 'print']
    }

    utils {
        dependsOn 'jquery'
        resource url: 'js/utils.js' 
    }

    forms {
        dependsOn 'core', 'utils'

        resource url: 'css/forms.css'
        resource url: 'js/forms.js'
    }
}

“ core”，“ utils”和“ forms”是我们应用程序模块的名称，您可能会猜到，“ forms”模块取决于“ core”和“ utils”。您还可以看到“ utils”模块取决于“ jquery”，这是jQuery插件提供的模块。最好以图表形式显示此结构：

如果我们对上述模块定义进行更深入的研究，可以看到使用“资源”和URL在模块内声明了各个资源。此URL是资源相对于项目中Web应用程序目录的位置。如果您愿意，还可以添加额外的属性，以实现对资源的细粒度控制，尤其是通过“处置”设置。

有两种标准配置：“头”，表示资源位于内部 element, and “defer”, which typically means the end of the body (although you have control over the exact placement as you’ll see shortly). By default, CSS files have a disposition of “head” while Javascript files use “defer”, but these defaults can be overridden on a per-resource basis.

现在，我们有了定义简单或复杂的资源模块网络的基础，这些网络跨越了应用程序和插件的边界。从用户的角度来看，甚至更好，这取决于插件，以确保其模块包含适当的资源和依赖项。

因此，Grails现在可能知道您的应用程序及其插件提供了哪些静态资源，但是仍然不知道哪些页面需要哪些资源。为此，您必须对GSP进行一些修改。

在页面中包含资源

如您所知，您以前必须在布局和视图中显式声明所有CSS和Javascript链接。那么资源插件如何改变呢？您无需声明指向各个资源的链接，而是声明页面需要哪些模块，这使内容更加简洁块。另外，您可以指定资源链接在页面中的位置。这是使用资源的非常简单的布局的示例：

如您所知， 标签告诉Grails在页面上包含哪些静态资源的同时标签指定链接应该去。你应该有两个标签：一入而另一个在 。配置为“ head”的任何资源都将转到第一个资源的位置，而配置为“ defer”的资源将插入第二个位置。

就是这样！大多数工作将用于确保模块定义正确。

到目前为止，我所介绍的内容对于CSS，Javascript和Favicon文件确实非常有效，因为Resources插件知道要为它们生成哪些链接以及将这些链接放在何处。但是内联图像和脚本会发生什么？

临时资源

Resources插件将内联图像和脚本称为“临时资源”。这些通常不会在模块中声明，而是在页面中遇到它们时进行简单处理。标准的嵌入式图片链接如下所示：

那么，如何使Resources插件知道图像文件呢？我们不！从Grails 2.0开始， 标记（用作上述方法）会自动将资源注册到插件（如果已安装）。这意味着映射器执行的所有魔术操作（我将稍后讨论）将应用于给定资源。而且当然， 可以用于任何类型的资源，而不仅仅是图像。

内联脚本有些不同，因为它们不是到外部文件的链接。但是他们仍然可以从Resources插件中受益。默认情况下，所有使用标签的行为将始终如一。但是如果您更换与标签，内联脚本将被移至第二个位置 （它们的默认配置为“延迟”）。您还可以声明一个明确的处置，以便脚本进入 。最重要的是，内联脚本保留了它们在页面中声明的顺序。

还有另一种特别资源需要提及：CSS。样式表是指定背景和其他类型的图像的常用方法，但是CSS文件不会作为GSP文件处理。资源插件知道这些链接吗？幸运的是。即使Resources插件修改了其管理的所有资源的URL路径，这对您来说都没有关系，因为该插件还会自动更新CSS文件中的链接。一切都会正常！

魔术贴图

单独使用Resources插件可使静态资源的管理比以前简单得多。但这只是故事的一部分。由于该插件知道所有资源并控制它们的链接生成，因此它可以执行额外的处理以添加有趣的行为。此处理由可扩展的映射器管道完成：

不要将此图作为绝对正确的参考：作为用户，确切的管道及其工作方式并不重要。关键是，您可以添加自己的映射器实现，也可以简单地安装提供一些映射器的插件。结果是一些非常强大的功能几乎可以实现零努力。

例如，假设您已经安装了Resources插件，定义了一些模块，并且将GSP视图和布局设置为使用适当的标记。只需安装Cached Resources和Zipped Resources插件，您将立即开始满足ySlow的一些建议，例如具有长期Expires标头和所有静态资源的gzip压缩（尽管您可以针对特定文件类型禁用gzip压缩例如图像，这些图像已经被压缩了）。这不应该那么容易，对吧？但它是。

结论

Resources插件已经存在了一段时间，并且已经在一些生产站点中使用。Grails网站上甚至在使用它。这也就不足为奇了：它将静态资源的改进管理与映射器管道结合在一起，该映射器管道具有易于使用但功能强大的功能。经验丰富的Web开发人员将立即注意到差异，并且您可以期望将来会有越来越多的插件提供资源支持。它甚至有自己的用户指南 。

由于该插件带来的优势，Grails 2使其成为了新Grails项目的默认插件，并将其集成到了一些核心标签中，例如 。但是，即使您还不能使用Grails 2，您仍然可以在较早版本的Grails中使用它，并且可以毫不费力地受益于出色的资源管理。它的主要设计原则之一是可以将它安装在任何Grails应用程序中而不会破坏任何东西。

无论使用哪种Grails版本，安装此插件都可以改善您作为Web开发人员的生活。那不是什么可闻的东西。
