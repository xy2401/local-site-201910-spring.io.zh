Green Beans ：Maven和Spring入门

工程
乔什·朗（Josh Long）
2011年1月17日

Apache Maven是一种流行的开源工具，它为项目构建管理提供了一种基于配置的约定方法。的确，Eclipse社区调查显示Maven的采用率从2009年的8％提高到2010 年的28％ ，突显了其在各种项目设置中的有用性。即使无需使用Maven即可使用Spring，也有很多理由向Spring开发人员推荐使用它。在本文中，我将向您展示如何开始使用Maven，以及如何将其成功用于Spring库，存储库和诸如SpringSource Tool Suite和Spring Roo之类的工具 。

Maven处理项目构建。如果您的项目遵循Maven的约定，则Maven可以相对轻松地提供很多功能和完善性。它是声明性的；您描述的是您要完成的工作，而不是您要如何完成。如果您使用诸如Make或Ant之类的古老构建工具，则此方法似乎会有所不同。

您可以在Maven项目配置中声明项目的依赖项。然后，代表您解决这些依赖关系并为您下载。这类似于许多不同操作系统中的打包系统。假设您使用的是OS X fink要么ports命令行工具。要更新操作系统的功能，用户可以选择一个程序包（例如，最新的安全补丁程序或该程序包的新版本）。 glib库）以安装在管理客户端中，然后指示客户端从众所周知的服务器（称为软件包存储库）下载并安装。下载软件包时，软件包管理器将查询软件包的清单，该清单会枚举该软件包所依赖的所有库（在其他软件包中找到）-其传递依赖项。那些也被下载。

有几种安装Maven的方法（如果尚未安装）。从Apache网站下载它。选择一个更新的版本。当前，许多人正在使用Maven 2.21或最近发布的Maven 3。下载您要使用的版本，然后将其解压缩到您选择的目录中。或者，有几种操作系统在软件包系统中提供Maven 2构建（不久之后，将提供Maven 3构建）。例如，在Ubuntu上，您可以运行sudo apt-get install maven2 。如果您使用的是SpringSource Tool Suite（可在此处免费下载 ），则不必担心，Maven已下载并包含在您的STS安装文件夹中。无论您如何在系统上获取Maven二进制文件，都应确保该二进制文件位于操作系统的搜索路径上。通常，这只是添加Maven安装文件的问题bin文件夹到您操作系统的PATH变量。为Maven安装本身创建一个系统变量（称为）也是一个好习惯MAVEN_HOME 。在Unix（包括OS X）或Linux计算机上，此设置看起来几乎相同。在我的机器（Ubuntu Linux机器）上，它看起来像这样：



export MAVEN_HOME=/home/jlong/bin/springsource/maven-2.2.1.RELEASE
export PATH=$PATH:$MAVEN_HOME/bin

要测试它，请打开一个新的shell并发出以下命令：



mvn –version

Maven入门


Maven项目采用标准目录结构，该目录结构至少看起来像这样：

./pom.xml
./src
./src/main
./src/main/java
./src/main/resources
./src/test
./src/test/java
./src/test/resources

目录结构的根是Maven期望的XML文件（始终称为pom.xml）。的pom.xml （POM是“项目对象模型”的缩写）描述了无法自动推断出的特定于您项目的事物，例如依赖性，项目名称等。

目录 说明目录的内容（相对于项目根）
src / main / java 包含项目的Java源代码
src / main / resources 包含项目的任何相对于类路径的资源（例如Spring应用程序上下文.xml文件）
src /测试/ java 包含测试类的Java源代码。该目录不会包含在最终版本中。本文中的所有测试将被编译，所有测试将运行。如果测试失败，它将中止构建。
src / test / resources 该目录不会包含在最终的Java构建中。该文件夹包含测试代码的所有与类路径相关的资源（例如Spring应用程序上下文.xml文件）。

让我们构建一个使用Spring Framework的简单Maven项目。在硬盘上设置上述目录结构。如果您使用的是类似Unix的操作系统，则可以使用以下命令来设置项目目录中的目录：



mkdir spring-example1;
cd spring-example1;
mkdir -p src/{test,main}/{java,resources}

创建一个名为的文本文件pom.xml 。在文件中输入以下内容：

<?xml version="1.0" encoding="UTF-8"?> 
<project xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 http://maven.apache.org/xsd/maven-4.0.0.xsd" xmlns="http://maven.apache.org/POM/4.0.0" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"> 
  <modelVersion>4.0.0</modelVersion> 

  <groupId>org.springsource.greenbeans.maven</groupId> 
  <artifactId>example1</artifactId> 
  <version>1.0-SNAPSHOT</version> 
 
  <name>Our Simple Project</name> 
   
</project>

现在让我们忽略该文件的内容。我们需要知道的是，该文件的内容足以唯一地标识Maven项目，并让Maven进行工作。让我们直接跳到收益：在项目的根文件夹中-与相同的文件夹中pom.xml -在命令行上运行以下命令：

mvn install

第一次运行可能需要一段时间。您会在屏幕上看到许多下载进度数字快速滚动。这些进度数字表明正在下载Maven的依赖项和插件。Maven没有构建“目标”的概念。相反，Maven拥有一组生命周期阶段的概念。这里， install是一个阶段。这些阶段按顺序调用，一个接一个。对于每个阶段，都有在每个阶段注册和调用的预包装插件。Maven已经配置好了默认插件，因此您通常不需要太多处理，但是了解正在发生的事情会有所帮助。用户可以覆盖或添加插件，并将其配置为在所需的任何阶段运行。当一个阶段被调用时，它之前的所有阶段都被首先调用。这是标准阶段，并解释每个阶段的意图。请记住，不同的项目可能在这些阶段配置不同的插件。

相 描述
验证 对项目本身进行健全性检查
编译 编译源代码
测试 运行已编译的测试类（委托给特定的单元测试插件运行程序，例如jUnit或TestNG）
包 从编译后的代码和类路径资源生成工件，并将其存储在target文件夹，位于项目的根目录
整合测试 处理打包的工件并将其部署到集成测试环境中
校验 运行检查以确认软件包有效
安装 将打包的工件安装在本地存储库中。您的本地存储库是一个文件夹，所有已下载的依赖项都保留在其中并缓存。尝试解决缓存中已经存在的依赖关系的后续构建将不会重新下载该依赖关系，而是使用本地存储库中的依赖关系。通常，它位于您的主目录下.m2 ，位于一个名为repository.因此，在我的系统上，这将是/home/jlong/.m2/repository
部署 将最终工件复制到另一个环境。通常，使用共享服务器，以便不同的团队可以共享共享的依赖项。

因此，Maven构建遵循以下各个阶段，并且每个项目最终都会通过运行这些阶段来生成工件。您可以通过指定一个 pom.xml文件中的元素。在上面的示例中，我们省略了packaging元素，将其保留为类型的包装jar 。其他包装类型包括war和ear 。

考虑到所有这些，我们从上面进行的简单调用应该看起来非常强大！

mvn install指示Maven除其他事项外，下载所有插件和依赖项（如果尚未下载和缓存），编译源代码，运行单元测试，构建.jar，将生成的.jar安装到相对于项目的目标文件夹中根目录，然后将生成的.jar安装到本地存储库中~/.m2/repository 。由于阶段是标准的，目录是标准的，并且默认的已配置插件是标准的，因此您应该能够执行任何Maven项目并运行mvn install并可以预测地获得经过测试的可用二进制文件。

有时，您希望Maven丢弃“目标”文件夹中的所有内容并开始清理。为此，可以在“安装”命令之前使用“清洁”命令。这将删除所有预构建的二进制文件，然后安装工件。



mvn clean install

证明你的身份！

所有Maven项目都可以通过结合使用它们的方式进行唯一标识artifactId ， groupId和version元素。此信息提供了项目的“坐标”。我们在中指定的信息pom.xml文件告诉Maven我们项目的坐标是什么。通过指定具有唯一坐标的依赖项，Maven可以在编译和执行期间自动将其他工件（依赖项）添加到项目的类路径中。这是座标的三个最重要部分。

坐标元素 描述
groupId 一个groupId可以是任何东西。可以将其视为用于构建的Java包。通常，它是您组织所在领域以及项目的组合。例如，Spring Integration项目有一个groupId的： org.springframework.integration
artifactIdartifactId是该特定工件在groupId.一种groupId对于一起描述整个系统的许多不同工件而言，可能是很常见的。ArtifactIds可以按用途，模块名称或任何其他区分符进行命名。Spring Integration File支持包括以下内容artifactId ： spring-integration-file
版 这是项目的版本。Maven版本可以是固定数字或SNAPSHOT。项目的最终GA版本可能具有1.0或3.5等版本。但是，在开发中，这些相同的项目被称为最终版本的快照。Maven可以使用SNAPSHOT后缀来适应此要求，该后缀告诉Maven“获取最新版本”。一个示例版本可能是1.0-SNAPSHOT，它翻译为“ 1.0的最新版本”。

我们的项目独特地标识了自己：


<groupId>org.springsource.greenbeans.maven</groupId>:<artifactId>example1</artifactId>:<version>1.0-SNAPSHOT</version>

一旦你installed在这个项目中，任何其他项目都可以通过将其作为依赖项添加到您的Maven pom.xml文件中来依赖它。这也适用于其他项目。让我们修改pom.xml以使其依赖于Spring Framework。您的新pom.xml文件将如下所示：



<?xml version="1.0" encoding="UTF-8"?> 
<project xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 http://maven.apache.org/xsd/maven-4.0.0.xsd" xmlns="http://maven.apache.org/POM/4.0.0" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"> 
  <modelVersion>4.0.0</modelVersion> 

  <groupId>org.springsource.greenbeans.maven</groupId> 
  <artifactId>example2</artifactId> 
  <version>1.0-SNAPSHOT</version> 
 
  <name>Our Simple Project</name> 

  <dependencies> 
       <dependency> 
            <groupId>org.springframework</groupId> 
            <artifactId>spring-context</artifactId> 
            <version>3.0.5.RELEASE</version>
       </dependency>
  </dependencies>
</project>

重新运行mvn install ，并且您应该开始看到Spring Framework库及其依赖的库下降了。检查输出，您会看到Maven正在从公共存储库下载依赖项， http://repo1.maven.org 。有许多公共存储库，默认情况下，Maven会参考一些。这些存储库只是带有目录的HTTP服务器。通常，您可以在浏览器中检查它们以搜索所需的特定依赖项。例如，如果我想查看所有可用的Spring Framework依赖关系，则可以浏览至URL http://repo1.maven.org/maven2/org/springframework/ 。要查看所有Spring Integration依赖项，请浏览至http://repo1.maven.org/maven2/org/springframework/integration/ 。要查看所有Spring Batch依赖项，请浏览至http://repo1.maven.org/maven2/org/springframework/batch/ ，等等。SpringSource还维护着几个Maven信息库，其中包含我们最新的依赖项，例如http://maven.springframework.org和spring-roo-repository.springsource.org

这些存储库都遵循通用布局，并且都包含有关存储在标准位置的每个工件的元数据。这种存储库结构已成为事实上的标准，现在对于其他使用它们的工具来说，它已经很常见了。除了Maven之外，还有其他更专门的构建工具，它们现在消耗和产生与Maven相同的元数据。

Maven的默认值足以满足大多数构建的需要，但有时您会发现有必要在构建阶段中定制Maven的行为，或者有时添加可以独立于任何阶段调用的功能。在Maven中，您可以使用插件来更改这些内容。我经常更改的一件事是编译器的语言支持。我通常要使用Java 5或更高版本，其中Maven 2的默认行为是1.4。我们将覆盖默认的Maven编译器行为，并配置特定的语言兼容性。下面显示的是更新的Maven项目文件。



<?xml version="1.0" encoding="UTF-8"?> 
<project xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 http://maven.apache.org/xsd/maven-4.0.0.xsd" xmlns="http://maven.apache.org/POM/4.0.0" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"> 
  <modelVersion>4.0.0</modelVersion> 

  <groupId>org.springsource.greenbeans.maven</groupId> 
  <artifactId>example2</artifactId> 
  <version>1.0-SNAPSHOT</version> 
 
  <name>Our Simple Project</name> 

  <dependencies> 
       <dependency> 
            <groupId>org.springframework</groupId> 
            <artifactId>spring-context</artifactId>
            <version>3.0.5.RELEASE</version>
       </dependency>
  </dependencies>

  <build>

    <plugins>

      <plugin>
        <groupId>org.apache.maven.plugins</groupId>
        <artifactId>maven-compiler-plugin</artifactId>
        <version>2.3.2</version>

        <configuration>
          <source>1.5</source>
          <target>1.5</target>
        </configuration>

      </plugin>
    </plugins>
  </build>
</project>

有许多可用的插件，许多其他插件仍由各种社区项目贡献。插件的一种常见用例是添加aspectj-compiler编译阶段的插件，以确保AspectJ方面已正确编译。另一个常见的要求是为基于WSDL的Web服务合同生成Java接口。通常，插件本身是有用的，并且不依赖于任何特定的构建阶段。Maven使您可以调用单个插件，而无需调用整个构建。插件通常提供它们提供的几个特定命令。要调用插件，请使用以下形式：

mvn [plugin-name]:[command-name]

插件名称可以是别名，也可以是完全限定的插件名称。来自Maven项目存储库的插件具有别名，并且易于使用。一个非常有用的插件是Maven依赖插件。依赖插件支持许多命令。人们的共同要求是获取附加到给定项目的所有依赖项（透明或其他）的清单。Maven依赖插件可以打印一棵树，该树显示项目的所有依赖关系。像这样调用它：

 mvn dependency:tree

将Maven与SpringSource Tool Suite一起使用

Maven鼓励可复制的版本。这些命令是标准的，阶段，插件配置和项目类型，都是标准的。任何安装了Maven的开发人员都应该能够获取项目的源代码及其pom.xml文件，并完全按照原始开发人员在其专用工作站上构建它的方式来对其进行重建。从理论上讲，应该没有必要去调试构建逻辑来调试构建文件，很少或不需要做一些技巧来设置构建所需的环境（属性文件，shell变量等）。这个

一个行家pom.xml文件指定构建项目所需的所有内容。它指定需要哪些库才能进行编译，它知道要使用的编译器，知道哪些源代码在哪里。它了解有关您的项目的所有信息。不足为奇的是，像SpringSource Tool Suite这样的好工具会包含功能（即M2Eclipse插件，它为Eclipse衍生物提供了出色的Maven支持）以直接导入Maven项目。首先，我们需要将项目纳入STS。为此：转到文件>导入> Maven>常规，然后浏览到包含您的文件夹的文件夹pom.xml



在此示例中，我们已将Maven项目导入STS环境，但是您也可以在STS中创建基于Maven的项目。 STS附带了许多可以访问的使用Maven的模板项目。只需转到File > New > Spring Template Project然后选择其中一个项目。它们也是基于Maven的，因此它们是快速入门的好方法。

现在，您已经在STS中运行了一个Maven项目，您可以立即开始编码。浏览“打包资源管理器”。您将在项目中看到一个“ Maven依赖关系”库，其中包含Maven pom中指定的依赖关系以及传递性依赖关系。您还将注意到，M2Eclipse插件已在Maven所需的STS中正确添加了四个源代码根。

至此，我们有一个可以正常运行的STS项目，该项目已经设置了类路径并可以正确构建。我不了解你，但是我的第一个想法是开始编写代码。这个项目感觉就像是一块干净的白色画布，只是乞求在上面绘画。毕竟，我有工作要做！通常，我将添加jUnit并保存pom.xml文件。

<dependency>
  <groupId>junit</groupId>
  <artifactId>junit</artifactId>
  <version>4.5</version>
</dependency>

此时，STS会自动更新项目中的Maven依赖项。

然后，我通常将添加一个日志记录框架并将其保存：


xml log4j log4j 1.2.16

然后，我将开始添加我需要特定于我的任务的依赖项。假设我已经阅读了上周关于Green Beans的文章，并且想利用JDBC和Spring的JdbcTemplate ，我补充说：


xml org.springframework spring-jdbc 3.0.5.RELEASE

我们出发了！让我们写一些代码。这是我个人最喜欢的功能之一M2Eclipse Maven集成开始了。亲爱的读者，您看，我在代码中犯了很多错误。 :-)我花了大量时间运行和调试代码。我通常会破坏一些东西，并且需要调试代码与框架之间的相互作用。Maven依赖项通常与随附的源代码一起部署，因此M2Eclipse插件可以自动为您下载源代码。为了了解这一点，让我们看一下JdbcTemplate.在STS中，转到Navigate > Open Type 。类型JdbcTemplate然后点击Enter 。STS首先会打开.class （默认行为），然后尝试在后台下载源代码。几秒钟后，您应该盯着正确的源代码JdbcTemplate 。然后，您可以检查代码并设置断点，然后在调试器中逐步执行。方便吗？


让Spring Roo做到

我们已经看到，SpringSource Tool Suite是使用Java（以及扩展为Spring和Maven）代码的理想环境。仅使用Maven和STS即可轻松开始基于Spring的应用程序。但是， Spring Roo更加容易，并且既已包含在STS中，也可以从STS内轻松使用。我将“ Spring Roo做到这一点”作为个人政策。Spring Roo为每个Spring Roo项目都设置了一个优雅的Maven构建，因此创建Spring Roo项目的真正举动会给您带来很多好处。让我们创建一个Spring Roo项目（当然，使用STS的向导！），然后看看Spring Roo对Maven的支持如何简化Maven构建的配置。

在STS中，转到File > New > Spring Roo Project 。这将启动Spring Roo向导。您会注意到，向导使您可以指定希望Maven参与的方式。保留默认值- Full Maven Build -指定。

与我们的第一个示例类似，这将为您提供一个新的Maven项目。您需要重复此步骤，其中包括了上一个示例中的资源文件夹。和以前一样，您拥有STS对Maven的所有支持，但是现在您还可以使用Spring Roo Shell来修改和修改Maven项目配置。首先，您可以使用Roo Shell安装新创建的项目：

perform command –mavenCommand install

这相当于跑步mvn install像我们以前那样您应该为任何新创建的Spring Roo项目运行它。Spring Roo还提供了方便，可编写脚本的Shell命令，用于添加和删除Maven依赖项。要添加Spring Integration框架（特别是文件系统特定的支持），您可以发出以下命令。


dependency add –groupId org.springframework.integration –artifactId spring-integration-file –version 2.0.0.RELEASE
perform package

的perform package命令会生成完整的工件，因此.war使用此命令构建的项目（例如）将可以立即部署到您的容器中。还有另一个命令dependency remove ，可用于从Maven项目中删除依赖项。
