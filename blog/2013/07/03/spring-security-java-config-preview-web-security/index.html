原始博客文章

在我以前的文章中 ，我介绍了Spring Security Java配置并讨论了该项目的一些后勤工作。在本文中，我们将通过一个非常简单的Web安全配置开始。然后，我们将使用已自定义的一些配置来使事情更加有趣。

您好网络安全

在本节中，我们将介绍基于Web的安全性的最基本配置。它可以分为四个步骤：

  • 更新您的依赖关系- 我们在之前的博客文章中使用Maven进行了演示
  • 提供Spring Security配置-在我们的示例中，这是通过WebSecurityConfigurerAdapter完成的
  • 确保已加载Spring Security配置-在我们的示例中，这是通过AbstractAnnotationConfigDispatcherServletInitializer完成的
  • 配置springSecurityFilterChain-在我们的示例中，这是通过AbstractSecurityWebApplicationInitializer完成的

WebSecurityConfigurerAdapter

@EnableWebSecurityannotation和WebSecurityConfigurerAdapter一起提供基于Web的安全性。通过扩展WebSecurityConfigurerAdapter和仅几行代码，我们可以执行以下操作：

  • 在访问我们应用程序中的任何URL之前，要求用户经过身份验证
  • 创建一个用户名为“ user”，密码为“ password”，角色为“ ROLE_USER”的用户
  • 启用HTTP基本和基于表单的身份验证
  • Spring Security将自动为您呈现一个登录页面和注销成功页面

@Configuration
@EnableWebSecurity
public class HelloWebSecurityConfiguration
   extends WebSecurityConfigurerAdapter {

  @Autowired
  public void configureGlobal(AuthenticationManagerBuilder auth) {
    auth
      .inMemoryAuthentication()
        .withUser("user").password("password").roles("USER");
  }
}

供您参考，它与以下XML配置类似，但有一些例外：

  • Spring Security将呈现登录名，身份验证失败URL和注销成功URL
  • 仅对HTTP POST处理login-processing-url
  • 仅对HTTP GET处理登录页面

“ WebApplicationInitializer的订购”
如果在调用AbstractSecurityWebApplicationInitializer之后添加了任何Servlet筛选器映射，则可能在springSecurityFilterChain之前意外添加了它们。除非应用程序包含不需要保护的Filter实例，否则springSecurityFilterChain应该在任何其他Filter映射之前。@Orderannotation可用于帮助确保以确定性顺序加载任何WebApplicationInitializer。

抽象安全性WebApplicationInitializer

最后一步是我们需要映射springSecurityFilterChain 。我们可以通过扩展AbstractSecurityWebApplicationInitializer和可选的重写方法来自定义映射来轻松实现此目的。

  • 在已配置的任何Servlet过滤器映射之前插入springSecurityFilterChain映射

CustomWebSecurityConfigurerAdapter

我们的HelloWebSecurityConfiguration示例演示了Spring Security Java配置可以为我们提供一些非常好的默认值。让我们看一些基本的自定义。


@EnableWebSecurity
@Configuration
public class CustomWebSecurityConfigurerAdapter extends
   WebSecurityConfigurerAdapter {
  @Autowired
  public void configureGlobal(AuthenticationManagerBuilder auth) {
    auth
      .inMemoryAuthentication()
        .withUser("user")  // #1
          .password("password")
          .roles("USER")
          .and()
        .withUser("admin") // #2
          .password("password")
          .roles("ADMIN","USER");
  }

  @Override
  public void configure(WebSecurity web) throws Exception {
    web
      .ignoring()
         .antMatchers("/resources/**"); // #3
  }

  @Override
  protected void configure(HttpSecurity http) throws Exception {
    http
      .authorizeUrls()
        .antMatchers("/signup","/about").permitAll() // #4
        .antMatchers("/admin/**").hasRole("ADMIN") // #6
        .anyRequest().authenticated() // 7
        .and()
    .formLogin()  // #8
        .loginUrl("/login") // #9
        .permitAll(); // #5
  }
}

假设我们调整AbstractAnnotationConfigDispatcherServletInitializer来加载我们的新配置，我们的CustomWebSecurityConfigurerAdapter将执行以下操作：

  • 允许与名为“ user”的用户进行内存身份验证
  • 允许使用名为“ admin”的管理用户进行内存身份验证
  • 忽略任何以“ / resources /”开头的请求。这类似于配置使用XML名称空间配置时， http @ security = none 。
  • 允许任何人（包括未经身份验证的用户）访问URL“ / signup”和“ / about”
  • 允许任何人（包括未经身份验证的用户）访问URL“ / login”和“ / login？错误”。在这种情况下，permitAll（）意味着允许访问formLogin（）使用的任何URL。
  • 以“ / admin /”开头的任何URL必须是管理用户。对于我们的示例，将是用户“ admin”。
  • 其余所有URL均要求用户成功通过身份验证
  • 使用Java配置默认设置基于表单的身份验证。当使用参数“用户名”和“密码”将POST提交到URL“ / login”时，将执行身份验证。
  • 明确声明登录页面，这意味着要求GET / login时要求开发人员呈现登录页面。

与XML命名空间的相似之处

看完我们稍微复杂一点的示例之后，您也许可以找到XML名称空间和Java配置之间的一些相似之处。以下是一些更有用的要点：

  • HttpSecurity与http名称空间元素非常相似。它允许为特定选择（在本例中为全部）请求配置基于Web的安全性。
  • WebSecurity与用于Web且不需要父级的任何Security命名空间元素非常相似（即security = none，debug等）。它允许配置影响所有Web安全的内容。
  • WebSecurityConfigurerAdapter是一个便捷类，它允许对WebSecurity和HttpSecurity进行自定义。我们可以多次扩展WebSecurityConfigurerAdapter（在不同的对象中）以复制具有多个http元素的行为。
  • 通过格式化我们的Java配置代码，它更容易阅读。它的读取方式类似于XML名称空间，其中“ and（）”表示有选择地关闭XML元素。

与XML命名空间的区别

您会注意到，XML和Java配置之间也有一些重要的区别。

  • 在＃1和＃2中创建用户时，我们不像在XML配置中那样指定“ ROLE_”。由于这种约定非常普遍，因此“ roles”方法会自动为您添加“ ROLE_”。如果您不希望添加“ ROLE_”，则可以使用Authority方法。
  • Java配置具有不同的默认URL和参数。创建自定义登录页面时，请记住这一点。结果是我们的URL更RESTful。此外，我们正在使用Spring Security来帮助防止信息泄漏并不是很明显。例如：
  • GET / login呈现登录页面，而不是/ spring_security_login
  • POST / login验证用户身份，而不是/ j_spring_security_check
  • username参数默认为username而不是j_username
  • password参数默认为password而不是j_password
  • Java配置可以轻松地将多个请求匹配器映射到相同的角色。这在＃4中很明显，其中我们映射了两个URL，任何人都可以访问
  • Java配置尝试删除冗余代码。例如，与其像在XML中那样在form-login元素和拦截URL元素中重复我们的/ login URL，我们只需声明用户应该可以访问与formLogin（）相关的任何URL，如＃所示。 5
  • 当像我们在＃6中那样使用hasRole方法映射HTTP请求时，我们不需要像在XML中那样指定“ ROLE_”。同样，这是一种惯例，因此hasRole方法会自动为您添加“ ROLE_”。如果您不想自动以“ ROLE_”作为前缀，则可以使用“ access”方法。
