Spring MVC中的异常处理

工程
保罗·查普曼
2013年11月1日

注意： 2018年4月修订

Spring MVC提供了几种互补的异常处理方法，但是，在教授Spring MVC时，我经常发现我的学生对它们感到困惑或不满意。

今天，我将向您展示可用的各种选项。我们的目标是尽可能不在 Controller方法中显式处理异常。它们是一个跨领域的问题，最好在专用代码中单独处理。

有三个选项：每个异常，每个控制器或全局。

可以在此处找到展示此处讨论要点的演示应用程序。
http://github.com/paulc4/mvc-exceptions 。有关详细信息，请参见下面的示例应用程序 。

注意： 演示应用程序已经过更新和更新（2018年4月）以使用Spring Boot 2.0.1，并且（希望）更易于使用和理解。我还修复了一些断开的链接（感谢您的反馈，很抱歉花了一段时间）。

春季靴

Spring Boot允许使用最少的配置来设置Spring项目，并且如果您的应用程序还不到几年，则很可能正在使用它。

Spring MVC不提供开箱即用的默认（备用）错误页面。设置默认错误页面的最常见方法一直是SimpleMappingExceptionResolver （实际上是从Spring V1开始）。我们将在后面讨论。

但是Spring Boot 确实提供了一个后备错误处理页面。

在启动时，Spring Boot会尝试找到一个映射/error 。按照约定，URL结尾/error映射到同名的逻辑视图： error 。在演示应用程序中，此视图依次映射到error.html胸腺模板。（如果使用JSP，它将映射到error.jsp根据您的设置InternalResourceViewResolver ）。实际的映射将取决于ViewResolver （如果有的话）您或Spring Boot已设置。

如果没有用于的视图解析器映射/error可以找到，Spring Boot定义了自己的后备错误页面-所谓的“ Whitelabel错误页面”（最小页面，其中仅包含HTTP状态信息和任何错误详细信息，例如未捕获的异常的消息）。在示例应用程序中，如果您重命名error.html例如， error2.html然后重新启动，您将看到它正在使用。

如果您正在发出RESTful请求（HTTP请求指定了除HTML之外的所需响应类型），Spring Boot将返回与错误信息相同的JSON表示形式，该信息与“ Whitelabel”错误页面中放入的信息相同。

$> curl -H "Accept: application/json" http://localhost:8080/no-such-page

{"timestamp":"2018-04-11T05:56:03.845+0000","status":404,"error":"Not Found","message":"No message available","path":"/no-such-page"}

Spring Boot还为容器设置了一个默认的错误页面，等效于
指令中web.xml （尽管实施方式大不相同）。Spring Boot回退错误页面仍会报告在Spring MVC框架之外引发的异常，例如从Servlet过滤器引发的异常。该示例应用程序还显示了一个示例。

在本文的末尾可以找到有关Spring Boot错误处理的更深入的讨论。

无论您使用的是带Spring Boot还是不带Spring Boot的Spring，本文的其余内容均适用 。

不耐烦的REST开发人员可能会选择直接跳到有关自定义REST错误响应的部分 。但是，他们应该阅读全文，因为大多数文章同样适用于所有Web应用程序（REST或其他）。

使用HTTP状态码

通常，处理Web请求时引发的任何未处理的异常都会导致服务器返回HTTP 500响应。但是，您自己编写的任何异常都可以用@ResponseStatus批注（支持HTTP规范定义的所有HTTP状态代码）。当从控制器方法引发带注释的异常，并且未在其他地方处理该异常时，它将自动导致使用指定的状态代码返回适当的HTTP响应。

例如，这是缺少订单的例外。

 @ResponseStatus(value=HttpStatus.NOT_FOUND, reason="No such Order")  // 404
 public class OrderNotFoundException extends RuntimeException {
     // ...
 }

这是使用它的控制器方法：

 @RequestMapping(value="/orders/{id}", method=GET)
 public String showOrder(@PathVariable("id") long id, Model model) {
     Order order = orderRepository.findOrderById(id);

     if (order == null) throw new OrderNotFoundException(id);

     model.addAttribute(order);
     return "orderDetail";
 }

如果此方法处理的URL包含未知的订单ID，则将返回熟悉的HTTP 404响应。

基于控制器的异常处理

使用@ExceptionHandler

您可以添加额外的（ @ExceptionHandler ）任何控制器的方法，以专门处理由请求处理（ @RequestMapping ）方法在同一控制器中。此类方法可以：

  1. 在没有异常的情况下处理异常@ResponseStatus批注（通常未编写的预定义异常）
  2. 将用户重定向到专用错误视图
  3. 建立完全自定义的错误响应

以下控制器演示了这三个选项：

@Controller
public class ExceptionHandlingController {

  // @RequestHandler methods
  ...
  
  // Exception handling methods
  
  // Convert a predefined exception to an HTTP Status code
  @ResponseStatus(value=HttpStatus.CONFLICT,
                  reason="Data integrity violation")  // 409
  @ExceptionHandler(DataIntegrityViolationException.class)
  public void conflict() {
    // Nothing to do
  }
  
  // Specify name of a specific view that will be used to display the error:
  @ExceptionHandler({SQLException.class,DataAccessException.class})
  public String databaseError() {
    // Nothing to do.  Returns the logical view name of an error page, passed
    // to the view-resolver(s) in usual way.
    // Note that the exception is NOT available to this view (it is not added
    // to the model) but see "Extending ExceptionHandlerExceptionResolver"
    // below.
    return "databaseError";
  }

  // Total control - setup a model and return the view name yourself. Or
  // consider subclassing ExceptionHandlerExceptionResolver (see below).
  @ExceptionHandler(Exception.class)
  public ModelAndView handleError(HttpServletRequest req, Exception ex) {
    logger.error("Request: " + req.getRequestURL() + " raised " + ex);

    ModelAndView mav = new ModelAndView();
    mav.addObject("exception", ex);
    mav.addObject("url", req.getRequestURL());
    mav.setViewName("error");
    return mav;
  }
}

在上述任何一种方法中，您都可以选择进行其他处理-最常见的示例是记录异常。

处理程序方法具有灵活的签名，因此您可以传递与Servlet相关的显而易见的对象，例如HttpServletRequest ， HttpServletResponse ， HttpSession和/或Principle 。

重要说明： Model可能不是任何参数@ExceptionHandler方法。相反，请使用ModelAndView如图所示handleError()以上。

例外与观点

向模型添加例外时要小心。您的用户不想看到包含Java异常详细信息和堆栈跟踪的网页。您可能具有明确禁止将任何异常信息放入错误页面的安全策略。确保覆盖Spring Boot白色标签错误页面的另一个原因。

确保异常记录有用，以便事件发生后由您的支持和开发团队进行分析。

请记住以下内容可能很方便，但这并不是生产中的最佳实践 。

将异常详细信息作为注释隐藏在页面源中可能会很有用，以帮助测试 。如果使用JSP，则可以执行以下操作以输出异常和相应的堆栈跟踪（使用隐藏的

是另一种选择）。

  <h1>Error Page</h1>
  <p>Application has encountered an error. Please contact support on ...</p>
    
  <!--
    Failed URL: ${url}
    Exception:  ${exception.message}
        <c:forEach items="${exception.stackTrace}" var="ste">    ${ste} 
    </c:forEach>
  -->

全局异常处理

使用@ControllerAdvice类

控制器建议允许您使用完全相同的异常处理技术，但将其应用于整个应用程序，而不仅是单个控制器。您可以将它们视为注释驱动的拦截器。

任何带注释的类@ControllerAdvice成为控制器建议，并支持三种方法：

  • 带有注释的异常处理方法@ExceptionHandler 。
  • 模型增强方法（用于向模型添加其他数据）
    @ModelAttribute 。请注意，这些属性不可用于异常处理视图。
  • 用以下注释的活页夹初始化方法（用于配置表单处理）
    @InitBinder 。

我们将只研究异常处理-搜索在线手册以获取更多信息@ControllerAdvice方法。

您上面看到的任何异常处理程序都可以在controller-advice类上定义-但是现在它们适用于从任何控制器抛出的异常。这是一个简单的示例：

@ControllerAdvice
class GlobalControllerExceptionHandler {
    @ResponseStatus(HttpStatus.CONFLICT)  // 409
    @ExceptionHandler(DataIntegrityViolationException.class)
    public void handleConflict() {
        // Nothing to do
    }
}

如果您想为任何异常使用默认处理程序，则会有些许皱纹。您需要确保框架处理注释的异常。代码如下：

@ControllerAdvice
class GlobalDefaultExceptionHandler {
  public static final String DEFAULT_ERROR_VIEW = "error";

  @ExceptionHandler(value = Exception.class)
  public ModelAndView
  defaultErrorHandler(HttpServletRequest req, Exception e) throws Exception {
    // If the exception is annotated with @ResponseStatus rethrow it and let
    // the framework handle it - like the OrderNotFoundException example
    // at the start of this post.
    // AnnotationUtils is a Spring Framework utility class.
    if (AnnotationUtils.findAnnotation
                (e.getClass(), ResponseStatus.class) != null)
      throw e;

    // Otherwise setup and send the user to a default error-view.
    ModelAndView mav = new ModelAndView();
    mav.addObject("exception", e);
    mav.addObject("url", req.getRequestURL());
    mav.setViewName(DEFAULT_ERROR_VIEW);
    return mav;
  }
}

更深入

HandlerExceptionResolver

在其中声明的任何Spring bean DispatcherServlet实现的应用程序上下文HandlerExceptionResolver将用于拦截和处理在MVC系统中引发且未被控制器处理的任何异常。该界面如下所示：

public interface HandlerExceptionResolver {
    ModelAndView resolveException(HttpServletRequest request, 
            HttpServletResponse response, Object handler, Exception ex);
}

的handler指产生异常的控制器（请记住， @Controller实例只是Spring MVC支持的一种处理程序。例如： HttpInvokerExporter和WebFlow Executor也是处理程序的类型）。

在后台，MVC默认情况下会创建三个此类解析器。这些解析器实现了上面讨论的行为：

  • ExceptionHandlerExceptionResolver将未捕获的异常与适当的匹配@ExceptionHandler处理程序（控制器）和任何控制器建议上的方法。
  • ResponseStatusExceptionResolver寻找未被注释的未捕获异常@ResponseStatus （如第1节所述）
  • DefaultHandlerExceptionResolver转换标准的Spring异常并将其转换为HTTP状态代码（我在上面没有提到，因为它是Spring MVC的内部功能）。

它们按照列出的顺序进行链接和处理-在内部，Spring创建一个专用的bean（ HandlerExceptionResolverComposite ） 去做这个。

注意，方法签名resolveException不包括Model 。这就是为什么@ExceptionHandler方法不能与模型一起注入。

您可以根据需要实施自己的HandlerExceptionResolver设置您自己的自定义异常处理系统。处理程序通常实现Spring的Ordered接口，因此您可以定义处理程序的运行顺序。

SimpleMappingExceptionResolver

长期以来，Spring提供了一种简单但方便的实现HandlerExceptionResolver您很可能已经发现您的应用已经使用了- SimpleMappingExceptionResolver 。它提供以下选项：

  • 映射异常类名称以查看名称-只需指定类名称，就不需要包。
  • 为任何未在其他地方处理的异常指定默认（备用）错误页面
  • 记录一条消息（默认情况下未启用）。
  • 设置名称exception属性添加到模型，以便可以在视图内使用
    （例如JSP）。默认情况下，此属性命名为exception 。调成null禁用。请记住，视图是从@ExceptionHandler方法无权访问异常，但定义为SimpleMappingExceptionResolver 做 。

这是使用Java配置的典型配置：

@Configuration
@EnableWebMvc  // Optionally setup Spring MVC defaults (if you aren't using
               // Spring Boot & haven't specified @EnableWebMvc elsewhere)
public class MvcConfiguration extends WebMvcConfigurerAdapter {
  @Bean(name="simpleMappingExceptionResolver")
  public SimpleMappingExceptionResolver
                  createSimpleMappingExceptionResolver() {
    SimpleMappingExceptionResolver r =
                new SimpleMappingExceptionResolver();

    Properties mappings = new Properties();
    mappings.setProperty("DatabaseException", "databaseError");
    mappings.setProperty("InvalidCreditCardException", "creditCardError");

    r.setExceptionMappings(mappings);  // None by default
    r.setDefaultErrorView("error");    // No default
    r.setExceptionAttribute("ex");     // Default is "exception"
    r.setWarnLogCategory("example.MvcLogger");     // No default
    return r;
  }
  ...
}

或使用XML配置：

  <bean id="simpleMappingExceptionResolver" class=
     "org.springframework.web.servlet.handler.SimpleMappingExceptionResolver">
    <property name="exceptionMappings">
      <map>
         <entry key="DatabaseException" value="databaseError"/>
         <entry key="InvalidCreditCardException" value="creditCardError"/>
      </map>
    </property>

    <!-- See note below on how this interacts with Spring Boot -->
    <property name="defaultErrorView" value="error"/>
    <property name="exceptionAttribute" value="ex"/>
        
    <!-- Name of logger to use to log exceptions. Unset by default, 
           so logging is disabled unless you set a value. -->
    <property name="warnLogCategory" value="example.MvcLogger"/>
  </bean>

defaultErrorView属性特别有用，因为它可以确保任何未捕获的异常都会生成合适的应用程序定义的错误页面。（大多数应用程序服务器的默认设置是显示Java堆栈跟踪，这是用户永远都不会看到的）。Spring Boot的“ white-label”错误页面提供了另一种做同样事情的方法。

扩展SimpleMappingExceptionResolver

扩展很普遍SimpleMappingExceptionResolver有几个原因：

  • 您可以使用构造函数直接设置属性-例如，启用异常日志记录并将记录器设置为使用
  • 通过覆盖覆盖默认日志消息buildLogMessage 。默认实现始终返回以下固定文本：
      处理程序执行导致异常
  • 通过覆盖使其他信息可用于错误视图doResolveException

例如：

public class MyMappingExceptionResolver extends SimpleMappingExceptionResolver {
  public MyMappingExceptionResolver() {
    // Enable logging by providing the name of the logger to use
    setWarnLogCategory(MyMappingExceptionResolver.class.getName());
  }

  @Override
  public String buildLogMessage(Exception e, HttpServletRequest req) {
    return "MVC exception: " + e.getLocalizedMessage();
  }
    
  @Override
  protected ModelAndView doResolveException(HttpServletRequest req,
        HttpServletResponse resp, Object handler, Exception ex) {
    // Call super method to get the ModelAndView
    ModelAndView mav = super.doResolveException(req, resp, handler, ex);
        
    // Make the full URL available to the view - note ModelAndView uses
    // addObject() but Model uses addAttribute(). They work the same. 
    mav.addObject("url", request.getRequestURL());
    return mav;
  }
}

此代码在演示应用程序中为ExampleSimpleMappingExceptionResolver

扩展ExceptionHandlerExceptionResolver

也可以扩展ExceptionHandlerExceptionResolver并覆盖其
doResolveHandlerMethodException方法相同。它具有几乎相同的签名（只是采用了新的HandlerMethod代替Handler ）。

为了确保它被使用，还要将继承的order属性（例如，在新类的构造函数中）设置为小于MAX_INT因此它可以在默认的ExceptionHandlerExceptionResolver实例之前运行（创建自己的处理程序实例比尝试修改/替换Spring创建的实例更容易）。有关更多信息，请参见演示应用程序中的ExampleExceptionHandlerExceptionResolver 。

错误和REST

RESTful GET请求也可能会生成异常，并且我们已经了解了如何返回标准的HTTP错误响应代码。但是，如果要返回有关错误的信息怎么办？这很容易做到。首先定义一个错误类：

public class ErrorInfo {
    public final String url;
    public final String ex;

    public ErrorInfo(String url, Exception ex) {
        this.url = url;
        this.ex = ex.getLocalizedMessage();
    }
}

现在我们可以从处理程序中返回一个实例作为@ResponseBody像这样：

@ResponseStatus(HttpStatus.BAD_REQUEST)
@ExceptionHandler(MyBadDataException.class)
@ResponseBody ErrorInfo
handleBadRequest(HttpServletRequest req, Exception ex) {
    return new ErrorInfo(req.getRequestURL(), ex);
}

什么时候使用？

和往常一样，Spring喜欢为您提供选择，那么您应该怎么做？这是一些经验法则。但是，如果您偏爱XML配置或注释，也可以。

  • 对于您编写的异常，请考虑添加@ResponseStatus给他们。
  • 对于所有其他例外，请实施@ExceptionHandler上的方法@ControllerAdvice类或使用的实例SimpleMappingExceptionResolver 。你可能有SimpleMappingExceptionResolver已经为您的应用程序进行了配置，在这种情况下，向其添加新的异常类可能比实现@ControllerAdvice 。
  • 对于控制器特定的异常处理，添加@ExceptionHandler控制器的方法。
  • 警告：在同一应用程序中，请小心混合太多这些选项。如果可以以多种方式处理同一个异常，则可能无法获得所需的行为。 @ExceptionHandler总是先选择Controller上的方法@ControllerAdvice实例。尚不确定处理哪种顺序的控制器建议。

Spring Boot和错误处理

Spring Boot允许使用最少的配置来设置Spring项目。当Spring Boot在类路径上检测到某些键类和包时，它会自动创建合理的默认值。例如，如果发现您正在使用Servlet环境，它将使用最常用的视图解析器，处理程序映射等设置Spring MVC。如果看到JSP和/或Thymeleaf，它将设置这些视图技术。

后备错误页面

Spring Boot如何支持本文开头所述的默认错误处理？

  1. 如果发生任何未处理的错误，Spring Boot会内部转发给/error 。
  2. 开机设置BasicErrorController处理任何要求/error 。控制器将错误信息添加到内部模型并返回error作为逻辑视图名称。
  3. 如果配置了任何视图解析器，它们将尝试使用相应的错误视图。
  4. 否则，将使用专用的View对象（使其独立于您可能使用的任何视图分辨率系统）。
  5. Spring Boot设置了一个BeanNameViewResolver以便/error可以映射到View同名。
  6. 如果你看Boot的ErrorMvcAutoConfiguration上课，您会看到defaultErrorView以名为的bean的形式返回error 。这是由BeanNameViewResolver 。

“ Whitelabel”错误页面故意最小且难看。您可以覆盖它：

  1. 通过定义错误模板-在我们的演示中，我们使用Thymeleaf，因此错误模板位于src/main/resources/templates/error.html （此位置由Spring Boot属性设置spring.thymeleaf.prefix -其他受支持的服务器端视图技术（例如JSP或Mustache）也存在类似的属性。
  2. 如果您不使用服务器端渲染
    2.1将自己的错误视图定义为名为的bean error 。
    2.1或通过设置属性禁用Spring Boot的“ Whitelabel”错误页面
    server.error.whitelabel.enabled至false 。而是使用容器的默认错误页面。

按照惯例，Spring Boot属性通常在application.properties要么application.yml 。

与SimpleMappingExceptionResolver集成

如果您已经在使用该怎么办SimpleMappingExceptionResolver设置默认值
错误视图？简单易用setDefaultErrorView()定义Spring Boot使用的相同视图： error 。

请注意，在演示中， defaultErrorView的属性SimpleMappingExceptionResolver故意设置为不error但是defaultErrorPage这样您就可以看到处理程序何时生成错误页面以及Spring Boot何时负责。通常两者都将设置为error 。

容器范围的异常处理

Spring Boot的后备错误页面也会报告在Spring Framework外部抛出的异常，例如从Servlet过滤器引发的异常。

为此，Spring Boot必须为容器注册一个默认错误页面。在Servlet 2中，有一个您可以添加到您的指令中web.xml去做这个。不幸的是，Servlet 3没有提供Java API等效项。相反，Spring Boot会执行以下操作：

  • 对于具有嵌入式容器的Jar应用程序，它将使用特定于容器的API来注册默认错误页面。
  • 对于作为传统WAR文件部署的Spring Boot应用程序，Servlet过滤器用于
    捕获进一步提出的异常并进行处理。
