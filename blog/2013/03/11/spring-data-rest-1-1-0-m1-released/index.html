Spring Data REST 1.1.0。M1发布

发布
乔恩·布里斯宾
2013年3月11日

Spring Data团队很高兴宣布使用RESTful语义将域对象导出到Web的演进中的下一个重大步骤：Spring Data REST 1.1.0。M1现在在SpringSource里程碑存储库中可用。

Spring Data REST主页 源于GitHub | 参考文件

将域对象导出到Web

Spring Data REST是一组Spring MVC组件，您可以将其添加到自己的Spring MVC应用程序中，这些应用程序使用RESTful HATEOAS语义将Spring Data Repository导出到Web。通过将存储库导出到RESTful URL（可通过几种不同方式配置），它提供了一致的交互API。

Spring Data REST通过直接编写一行定义扩展Spring Data的接口的代码来支持顶级实体（由Spring Data Repository直接管理的域对象）的CRUD。 CrudRepository接口。完成后，您的实体便具有完整的RESTful语义。您可以使用遵循HATEOAS原理的可发现的标准URL创建新的，更新现有的并删除它们。这意味着访问您的Spring Data REST应用程序的用户代理不需要预先了解要导出的资源。通过连续调用JSON中提供的URL，它可以发现存在哪些实体以及这些实体上存在什么关系。这些“链接”是HATEOAS REST应用程序的真正基础和强大功能。

从头开始的变化

1.1版实际上是从头开始重写的。不仅比1.0更容易配置，并且更好地符合Spring MVC对过渡到Spring 3.2的期望，而且Spring Data REST内部的最大变化是，它现在不仅支持JPA，还支持其他类型的Spring Data存储库实现。无论使用哪种支持数据存储，CRUD和管理关系（如果数据存储支持）的HTTP语义都保持不变。

这意味着现在可以在同一Spring Data REST应用程序中导出JPA实体和MongoDB实体，并使用通用URL结构以及所有实体和集合的标准Spring HATEOAS资源表示形式访问这些实体。访问那些RESTful URL的用户代理不需要任何特殊知识，也无需了解支持实体由哪个数据存储管理，最重要的是，您无需编写任何代码即可获得该功能！

MongoDB支持

Spring Data REST 1.1现在支持导出MongoDB CrudRepository实现。相同的HTTP语义适用于MongoDB @Document实体，适用于JPA实体。GET，POST，PUT，DELETE当然受支持，但是也是如此@DBRef 。您可以使用GET，POST，PUT和DELETE查看和管理两个文档之间的关系，并可以根据您的导出导出方法@Query定义。请参考spring-data-mongodb参考文档 ，以获取关于对象映射与JPA样式映射的不同之处以及查询定义如何工作的全部详细信息。

Gemfire支持

Spring Data REST 1.1现在支持将使用高性能Gemfire数据库的实体导出到不同的区域。阅读Spring Data Gemfire文档 ，以获取有关大量配置选项的完整说明，以及Gemfire中的POJO映射与其他映射技术有何不同。

下一个Neo4J支持

Spring Data REST 1.1现在设置为支持Neo4J GraphRepository带有spring-data-neo4j的下一版本2.3。当这些内容普遍可用时（应该在Spring Data REST 1.1 RELEASE的一般可用性左右或之前），您将可以访问@NodeEntity以及它们之间的关系使用标准HATEOAS语义，就像处理其他数据存储区一样。

将其添加到您现有的应用中

Spring Data REST的设计方式使您可以根据需要为Spring Data REST应用程序创建整个应用程序。毕竟，这只是一个标准的Spring MVC Web应用程序。但是，当您将Spring Data REST添加到自己的服务中时，事情会变得非常有趣。

Spring HATEOAS的作者Oliver Gierke创建了一个示例应用程序，演示了在现代Web应用程序中HATEOAS原理的使用。它称为spring-restbucks ，是REST服务的系统神学中描述的Restbucks应用程序的实现：Jim Webber，Savas Parastatidis和Ian Robinson 撰写的 REST in Practice 。

Mixin REST服务

通过将Spring Data REST与您的其他RESTful服务混合，您可以在Spring Data REST导出的那些域对象（这些对象无需编写任何代码即可公开）与这些服务之间实现无缝集成。不代表实际实体，而是过程。您可以在spring-restbucks应用程序中看到一个付款服务如何与域对象CRUD交互的示例，其中信用卡付款处理由自定义控制器处理，而对象CRUD由Spring Data REST处理。您的自定义控制器实际上可以搭载在Spring Data REST URL上，从而无论URL是否引用您的自定义控制器，Spring Data REST JPA存储库或其他任何内容，都可以在整个应用程序中维护一致且简单的URL结构。支持的存储库样式。

不是唯一的

与Spring Data REST并非两全其美。如果您不希望所有存储库都公开给Web客户端，那就没问题了！您可以通过几种不同的方法来关闭存储库功能。您可以将注释嵌入源代码中，或者，如果您无权访问或根本无法添加Spring Data REST注释，则可以使用流畅的DSL风格的配置来告诉Spring Data REST应该如何公开您的资源。在您的应用程序中使用Spring Data REST并不是仅一种工作方式的唯一承诺。Spring Data REST的结构使其可以很好地与现有应用程序配合使用，因此您可以合并所需的Spring Data REST中的这些功能，同时仍保留您在Spring MVC中创建的所有自定义编码服务控制器。

JSONP支持移至过滤器

Spring Data REST 1.0中内置的JSONP支持已从核心框架中删除，而不是即将推出的通用JSONP Serlvet筛选器，它比1.0版中实现JSONP的方式要好得多。当该过滤器普遍可用时，不仅可以将JSONP支持添加到Spring Data REST，而且实际上可以添加任何基于Servlet的REST资源。
