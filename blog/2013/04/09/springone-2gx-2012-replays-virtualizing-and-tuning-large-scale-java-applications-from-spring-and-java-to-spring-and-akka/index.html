SpringOne 2GX 2012重播：虚拟化和优化从Spring和Java到Spring和Akka的大规模Java应用程序

新闻与活动
彼得·汉弗莱（Pieter Humphrey）
2013年4月9日

虚拟化和优化大型Java应用程序



本次会议分享了许多在vSphere上运行Java的经过生产验证的方法。涵盖如何为大型部署调整JVM和VM的大小。该课程将专门讨论GC调优，并展示如何使用过去15年调优JVM的实际经验开发的GC配方来调优各种JVM。


在本节中讨论了三个关键趋势和相关的调整技术。关键趋势是：整合，弹性和灵活性以及性能


整合我们的许多客户发现他们的中间件部署已经激增，并且正成为与更高成本相关的管理挑战。我们看到了希望将虚拟化作为减少服务器实例数量的一种方式的客户的趋势。同时，客户正在利用整合机会来合理化服务特定负载所需的中间件组件的数量。中间件组件最通常在Java虚拟机（JVM）中运行，观察到的JVM实例规模为100到1000个，并且为JVM实例整合提供了许多机会。因此，中间件虚拟化提供了两次整合的机会-一次整合服务器实例，其次，整合JVM实例。这种趋势意义深远，因为地球上的每个IT部门都在考虑节省整合成本。酒店行业的一位客户经历了整合服务器占用空间的过程，同时整合了许多小于1GB堆的小型JVM。他们将许多较小的1GB JVM分为2类，分别为4GB和6GB。他们以这种方式执行合并，以使应用程序可用的RAM的净总量等于原始RAM的总量，但JVM实例更少。他们在提高性能和保持良好的SLA的同时完成了所有这些工作。由于减少了必须管理的JVM实例的数量，并且简化了有助于轻松实现SLA的环境，他们还大大降低了管理成本。

保险行业的另一个客户能够达到与上述客户相同的价格，但是另外还能够在开发和QA环境中超额使用CPU，以节省第三方软件许可成本。另一方面，有时我们会遇到有合法业务要求的客户，这些客户需要为一个应用程序维护一个JVM，和/或每条业务线维护一个JVM。在这些情况下，您无法真正整合JVM实例，因为这将导致一个应用程序的业务生命周期与另一业务部门的生命周期混杂在一起。但是，虽然这些客户没有从通过JVM合并消除额外的JVM实例中受益，但他们的确受益于更充分地利用服务器硬件上的可用计算资源，否则这些资源在未虚拟化的环境中将无法得到充分利用。


弹性和灵活性查找具有季节性需求的应用程序越来越普遍。例如，我们的许多客户开展了各种营销活动，将季节性流量推向他们的应用程序。使用VMware，您可以通过在需要时自动置备新的虚拟机和中间件组件，然后在负载减轻时自动关闭这些VM来处理这种流量突发事件。此外，对于支持云时代规模和正常运行时间的中间件而言，在不引起中断的情况下更改更新/补丁硬件的能力至关重要。VMware VMotion使您能够移动虚拟机，而无需停止施加器和/或虚拟机。在管理大规模中间件部署时，仅此灵活性就使中间件虚拟化值得。财务领域的一位客户每天处理数百万笔交易，因此经常使用VMotion安排其硬件升级，而无需停机。否则，原本计划中的业务停机时间将十分昂贵。


性能客户通常会在虚拟化时报告中间件平台性能得到改善。性能的提高部分归因于客户通常会在虚拟化项目期间更新的更新硬件。由于强大的VMware虚拟机管理程序，性能也有所提高。近期一位表现出色的客户提供了以下证词


“通过我们的OrderExpress项目，我们升级了中间件服务，商务，门户，WCM，服务层，DB2数据库；从AIX迁移到Linux；在VMware上虚拟化；将应用程序移动到三层DMZ中；使我们的交易量增加了150％以上；并添加了重要的新功能，这些功能极大地改善了客户体验。一次更改如此广泛的技术组件是一个巨大的挑战。但是，使用VMware vSphere和其他体系结构更改，我们成功地将性能提高了300％以上；降低了数百万的成本；改进的安全性，可用性和可伸缩性；以及我们计划如何继续发展该应用程序，以保持每年30％以上的增长。”


– Cardinal Health的高级企业架构师Jeff Battisti


在本次会议中，我将展示一些针对小型和大型JVM的中间件组件的实际JVM和VM大小。还将详细介绍我多年来开发的GC调优配方，该配方已被证明可以处理4GB至88GB以上的JVM堆大小。当然，内存数据库的引入推动了拥有更大的JVM的趋势，因此为什么我们将讨论调整JVM，VM和它们所部署的硬件平台的最佳方法是什么。


我认为选型问题是我们客户群中最常见的问题，因此，我计划在本届会议期间重点关注该问题。

从Spring + Java到Spring + Akka-探索之旅

基于参与者的并发模型是一种范式转移。什么是范式转换？

科学统治理论中基本假设或范式的改变。范式的转变使简单的想法难以理解。尽管Actor模型是一个简单且具有革命性的想法，但程序员很难看到实际的好处和用法。同样，Scala编程语言在简化基于actor的设计方面带来了很多好处，但提供了一种新的编码样式。解决此问题的一种可能方法是根据旧范式评估新范式。在本演示中，我们将使用一个基于Spring的可运行Web应用程序，并使用actor和Scala逐步实现它。我们会对新想法表示怀疑和怀疑，但与此同时，我们将保持开放的态度。我们将使用spring的知识来学习基于参与者的并发模型。同时，我们将学习Actor，Scala背后的最佳实践，并将其与Spring框架相结合。
