Spring XD 1.0里程碑1发布

工程
马克·波拉克
2013年6月12日

今天我们很高兴地宣布Spring XD的1.0 M1版本（下载） 。Spring XD是一个统一的，分布式的，可扩展的系统，用于数据摄取，实时分析，批处理和数据导出。该项目的目标是简化大数据应用程序的开发。

从10,000英尺的角度来看，大数据应用程序与企业集成和批处理应用程序共享许多特征。过去6年来，Spring通过Spring Integration和Spring Batch项目为构建集成和批处理应用程序提供了行之有效的解决方案。Spring XD在此基础上构建，并提供了一个轻量级的运行时环境，可通过简单的DSL轻松配置和组装该环境。

在此博客中，我们将介绍Spring XD的关键组件，即用于声明它们的Streams，Jobs，Taps，Analytics和DSL，以及运行时体系结构。可以在XD指南中找到更多详细信息。

流


流定义了如何收集，处理，存储或转发数据。例如，流可以收集syslog数据，对其进行过滤，然后将其存储在HDFS中。Spring XD提供了DSL来定义流。DSL允许您使用UNIX管道过滤器语法来开始简单的操作，以构建线性处理流程，但也可以使用扩展语法来描述更复杂的流程。

源和汇


一个简单的线性流包括以下序列：输入源，（可选）处理步骤和输出接收器。作为一个简单的示例，请考虑从HTTP源写入文件接收器的数据收集。描述该流的DSL是

http | file

您告诉Spring XD通过向默认在端口8080上运行的XD Admin Server发出HTTP请求来创建流。在M2版本中，我们将提供一个与XD进行通信的交互式外壳，但是对于M1，最简单的方法是使用“ curl”与XD进行交互。

curl -d "http | file" http://localhost:8080/streams/httptest

流的名称为httptest ，要侦听的默认HTTP端口为9000 ，默认文件位置为/tmp/xd/output/${streamname} 。

如果使用curl在端口9000上发布一些数据

curl -d "hello world" http://localhost:9000

您将在文件/tmp/xd/output/httptest看到字符串hello world

要更改默认值，您可以传入选项参数

http --port=9090 | file --dir=/var/streams --name=data.txt

M1中支持的源是文件，时间，HTTP，尾部，Twitter搜索，Gemfire（连续查询），Gemfire（缓存事件），Syslog和TCP。支持的接收器包括日志，文件，HDFS，Gemfire分布式数据网格和TCP。要将syslog数据捕获到HDFS，只需简单地使用DSL

syslog | hdfs --namenode="http://192.168.1.100:9000"

您还可以添加自己的自定义源和接收器。遵循一个简单的方法，就可以添加Spring Integration中现有的入站和出站通道适配器。未来的版本将增加对MQTT，RabbitMQ，JMS和Kafka的支持。我们希望提出请求，以贡献您喜欢的源和接收器模块。

Stream的编程模型基于Spring Integration。输入源将外部数据转换为由标头组成的消息，该标头包含键值对和有效载荷，可以是任何Java类型。消息通过消息通道流经流。下面显示了具有输入源，处理步骤和输出接收器的流的情况。

在DSL中，管道符号对应于将数据从每个处理步骤传递到下一步骤的通道。Spring XD中的通道可以在内存中，也可以由中间件（例如Redis，JMS，RabbitMQ等）支持。这允许一个简单的分布式处理模型，稍后将进行讨论。


表示带有处理步骤的流的DSL表示形式为

source | filter | transform | sink

M1中支持的处理器是过滤器，转换器，json-field-extractor，json-field-value-filter和脚本。过滤器和转换器处理器支持使用Spring Expression Language（SpEL）和Groovy。要将上一个示例中的HTTP请求的有效负载转换为大写，请使用SpEL，

http | transform --expression=payload.toUpperCase() | file

脚本处理器还允许您执行定制的Groovy代码。

水龙头


点击可让您“收听”另一流中的数据，并在单独的流中处理数据。类似于电话窃听，原始流不受水龙头的影响，并且不知道其存在。WireTaps是EAI模式的标准目录的一部分，并且是 Spring XD使用的Spring Integration框架的一部分 。


抽头可以从目标流的处理管道中的任何点消耗数据。例如，如果您有一个名为mystream的流，则定义为

source | filter | transform | sink

您可以使用DSL创建水龙头

tap mystream.filter | sink2

这将在应用滤波器之后但在变压器之前利用流的数据。因此，未转换的数据将被发送到接收器2。

例如，如果使用以下命令创建名为httpstream的流：

curl  -d "http --port=9898 | filter --expression='payload.length() > 5'
                           | transform --expression=payload.toUpperCase()
                           | file"  http://localhost:8080/streams/httpstream

然后使用以下命令在名为httptap的流上创建一个水龙头，将过滤后的数据流写入一个单独的文件：

curl -d "tap httpstream.filter | file --dir=/tmp --name=filtered.txt" http://localhost:8080/streams/httptap

发布数据，例如

curl -d "hello world" http://localhost:9898
curl -d "he" http://localhost:9898
curl -d "hello world 2" http://localhost:9898

将导致与HELLO WORLD和HELLO WORLD 2中的文件/tmp/xd/output/httpstream和降低套管当量/tmp/filtered.txt 。文本'he'将不会出现在两个文件中。

主要用例是在通过主要流提取数据的同时执行实时分析。例如，考虑消耗Twitter搜索结果并将其写入HDFS的数据流。可以在将数据写入HDFS之前创建分接头，然后将数据从分接头传送到计数器，该数据与推文中提及特定主题标签的次数相对应。

分析工具


询问10个开发人员什么是“实时分析”，您将获得20个答案。答案从非常简单（但非常有用）的计数器到移动平均值，汇总计数器，直方图，时间序列，机器学习算法再到嵌入式CEP引擎。Spring XD打算作为与多种后端存储技术一起使用的通用类库，支持这些指标和分析数据结构的广泛范围。它们也作为Xink的一种接收器暴露于XD中，以用于DSL表达式中。


在M1版本中，支持计数器，字段值计数器，量规和富量规。这些指标可以存储在内存中或Redis中。有关更多详细信息，请参见用户指南的JavaDocs和Analytics（分析）部分 ，以及将来版本中将实现的内容的列表。


例如，考虑收集一条推文流中主题标签频率的实时计数的情况。为此，请使用SpringXD创建一个新的流定义，该定义使用Twitter搜索源模块并将其命名为“ spring”

curl -d "twittersearch --query='spring' --consumerKey=<consumer-key> --consumerSecret=<consumer-secret>
           | file" http://localhost:8080/streams/spring

这会将推文存储在本地文件系统中。注意，要获取consumerKey和consumerSecret，您需要注册一个Twitter应用程序。如果尚未设置，则可以在Twitter Developers站点上创建一个应用程序以获取这些凭据。

接下来，在twittersearch源的输出上创建一个名为“ springtap”的水龙头，以计算推文中主题标签的频率。

curl -d "tap spring.twittersearch | field-value-counter
                                     --fieldName=entities.hashTags.text
                                     --counterName=hashTagFrequency" http://localhost:8080/streams/springtap

领域entities.hashTags.text是通向在的JSON表示的主题标签弹簧社会资料Tweet底层实现使用的对象。要查看前5个主题标签，请使用redis-cli查看名为fieldvaluecounters.hashTagFrequency.的排序集的内容fieldvaluecounters.hashTagFrequency.请注意，收集具有主题标签实体的足够的推文通常需要几分钟。

> redis-cli

建筑

Spring XD具有两种操作模式-单节点和分布式。第一个是处理所有处理和管理的单一过程。此模式可帮助您轻松入门，并简化了应用程序的开发和测试。分布式模式允许将处理任务分散在一组计算机上，并且管理服务器发送命令以控制在群集上执行的处理任务。

M1版本中的分布式体系结构很简单。流的每个部分（称为模块）都可以在其自己的容器实例中执行。使用Redis队列在模块之间传递数据。有关更多详细信息，请参见“ 体系结构”部分 。此版本的主要重点是正确实现抽象，例如使DSL中的管道符号可跨各种传输插入。其他传输和性能改进将在将来的版本中以及在Hadoop群集中执行时出现。

还有更多


这篇文章中未涉及的其他一些主题是Tuple数据结构的介绍以及如何创建自定义处理器 。下一个版本的重要部分将是对XDContainer的支持，以运行Spring Batch作业。这些作业可用于帮助将数据从HDFS导出到关系数据库，以及协调群集上MapReduce，Pig，Hive或Cascading作业等Hadoop作业的执行。我们还将为度量标准提供其他库，例如聚合计数器，基于HTTP / JMX的管理以及基于Reactor项目的一些高性能源，敬请期待！


我们将继续努力争取最终的Spring XD 1.0.0版本，我们希望能收到您的反馈。如有任何疑问，请使用Stackoverflow （标记： springxd ），并报告任何错误或改进，请使用Jira Issue Tracker或提交GitHub问题 。
