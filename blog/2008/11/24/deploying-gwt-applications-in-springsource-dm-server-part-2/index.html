在SpringSource dm Server中部署GWT应用程序-第2部分

工程
本·科里
2008年11月24日

介绍

这是三个博客系列文章中的第二篇，它描述了在SpringSource dm Server™中逐步构建和部署GWT应用程序的方法。第一个博客介绍了从示例GWT应用程序创建简单WAR文件的过程。下一个博客将探讨如何将我们在第1部分中创建的WAR文件转换为“共享库” WAR。这意味着我们将把应用程序的GWT依赖关系外部化为OSGi捆绑包，以便可以被任意数量的GWT应用程序共享。您可以将其视为具有GWT远程处理功能的dm服务器扩展。

如第1部分中所述 ，我在第二篇博文中没有使用Spring Framework ，而是专注于SpringSource dm Server™和SpringSource Tool Suite来部署“纯” GWT。

另请参阅第1部分 ，以获得GWT StockWatcher示例的背景以及我使用的软件。

快速赶上


在第1部分中 ，我们从头开始将GWT StockWatcher示例应用程序构建为Eclipse项目，然后将代码生成到Dynamic Web项目中，然后将其部署到dm Server中。最后，我们将Dynamic Web项目导出到WAR文件中，并将其部署在STS之外。

这里描述的逐步方法将基于我们在第1部分中所做的工作，而不是从头开始。我们现在要更改的第1部分中唯一要做的就是删除对gwt-servlet.jar库的显式依赖。

步骤1：将我们的GWT依赖项转换为OSGi捆绑包


首先，多一点背景。“共享库”方法的整个概念是使用OSGi捆绑软件之间的显式导入和导出在dm Server中创建依赖关系图。使用诸如我们的StockWatcher样本之类的小WAR，这基本上只是一个有趣的学术练习。但是，鉴于许多商业Web项目都以大型WAR文件提供，这些WAR文件与数十个甚至数百个相关的jar文件打包在一起，因此从足迹的角度出发，将这些依赖关系分解为可共享的资源不仅有意义，而且还可以进行打包，版本控制和维护应用程序明显减少了痛苦。

好消息是，创建这些依赖项的大量工作已经为您完成。SpringSource Enterprise Bundle存储库包含大多数常用库的“捆绑”版本。但是，在撰写本文时，我们的GWT依赖项是一个库的示例，您必须自己将其打包成一个包。对我们来说幸运的是，Eclipse 3.4使过程非常简单。

-右键单击Project Explorer，然后选择“新建”->“其他”->“插件开发”->“从现有Jar存档中插件”。
-点击“添加外部”，然后浏览gwt-servlet.jar 。
-点击“下一步”，并将项目命名为“ com.google.gwt”。
-设置插件版本以反映GWT版本，在本例中为1.5.3。
-选择“分析库内容并添加依赖项”
-选择Equinox OSGi框架

单击“完成”，不必再切换到“插件开发”透视图。我们将再次直接导出捆绑包。

现在，您应该看到生成的MANIFEST.MF文件的概述，在该文件中定义了捆绑包的依赖关系。您会在“运行时”标签中看到该工具已将所有com.google.gwt ..软件包添加为导出。您还将在“ Dependencies”选项卡中看到，已经确定该捆绑包需要几个javax.servlet ..包和一个junit包。

我们现在将删除JUnit依赖项，因为在dm Server中默认情况下将不解决它。当然，如果我们愿意，我们可以向dm Server添加一个JUnit捆绑软件来满足依赖性，或者可以通过添加required：= optional来使依赖性成为可选 。选择junit.framework包，单击“删除”，然后保存。

值得指出的是，这是将JAR文件转换为捆绑包的粗略方法，并且不能保证在所有情况下都能正常工作。一方面，我们可能不希望导出所有软件包。但是，更重要的是，有一个或两个源级别的陷阱可能无法在OSGi中很好地发挥作用，例如使用Class.forName（） 。如有疑问，请始终先访问SpringSource Enterprise Bundle信息库，而不要尝试自己动手做。

最后，我们需要将插件项目导出为OSGi捆绑包。选择“导出”->“插件开发”->“可部署的插件和片段”。选择一个方便的位置作为输出目录。

现在，我们应该在/plugins/com.google.gwt_1.5.3.jar中看到一个文件。下一步是重命名文件，使其与dm Server中其他命名捆绑软件的格式一致：只需将下划线更改为破折号： com.google.gwt-1.5.3.jar 。如果不进行此更改，STS的当前版本将无法识别捆绑软件版本。如果您想跳过此步骤，可以在此处下载。

最后，将捆绑软件复制到dm Server捆绑软件存储库： / repository / bundles / usr 。此处的任何捆绑软件都可以是dm Server中其他捆绑软件的依赖项，但重要的是，它也将在STS中显示为依赖项，正如我们将在下一步中看到的那样。

此时，您可以从com.google.gwt项目已达到其目的的工作空间中删除它。

步骤2：将我们的WAR项目迁移到新的OSGi捆绑包


希望您现在应该安装dm服务器， 并在其存储库中使用com.google.gwt-1.5.3.jar 。

下一步是破坏我们的应用程序！

右键单击StockWatcherWar，选择“属性”->“ Java EE模块依赖项”，取消选择gwt-server.jar文件，然后单击“确定”。现在，我们消除了WAR对GWT的明确依赖，并给自己带来了许多新问题，其中大多数是编译器错误。

因此，我们需要将Dynamic Web项目转换为dm Server捆绑软件，以便它可以从新创建的GWT捆绑软件中显式导入所需的依赖项。为此，请在StockWatcherWar上单击鼠标右键，然后选择“ Spring Tools”->“ Add OSGi Bundle Project Nature”。您应该注意到，该项目现在带有一个非常重要的“ S”符号，并且MANIFEST.MF也更改了字符：

它被破坏是因为它看起来不像捆绑清单。让我们添加一些合理的默认值（您可以剪切并粘贴以下内容，或使用选项卡中的字段）。请注意，您也可以在编辑器中使用ctrl-space来提示您有效的选项。


清单版本：1.0
Bundle-ManifestVersion：2
捆绑名称：com.google.sample.stockwatcher
Bundle-SymbolicName：com.google.sample.stockwatcher
捆绑版本：1.0.0
捆绑包描述：共享库StockWatcher演示

因此，这解决了一个问题，因为我们的捆绑清单现在看起来很健全。但是，我们尚未定义对GWT包的依赖关系，因此我们仍然遇到许多编译器错误。

选择“依赖关系”选项卡，然后单击“添加”。在顶部，您应该看到com.google.gwt捆绑包已经神奇地出现了。现在，重要的是要了解此列表中显示的GWT包与我们在步骤1中创建的插件项目无关。如果您从工作空间中删除该项目，该捆绑包仍将出现。它之所以有效，是因为dm Server安装被设置为项目的运行时，因此该工具正在dm Server的存储库中查找您可能要导入的任何捆绑包。如果这不起作用，则可能是GWT捆绑包未复制到正确的位置，或者WAR项目的运行时未设置为dm Server。在“属性”->“目标运行时间”中检查后者。

接下来，选择GWT捆绑包，单击“确定”，然后保存清单。现在，您应该看到GWT包已添加到名为“ Bundle Dependencies”的新类路径元素中：

您现在应该注意到，大多数错误已得到解决，但是仍然存在一些错误。这是因为我们的GWT捆绑包依赖于javax.servlet API，并且在我们拥有这些可用类之前，类型层次结构是不完整的。

那么，为什么在构建路径中只有gwt-servlet.jar时我们以前没有这个问题呢？好吧，在进行更改之前，WAR将在其构建路径中获取所有内容，其中包括dm Server Target Runtime提供的所有JAR。但是，您可能已经注意到，当我们添加OSGi Bundle Project Nature时，构建路径上的dm Server JAR消失了。这是因为，通过成为OSGi捆绑包，我们进入了一个不同的依赖世界，其中必须显式声明导入和导出。

声明我们对javax.servlet API的依赖是导入另一个捆绑包的简单问题。您将在列表中以com.springsource.javax.servlet的形式查看它。添加两个捆绑包后，保存清单，现在应该可以解决所有错误。！

如果现在导航至MANIFEST.MF选项卡，则可以看到所有按钮按下的效果：清单中绿色的“导入包”条目。注意，我们可以在“ Import-Bundle”之后轻松地使用ctrl-space，并且会建议所有可能的导入：

如果要查看我的项目，可以在此处下载压缩的副本。它包括运行时配置，以使用嵌入式Tomcat在托管模式下启动，在dm Server托管模式下启动以及启动GWT编译器。请注意，嵌入式Tomcat的托管模式可以正常工作，而无需修改捆绑软件WAR项目。这是因为dm Server中的javax.servlet包不再位于构建路径上。要使用我的项目，您将需要GWT_ROOT_INSTALL变量，并且可能需要选择dm Server Target Runtime实例，如第1部分中所述。

步骤3：部署到dm服务器


部署“共享库”应用程序的步骤与第1部分中的步骤7和8完全相同，因此没有必要再次详细介绍这些步骤。当然，主要的区别在于我们已经成功地将依赖关系分解为一个库，该库可以由多个GWT应用程序共享。如果下载我的StockWatcher共享库WAR ，您会发现它现在是290k而不是890k。

只是为了好玩，让我们看一看幕后花絮，看看我们的互动包。

dm Server启动时，它会通知您可以使用telnet客户端访问其OSGi控制台：

步骤4：在共享库上部署其他GWT应用程序


值得注意的是，并非所有GWT应用程序都需要使用gwt-servlet.jar 。只有使用某种形式的远程处理的GWT应用程序才对此库具有依赖性。Google在其发行版中提供了许多示例应用程序，其中大多数仅生成为javascript和html，因此无需使用Java代码即可进行部署。将这些示例转换为WAR文件很简单，只需按照第1部分第5步中所述将Compile脚本运行到Dynamic Web项目中，然后创建合适的web.xml。

但是，GWT分发中包含的确实使用一些简单远程处理的一个示例称为DynaTable。使用这些博客中描述的相同步骤和原则，我将此示例转换为Shared Libraries WAR文件。要查看我做了什么，可以下载压缩的项目或WAR文件并进行查看。这非常简单地演示了在dm Server中运行多个应用程序的原理，这两个应用程序都共享我们创建的GWT包。
