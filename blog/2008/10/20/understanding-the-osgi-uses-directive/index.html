了解OSGi使用指令

工程
格林·诺明顿
2008年10月20日

如果您为SpringSource dm Server或任何其他OSGi平台构建应用程序，则可能不久就会遇到using指令。除非您对指令的用途有清楚的了解，否则您将不知道何时编写该指令，并且您将只能猜测由于使用冲突而导致束无法解决的时间。这篇文章应该给你的使用指令，当使用它，以及如何调试用途冲突的全面理解。

捆绑分辨率

OSGi的设计宗旨是，一旦“解决”了捆绑包，通常就不应该遇到由于类型不匹配而导致的类强制转换异常和类似问题。这是非常重要的，因为OSGi为每个捆绑软件使用了一个类加载器，因此有足够的机会使用户暴露于各种类型的不匹配。

必须牢记任何Java类型（例如类或接口）都必须由类装入器装入，然后才能在运行时使用。运行时类型是由该类型的完全限定的类名称和定义该类型的类加载器的组合来定义的。如果两个不同的类加载器定义了相同的完全限定的类名称，则这将产生两种不兼容的运行时类型。如果两种类型接触，这种不兼容性会导致运行时错误。例如，尝试将这些类型之一转换为另一类型将导致类转换异常。

OSGi不是唯一的基于类加载器的Java模块系统，但它是迄今为止最成熟的。这意味着OSGi的设计者已经对这些问题进行了长时间的认真思考，并将解决方案包括在OSGi规范中。OSGi的设计是在称为解决方案的过程中，在应用程序代码运行之前解决这些问题。解决方案类似于使用强类型编程语言（例如Java）进行的编译，因为在应用程序代码开始运行之前，已经对某些类别的问题进行了诊断。有时，让您的捆绑包解决可能会有些令人头疼，但是它比诊断运行时错误（例如类强制转换异常）要好。

那么解决捆绑商品意味着什么？这意味着满足捆绑包的依赖性，通常是通过找到捆绑包来导出给定捆绑包导入的包并满足各种约束。最明显的约束是，每个导出的软件包版本应位于软件包导入的软件包版本范围内。另一种约束是可以在包导入上指定任意属性，然后必须在相应的包导出上匹配属性。

多个捆绑出口的包裹

使用约束（我们将很快解决）旨在消除由多个捆绑包导出的软件包引起的类型不匹配的类别。当使用一个捆绑软件中的类型而又需要另一个捆绑软件中的类型时，会发生类型不匹配，因为运行时类型不兼容。例如，当尝试使用另一个包中具有相同类名的不同类型来尝试从一个包中转换类型时，将发生类转换异常。怎么会这样由于捆绑包不能从一个以上的捆绑包中导入同一软件包，因此必须使用其他方式来联系冲突类型。这是通过将类型“传递”到另一个包中的类型而发生的。

一种类型可以通过另一种类型传递。第一种方法是一种类型明确地引用另一种类型。例如，在包org.bar 律师类型的方法可以是指在org.foo封装的Foo类型：

公共Foo getFoo（）;

一个类型可以通过另一个类型传递的第二种方式是通过超类型隐式传递。例如，方法的签名可以引用超类型：

公共对象getFoo（）;

在隐式情况下，超类型的实例将在某个时候转换为冲突类型。

这就是在Java代码级别上发生这种类型不匹配的方式。让我们考虑一下捆绑包的外观。

所需的Foo类型可以由与导出org.bar包相同的捆绑包（ B ）导出：

bundle-symbolicname：B
bundle-manifestversion：2
出口包装：org.foo，org.bar

或通过另一个束（ F ）：

bundle-symbolicname：B
bundle-manifestversion：2
出口包装：org.bar
导入包：org.foo


bundle-symbolicname：F
bundle-manifestversion：2
出口包装：org.foo

引入了uses指令，以便OSGi可以在分发包解析期间诊断上述类型的不匹配。

用途指令

为了在解析期间检测到上述类型的潜在类型不匹配，在相应的包清单中声明了Java代码级别的显式或隐式类型引用。包含引用类型的包的导出标记有一个use指令，该指令声明所引用类型的包。

在上面的示例中， org.bar包的导出被声明为使用 org.foo包：

…
出口包装：org.bar;用途：=“ org.foo”
…

请注意， 使用指令中指定的一个或多个程序包是由包含使用指令的捆绑清单导出或导入的。所以以下清单
已验证：

…
出口包装：p；用途：=“ q，r”，q
导入包：r
…

而以下清单无效（因为它既不导出也不导入包q ）：

…
出口包装：p；用途：=“ q，r”
导入包：r
…

传递性用途

类型引用是可传递的。例如，如果类型A引用类型B ，而类型B引用另一个类型C ，则A的用户可以通过B获得对C的引用。

由于类型引用是可传递的，因此OSGi会自动考虑这一点。它形成了uses指令的可传递闭包 。这意味着为每个类型引用编写一个use指令就足够了，OSGi将负责传递类型引用。

例如，尽管捆绑包体现为：

…
出口包装：p;用途：=“ q，r”，q;用途：=“ r”
导入包：r
…

是正确的，以下捆绑清单足以捕获从包p到q ，从q到r以及（从）从p到r的类型引用：

…
出口包装：p;用途：=“ q”，q;用途：=“ r”
导入包：r
…

诊断使用冲突

捆绑包解析过程旨在满足所有约束，因此，如果捆绑解决方案不能满足所有使用约束的方式满足依赖关系，则它将仅报告使用冲突。在这些情况下，SpringSource dm Server发出的诊断有助于解决问题。

让我们看一个虚构的例子来理解原理。假设我们正在开发两个实用程序捆绑包F和B，它们将从客户端捆绑包C中调用。假设我们引入了F的第二个版本，并尝试在服务器上部署具有以下清单的捆绑包。

清单版本：1.0
Bundle-ManifestVersion：2
Bundle-SymbolicName：F
捆绑版本：1
捆绑包名称：F Bundle
导出包：org.foo; version = 1


清单版本：1.0
Bundle-ManifestVersion：2
Bundle-SymbolicName：F
捆绑版本：2
捆绑包名称：F Bundle
导出包：org.foo; version = 2


清单版本：1.0
Bundle-ManifestVersion：2
Bundle-SymbolicName：B
捆绑版本：1
捆绑包名称：B捆绑包
导出包：org.bar;用途：=“ org.foo”
导入包：org.foo; version =“ [1,2）”


清单版本：1.0
Bundle-ManifestVersion：2
Bundle-SymbolicName：C
捆绑版本：1.0.0
捆绑名称：C Bundle
导入包：org.bar，org.foo; version =“ [2,3）”

当我们尝试部署捆绑软件C时，dm Server发出以下日志消息：

无法从位置“文件：/xxx/C.jar/”安装应用程序。无法满足版本“ 1.0.0”中束“ C”的约束。
无法解决：C
解析器报告：
捆绑：C_1.0.0-使用冲突：导入包：org.bar;版本=“ 0.0.0”
可能的供应商：B_1.0.0-出口包装：org.bar;版本=“ 0.0.0”
可能的冲突：org.foo
。

该行：

捆绑：C_1.0.0-使用冲突：导入包：org.bar;版本=“ 0.0.0”

告诉我们，与捆绑软件C导入org.bar包有关的使用约束冲突。换句话说， C尝试使用的org.bar导出具有无法满足的使用约束。

该行：

可能的供应商：B_1.0.0-出口包装：org.bar;版本=“ 0.0.0”

告诉我们正在考虑哪个org.bar供应商。

该行：

可能的冲突：org.foo

告诉我们哪个包装违反了使用限制。

从细节退一步，我们知道使用冲突是由于包C导入的软件包org.foo与包B导入的版本不同。某种原因必须导致使用不同版本，并且当我们仔细检查软件包的导入时，我们意识到我们忘记升级B来使用F的最新版本。

将B的清单更新为：

清单版本：1.0
Bundle-ManifestVersion：2
Bundle-SymbolicName：B
捆绑版本：1
捆绑包名称：B捆绑包
导出包：org.bar;用途：=“ org.foo”
导入包：org.foo; version =“ [2,3）”

我们现在可以成功部署捆绑软件C。

诊断复杂的使用冲突

我们的化妆使用冲突非常简单，以至于我们可以长时间凝视各种捆绑物清单来发现问题。对于更复杂的使用冲突，例如，在uses指令中列出了许多可能相互冲突的软件包时，您可以通过使用Equinox控制台（ 远程登录到端口2401）检查已成功安装的捆绑包（dm）来加快进度。服务器将卸载无法成功部署的所有捆绑软件。

您可以通过发出以下Equinox控制台命令来获取已安装捆绑软件的列表：

osgi> ss

在我们的伪造问题中，它显示如下内容：

…
82有效F_1.0.0
84有效F_2.0.0
85个活动B_1.0.0

要显示B问题的捆绑清单：

osgi>标头85

并显示org.foo软件包的进口商和出口商：

osgi>包org.foo

摘要

本文探讨了对uses指令的需求，如何使用它来提供对某些类型的类型不匹配错误的早期诊断，以及如何使用dm Server诊断和Equinox控制台深入了解使用约束冲突。

你可能会觉得用途指令是更多的麻烦比它的价值，但是当你考虑你的应用程序运行时追查原因可能是一个不起眼的类转换异常的选择，你应该开始看到用途的理由。实际上，一旦达到了拥有多个版本的应用程序模块的地步，任何不提供等效使用限制的Java模块系统都可能在运行时为您提供类强制转换异常。OSGi的使用指令解决了的Java模块化的一个普遍问题。
