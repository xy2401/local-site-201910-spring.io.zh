Spring动态模块达到1.0！

工程
阿德里安·科耶尔（Adrian Colyer）
2008年1月25日

嗯，它花了比我们最初预期的时间长得多的时间，但是我很高兴地说Spring Dynamic Modules项目今天达到了其1.0里程碑。当我在2006年9月首次发布有关该主题的文章时（“ Spring OSGi支持不断增强 ”），最初的规范只是针对Spring Framework 问题的附件，并且与更广泛的OSGi社区的联系才刚刚形成。 。

快进了18个月， Spring Dynamic Modules已成为Spring产品组合中的一个完整项目，来自SpringSource，BEA和Oracle的提交者。BEA和Oracle都在使用Spring Dynamic Modules来构建自己的基于OSGi的产品（例如，参见“ WebLogic Event Server-为什么使用Spring ”），并且Spring Dynamic Modules讨论组有近1000个成员。OSGi联盟本身已经成立了企业专家组 ，而SpringSource是活跃的成员。

每当我们在会议上谈论OSGi和Spring动态模块时，人们的兴趣就很高。那么，大惊小怪的是什么？

OSGi服务平台是经过验证的轻量级运行时，提供了“ Java动态模块系统”。从嵌入式设备，汽车和电信应用程序到IBM，BEA和Oracle等大量企业中间件产品的基础，它已被广泛使用。Spring Dynamic Modules致力于使您能够在构建自己的企业应用程序时利用OSGi服务平台 。使用Spring动态模块构建并在OSGi服务平台上运行的应用程序由一组对等的Spring应用程序上下文（每个OSGi捆绑包）组成，这些上下文通过OSGi Service Registry进行交互。它是一种面向服务的体系结构，但是使用常规的Java引用访问服务，因此完全在VM内部。以这种方式构造的应用程序的好处包括：


  • 强大的模块化性：每个模块的内部组件都是私有的，只有您选择导出到模块外部的包和bean（服务）才可公开供其他模块使用。

  • 动态模块生命周期：您可以使用标准OSGi服务平台生命周期工具独立安装，卸载，启动，停止和更新每个模块（应用程序上下文）。Spring Dynamic Modules提供了对跨越模块边界的服务引用的智能管理，因此即使提供该服务的模块已就位更新，您也可以继续使用恒定引用。

  • 支持模块版本控制。可以同时部署同一Java包（以及库）的多个版本。OSGi平台负责将这些库的用户绑定到适当的版本，并确保您仅看到实现版本兼容接口的服务。

  • 对已安装模块，它们导入和导出的服务和软件包以及如何在运行时将模块连接在一起的操作进行洞察。


在其他模块保持运行状态的同时就地更新模块的能力可以极大地提高开发效率。

在1.0版本中，我们为模块生命周期和服务集成提供了可靠的解决方案。当使用OSGi Service Platform上的现有企业库（不是编写要在OSGi上运行）时，对于类和资源加载以及可见性，仍然存在一些陷阱。我们相信我们已经在最近发布的Spring Framework 2.5中为Spring本身解决了所有这些问题。Spring Dynamic Modules下一版本的重点将在于为其他常用企业库提供实用的解决方案，以使编写在OSGi上运行Spring的Web应用程序变得更加容易。
