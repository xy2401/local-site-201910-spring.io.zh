Spring Dynamic Modules 1.0在这里

工程
Costin Leau
2008年1月25日

我很高兴地报告（与Adrian一起 ），在3个里程碑和2个候选发布版本之后， Spring Dynamic Modules （以前称为Spring OSGi）1.0已发布 。

自从我上一篇文章 （大约1.0 M1）以来，已经改进或添加了许多功能；我将在以后的文章中进一步讨论它们（也有参考文档详细解释了该库），所以我仅举几例：

-一致性

我们希望提供一个功能强大，简单且一致的编程模型。这就是为什么Spring Dynamic Modules建立在Spring之上并使用其可靠的概念，可靠性和普遍性的原因。

-高度非侵入性

推荐的使用Spring DM的方法是不要在代码中使用其类，也不要在包清单中包含任何此类的导入。如果您不是在代码内部使用Spring，而是仅用于应用程序配置，则适用相同的规则。Spring DM为您创建了应用程序上下文，因此您无需依赖Spring或Spring DM。而且不用担心诸如自定义名称空间或XML模式之类的东西-我们已经介绍了它们。

-OSGi服务动态生命周期管理

这是Spring DM最重要的功能之一-能够像使用普通 bean一样与OSGi服务进行交互。您无需编写任何代码即可发布和使用OSGi服务。我们将为您处理动态-您将拥有完全的控制权（将来会对此有更多的了解）。

-更智能的集成测试框架

由于我们在内部广泛使用了Spring-DM集成测试，因此我们改进了默认设置，maven集成，并使自动清单生成比以前更快，更智能。例如，框架自动确定测试包中可用的类，并且不会为其生成导入。

-简单的捆绑互动

Andy Piper（ 博客 ）基于模块生命周期和Spring Bean依赖性添加了一种简单的声明式安装/启动/停止/更新捆绑包的方法。

-托管的启动/关闭上下文创建

在OSGi中，应用程序分为相互依赖的各种模块（也称为捆绑包）。这将在模块之间创建一个依赖关系树，这在启动和关闭过程中变得很重要。传统上，这可以通过根据依赖关系顺序安装和启动捆绑包来解决，但是，这不能完全解决问题。如OSGi规范所建议，初始化时间较长的OSGi服务（例如连接池）应依赖于不同于用于启动和停止捆绑软件的线程的线程。这意味着，如果启动捆绑软件，则并不意味着其服务已启动。并不是每个应用程序都准备好在启动过程中等待其所需的服务-实际上，很少。这意味着捆绑软件将失败，因为它依赖于几毫秒后发布的服务（默认情况下，OSGi是VM平台，事情发生的速度非常快）。

这种行为并不罕见-实际上，在具有多个捆绑包的多核平台上启动时非常普遍。Spring DM通过确定依赖关系（通过Spring配置）并在创建应用程序上下文之前等待它们变得可用来解决此问题。在关闭时，Spring DM将根据上下文的依赖关系顺序停止上下文，因此将使用类似的过程，因此您不必担心启动或停止捆绑软件。

-无线程依赖关系等待

我不能不讨论依赖机制，而不必提及由Hal Hildebrand实现的用于等待依赖的“无线程”方法（我知道这听起来有点像矛盾，我们正在为其设计一个漂亮的标题）（请参阅他的博客 ） 。由于模块需要各种服务才能正常启动，因此需要某种等待/监视，这通常意味着使用线程。

但是，在OSGi平台上可以有（并且将会有）多个模块（很容易地有几十个模块）-每个模块使用等待线程根本无法扩展。我们努力工作的一件事是改进此模型，我相信我们提供了一个非常好的解决方案-在等待过程中完全不使用线程。这意味着无论部署了3个捆绑包还是300个捆绑包，除非实际启动模块，否则都不会花费CPU时间。

 

Spring Dynamic Modules不仅涉及“ Spring”化的API，还涉及不同的运行时环境。

 

关于工具，Spring IDE支持Spring DM名称空间，并且（感谢Christian ）还为Eclipse PDE提供了特定于Spring-DM的目标 ，这是Spring IDE夜间构建中可用的功能（有关安装和使用插件的更多信息， 请参见参考文档）。

 

未来发展方向

 

那么，现在已经发布了1.0，下一步是什么？涉及的领域很多：

网路支援

OSGi平台提供了专用的Http服务，但使用它需要编码。资源加载，JSP生成和部署之类的事情可以大大简化。这是1.1版本的主要焦点。

坚持不懈

现代的持久性工具提供了诸如延迟加载之类的高级功能，这些功能依靠类生成和代理来改变由OSGi环境强制执行的模块化边界。我们希望解决此问题，并且像使用Web支持一样，无论使用简单的JDBC或/和ORM工具，都可以提供流畅的体验。

行动计划

继持久性问题之后，我们正在寻求在OSGi中执行通用AOP的解决方案。要破解并正确做到这一点很难，需要内部OSGi平台支持。好消息是，像Equinox Aspects之类的项目已经处于领先地位 ，而OSGi企业专家组 （EEG）则将这个问题引起关注。
