域对象依赖注入功能的新改进

工程
拉姆尼瓦斯·拉达德（Ramnivas Laddad）
2008年1月24日

Spring的依赖项注入（DI）机制允许配置在应用程序上下文中定义的bean。如果您想将相同的想法扩展到非bean，该怎么办？Spring对领域对象DI的支持利用AspectJ编织将DI扩展到任何对象，即使它是由Web或ORM框架创建的。由于域对象现在可以与注入的对象协作，因此可以创建域行为丰富的对象。在此博客中，我将讨论该领域中Spring Framework 的最新改进。

域对象DI背后的核心思想非常简单：AspectJ编织的方面选择与创建或反序列化符合特定规范的任何对象相对应的连接点。这些连接点的建议将依赖项注入正在创建或反序列化的对象中。当然，细节在于魔鬼。例如，如何选择与反序列化相对应的连接点，或者如何对每个对象仅注入一次依赖关系？通过提供一些预先编写的方面，Spring使开发人员免受所有这些细节的影响。

当前，大多数Spring用户使用@Configurable注释来指定可配置的类。随着即将发布的Spring 2.5.2的最新改进（ 从每晚版本379开始可用） ，您还有更多选择，使此功能更强大。新的改进遵循“使简单事物变得简单，使复杂事物成为可能”的原则。根据您对AspectJ的熟悉程度和期望的设计复杂程度，这些选项之一会很好地起作用。图1显示了新的方面层次结构，它将简单性和灵活性结合在一起。

图1：域对象依赖项注入方面的继承层次结构。

那么这些方面都提供了什么？让我们自下而上。


简单的事情很简单：AnnotationBeanConfigurerAspect


AnnotationBeanConfigurerAspect启用域对象DI，而无需任何用户AspectJ代码。因此，对于许多开发人员来说，这是最简单的选择。在这方面，您可以使用@Configurable批注对需要依赖项注入的类进行批注。例如，您可以对Order类进行如下注释：


@Configurable
public class Order {
    private transient MailSender mailSender;
    
    public void setMailSender(MailSender mailSender) {
        this.mailSender = mailSender;
    }
    
    public void process() {
        ... 
        mailSender.send(...);
        ...
    }
}

接下来，您指导Spring如何配置Order类型的对象。这些指令遵循原型Bean的标准Bean定义，如下所示：


<context:spring-configured/>
    
<bean class="example.Order" scope="prototype">
    <property name="mailSender" ref="externalMailSender"/>
</bean>
    
<bean id="externalMailSender" ...>
    ...
</bean>

现在，在创建或反序列化任何Order时 ，Spring都将使用externalMailSender bean设置所创建对象的mailSender属性。

Spring 2.5具有一个新的基于注解的配置选项 ，该选项允许消除或减少附带的XML。基于@Configurable注释的DI也从中受益。例如，您可以将mailSender属性标记为@Autowired ，如下所示：


@Configurable
public class Order {
    private transient MailSender mailSender;
    
    @Autowired
    public void setMailSender(MailSender mailSender) {
        this.mailSender = mailSender;
    }
    
    public void process() {
        ... 
        mailSender.send(...);
        ...
    }
}

您可以通过注释字段本身来消除设置器，将上述代码减少为：


@Configurable
public class Order {
    @Autowired private transient MailSender mailSender;
    
    public void process() {
        ... 
        mailSender.send(...);
        ...
    }
}

无论哪种情况，伴随的XML配置都将简化为以下内容（请注意使用 ）：


<context:spring-configured/>
    
<context:annotation-config/>
    
<bean id="externalMailSender" ...>
    ...
</bean>

有关此域对象DI选项的更多详细信息，请参见使用AspectJ通过Spring依赖注入域对象 。


可能的复杂事情：AbstractInterfaceDrivenDependencyInjectionAspect


AnnotationBeanConfigurerAspect的基本方面AbstractInterfaceDrivenDependencyInjectionAspect使用接口而非注释来标记可配置类。尽管看起来像是一个肤浅的变化，但它提供了一些有趣的选项，例如使用域接口和注释来指定依赖项注入，通过绕过反射来提高注入性能以及利用多个方面来配置对象。

在设计级别，此方面将配置其类型实现ConfigurableObject接口的任何域对象。虽然直接使用类型来实现ConfigurableObject接口无疑是一个有效的选择，但一种不错的选择是在另一个方面使用声明父声明（ AbstractInterfaceDrivenDependencyInjectionAspect的一个子方面将是一个逻辑选择）。该语句将声明一个可配置类，以实现ConfigurableObject接口。这使您的域类免受特定于Spring的构件的影响，同时受益于DI机制。让我们看一个这样用法的例子。

考虑上一节中的Order类。除了使用@Configurable之外 ，还可以让它实现特定于域的MailSenderClient接口，该接口表示它使用MailSender 。


public class Order implements MailSenderClient {
    private transient MailSender mailSender;
            
    public void setMailSender(MailSender mailSender) {
        this.mailSender = mailSender;
    }
            
    public void process() {
        ... 
        mailSender.send(...);
        ...
    }
}

接下来，编写一个AbstractInterfaceDrivenDependencyInjectionAspect的子方面，以将依赖项注入到任何MailSenderClient对象中。


public aspect MailClientDependencyInjectionAspect extends 
    AbstractInterfaceDrivenDependencyInjectionAspect {
    private MailSender mailSender;
    
    declare parents: MailSenderClient implements ConfigurableObject;
            
    public pointcut inConfigurableBean() : within(MailSenderClient+);
    
    public void configureBean(Object bean) {
        ((MailSenderClient)bean).setMailSender(this.mailSender);
    }
            
    public void setMailSender(MailSender mailSender) {
        this.mailSender = mailSender;
    }
}

在方面中使用了两种AspectJ构造：


  1. 声明父母声明使MailSenderClient实现ConfigurableObject接口，使其可以通过AbstractInterfaceDrivenDependencyInjectionAspect获得DI。

  2. inConfigurableBean（）仅在MailSenderClient的子类型中选择连接点，因此将Aspect的适用性限制为仅匹配类型。

configureBean（）方法通过直接调用适当的设置器来执行注入到Bean中的操作。当然，适合于bean配置的任何其他逻辑（例如，调用多参数方法或调用任何初始化方法）都可以正常工作。请注意，以这种方式使用的直接调用可以避免反射，并且如果域对象创建的速度很高，则可以显着提高性能。

您需要配置MailClientDependencyInjectionAspect方面实例本身以注入其依赖项– mailSender属性。Spring的方式是为方面创建一个bean，并在应用程序上下文中对其进行配置：


<bean class="example.MailClientDependencyInjectionAspect" 
        factory-method="aspectOf">
    <property name="mailSender" ref="externalMailSender"/>
</bean>
    
<bean id="externalMailSender" ...>
    ...
</bean>

关于这方面还有一些其他模式：


  • 使用多个方面来配置一个对象（例如，每个“客户端”接口一个）。

  • 使用域批注而不是域接口或@Configurable批注来指定可配置的类型。

  • 使用基于hasmethod（）的类型模式（当前是AspectJ 5中的实验性功能，在AspectJ 6中将成为常规功能 ），以避免使用DI相关的类型或注释。

  • 使用基于AspectJ的mixins为客户端接口提供默认实现，并避免重复的设置程序。

但是，让我们将这些想法保存到另一个博客条目中。


需要时的灵活性：AbstractDependencyInjectionAspect


最后，这是最灵活的基本方面。这方面要求您对AspectJ切入点语言有深入的了解。但是，除了极端自定义方案（例如自定义反序列化事件）以外，您不会直接创建此基本方面的子方面。相反，您将使用我们前面讨论的子方面之一。

该方面声明了一个子方面可以定义的六个切入点：


  1. beanConstruction（Object bean） ：选择bean结构。典型的实现将选择对象初始化连接点。

  2. beanDeserialization（Object bean） ：选择bean反序列化。典型的实现将选择必须存在于注入对象中的readResolve（）方法。如果使用的是非标准反序列化（不调用readResolve（） ），则可以使用此切入点选择适当的替代方法。

  3. inConfigurableBean（） ：选择可通过定义方面配置的bean中的连接点。典型的实现将使用具有适当类型模式的inside （）切入点。

  4. preConstructionConfiguration（） ：为需要在构造之前注入依赖项的bean选择连接点。该切入点的默认实现不选择任何连接点（在构造函数运行后，bean将注入依赖项）。

  5. mostSpecificSubTypeConstruction（） ：选择与最具体的子类型相对应的连接点。缺省实现使用连接点签名来确定构造函数是否表示要注入的Bean的类型层次结构中最具体的一个。然后，此信息与preConstructionConfiguration（）切入点结合使用，以在注入依赖项之前或之后使用建议。

  6. minimumSpecificSuperTypeConstruction（） ：选择与最小特定超类型相对应的连接点。

这方面还定义了一个抽象方法configureBean（Object bean） ，其实现应指定与依赖项注入相对应的逻辑。

因此，您有所有选项可以在应用程序中启用域对象DI。如果您对DDD感兴趣，或者需要将DI扩展到域对象，则必须查看这些新方面。根据您的特定需求和AspectJ知识，您会发现其中之一有助于创建优雅的解决方案。
