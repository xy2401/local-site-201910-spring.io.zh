在Spring MVC中使用混合注释和XML方法进行请求映射

工程
罗森·斯托扬切夫（Rossen Stoyanchev）
2008年3月24日

在Spring 2.5中，可以使用批注来配置Web应用程序的所有部分。在Web层中，看到应用的注释特别有趣，在Web层中，开发人员传统上依赖SimpleFormController和MultiActionController进行表单页面处理。注释的引入创建了第三个选项，该选项不需要基类，同时仍提供了先前方法的灵活性。

尽管可以很容易地看到使用带注释的POJO来实现Controller的优雅之处，但是在URL到Controller的映射中，好处并不明显。使用注释定义所有URL映射规则会是什么样？确实，对于Spring MVC应用程序的开发人员来说，集中式配置在这一领域非常有效。

让我们回顾一下Spring 2.0的URL到Controller映射选项：

  1. Bean名称方法（BeanNameUrlHandlerMapping）。每个bean的名称都包含它服务的路径。尽管它很简单，但是如果与粗粒度的servlet映射（例如“ / browse / *”，“ / order / *”，“ / reports / *”等）结合使用，这种方法就可以扩展。
  2. 集中化方法（SimpleUrlHandlerMapping）。查看URL模式和控制器映射的中心位置。
  3. 配置方法上的约定（ClassNameUrlHandlerMapping）。将URL路径与类名匹配。因此，“ / accounts / *”被映射到类型为AccountsController的MultiActionController。不需要显式映射。

Spring 2.5以@RequestMapping注释的形式添加了第四个选项，可以将其放置在类或方法上。当两者都放在方法级映射上时，将缩小类级映射。

这是一个SimpleFormController样式的工作流，其中通过请求方法缩小了方法级别的映射：


@Controller 
@RequestMapping("/editAccount")
public class EditAccountController {

    @RequestMapping(method=RequestMethod.GET)
    public Account setupForm(@RequestParam("id") Long id) {
        ...
        return account;
    }

    @RequestMapping(method=RequestMethod.POST)
    public String processSubmit(Account account) {
        ...
        return "redirect:/success.htm";
    }
}

这是一个MultiActionController样式的委派，其中通过请求方法和相对路径来缩小方法级映射：


@Controller 
@RequestMapping("/accounts/*")
public class AccountsController {

    @RequestMapping(method=RequestMethod.GET)
    public List<Account> list() {...}

    @RequestMapping(method=RequestMethod.GET)
    public Account show(@RequestParam("id") Long id) {...}

    @RequestMapping(method=RequestMethod.POST)
    public String create(Account account) {...}
    ... 
}

如您所见，由于代码中嵌入了“ / accounts / *”映射，因此很难为Controller映射实施应用程序范围的约定。至少并非没有严格的纪律。幸运的是，有一种方法可以将外部HandlerMapping与方法级别的@RequestMapping注释结合在一起。以下是说明此方法如何工作的示例：


<bean class="org.springframework.web.servlet.handler.SimpleUrlHandlerMapping">
    <property name="mappings">
        <value>
            /accounts/*=accountsController
        </value>
    <property>
</bean>


@Controller
public class AccountsController {

    @RequestMapping(method=RequestMethod.GET)
    public List<Account> list() {...}

    @RequestMapping(method=RequestMethod.GET)
    public Account show(@RequestParam("id") Long id) {...}

    @RequestMapping(method=RequestMethod.POST)
    public String create(Account account) {...}
    ... 
}

此处，控制器映射位于基于XML的中央映射中，而操作方法映射则通过注释指定。这种方法可以描述为具有基于注释的方法分派的POJO MultiActionController。实际上，在SpringSource的最近一次通信中，Juergen指出，为MultiActionController提供基于注释的替代方法是一个明确的Spring 2.5设计目标，因此我想在那里并不奇怪！此外，正在进行的工作允许您将方法级别的@RequestMapping批注与ControllerClassNameHandlerMapping约定结合起来（请参阅SPR-4129 ）。

那么这一切的意义何在？

一种基于XML的用于配置Web层的方法可能会比较吵杂，但集中化的外部化配置确实有其用途。从具有深层继承层次结构的特定于框架的基类扩展来实现控制逻辑也会带来噪音，我们通常认为，如果可以帮助的话，应该避免这样做。在Spring MVC 2.5中，注释可以通过将方法映射规则封装在Controller类中来解决这些问题，并且还允许您将Controllers实现为POJO。此外，上述混合方法显示了如何充分利用外部化配置和基于注解的配置。
