Spring Java配置-M3中的新增功能

工程
克里斯·比姆斯
2008年3月27日

今天是Spring Java Configuration项目 （简称JavaConfig）的第三个里程碑版本。该版本包含许多错误修复和新功能-我将在下面重点介绍一些最有趣的更改，但首先让我快速回顾一下JavaConfig的全部含义。

如果您有使用Spring的经验，那么以下XML配置片段可能会很熟悉。假设我们正在查看一个名为application-config.xml的文件：

当然，此XML配置最终将用作Spring ApplicationContext实例化和配置我们的bean的一组指令：


ApplicationContext ctx = new ClassPathXmlApplicationContext("application-config.xml");
OrderService orderService = (OrderService) ctx.getBean("orderService");

JavaConfig只是提供了另一种配置Spring IoC容器的机制，这次是使用纯Java，而不是要求XML来完成工作。让我们将上面的配置移植到JavaConfig：


@Configuration
public class ApplicationConfig {
	public @Bean OrderService orderService() {
		return new OrderService(orderRepository());
	}

	public @Bean OrderRepository orderRepository() {
		return new OrderRepository(dataSource());
	}

	public @Bean DataSource dataSource() {
		// instantiate and return an new DataSource ...
	}
}

与原始XML文件一样，此类仅是关于如何构造应用程序的各种组件的一组指令。我们将这些指令提供给ApplicationContext实现，该实现专门用于读取和执行基于Java的配置指令：


JavaConfigApplicationContext ctx = new JavaConfigApplicationContext(ApplicationConfig.class);
OrderService orderService = ctx.getBean(OrderService.class);

就是这样！好吧，差不多。当然，JavaConfig还有很多东西，但是大多数情况下，功能集是1：1，与Spring的XML配置中可用的功能相同。有关如何使用JavaConfig的完整详细信息，请参阅参考文档 。如果您不熟悉JavaConfig，请务必查看“ 快速入门”部分。

无论如何，JavaConfig的好处显而易见：


  • 它是纯Java，因此不需要XML

  • 您可以在配置代码中获得面向对象的所有好处

  • 类型安全且易于重构

  • 您仍然可以获得核心Spring IoC容器的全部功能

考虑到这一点，让我们看一下M3版本中的更改内容：

AnnotationApplicationContext已弃用

几乎不是“新功能”，但要提及此更改很重要，因为我下面将讨论的大部分内容都围绕JavaConfigApplicationContext ，后者是AnnotationApplicationContext的后继者。为什么要进行此更改？ AnnotationApplicationContext与Spring 2.5的Annotation-Driven Injection工具造成了重大的命名冲突。JavaConfig提供了一种与注释驱动注入不同的配置方法，因此我们希望通过完全重命名该类来使这种区别清晰可见。在1.0.0.rc1发行版之前， AnnotationApplicationContext将保持弃用状态，届时它将被永久删除。

类型安全性改进

尽管上述JavaConfigApplicationContext的行为在很大程度上与其前身相似，但它还引入了类型安全的getBean（）方法，该方法充分利用了泛型。现在可以使用以下代码（从现在开始，这是与JavaConfig结合使用的首选方法）：


JavaConfigApplicationContext context = new JavaConfigApplicationContext(AppConfig.class);
OrderService orderService = context.getBean(OrderService.class);

妈，不要铸造！也没有基于字符串的查找。当然，这将是一个问题， “如果在上下文中配置了两个或多个OrderService类型的对象，该怎么办？” 这是一种很容易发生的情况，有多种解决方法。为了简短起见，我只请有兴趣的人员阅读参考文档的歧义消除选项部分。

这些类型安全的getBean（）方法也已装配到ConfigurationSupport基类上，从而可以实现以下目的：


@Configuration
public class ApplicationConfig extends ConfigurationSupport {
	public @Bean OrderRepository orderRepository() {
		return new JdbcOrderRepository(this.getBean(DataSource.class));
	}
}
主要文档更新

我们一直在努力使JavaConfig的文档达到与Spring著名的质量相当的水平。如上所示，参考文档以HTML以及PDF格式提供。请注意，本文档也打包为可通过SourceForge获得的常规zip发行版的一部分。与M3中的所有其他功能一样，您对文档的反馈将有助于在项目前进至1.0 GA版本时对其进行改进。

在Web层中支持JavaConfig

在此发行版之前，必须通过XML“引导” JavaConfig，以便与Spring的ContextLoaderListener和DispatcherServlet类结合使用。为了解决此限制，已添加JavaConfigWebApplicationContext 。只需在web.xml中将此类指定为contextClass参数，就可以直接使用@Configuration类：



    
    
        contextClass
        org.springframework.config.java.context.JavaConfigWebApplicationContext
    
    
    
        contextConfigLocation
        example.RootApplicationConfig
    
    
    
        org.springframework.web.context.ContextLoaderListener
    
    
    
        dispatcher-servlet
        org.springframework.web.servlet.DispatcherServlet
        
        
            contextClass
            org.springframework.config.java.context.JavaConfigWebApplicationContext
        
        
        
            contextConfigLocation
            example.web.WebBeansConfig

实际上，很多人可能会希望继续使用XML和JavaConfig的组合，尤其是在Web应用程序中。这种方法仍然可以很好地工作（请参阅结合配置方法 ），但是对于团队而言，重要的是，如果用户愿意，我们可以为他们提供真正的“无XML”方法。这种变化消除了这种可能性。

带有新的@Import批注的模块化改进

很像Spring XML配置的元素，现在可以让一个@Configuration类导入另一个（以及所有其bean定义）：


@Configuration
public class FooConfig {
	public @Bean Foo foo() { ... }
	public @Bean Bar bar() { ... }
}

@Import(FooConfig.class)
@Configuration
public class ApplicationConfig {
	public @Bean ServiceA serviceA() { ... }
}

JavaConfigApplicationContext ctx = new JavaConfigApplicationContext(ApplicationConfig.class);
// foo, bar, and serviceA beans will all be available
ctx.getBean(ServiceA.class); // works
ctx.getBean(Foo.class); // works too

该功能只是提供了另一个工具，可以有效地将@Configuration类模块化。

使用@ExternalValue和@ResourceBundles外部化值

一些人建议在JavaConfig中引入与PropertyPlaceholderConfigurer等效的功能，以便在配置过程中访问属性文件中的值。M3提供了这一点。假设我们有一个典型的数据源 ，需要它的JDBC URL，用户名和密码。通常，这些值存储在属性文件中。对于下面的示例，该属性文件将在com / acme / datasource.properties的类路径中提供。该文件的内容如下：

datasource.url=jdbc:localhost:...
datasource.username=scott
datasource.password=tiger

使用@ResourceBundles和@ExternalValue ，我们现在可以从JavaConfig中访问以下属性：


@Configuration
@ResourceBundles("classpath:/com/acme/datasource")
public abstract class ApplicationConfig {
	public @Bean OrderService orderService() {
		return new OrderServiceImpl(orderRepository());
	}

	public @Bean OrderRepository orderRepository() {
		return new JdbcOrderRepository(dataSource());
	}

	public @Bean DataSource dataSource() {
		return new DriverManagerDataSource(url(), username(), password());
	}

	abstract @ExternalValue("datasource.url") String url();
	abstract @ExternalValue("datasource.username") String username();
	abstract @ExternalValue("datasource.password") String password();
}

这里需要注意几件事：看看@ResourceBundles批注的值如何不以.properties结尾？这是因为JavaConfig在下面使用了Spring的国际化基础结构，并将根据当前语言环境查找datasource.properties的变体，例如datasource_en.properties 。同样，尽管此示例将字符串值提供给@ExternalValue批注，但默认设置是基于方法名称查找属性。因此，如果我们没有提供@ExternalValue（“ datasource.url”）String url（） ，而是仅提供@ExternalValue String url（） ，则JavaConfig会查找名为'url'的属性。
