在GWT客户端代码中启用测试驱动开发

工程
伊文·富尔德
2008年2月19日

在过去的几个月中，我一直与各种客户一起使用Google Web Toolkit [GWT]进行项目。我喜欢GWT主要是因为Java到javascript编译器。这是让普通的Java开发人员无需学习新语言即可创建RIA的大门的关键。

我一直是测试驱动开发的忠实拥护者，令我一眼失望的是，看起来TDD和GWT不会一起玩。

测试GWT代码有点问题。问题的核心是GWT代码在运行之前已被编译为javascript。在许多情况下，使用GWT.create（）语句来挂钩动态绑定机制。在正常的Java环境中执行时，此语句会导致异常。

即使使用EasyMock之类的模拟库来模拟罪魁祸首，如果例如从构造函数中触发它，您仍然很可能会遇到问题。为所有小部件创建接口的开销太大，即使这样做，也将无法测试其中具有GWT.create（）的特定类。

GWT在GWTTestCase中提供了针对此问题的解决方案，但是此类本身有很多问题。仅举几个：


  • 它从testrunner内部启动托管模式（使其变慢）

  • 它需要在GWT编译的代码上运行（使其变慢）

  • 它等待带有超时的异步服务（使其变慢）


它提供了一种对客户端代码进行集成测试的不错的机制，但是对于单元测试而言，它太过膨胀了。

当您进行测试驱动的开发时，您需要至少能够编写具有以下属性的测试：


  • 它们立即从IDE运行（无需切换，无需等待）

  • 它们运行速度很快（上限大约为10秒）

  • 他们可以隔离测试单元（模拟或存根依赖性）


所有这些要求都是关于在测试模式和开发模式之间快速轻松地切换。因为如果可能的话，您可以全神贯注于解决问题。从个人经验来看，我可以肯定地说这是一种可靠的方式，但是您的里程可能会有所不同。无论如何，如果这些基本要求得不到满足，您将在开发过程中分心，这消除了TDD的本质，即前期测试的优势。

那么，我们该怎么做，以防止由于缺乏测试而使代码变成意大利面条，而又将那些缓慢的GWTTestCases保持在最低水平？

MVC进行救援

MVC是一种资深模式。它有许多应用程序，除了名称之外，其中一些与原始MVC模式几乎没有任何共同之处，但我要指出的总体方向最好由Martin Fowler以Humble View的名称概括。

我什至更简短的总结是：视图通常很难测试，因此它们应该知道并且做得尽可能少。在GWT中，视图与Widget相同。现在这里有一个小问题：在客户端上运行的所有内容都由Widget拥有。大多数GWT代码几乎都具有小部件中的所有逻辑，因此大多数使用GWT的开发人员都扩展了小部件并仅添加了更多逻辑。

我的建议很简单：不要去那里。类型安全性很容易获得IDE的支持，它并不能阻止您编写错误的代码。因此，如果您不进行单元测试，那么您的代码将最终变得一团糟。

如果您不遵循在视图（小部件子类）中放置逻辑的方法，则需要在另一个地方放置逻辑。那就是Controller进来的地方。回到我之前提到的陷阱，我们将需要从Widget或EntryPoint引导控制器。真的没有办法解决这个问题，所以让我们做一下，看看它看起来有多糟糕：


public class NoteEditor extends Composite {
 public NoteEditor() {
 //do the dependency injection stuff
 NoteModel noteModel = new NoteModel();
 NoteEditorController controller =
 new NoteEditorController(noteModel, NoteService.App.getInstance(), new AlertCallback());
 //…
 }
}

如您所见，我在这里的代码中做了一些依赖注入，因为我们在客户端上还没有Spring。我说了，因为稍后可以使用GWToolbox或Rocket。该服务使用GWT.create（）进行引导，并且AlertCallback用于将控制器与Window分离，以偶尔发出警报。我要说的还算不错，无需编写此代码的测试就可以轻松入睡。麻烦还没有完全解决，因为我们要在视图中使用的任何元素（按钮，标签等）都需要在视图中实例化，然后在控制器中注册：


controller.registerDetailViewSelector(new DeckPanelSelector(detailView));
detailView.addStyleName(“detailPanel”);
main.add(detailView, DockPanel.CENTER);
//some buttons at the bottom of the screen
buttonPanel.add((Widget)controller.registerClearButton(new Button(“Clear”)));
buttonPanel.add((Widget)controller.registerLoadButton(new Button(“Load existing Note”)));

控制器接受其界面（SourcesClickEvents）下的按钮，作为奖励，我们可以使用外观不同的Widget替换按钮，而无需更改控制器。这里没有什么新鲜的，这正是MVC所关注的关注点分离。老实说，我通常更喜欢编写一个测试来检查注册是否正确进行，但是如果没有GWTTesCase，这是无法完成的。现在是时候让我们的IDE为我们创建该控制器+方法并编写测试，以便我们实现逻辑了。例如，加载按钮的测试如下所示：


 public void testLoadButtonPressed_success() throws Exception {
 final Foo expectedFoo = new Foo(“expected”);
 fooServiceMock.loadFoo(isA(String.class), isA(AsyncCallback.class));
 expectLastCall().andAnswer(new IAnswer() {
 public Object answer() throws Throwable {
 ((AsyncCallback) getCurrentArguments()[1]).onSuccess(expectedFoo);
 return null;
 }
 });
 fooModelMock.setFoo(expectedFoo);
 replay(allMocks);
 loadButton.fireClick();
 verify(allMocks);
 }

然后出发！我附上了一个简短的示例来展示这个想法。您可以将其用作实验的起点。

2008-10更新：示例的来源已过时。一般建议仍然成立。

总结一下GWT风格的MVC模式的快速草图。但是，当然，只要您将逻辑保持在可测试的类中 ，就可以选择自己的口味。
