实施企业集成模式第0部分

工程
伊文·富尔德
2008年5月19日

关于Spring Integration的讨论结束后，我对澄清和样本提出了很多疑问。为了满足需求，我将开始一个小系列，介绍如何使用Spring Integration实现不同的集成模式。第一篇文章将重点介绍基础知识。它将向您展示如何起步，跑步并逐步浏览其中一个示例。

如果您之前从未听说过Spring Integration，那么最好先阅读一下Mark Fisher撰写的介绍性博客，或者通过浏览项目网站来熟悉一下它。一般来说

让我从一个免责声明开始：您将在这里看到的模式仅是对Spring Integration必须提供的功能的说明。语法可能因版本不同而有所不同，并且系统上可能需要采取额外的预防措施。

参与Spring Integration的第一步是下载项目并查看示例。因此，让我们开始：

  1. 请查看svn的源代码： svn co https://src.springframework.org/svn/spring-integration/trunk如果您对使用不断变化的代码库感到不舒服，则还可以使用https上的里程碑之一： //src.springframework.org/svn/spring-integration/tags/，但在撰写此博客时，M4尚未发布，我仍然想向您展示最新功能。阅读本文时，请不要忘记检查M4是否尚未退出。上次我检查它即将孵化。
  2. 在整个项目上运行构建：
    $ cd build-spring-integration
    $蚂蚁
  3. 在Eclipse中导入项目，并配置IVY_CACHE变量以指向您的常春藤缓存。这可能是build-spring-integration旁边的目录。
  4. 打开HelloWorldDemo并作为Java应用程序运行。如果您已完成此操作，则将看到语法更改。

如果可以，则可以启动并运行。接下来，您可以探索不同的样本。我将仅描述咖啡馆示例，因为这是最有趣的配置。

在示例项目的cafe软件包中，您将找到与著名的星巴克故事的实现相关的所有代码。为了向您介绍该体系结构，我将按照从开始（Cafe）到结束（Barista）的顺序进行操作：

首先，订单由CafeDemo创建。它包含两种饮料，一种是热的，一种是冷的。然后将订单包装在一条消息中，并放入“订单”渠道。侦听“订单”通道的端点是split方法中的拆分器（OrderSplitter），该订单被拆分为多个Drink，然后由框架将它们包装成单独的消息。

拆分并放入“饮料”频道后，饮料将由DrinksRouter处理。该路由器负责将热饮放到“ hotDrinks”频道上，将冷饮放到“ coldDrinks”频道上。这两个通道均由不同的端点处理，这些端点分别在Barista上使用prepareHotDrink和preprareColdDrink方法。

一种 cafeDemo.xml中使用它来引导OrderSplitter和DrinkRouter。

OrderSplitter上有一些注释：

  • @MessageEndpoint注释告诉Spring Integration在打开注释配置时将这种类型的bean包装在端点中。另外，它是@Component的子类型，因此您不需要为Spring编写xml bean定义就可以将其拾取。
  • @Splitter批注告诉spring使用splitOrder方法将包含Order作为有效负载的消息拆分为多个包含该Order的饮料的消息。

@MessageEndpoint(input=“orders”, output=“drinks”)
public class OrderSplitter {

@Splitter
public List<Drink> split(Message<DrinkOrder> orderMessage) {
    return orderMessage.getPayload().getDrinks();
}

}

同样，DrinkRouter带有注解，指示Spring Integration将其用作路由器。从@Router注释的方法返回的字符串被Spring Integration解释为输出通道。
@MessageEndpoint(input=“drinks”)
public class DrinkRouter {

@Router
public String resolveDrinkChannel(Drink drink) {
    return (drink.isIced()) ? "coldDrinks" : "hotDrinks";
}

}

路由之后，取决于温度，消息在hotDrinks或coldDrinks通道上可用。
