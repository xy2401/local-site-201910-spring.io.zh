完成图：Spring，OSGi和SpringSource应用程序平台

工程
阿德里安·科耶尔（Adrian Colyer）
2008年5月1日

** 5月2日更新的案例研究：-有关详细信息，请参见此帖子的底部**
我敢肯定，大多数阅读此博客的人都会在昨天看到SpringSource Application Platform的发布。如果没有，请确保查看Rob的博客文章 ，其中描述了一些动机，编程模型和路线图。

我想在本文中直接回答几个常见问题。之后，我将描述另外两个激动人心的公告，它们补充了SpringSource Application Platform本身，但昨天没有成为头条新闻：SpringSource Enterprise Bundle存储库和Eclipse的Application Platform工具。这些共同完成了有关使用Spring基于OSGi的企业应用程序开发的故事。

我在过去24小时内多次听到的问题是：OSGi有什么问题-为什么我们不能仅使用香草OSGi服务平台（例如Equinox，Felix或Knopflerfish）而不是SpringSource应用程序平台？

OSGi绝对没有错。

OSGi是一个很好的基础和服务平台-这就是我们和许多其他人选择在其基础上进行构建的原因。它已在广泛的行业和应用程序中得到证明 ，并为诸如Eclipse和IBM的WebSphere等应用程序以及其他多家供应商的中间件堆栈提供了基础。

直接按照OSGi规范API进行编程缺少我们对企业应用程序所期望的某些素质-例如使用依赖注入和创建易于在容器外部进行单元测试和集成测试的应用程序的能力。直接按照OSGi规范进行编程的API也会迫使您以相对较低的水平来处理OSGi平台的动态性-在运行时停止，启动，安装和更新所依赖的模块和服务时，您会怎么做？但是这里没有任何基本障碍，我们无法通过Spring Dynamic Modules项目克服。使用Spring动态模块构建的应用程序可以在任何标准OSGi服务平台上运行，并且我们针对Equinox，Felix和Knopflerfish测试了所有构建。我们致力于确保Spring动态模块和基于Spring的编程模型保持运行时中立。随着SpringSource应用程序平台的推出，这一立场不会改变。

现有企业库也绝对没有错。

好吧在某些情况下，还有一些不足之处，但总的来说，我们知道如何使它们适合企业应用程序开发需求。

那是什么问题呢？

如果OSGi运行良好，并且现有的企业库可以满足我们的需求，那么问题出在哪里？当您尝试将 OSGi服务平台与一组未考虑OSGi编写的现有企业库相结合时，就会遇到困难。这不是OSGi的错，它有一个出色的模型，可提供出色的模块化，版本控制和操作控制。这也不是企业库的错-它们不是写在OSGi下运行的。但是，使OSGi如此吸引人的东西打破了那些企业库的开发人员所做的假设。例如，OSGi的模块化模型使您无法看到其他捆绑软件的私有部分。在您意识到企业库不再能看到您的应用程序类型之前，这就是您想要的。许多事情可能会中断：从公用日志记录到jsps，从标记库到数据源，从加载时编织到组件扫描，从资源加载到orm映射。清单还在继续……（是的，当您将应用程序代码和它需要的所有库打包到一个捆绑包中时，您可以使许多事情工作，但这很重要！）。

这就是为什么您看到很多人都在OSGi之上构建的原因，但是很少有将OSGi好处传递到应用程序编程模型中的情况（Eclipse RCP是一个罕见的例外）。当您在OSGi之上构建但不必为最终用户应用程序开发暴露该模型时，则可以构建到OSGi模型并使其正常工作。当您需要提供一个可以使用大量现有企业库的平台时，这是一个不同的局面。如果我们可以抛弃所有这些，然后从为OSGi明确编写的库重新开始，那很好。例如，我们已经确保Spring Framework 完全能够在OSGi服务平台内运行。但这不是一个现实的主张。或者，我们可以等待现有库的开发人员将它们全部转换为可以在OSGi下运行（如我们对Spring所做的那样）。但是，除非其他所有人也这样做，他们这样做的动机是什么？因此，我们似乎陷入了困境。在过去的一年中，OSGi企业专家组花了很多时间讨论这个问题。
这是SpringSource Application Platform解决的问题：-通过使具有标准OSGi语义的标准OSGi捆绑包与现有企业应用程序库一起使用，它将企业应用程序开发引导到OSGi领域。

我还想再次强调一下，该平台不仅是关于OSGi的：OSGi支持是我们最兴奋的功能之一，但是SpringSource Application Platform还是一个用于部署标准war文件的出色服务器平台。我们将在以后的文章中描述平台在这种情况下提供的好处。

希望这篇文章有助于消除围绕SpringSource Application Platform与OSGi的确切联系所引起的一些困惑。如果到目前为止您一直在关注，那么您可能会遇到另一个潜伏的问题：使现有企业库在OSGi下工作很好，但是您不需要将其所有jar文件都转换为OSGi捆绑包为了能够部署它们？是的你是。如果要正确版本化所有导入和导出并确保您具有正确的符号名称等，事实证明这是很多工作。好消息是，对于数百种常用的企业应用程序库，我们已经为您完成了艰苦的工作，并在SpringSource Enterprise Bundle信息库中提供了OSGi就绪版本。

SpringSource企业捆绑存储库


SpringSource Enterprise Bundle信息库既是可从Ivy和Maven使用的信息库，又是企业库的在线可搜索数据库。您可以在www.springsource.com/repository上找到它。您可以按名称浏览捆绑软件，或仅键入搜索词以查找具有匹配名称，导出的软件包，类或资源的捆绑软件。您还可以看到任何捆绑软件的最小（仅满足所需的依赖）和最大（满足尽可能多的可选依赖）。

从常见问题解答：

“ SpringSource Enterprise Bundle信息库是一个开源库的集合，这些开源库通常用于使用Spring Framework开发企业Java应用程序。该存储库包含jar文件（捆绑包）和库定义（“ .libd”）文件。一个库定义了捆绑软件的集合，这些捆绑软件通常出于某种目的而一起使用（例如“ Spring Framework”库）。存储库中包含数百个捆绑软件。”
存储库满足以下条件：


  • 存储库中的每个jar文件都是有效的OSGi捆绑包。您从存储库下载的任何jar都可以按原样部署到OSGi服务平台和SpringSource应用程序平台中。它也可以用作OSGi之外的常规jar文件。

  • 每个捆绑软件和库都有与之关联的完整版本信息。捆绑软件的软件包导出信息包含版本信息，捆绑软件的软件包导入信息包含完整的版本范围兼容性信息。

  • 存储库在传递上已完成。保证任何捆绑软件的强制性依赖性也都在存储库中。存储库中任何捆绑软件的大多数可选依赖项也将存在。任何库定义中列出的捆绑软件都保证在资源库中。

  • 存储库是自洽的。在将任何人工制品上载到存储库之前，我们验证可以与存储库中的所有其他捆绑包一起在OSGi服务平台（使用与SpringSource Application Platform相同的配置文件）中安装，解析和启动该人工制品。

  • 可以从基于Ivy和Maven的构建中使用该存储库。


为了保持这些保证，我们围绕将文物发布到存储库建立了治理模型。有一个JIRA实例，您可以针对该实例提出要包含的其他库的请求，并报告现有已发布文物的任何问题（与OSGi清单等有关）。

应用开发工具：


到目前为止，我们已经讨论了用于将应用程序开发为OSGi捆绑包的基于Spring的编程模型，可部署到OSGi Service Platform中的企业库的可用性以及使这些旧式库能够在Windows 2000中运行的运行时（SpringSource Application Platform）。 OSGi运行时。难题中缺少的一环是开发人员工具，这些工具可以简化基于OSGi的应用程序的创建。

Eclipse已经内置了OSGi开发工具。由于每个Eclipse插件也是OSGi捆绑包，因此可以将Eclipse PDE工具（插件开发环境工具）用于OSGi应用程序开发。但是，这些工具主要是为Eclipse插件的开发而设计的事实表明了这一点，并且在将它们用于OSGi应用程序开发时会遇到一些常见的挫败感。一种是META-INF / MANIFEST.MF文件只能放在项目的根目录下-与Ivy和Maven等构建工具不能很好地配合，另一种是您只能使用一个目标整个工作空间的平台（要开发的捆绑软件集合）。什么是伟大的有关PDE的工具，和你真正需要的，是他们打造的编译类路径从OSGi清单项目-让你不必在编译，测试和运行时之间的类路径和类知名度的差异。

除了SpringSource Application Platform之外，我们还发布了一组Eclipse插件（可从SpringSource Application Platform下载页面获得），这些插件使OSGi应用程序的开发更加容易，尤其是针对SpringSource Application Platform的应用程序。您的META-INF / MANIFEST.MF文件可以位于任何源目录中，并且这些工具从清单条目构建编译类路径。但是，您可以将项目与为Eclipse定义的SpringSource Application Platform服务器关联（而不是使用单个目标平台）（使用WTP工具）。然后，从清单中的import语句派生项目的类路径，并针对工作空间中的其他捆绑软件项目以及关联服务器中安装的捆绑软件进行解析。您在编译时对类路径和依赖项的解释与运行时完全相同。当然，正常的“部署到服务器”选项也可以使用。

这是服务器在Eclipse中运行时的外观：

此屏幕快照显示了如何使用“捆绑依赖项”类路径容器管理类路径。请注意，清单文件中尚未导入的软件包是如何变灰的，以指示您当前无法访问它们。

更好的是我们如何利用OSGi的模块化。一组项目（每个捆绑包一个）组成您的应用程序。当您在项目中进行任何更改时，一个额外的增量生成器将分析资源增量并在SpringSource Application Platform中对正在运行的捆绑软件进行实时更新-因此，您将始终使用最新的代码进行运行。这极大地提高了生产力，并带来了出色的开发经验。

案例分析


Matt Raible发布了一篇关于他的冒险的博客条目 ，他试图使用Freemarker在OSGi下运行Spring Web应用程序-而不使用SpringSource Application Platform。这似乎是一个很好的挑战应用程序，用于测试我上面所说的有关使现有企业库正常工作的内容。好消息是，此应用程序在SpringSource应用程序平台上非常愉快地运行。这是我要使其正常工作所遵循的步骤（总时间，大约10分钟）：

  • 从Matt的博客下载zip文件

  • 运行“ MVN”

  • cp target / mpapp.war到平台的提取目录

  • 启动平台：bin / startup.sh。


我将以下输出输出到控制台：
com.springsource.platform.deployer.core.DeploymentException: Unable to satisfy constraints of ‘myapp’ version ‘0.0.0’:

Cannot resolve: myapp  Unsatisfied leaf constraints:

Bundle: myapp_0.0.0 - Import-Package: org.springframework.osgi.web.context.support; version=“0.0.0”

Did you mean: ‘org.springframework.osgi.context.support’?

Bundle: myapp_0.0.0 - Import-Package: freemarker.ext.servlet; version=“0.0.0”

Did you mean: ‘javax.servlet’?

Bundle: myapp_0.0.0 - Import-Package: freemarker.core; version=“0.0.0”

Did you mean: ‘org.hamcrest.core’?

Bundle: myapp_0.0.0 - Import-Package: freemarker.template; version=“0.0.0”

Did you mean: ‘org.antlr.tool’?

Bundle: myapp_0.0.0 - Import-Package: freemarker.cache; version=“0.0.0”

Did you mean: ‘org.apache’?

这些是预期的消息，因为我没有在平台中安装freemarker或osgi.web.context支持包。

  • 转到http://www.springsource.com/repository 。在搜索框中输入“ freemarker”，找到一个匹配的条目，然后单击链接进行下载。将下载的捆绑软件复制到存储库/捆绑软件/ usr

  • 简化清单以指向平台上的新捆绑包和库。原始清单如下所示：


Import-Package: javax.servlet,javax.servlet.http,javax.servlet.resources,javax.swing.tree,

javax.naming,org.w3c.dom,org.apache.commons.logging,javax.xml.parsers;resolution:=optional,

org.xml.sax;resolution:=optional,org.xml.sax.helpers;resolution:=optional,

org.springframework.osgi.web.context.support, org.springframework.context.support,

org.springframework.web.context, org.springframework.web.context.support,

org.springframework.web.servlet, org.springframework.web.servlet.mvc,

org.springframework.web.servlet.mvc.support, org.springframework.web.servlet.view,

org.springframework.ui, org.springframework.web.servlet.view.

freemarker, freemarker.cache,freemarker.core,freemarker.template,freemarker.ext.servlet

我将其归结为：
Import-Package: org.apache.commons.logging

Import-Library: org.springframework.spring;version=“[2.5.4,3.0.0)”

Import-Bundle: com.springsource.freemarker;version=“2.3.12”

当我们知道您正在部署Web应用程序时，通常需要的导入将在部署时自动添加。使用Import-Library和Import-Bundle，您可以在单个语句中方便地引用库和包。我还删除了“ Bundle-Classpath”条目，因为该应用程序平台将自动检测WEB-INF / lib中的库并将它们添加到Bundle类路径中。

  • 我编辑了web.xml并注释了context-param声明，因为这里不需要使用自定义应用程序上下文类型

  • 再次运行“ mvn”，然后将myapp.war复制到拾取目录。

  • Application Platform自动重新部署应用程序

  • 将浏览器指向http：// localhost：8080 / myapp / 。成功！


我认为这很好地展示了该平台的价值主张，可以平滑使企业库在OSGi下工作的路径。
