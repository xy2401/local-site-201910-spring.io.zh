使用SpringSource Application Platform的供应存储库

工程
安迪·威尔金森
2008年5月9日

SpringSource应用程序平台的主要优点之一是它能够根据需要提供依赖项。这样做的好处有两方面：确保平台的内存占用空间尽可能小，并且可以在不将所有依赖项封装在整体部署单元（例如WAR文件）中的情况下部署应用程序。要利用这些功能，您需要了解平台的配置存储库，而本博客旨在提供此功能。

供应存储库在哪里，它如何工作？


默认情况下，可以在安装根目录的存储库目录中找到平台的配置存储库：


如您所见，这里有三个主要目录： bundles ， installed和library 。 安装的文件仅供平台内部使用，因此我们将在此处重点介绍bundles和库目录。每个包含多个子目录以分隔不同类型的依赖项：


  • ext包含平台随附的外部依赖关系，但不属于平台本身。

  • 子系统包含构成平台的所有子系统。

  • usr最初为空，旨在包含用户添加的依赖项，即，平台尚未提供的应用程序所依赖的任何内容。


平台在其初始启动期间会在存储库目录结构中搜索捆绑软件和库。我将在稍后的条目中讨论如何配置此搜索。当在存储库中找到包和库时，它们的符号名，导出的包等详细信息将添加到存储库的内存索引中。完成扫描后，内存中索引将缓存到磁盘。在开发过程中，尽量减少平台的启动时间是我们的首要任务。这种缓存使平台可以在启动过程中节省一些时间：它可以跳过扫描，除非它检测到存储库的内容已更改。

运行时供应


在普通的OSGi环境中，捆绑包的依赖关系只能由环境中已经安装的其他捆绑包来满足。例如，如果尚未安装任何导出org.apache.commons.dbcp软件包的软件包，则安装和启动该软件包将失败。对于用户而言，这可能是真正的痛苦，因为他们必须手动安装捆绑软件的所有依赖项。值得庆幸的是，SpringSource应用程序平台通过在需要的基础上动态安装依赖项来显着改善了这一点。

当平台启动已部署的应用程序时，其捆绑包将安装到Equinox中。然后，平台要求Equinox提供所有捆绑软件未满足的依赖关系的列表，并尝试满足它们。让我们通过简单的示例场景来分析此过程：


  1. 平台将导入org.apache.commons.dbcp的捆绑软件安装到Equinox中。

  2. 平台向Equinox询问捆绑软件不满意的依赖关系，并被告知无法满足org.apache.commons.dbcp的导入要求。

  3. 平台在其配置存储库的索引中搜索一个捆绑包，该捆绑包导出org.apache.commons.dbcp

  4. 平台安装将org.apache.commons.dbcp导出到Equinox的捆绑软件

  5. 满足原始捆绑软件的依赖性后，平台成功启动了原始捆绑软件。


我们希望这将使我们的用户的生活变得更加轻松：只要您的应用程序的依赖项对平台可用，您就可以轻松部署应用程序并开始工作！不需要费时的手动安装所有应用程序依赖项的过程。此过程的一大优点是，仅在需要时才安装依赖项。您可以根据需要将任意数量的捆绑软件添加到预配置存储库，而几乎不影响平台的内存占用。

如果平台无法满足其存储库中的依赖关系，则会生成一条日志消息，其中详细说明了无法满足的依赖关系。掌握了这些信息后，您可以使用SpringSource Enterprise Bundle信息库来掌握您的需求。我们希望存储库提供尽可能完整的捆绑软件集：如果您需要依赖项，并且该依赖项不可用，请告知我们 。

将项目添加到预配置存储库


因此，您已经下载了一个或两个依赖项，现在您要将它们添加到平台中。您需要做的就是将它们复制到存储库中的相应目录中。例如，通常将新的捆绑软件添加到repository / bundles / usr目录，并且通常将新的库描述符添加到repository / libraries / usr目录。

如上所述，平台使用其在启动期间填充的配置存储库的内存中索引。除此之外，每当部署应用程序时，平台还将检查其对存储库的视图是否为最新。如果要安装具有新依赖关系的应用程序，只需将依赖关系复制到存储库中的适当位置，然后部署您的应用程序。在部署过程中，平台会注意到存储库已更新，并将刷新其视图。这意味着您不需要每次要安装具有新依赖性的应用程序时都花时间重新启动平台。

在安装之间共享配置库


可以轻松配置平台在预配置存储库中搜索项目的位置，以适合您的需求。例如，多个Platform实例有可能共享部分或全部的资源调配存储库，因此您无需浪费精力在Platform安装之间维护重复的捆绑集。

可以在config / platform.config文件中配置平台在创建其配置存储库时扫描的位置。在没有任何特定配置的情况下使用的默认配置为：


"provisioning" : {
    "searchPaths": [ 
        "repository/bundles/subsystems/{name}/{bundle}.jar",
        "repository/bundles/ext/{bundle}",
        "repository/bundles/usr/{bundle}",
        "repository/libraries/ext/{library}",
        "repository/libraries/usr/{library}"
    ]
}

平台会将任何相对路径解释为相对于其安装根目录，并且也支持绝对路径。大括号内的搜索路径中的条目只是通配符，例如，子系统搜索路径存储库/捆绑包/子系统/{name}/{bundle}.jar将在任何子系统目录的目录中找到任何名称以.jar结尾的文件。直接子目录。

希望您能看到在平台之间共享部分或全部配置存储库是一项微不足道的更改。例如，要使平台在文件系统的根目录以及其自己的子系统和ext目录中搜索名为shared-bundles的目录，您需要做的就是将以下JSON （JavaScript对象符号）代码段添加到platform.config文件：


"provisioning" : {
    "searchPaths": [
        "repository/bundles/subsystems/{name}/{bundle}.jar",
        "repository/bundles/ext/{bundle}",
        "/shared-bundles/{bundle}",
        "repository/libraries/ext/{library}",
        "repository/libraries/usr/{library}"
    ]
}

通过使用此配置配置两个或多个Platform安装，可以使它们共享/ shared-bundles中的捆绑包 。通过将所有搜索路径配置为指向共享位置，可以轻松地将其更进一步，从而您根本不需要管理每个平台的配置库。

下一步是什么？


我们计划通过提供一些工具或脚本来简化从同一组二进制文件运行平台的多个实例的过程。这些将为您完成大部分工作，并提供合理的默认配置，然后您可以对其进行调整以满足特定的需求。

我们还打算通过允许将平台配置为在尝试满足依赖关系时选择搜索远程存储库，从而结合平台按需供应和SpringSource Enterprise Bundle存储库的功能。如果在远程存储库中找到依赖项，则平台将自动处理其下载和安装。希望这将使开发人员的生活变得更轻松，尤其是在应用程序开发的初始阶段，因为定期添加新的依赖项。
