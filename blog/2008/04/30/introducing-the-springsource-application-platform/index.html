介绍SpringSource应用程序平台

工程
罗伯·哈罗普
2008年4月30日

经过几个月的狂热编码，我很高兴地宣布SpringSource Application Platform 1.0的beta版。

从2007年初开始，我们开始讨论Enterprise Java成为代名词的单片和重量级应用服务器的可能替代方案。客户正在寻找一个足够轻巧，模块化和灵活的平台，以满足他们的开发和部署需求。

Spring和Tomcat的配对表明，开发人员和操作员可以在生产中成功使用轻量级平台。尽管这种组合取得了成功，但缺乏模块化和对非Web应用程序的明确支持限制了其适用性和灵活性。

我们着手构建SpringSource应用程序平台来满足这些要求并消除这些限制。

平台的核心是动态模块内核（DMK）。DMK是基于OSGi的内核，它充分利用了OSGi平台的模块化和版本控制功能。DMK建立在Equinox之上，并扩展了其用于供应和库管理的功能，并为平台提供了核心功能。

为了保持最小的运行时占用，DMK设置子系统按需安装了OSGi捆绑包。这允许将应用程序安装到正在运行的平台中，并可以从外部存储库中满足其依赖关系。这不仅消除了手动安装所有应用程序依赖项的麻烦（这将是乏味的），而且还将内存使用量降至最低。

DMK本身需要运行的捆绑软件最少，并且配置了配置文件以精确控制所加载的其他模块的集合。例如，DMK不需要存在Tomcat，但是默认的平台配置文件包括Tomcat，以允许部署Web应用程序。如果要在没有Tomcat的情况下运行平台，则只需编辑配置文件就不会安装它。（如果尝试这样做，请记住，删除Web支持意味着将不再部署Web模块，因此请删除拾取目录的内容，以便该平台在启动时不会尝试安装Admin和启动屏幕应用程序。）安装了管理控制台的默认平台配置仅占用15MB的内存。

我一直对Enterprise Java感到沮丧的是，应用程序经常被人为地塞入人为的孤岛，并且缺少对不同应用程序类型的明确支持。考虑一个在线商店的应用程序。该应用程序具有Web前端，消息驱动的订单处理模块，批处理驱动的库存重新订购模块和B2B Web服务模块。如今，许多类似这样的应用程序将被打包为WAR或EAR，并且模块看起来非常相似，几乎不支持模块类型的差异。有趣的是，许多人将其称为Web应用程序，而不是带有Web模块的应用程序。

在SpringSource应用程序平台中，应用程序是模块化的，每个模块都有一个个性 ，以描述它是哪种模块：Web，批处理，Web服务等。平台以个性化方式部署每个个性化的模块。例如，在Tomcat中使用Web上下文配置了Web模块。可以独立于其他模块更新应用程序中的每个模块，同时保留作为较大应用程序一部分的身份。无论您要构建哪种应用程序，编程模型都将保留标准的Spring和Spring DM。

在1.0 Platform版本中，我们支持Web和捆绑包个性，这使您可以构建复杂的Web应用程序。将来的版本将包含对更多个性的支持，如稍后所述。

建筑应用

该平台支持以三种形式打包的应用程序：

  1. Java EE战争
  2. 原始OSGi捆绑包
  3. 平台存档（PAR）

平台直接支持标准WAR文件。在部署时，WAR文件将转换为OSGi捆绑软件并安装到Tomcat中。遵守所有标准WAR合同，并且您现有的WAR文件应直接放入并部署，无需更改。

任何与OSGi兼容的捆绑包都可以直接部署到平台中，并且可以充分利用Import-Package和Require-Bundle引用的所有依赖项的即时配置。

建议使用PAR格式为平台打包和部署应用程序。PAR只是在标准JAR文件中分组在一起的OSGi捆绑包（模块）的集合，以及唯一标识应用程序的名称和版本。PAR文件作为一个单元部署到平台中。平台将从PAR中提取所有模块并进行安装。第三方依赖项将根据需要即时安装。

与将捆绑软件直接部署到平台相比，PAR格式具有三个主要优点。首先，这很容易。一个普通大小的企业应用程序可能包含12个以上的捆绑包-手动部署它们将非常麻烦。其次，PAR文件在应用程序中的所有捆绑软件上形成了明确的范围，从而防止了使用重叠类型或服务的应用程序彼此冲突。一些高级功能（例如加载时编织）也使用此作用域，以确保编织一个应用程序不会干扰另一个应用程序的编织。最后，PAR构成逻辑分组，以定义哪些模块是应用程序的一部分以及该应用程序具有哪些第三方依赖性。管理工具使用此分组来提供应用程序的详细视图。一个典型的PAR应用程序如下所示：

应用程序中模块之间的依赖关系通常使用Import-Package和Export-Package表示 。对第三方库的依赖关系可以用相同的方式表示，但是对于许多库而言，这很容易出错并且很耗时。当使用诸如Hibernate之类的库时，通常将需要导入比最初预期更多的包。为了解决这个问题，您可以使用Require-Bundle ，但这具有一些语义上的粗糙边缘，例如拆分包，其中逻辑包被拆分为两个或多个类加载器，从而在运行时引起问题。该平台引入了两种引用第三方依赖项的新机制： Import-Bundle和Import-Libary 。Import-Bundle与Require-Bundle相似，除了它可以防止拆分包和Require-Bundle的其他问题。Import-Library提供了一种在单个声明中引用一组捆绑软件（例如，Spring Framework中的所有捆绑软件）导出的所有软件包的机制：

Bundle-SymbolicName: com.myapp.dao.jdbc
Bundle-Version: 1.0.0
Import-Bundle: org.apache.commons.dbcp;version="1.2.2.osgi"
Import-Library: org.springframework.spring;version="2.5.4.A"

在这里，我有一个模块包，它依赖于Commons DBCP包和Spring Framework库。Spring框架库包含在您的应用程序中使用Spring所需的所有捆绑软件。

Import-Library和Import-Bundle在后台扩展为Import-Package ，因此与标准OSGi语义一致。

平台了解模块的个性，由此可以推断出如何配置模块的执行环境。部署Web模块时，将自动创建典型的Spring MVC应用程序所需的所有servlet基础结构，而无需在应用程序之间重新创建此样板代码。在1.0最终版本中，更多智能将添加到Web模块个性中，以支持Spring Web Flow等其他技术。

无论选择哪种打包格式，编程模型都只是Spring Framework和Spring Dynamic Modules，而其他Spring Portfolio产品则在上面运行。

可维修性

可维护性是整个工程团队的关键考虑因素。我们花费了大量时间来担心日志消息的格式和堆栈跟踪的大小，以确保尽可能轻松地诊断应用程序问题。每当发现重复性和耗时的任务时，我们都在寻找一种使之自动化或完全删除的方法。

为了帮助进行问题诊断，该平台在日志和跟踪消息之间进行了很大的区分。日志消息旨在供最终用户使用，并允许您访问最重要的故障信息，而不必遍历千兆字节的跟踪内容。所有应用程序故障都会显示在日志输出中并进行编码-这些代码可作为访问知识库或支持内容的便捷方式。要了解为什么这样做如此有用，请考虑平台启动的以下输出：

[2008-04-29 12:12:04.037] main                     <SPKE0000I> Boot subsystems installed.
[2008-04-29 12:12:06.013] main                     <SPKE0001I> Base subsystems installed.
[2008-04-29 12:12:07.396] platform-dm-1            <SPPM0000I> Installing profile 'web'.
[2008-04-29 12:12:07.674] platform-dm-1            <SPPM0001I> Installed profile 'web'.
[2008-04-29 12:12:07.721] platform-dm-14           <SPSC0000I> Creating ServletContainer on port 8080
[2008-04-29 12:12:08.036] platform-dm-10           <SPPM0002I> Platform open for business with profile 'web'.
[2008-04-29 12:12:09.405] fs-watcher               <SPSC1000I> Creating web application '/admin'
[2008-04-29 12:12:09.652] async-delivery-thread-1  <SPSC1001I> Deploying web application '/admin'

输出的页面和页面不再详细说明启动调用链中每种类型的内部工作。相反，您得到的消息实际上是有意义的 。当然，这并不意味着您无法找出每种类型的作用：我们仍然保持广泛的跟踪。确实，由于我们从跟踪中分离出了最重要的日志消息，因此我们的跟踪可能会变得更加冗长，因为它打算减少读取的频率。

平台中的严重故障会生成服务转储，可以将其转交给支持人员以帮助诊断过程。此转储包含来自平台中所有主要子系统的状态，并通过AspectJ进行了挂钩，以确保我们选择了可能引发未经检查的Exception的每个可能的站点。

该平台还积极监视JVM中的问题并触发转储。在1.0版本中，此功能仅限于死锁监视，但在将来的版本中将包括内存，锁争用和CPU争用。

路线图

这个beta版本只是SpringSource应用程序平台的开始。我们已经初步定义了涵盖2.0版本的路线图，重点放在五个主要领域：管理控制台，中间件集成，其他个性，集群和DMK 2.0。

通过管理控制台，我们希望利用平台拥有的所有应用程序知识，并将其提供给最终用户。管理控制台将允许从模块级别开始一直运行到Bean级别的应用程序的详细检查。应用程序将链接到它们所依赖的库和捆绑软件，并且管理控制台将提供动态升级这些库的功能。通过了解应用程序包含的不同类型的Bean，管理控制台将提供对应用程序内部的详细了解，并允许对某些应用程序组件进行动态重新配置。

为了支持典型的企业部署方案，该平台的2.0版本将为常见的企业中间件组件（例如Oracle，TIBCO EMS和IBM MQSeries）提供显式支持。与管理控制台的集成将允许在几分钟（而不是几小时或几天）内建立与这些中间件提供商的连接。应用程序将能够使用OSGi服务注册表和Spring DM的访问这些连接。 。

2.0版将引入其他特性，以覆盖批处理，Web服务和SOA应用程序。

群集是2.0版的一项重要功能，群集范围内的单个系统映像（SSI）是至关重要的部分，并且路线图上也包括负载平衡和群集缓存。使用SSI支持，管理和部署操作将在整个群集中传播。最终，我们希望支持整个集群中的每个模块更新，如果升级中断了已部署的应用程序，则需要完整的集群回滚。

DMK 2.0将包括对并发子系统的改进，以支持频繁更改的大量捆绑软件，并支持具有更复杂相互依赖性的大量模块。还包括对供应子系统的更新以及对内存，线程，IO和CPU的集成资源管理和监视。
