对Spring Data做出反应

工程
马克·帕鲁奇
2016年11月28日

上周的Spring Data Kay M1是支持响应数据访问的第一个版本。它最初支持的存储集-MongoDB，Apache Cassandra和Redis-都已经发布了反应性驱动程序，这使它们成为此类原型的自然候选者。让我们更详细地研究新的编程模型和构成该支持的API。

反应库

存储库编程模型是Spring Data用户通常处理的最高级抽象。它们通常由Spring Data提供的接口中定义的一组CRUD方法和特定于域的查询方法组成。这是反应式Spring Data存储库定义的样子：

public interface ReactivePersonRepository
  extends ReactiveCrudRepository<Person, String> {

  Flux<Person> findByLastname(Mono<String> lastname);

  @Query("{ 'firstname': ?0, 'lastname': ?1}")
  Mono<Person> findByFirstnameAndLastname(String firstname, String lastname);
}

如您所见，与您以前的习惯并没有太大区别。但是，与传统的存储库接口相比，反应式存储库使用反应式类型作为返回类型，并且也可以对参数类型进行操作。新引入的CRUD方法ReactiveCrudRepository ，当然也可以使用这些类型。

默认情况下，反应式存储库使用Project Reactor类型，但也可以使用其他反应式库。我们提供了一个自定义的信息库基础界面（例如RxJava2CrudRepository ），并根据查询方法（例如RxJava的Observable和Single 。其余基本上保持不变。但是请注意，当前里程碑尚不支持分页，并且您当然必须在类路径上具有必需的反应式库才能激活对特定库的支持。

激活反应式Spring数据

与阻塞世界一样，对响应式Spring Data的支持通过以下方式激活： @Enable…注释以及一些基础架构设置：

@EnableReactiveMongoRepositories
public class AppConfig extends AbstractReactiveMongoConfiguration {

  @Bean
  public MongoClient mongoClient() {
    return MongoClients.create();
  }

  @Override
  protected String getDatabaseName() {
    return "reactive";
  }
}

了解我们如何使用其他基类进行基础结构配置，因为我们需要使用MongoDB异步驱动程序。

Spring Cloud Data Flow 1.1 GA发布

发布
马克·波拉克
2016年11月23日

我很高兴代表团队宣布GA发布Spring Cloud Data Flow 1.1。按照入门指南中的链接下载本地服务器实现和Shell，以创建Stream和Tasks。

1.1 GA版本的主要亮点包括：

  • 基于Spring Boot 1.4，Spring Cloud Camden SR2，Spring Integration 4.3和Spring Cloud Task 1.1发行版的改进。

  • 添加基于LDAP ， Basic和File的后端身份验证

  • OAUTH支持的身份验证的改进

  • SSL现在支持LDAP身份验证

  • 为非OAUTH后端身份验证方法（例如LDAP，Basic和基于文件的选项）添加基于表单的登录页面

  • 添加了通过YAML文件传递应用程序特定属性的功能 。当部署需要设置许多部署属性的流时，这特别有用。

  • 内存，磁盘和cpu的可移植部署属性已准备就绪，可在各种运行时实现中提供支持。

  • 核心REST API的文档

  • 更新了参考文档，显示了如何与Spring Boot Admin集成以可视化服务器指标以及如何将Stream / Task应用程序指标导出到外部应用程序监视工具 。

Spring Cloud Task 1.1.0。现在可以发布

发布
迈克尔·米内拉
2016年11月22日

我们很高兴地宣布Spring Cloud Task 1.1.0。现在可以通过Github和Pivotal下载存储库获得RELEASE。非常感谢所有对此版本做出贡献的人。

Spring Cloud Task 1.1.0提供以下功能：

这是1.1.0版的通用版本（GA）。此版本解决了该项目的关键增强功能，可以更好地覆盖云环境中任务的操作问题。1.1.0系列的新增功能包括：

  • 更新的错误处理 -1.0.x存储的堆栈跟踪是任务执行过程中的结果TaskExecution#exitMessage字段，要求该字段执行双重职责。首先，它可用于编排任务（类似于StepExecution#exitStatus在Spring Batch中）以及用于调试的堆栈跟踪存储。在1.1.0版本中，错误消息已移至新字段TaskExecution#errorMessage这样每个字段都有自己专用的用途。

  • 更新了分区批处理作业的自定义选项 -在Spring Cloud Task的1.0.x行中，将工作程序作为任务启动时，无法自定义提供给他们的命令行参数。在像CloudFoundry这样的环境中，这是一个问题，您可以在其中使用命令行参数来自定义配置，而无需重新推送您的应用程序。在新的1.1.0版本中，我们提供了通过以下命令来自定义命令行参数的功能CommandLineArgsProvider在功能上类似于EnvironmentVariablesProvider在1.0.2中引入。

  • 外部执行ID持久性 -在云环境中运行任务时，通常有一个单独的ID与与基础结构关联以执行任务。YARN在历史记录服务器中使用的ID或CloudFoundry使用的任务ID是这些ID的示例。Spring Cloud Task 1.1.0现在提供了持久化此外部执行ID的功能，以便能够将一个人映射到另一个人。

  • 记录任务执行的请求 -在某种形式的基础架构（如Kubernetes或CloudFoundry）上启动任务时，请求任务执行的时间与实际启动之间可能会有一定的滞后。如果平台存在问题，该任务可能根本无法启动，这使该问题更加复杂。如果发生这种情况，您仍然需要记录请求。 Spring Cloud Task 1.1.0引入了一种功能，它可以记录任务预期在实际开始执行之前开始。允许启动器记录此请求，任务本身将相应地更新其状态。

  • 任务存储库的其他数据库支持 -1.1.0引入了社区贡献的模式，以支持DB2作为任务存储库的数据存储。
