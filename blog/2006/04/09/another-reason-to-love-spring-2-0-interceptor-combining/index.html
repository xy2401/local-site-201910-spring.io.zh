喜欢Spring 2.0的另一个原因：拦截器组合

工程
本·黑尔
2006年4月9日

最近，我正在一个项目中，该项目的Swing客户端通过RMI与服务层进行通信。服务层标记有事务，一切似乎都正常。但是，每当我们在Hibernate DAO层上遇到一个异常时，Spring都会将该异常转换为运行时异常，并将其作为RemoteException一直传播到整个堆栈并贯穿RMI连接。每当反序列化异常时，客户端上都会有一个异常（与RemoteException分开）。决定仅介绍一个方面。子类化ServiceAccessException的任何异常将被传递给客户端，而其他任何异常将被转换为FilteredServiceAccessException （ ServiceAccessException的子类），然后引发该异常。这导致内容丢失，因此我们确保将原始异常记录在可能有用的服务器上，并让客户端显示一个通用对话框，以便用户大致了解发生了什么。

现在，这是一个相当不错的计划，并且在我们尝试实施之前，它似乎可以正常运行。我们正在使用一种自动代理任何具有@Transactional的 bean的魔术方法来获取我们的事务代理。我们可能已经更新了该自动代理的定义，以确保添加了有关此异常过滤的建议（请考虑TransactionProxyFactoryBean中的 setPreInterceptor ），但是自动代理所捕获的不仅仅是服务层。

那那把我们留在哪里呢？我们可以A）显式声明对TransactionProxyFactoryBean的每次使用，B）进行两组不同的自动代理，并使它们彼此互斥，或者C）现在忽略此要求，并希望发生一些神奇的事情。由于产品距离消费者还有六个月的时间，我尝试遵循杰里米·米勒（Jeremy Miller）向我介绍的“ 最后一个负责任时刻 ”的原则，因此我决定以选项A作为我的备份计划来解决这个问题（最好别无选择，魔法的两倍）。

瞧，Spring 2.0解决了我的问题。我一生无法找到阅读的地方，但是从2.0的里程碑之一开始，当代理一个bean时，代理工厂现在可以检测到该bean已经具有代理，而只需将预期的拦截器添加为另一个拦截器即可。 （如果您知道它在哪里，请在评论中保留链接）。这意味着我可以使用新的魔术（ ），只需添加一个具有所需切入点的方面，就不必担心事务代理和AOP代理会交叉使用。不太清楚这是怎么回事？举个例子。首先是接口和实现。


package interceptorcombiningexample;

import org.springframework.transaction.annotation.Transactional;

@Transactional
public interface ExampleTarget {

	void exampleMethod();

}


package interceptorcombiningexample;

public class DefaultExampleTarget implements ExampleTarget {

	public void exampleMethod() {
	}
}

注意，该接口被标记为@Transactional 。稍后，我们将使用它来进行一些神奇的自动代理。接下来，我们将看一下bean的定义。


<?xml version="1.0" encoding="UTF-8"?>
<beans xmlns="http://www.springframework.org/schema/beans"
	xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
	xmlns:aop="http://www.springframework.org/schema/aop"
	xmlns:tx="http://www.springframework.org/schema/tx"
	xsi:schemaLocation="http://www.springframework.org/schema/beans
       http://www.springframework.org/schema/beans/spring-beans.xsd
       http://www.springframework.org/schema/aop 
       http://www.springframework.org/schema/aop/spring-aop.xsd
       http://www.springframework.org/schema/tx
       http://www.springframework.org/schema/tx/spring-tx.xsd">

	<tx:annotation-driven />

	<aop:config>
		<aop:aspect id="exampleAspect" ref="exampleAdvice">
			<aop:before method="exampleAdvice"
				pointcut="execution(* interceptorcombiningexample.ExampleTarget.exampleMethod())" />
		</aop:aspect>
	</aop:config>

	<bean id="exampleAdvice"
		class="interceptorcombiningexample.ExampleAdvice" />

	<bean id="exampleTarget"
		class="interceptorcombiningexample.DefaultExampleTarget" />

	<bean id="transactionManager"
		class="interceptorcombiningexample.DummyTransactionManager" />

</beans>

您会注意到，我们设置了注释驱动的事务，该事务将自动围绕DefaultExampleTarget建立一个代理。此外，我们定义了另一个方面，该方面需要代理同一个DefaultExampleTarget bean。最后，让我们看一下我们的可执行类。


package interceptorcombiningexample;

import org.springframework.aop.Advisor;
import org.springframework.aop.framework.Advised;
import org.springframework.context.ApplicationContext;
import org.springframework.context.support.ClassPathXmlApplicationContext;

public class InterceptorCombiningExample {

	public static void main(String[] args) {
		ApplicationContext ctx = new ClassPathXmlApplicationContext(
				"classpath:interceptorcombiningexample/applicationContext.xml");

		ExampleTarget target = (ExampleTarget) ctx.getBean("exampleTarget");
		if (target instanceof Advised) {
			Advised advised = (Advised) target;
			System.out
					.println("Advisor count: " + advised.getAdvisors().length);
			for (Advisor advisor : advised.getAdvisors()) {
				System.out.println("Advisor type: "
						+ advisor.getAdvice().getClass().getName());
			}
		}
	}

}

该类利用了Spring代理机制的一个很好的小功能。任何由Spring创建的代理都可以转换为Advised接口。该界面将使您能够访问代理中的所有拦截器。当我们继续运行此类时，输出显示：

Advisor count: 3
Advisor type: org.springframework.aop.interceptor.ExposeInvocationInterceptor
Advisor type: org.springframework.transaction.interceptor.TransactionInterceptor
Advisor type: org.springframework.aop.aspectj.AspectJMethodBeforeAdvice

由此可见，代理中不仅包含TransactionInterceptor ，还包含AspectJMethodBeforeAdvice 。

重要的是要知道，这不应影响已经尝试执行相同操作的任何实现。这应该使所有一直在等待“最后一个负责任时刻”解决该问题的人的生活更加轻松。 :)

PS与上一篇文章一样，我在此示例中包括了项目的存档，因此您可以在需要时查看其余的代码。
