Spring Integration 4.3 M2可用

发布
阿尔特姆·比兰（Artem Bilan）
2016年4月26日

我很高兴地宣布Spring Integration 4.3.0.M2现在可以从Spring里程碑存储库中获得 。这个版本关闭了大约50个JIRA并且包含了将近100次提交 。

感谢所有贡献的人。特别是对Spring Cloud Stream团队而言，其全面的Spring Integration使用会影响项目方向。

自第一个里程碑以来的一些关键功能：

  • 现在可以将Spring Integration运行时对象模型以及组件度量标准公开为graph ，可用于可视化集成应用程序的当前状态。在网络容器中运行时， @EnableIntegrationGraphController注释，以及IntegrationGraphServer bean，创建服务以通过REST协议检索模型和状态，例如JSON表示法：

  • 持久的MessageStore现在支持Lazy-Load的算法（默认启用） MessageGroup恢复。当处理具有聚合等操作的大型持久性组时，这会带来显着的性能优势。

  • 服务激活器现在支持async选项。如果服务返回ListenableFuture和async是true ，调用线程将立即释放，并在该线程上（从您的服务中）发送答复消息，以完成将来的任务。在此基础上，我们还提供了AsyncAmqpOutboundGateway和async模式JmsOutboundGateway下游流在回复侦听器容器线程上运行。

2016年4月19日

在先前的Reactive Spring和Reactor Core 3.0博客文章之后，我想根据我们在处理Spring Framework 5即将到来的Reactive支持时所学到的知识，来解释为什么Reactive类型很有用，以及它们如何与其他异步类型进行比较。

为什么要使用反应型？

响应类型并非旨在让您更快地处理请求或数据，实际上，与常规阻止处理相比，它们会带来较小的开销。它们的优势在于能够更有效地并发处理更多请求，并能更有效地处理具有延迟的操作，例如从远程服务器请求数据。它们通过本机处理时间和延迟，而无需消耗更多资源，从而使您能够提供更好的服务质量和可预测的容量计划。与等待结果的同时阻塞当前线程的传统处理不同，等待的Reactive API不花费任何成本，仅请求它能够处理的数据量并带来新的功能，因为它处理数据流，而不仅仅是处理单个元素。一个。
