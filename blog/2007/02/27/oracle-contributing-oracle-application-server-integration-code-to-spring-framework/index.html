Oracle将Oracle应用服务器集成代码贡献给Spring Framework

工程
罗德·约翰逊
2007年2月27日

关于包含Spring的应用服务器的主题，另一个更新。Oracle一直在致力于与其应用程序服务器的增值集成。

这类似于WebLogicJtaTransactionManager中的WebLogic 8.1及更高版本。在OC4J环境中，应使用OC4JJtaTransactionManager代替通用JtaTransactionManager，它具有以下优点：


  • 直接访问事务管理器和帮助程序类，而无需使用
    JNDI查找

  • 自动检测服务器版本以充分利用不同版本中的不同事务管理器实现

  • 控制事务隔离级别：JTA中没有的非常有用的功能

对于熟悉JTA的人来说，使用UserTransaction就像在Java EE中以编程方式控制事务时所做的那样，存在一些巨大的差距，考虑到近十年来J2EE最初设想几乎没人愿意做的现在已经过时的假设，这也许可以理解。没有EJB的事务管理。

问题在于，只有在TransactionManager上才能执行某些操作，例如暂停事务（例如，获取“需要新的”语义）。该接口在JTA规范中已标准化，但是与UserTransaction不同，它没有提供众所周知的JNDI位置或其他获取它的方式。使用JTA根本无法控制隔离级别或特定于服务器的“事务命名”（用于监视或其他目的）。

因为Spring提供了丰富的，可移植的事务抽象，所以它在驱动JTA和其他API的幕后进行工作，因此您的代码不需要了解基础架构的任何知识，因此可以利用任何专有API来提供更多的控制或更高的效率。Spring同时支持声明式和程序性事务，因此您可以将事务管理分层到对它一无所知的POJO上。或者，如果事务管理是您业务逻辑的一部分，则可以使用更优雅的API，该API提供的功能比JTA多，并且与JNDI无关，并且操作起来不那么冗长。

该代码将进入Spring核心，并随Spring 2.0.3一起提供。感谢Oracle对Spring的持续支持。Spring在其融合中间件平台和SCA策略中扮演着重要角色。当然，Interface21还是SCA的创始合作伙伴 ，我们正在与Oracle，BEA，IBM和其他公司合作进行Spring的SCA绑定。正如Oracle SCA资深人员Greg Pavlik 去年在博客上写道 ，这里有很多好处，Spring应用程序开箱即用即可使用SCA：

最后，从Java程序员的角度来看，一些非常有趣的消息：现在有一个Spring集成，它允许基于Spring的应用程序直接绑定到基于SCA的SOA环境。随着Spring在许多组织中成为构建J2EE应用程序的事实上的标准，我们为这些投资打开了基于透明SCA的集成之门。另外，现在有一个实用的开源故事供Java开发人员使用SCA，而不必担心新的学习曲线或许多新的结构。在Spring中，可能只是POJO：一直到乌龟。我有很多人直接问我有关Java编程和SCA的问题。春天是一个很好的答案。

这使我想起：SCA Java工作组中发生了一些有趣的事情，而Adrian（在Interface21上为此付出了所有艰苦的工作！）或者我们有时间的时候应该发布适当的更新…

Oracle还具有有关HA功能的更多想法。肯定有一些有趣的可能性，尤其是在RAC周围，我们的许多大型企业客户都使用RAC。他们对有关Spring和Oracle技术之间集成的更多想法非常感兴趣，因此请在此处或通过您的Oracle联系人反馈任何想法。Oracle还维护了有关Spring集成的出色资源页面 。
