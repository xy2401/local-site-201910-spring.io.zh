那么与Spring-OSGi有什么关系呢？

工程
Costin Leau
2007年4月5日

欢迎到我的博客！
这是我有史以来的第一次。我设法抵制写博客的冲动，但是由于有很多人鼓励我写关于i21所做的事情，所以我决定尝试一下。这就是Spring-OSGi昨天晚上（EET时区）首次发布的事实。

从去年八月开始，我就参与了Spring-OSGi的工作，这是一个很大的旅程。这是我从事过的最具挑战性的项目之一，我很高兴将它作为里程碑发布给公众。非常感谢大家参与制作这一点，特别是我的团队队友-阿德里安，安迪和哈尔！

在这篇文章中，我想让您了解一下Spring 1.0 M1目前提供的功能。我将跳过OSGi的介绍，因为互联网上有很多优秀的资料（请参阅底部的链接）。

Spring-OSGi的基本思想是简化在OSGi环境中构建/编写/部署Spring应用程序的过程。也就是说，拥有Spring提供的基于POJO的综合编程模型（IoC，AOP，服务抽象），可以在动态执行环境中透明地工作，该环境注重版本控制和模块化。

采用OSGi时面临的最大挑战之一是应对其动态特性。服务（是简单的对象实例）来来往往，您的应用程序必须处理这些。解决方案不是直接的，视情况而定，并且像异常处理和事务一样需要整个应用程序范围。由上述模块强制实施的类加载限制，与AOP结合使用，可能会引起很多麻烦，并迫使开发人员创建黑客，从而使OSGi提供的好处荡然无存。
这些只是我们在Spring-OSGi中要解决的问题的一些示例，最终，这些示例应允许顺利采用OSGi。

让我们看一下在1.0 M1中可以找到的一些功能：

  1. OSGi应用程序上下文

OSGi基于捆绑软件 ，而捆绑软件仅是带有一些专用清单条目的罐子。它们是导出和导入类包和/或服务的模块，单元。
一个应用程序可以由一个或多个捆绑软件组成。Spring-OSGi提供了基于捆绑软件及其生命周期构建的应用程序上下文，使您可以访问应用程序所在的OSGi上下文，OSGi 自定义范围以及额外的 Aware接口。与其他接口一样，该接口也提供了进行依赖关系查找的功能，在完全支持OSGi服务依赖关系注入之前，您应该三思而行。

  1. 资源抽象

类加载不是OSGi中曾经使用的类-例如，类路径具有不同的含义，因为它可以由多个包组装而成（反过来，可以在多个类路径中使用它们）。因此，由于运行环境发生了很大变化， getClass（）。getResource（）可能会有不同的结果。这是您在寻找课程时可能会得到的示例：

春分 ： bundleentry：//5/my/package/MyClass.class （也可以是bundleresource：// ）
Knopflerfish ： 包：//13/my/package/MyClass.class
Felix ： 包：//18.0/0/my/package/MyClass.class

依托URL方案是不可移植因此在春天的OSGi做的第一件事就是通过简单的封装低级别的访问，但有效的资源接口，所以无论你使用，你可以找到你的文件是什么OSGi实现。此外，还可以进行样式样式查找，例如myFolder / *（实际上，我们使用/ META-INF / spring / *来检测“由弹簧驱动”的束）。

  1. 动态服务支持

假设您具有以下应用程序上下文：

大多数应用程序都有几层是OSGi捆绑包的极佳候选层，因为一个层可以简单地将DAO类放在一个捆绑包（dao捆绑包）中，将服务层放在另一个捆绑包（服务捆绑包）中，因此在更新DAO实现时（例如上面的badPerformerDAO已替换为excellentPerformerDAO ），或者正在部署不同版本的应用程序，无需重新启动应用程序：这是选择OSGi的最佳原因之一！

但是，要利用OSGi功能，这些对象必须成为服务-也就是说，必须在对象（客户）必须“消费”它们之前在OSGi平台上注册这些对象。这是一种类似于SOA的方法，避免了模块之间的紧密耦合，因此当捆绑包关闭时，捆绑包发布的服务就会消失。这意味着首先必须使用OSGi API进行注册和查找，但还必须处理由于服务来回而导致的故障。

Spring-OSGi允许在不进行任何代码更改的情况下直接在应用程序中导出和导入任何类型的对象，从而在此领域提供了极大帮助。

为了使Spring-OSGi适应OSGI环境，必须添加两行配置：

  1. 指示框架寻找什么OSGi服务
  2. 将现有的bean导出为OSGi服务

显然，可以对dataSource依赖项执行相同的操作-将其外部化为OSGi捆绑包，只需将直接引用替换为osgi：reference即可 。没有新的API可以处理，也没有尝试/捕获/最终解决的异常，尤其是内置的查找行为。
可以指示Spring-OSGi以便除非找到实现daoInterface的服务即可启动应用程序上下文-也就是说，可以满足dao依赖关系。而且，在运行时，如果服务消失，Spring-OSGi将根据您的配置（重试次数和超时）自动寻找新的实现：如果在拥有包更新时正是在dao bean上调用了调用（例如升级，而不是得到一个讨厌的，显然是莫名其妙的异常goodPerformerDAO到excellentPerformerDAO），一会就搞定造成新的服务查找潜移默化延迟。与往常一样，行为已完全配置。

在某种程度上，导出器/导入器功能类似于Spring远程处理 ，区别在于不涉及远程处理-一切都在同一个VM中运行，并且从未涉及任何序列化。

  1. 整合测试

测试非常重要（甚至至关重要），尤其是在将应用程序迁移到新环境中时，因为很多事情理所当然会失败：我们在开发的早期就已经经历了这一点。这是一个很大的问题，因为在谈论OSGi时，测试绝非易事或自动化，因为必须启动执行环境（无需标准API）并设置（安装捆绑软件，测试取决于）。但是，棘手的是，测试本身必须经过OSG标准化-放置在声明依赖项的清单旁边，并放入捆绑包中，然后将其安装并启动到OSGi平台中。

认识AbstractOsgiTests ＆ co ：


public class SimpleIntegrationTests extends AbstractConfigurableBundleCreatorTests
{

  public void testInstalledBundles() {
    // get access to the OSGi bundle context
     Bundle[] bundles = getBundleContext().getBundles();

     getBundleContext().installBundle(someBundleLocation);
     assertEquals(bundles.length()+1, getBundleContext().getBundles().length());
  }

  // specify the bundles to install  
  protected String[] getBundles() {
        return new String[] {
                "org.springframework.osgi, commons-collections.osgi, 3.2-SNAPSHOT",
                "org.springframework.osgi, org.springframework.osgi.test.simple.service,1.0-SNAPSHOT"};
    }
}

AbstractOsgiTests建立在JUnit之上，因此您可以直接从IDE编写和运行OSGi集成测试。整个设置由测试基础结构处理，因此您不必：无需为测试编写MANIFEST.MF ，进行任何打包或部署-一切都会自动处理。而且速度非常快！实际上，在Spring-OSGi代码内部花费了不到10％的启动时间-其余的时间由OSGi平台本身使用。我们的大多数集成测试均在1-3秒内完全执行。均支持Equinox，Knopflerfish和Felix。

好吧，我认为对于第一个条目就足够了……在以后的条目中，我将详细介绍Spring-OSGi功能。我希望我已经使您的好奇心足以使1.0 M1旋转（请注意，这是第一个里程碑，它具有一些“粗糙”的特征）。

谢谢阅读！
