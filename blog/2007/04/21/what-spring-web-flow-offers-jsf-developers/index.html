Spring Web Flow为JSF开发人员提供什么

工程
基思·唐纳德
2007年4月21日

Spring Web Flow与Spring Framework本身非常相似，是一种独特的集成技术。我们的大多数用户将其视为可以嵌入在任何环境中的通用ApplicationController 。我们支持基于Servlet和Portlet的应用程序，并与领先的Web框架Struts，Spring MVC和Java Server Faces集成。我知道甚至有一些团队在Flex环境中使用Spring Web Flow。在以上每种环境中，Spring Web Flow都集成在一起，以提供更好的模型来实现导航逻辑和管理应用程序状态。

我们的用户之所以喜欢它，是因为他们可以编写一次控制流，然后在任何地方重复使用。在这个日新月异的Web框架中，Spring Web Flow为它们提供了一个现代的框架来学习和构建知识，工具和周围的扩展。它从一开始就被设计为扮演这个角色，我很高兴看到Web Flow集成在多个层面上都在增长。

Java Server Faces（JSF）社区是我们集成发展的一个重要领域。从Spring Web Flow 1.0.3开始 ，我们的JSF集成与Spring社区的期望相提并论，并提供了JSF开发人员最需要的东西。该博客将说明集成增强功能，以向您展示Spring Web Flow对JSF开发人员的影响。

适合

Spring Web Flow是一个控制器框架，提供了用于在Web应用程序中实现用户界面控制流的语言和运行时。Java Server Faces是一个用户界面组件框架，由一个标准API和两个实现组成-Sun参考实现和Apache MyFaces。作为JCP支持的标准API规范，JSF提供了扩展点，允许产品供应商插入并竞争其扩展。当用作JSF扩展时，Spring Web Flow承担两项职责：处理视图导航规则和管理与正在进行的用户交互（也称为对话）相关的状态。这种集成将Web Flow在导航和状态管理方面的优势与JSF的优势相结合，而JSF则是不断发展的用户界面组件库生态系统。图片中的Web Flow可以像以前一样继续使用所有JSF组件和视图。有了Web Flow，JSF开发人员就可以从功能更强大的导航模型中受益，该导航模型可以缓解传统上与手动管理对话状态相关的麻烦。

我可以不停地讨论Web Flow的特定功能集。我将尝试着重强调对JSF开发人员最重要的功能，而不是那样做。

在导航处理领域，Web Flow提供了：


  • 实现动态导航规则的能力，该规则可在不重启服务器的情况下即时更改
  • 流定义语言中内置了完整的前进，后退，刷新，重定向和递归导航功能
  • 通过流定义概念对导航逻辑进行模块化和封装

基本上，Web Flow可以解决这个可怜的人使用JSF的基本导航功能所遇到的所有问题。正如我们的一位主要用户指出的那样 ，Web Flow可以完全替代JSF的默认“以转发为中心”的导航模型：


Jeremy Grelle：即使在我们使用更简单的页面和菜单的情况下，我也一直在使用SWF替代JSF导航规则。我很高兴能够将其推到极致，因为对于我的团队而言，在多个位置定义导航规则会有点令人困惑。

在状态管理领域，Web Flow引入了一些对话性作用域，以补充JSF的现有请求，会话和应用程序作用域。这些范围是


  • 对话范围-在用户对话期间有效的托管范围
  • 流作用域-在对话中执行流的持续时间内存在的托管作用域
  • Flash作用域-在参与流程的视图期间有效的托管作用域

这些范围是“托管”的，因为Web Flow会自动为您清除它们。对话范围是存储模型状态以在用户交互（例如帐户注册向导）结束时保留的状态的理想选择。事实证明，Flash范围对于集成Ajax库（例如Ajax4JSF和ICEFaces）的用户非常有用，在这些库中，来自同一视图的多个请求是常见的。在这种情况下，Flash作用域允许对来自同一页面的细粒度Ajax请求进行模型状态处理，然后在下一次导航时自动将其清除。

回到我们的主要用户所说的话：


Jeremy Grelle：我发现即使使用JSF中的简单页面，您也经常需要对页面的多个请求进行管理的模型状态……JSF社区中有许多解决此问题的方法（Tomahawk的saveState标记，Shale对话框）框架等），但我个人认为SWF的多个作用域是最可靠，最优雅的解决方案。

该方法

希望到现在为止，您已经对将Web Flow用作JSF扩展所带来的好处有了一个很好的了解。现在，我想详细介绍一下集成的工作原理，然后通过向您展示如何一起使用Web Flow和JSF来结束本文。

首先了解Web Flow如何插入JSF，需要了解基本的Web Flow构造。Spring Web Flow中的控制器功能单元（如帐户注册向导或客户主数据/明细编辑器）称为流程定义 。这种控制器的运行时实例称为流执行 。在运行时启动新的流程执行，以允许单个用户参与与应用程序的对话。流程执行处理选择用户的初始视图，然后响应用户事件以执行应用程序行为并确定要显示的下一个视图。它还管理与用户对话框关联的状态。

将Web Flow与JSF集成的很大一部分是使其流执行生命周期适合JSF生命周期。这部分是通过实现自定义的PhaseListener来实现的，该自定义的PhaseListener可以在客户端请求时处理启动新流程执行，以及在还原JSF视图期间还原现有流程执行。

例如，通过访问如下网址：


http://localhost:8080/accounts/servlet.faces?_flowId=register-account-flow

…您将开始一个新的“注册帐户流”。该请求首先到达FacesServlet，从而启动JSF生命周期。在还原任何视图之前，FlowPhaseListener会启动“ register-account-flow”，最终确定要显示的初始JSF视图。

一旦选择了初始JSF视图，就将其呈现。呈现通常发生在使用Spring Web Flow的默认“ 总是在暂停时重定向”设置的自动重定向之后，该设置将选定的视图与可以安全刷新并在以后没有浏览器警告的情况下导航回到的URL关联起来。

进行渲染时，JSF UI组件可以完全访问在Web Flow的任何对话范围中管理的bean，还可以访问标准JSF会话和应用程序范围。组件值绑定是透明的，这意味着JSF视图开发人员不需要知道在哪个bean中管理对象，他们只需要知道bean的名称即可。通过实现另一个JSF扩展点，即自定义VariableResolver ，实现了此功能。

参与流程执行的视图呈现后，用户可以决定要执行的操作。如果她决定单击浏览器的刷新按钮，那很好-流执行将在其稳定的URL处刷新，并重新呈现相同的JSF视图。如果她决定执行触发同一页面上某些Ajax请求的操作，那也很好-流执行再次自动恢复，并且参与Ajax调用的JSF组件可以以线程安全的方式更新流管理状态无缝（无双关语）。

一旦用户决定调用UI命令（例如通过单击命令链接），就会启动标准JSF回发生命周期。在处理了UI组件验证并更新了模型值之后，将JSF操作结果作为信号发送给已还原流执行的视图状态。流程从那里接管，通过调用适当的应用程序行为并根据流程定义的导航规则选择下一个视图来处理事件。该Web Flow导航步骤由JSF集成中的最后一个关键构造（ 自定义NavigationHandler）处理 。

总而言之，从JSF视图和组件开发人员的角度来看，它只是标准的JSF-不管您是将JSP还是Facelets与XYZ组件提供程序一起使用都无关紧要。Web Flow执行将自动为您恢复，访问会话状态是完全透明的。Web Flow事件只是标准的JSF UI命令结果。从控制器开发人员的角度来看，这都是Web Flow。结果，faces-config.xml中冗长而有限的导航规则被基于更加丰富的流定义语言的模块化，简洁的流定义所取代。最重要的是，您将获得本机JSF组件模型的所有优点以及Web Flow导航模型的所有优点。

入门

在JSF环境中开始使用Spring Web Flow的最佳方法是下载1.0.3版本并部署sellitem-jsf示例，该示例也可用于在线测试 。该示例可作为Eclipse动态Web项目导入，以便在Eclipse中轻松部署。此外，该示例还启用了Spring IDE 2.0，这是一个Eclipse插件，其中包括Graphical Web Flow Editor 。

未来

我们不仅仅是偶然地来到这里，我们还与客户每天在我们的论坛和支持中心一起使用Spring Web Flow和JSF构建应用程序，从而来到这里。特别感谢Jeremy Grelle，James Clinton，Ed Burns，Craig McClanahan和Colin Sampaleanu为将Spring Web Flow JSF集成到今天的坚实基础而做出的所有贡献。

Spring Web Flow 1.1的工作已经开始，并且更高级别的集成是令人兴奋的路线图中的一个主要主题。在JSF方面值得注意的是，我们将在流定义中添加对统一表达语言（EL）的支持，并支持利用Spring 2.0作为所有范围（包括提供的对话范围）的JSF受管bean的全面提供者。由Spring Web Flow提供。这项工作是对新的核心功能的补充，这些新功能将使所有环境中的Web Flow用户受益，例如对对话范围的持久性上下文的支持。参加论坛上的1.1路线图讨论 ，以获取更多信息并参与进来！
