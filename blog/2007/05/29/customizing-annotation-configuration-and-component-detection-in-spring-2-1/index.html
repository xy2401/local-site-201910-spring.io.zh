两个星期前，我在博客中介绍了Spring 2.1的新注释驱动的依赖项注入功能，并提到我将在“本周后期”提供更多信息。事实证明，这有点乐观，但可喜的是，与此同时，该功能已经发展了很多。因此，要遵循此处的示例，您将需要下载2.1-M2官方发行版 （或者，如果您是最早阅读此更新条目的人之一，并且M2尚不可用，则您至少应每晚进行构建＃ 115，您可以在此处下载）。

我要演示的第一件事是如何在不使用任何XML的情况下创建应用程序上下文。对于那些使用过Spring的BeanDefinitionReader实现的人，这看起来非常熟悉。但是，在创建上下文之前，我们在类路径上需要一些“候选” bean。继续上一个博客中的示例，我有以下两个接口：Java公用接口GreetingService {String greet（String name）; }``````java公共接口MessageRepository {String getMessage（String language）; } ... ...以及以下相应的实现：Java java @Component公共类GreetingServiceImpl实现GreetingService {@Autowired private MessageRepository messageRepository; public String greet（String name）{Locale locale = Locale.getDefault（）;如果（messageRepository == null）{返回“抱歉，没有消息”； }字符串message = messageRepository.getMessage（locale.getDisplayLanguage（））;返回消息+“” +名称； java @Repository公共类StubMessageRepository实现MessageRepository {Map messages = new HashMap （）; @PostConstruct public void initialize（）{messages.put（“ English”，“ Welcome”）; messages.put（“ Deutsch”，“ Willkommen”）; } public String getMessage（String language）{return messages.get（language）; }}``

现在按照承诺...无需任何XML即可组装这个公认的琐碎“应用程序”：```java Locale.setDefault（Locale。德语）; GenericApplicationContext context = new GenericApplicationContext（）; ClassPathBeanDefinitionScanner扫描程序=新的ClassPathBeanDefinitionScanner（上下文）; Scanner.scan（“博客”）; //参数是'basePackage'context.refresh（）; GreetingService greetingService =（GreetingService）context.getBean（“ greetingServiceImpl”）;字符串消息= greetingService.greet（“ Standalone Beans”）; System.out.println（消息）;结果是：Java Willkommen独立Beans

本质上，这与使用新的“ context”名称空间中的组件扫描 XML元素时完全一样的行为（如我先前的博客所示 ）。但是，我想重点介绍一些较新的功能以及自定义选项。首先，我将从StubMessageRepository中删除@Repository批注，然后重新运行测试以产生以下异常：```java org.springframework.beans.factory。BeanCreationException：创建名称为'greetingServiceImpl'的bean时出错：字段的自动装配失败；嵌套的异常是org.springframework.beans.factory。BeanCreationException：无法自动连线字段：私人博客。MessageRepository博客。GreetingServiceImpl.messageRepository;嵌套的异常是org.springframework.beans.factory。NoSuchBeanDefinitionException：没有[blog类型的唯一bean。[MessageRepository]定义：预期的单个bean，但是找到0```显然@Autowired注释默认情况下指示必需的依赖关系，但是可以通过添加值为'false'的'required'参数来轻松切换，例如： java @Component公共类GreetingServiceImpl实现GreetingService {@Autowired（required = false）私有MessageRepository messageRepository; ...修改后的结果：java抱歉，没有消息。

显然，由于上下文中没有可用的DataSource，因此导致了自动装配失败的连锁反应。但是，作为坚信测试驱动开发的人，我想在建立基础结构之前对实现进行单元测试 。幸运的是，扫描仪是相当可定制的，我可以提供过滤器，例如：```java Locale.setDefault（Locale。德语）; GenericApplicationContext context = new GenericApplicationContext（）; boolean useDefaultFilters = false; ClassPathBeanDefinitionScanner扫描程序=新的ClassPathBeanDefinitionScanner（上下文，useDefaultFilters）; Scanner.addExcludeFilter（new AssignableTypeFilter（JdbcMessageRepository.class））; Scanner.addIncludeFilter（new AnnotationTypeFilter（Component.class））; Scanner.addIncludeFilter（new RegexPatternTypeFilter（Pattern.compile（“ blog \\。Stub。 ; System.out.println（message）;```如您所见，我禁用了'defaultFilters'并明确添加了自己的。在这种情况下，这不是完全必要的，因为默认值包括@Component和@Repository批注，但是我想显示各种过滤选项-不仅包括批注，还包括可分配的类型，甚至是正则表达式。当然，主要目的是为了支持存根而禁用MessageRepository的JDBC版本，根据我的结果，这就是发生的情况：```java Willkommen Standalone Beans```

到目前为止，您已经看到0个候选bean将导致自动装配失败，除非@Autowired的'required'参数设置为false 。鉴于自动装配遵循“按类型”语义，所以无论所需参数的值如何，超过1个bean都会导致失败。例如，将@Repository批注添加回StubMessageRepository并重新运行上一个示例后，我收到以下异常：```java org.springframework.beans.factory。BeanCreationException：创建名称为'greetingServiceImpl'的bean时出错：字段的自动装配失败；嵌套的异常是org.springframework.beans.factory。BeanCreationException：无法自动连线字段：私人博客。MessageRepository博客。GreetingServiceImpl.messageRepository;嵌套的异常是org.springframework.beans.factory。NoSuchBeanDefinitionException：没有[blog类型的唯一bean。定义了[MessageRepository]：期望的单个bean，但是找到了2```可以通过切换到'by-name'语义来解决-通过Spring 2.1对JSR-250 @Resource批注的支持来实现：公共类GreetingServiceImpl实现GreetingService {@Resource（name =“ jdbcMessageRepository”）私有MessageRepository messageRepository; ...```
