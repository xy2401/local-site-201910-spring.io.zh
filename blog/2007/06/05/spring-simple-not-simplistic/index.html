Spring ：简单而不是简单...

工程
阿列夫·阿伦森（Alef Arendsen）
2007年6月5日

在上周的培训中，我第一次使用了Spring Web Services的第一个Release Candidate。自Arjen发布他的珍贵RC1以来，已经过去了不到两个星期，所以很高兴向一些参与者展示这一新产品。

在Web服务部分之前，我们做了一些JMX和远程处理，展示了Spring的导出器功能。您可能知道，这使您只需少量的声明性配置即可将任何Spring托管的bean导出到远程端点或JMX注册表：


<bean id="myService" class="com.mycompany.MyServiceImpl">
    <property name="myDao" ref="myDao"/>
</bean>

<bean class="org.springframework.remoting.rmi.RmiServiceExporter">
	<property name="serviceName" value="myService"/>
	<property name="serviceInterface" value="com.mycompany.MyService"/>
	<property name="service" ref="myService"/>
</bean>

这一切都非常容易，并且与到目前为止观众通常对Spring的看法是一致的：Spring使所有以前使用的东西变得非常简单。它只需要几行XML（或我们当前正在实现的其他一些配置选项）就可以了。这当然是正确的，但是我不想看到人们离开Core Spring培训的印象。

幸运的是，Arjen以Spring Web Services的形式给我提供了一些很棒的资料，一劳永逸地告诉我们，我们希望您想到Spring的方式与“完全不同，它只是添加几行XML，然后我们完成了。不像那样！

回到上周我进行的培训；当我开始平常谈论Web服务时，我们才刚完成JMX。简短的版本基本上说： Web服务！=远程处理 ，您最好学会忍受这样一个事实，那就是，这意味着实现Web服务必须要做的事情！=将服务导出到RMI端点所需要做的事情 （我怀疑很可能是！=在所有情况下都可以用>代替。

我和Arjen正在撰写一篇文章，将对此进行更详细的解释。在此之前， Spring Web Services参考手册和Arjen的个人博客应该为您提供有关所有这些背后原因的大量信息。

上周令我感到非常惊喜的是，一位参加培训的学员突然打断了我，说： “嘿，我知道，Spring的方法不一定是简化我的代码或减少代码量，这是要消除不必要的复杂性，让我专注于重要的事情。” 。

这个家伙（托马斯，是的，是你:)），真是太不可思议了。Spring Portfolio的每个部分都致力于做到这一点：

依赖注入和AOP致力于提供一种干净而强大的方法来注入依赖并实现交叉约束行为。模块化和关注点分离是很容易实现的事情，是非常重要的问题 。Spring的DI容器和AOP设施专注于使您实现模块化和良好的关注点分离。他们并不是主要专注于简化代码。我几乎可以说这是使用DI和AOP的结果之一，这是一个很好的副作用；-)

Spring的远程导出器旨在提供一种从远程位置调用服务器端代码的几乎透明的方式。当想要实现基于RMI的远程端点时，我们暗含的是说我们希望将服务与客户紧密耦合。Spring的远程导出器并不是仅用四行XML来设计的。他们旨在获得这种几乎透明的编程体验。同样，它只能用四行XML来完成，这是一个很好的副作用。

现在最重要的是……当我们实施Web服务时，暗指我们希望将我们的服务与可能正在使用我们的服务的任何潜在客户进行松散耦合 。我们不是（也不应该）正在寻找一种透明的方式来使用和标记发出方法调用。版本管理，灵活性和保持向后兼容的能力是实施功能强大的Web服务的关键，该Web服务经受了时间的考验，并将客户与我们的服务松散地联系在一起。Spring Web Services旨在解决这些问题并使您能够非常轻松地解决它们。这并不是要简化您的代码！尽管我非常确定，在许多情况下，使用Spring Web Services时最终得到的代码将尽可能地简单易懂！

上周五，我带着灿烂的笑容走到车上。我们结束了这次培训，理解了最重要的一课： Spring并不是为了简化代码而设计的；它旨在让您专注于重要的事情。作为副作用，您可以简化代码，这当然是我们都可以肯定的！
