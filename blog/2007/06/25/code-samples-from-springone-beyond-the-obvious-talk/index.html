SpringOne的代码示例“超越显而易见”

工程
乔里斯·奎珀斯
2007年6月25日

上周在SpringOne上，我和Alef进行了关于使用Spring处理复杂应用程序的演讲。在这种情况下，复杂性在结构和动态两方面都得到了考虑。至于演讲的结构性部分，我在之前的博客文章中介绍了该话题。动态部分说明了一些可能的解决方案，以解决部署环境之间的差异。 （测试，验收，生产等）
很多人问我是否可以向他们提供我在谈话中进行的演示的来源。我已将源代码附加到此博客条目中，并将简要说明这一切如何工作。希望这次演讲本身最终会在今年晚些时候在parleys.com上发布，因此您可以获取有关该主题的更多背景信息。

环境意识：支持您的运行时环境

在演讲中，我认为，通过使用按环境生成部署单元的自定义构建脚本来处理环境之间的差异有几个缺点。其中之一是，当应用程序从例如接受环境升级到生产环境时，您将无法重用构建工件（war或ear文件）。为了解决这些缺点，您必须使您的应用程序环境敏感 ：提出某种方法来确定您的应用程序在哪种环境中运行，然后配置您的应用程序以对此做出反应。

有几种方法可以做到这一点：在Spring应用程序中，启用依赖于环境的行为的明显方法是为每个需要非默认配置的环境使用不同的bean定义。我已经展示了三种方法：


  • 首先是为某些环境加载额外的上下文配置文件。通过使用bean定义继承和覆盖，您可以在配置时执行适当的操作。当您使用ContextLoaderListener配置Web应用程序上下文时，这需要一些额外的代码，这些代码显示在所附示例的Web项目中。顺便说一句，这个想法应该归功于我的一位前同事（以斯拉，谢谢！）
  • 第二种是使用JavaConfig而不是XML来定义与环境有关的bean。然后，确定要创建哪些bean的位置将从确定要加载到JavaConfig类的配置文件中移出。
  • 最后一个示例结合了前两个示例，以确定要使用的JavaConfig类，其中为每个环境创建了扩展公共基本配置的子类。这展示了在实际的类中进行配置的强大功能：可以使用常规的Java继承和多态性，这模糊了bean定义继承和类继承之间的区别。

例如，其他方式可能包括自定义FactoryBean实现。

样本

这些示例演示了如何使用简单的服务和dao来通过名字查找一个人的姓氏。在默认情况下，仅定义两个人（Alef和I）。在开发环境中，我们将使用静态dao实现，而不是始终返回John Doe的实现 。在接受环境中，定义了第三人（我们的同事Arjen）。集成测试检查我们的应用程序在每种环境中是否显示了正确的行为。

有几种方法可以检查应用程序在哪个环境中运行：对于这些示例，我使用了EnvironmentUtils类正在使用的系统属性。例如，其他方法可能是检查数据库或JNDI注册中心。

而不是只拉上了我的Eclipse工作区中，我创建了一个Maven的示例 更新 ：戴夫Turanski亲切更新的例子来春3.0：一些人曾试图说服我最近给Maven的第二次机会，我从它转过身去之后几年前令人反感。我必须承认，Maven 2实际上比Maven 1好很多，尽管用于Web项目的Eclipse工作区工件的生成可以使用一些改进（例如对WTP 2的支持）。我也错过了其他一些东西，但这可能只是我完全缺乏经验。好消息是，我现在不必将所有依赖项打包在示例中。

安装Maven之后，您可以通过从根目录运行“ mvn test”来运行示例测试。要从样本生成Eclipse项目，请运行以下命令：'mvn -DdownloadSources = true eclipse：eclipse'。要构建罐子和战争，请使用“ mvn软件包”或“ mvn安装”将构建工件添加到本地存储库。

战争包含一个可通过'/ [context-root] / env'访问的servlet：要查看每种环境的不同，请使用一个名为'app.env'的系统属性启动servlet容器，该属性包含环境字母（D，T ，A或P）。因此，将-Dapp.env = D添加到服务器的启动行将导致该应用程序在开发模式下运行，这意味着返回的所有姓氏均为“ Doe”。
