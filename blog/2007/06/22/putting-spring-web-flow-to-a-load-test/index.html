将Spring Web Flow进行负载测试

工程
罗森·斯托扬切夫（Rossen Stoyanchev）
2007年6月22日

对Web Flow应用程序进行负载测试类似于对任何其他Web应用程序进行负载测试-我们将使用负载测试工具来模拟并发客户端访问级别的提高，以捕获基本的性能统计信息。

使用Web Flow，负载测试将有两个重要的注意事项：

  1. 负载测试必须维护独立的“ cookie存储区”，以便每个客户端请求都可以承载独立的HTTP会话。
  2. 我们需要一种机制，用于从初始响应中提取唯一的流执行密钥，并使用它来定制同一流会话中的后续请求。

Apache JMeter是一个开放源代码性能测试工具，可以满足这两个方面的考虑。

对于1），我们在每个行使Web Flow功能的测试组的根部添加一个HTTP Cookie Manager元素。Cookie管理器确保每个模拟的客户端请求可以具有独立于其他客户端请求的自己的cookie，从而允许servlet容器通过jsessionid cookie跟踪独立的HTTP会话。

对于2），我们在启动流程的HTTP请求元素之后立即添加一个正则表达式提取器。Extractor的目的是解析HTTP响应，使用我们提供的正则表达式查找某些文本，并使该文本可用作变量供后续的HTTP Request元素使用。这是一个示例正则表达式提取器配置：

Reference Name: flowExecutionKey
Regular Expression: name="_flowExecutionKey“ value=”(.*)"
Template: $1$
Match No.: 0

通过以上配置，我们现在可以将变量$ {flowExecutionKey}嵌入到同一流会话的后续HTTP请求元素中。




现在，我们使用它来对Web Flow进行负载测试。为了正确地行使代表性的Web Flow功能，我创建了一个示例Web Flow应用程序，以模仿6步购物车流程，以收集用户输入的送货地址，送货选项，信用卡，账单地址，订单确认以及订单摘要。结束。该流程中的各个步骤包括数据绑定和验证，视图状态，操作状态，决策状态和子流程状态-我们希望在典型的Web Flow应用程序中找到这些内容。但是，该应用程序使用存根而不是实际的数据库访问代码来避免在整体统计信息中包含此类数字。在此测试中，我们仅关注Web Flow。

构建应用程序并创建JMeter脚本后，我添加了一个Aggregate Report元素来记录具有不同负载水平的测试的性能统计信息。

使用运行Ubuntu和Apache Tomcat 5.5版的Lenovo T60双核笔记本电脑作为配置为最多150个并发连接的servlet容器，我观察到以下结果：

实际负载测试应在实际硬件上并根据实际用例进行。没有替代品。但是，我们可以从上述数字得出某些结论。

以上数字表明，即使同时使用的用户数量大大增加，使用核心Web Flow功能时吞吐量仍保持稳定。90％用户的响应时间不到一秒钟。随着负载的增加，最差的响应时间会增加，但是考虑到用于测试的硬件不足，这不足为奇。

使用以上技术，您可以负载测试您自己的Web Flow应用程序。
