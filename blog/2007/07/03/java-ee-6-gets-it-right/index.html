Java EE 6实现正确

工程
罗德·约翰逊
2007年7月3日

Java EE 6提案 （JSR 316）于今天发布。我相信，这将是自该平台近10年来发布以来最重要的版本，该技术的用户应该对此表示欢迎。Interface21很高兴成为此JSR的支持者，我期待为它做出贡献。

Java EE（在其大部分历史中被称为J2EE）在创建Java中间件市场方面发挥了重要作用。但是，在过去的十年中，该平台出现了一些重要问题，例如：



  • 对Java EE兼容服务器的需求是一系列功能，而绝大多数用户对此并不感兴趣


  • 自从设想了J2EE以来，企业需求已发生变化，而“一种尺寸适合所有模型”这一事实越来越不适合了。

  • 框架的出现（尤其是开源的框架）极大地增强了企业Java的事实，这些框架使开发人员更加高效，其生产应用程序更加高效和可维护。


  • 诸如Ruby on Rails甚至.NET之类的新挑战表明，在快速变化和创新的时代，舒适的2-3年发布周期危及整个平台

Java EE 6是该平台的重要修订版，具有解决所有这些问题的潜力。

它还有可能解决另一个问题：如果EE供应商需要针对大多数客户永远不会使用的大量功能进行认证，这意味着他们很难跟上规格，这是事实。进行稳定升级是一项挑战，而且重要的是，Java EE市场上新进入者的可能性为零。最后一点是有关的，因为让EE服务器成为老牌运营商的特权并不符合用户的利益。要证明进行新平台发布的困难：在这一点上，据我所知， BEA是唯一获得Java EE认证的市场领导者，尽管Java EE 5规范已经确定了几个月。Java EE 5的最有价值的新部分（例如JPA）已经在最终发行版的WebLogic中准备了好几个月，无法在GA产品中发布，而有关某些技术的问题已解决，大多数WebLogic用户可能永远不会碰到。

我的解释足够多：让我们直接去看一下提案，并从消息来源那里听听：

在过去的8年中，Java EE平台已经成长和成熟，现在可以满足各种企业和Web应用程序开发需求。此外，Java EE平台为与该平台一起使用的其他技术，框架和应用程序营造了一个生机勃勃的社区和市场。其中一些提供了平台中缺少的功能。其他提供平台设施的替代方案。此发行版的主要主题是作为Java EE整体格局的一部分来包含和支持这些技术，同时还将继续简化平台以更好地面向更广泛的开发人员。为此，我们为此发行版提出了两个目标- 可扩展性和配置文件 。

几年前，这本来是异端。当时我是一个异端，但是我从来不想成为一个异端。Java EE最终考虑到了更大的前景，而不是想像它会为所有用户需求提供良好的解决方案。在Java EE 6中，部分重新诠释了EE平台，将其作为协议的中心点，以支持解决方案的选择：肥沃的土壤，使数千朵鲜花绽放。令人鼓舞的是，Sun看到了这种更大的构想：考虑他们对JRuby的拥抱，并认识到像EE平台一样，JVM是生态系统的基础，而不是局限于一种语言。

Java EE平台不能无限制地包含Web和企业应用程序开发人员所需的所有有趣和有用的技术，这是不合适的。取而代之的是，我们认为希望使更多的这些技术能够干净地分层或插入Java EE应用服务器。通过添加更多的可扩展性点和更多的服务提供商接口，这些其他技术可以干净高效地插入平台实现中，并且对于开发人员而言，就像在平台中内置的功能一样易于使用 。

再次，很棒的东西。与服务器供应商长期使用这些“有趣和有用的技术”来混淆用户并阻止创新相比，该平台附带的技术与服务器基础结构“更好地集成”的谬论。（幸运的是，几乎所有的公司都已经停止这样做了很长时间：例如，考虑IBM在Spring on WebSphere认证中的作用 ）。

这里的关键是创新。有些技术自然会以不同的速度发展：一方面，尤其是服务器基础结构和有线协议（变化相当缓慢，并受益于标准化）；而不是说在它需要更改的方式坐在上面更快的速度和标准化已经在很大程度上相对于失败“有趣，有用”的技术。

Java EE平台的影响范围是如此之广，以至于失去了最初的关注点。为了将Java EE平台重新聚焦于特定类别的开发人员和应用程序，我们建议引入Java EE平台配置文件。概要文件将引用JCP流程定义的Java EE平台，并且可能包括Java EE平台技术的子集，不属于基本Java EE平台的其他JCP技术，或两者。除了定义基本的Java EE平台之外，该规范还将定义在Java EE Profiles中引用Java EE平台技术的规则。

个人资料是向前迈出的重要一步。最后，在用户开始构建应用程序的两年之前，用户将有权购买所需的东西，而不是规格委员会想要的东西。现在是时候以健康的竞争取代苏联式的指挥经济。继续我先前的观点：在以前的版本中，EE平台试图指示您如何构建应用程序，就像苏联五年计划一样，而在EE 6中，该平台的作用更像法律框架中的作用。一个西方国家，可以确保人们可以竞争并作为个人自由行动，同时就所有人的利益达成游戏规则。

显然，企业Java技术的用户已经确定了许多概要文件。例如，当今的Web应用程序，SOA应用程序和金融中间件对服务器基础结构的要求有很大不同，尽管Java EE的不同部分为每个应用程序提供了价值。尤其是到目前为止，批处理和无头中间件用户已被Java EE所忽略。最后，在Java EE 6的潜力之内，他们的希望已经显现。

概要文件的使用是解决Java EE平台规模不断扩大的一种工具。同样，Java EE平台中包含的某些技术不再像引入平台时那样具有相关性。需要一种方法，以谨慎有序的方式从平台“修剪”这些技术，以最大程度地减少对使用这些技术的开发人员的影响，同时使平台变得更加强大。

我希望这是可以遵循的。假设您有一台Java EE 5服务器（您的供应商可能会或无法在可接受的时间内为您提供）。您是支持EJB CMP 2.0甚至1.1的产品的骄傲拥有者。还记得CMP 1.1中的公共实例变量吗？您仍然可以享受它们。这是一个丑陋的史前史，值得忘记，而不是使今天的产品膨胀。如果那里仍然有任何应用程序在运行，它们可以在较旧的服务器上爬出它们存在的最后几年，直到有人摆脱困境。

读提案“ EJB CMP-有效地被Java Persistence代替”是很棒的。关于此版本的一个重要主题是使EE在2007年的世界中变得更加重要，消除或制造可选的旧故障就很好地说明了这一点，对于将产品引入其中的用户和供应商而言，它们具有真正的实际好处。

Interface21期待与Sun以及EE 6总体小组和相关专家小组的其他公司和个人合作，以使EE 6成为一个强大的平台。

如我所说，我从不想成为异端。从本质上讲， 没有EJB的J2EE的愿景（我与Juergen Hoeller共同撰写）不是废除J2EE，而是通过诚实对待桶中的烂苹果以及通过创新来解决标准化的需要来帮助J2EE繁荣发展。Java EE 6提议似乎允许这种情况发生，我很高兴我可以加入这个聚会。让我引用那本书，其中大部分是大约四年前写的。请注意，重点不仅在于批评EJB，还强调真正重要的是J2EE的概况：

J2EE仍然是一个相对较年轻的技术。不完美并不奇怪。现在是时候评估一下它的工作原理和效果不佳的地方，以便我们消除负面影响并享受正面影响。因为J2EE包含很多内容，所以从本质上讲，这意味着识别提供最大价值的J2EE子集，以及我们需要最有效地利用它的一些补充基础结构。...您可能会想，“没有EJB的J2EE还剩下什么？”答案是：很多。J2EE不仅仅是EJB。许多J2EE开发人员对此表示反对，当他们在您的办公桌上看这本书时，他们会告诉您，但是对EJB的功能以及J2EE的总体功能的冷淡分析表明，EJB只是更大更重要的一部分。 。J2EE本质上是关于标准化一系列企业服务的，例如命名和目录服务（JNDI），事务管理提供可能跨越不同事务资源（JTS和JTA）的标准API，与遗留系统的连接（JCA），资源池以及线程管理。J2EE的真正力量在于这些服务，并且这种标准化为行业提供了出色的服务。

我相信支持Java EE 6（鉴于其目标）与我对J2EE的立场是一致的，J2EE在没有EJB和其他地方的J2EE中表示。例如， 没有EJB的J2EE的另一个中心主题是标准化应发挥的作用，以及在各个框架之间进行竞争以促进创新的更好方法：

尽管开放（或至少部分开放）规范过程是积极的，但我认为J2EE相对于.NET的最大优势之一是Java / J2EE开源软件的丰富性...如我们所见，关于J2EE架构的正统智慧与实际经验不符。某些J2EE规范在较小程度上也是如此。我觉得我们在J2EE平台的发展中处于重要的十字路口。它显然需要发展和创新才能生存和繁荣。然而，对基本企业基础结构（如JTA，JAXB，JDBC和Java语言本身）的规范意味着，在不破坏采用一致的，标准的方法来解决最棘手的低级问题的好处的前提下，可以在如何访问该基础结构方面进行创新企业软件。

当我在2003年写下这些字眼时，我没想到Java EE专家组最终会使用术语“可扩展性”来捕捉相同的想法。这是一个惊喜。

我相信企业Java社区应该欢迎Java EE 6，并且应该欢迎Sun愿意与时俱进，并做出选择来增强整个企业Java平台。J2EE / Java EE有很多优点，但是其中一些问题使它难以理解。Java EE 6应该改变这一点！
