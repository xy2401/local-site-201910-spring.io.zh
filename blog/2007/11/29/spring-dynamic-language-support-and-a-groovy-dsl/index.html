Spring动态语言支持和Groovy DSL

工程
戴夫·瑟
2007年11月29日

自从Spring 2.0中引入Spring动态语言支持以来，它一直是Groovy的有吸引力的集成点，并且Groovy提供了丰富的环境来定义域特定语言（DSL）。但是Spring参考手册中Groovy集成的示例范围有限，并且没有显示Spring中针对DSL集成的功能。在本文中，我将展示如何使用这些功能，并作为示例，我们使用Grails发行版中的Groovy DSL将bean定义添加到现有的ApplicationContext中。

Groovy Beans

Spring动态语言集成的基本功能在XML的“ lang”命名空间中公开。您可以做的最直接的事情是将Spring组件定义为Groovy Bean，位于单独的文件中或XML的内联中。 Spring参考指南（ http://static.springframework.org/spring/docs/2.5.x/reference/index.html ），因此我们无需赘述过多，但为完整起见，我们不妨看一个简单的示例。

假设我们有一个Java接口

public interface Messenger {

	String getMessage();

}

这是Groovy中的内联bean定义，它实现了接口

<?xml version="1.0" encoding="UTF-8"?>
<beans xmlns="http://www.springframework.org/schema/beans"
	xmlns:lang="http://www.springframework.org/schema/lang"
	xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
	xsi:schemaLocation="http://www.springframework.org/schema/lang http://www.springframework.org/schema/lang/spring-lang-2.5.xsd
	http://www.springframework.org/schema/beans http://www.springframework.org/schema/beans/spring-beans-2.5.xsd">

	<lang:groovy id="messenger">
<![CDATA[
class GroovyMessenger implements spring.Messenger {

	def String message;
}
]]>
	</lang:groovy>

</beans>

请注意，由于Groovy为所有属性定义了公共getter和setter，因此我们实际上不需要显式编写getMessage（）方法。还要记住，Spring动态语言支持的一个功能是内联Groovy代码也可以被拉到一个单独的源文件中（使用lang：groovy元素的script-source属性）。

Spring动态语言支持的另一个功能是脚本可以超越仅定义类的范围。您还可以编写一个Groovy脚本进行一些处理，最后返回对象的实例。例如，如果我们已经有一个名为JavaMessenger的Messenger实现：

<lang:groovy id="messenger">
<![CDATA[
def messenger = new JavaMessenger("Hello World!")
messenger
]]>
</lang:groovy>

这具有通过特定消息公开JavaMessenger实例的效果-一个简单的示例，但是是查看所公开功能的好方法。使用这种技术可以使我们超越Spring中常规的bean创建模式，并在返回对象之前进行脚本中所需的尽可能多的处理。

Spring在后台创建了groovy.util的实例。脚本，其run（）方法在脚本末尾返回对象。当我们开始考虑如何集成DSL时，这将变得很重要。

自定义Groovy对象

要进入DSL领域，我们需要研究的下一个功能是在Groovy对象作为Spring组件公开之前对其进行自定义的能力。我相信，Rod Johnson和Guillaume Laforge在接近Spring 2.0版本开始时的会议（不是2.0版本）中的一次会议之后增加了此功能。Guillaume对领域特定语言的兴趣使他发现，Spring在处于任何人有机会使用Groovy对象（或其类）之前能够操纵并向Groovy对象（或其类）添加行为的有利位置，因为Groovy是一种动态语言。这是一个非常有力的习语。

他们想出的机制是GroovyObjectCustomizer接口，可以在Groovy对象暴露给Spring容器客户端之前将其应用于Groovy对象。该界面如下所示：

public interface GroovyObjectCustomizer {

	void customize(GroovyObject goo);

}

并在实例化之后（如果是脚本）将其应用于Groovy对象，然后再运行它。这使我们可以在释放对象之前使用对象的方法和属性进行游戏。

要应用定制器，我们要做的就是在Groovy bean定义中添加对它的引用：

<lang:groovy id="messenger" script-source="classpath:..." customizer-ref="customizer"/>

<bean id="customizer" class="..."/>

领域特定的语言-BeanBuilder

Grails对于Spring组件有一个不错的DSL，称为BeanBuilder（请参阅此处以了解更多详细信息）。它使我们能够以一种自然而简洁的方式在Groovy中构建Spring ApplicationContext。根据Graeme Rocher的说法，在最新版本的Grails中，BeanBuilder也可以在不依赖于Web框架的情况下工作-您只需在类路径上使用Grails Core和Groovy。所以，现在是一个好时机，看看我们是否能够整合与Spring的BeanBuilder（如也已指出，在春季论坛在这里 ）。（实际上，如果没有Servlet API和Spring webflow jar，我实际上无法使该示例与Grails 1.0-rc1一起使用，但可能可以在rc2或1.0 final中使用。）

Groovy中使用领域特定语言的表达式通常采用闭包的形式，因此自然可以使用Spring集成中的Script模式来定义闭包。对于BeanBuilder，它看起来像这样：

<lang:groovy id="beans">
<![CDATA[
beans = {
	messenger(JavaMessenger) {
		message = "Hello World!"
	}
	// ... more bean definitions here ...
}
]]>
</lang:groovy>

这将产生一个Script对象，该对象本身返回一个包含bean定义的闭包（称为“ beans”）。Bean的定义之一是我们的朋友Messenger。我们理想的情况是希望能够获得这些bean定义并将它们与当前ApplicationContext合并。为此，我们将需要使用GroovyObjectCustomizer。

一个基本的GroovyObjectCustomizer

这是定制程序的基本内容，它将从脚本化的Groovy对象获取闭包并从中创建应用程序上下文：

public class BeanBuilderClosureCustomizer implements GroovyObjectCustomizer {

	public void customize(GroovyObject goo) {
		createApplicationContext(goo.run())
	}
	
	private ApplicationContext createApplicationContext(Closure value) {
		BeanBuilder builder = new BeanBuilder()
		builder.beans(value)
        builder.createApplicationContext()
	}

}

它对它创建的应用程序上下文不做任何事情-只是创建它并使它蒸发。它还不执行任何错误检查，但我们可以稍后添加。定制程序是用Groovy编写的，因此我们可以直接调用goo.run（）而不转换为Script。

改进的GroovyObjectCustomizer

现在，让我们改进实现，以便将Bean定义从BeanBuilder传递到封闭的ApplicationContext。

public class BeanBuilderClosureCustomizer implements GroovyObjectCustomizer {

	public void customize(GroovyObject goo) {
		addbeanDefinitions(createApplicationContext(goo.run()))
	}
	
	private void addBeanDefinitions(ApplicationContext context) {
		DefaultListableBeanFactory scriptBeanFactory = context.autowireCapableBeanFactory
		for (name in  scriptBeanFactory.getBeanDefinitionNames()) {
			BeanDefinition definition = scriptBeanFactory.getBeanDefinition(name)
			applicationContext.autowireCapableBeanFactory.registerBeanDefinition(name, definition)
		}
	}

    // createAppicationContext defined here....
}

有什么可能更简单？

至此，我们可以加载以下Spring配置：

<beans>

	<lang:groovy id="beans" customizer-ref="customizer">
<![CDATA[
beans = {
	messenger(JavaMessenger) {
		message = "Hello World!"
	}
	// ... more bean definitions here ...
}
]]>
	</lang:groovy>

	<bean id="customizer" class="BeanBuilderClosureCustomizer"/>

</beans>

然后拿出使者并使用它。在示例（请参阅附件）中，我们让Spring 2.5 TestContextFramework负责创建ApplicationContext并将依赖项注入测试用例（因此无需任何依赖项查找）。

使用当前上下文作为父级

作为使BeanBuilderClosureCustomizer更加有用的最后一项调整，我们将对其进行修改，以使用封闭的ApplicationContext作为BeanBuilder中Bean定义的父级。为此，我们只需要在定制器中引用父级，因此我们需要实现ApplicationContextAware并使用该引用来构造BeanBuilder：

public class BeanBuilderClosureCustomizer implements GroovyObjectCustomizer,
		ApplicationContextAware {

	def ApplicationContext applicationContext;

	public void customize(GroovyObject goo) {
		addbeanDefinitions(createApplicationContext(goo.run()))
	}
	
	private ApplicationContext createApplicationContext(Closure value) {
		BeanBuilder builder = new BeanBuilder(applicationContext)
		builder.beans(value)
		builder.createApplicationContext()
	}

    // addBeanDefinitions defined here....
}

由于BeanBuilderClosureCustomizer是用Groovy编写的，因此我们不需要为applicationContext属性定义显式的getter和setter-它们是由Groovy自动生成的。

BeanBuilderClosureCustomizer现在可以使用了（可能还会进行一些其他的错误检查）。Groovy的真正好处在于，它可以作为JVM字节码在jar文件中进行编译和运送。因此，我要做的就是确保在打包项目时附带生成的类文件。该示例通过将Groovy bean编译到与Java编译器使用的目标目录中来完成此操作。

在父上下文中引用Bean

在我们的Groovy DSL中的父上下文中引用bean也很整洁。 Grails允许我们已经通过使用BeanBuilder DSL中的“ ref”关键字来做到这一点，例如

<lang:groovy id="beans" customizer-ref="customizer">
<![CDATA[
beans = {
	messenger(JavaMessenger) {
		message = ref("helloMessage")
	}
	// ... more bean definitions here ...
}
</lang:groovy>

在这里，我们从父上下文中的bean定义中加载了消息。

样本项目

要运行该示例，只需解压缩zip文件 ，或使用Eclipse将其导入到现有工作区中（文件->导入…->现有项目…）。如果您具有Eclipse的m2插件，则应立即使用。如果不是，则可以使用m2 Eclipse插件生成Eclipse元数据（“ mvn eclipse：eclipse”）。如果您不使用Maven或Eclipse，则只能靠自己，但可以在pom.xml中找到顶级项目依赖项。

由于项目在单元测试中使用JSR-250批注进行依赖项注入，因此您将需要该API。最简单的方法是使用Java 6运行和编译。例如在* NIX命令行上

$ JAVA_HOME=<path-to-JDK-1.6> mvn clean test

脚注：实际上，当我在上面说到我可以加载包含内联脚本的配置时，我撒谎了-由于在2.5.1中已修复的错误（请参见JIRA ），因此在Spring 2.5中不起作用。解决方法（根据示例）是使用外部文件存储脚本。
