具有命名切入点的AOP上下文绑定

工程
本·黑尔
2007年3月29日

Spring AOP中有很多新功能，包括AspectJ切入点语言， 命名空间，以及@AspectJ语法支持。但是到目前为止，最强大的方面之一（原谅双关语）是AOP上下文绑定。

例如，假设您要建议采用String作为参数的方法。


public interface HelloService {
	String getHelloMessage(String toAddHello);
}

为了建议这种方法，您需要编写一个切入点，以查找String返回类型， HelloService接口的所有实现以及getHelloMessage（String）方法。


@Before("execution(public java.lang.String aop.HelloService+.getHelloMessage(String))")

但是，如果您想在内部应用需要String参数的建议该怎么办？


public void filter(String input) {
	if (obscenities.contains(input)) {
		throw new IllegalArgumentException("Obscenities such as '" + input + "' will not be tolerated!");
	}
}

这就是AOP上下文绑定起作用的地方。事实证明，Spring支持相当多的AspectJ切入点语言。这里涉及到我们的部分是args（）运算符。该运算符可以允许我们选择一个参数并将该参数绑定到建议中。因此，当切入点和建议结合在一起时，这就是您所看到的。


@Before("execution(public java.lang.String aop.HelloService+.getHelloMessage(String)) && args(input)")
public void filter(String input) {
	if (obscenities.contains(input)) {
		throw new IllegalArgumentException("Obscenities such as '" + input + "' will not be tolerated!");
	}
}

现在，这很酷。您可以从建议的方法到建议中对参数进行强类型和命名绑定。在更复杂的示例中，可以绑定多个上下文片段，例如其他参数，被调用的对象以及更多的建议。我要指出的一件事是，它使我无时无刻不在感到沮丧，因为args（）运算符中的参数名称与advice方法中的参数名称相对应。

这种特定配置的问题在于，大多数情况下，您不会使用嵌入式切入点创建建议声明。通常，您会将切入点外部化为命名切入点。为此，您需要引入另一个间接级别。也就是说，命名切入点后跟建议定义。


@Pointcut("execution(public java.lang.String aop.HelloService+.getHelloMessage(String)) && args(helloInput)")
public void helloService(String helloInput) {}



@Before("helloService(input)")
public void filter(String input) {
	if (obscenities.contains(input)) {
		throw new IllegalArgumentException("Obscenities such as '" + input + "' will not be tolerated!");
	}
}

在此示例中要注意的重要一点是，需要使用命名的切入点将上下文类型作为参数，以便可以将参数传播到建议。您会看到helloService（String）带有字符串，因此Before建议可以引用helloService（input） 。

现在，有些人可能会问为什么还要使用AOP上下文绑定。的确，您只需在通知中添加JoinPoint参数，Spring AOP就会自动将其绑定。该对象将包含所有参数返回值，等等，您可以通过上下文绑定获得。但是，在处理上下文信息时，您真正得到的只是Objects和Object []项目。您仍然最终会强制转换并处理潜在的异常。使用上下文绑定，您仅获得所需的内容（无需在JoinPoint数据结构中随意查找），并且该数据是强类型的。更少的代码==更少的维护==更少的成本！（但这就是AOP的全部意义，对吧？）
